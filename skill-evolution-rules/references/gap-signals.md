# Skill 缺口信号

## 先区分三类问题

### 1. 业务问题

典型特征：

- 需求本身没讲清
- Bug 证据不够
- 测试环境没准备好
- 代码逻辑本身没写对

处理原则：

- 回到需求域、Bug 域、编码域或测试域
- 不要误判成 skill 缺口

### 2. 工程问题

典型特征：

- 工具链报错
- 依赖没装好
- 环境变量缺失
- CI、编译、运行环境异常

处理原则：

- 先解决工程环境问题
- 不要把环境问题包装成 skill 演进

### 3. Skill 缺口问题

典型特征：

- 已经命中某个 skill，但它没有覆盖当前稳定高频场景
- skill 只有抽象原则，没有足够执行流程
- skill 没说明边界，导致与相邻 skill 来回推诿
- skill 没给出判定标准，执行结果无法收口
- skill 没给出 references，复杂场景只能临时补口头规则
- 同一个空档在多个任务里反复出现

处理原则：

- 启动 `skill-evolution-rules`
- 明确目标 skill、缺口类型和最小回补方式

## 缺口类型清单

- `description` 不准：自动触发信号过宽、过窄、容易误命中
- 流程不全：没有“先做什么、默认流程、阻断条件”
- 边界不清：与相邻 skill 重叠或互相漏接
- `references` 不足：正文太轻，但没有补充细则
- 归档缺失：不知道结果该写到哪里
- 验收缺失：补完后不知道如何判断 skill 已经可用

## 阻断级判定

属于阻断级：

- 不补 skill 就会直接写错代码、测错方向或归档错位置
- 同一个 gap 已经影响当前任务的正确推进
- 当前 gap 会导致多个 skill 冲突，且总控层无法稳定裁决

不属于阻断级：

- 只是表述还可以更精炼
- 只是低频细节还没补例子
- 当前任务能在不引入错误的前提下继续完成，gap 可在收尾时补
