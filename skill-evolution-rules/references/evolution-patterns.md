# Skill 演进模式

## 用途

用于快速判断当前 skill 体系调整属于哪一种模式，避免一上来就直接重写目录或正文。

## 五类演进模式

### 1. 新增

适用场景：

- 现有 skill 无法承接某类稳定且高频的任务。
- 新规则已经形成独立职责，继续塞进旧 skill 会造成边界混乱。

输出重点：

- 新 skill 名称
- 触发 description
- 与相邻 skill 的边界

### 2. 拆分

适用场景：

- 一个 skill 同时承担多个阶段或多个位点职责。
- 一个 skill 的 `description` 已经过宽，容易误触发。
- 一个 skill 的 `references/` 继续增长后已无法维护。

输出重点：

- 拆成几个 skill
- 每个 skill 各自负责什么
- 原 skill 中哪些文件要迁出

### 3. 合并

适用场景：

- 两个或多个 skill 长期重复触发、规则重复、边界过近。
- 维护成本明显高于保留独立 skill 的收益。

输出重点：

- 合并后的统一职责
- 哪些旧 skill 退休
- 合并后如何避免 description 过宽

### 4. 迁移

适用场景：

- 某些规则继续保留在当前 skill 中已经不合适，但不值得单独建 skill。
- 某些 references 更适合进入相邻 skill。

输出重点：

- 迁移源
- 迁移目标
- 迁移后需要同步更新哪些引用入口

### 5. 退休

适用场景：

- 某个 skill 已被更稳定的 skill 完全取代。
- 某个 skill 对应的研发流程已不再使用。

输出重点：

- 退休原因
- 替代 skill
- 历史 references 是否保留
