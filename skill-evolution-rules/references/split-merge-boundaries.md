# 拆分与合并边界

## 用途

用于判断一个 skill 是应该继续保持独立、继续细分，还是应该并回相邻 skill。

## 应该拆分的信号

- 一个 skill 同时覆盖需求、Bug、编码、测试等多个阶段。
- 一个 skill 的 `description` 需要靠长篇枚举才能说明触发条件。
- 一个 skill 的 `SKILL.md` 已经开始承担多个小 skill 的职责。
- 一个 skill 的 references 按阶段、语言、位点不断膨胀，已经难以按需读取。

## 不应轻易拆分的信号

- 新增的规则只是现有 skill 的一个小检查点。
- 该规则没有稳定触发信号，只是偶发补充说明。
- 拆分后会让两个 skill 几乎总是一起触发，失去独立意义。

## 应该合并的信号

- 两个 skill 长期重复触发，且职责几乎一致。
- 两个 skill 的 references 大量重叠。
- 保留独立 skill 只会增加维护成本，不会提高触发准确性。

## 不应轻易合并的信号

- 合并后会把 `description` 变得过宽。
- 合并后会把不同阶段、不同位点或不同问题类型重新混成一团。
- 合并后会让小 skill 被总控层或大 skill 吞没。
