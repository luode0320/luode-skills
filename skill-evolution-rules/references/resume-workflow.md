# 回补后恢复流程

## 目标

让“发现 skill 缺口 -> 回补 -> 继续当前任务”形成固定闭环，而不是补完后上下文再次漂移。

## 恢复顺序

1. 先完成目标 skill 的修改。
2. 再校验该 skill 的基础结构是否通过。
3. 如果修改影响了总规划、总清单或路由，再同步更新这些文件。
4. 重新加载被修改的 skill。
5. 回到原任务，但不要从中间随意续写，而是从受影响的步骤重新执行。
6. 重新确认当前 gap 是否已经消失，再继续编码、测试或交付。

## 常见恢复点

### 需求域

- 从需求接入或缺口识别重新开始
- 不直接沿用旧的模糊结论继续实现

### Bug 域

- 从复现、静态定位或运行时诊断的受影响步骤重新开始
- 不直接沿用旧的猜测性结论继续修

### 编码域

- 从当前位点 skill 的规则重新过一遍
- 必要时回退到“先审再写”的状态

### 测试域

- 从测试策略或功能验证入口重新过一遍
- 不直接沿用旧的测试资产组织和结论

## 验收重点

- 当前任务是否不再因为同一 skill gap 卡住
- 当前 skill 是否能更稳定地命中正确场景
- 当前 skill 与相邻 skill 的边界是否更清晰
