# Skill 演进决策矩阵

## 默认优先级

1. 先补现有 skill 的 `description`
2. 再补现有 skill 的流程、边界或判定标准
3. 再补现有 skill 的 `references/`
4. 只有前面三种都承接不住，才新增独立 skill

## 决策表

| 场景 | 默认动作 | 说明 |
| --- | --- | --- |
| 触发信号写得不准，导致经常误命中或漏命中 | 补当前 skill 的 `description` | 不要急着新建 skill |
| 主流程已有，但复杂场景缺细则 | 补当前 skill 的 `references/` | 这是最常见的回补方式 |
| 当前 skill 缺“先做什么 / 阻断条件 / 通过标准” | 补当前 skill 正文 | 优先完善现有骨架 |
| 与相邻 skill 经常边界打架 | 同时补两边边界，必要时由总控层补路由 | 先清边界，不默认拆新 skill |
| 当前场景高频、稳定、长期独立，现有 skill 明显承接不住 | 新增独立 skill | 必须能说清职责边界 |
| 只是一次性特例，未来基本不会重复 | 不新增 skill，只记录处理说明 | 避免 skill 膨胀 |
| 问题本质仍是业务不清、Bug 未定位或测试条件不足 | 不改 skill，回主域处理 | 这是业务问题，不是 skill 问题 |

## 何时建议新增独立 skill

必须同时满足：

- 场景高频，不是偶发
- 职责稳定，不会频繁漂移
- 与相邻 skill 的边界能写清
- 单纯补现有 skill 会让它职责明显膨胀

只满足其中一部分时：

- 优先补现有 skill

## 最小化修改原则

- 能补一段 `description`，就不要先拆 skill
- 能补一个 `references/*.md`，就不要先重写整个 `SKILL.md`
- 能在相邻两个 skill 补边界说明，就不要先新增第三个 skill
- 只有当前结构已经稳定失衡，才建议拆分
