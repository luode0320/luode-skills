# 本地材料回溯工作流

## 适用条件

当没有真实持久记忆后端时，改用本地材料做近似历史回溯。

## 建议顺序

### 1. 先查 `ment/`

优先看：

- 需求文档
- 需求补齐文档
- 需求变更文档
- 验收文档

这些通常最接近当时的业务语境。

### 2. 再查 Bug / 测试 / 交付文档

重点看：

- Bug 分析与修复记录
- 功能验证和回归测试说明
- 交付摘要和评审结论

### 3. 再查 Git 历史

优先确认：

- 什么时候改过类似代码
- 哪些提交最可能与当前问题有关
- 当时改动范围和说明是什么

### 4. 最后查仓库分析文档

例如：

- 总规划文档
- 巡检文档
- 复盘文档
- 其它根目录说明

## 输出要求

- 明确标注这是“基于本地文档 / Git 记录回溯得到”的结论。
- 指出核心依据来自哪些文件或哪些提交。
- 如果材料不完整，直接说明不确定点。

## 边界提醒

- 本地回溯只能近似补历史，不等于真实永久记忆。
- 如果当前仓库历史已经缺失，不要编造完整故事。
