# Props 与事件契约规则

## props 设计

- props 名称要表达语义，不要用模糊缩写
- props 尽量稳定，避免把一大堆零散布尔值堆成状态爆炸
- 如果多个 props 总是一起出现，考虑合并为更清晰的对象或语义模式

## events / emits / callbacks

- 事件命名要表达“发生了什么”，不是“父组件该怎么做”
- 子组件负责发出事实，父组件决定后续动作
- 不要在子组件里偷偷修改父层状态来源

## slots / children

- 只有确实需要可插拔结构时再开放 slots / children
- 不要为了“灵活”把组件 API 做成难以理解的万能容器

## 常见反模式

- props 层层透传五六层
- 回调命名模糊，如 `onChangeData`、`onHandle`
- 同时开放太多定制口子，最后没人知道标准用法是什么
