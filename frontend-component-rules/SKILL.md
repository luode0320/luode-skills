# 前端组件规则

只在需要判断前端组件应该怎么拆、状态该放哪、组件职责是否清晰时使用这个 skill。
如果当前主要问题是页面视觉风格、落地页叙事或整体 UI 气质，请转交 `frontend-skill`；如果是通用代码最小改动和风格问题，请交给编码基线域。

## Skill 作用与适用场景

- 规范前端组件的拆分边界和职责单一性。
- 控制 props、状态、事件和样式的承载层次。
- 避免把页面、数据、交互和样式全部挤进一个组件。
- 让前端规则也能以独立 skill 形式被命中，而不是只靠页面设计种子。

## 自动触发信号

- 新增或修改 React / Vue / Web 组件、组合组件或页面内局部模块。
- 组件 props 过多、状态归属不清、事件层层透传或复用边界混乱。
- 准备把页面逻辑拆组件，或把组件合并 / 下沉 / 上提。
- 评审指出某个组件职责混杂、难测试或难复用。

## 进入后先做什么

1. 先确认当前组件到底承担展示、交互编排、状态管理还是数据装配。
2. 划清哪些状态属于局部组件，哪些应上提或下沉。
3. 检查 props、事件、样式和副作用是否已经越界。
4. 决定拆分、合并还是保持现状，并说明依据。

## 默认执行流程

1. 默认先读 `references/component-splitting.md`，先判断组件该不该拆、按什么维度拆。
2. 如需继续展开，再读 `references/component-state-boundaries.md`，需要决定状态归属、props 和事件边界。
3. 需要对照边界或正反例时，再读 `references/component-examples.md`，需要对照组件拆分正反例。
4. 输出 组件边界建议、状态归属结论和需要调整的组件结构。
5. 组件结构稳定后，再回到实际前端实现；若页面整体视觉方向仍有争议，可并行参考 `frontend-skill`。

## 权责边界与不负责事项

- 只负责组件层结构，不替代 `frontend-skill` 处理页面视觉设计。
- 不为了“组件化”而过度抽象，造成简单页面也被拆碎。
- 不把全局状态、页面状态和局部交互状态混放而不解释。
- 不绕开最小改动原则做大面积无关前端重构。

## 需要暂停并确认的条件

- 组件职责本身还不清，拆分结论无法稳定。
- 当前页面结构和数据流大幅波动，过早组件化会频繁返工。
- 为了通用复用提前抽象，但真实复用场景并不存在。
- 组件边界调整会牵动大量无关页面，超出本次任务范围。

## 执行通过 / 驳回标准

- 通过：组件职责清晰、状态归属明确、props 和事件传递可解释，并且没有明显的过度抽象或职责混杂。
- 驳回：组件继续承担多重职责、状态边界混乱、props 透传失控或为了复用做了空泛抽象。

## 执行结果归档要求

- 如组件边界调整影响多个页面或公共组件，应将结论记录到 `analysis/` 或评审记录。
- 归档内容至少包含组件职责、状态归属、拆分理由和影响范围。
- 局部简单组件调整可不单独归档，但应在评审中可追溯。

## references 读取规则

- 默认先读 `references/component-splitting.md`。
- 只有在 判断状态边界和通信方式 时，再读 `references/component-state-boundaries.md`。
- 只有在 对照样例或判断是否过度抽象 时，再读 `references/component-examples.md`。
