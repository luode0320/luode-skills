# 发布上线规则

只在准备把变更真正推向生产、灰度环境或对外可见流量时使用这个 skill。
如果当前实现和测试还没收口，请先留在编码审查域或测试域；如果只是整理发布说明，请转交 `delivery-summary-rules`。

## Skill 作用与适用场景

- 规范上线前检查、灰度节奏、放量条件和回滚预案。
- 把发布动作与测试动作、交付说明动作拆开。
- 降低上线后不可控风险，避免无门槛直接全量投放。
- 让交付域具备真正的“发布收口”能力。

## 自动触发信号

- 代码和测试已基本完成，准备发布、灰度、上线或分批放量。
- 需要确认开关、配置、数据库、缓存、任务开跑和监控观察点。
- 需要决定先灰度还是直接发布、何时放量、何时回滚。
- 用户明确要求给出上线检查表和回滚条件。

## 进入后先做什么

1. 先确认测试结论、发布前置条件和环境准备已具备。
2. 识别本次发布涉及的高风险点、开关点和观测点。
3. 确定发布节奏，是直接上线、灰度放量还是分批投放。
4. 写清回滚触发条件和执行路径。

## 默认执行流程

1. 默认先读 `references/release-checklist.md`，先检查上线前最少需要确认哪些事项。
2. 如需继续展开，再读 `references/rollout-risk-controls.md`，需要设计灰度和放量节奏。
3. 需要对照边界或正反例时，再读 `references/rollback-gates.md`，需要明确何时必须回滚以及回滚门槛。
4. 输出 发布检查结论、放量策略和回滚门槛。
5. 发布策略确认后，交由具体发布流程执行；若上线前发现验证缺口，应回流测试域；上线后交付说明由 `delivery-summary-rules` 收口。

## 权责边界与不负责事项

- 只负责发布与灰度规则，不代替测试域补验证。
- 不代替 `git-collaboration-rules` 处理提交、分支和 PR 整理。
- 不把交付总结和发布动作混成一份清单。
- 不在回滚条件不清时强行推进上线。

## 需要暂停并确认的条件

- 关键测试未完成或结论仍不稳定。
- 上线依赖的配置、开关、数据初始化或监控点未准备好。
- 没有明确回滚门槛和回滚路径。
- 高风险变更却计划直接全量上线，且没有观察窗口。

## 执行通过 / 驳回标准

- 通过：能够明确说明上线前检查是否通过、采用什么放量策略、观察什么指标，以及何种条件下必须回滚。
- 驳回：上线策略只有“发上去看看”，没有前置检查、没有灰度控制，也没有回滚口径。

## 执行结果归档要求

- 将发布检查、灰度计划和回滚门槛记录到 `delivery/`、上线单或 `analysis/`。
- 归档内容至少包含发布时间窗、发布方式、关键风险点、观察点和回滚条件。
- 如果发布计划临时调整，应保留变更原因和决策记录。

## references 读取规则

- 默认先读 `references/release-checklist.md`。
- 只有在 设计放量节奏和风险控制 时，再读 `references/rollout-risk-controls.md`。
- 只有在 判断回滚条件或对照样例 时，再读 `references/rollback-gates.md`。
