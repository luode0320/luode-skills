# 代码归位审查规则

只在“代码已经写完，需要确认它是不是放对地方、归对层、没有越层依赖”时使用这个 skill。
如果当前是编码前还没决定该放哪一层，请优先用 `package-structure-rules`，不要把前置设计问题拖到这里才发现。

## Skill 作用与适用场景

- 检查文件归属、模块归属和目录落点是否合理。
- 检查跨层调用、依赖方向和职责归位是否正确。
- 识别“位置看起来合理，但职责实则错位”的伪通过情况。
- 检查 `utils` / `common` / `global` / `middleware` 是否按子目录分层，避免根目录直接堆实现文件。
- 检查 Go `internal/service` 是否按业务子目录分层，避免在根目录堆业务实现文件。
- 对 Go 项目检查 `*_test.go` 是否出现在 `test/` 之外，防止违反测试落点基线。
- 检查是否存在 500 行及以上且持续膨胀的业务文件，避免把多职责长期堆在单文件中。
- 作为编码审查域进入测试前的最后一道结构关口。

## 自动触发信号

- 新增文件、移动文件或扩展模块。
- 出现跨层调用、工具类落点争议或目录归属争议。
- 编码完成后需要确认实现是不是放对了层、对了目录。
- Go 代码改动涉及测试文件，需确认 `*_test.go` 没有落在禁放目录。
- Go 业务实现持续增加，怀疑 `internal/service` 出现大平层堆叠。
- 单文件超过 500 行且本轮仍继续新增方法或流程分支。

## 进入后先做什么

1. 先识别当前文件属于入口层、业务层、数据层、公共层还是测试层。
2. 再检查它与相邻目录和模块的职责是否一致。
3. 确认依赖方向是否越层或反向。
4. 检查 `utils` / `common` / `global` / `middleware` 根目录是否直接出现实现文件。
5. 检查 Go `internal/service` 根目录是否直接出现大量业务实现文件，而未按子目录分层。
6. 对 Go 项目执行禁放扫描：确认 `test/` 外不存在 `*_test.go`。
7. 检查本轮目标文件是否已达 500 行及以上且仍持续堆功能，判断是否应拆分到多个文件或子目录。
8. 判断是否存在“文件位置正确但逻辑职责错误”的情况。

## 默认执行流程

1. 默认先读 `references/placement-checklist.md`，检查文件和模块归属。
2. 如果涉及依赖方向和跨层调用，再读 `references/dependency-direction.md`。
3. 如果需要对照正反例，再读 `references/placement-examples.md`。
4. 若为 Go 项目，补充执行 `*_test.go` 禁放扫描并记录命中路径。
5. 对本轮主要变更文件补充检查行数和职责密度，识别是否命中“500+ 行持续膨胀”风险。
6. 输出归位结论、越层问题和需回改的文件。
7. 归位未通过时，不进入测试域。

## 权责边界与不负责事项

- 只负责进入测试前的归位复查，不代替 `package-structure-rules` 做编码前结构设计。
- 不代替 `implementation-review-rules` 做实现质量自审。
- 不代替 `cleanup-format-review-rules` 做噪音清理。
- 不代替测试域判断功能是否正确。

## 需要暂停并确认的条件

- 当前代码跨越多个层级，无法单点纠偏。
- 纠正归位会显著扩大本次改动范围。
- 模块职责本身在项目里就有冲突，无法直接判定应该跟随哪一侧。
- 当前问题已不是“归位复查”，而是更上层的架构调整。
- Go 场景扫描到 `test/` 外存在 `*_test.go`，但是否改 seam 方案尚未达成一致。
- 500+ 行文件拆分会引发跨模块调用改造，且当前迭代窗口无法承接完整拆分时。

## 执行通过 / 驳回标准

- 通过：文件、模块、目录和依赖方向都与职责一致，没有明显越层或错位；对 500 行及以上且持续膨胀的文件已完成拆分或给出本轮可执行拆分方案。
- 驳回：位置、职责、依赖方向存在明显错位，`utils` / `common` / `global` / `middleware` 根目录直接放实现文件，Go `internal/service` 根目录直接堆业务实现文件，或 Go 项目中 `test/` 之外存在 `*_test.go`；或 500 行及以上文件继续堆功能却没有拆分动作/方案；即使功能能跑也不能视为通过。

## 执行结果归档要求

- 将归位结论、越层问题和回改建议记录到 `review/`。
- 归档内容至少包含检查对象、归属结论、依赖方向和下一步动作。
- 如果本次归位检查无争议，可以简化记录，但应保留结论。

## references 读取规则

- 默认先读 `references/placement-checklist.md`。
- 只有在判断依赖方向时，再读 `references/dependency-direction.md`。
- 只有在对照正反例时，再读 `references/placement-examples.md`。
