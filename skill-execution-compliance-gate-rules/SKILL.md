# Skill 执行完整性闸门规则

只在“skill 已触发但执行闭环可能不完整”时使用这个 skill。
目标不是重复实现业务，而是在收口前补齐“可执行规则未落实时的可行动反馈”。

本 skill 与 `code-change-finalization-gate-rules` 是同源拆分：本 skill 负责“其他已命中 skill 是否被完整执行”这条通用合规链，代码/测试改动的最终收口闸门（注释双 skill、测试目录、implementation-review 收口、真实运行验证、router 提交前检查、用户手改保护）由 `code-change-finalization-gate-rules` 负责，两者互不重复检查。

## Skill 作用与适用场景

- 对本轮已触发的 skill 做末端合规检查，确认是否存在可执行但未执行的规则。
- 当无法一次性补齐原执行计划内必需规则或阻断级缺口时，不强行“假完成”，而是记录并校验可交接的事实。
- 把“静默跳过必需规则”变成“可校验的计划内缺口或共享阻断事实”，避免用户误以为已全部收口。
- 真实 `blocked/manual_handoff` 时，只校验 `artifact-delivery-gate-rules/references/task-blocker-closure-contract.md` 的完整性、去重与唯一渲染条件；面向用户的阻断区块和解决计划仅由 `reasoning-summary-structure-rules` 渲染。
- 非阻断时，不得因本 skill 生成任何扩散性后续。
- 收到用户明确结束指令（如“结束”“停止”“到此为止”“不要继续”“不要下一步建议”“不要扩散”）时，结束指令优先于默认建议模板；没有阻断级缺口时直接最小收口，且不得输出下一步区块或等待类占位文案。
- 若当前运行环境存在 goal / plan / task 等显式状态收口机制，且主任务已满足原始用户目标或已满足该机制的阻断条件，必须确认已执行对应真实收口动作；未执行时不得给“已完成 / 等待用户新指令”结论。Codex goal 的 `get_goal` / `update_goal` 仅是其中一种特例。

## 自动触发信号

- 本轮任务已命中某个 skill，且出现“只做了部分规则”的反馈或风险。
- 用户明确提出“检查是否按 skill 全部执行”或“为什么只按部分规则实现”。
- 任务准备结束，但仍存在原执行计划内未完成必需项或阻断项；若已经没有三类合法后续，则不得触发后续内容输出。
- 多个 skill 同时命中，可能出现边界冲突导致部分规则被跳过。
- 本轮发生非预期执行失败、退出成功但产物不可信，或对已知失败进行了恢复尝试。
- 本轮发生任一工具调用，且最终回复需要给出可核验的 skill 执行证据。
- 当前运行环境存在显式状态收口机制，需要确认是否已经真实收口。

## 进入后先做什么

1. 列出本轮实际命中的 skill 清单，不要臆造未命中的 skill。
2. 为每个 skill 判断“当前阶段应执行的规则”，只检查当前阶段可落地项。
3. 按 `references/applicability-and-gap-check.md` 识别“可执行但未执行”的规则缺口。
4. 识别计划内未完成必需项与真实阻断候选，判断缺口是否阻断当前任务正确性。
5. 若已真实收口为 `blocked/manual_handoff`，校验共享契约的状态、阶段、依据与证据、已尝试与停止条件、影响、1-3 条解决计划、恢复验证 / 重入点；不得在本 skill 内另行生成用户文案。
6. 合并主任务状态与缺口状态；真实阻断只交由 `reasoning-summary-structure-rules` 按共享契约渲染，非阻断时不存在原执行计划内未完成必需项或用户显式建议请求则强制无下一步。
7. 若当前运行环境存在显式状态收口机制，额外核验是否已经完成真实的状态收口动作；若没有，则本轮仍不得视为完成。此项与代码改动无关，由本 skill 统一承接，`code-change-finalization-gate-rules` 不重复判断。

## 默认执行流程

1. 收集证据：本轮编辑、测试、日志、结论说明。
2. 识别规则：从目标 skill 中提取当前阶段应执行规则（尤其是含“必须/默认/先做/通过标准/阻断条件”的规则）。
3. 逐条核对：判断每条规则是“已执行 / 未执行但可执行 / 当前阶段不适用”。
3.1 若出现非预期执行失败，核对 `execution-failure-learning-rules` 是否完成分类、active 案例检索、有限重试、同输入复验、candidate/active 状态判断和业务 Bug/Skill gap 分流。
4. 对测试相关任务执行禁放扫描：检查禁放目录是否存在 `*_test.go`（默认禁放目录为 `doc/5-tests/` 之外路径）。
5. 形成校验结论：分别标记“计划内必需项”“真实阻断候选”与“用户显式要求的建议”；不得在本 skill 生成用户可见的恢复计划。
6. 若真实状态为 `blocked/manual_handoff`，校验共享阻断事实满足契约且本轮最终回复只由 `reasoning-summary-structure-rules` 输出一个阻断区块；同根因不得重复计划。
7. 输出给用户：本 skill 只报告 PASS/FAIL 与缺失字段；默认不补充“后续内容”、阻断区块或解决计划。不存在计划内未完成必需项或用户显式建议请求时禁止输出“等待用户新指令”“无需继续动作”等占位文案。
8. 若本轮发生任一工具调用，最终回复必须包含 `Skill 执行证据`（命中项、执行动作、对应产物/命令）；缺失则判定失败并触发一次自动补救重试（先补证据后结束）。
9. 若当前运行环境存在显式状态收口机制，则最终回复前必须确认：不是只写了“后续内容”或完成文案，而是已经真实执行了对应的状态收口动作。

## 阻断判定与处理

属于阻断级：

- 缺口会直接导致实现错误、测试结论失真或归档位置错误。
- 不补该规则就无法继续当前阶段的正确推进。
- 本轮发生工具调用但最终回复缺少 `Skill 执行证据`，且未完成一次自动补救重试。
- 当前运行环境存在显式状态收口机制，但原始用户目标已完成后未真实执行对应完成收口，或满足 blocked / failed / cancelled 条件后未真实执行对应状态收口。

属于非阻断级：

- 缺口不影响当前结论正确性，但应在后续迭代补齐。
- 可先完成当前任务收口；只有用户明确要求建议/backlog，才把这类内容作为后续内容输出。

## 权责边界与不负责事项

- 只负责其他已命中 skill 的执行完整性闸门与合法后续内容判定，不代替具体业务实现。
- 不强制要求一次性补齐所有历史遗留规则，优先处理当前阶段可执行缺口。
- 不把“当前阶段不适用”的规则误报为未执行缺口。
- 不输出冗长核对报告，除非用户明确要求展开明细。
- 不因为执行本 skill 而丢弃其他来源的主要建议。
- 本轮存在代码/测试改动时的最终收口检查（注释双 skill、测试目录、implementation-review 最低收口、真实运行验证、router 提交前风格检查、用户手改保护）由 `code-change-finalization-gate-rules` 负责，本 skill 不重复检查。

## 执行通过 / 驳回标准

- 通过：已完成其他已命中 skill 的执行完整性核对（含缺口分类、blocked/manual_handoff 共享契约校验、Skill 执行证据、运行时状态真收口核验）；若存在原执行计划内未完成必需项、阻断项，或用户明确要求后续建议/backlog，则输出合并后的 1-3 条后续内容；若不存在三类合法后续，则强制无下一步并直接结束。
- 驳回：只说“还有规则没做完”，但没有给出可执行的计划内必需动作或阻断恢复动作；或本轮发生任一工具调用却未给出 Skill 执行证据也未完成自动补救重试；或当前运行环境存在显式状态收口机制却只输出完成文案而未真实执行对应收口动作。
- 驳回：原始用户目标已完成，或用户明确要求结束，却仍默认追加“下一步状态 / 下一步建议 / 等待用户新指令 / 任务已完成，无需继续动作 / 当前无需补充动作”等占位或扩散性后续。

## references 读取规则

- 默认先读 `references/applicability-and-gap-check.md`。

## 回到主流程的重启点

- 若为阻断级：先执行“后续内容”中的高优先级动作，再从当前阶段入口重新开始。
- 若为非阻断级：当前任务可收口；若用户未显式要求建议/backlog，则直接结束，不默认追加后续区块。

## 输出要求（简化版）

本 skill 不拥有面向用户的“后续内容”或“任务阻断收口”模板。真实阻断时只输出合规 PASS/FAIL 与共享契约缺失项；最终用户区块由 `reasoning-summary-structure-rules` 唯一渲染。

约束：

- 上述结构是普通 Markdown 示例，不得包进 ` ```text ` 或其他代码围栏；只有真实代码、命令、配置、日志、JSON/YAML 等片段才使用代码围栏。
- 合规结论必须可追溯到原执行计划、完成定义、验收标准或共享阻断契约，不给泛化空话。
- 缺失状态、阶段、依据与证据、已尝试与停止条件、影响、1-3 条解决计划、恢复验证 / 重入点中任一字段，或渲染 owner 不是 `reasoning-summary-structure-rules`，均为 FAIL。
- `limited/not_applicable` 不得被本 skill 升级为真实阻断；若主任务无阻断级缺口和计划内未完成必需项，且用户未显式要求建议/backlog，默认直接结束。
- 可选优化、额外整理、未来迭代、体验提升、文档再润色等内容，若不属于原计划必需项，不得作为默认后续内容输出。
- 禁止为了占位输出“等待用户新指令”“无需继续动作”“当前无需补充动作”“无下一步”等文案；无下一步时直接结束。
