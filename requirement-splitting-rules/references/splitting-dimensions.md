# 需求拆分维度

## 常见拆分维度

- 按独立子系统 / 子项目：例如聊天、文件、计费、分析这类相对独立的大能力先拆开。
- 按业务流程：录入、处理、展示、通知、回写等阶段形成垂直闭环。
- 按角色：管理员、普通用户、外部调用方分别形成可验证业务结果。
- 按风险：先做低风险闭环，再做高风险扩展。
- 按依赖：先明确上游业务输入，再安排下游消费。
- 不默认按前端、后端、数据库等技术层水平切块；只有该技术层本身能独立交付业务价值时才可单列。

## 粒度原则

- 每个子项都应描述清楚用户价值、范围、业务输入输出和验收意图。
- 每个子项都应形成相对独立的“需求澄清 -> 业务结果 -> 前置验收 -> 实施规划”闭环。
- 不拆出高度耦合、不能独立验收的伪子任务。
- 原需求覆盖多个独立子系统时，先拆子项目，再细化当前优先子项目。

## `SLICE-*` 业务切片契约（强制）

每个垂直切片必须有唯一 `SLICE-*`，由需求拆分 Owner 冻结：目标与用户价值、所属 `REQ/RULE`、范围/非范围、业务输入输出、角色、业务依赖、权限/兼容、异常与边界、验收意图、优先级、责任人、决策证据和需求级回退条件。

| 检查项 | 强制标准 |
| --- | --- |
| 唯一归属 | 一个需求点只能归属一个主切片；跨切片共享能力必须单列依赖切片 |
| 业务可交付性 | 每个切片都能产生可独立说明、可进入前置验收的业务结果 |
| 依赖关系 | 业务依赖图无环；每条边写明输入、输出、阻断条件和推荐顺序 |
| 粒度 | 不拆成无法独立验收的伪任务，也不把多个独立系统合并为一个切片 |
| 当前闭环 | 明确当前优先 `SLICE-*`；其余切片只保留足够的顶层边界和依赖 |
| 图片资产 | 需要图片时声明图片决策和业务用途；物理路径、`IMG-*` 与最终核验交给外部 Owner |

## 实施规划移交契约（强制）

需求拆分完成后，`implementation-planning-rules` 必须为当前优先 `SLICE-*` 冻结以下字段：精确文件/符号落点、技术依赖 DAG、实施总览、实施周期、最小任务、local 真实测试命令、样本与断言、失败预期、清理、任务级回滚、完成/停止条件和证据编号。

这些实施字段是必需保护语义，但不由需求拆分 Owner 定义；缺少任一必需字段时，由实施规划阶段标记 `blocked`，不得反向把技术任务细则塞回需求切片。

## 切片收口闸门

1. P0/P1 需求、边界、权限或兼容结论未决时，不得把切片声明为可执行。
2. 切片范围、业务依赖、顺序或责任变化时，更新拆分表和业务依赖图，并向验收与实施规划 Owner 发出失效信号。
3. 每个切片后续必须由实施规划转成逐任务“实现 -> 真实测试 -> 审查 -> 验收”闭环；需求拆分通过不等于实现计划已经通过。
