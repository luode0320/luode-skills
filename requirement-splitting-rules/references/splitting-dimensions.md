# 需求拆分维度

## 常见拆分维度

- 按独立子系统 / 子项目：例如聊天、文件、计费、分析这类相对独立的大能力先拆开。
- 按模块：前端、后端、配置、数据、测试支撑分别拆开。
- 按流程：录入、处理、展示、通知、回写等阶段拆开。
- 按角色：管理员、普通用户、外部调用方分开。
- 按风险：先做低风险闭环，再做高风险扩展。
- 按依赖：先做上游准备，再做下游消费。

## 粒度原则

- 每个子项都应能描述清楚输入、输出和验收点。
- 每个子项都应尽量形成相对独立的“需求澄清 -> 实现 -> 验证”闭环。
- 不拆出高度耦合却无法单独落地的伪子任务。
- 如果原需求一次覆盖多个独立子系统，应先拆子项目，再细化单个子项目内部结构。

## 极致完整性切片契约（强制）

每个垂直切片必须有唯一 `SLICE-*`，并冻结：目标与用户价值、所属 `REQ/RULE`、范围/非范围、输入输出、文件/符号落点、依赖前置、权限/兼容、异常与边界、测试命令、样本与断言、清理、回滚、责任人、完成条件和证据编号。

| 检查项 | 强制标准 |
| --- | --- |
| 唯一归属 | 一个需求点只能归属一个主切片；跨切片共享能力必须单列依赖切片 |
| 可交付性 | 每个切片都能独立完成“实现 -> 真实测试 -> 审查 -> 验收”闭环 |
| 依赖关系 | 依赖图无环；每条边写明输入、输出、阻断条件和顺序 |
| 粒度 | 不拆成无法单独落盘/验证的伪任务，也不把多个独立系统合并为一个任务 |
| 追踪 | `REQ/RULE -> SLICE -> CYCLE/TASK -> TEST -> EVIDENCE` 双向覆盖 100% |
| 图片资产 | 需要图片时统一使用 `doc/data/images/`、`IMG-*` 和当前文档位置的相对路径；图片仅作为视觉证据，不替代 Mermaid 结构/依赖图 |

### 切片收口闸门

1. 切片缺少文件/符号落点、真实测试入口、失败预期、清理或回滚任一项，状态为 `blocked`。
2. 切片范围、依赖、顺序或责任发生变化时，必须同步更新拆分表、依赖图、实施周期和验收映射。
3. 未决的 P0/P1 需求、边界或兼容结论不得被拆成可执行任务；必须回开需求/缺口/边界文档。
