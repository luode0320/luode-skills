# 需求拆分维度

## 常见拆分维度

- 按独立子系统 / 子项目：例如聊天、文件、计费、分析这类相对独立的大能力先拆开。
- 按模块：前端、后端、配置、数据、测试支撑分别拆开。
- 按流程：录入、处理、展示、通知、回写等阶段拆开。
- 按角色：管理员、普通用户、外部调用方分开。
- 按风险：先做低风险闭环，再做高风险扩展。
- 按依赖：先做上游准备，再做下游消费。

## 粒度原则

- 每个子项都应能描述清楚输入、输出和验收点。
- 每个子项都应尽量形成相对独立的“需求澄清 -> 实现 -> 验证”闭环。
- 不拆出高度耦合却无法单独落地的伪子任务。
- 如果原需求一次覆盖多个独立子系统，应先拆子项目，再细化单个子项目内部结构。
