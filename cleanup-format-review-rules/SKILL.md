# 清理格式审查规则

只在“进入测试前先把代码噪音清掉、基础格式拉齐”时使用这个 skill。
如果当前改动已经涉及功能逻辑变化，不要把它伪装成格式清理。

## Skill 作用与适用场景

- 清理未使用导入、未使用变量、未使用方法引用和死代码。
- 清理调试残留、临时代码和无意义打印。
- 统一多余换行、格式不一致和代码块排版问题。
- 保障最低可读格式达标：缩进层级清晰、同级语句对齐、尾随空格清理、连续空行收敛。
- 防止把明显代码噪音带进测试和交付阶段。
- 对前端代码改动，必须检查 import 排序/分组、重复或未使用 import、模板 / script / style 区块排版、缩进、连续空行和 Prettier/ESLint 可自动发现的格式问题。

## 自动触发信号

- 代码实现已完成，准备进入测试前自审。
- 出现未使用导入、变量、方法引用或临时调试语句。
- 出现多余空行、排版混乱、格式不一致等可自动整理问题。
- 出现“能构建通过，但代码排版明显混乱”的情况（例如同一代码块缩进错位、同级列定义缩进不一致）。
- 需要确认某次改动是不是“只做清理”，还是已经越界到功能性修改。
- 新增或修改 `.vue`、`.tsx`、`.jsx`、`.ts`、`.js`、`.html`、`.css`、`.scss`、`.less` 前端代码后准备声明完成或进入测试。

## 进入后先做什么

1. 先区分当前问题是代码噪音还是功能逻辑。
2. 列出未使用项、调试残留和格式问题。
3. 明确哪些可直接清理，哪些会碰到功能边界。
4. 对会影响行为的清理动作先停下，不混在这一轮做。
5. 对新增或修改代码执行“最低格式达标检查”，至少确认缩进、对齐和空行符合项目既有风格。
6. 若涉及前端代码，优先使用项目现有 formatter/linter（如 Prettier、ESLint、stylelint、npm/yarn/pnpm lint 或 format 脚本）；若没有可用命令，必须人工核对 import 分组、未使用 import、区块排版和缩进空行。

## 默认执行流程

1. 默认先读 `references/cleanup-scope.md`，确认本轮允许处理哪些噪音问题。
2. 如果涉及调试残留和临时代码，再读 `references/debug-artifact-rules.md`。
3. 如果需要判断是否越界到功能修改，再读 `references/cleanup-examples.md`。
4. 执行最小格式化动作并复检改动位点排版（以项目现有工具为准；无工具时至少人工完成缩进/对齐/空行清理）。
5. 输出可直接清理项、需回到编码阶段的问题和禁止混入的功能修改。
6. 清理未完成前，不进入测试域。

## 权责边界与不负责事项

- 只负责清理噪音和基础格式，不代替 `implementation-review-rules` 做实现质量判断。
- 不代替 `syntax-check-review-rules` 做语法 / 类型 / 引用完整性校验。
- 不允许把功能性修改、结构重构或命名大改包装成“顺手清理”。
- 不负责决定目录归位和跨层依赖方向。

## 需要暂停并确认的条件

- 某项“清理”会改变行为、分支或数据流。
- 当前目录存在大量历史噪音，但本次任务只涉及很小改动范围。
- 调试残留是否需要降级为正式日志仍不清楚。
- 清理动作已经扩散成大面积无关格式化。

## 执行通过 / 驳回标准

- 通过：未使用项、调试残留、死代码和明显格式噪音已清理；新增/修改代码满足最低格式达标（缩进、对齐、空行可读）；且未混入功能修改。
- 驳回：借清理之名扩大改动范围，或把调试残留、临时代码和明显排版混乱代码继续带入测试；前端代码存在明显 import 顺序/分组混乱、未使用 import、模板/script/style 区块排版混乱、缩进或连续空行问题却仍声明完成。

## 执行结果归档要求

- 将清理结论、未处理项和需回流的功能性问题记录到 `review/`。
- 归档内容至少包含已清理项、未清理原因和下一步动作。
- 如果只是轻量整理且无争议，可以简化记录，但应保留结论。

## references 读取规则

- 默认先读 `references/cleanup-scope.md`。
- 只有在处理调试残留和临时代码时，再读 `references/debug-artifact-rules.md`。
- 只有在判断是否越界时，再读 `references/cleanup-examples.md`。
