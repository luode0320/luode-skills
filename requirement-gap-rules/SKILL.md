# 需求缺口识别规则

只在判断“当前需求信息够不够支撑继续实现”时使用这个 skill。
如果当前争议已经变成需求边界、历史问题归属或验收标准细化，请转给相邻需求类 skill。

## Skill 作用与适用场景

- 识别需求描述中的关键缺失项。
- 识别目的、约束、成功标准、关键前提和方案选择是否仍然缺失或未收敛。
- 判断这些缺失是否足以阻断编码继续推进。
- 把模糊需求转成可确认的缺口清单。
- 检查当前需求文档中是否仍残留 `TBD/TODO/待补`、自相矛盾或双重解读都成立的描述。
- 将缺口分析结果持续更新到同一个需求文档，而不是散落成独立记录。
- 防止在信息不足时直接盲写实现。

## 自动触发信号

- 用户提出了新需求，但目标、输入输出或流程不完整。
- 缺少字段定义、数据结构、业务规则或交互细节。
- 缺少上下游依赖、环境前提、外部接口前提。
- 缺少验收标准、限制条件或兼容要求。
- 当前需求虽然写了文档，但仍存在多种合理解释、关键方向尚未选定或关键约束未说清。
- 当前需求文档里还有占位词、明显歧义或前后矛盾，继续推进会导致实现方向不稳定。

## 进入后先做什么

1. 先判断当前需求是否已经具备最小实现信息。
2. 列出缺失项，而不是直接假设。
3. 判断缺失项属于可继续推进的小不确定，还是必须阻断的关键缺口。
4. 检查当前需求文档是否仍存在占位词、矛盾、双重解读或未收敛的关键方向。
5. 先找到当前需求对应的需求主文档；如果还没有，就按 `artifact-storage-rules` 的路径与命名模板初始化同一份文档。
6. 输出待确认项、影响范围和建议确认顺序，并回写到该文档中。

## 默认执行流程

1. 默认先读 `references/missing-info-checklist.md`，检查缺失的是哪一类信息。
2. 再读 `../artifact-storage-rules/references/path-map.yaml` 与 `../artifact-storage-rules/references/update-policy.md`，确认需求主文档根目录、入口文件模板和“同一需求持续更新同一文档”的策略。
3. 如果需要判断是否必须暂停，再读 `references/pause-triggers.md`。
4. 如果需要对照类似场景的处理方式，再读 `references/requirement-gap-examples.md`。
5. 找到当前需求对应的需求主文档，并在其中更新缺失项清单、阻断结论、继续推进条件、建议确认问题和仍未收口的歧义点。
6. 如果该需求还没有文档，则按 `artifact-storage-rules` 的中央目录与入口模板创建同一份需求文档后再更新。
7. 若发现多个合理方向都说得通但还没收敛，应先要求用户确认方向，必要时回流 `requirement-intake-rules` 输出方案比选，而不是假设一种继续推进。
8. 在关键缺口未补齐前，不进入编码阶段。

## 权责边界与不负责事项

- 只负责识别“缺了什么”，不负责判断“这是不是当前需求范围”，那属于 `requirement-boundary-rules`。
- 不负责把验收标准完全写实，那属于 `acceptance-criteria-rules`。
- 不把所有普通不确定项都放大成阻断；只拦关键缺口。
- 不把“多个合理方向尚未收敛”误判成已经可实现的稳定需求。
- 不代替总控层做需求域与 Bug 域的分流裁决。
- 不为同一个需求再额外创建第二份缺口文档；缺口分析只能更新同一份需求主文档。

## 需要暂停并确认的条件

- 目标不清，无法判断要做什么。
- 输入输出不清，无法设计接口或处理流程。
- 核心字段、核心规则或前置依赖缺失。
- 验收标准缺失到无法判断做完算不算完成。
- 多种合理解释都成立，但还没有做方向选择。
- 需求文档中仍存在关键占位词、明显歧义或前后矛盾。

## 执行通过 / 驳回标准

- 通过：能明确列出缺失项、判断其严重程度、说明是否可以继续推进，并将结论回写到同一份需求主文档；同时指出当前文档中残留的占位词、歧义或未收敛方向。
- 驳回：对明显缺口视而不见直接进入实现，或把轻微不确定项过度放大成全面阻断，或把未收敛的关键方向误判成已明确需求。

## 执行结果归档要求

- 将关键缺口、阻断结论、待确认问题记录到当前需求对应的需求主文档中。
- 所有需求域相关 skill 都应复用同一个需求文档；补齐、拆分、边界、变更都在同一份文档持续更新，不重复新建平行文档。
- 需求主文档的根目录、入口文件模板和复用策略统一遵循 `../artifact-storage-rules/references/path-map.yaml` 与 `../artifact-storage-rules/references/update-policy.md`。
- 归档内容至少包含当前需求、缺失项、影响面、是否阻断、仍存歧义点和下一步确认动作。
- 如果只是轻微补问且不影响推进，也应更新同一份需求文档中的“待确认项”或“风险说明”，而不是另开记录。

## references 读取规则

- 默认先读 `references/missing-info-checklist.md`。
- 在定位当前需求主文档、创建初始文档或判断是否继续更新同一文档时，先读 `../artifact-storage-rules/references/path-map.yaml` 与 `../artifact-storage-rules/references/update-policy.md`。
- 只有在判断是否必须暂停时，再读 `references/pause-triggers.md`。
- 只有在需要对照正反例时，再读 `references/requirement-gap-examples.md`。
