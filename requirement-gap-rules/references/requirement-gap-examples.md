# 需求缺口正反例

## 正例

### 正例 1：接口字段没给全

- 当前需求：新增接口。
- 已知问题：核心字段、校验规则和返回结构都不完整。
- 处理：列出缺失字段和待确认项，更新到同一个 `ment/` 需求文档中，暂停进入实现。

### 正例 2：页面目标清楚，但少量文案未定

- 当前需求：页面流程、数据和交互已清楚。
- 已知问题：个别展示文案尚未定稿。
- 处理：在同一个 `ment/` 需求文档中标记风险，可继续推进主体实现。

## 反例

### 反例 1：完全靠猜继续写

- 目标、字段、规则都不清。
- 仍直接进入编码。
- 结论：驳回。

### 反例 2：把所有小问题都拦成暂停

- 主流程已清楚，只剩个别非核心细节未定。
- 仍把任务整体拦下。
- 结论：过度阻断，不通过。

### 反例 3：同一个需求分散成多份文档

- 接入写一份，缺口再新建一份，后续拆分和变更又各写一份。
- 结果：同一需求信息分散，后续难追溯。
- 结论：驳回，应统一回写到同一份 `ment/` 需求文档。
