# 需求缺口正反例

## 正例

### 正例 1：接口字段没给全

- 当前需求：新增接口。
- 已知问题：核心字段、校验规则和返回结构都不完整。
- 处理：先确认这些信息无法继续通过 discovery 获得，再创建临时缺口文档，列出缺失字段、已侦察证据和待确认项，暂停进入实现；用户确认后回填主需求文档并删除临时缺口文档。

### 正例 2：页面目标清楚，但少量文案未定

- 当前需求：页面流程、数据和交互已清楚。
- 已知问题：个别展示文案尚未定稿。
- 处理：在同一个 `doc/需求/` 主需求文档中标记风险，可继续推进主体实现，不必创建临时缺口文档。

### 正例 3：存在多个合理方向，但还没选定

- 当前需求：要做一个新流程，但“单页完成”与“分步向导”两种方向都合理。
- 已知问题：如果不先选方向，后续字段组织、页面结构和验收口径都会不同。
- 处理：在临时缺口文档中明确记录“方向未收敛”和拍板问题，暂停继续实现，回到需求接入层做方案比选；确认后回填主需求文档。

## 反例

### 反例 1：完全靠猜继续写

- 目标、字段、规则都不清。
- 仍直接进入编码。
- 结论：驳回。

### 反例 2：把所有小问题都拦成暂停

- 主流程已清楚，只剩个别非核心细节未定。
- 仍把任务整体拦下。
- 结论：过度阻断，不通过。

### 反例 3：把可侦察问题直接丢给用户

- GitHub、官网或 API 文档里本可补齐的信息还没查。
- 就直接追问用户。
- 结论：驳回，应先回流 discovery。

### 反例 4：临时缺口文档变成长期入口

- 用户已确认缺口，但没有回填主需求文档，也没删除临时缺口文档。
- 结果：主文档和缺口文档长期并存、信息漂移。
- 结论：驳回，应回填主文档并删除临时缺口文档。

### 反例 5：文档还有占位词却当成已清晰需求

- 需求文档里还写着 `TBD`、`待补`，或者同一条规则前后表述互相冲突。
- 仍然直接进入编码。
- 结论：驳回，应先补齐缺口并更新主需求文档。
