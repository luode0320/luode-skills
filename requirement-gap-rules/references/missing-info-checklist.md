# 需求缺失信息清单

## 用途

用于判断当前需求缺少的是哪一类关键信息。

## 常见缺失项

- 目标缺失：不知道最终要解决什么问题。
- 动因缺失：不知道为什么现在要做、为什么值得做。
- 输入缺失：不知道输入来源、参数、字段或触发条件。
- 输出缺失：不知道应返回什么结果、页面效果或状态变化。
- 流程缺失：不知道关键业务步骤、分支或顺序。
- 规则缺失：不知道校验规则、状态规则、权限规则或异常规则。
- 依赖缺失：不知道依赖哪个接口、服务、表结构、配置或环境。
- 约束缺失：不知道兼容、权限、环境、性能或时间限制。
- 成功标准缺失：不知道做到什么算“已满足需求”。
- 方向未收敛：存在多个合理方案，但还没有选定当前推荐方向。
- 文档缺口：需求文档里仍有 `TBD/TODO/待补`、歧义句或前后矛盾。
- 验收缺失：不知道什么结果算完成、什么场景必须覆盖。
- 图片决策缺失：未声明是否需要图片，或未写 `N/A + 原因 + 证据`。
- 图片资产缺失：声明需要但缺 `doc/data/images/` 文件、`IMG-*` 清单、非空 alt、可解析相对路径、版本/来源/版权或签名证据。

## 判断原则

- 缺失项只要会直接影响设计或实现，就不能忽略。
- 只要缺失项会让不同实现方向都看起来合理，就应先确认。
- 只要文档中还存在关键占位词、矛盾或双重解读，就不应假装需求已经稳定。
- 对能通过项目、代码、数据库、上下游、GitHub、网站或官方 API 文档补齐的信息，必须先回流 discovery，不直接追问用户。
- 缺口分析结果默认先写入当前需求对应的临时缺口文档；用户确认后再回填同一份 `doc/2-需求/` 主需求文档，并删除临时缺口文档。

## 极致完整性缺口矩阵（强制）

每个缺口必须登记为唯一 `GAP-*`，并逐项填写：缺口类别、缺失字段、已侦察证据、来源 `SRC-*`、影响的 `REQ/RULE/AC`、严重级别、责任人、补齐动作、截止时间、阻断状态、回填位置、验证命令、完成证据、失败预期、清理与回滚。

| 级别 | 判定 | 强制动作 |
| --- | --- | --- |
| `P0` | 无法确定目标、权限、数据、接口、核心规则、异常或安全边界 | 立即 `blocked`；不得进入验收、实施计划或编码 |
| `P1` | 会导致多个实现方向、兼容性或测试结论不同 | 由授权决策人确认；确认前只允许侦察与文档补齐 |
| `P2` | 不影响主链路但会影响质量、可观测、运维或交付 | 写明默认值授权、有效期、风险和复核证据 |

### 缺口关闭闸门

1. 可由项目、代码、数据、上下游或官方资料获得的缺口必须先走 discovery，并记录查询入口和原始证据；不得直接向用户索取可自动获得的信息。
2. 缺口关闭后必须回填同一份需求主文档，更新 `unresolved_decisions`、图形、影响矩阵和 `REQ/RULE -> AC` 追踪；临时缺口文档只在确认后删除。
3. 任何缺口不得以“执行模型自行判断”关闭；缺失项必须写 `N/A + 原因 + 证据`，否则视为未关闭。
4. 图片缺口必须校验 `doc/data/images/` 唯一根目录和 `../data/images/<file>` 相对路径；直接 `doc/data/`、历史需求域图片目录、绝对/远程/Base64/HTML/反斜杠/越界路径均保持未关闭。
