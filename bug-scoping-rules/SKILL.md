# Bug 范围界定规则

只在需要回答“这个问题归谁、先从哪条链路定位、影响到哪里”时使用这个 skill。
如果当前还没有复现或入口信息不足，请先补上；如果已经在具体代码中形成根因证据，请转交 `bug-root-cause-rules`。

## Skill 作用与适用场景

- 确定问题归属模块、链路、页面、接口或服务边界。
- 划出直接影响面、间接影响面和疑似无关区域。
- 帮助选择最优先的定位入口和证据来源。
- 避免多人同时在大范围盲查。

## 自动触发信号

- 问题横跨前后端、多个服务或多个模块，归属不清。
- 需要判断异常是页面展示问题、接口返回问题、数据链路问题还是环境问题。
- 团队对先查哪里存在争议。
- 需要明确本次问题会影响哪些用户、流程或上下游。

## 进入后先做什么

1. 先基于现象和复现路径列出可能涉及的模块与链路。
2. 划分直接触达区、间接依赖区和暂不相关区。
3. 判断优先观察哪一层最可能先暴露证据。
4. 形成归属与影响范围结论，而不是立刻下根因判断。

## 默认执行流程

1. 默认先读 `references/scoping-dimensions.md`，先按统一维度划定问题范围。
2. 如需继续展开，再读 `references/ownership-boundaries.md`，需要区分归属边界和优先入口。
3. 需要对照边界或正反例时，再读 `references/scoping-examples.md`，需要对照范围界定正反例。
4. 输出 归属判断、影响范围和优先定位入口。
5. 范围界定后转 `bug-root-cause-rules` 做静态定位；若仅靠静态仍无法收敛，再转 `bug-runtime-debug-rules`。

## 权责边界与不负责事项

- 只负责归属与影响面，不代替根因分析。
- 不把“在这个模块看到异常”直接等同于“根因就在这里”。
- 不负责决定修复方案，那属于 `bug-fix-proposal-rules`。
- 不把所有上下游都纳入问题范围，必须有证据区分主次。

## 需要暂停并确认的条件

- 基础现象和复现步骤仍不清，无法可靠划范围。
- 问题同时涉及多个团队边界，但责任归属证据不足。
- 影响面和定位入口判断差异极大，需要更多证据支撑。
- 现象本身已变化，旧的范围判断可能失效。

## 执行通过 / 驳回标准

- 通过：能够说明问题最可能属于哪条链路、影响哪些范围、应优先从哪里定位，以及哪些区域暂时不应投入。
- 驳回：把范围界定写成泛泛大图，没有优先级，也没有把直接影响和间接影响分开。

## 执行结果归档要求

- 将归属判断、影响范围和优先入口记录到 `analysis/`。
- 归档内容至少包含问题链路、模块范围、影响对象和下一步定位建议。
- 如果结论涉及多团队协作，应写明当前证据基础，避免后续扯皮。

## references 读取规则

- 默认先读 `references/scoping-dimensions.md`。
- 只有在 判断模块归属和优先定位入口 时，再读 `references/ownership-boundaries.md`。
- 只有在 需要对照样例或判断范围是否过宽 时，再读 `references/scoping-examples.md`。
