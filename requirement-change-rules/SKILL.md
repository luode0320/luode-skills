# 需求变更确认规则

只在原需求已经进入澄清、拆分或实现后，又出现新的业务条件、范围调整或优先级变化时使用这个 skill。
如果当前问题不是“需求变了”，而是“原实现有缺陷”，请转交 Bug 域；如果只是原需求信息不全，请转交 `requirement-gap-rules`。

## Skill 作用与适用场景

- 把编码中途出现的新增条件、优先级调整和范围修正重新归类。
- 判断变更是轻量补充、边界变化、实现重做，还是应拆成新任务。
- 重算对已有拆分、实现、测试和交付结论的影响。
- 当变更推翻原有需求收敛结论时，要求先更新同一份需求文档并重新确认，避免边改边漂移。
- 将需求变更结论更新到当前需求对应的同一份需求主文档中。
- 防止“边做边改”失控，导致范围偷偷扩大。

## 自动触发信号

- 用户在实现过程中追加字段、规则、页面、接口或异常分支。
- 原需求优先级、默认值、兼容要求、交付物形态发生变化。
- 当前已有方案因为新条件加入而需要重算影响范围。
- 团队需要判断这是继续在当前任务处理，还是应拆新任务。

## 进入后先做什么

1. 先写清本次新增或调整的到底是什么，而不是直接改代码。
2. 判断它属于信息补充、范围变化、优先级变化还是交付物变化。
3. 对照已有结论，确认会影响哪些需求、实现、测试和交付环节。
4. 找到当前需求对应的需求主文档；如果还没有，就按 `artifact-storage-rules` 的路径与命名模板初始化同一份文档。
5. 决定继续当前任务、回退前序结论，还是拆出独立变更项，并准备回写到该文档。

## 默认执行流程

1. 默认先读 `references/change-classification.md`，先判断这次变更属于哪一类。
2. 再读 `../artifact-storage-rules/references/path-map.yaml` 与 `../artifact-storage-rules/references/update-policy.md`，确认需求主文档根目录、入口文件模板和同文档复用策略。
3. 如需继续展开，再读 `references/impact-recheck.md`，需要重算对已有结论的影响。
4. 需要对照边界或正反例时，再读 `references/change-decision-examples.md`，需要对照变更处理方式样例。
5. 输出变更分类结论、影响面重算结果和处理建议，并更新到当前需求对应的需求主文档。
6. 如果该需求还没有文档，则按 `artifact-storage-rules` 的中央目录与入口模板创建同一份需求文档后再更新。
7. 如果变更改动了关键目标、关键约束、主方案结论或成功标准，应先在同一份需求文档中完成更新、自审并请用户重新确认，再继续实现。
8. 若变更导致边界重算，转交 `requirement-boundary-rules`；若变更后任务体量明显增大，转交 `requirement-splitting-rules`；若需要重写验收标准，转交 `acceptance-criteria-rules`。
9. 若变更推翻原先确认过的实施顺序、文件落点或阶段闭环，应在更新需求文档后回流 `implementation-plan-rules`，重写同一需求对应的实施计划文档。

## 权责边界与不负责事项

- 只负责需求变更识别与重算，不替代 `requirement-gap-rules` 做初始缺口识别。
- 不把历史故障或实现错误包装成需求变更，疑似缺陷应转 Bug 域。
- 不直接批准超大范围扩写，应在必要时要求拆新任务。
- 不代替项目管理做排期确认，但要明确变更会影响哪些已有结论。
- 不为同一个需求再额外创建变更文档；变更只能更新同一个需求主文档。

## 需要暂停并确认的条件

- 新增条件表述含糊，无法判断到底改了什么。
- 变更会推翻已有需求边界、实现方案或测试计划，但尚未重新确认。
- 当前任务已经因多次变更失去原始目标，疑似需要重新接入或拆分。
- 变更来源和决策权不清，无法确认是否应纳入当前任务。

## 执行通过 / 驳回标准

- 通过：能够说清这次变化是什么、属于哪类变更、影响哪些既有结论，以及应继续当前任务还是拆出新项，并将结论回写到同一个需求主文档。
- 驳回：直接把新增条件塞进实现里，没有重算影响范围，或把历史 Bug、信息缺口、边界争议误判成需求变更。

## 执行结果归档要求

- 将变更内容、影响面和处理决策记录到当前需求对应的需求主文档。
- 所有需求域相关 skill 都应复用同一个需求文档；需求变更只是在同一份文档中追加“变更记录 / 影响重算 / 决策结论”。
- 需求主文档的根目录、入口文件模板和复用策略统一遵循 `../artifact-storage-rules/references/path-map.yaml` 与 `../artifact-storage-rules/references/update-policy.md`。
- 归档内容至少包含变更来源、变更分类、受影响环节和后续动作。
- 如果变更推翻了原拆分或原验收标准，应在同一份文档中保留新旧版本差异说明。
- 如果变更已经影响原先确认过的需求方向或范围，应在同一份文档中记录“需重新确认”的状态，而不是默认沿用旧确认。

## references 读取规则

- 默认先读 `references/change-classification.md`。
- 在定位当前需求主文档、创建初始文档或判断是否继续更新同一文档时，先读 `../artifact-storage-rules/references/path-map.yaml` 与 `../artifact-storage-rules/references/update-policy.md`。
- 只有在 重算变更对需求、实现、测试和交付的影响 时，再读 `references/impact-recheck.md`。
- 只有在 需要对照正反例或判断是否应拆新任务 时，再读 `references/change-decision-examples.md`。
