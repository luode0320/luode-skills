# 变更影响重算

## 至少检查四个面向

- 需求：是否推翻原边界、原拆分或原验收标准。
- 实现：是否需要新增文件、模块、接口、数据结构或重做已有实现。
- 测试：是否需要新增功能验证、回归验证或测试资源调整。
- 交付：是否影响提交流程、发布窗口、说明文档或回滚准备。
- 图片资产：是否新增/替换截图、UI/原型或视觉证据，是否影响 `IMG-*` 版本、共享引用、敏感/版权状态、`doc/data/images/` 路径、孤儿扫描和清理/回滚。

## 决策输出

- 继续当前任务处理。
- 当前任务只接一部分，其余拆新任务。
- 回退到需求域重新澄清或拆分。
- 以上重算结论都要更新到当前需求对应的同一份 `doc/2-需求/` 文档中。

## 极致完整性变更重算契约（强制）

每次变更必须生成唯一 `CHG-*`，并记录变更来源、原值/新值、决策人、状态、影响的 `SRC/DEC/REQ/RULE/AC/SLICE/CYCLE/TASK`、文件/符号、图表、测试、交付、清理、回滚、证据和复核时间。

| 影响面 | 必须重算 |
| --- | --- |
| 需求/规则 | 范围、非范围、优先级、权限、兼容、异常、数据和 `unresolved_decisions` |
| 实施 | 文件/符号、依赖 DAG、周期归属、任务顺序、执行边界和回滚点 |
| 验收/测试 | `REQ/RULE -> AC` 覆盖、主/异常/边界场景、样本、断言、失败预期、清理 |
| 交付 | 图形、目录树、链接、证据、版本、发布窗口、风险和回退方案 |
| 图片资产 | 图片决策、真实生成/来源、`IMG-*`、版本、非空 alt、相对路径、引用章节、共享消费者、敏感/版权、孤儿和回滚 |

### 变更回开闸门

1. 变更影响已确认的目标、范围、接口、数据、权限、兼容、验收或任务时，必须回开受影响文档并保留旧值与新值差异；不得只在聊天中说明。
2. `P0/P1` 变更未获授权前状态为 `blocked`，禁止继续编码或验收；`P2` 变更也必须登记默认值授权、有效期和复核证据。
3. 变更完成后必须重新运行文档机器校验和受影响行为测试；任一证据缺失，原任务不得标记完成。
4. 图片变更必须重新检查 `doc/data/images/` 文件签名、`../data/images/<file>` 相对引用、`IMG-*` 清单和全仓消费者；直接 `doc/data/`、历史需求域图片目录、绝对/远程/Base64/HTML/反斜杠/越界路径均阻断。
