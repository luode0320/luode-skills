# 查询性能与样例

## 正例

- 在访问层清楚区分单条读取、分页列表、批量写入和事务型写入。
- 慢查询先确认筛选条件、排序方式、返回字段和索引适配，再决定是否重写 SQL。
- 批量任务采用分批处理和可恢复机制，不把全量更新塞进一次事务。
- 读写逻辑明确职责边界，不在 Repository 里埋复杂业务流程。

## 反例

- 为了少写几行代码，把复杂业务判断全部塞进 SQL 或 Repository。
- 查询慢就盲目加事务、加锁或强行改成复杂 QueryBuilder。
- 列表查询不限制返回范围，分页语义不稳定。
- 批量更新没有条件、没有分批、没有失败恢复说明。

## 判定提示

- 如果访问层代码看起来像业务服务，通常已经越界。
- 如果事务里包含外部调用、消息发送或长时间等待，通常边界过宽。
- 如果性能问题本质来自 schema、缓存或接口层，不要硬留在 query 规则里解决。
