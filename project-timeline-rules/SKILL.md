# 项目时间线规则

只在需要按项目维度回看长期历史、阶段演进和关键决策时使用这个 skill。
如果用户只是想知道“这次交付做了什么”，那是 `delivery-summary-rules` 的职责，不是这里。

## Skill 作用与适用场景

- 输出项目开发历程、技术演进和关键决策链路。
- 回看项目中重要里程碑、重大 Bug、重构阶段和阶段性突破。
- 帮团队在复盘、接手、演示或战略判断时快速理解项目历史。
- 在没有真实记忆后端时，退化为基于本地材料和 Git 历史的时间线报告。

## 自动触发信号

- 用户要求“生成项目时间线报告”“总结整个项目历史”“回顾这个项目是怎么做起来的”。
- 用户要求“看一下这个项目经历了哪些阶段”“有哪些关键决策和转折点”。
- 当前任务是项目复盘、项目介绍、长期演进分析，而不是单点需求或单点 Bug。

## 进入后先做什么

1. 先确认用户要的是完整项目史、某个阶段的历史，还是某类主题的时间线。
2. 明确项目名，不要把 worktree 名误当成真实项目名。
3. 先判断是否存在真实持久记忆后端。
4. 如果没有后端，明确说明将降级为本地材料时间线。
5. 先评估材料体量，再决定是做完整报告还是阶段性报告。

## 默认执行流程

1. 默认先读 `references/timeline-scope-boundary.md`，确认这次要做的是哪种时间线。
2. 如果存在真实记忆后端，再读 `references/backend-timeline-workflow.md`。
3. 如果没有真实记忆后端，先读 `../artifact-storage-rules/references/path-map.yaml`，确认本地时间线扫描根目录和报告保存根目录。
4. 如果没有真实记忆后端，再读 `references/local-material-timeline.md`。
5. 如果需要了解来源和吸收边界，再读 `references/source-notes.md`。
6. 输出时明确写出数据来源、覆盖时间范围、缺失阶段和报告可信度。

## 权责边界与不负责事项

- 只负责长期历史与阶段演进，不负责当前一次交付的提交摘要。
- 不代替当前需求验收、当前 Bug 结论和当前测试结论。
- 不在材料不足时伪造完整项目发展史。
- 不把阶段性回顾写成“当前实现已正确”的证明。

## 需要暂停并确认的条件

- 用户没有说明是要全项目报告，还是只要某阶段回顾。
- 材料体量过大，但用户其实只需要短摘要。
- 没有真实后端，本地材料也不足以支撑长期时间线。
- 当前项目名或时间范围判断不准，继续分析会把别的项目历史混进来。

## 执行通过 / 驳回标准

- 通过：能明确给出项目范围、时间顺序、关键阶段、关键问题和关键决策，并标清数据来源与缺口。
- 驳回：把零散文档拼成看似完整的项目史，或把当前交付摘要误写成长期演进报告。

## 执行结果归档要求

- 如果本次输出用于复盘、交接、汇报或长期记录，应单独保存为分析文档。
- 不要把长期时间线混进 `artifact-storage-rules` 约定的需求文档或验收文档中。
- 本地时间线扫描根目录和分析文档默认保存根目录统一遵循 `../artifact-storage-rules/references/path-map.yaml`。
- 归档内容至少包含：项目范围、时间范围、数据来源、关键阶段、主要结论和缺失说明。

## references 读取规则

- 默认先读 `references/timeline-scope-boundary.md`。
- 只有在真实记忆后端可用时，再读 `references/backend-timeline-workflow.md`。
- 只有在需要本地降级时间线或决定分析文档保存根目录时，再读 `../artifact-storage-rules/references/path-map.yaml`。
- 只有在需要本地降级时间线时，再读 `references/local-material-timeline.md`。
- 只有在需要了解来源和吸收边界时，再读 `references/source-notes.md`。
