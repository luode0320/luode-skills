# 前端 UI 视觉规则

只在任务会改变界面“看起来如何、交互起来如何、响应起来如何”时使用这个 skill。
如果当前只是前端组件拆分、状态归属、接口对接或纯逻辑修复，不要让这个 skill 抢占职责。

## Skill 作用与适用场景

- 为页面、面板、后台、移动端界面、落地页确定视觉方向和信息层级。
- 在编码前先确定页面的用途、受众、语气和一个足够明确的视觉记忆点，避免做成模板化前端。
- 约束颜色、字体、间距、图标、阴影、圆角、动效和状态反馈的统一性。
- 保障移动端适配、可访问性、交互反馈和暗黑模式等基础体验。
- 约束“设计表达强度”和“实现复杂度”要匹配，不把极简页面写得花乱，也不把需要强表达的页面做得像通用后台。
- 在编码落地前先判断风格方向，在编码完成前再做一轮 UI 自审。
- 合并保留外部 UI/UX 设计种子的搜索能力，在需要时可直接在本 skill 内完成设计方向检索。
- 将外部前端设计种子转化为适合你们团队长期维护的稳定规则。

## 自动触发信号

- 新增或修改页面级前端文件，如 `.tsx`、`.jsx`、`.vue`、`.html`。
- 新增或修改界面样式文件，如 `.css`、`.scss`、`.less`、主题变量、设计 token。
- 修改 Tailwind 类名、颜色变量、字体变量、间距变量、阴影、圆角、动画或断点。
- 新增或修改导航栏、侧边栏、表单、弹窗、卡片、列表、表格、图表、空态、加载态。
- 用户明确要求“更好看”“更专业”“更像产品”“更有层级”“更适合移动端”。

## 进入后先做什么

1. 先判断这是页面视觉问题、交互体验问题，还是组件工程问题。
2. 明确当前页面的产品类型、用户目标和主操作，不要一上来就拼效果。
3. 先明确页面语气、品牌气质或情绪方向，并定义“这页最该被记住的一处设计点”。
4. 优先确定信息层级、主 CTA、内容密度和布局骨架，再决定颜色和动画。
5. 先保证可访问性、触达面积、响应式和加载反馈，再决定装饰性表达。
6. 统一颜色、字体、间距、图标和状态表现，不要局部各写各的。

## 默认执行流程

1. 默认先读 `references/ui-priority-model.md`，先按优先级确定本次界面该先修什么。
2. 如果当前任务是新页面、新首页、新后台或要重新定界面气质，先读 `references/aesthetic-direction-rules.md`，明确用途、语气、记忆点和表达强度。
3. 如果当前任务要进一步匹配页面类型、产品场景和风格边界，再读 `references/page-style-and-scenario.md`。
4. 如果内部规则仍不足以确定风格方向、产品风格映射或配色字体方案，再读 `references/design-search-workflow.md`，并按需运行本 skill 内置的 `scripts/search.py`。
5. 如果当前任务涉及版式、断点、容器、手机端适配和滚动体验，再读 `references/layout-responsive-rules.md`。
6. 如果当前任务涉及点击反馈、无障碍、动效、加载态和表单交互，再读 `references/interaction-accessibility-rules.md`。
7. 如果当前任务涉及主题、配色、字体、图标和设计 token，再读 `references/color-typography-icon-rules.md`。
8. 如果当前任务涉及表单、导航、图表、数据展示和空态，再读 `references/forms-nav-data-display-rules.md`。
9. 完成前默认再读 `references/ui-delivery-checklist.md` 做一轮交付前 UI 自审。

## 权责边界与不负责事项

- 只负责前端视觉表达、交互体验和页面级 UI 质量，不代替组件工程设计。
- 不负责接口字段设计、状态机设计、数据库规则和后端逻辑。
- 不因为“想做得炫”就牺牲可读性、性能、可访问性和可维护性。
- 不把颜色、字体、圆角、阴影写成到处散落的原始值，优先收敛为统一 token。
- 不把页面风格做成每个模块各自为政；同一产品线应有统一视觉语言。
- 不把所有前端都做成同一种“AI 味模板”，尤其避免空泛渐变、默认字体和无场景来源的流行风格堆叠。
- 不绑定任何特定 UI 组件库的默认外观；可以重塑组件视觉或替换组件实现，但必须保持交互一致性与可维护性。

## 需要暂停并确认的条件

- 页面目标不清，无法判断主内容和主 CTA。
- 用户只说“做得高级一点”，但产品类型、受众和气质完全没说明。
- 项目已有明确设计系统或品牌规范，但当前修改方向和它明显冲突。
- 视觉要求与可访问性、移动端可用性或性能约束发生明显冲突。

## 执行通过 / 驳回标准

- 通过：能明确界面层级、主操作、风格方向、视觉记忆点、布局骨架、关键状态和移动端适配，并让颜色、字体、间距、图标和交互反馈保持一致。
- 驳回：只在局部堆渐变、阴影和动画，却没有解决层级、可读性、响应式、触达面积和可访问性问题；或整体仍像无场景差异的通用 AI 模板。

## 执行结果归档要求

- 如果本次修改形成了可复用的页面视觉方向、主题 token、间距体系或组件状态约束，应记录到项目现有设计说明、主题文档或对应需求文档中。
- 如果项目已经有设计系统或主题文件，优先更新原有文档，不重复新建平行设计说明。
- 归档内容至少包含页面类型、视觉方向、主色策略、字体策略、关键交互状态和响应式约束。
- 如果本次风格方向显著依赖内置搜索结果，建议同时记录使用了哪些查询词和最终采用了哪套风格判断。
