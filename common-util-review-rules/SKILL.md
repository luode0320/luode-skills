# 公共工具检查镜像规则

## 目标

只检查当前逻辑是否满足 `common-util-rules` 的公共工具判定标准，不执行抽取或改写。

## 源规则

以下判定严格沿用源 skill：

- 是否真的属于通用工具、常量或公共组件
- 是否存在同语义重复封装
- `utils` / `common` / `global` / `middleware` 是否应先拆子目录
- 是否命中 7 天冻结
- 是否不应为了复用提前抽公共接口

## 自动触发信号

- 新增或修改工具类、工具函数、辅助方法、公共组件。
- 打算把业务代码抽成 `utils`、`common`、`shared`、`helpers` 等目录。
- 不确定一段逻辑该留在业务层还是抽到公共层。
- 发现仓库中可能已经有可复用实现。

## 进入后先做什么

1. 先判断这段逻辑是否脱离当前业务后仍然成立。
2. 再判断它是通用工具、稳定常量、通用组件，还是当前业务逻辑。
3. 明确是否存在重复封装。
4. 判断是否命中 7 天冻结。

## 默认执行流程

1. 先读取 `common-util-rules`。
2. 如需继续判断落点和调用方式，再读源 skill 的对应 references。
3. 如需对照正反例，再读源 skill 的对应 references。
4. 输出应抽 / 不应抽结论、落点建议和重复封装风险。
5. 不直接改代码，只输出抽取边界。

## 权责边界与不负责事项

- 不代替 `package-structure-rules` 决定整个项目分层结构。
- 不代替具体位点 skill 设计接口、数据库、错误处理或日志实现。
- 不把业务逻辑伪装成公共工具。
- 不允许已存在公共工具时再次平行封装同语义函数。

## 需要暂停并确认的条件

- 当前代码只在一个业务流程内成立，却计划抽成公共层。
- 抽取后需要把业务上下文大量参数化，导致调用复杂度上升。
- 当前只有一个稳定实现，却计划为了“规范”新增接口定义、接口实现体和装配代码。
- 抽取会让公共层开始依赖业务层或具体位点层。

## 执行通过 / 驳回标准

- 通过：能明确说明这段代码为何通用、为何应抽、应放在哪、调用边界是什么。
- 驳回：只是因为“看起来可以复用”就提前抽象，或把业务逻辑塞进公共工具层。

## 执行结果归档要求

- 如果本次新增了公共工具、公共常量或公共组件，将结论记录到 `analysis/` 或 `review/`。
- 归档内容至少包含抽取理由、使用范围、落点和不应继续扩张的边界说明。
- 如果结论是不应抽，也应在必要时记录保留在业务层的原因。

## 读取规则

- 默认先读 `common-util-rules`。
- 只有在判断存放位置和调用方式时，再读源 skill 的对应 references。
- 只有在对照正反例时，再读源 skill 的对应 references。
