# 实现自审范围

## 用途

用于界定测试前第一轮实现自审到底看什么，不看什么。

## 应检查

- 当前实现是否符合最小改动原则。
- 最近改动代码是否仍残留明显可直接收口的表达层冗余。
- 是否坚持可读性优先，而不是炫技式写法、隐式逻辑或过度优化。
- 函数或方法是否保持单一职责，核心逻辑是否拆成可复用原子方法。
- 单个函数是否通常控制在 50 行以内；超出时是否有清晰结构和中文注释解释。
- 单文件是否达到 500 行及以上且仍在持续扩展；命中时是否已按职责拆分文件或给出可执行拆分方案。
- 是否存在嵌套三元、过密链式调用、无价值包装函数或“为了短而短”的表达。
- 是否存在魔法值、魔法字符串、魔法状态、硬编码超时、硬编码地址、硬编码开关等问题。
- 是否存在未使用变量、未使用导入、空分支、重复判断和明显冗余逻辑。
- 依赖选择是否成熟、活跃、文档完善、版本明确，是否存在随意引入未版本化依赖的问题。
- 命名是否语义明确，是否存在无意义缩写。
- 若生态不适合中文命名，是否采用稳定可读的中文拼音或明确英文业务词。
- 错误处理是否语义明确、可追踪、可国际化扩展；错误响应是否默认使用英文。
- 日志是否具备模块、操作、对象或业务上下文；日志文案是否优先使用中文。
- 方法级注释中是否明确标注“最近修改时间”，格式是否为北京时间 `yyyy-MM-DD HH:mm:ss`。
- 注释、文档与新增代码文件是否默认使用 UTF-8；注释是否使用中文。
- 实现风格是否与项目现有写法明显冲突。
- 实现结构是否存在显著可维护性问题。
- Go 项目是否存在 `test/` 外 `*_test.go` 落点违规（命令示例：`rg --files -g "*_test.go" | rg -v "^test/"`）。
- Go 项目本轮改动是否命中 `internal/service/[^/]+.go` 根目录落点（命令示例：`git diff --name-only HEAD -- | rg "^internal/service/[^/]+\\.go$"`）。
- Go 项目 `internal/service` 是否散落请求/响应/第三方结果结构体（命令示例：`rg -n "^\\s*type\\s+[A-Za-z_][A-Za-z0-9_]*\\s+struct\\s*{" internal/service`，并人工复核结构体语义）。

## 不在本轮主检查范围

- 语法 / 类型 / 构建错误的完整细项校验。
- 代码格式化工具的具体执行。
- 代码目录归位和跨层依赖方向审查。
- 功能正确性验证。
- 最近改动范围之外的大面积历史清理。

## 输出补充

- Go 场景下，输出中应包含 `*_test.go` 禁放扫描结果（命中路径或“未命中”）。
- Go 场景下，输出中应包含 `internal/service` 根目录落点扫描结果（命中路径或“未命中”）。
- Go 场景下，输出中应包含 `internal/service` 结构体散落扫描结果（可疑路径 + 复核结论）。
