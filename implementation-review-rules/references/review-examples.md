# 实现自审正反例

## 统一判定字段（所有审查文档必须包含）

审查文档落盘时必须包含以下 4 个字段，用于提交闸门校验：

- 审查结论: 通过/阻断
- 审查范围: <文件列表或等价范围说明>
- 是否允许提交: 是/否
- 阻断问题: <P0/P1 摘要，没有则写无>

通过示例：

```text
审查结论: 通过
审查范围: internal/service/order/create_service.go, internal/entity/order/request.go
是否允许提交: 是
阻断问题: 无
```

阻断示例：

```text
审查结论: 阻断
审查范围: internal/service/order/create_service.go
是否允许提交: 否
阻断问题: [P1] create_service.go:45 - 请求结构体散落在 service 层
```

阻断文档还必须包含共享契约规定的“任务阻断收口”。P0/P1 才属于实现自审的真实阻断；P2/P3、`limited` 与 `not_applicable` 继续使用原结论，不生成 `BLK-*`。

```markdown
## 任务阻断收口

- 阻断记录 ID: `BLK-REVIEW-001`
- 任务状态: `blocked`
- 阻断阶段: 实现自审
- 阻断依据与证据: [P1] 请求结构体散落在 service 层；`implementation-review.structure_placement`；`internal/service/order/create_service.go:45`
- 已尝试与停止边界: 已确认结构体定义及调用方；未迁移以避免在自审阶段扩大变更；证据: 审查记录 `REV-001`
- 影响: 不能进入测试域，也不能宣布实现自审完成
- 解决计划: 1. 责任方: 编码实现者；前置条件: 确认 entity 归属；动作: 将请求和响应结构体迁移至 `internal/entity/order/` 并更新引用；完成判据: service 层不再含该数据结构定义；验证入口: 重新执行结构体扫描并复审当前 diff
- 恢复后重入点: 实现自审的结构归位检查
- 去重键: `implementation-review.structure_placement + internal/service/order/create_service.go:45`
```

同一根因已存在 `BLK-*` 时，仅引用该记录和恢复后复审入口，不重复列出解决计划。

## 正例

### 正例 1：实现完成且可读性优先

- 改动范围受控，没有顺手扩大无关修改。
- 方法职责单一，主流程拆成多个可复用原子方法。
- 命名、注释和日志足够支撑理解与追踪。
- 常量、状态、超时和配置没有硬编码散落。
- 结论：可进入后续审查和测试。

### 正例 2：函数略长但结构清晰

- 单个函数略微超过 50 行。
- 但内部结构按步骤拆段，并有中文注释解释为什么暂时不继续拆分。
- 关键常量已抽离，错误处理和日志上下文完整。
- 结论：可通过，但应持续关注后续是否还需继续拆分。

### 正例 3：依赖使用克制且可审计

- 优先复用成熟第三方库，没有重复造轮子。
- 引入的依赖版本明确、来源清楚、文档完善。
- 错误消息默认英文响应，日志保留中文上下文说明。
- 方法注释中标注了“最近修改时间：2026-03-21 14:25:30”。
- 结论：可进入后续审查和测试。

### 正例 4：Go 测试落点自审通过

- 执行 `rg --files -g "*_test.go" | rg -v "^doc/5-tests/"` 无命中。
- 本轮 Go 测试文件均位于 `doc/5-tests/<时间戳>/` ASCII 镜像路径。
- 结论：可进入后续审查和测试。

### 正例 5：Go 服务实现按业务子目录落位

- 执行 `git diff --name-only HEAD -- | rg "^internal/service/[^/]+\\.go$"` 无命中。
- 本轮服务实现落在 `internal/service/order/`、`internal/service/user/` 等子目录。
- 结论：可进入后续审查和测试。

### 正例 6：Go 服务结构体归位到 entity

- 执行 `rg -n "^\\s*type\\s+[A-Za-z_][A-Za-z0-9_]*\\s+struct\\s*{" internal/service` 后人工复核，无请求/响应/第三方结果结构体散落在 `internal/service`。
- 本轮新增请求/响应结构体放在 `internal/entity/order/`、`internal/entity/user/` 等业务子目录。
- 结论：可进入后续审查和测试。

### 正例 7：Go 局部变量逐行声明

- 函数内需要预声明变量时，使用 `var order *model.AdminTenantOrder`、`var err error` 逐行声明。
- 未使用 `var (...)` 分组声明局部变量。
- 结论：可进入后续审查和测试。

### 正例 8：Go 多参数函数签名单行或参数结构体

- 多参数函数优先保持单行签名，调用方能一眼看全参数语义。
- 当参数继续增加时，提炼 `HandleStageFailureParams` 等参数结构体后再传参。
- 结论：可进入后续审查和测试。

### 正例 9：第三方 API 响应先探测后结构体建模

- 文档仅有 curl 示例时，先通过探测脚本获取脱敏响应样例。
- 基于样例定义响应结构体并完成结构体反序列化，不使用 map key 硬编码读取。
- 结论：可进入后续审查和测试。

## 反例

### 反例 1：功能写完但顺手改太多

- 本次目标很小，却顺手改了大段旧逻辑。
- 改动面已经超出最小改动原则。
- 结论：驳回，先回改。

### 反例 2：功能写完但命名、注释和结构明显失衡

- 关键业务分支无中文说明。
- 方法名、变量名大量使用 `tmp`、`data`、`obj`、`res`、`mgr` 之类弱语义名称。
- 方法级注释缺少“最近修改时间”字段。
- 结论：驳回，先修实现质量。

### 反例 3：魔法值和硬编码散落

- 超时时间直接写 `3000`、状态直接写 `"2"`、接口地址直接写在业务代码中。
- 特性开关和文案键没有统一抽离。
- 结论：驳回，先做常量、配置或枚举治理。

### 反例 4：错误处理和日志无语义

- 直接 `panic("fail")`、`throw "error"` 或返回没有上下文的错误。
- 日志只写“开始处理”“执行失败”，没有模块、操作、对象信息。
- 对外错误消息中英文混乱，无法支撑国际化扩展。
- 结论：驳回，先修错误处理和日志。

### 反例 5：冗余逻辑和依赖治理失控

- 代码里保留未使用变量、未使用导入、空分支和重复判断。
- 为一个简单工具能力引入来源不明、未版本化或维护状态不清的依赖。
- 结论：驳回，先清理冗余实现并回收不合规依赖。

### 反例 6：Go 测试落点违规

- 扫描命中 `internal/service/order_service_test.go`（位于 `doc/5-tests/` 之外）。
- 试图以白盒同包为由保留源码目录 `*_test.go`。
- 结论：驳回，先改 seam 并迁移测试落点。

### 反例 7：Go 服务层根目录直落实现

- 扫描命中 `internal/service/create_order.go`。
- 本轮业务实现直接落在 `internal/service/` 根目录，未按业务域拆分子目录。
- 结论：驳回，先迁移到 `internal/service/<domain>/`。

### 反例 8：Go 服务实现散落请求/响应结构体

- 扫描命中 `internal/service/order/create_service.go` 中新增 `CreateOrderRequest`、`CreateOrderResponse`。
- 这些结构体属于数据定义，不应混在服务行为实现文件。
- 结论：驳回，先迁移到 `internal/entity/order/` 后再进入测试。

### 反例 9：Go 局部变量使用分组声明

- 在函数体内写了 `var ( order *model.AdminTenantOrder; err error )`。
- 该写法不符合当前项目局部变量声明约定。
- 结论：驳回，改为逐行单独 `var` 声明后再进入测试。

### 反例 10：Go 多参数函数签名直接换行

- 仅因参数偏多就使用多行参数列表，未先尝试单行签名。
- 也没有提炼参数结构体收口语义。
- 结论：驳回，先改为单行签名或参数结构体。

### 反例 11：第三方 API 响应长期 map 解析

- 用 `map[string]any` 承接第三方响应并通过 key 字符串层层断言读取。
- 没有探测脚本证据，也没有结构体建模计划。
- 结论：驳回，先补探测样例并改为结构体解析。
