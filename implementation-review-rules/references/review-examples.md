# 实现自审正反例

## 正例

### 正例 1：实现完成且可读性优先

- 改动范围受控，没有顺手扩大无关修改。
- 方法职责单一，主流程拆成多个可复用原子方法。
- 命名、注释和日志足够支撑理解与追踪。
- 常量、状态、超时和配置没有硬编码散落。
- 结论：可进入后续审查和测试。

### 正例 2：函数略长但结构清晰

- 单个函数略微超过 50 行。
- 但内部结构按步骤拆段，并有中文注释解释为什么暂时不继续拆分。
- 关键常量已抽离，错误处理和日志上下文完整。
- 结论：可通过，但应持续关注后续是否还需继续拆分。

### 正例 3：依赖使用克制且可审计

- 优先复用成熟第三方库，没有重复造轮子。
- 引入的依赖版本明确、来源清楚、文档完善。
- 错误消息默认英文响应，日志保留中文上下文说明。
- 方法注释中标注了“最近修改时间：2026-03-21 14:25:30”。
- 结论：可进入后续审查和测试。

### 正例 4：Go 测试落点自审通过

- 执行 `rg --files -g "*_test.go" | rg -v "^test/"` 无命中。
- 本轮 Go 测试文件均位于 `test/<时间戳>/` ASCII 镜像路径。
- 结论：可进入后续审查和测试。

### 正例 5：Go 服务实现按业务子目录落位

- 执行 `git diff --name-only HEAD -- | rg "^internal/service/[^/]+\\.go$"` 无命中。
- 本轮服务实现落在 `internal/service/order/`、`internal/service/user/` 等子目录。
- 结论：可进入后续审查和测试。

## 反例

### 反例 1：功能写完但顺手改太多

- 本次目标很小，却顺手改了大段旧逻辑。
- 改动面已经超出最小改动原则。
- 结论：驳回，先回改。

### 反例 2：功能写完但命名、注释和结构明显失衡

- 关键业务分支无中文说明。
- 方法名、变量名大量使用 `tmp`、`data`、`obj`、`res`、`mgr` 之类弱语义名称。
- 方法级注释缺少“最近修改时间”字段。
- 结论：驳回，先修实现质量。

### 反例 3：魔法值和硬编码散落

- 超时时间直接写 `3000`、状态直接写 `"2"`、接口地址直接写在业务代码中。
- 特性开关和文案键没有统一抽离。
- 结论：驳回，先做常量、配置或枚举治理。

### 反例 4：错误处理和日志无语义

- 直接 `panic("fail")`、`throw "error"` 或返回没有上下文的错误。
- 日志只写“开始处理”“执行失败”，没有模块、操作、对象信息。
- 对外错误消息中英文混乱，无法支撑国际化扩展。
- 结论：驳回，先修错误处理和日志。

### 反例 5：冗余逻辑和依赖治理失控

- 代码里保留未使用变量、未使用导入、空分支和重复判断。
- 为一个简单工具能力引入来源不明、未版本化或维护状态不清的依赖。
- 结论：驳回，先清理冗余实现并回收不合规依赖。

### 反例 6：Go 测试落点违规

- 扫描命中 `internal/service/order_service_test.go`（位于 `test/` 之外）。
- 试图以白盒同包为由保留源码目录 `*_test.go`。
- 结论：驳回，先改 seam 并迁移测试落点。

### 反例 7：Go 服务层根目录直落实现

- 扫描命中 `internal/service/create_order.go`。
- 本轮业务实现直接落在 `internal/service/` 根目录，未按业务域拆分子目录。
- 结论：驳回，先迁移到 `internal/service/<domain>/`。
