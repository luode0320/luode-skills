# 自审边界说明

## 用途

用于区分实现自审内置的四组测试前收口检查与功能验证的边界。

## 与内置语法/类型/引用检查的边界

- 当前 skill 既看“实现质量是否像样”，也直接拦“能不能编译、类型对不对、引用是否缺失、构建是否明显失败”。
- 如果项目已有 `tsc`、`vue-tsc`、`go test`、`npm lint`、`pnpm lint` 等命令，应优先借助这些工具完成静态校验，但对外仍统一归口到当前 skill。

## 与内置格式清理的边界

- 当前 skill 要识别并驳回未使用变量、未使用导入、空分支、调试残留和明显冗余逻辑。
- 当前 skill 也直接负责判断“最低格式是否达标”，例如缩进、空行、导入排序和基础排版。
- 若项目有 formatter/linter，优先调用并吸收结果；没有工具时，仍要人工完成最低格式达标判断。

## 与内置目录归位/依赖方向检查的边界

- 当前 skill 直接负责测试前的目录归位、模块落点和依赖方向复查。
- 但如果问题已经上升为“编码前结构设计”或“跨模块架构重排”，仍应回到 `package-structure-rules` 或更上层设计讨论。
- Go 场景下，当前 skill 需要直接收口 `doc/tests/` 外 `*_test.go`、`internal/service/*.go` 根目录落点、`internal/service` 结构体散落、局部变量 `var (...)`、多参数签名直换行与第三方 API 响应 map 硬编码解析等问题。

## 与注释规范的边界

- 当前 skill 会检查注释是否缺失、是否使用中文、是否包含“最近修改时间”字段，以及是否足以支撑理解。
- 更细的注释放置颗粒度、字段注释写法和步骤注释规则，可继续由专门的注释规则承接。

## 与测试的边界

- 当前 skill 不验证功能正确性。
- 当前 skill 只决定是否具备进入测试的实现质量前提。

## 处理原则

- 不要把“功能能跑”误判成“实现已经合格”。
- 不要把纯工具层输出当成通过结论本身；当前 skill 仍要给出统一的通过/驳回判断。
- 不要用“后面再清理”容忍明显的冗余逻辑、魔法值和无语义命名进入测试阶段。
