# 自审边界说明

## 用途

用于区分实现自审与后续语法检查、格式清理、代码归位和测试验证的边界。

## 与语法检查的边界

- 当前 skill 看“实现质量是否像样，是否适合进入测试前流程”。
- 语法检查看“能不能编译、类型对不对、引用是否缺失、构建是否通过”。

## 与格式清理的边界

- 当前 skill 要识别并驳回未使用变量、未使用导入、空分支和明显冗余逻辑。
- 格式清理更偏向执行机械化整理，例如格式统一、空行规范、导入排序和自动清理。
- 也就是说，这个 skill 负责判断“该不该改”，不负责替代工具完成所有“怎么改”。

## 与代码归位审查的边界

- 当前 skill 不负责完整目录归位和依赖方向纠偏。
- 但如果实现为了追求“方便”把常量、配置、错误、枚举、日志助手乱塞位置，当前 skill 仍应指出这是实现质量问题。

## 与注释规范的边界

- 当前 skill 会检查注释是否缺失、是否使用中文、是否包含“最近修改时间”字段，以及是否足以支撑理解。
- 更细的注释放置颗粒度、字段注释写法和步骤注释规则，可继续由专门的注释规则承接。

## 与测试的边界

- 当前 skill 不验证功能正确性。
- 当前 skill 只决定是否具备进入测试的实现质量前提。

## 处理原则

- 不要把“功能能跑”误判成“实现已经合格”。
- 不要把纯工具层的格式动作，误判成实现质量审查的全部内容。
- 不要用“后面再清理”容忍明显的冗余逻辑、魔法值和无语义命名进入测试阶段。
