# 格式清理检查

## 用途

用于定义 `implementation-review-rules` 内置的基础格式与清理闸门，确保进入测试前已经完成最小去噪和排版收口。

## 必查范围

- 未使用导入
- 未使用变量
- 未使用方法引用
- 死代码
- 多余空行
- 明显格式不一致
- 前端 import 排序/分组混乱、重复或未使用 import
- `.vue` 模板 / script / style 区块排版、缩进和连续空行问题
- `.tsx`、`.jsx`、`.ts`、`.js`、`.html`、`.css`、`.scss`、`.less` 中可由 Prettier、ESLint、stylelint 或人工稳定判断的格式问题

## 调试残留处理

- 临时调试代码默认不应进入测试和交付阶段。
- 如果某条日志确有长期价值，应转为正式日志而不是原样保留。
- 断言和诊断检查若只是排障手段，应在任务结束前明确是删除、降级为正式日志，还是保留为正式校验。

## 处理原则

- 只清理噪音，不借机改业务逻辑。
- 若项目 formatter/linter 会触发大面积历史格式化，只处理本轮改动文件。
- 发现“功能修复伪装成格式清理”时，直接判定越界并回到实现阶段重新处理。

## 正反例

### 正例 1：删除未使用导入和临时打印

- 不改变功能行为。
- 只是去噪。

### 正例 2：统一当前小范围改动的排版

- 只整理本次改动相关代码块。
- 不扩散到全目录。

### 反例 1：借格式化重排大段旧逻辑

- 当前任务只改一个点，却把整文件结构大改。
- 结论：越界。

### 反例 2：把功能修复算作清理

- 删除“看起来多余”的代码，实则改变业务行为。
- 结论：不属于本轮清理。
