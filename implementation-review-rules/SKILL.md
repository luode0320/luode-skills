# 实现自审规则

只在“代码已经写完，但还没进测试，需要先做一轮测试前实现收口”时使用这个 skill。
如果当前问题是功能是否正确，请转交测试域。

## Skill 作用与适用场景

- 作为编码完成后的唯一自动测试前审查闸门。
- 在进入测试前检查当前实现是否已经达到“可读、可维护、可追溯、可继续演进”的最低质量。
- 在进入测试前统一收口基础格式、语法/类型/引用以及目录归位/依赖方向，不再依赖并列审查 skill。
- 防止“功能看起来写完了”就直接跳进测试阶段，导致后续返工成本放大。
- 把实现层质量问题尽量拦在测试前，而不是留到联调、回归或上线总结阶段。
- 检查格式化日志调用是否遵循项目约定：能一行则一行；必须换行时，保留日志模板在首行、参数从下一行开始，不把参数拆成碎片多行。
- 作为注释规则的末端闸门，强制检查本轮改动位点是否完成 `comment-placement-granularity-rules` 与 `comment-completion-gate-rules` 的双重要求。
- 在进入测试前统一完成「核心自审要求」列出的实现质量、Go 专项落点、Swagger 同步、复用与回归风险检查；每项属于硬阻断还是默认自查，一律以本文件「分级口径」为准，不在本节逐条重复。

## 自动触发信号

- 功能代码、Bug 修复代码或重构代码刚完成。
- 准备进入测试、联调或回归验证前。
- 需要确认当前实现是否已经达到可交给测试域的最低质量。
- 代码评审前希望先做一轮作者自审，避免明显实现问题流入正式评审。
- Go 代码改动涉及测试落点、服务层落点、结构体归位、局部变量声明、函数签名或第三方响应解析等专项场景（判定是硬阻断还是默认自查，以「核心自审要求」对应编号与「分级口径」为准）。
- 后端 HTTP API 已新增或修改，需确认 Swagger/OpenAPI 已同步。
- 代码已改但尚未确认“改动位点注释补齐”是否完成。
- 本轮目标文件已达 500 行及以上且仍在新增方法、分支或流程步骤。
- 本轮代码虽然可构建，但存在明显排版混乱风险，尚未完成最低格式清理。

## 进入后先做什么

1. 先确认实现目标是否已经落地，不在目标未收敛时机械做实现自审。
2. 再检查实现是否偏离最小改动原则，是否顺手扩大了无关改动面。
3. 优先检查最近改动代码是否仍存在明显的表达层冗余、嵌套三元、过密一行逻辑和无价值包装。
4. 按“可读性、结构、命名、常量抽离、依赖选择、错误处理、日志、注释、编码规范”逐项检查。
5. 核验本轮改动是否已执行注释双 skill：`comment-placement-granularity-rules` 的注释必要性 / 放置位置 / 字段注释 / 过期注释治理检查，以及 `comment-completion-gate-rules` 的改动位点注释补齐、`[参数]` / `[返回]`、最近修改时间、最近修改时间后的改动原因、步骤流程至少顶层 `1` 是否齐全（复杂步骤可含 `1.1/1.2` 细分，多步骤补齐 `1/2/3`）；若方法体存在流程型普通注释但无顶层编号，或字段定义 / 结构体字面量字段映射 / 关键配置对象缺少中文说明，或补丁位点没有“做了什么 + 为什么要加”注释，直接判定不通过。
6. 检查存在步骤流程的函数/方法是否“编号注释就近落位”，禁止函数头部集中写 `1/2/3` 却不在步骤处落位。
7. 判断当前实现是否已经具备进入语法检查和测试阶段的基础条件。
8. 对 Go 项目执行 `*_test.go` 禁放扫描，发现 `doc/5-tests/` 外命中项则先回改。
9. 对 Go 项目执行服务层落点扫描，若本轮改动命中 `internal/service/[^/]+.go` 则先改为业务子目录。
10. 对 Go 项目执行服务实现结构体扫描并人工复核，若请求/响应/第三方结果结构体散落在 `internal/service`，则先迁移到 `internal/entity/<domain>/`。
11. 对 Go 项目执行局部变量分组声明扫描并人工复核，若函数/方法内命中 `var (...)`，则改为逐行单独 `var` 声明。
12. 对 Go 项目执行多参数函数签名扫描并人工复核，若直接采用多行参数列表，则先改为单行签名或提炼参数结构体。
13. 对 Go 项目执行第三方响应 map 解析扫描并人工复核，若命中 `map[string]any` + key 硬编码读取第三方响应，则改为结构体解析；若结构未知，先补探测脚本或暂停反馈用户。
14. 如果本轮改动涉及后端 HTTP API，检查 `api-swagger-rules` 要求的 Swagger/OpenAPI 是否已经同步。
15. 检查本轮重点文件是否达到 500 行及以上且持续膨胀；若命中，要求拆分到多文件并在必要时升级为子目录拆分。
16. 检查新增工具代码是否与既有 util 同语义重复；若可复用实现已存在，要求改为复用并移除重复封装。
17. 检查本轮是否直接修改“最近修改超过7天”的高复用通用代码旧行为；命中则要求改为新增兼容实现。
18. 在进入测试前，完成最低格式达标检查：至少覆盖缩进、对齐、空行、未使用导入、未使用变量、调试残留和明显死代码；未完成时先回改。
19. 在进入测试前，完成基础语法/类型/引用检查；未完成时先补检。
20. 在进入测试前，完成目录归位和依赖方向检查：若存在明显错位、越层或落点违规，先回改。
21. 在进入测试前，列出本轮改动影响的真实用户路径、接口路径或批处理 / 导出入口，并为每条路径预标记验证状态：`待真实运行验证`、`仅静态验证`、`不适用`；`仅静态验证` 必须写明为什么无法运行，且不得作为功能验证通过依据。

## 核心自审要求

> 分级口径（本节唯一权威分级来源，供实施计划冻结与执行期放行判定引用）：
> - 保留为硬阻断（不满足必须停下、不能放行）：真实运行验证 vs 仅静态验证的降级声明与受影响运行路径列出；注释双闸门（`comment-placement-granularity-rules` + `comment-completion-gate-rules`，仅本轮真实改动位点）；Go 目录归位与分层落点（`doc/5-tests/` 外 `*_test.go`、`internal/service` 根目录业务实现、请求/响应/第三方结果结构体归位 `internal/entity/<domain>/`）；后端 HTTP API 改动的 Swagger/OpenAPI 同步；重复造轮子拦截；7 天内高复用通用代码直接改旧行为拦截；基础语法/类型/引用与目录归位/依赖方向的测试前收口。
> - 降级为“模型默认自查 + 提示改进”（默认不作为硬阻断，仅当实施计划已把该项冻结为放行必须项、或用户在本轮明确要求时才升级为硬判）：可读性优先、单一职责、命名语义化、错误处理表达、日志可追溯性表达、魔法值治理、冗余逻辑清理、编码风格等通用质量判断；500 行及以上文件拆分；Go 表达层风格项（函数内 `var (...)` 分组、多参数多行签名、`map[string]any` + key 硬编码解析）。
> - 下方编号要求中，降级项的“应 / 默认应 / 优先”表述按本口径处理：默认给出改进提示而非驳回；保留项仍按原文硬阻断执行。

1. 可读性优先于炫技和过度优化。
说明：优先选择结构清晰、命名直观、职责单一、便于维护的实现；避免过度封装、过度简化、过度链式调用和隐式逻辑。

2. 函数或方法保持单一职责。
说明：核心逻辑应拆成可复用的原子方法；单个函数通常不超过 50 行，确有必要超出时，应以清晰结构和中文注释说明原因。

2.1 文件规模与职责边界必须可维护。
说明：单文件达到 500 行及以上且仍持续新增功能时，默认应按职责拆分到多个文件；业务流程明显增多时，应进一步按业务域拆子目录/子包，不得长期把多职责堆在一个文件中。

3. 禁止魔法值、魔法字符串、魔法状态和硬编码配置。
说明：常量、枚举、超时时间、接口地址、状态码、文案键、特性开关等统一抽离到配置文件、常量模块或枚举定义中。
补充：常量/枚举定义应优先使用显式数值，禁止把对外稳定的状态定义写成 `iota` 递增枚举；若需要长期兼容，应保留清晰的值与注释，不依赖位置顺序表达语义。
补充：单点使用的简单查询条件不应为了“去重”而抽成 const/var；若 SQL 只在一个 `Where` / `Join` / `Raw` 使用点出现，优先就地书写，保持查询语义贴近调用点。

4. 新增或修改代码时必须清理明显冗余实现。
说明：未使用的变量、未使用的导入、空分支、重复判断、显著重复代码和无意义中转逻辑都应在进入测试前移除。

4.1 最近改动代码若存在明显表达层冗余，应在行为不变前提下先收口。
说明：例如嵌套三元、过密链式调用、无价值包装函数、仅描述显而易见行为的废注释；但不允许借机扩大为跨模块重构。

5. 优先使用成熟第三方依赖，避免重复造轮子。
说明：优先选择成熟、活跃、文档完善、版本明确的依赖；禁止随意引入未版本化、维护状态不清或来源不明的依赖。

6. 命名必须语义化。
说明：禁止无意义缩写；若目标语言或生态不适合中文字符，则使用稳定、可读的中文拼音或明确英文业务词。

7. 错误处理必须语义明确、可追踪、可国际化扩展。
说明：禁止直接抛出无语义 `panic` 或 `throw`；错误消息默认使用英文响应，并保留足够的业务上下文。

8. 日志输出必须具备可追溯性。
说明：日志至少体现模块、操作、对象或业务上下文；避免“开始处理”“执行失败”这类无语义日志；日志打印优先使用中文；格式化日志优先单行，必须换行时采用“模板一行、参数下一行”，不要拆成碎片式多行。

9. 新增或修改代码时，方法级注释必须标注最近修改时间并追加本次改动原因简述。
说明：字段名固定为“最近修改时间”，格式为北京时间 `yyyy-MM-DD HH:mm:ss`，例如 `2026-03-21 14:25:30 将查询1小时改成10小时`。

10. 注释、文档与新增代码文件默认使用 UTF-8，注释必须使用中文（除标准协议字段/第三方固定错误原文外）。
说明：不要混用不稳定编码；不要在注释中混入无必要的英文模板式废话；第三方固定错误原文允许保留，但应补充中文解释。

11. Go 测试落点必须满足统一禁放规则。
说明：`doc/5-tests/` 之外不得出现 `*_test.go`；若存在白盒诉求，先补可外部测试 seam（导出接口 / 依赖注入 / 测试钩子），再按测试规则落到 `doc/5-tests/<时间戳>/` ASCII 镜像路径。

12. Go 服务实现默认按业务子目录落位。
说明：本轮改动中不得将业务实现直接新增/修改为 `internal/service/*.go` 根目录文件；应落到 `internal/service/<domain>/` 下。

12.1 Go 请求/响应/第三方结果结构体默认归位到 `internal/entity/<domain>/`。
说明：`internal/service` 实现文件应聚焦服务行为与必要服务接收器结构体；若在 `internal/service` 出现请求结构体、响应结构体、第三方结果结构体，判定为不通过并优先迁移归位。

12.2 Go 函数/方法内局部变量预声明禁止 `var (...)` 分组写法。
说明：需要预声明多个局部变量时，必须逐行单独 `var` 声明（如 `var order *model.AdminTenantOrder`、`var err error`）；包级历史分组声明不作为本条默认阻断项。

12.3 Go 多参数函数签名优先单行，参数过多时优先参数结构体。
说明：禁止仅因参数偏多就直接改为多行参数列表；应先保持单行签名，若单行可读性明显下降，则提炼 `xxxParams` 结构体收口参数语义。

12.4 第三方 API 响应解析优先结构体，禁止长期 map 硬编码。
说明：先读取 `review_acceptance_gates` 判断当前改动是否涉及并要求第三方响应验证。适用且当前必须完成时，响应可确认就必须定义结构体并解析；响应结构未知时先用探测脚本取样。若第三方验证不适用，记录原因和依据后继续；若受限但有替代验证，允许继续准备但不能正式放行。

13. 存在步骤流程的函数/方法必须逐步就近落位编号注释。
说明：当代码存在 1 个及以上步骤时，顶层至少 `1.` 注释必须紧贴对应步骤代码入口；若存在多个连续步骤，需继续补齐 `2.` `3.`；顶层步骤内可继续使用 `1.1` `1.2` 细分；若在函数头部集中编号但后续步骤无对应编号注释，判定为不通过。

14. 本轮改动必须完成注释双 skill 闸门。
说明：新增或修改代码后，若未完成 `comment-placement-granularity-rules` 的注释必要性 / 放置位置 / 字段注释 / 过期注释治理检查，或未完成 `comment-completion-gate-rules` 的改动位点注释、方法最近修改时间、最近修改时间后的改动原因、`[参数]` / `[返回]`、步骤流程至少 `1`（多步骤 `1/2/3`）就近注释、字段定义或结构体字面量字段中文说明、补丁位点“做了什么 + 为什么要加”注释，判定为不通过；只满足其中一部分不算通过。

15. 后端 HTTP API 改动必须同步 Swagger/OpenAPI。
说明：如果本轮改动新增或修改了接口路径、请求 DTO、响应 DTO、错误响应、tag、摘要、鉴权要求或调试入口，但没有同步 Swagger/OpenAPI 文档、注解或暴露策略，判定为不通过。

16. 公共工具必须优先复用，禁止同语义重复封装。
说明：新增工具方法前应检索既有 util 实现；若已有等价能力（相同输入输出语义或可通过轻微参数化复用），必须复用现有实现，不得在业务文件或其他工具文件再写一份“几乎相同”的实现。

17. 高复用通用代码命中 7 天冻结时默认“可新增不可改旧行为”。
说明：当通用代码被多处引用且最近修改时间超过7天，默认不允许直接修改既有行为；应通过新增函数、新参数开关或新版本入口实现需求，并保留旧行为兼容。

18. 受影响运行路径必须显式列出。
说明：本轮代码改动完成后，必须在进入测试域前列出会被用户或上游真实触发的路径，例如 HTTP API、页面交互、导出任务、查询入口、提交入口、定时任务或 CLI 命令。实现自审只负责列清路径与验证准备状态，不替代功能验证；但如果核心路径没有被列出，后续测试容易漏跑，判定为测试前收口不完整。

## 默认执行流程

1. 默认先读 `references/review-scope.md`，确认本轮自审覆盖哪些实现层问题。
2. 如果涉及与语法检查、格式清理、代码归位或测试验证的边界，再读 `references/review-boundaries.md`。
3. 需要检查基础格式与清理项时，再读 `references/format-cleanup-checks.md`。
4. 需要检查语法、类型、引用与明显构建失败风险时，再读 `references/syntax-and-reference-checks.md`。
5. 需要检查目录归位、职责对位与依赖方向时，再读 `references/placement-and-dependency-checks.md`。
6. 如果需要对照通过 / 驳回样例，再读 `references/review-examples.md`。
7. 对 Go 项目补充执行禁放扫描并记录命中路径：`rg --files -g "*_test.go" | rg -v "^doc/5-tests/"`。
8. 对 Go 项目补充执行服务层落点扫描并记录命中路径：`git diff --name-only HEAD -- | rg "^internal/service/[^/]+\\.go$"`。
9. 对 Go 项目补充执行服务实现结构体扫描并记录可疑路径：`rg -n "^\s*type\s+[A-Za-z_][A-Za-z0-9_]*\s+struct\s*{" internal/service`，随后人工复核是否为请求/响应/第三方结果结构体。
10. 对 Go 项目补充执行局部变量分组声明扫描并记录可疑路径：`rg -n "^\s+var\s*\($" --glob "*.go"`，随后人工复核是否位于函数/方法内部。
11. 对 Go 项目补充执行多参数函数签名扫描并记录可疑路径：`rg -n "^\s*func .*\\($|^\s*func .*,\s*$" --glob "*.go"`，随后人工复核是否为“多行参数列表且未提炼参数结构体”。
12. 对 Go 项目补充执行第三方响应 map 解析扫描并记录可疑路径：`rg -n "map\\[string\\](any|interface\\{\\})" --glob "*.go"`，随后人工复核是否用于第三方响应 key 硬编码读取。
13. 核验本轮改动位点注释补齐是否完整（注释双 skill：`comment-placement-granularity-rules` 的必要性 / 位置 / 字段注释 / 过期注释治理 + `comment-completion-gate-rules` 的改动位点注释、`[参数]` / `[返回]`、最近修改时间、最近修改时间后的改动原因、步骤流程至少顶层 `1` 且多步骤补齐 `1/2/3` 的就近注释（可含 `1.1/1.2` 细分）、注释顺序）。
14. 对本轮重点文件补充检查行数与职责密度，识别是否命中“500+ 行持续膨胀”风险。
15. 如果本轮涉及后端 HTTP API，补充检查 Swagger/OpenAPI 注解、文档定义或暴露配置是否同步。
16. 补充执行工具复用扫描并人工复核：若本轮新增工具函数，使用 `rg -n \"func\\s+.*(util|helper|normalize|format|convert|build|parse|validate)\" --glob \"*.go\"`（按语言替换）结合语义比对检查是否存在可复用实现。
17. 补充执行通用代码 7 天冻结检查：对改动到的通用文件执行 `git log -1 --format=%ct -- <file>`，若超过 7 天且属于多处复用代码，人工复核是否直接改了旧行为。
18. 补充确认基础格式、语法/类型/引用、目录归位/依赖方向 4 组检查均已完成；任一缺失，判定当前轮次不可进入测试。
19. 补充输出“受影响运行路径清单”：每条路径至少包含路径 / 入口、触发方式、预期验证方式和当前状态（`待真实运行验证` / `仅静态验证` / `不适用`）；该清单交给 `functional-validation-rules` 继续执行真实验证。
20. 输出通过项、驳回项、风险项和需要回改的实现问题。
21. 自审未通过时，不进入测试域；仅 P0/P1 导致的真实阻断，才在审查文档中按 `../artifact-delivery-gate-rules/references/task-blocker-closure-contract.md` 创建或引用去重后的 `BLK-*` 记录，保留最小回改计划与复审入口。P2/P3、`limited` 和 `not_applicable` 不创建阻断记录。

## 权责边界与不负责事项

- 负责统一测试前实现收口，但不要求替代项目已有 formatter / lint / typecheck 命令本身；若项目有工具，应优先调用并吸收结果。
- 可以要求清理未使用变量、未使用导入、空分支和明显冗余逻辑，也可以直接把这些问题判定为未通过。
- 不把功能正确性验证当成自己的职责，那属于测试域。
- 不在实现目标、需求边界尚未明确时，假装通过“实现自审”掩盖需求问题。
- 不代替 `api-swagger-rules` 设计 Swagger/OpenAPI 方案本身，但可以把“未同步 Swagger/OpenAPI”判定为自审不通过。
- 不允许以“赶进度”为由放过已存在公共工具却重复封装的改动。
- 不允许以“赶进度”为由放过“最近修改超过7天的高复用通用代码被直接改旧行为”的改动。
