# 实现自审规则

只在“代码已经写完，但还没进测试，需要先做一轮实现质量自审”时使用这个 skill。
如果当前问题是语法 / 类型 / 引用错误，请转交 `syntax-check-review-rules`；如果是未使用导入、调试残留和格式噪音，请转交 `cleanup-format-review-rules`。

## Skill 作用与适用场景

- 作为编码完成后的第一道自审关口。
- 检查当前实现是否符合编码规范、注释规范、风格一致性和最小改动原则。
- 在进入测试前尽早发现实现层面的明显质量问题。
- 防止“功能看起来写完了”就直接跳进测试阶段。

## 自动触发信号

- 功能代码、Bug 修复代码或重构代码刚完成。
- 准备进入测试、联调或回归验证前。
- 需要确认当前实现是否已经达到可交给测试域的最低质量。

## 进入后先做什么

1. 先确认实现目标是否已经落地。
2. 再检查实现是否偏离最小改动原则。
3. 检查命名、注释、风格一致性和可读性是否明显失衡。
4. 判断是否已具备进入语法检查和测试阶段的基础条件。

## 默认执行流程

1. 默认先读 `references/review-scope.md`，确认本轮自审覆盖哪些实现层问题。
2. 如果涉及最小改动、注释、风格一致性冲突，再读 `references/review-boundaries.md`。
3. 如果需要对照通过 / 驳回样例，再读 `references/review-examples.md`。
4. 输出通过项、驳回项和需要回改的实现问题。
5. 自审未通过时，不进入测试域。

## 权责边界与不负责事项

- 只负责实现层质量自审，不代替 `syntax-check-review-rules` 做语法和类型层校验。
- 不代替 `cleanup-format-review-rules` 清理未使用导入、死代码和格式噪音。
- 不代替 `code-placement-review-rules` 做目录归位和依赖方向审查。
- 不把功能正确性验证当成自己的职责，那属于测试域。

## 需要暂停并确认的条件

- 当前实现已经明显偏离最小改动原则。
- 注释、命名、结构问题已经严重到需要回到编码阶段重做。
- 当前实现存在多处质量争议，且不同修法会影响后续测试或评审判断。

## 执行通过 / 驳回标准

- 通过：实现目标达成，且没有明显违反最小改动、命名、注释、风格一致性和实现可读性的重大问题。
- 驳回：虽然功能写完，但实现层质量明显不达标，继续进测试只会放大返工成本。

## 执行结果归档要求

- 将自审结论、驳回项和回改建议记录到 `review/`。
- 归档内容至少包含检查范围、通过项、未通过项和下一步动作。
- 如果只是轻量实现且自审无争议，可以简化记录，但应保留结论。

## references 读取规则

- 默认先读 `references/review-scope.md`。
- 只有在判断与相邻审查 skill 的边界时，再读 `references/review-boundaries.md`。
- 只有在需要正反例时，再读 `references/review-examples.md`。
