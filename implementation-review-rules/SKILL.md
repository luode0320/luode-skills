# 实现自审规则

只在“代码已经写完，但还没进测试，需要先做一轮实现质量自审”时使用这个 skill。
如果当前问题是语法 / 类型 / 引用错误，请转交 `syntax-check-review-rules`；如果当前问题是目录归位、分层职责或跨层依赖，请转交 `code-placement-review-rules`；如果当前问题是功能是否正确，请转交测试域。

## Skill 作用与适用场景

- 作为编码完成后的第一道实现质量自审关口。
- 在进入测试前检查当前实现是否已经达到“可读、可维护、可追溯、可继续演进”的最低质量。
- 防止“功能看起来写完了”就直接跳进测试阶段，导致后续返工成本放大。
- 把实现层质量问题尽量拦在测试前，而不是留到联调、回归或上线总结阶段。
- 作为注释规则的末端闸门，强制检查本轮改动位点是否完成 `code-comment-rules` 要求。
- 在进入测试前提前发现 Go 测试落点违规（`test/` 外 `*_test.go`），降低后续返工。
- 在进入测试前提前发现 Go 服务层落点违规（本轮改动直接落 `internal/service/*.go`），避免形成大平层。
- 在进入测试前提前发现“HTTP API 已改，但 Swagger/OpenAPI 仍未同步”的接口契约漂移问题。
- 在进入测试前拦截“500+ 行文件继续堆功能”的结构风险，要求按职责拆分文件，必要时拆子目录。

## 自动触发信号

- 功能代码、Bug 修复代码或重构代码刚完成。
- 准备进入测试、联调或回归验证前。
- 需要确认当前实现是否已经达到可交给测试域的最低质量。
- 代码评审前希望先做一轮作者自审，避免明显实现问题流入正式评审。
- Go 代码改动涉及测试文件，需确认没有在源码目录落地 `*_test.go`。
- Go 代码改动涉及服务实现，需确认本轮改动未把业务实现直接放在 `internal/service/` 根目录。
- 后端 HTTP API 已新增或修改，需确认 Swagger/OpenAPI 已同步。
- 代码已改但尚未确认“改动位点注释补齐”是否完成。
- 本轮目标文件已达 500 行及以上且仍在新增方法、分支或流程步骤。

## 进入后先做什么

1. 先确认实现目标是否已经落地，不在目标未收敛时机械做实现自审。
2. 再检查实现是否偏离最小改动原则，是否顺手扩大了无关改动面。
3. 优先检查最近改动代码是否仍存在明显的表达层冗余、嵌套三元、过密一行逻辑和无价值包装。
4. 按“可读性、结构、命名、常量抽离、依赖选择、错误处理、日志、注释、编码规范”逐项检查。
5. 核验本轮改动是否已执行 `code-comment-rules`：改动位点注释补齐、最近修改时间、`[参数]` / `[返回]`、多步骤 `1/2/3` 是否齐全。
6. 检查多步骤函数/方法是否“编号注释就近落位”，禁止函数头部集中写 `1/2/3` 却不在步骤处落位。
7. 判断当前实现是否已经具备进入语法检查和测试阶段的基础条件。
8. 对 Go 项目执行 `*_test.go` 禁放扫描，发现 `test/` 外命中项则先回改。
9. 对 Go 项目执行服务层落点扫描，若本轮改动命中 `internal/service/[^/]+.go` 则先改为业务子目录。
10. 如果本轮改动涉及后端 HTTP API，检查 `api-swagger-rules` 要求的 Swagger/OpenAPI 是否已经同步。
11. 检查本轮重点文件是否达到 500 行及以上且持续膨胀；若命中，要求拆分到多文件并在必要时升级为子目录拆分。

## 核心自审要求

1. 可读性优先于炫技和过度优化。
说明：优先选择结构清晰、命名直观、职责单一、便于维护的实现；避免过度封装、过度简化、过度链式调用和隐式逻辑。

2. 函数或方法保持单一职责。
说明：核心逻辑应拆成可复用的原子方法；单个函数通常不超过 50 行，确有必要超出时，应以清晰结构和中文注释说明原因。

2.1 文件规模与职责边界必须可维护。
说明：单文件达到 500 行及以上且仍持续新增功能时，默认应按职责拆分到多个文件；业务流程明显增多时，应进一步按业务域拆子目录/子包，不得长期把多职责堆在一个文件中。

3. 禁止魔法值、魔法字符串、魔法状态和硬编码配置。
说明：常量、枚举、超时时间、接口地址、状态码、文案键、特性开关等统一抽离到配置文件、常量模块或枚举定义中。

4. 新增或修改代码时必须清理明显冗余实现。
说明：未使用的变量、未使用的导入、空分支、重复判断、显著重复代码和无意义中转逻辑都应在进入测试前移除。

4.1 最近改动代码若存在明显表达层冗余，应在行为不变前提下先收口。
说明：例如嵌套三元、过密链式调用、无价值包装函数、仅描述显而易见行为的废注释；但不允许借机扩大为跨模块重构。

5. 优先使用成熟第三方依赖，避免重复造轮子。
说明：优先选择成熟、活跃、文档完善、版本明确的依赖；禁止随意引入未版本化、维护状态不清或来源不明的依赖。

6. 命名必须语义化。
说明：禁止无意义缩写；若目标语言或生态不适合中文字符，则使用稳定、可读的中文拼音或明确英文业务词。

7. 错误处理必须语义明确、可追踪、可国际化扩展。
说明：禁止直接抛出无语义 `panic` 或 `throw`；错误消息默认使用英文响应，并保留足够的业务上下文。

8. 日志输出必须具备可追溯性。
说明：日志至少体现模块、操作、对象或业务上下文；避免“开始处理”“执行失败”这类无语义日志；日志打印优先使用中文。

9. 新增或修改代码时，方法级注释必须标注最近修改时间。
说明：字段名固定为“最近修改时间”，格式为北京时间 `yyyy-MM-DD HH:mm:ss`，例如 `2026-03-21 14:25:30`。

10. 注释、文档与新增代码文件默认使用 UTF-8，注释必须使用中文（除标准协议字段/第三方固定错误原文外）。
说明：不要混用不稳定编码；不要在注释中混入无必要的英文模板式废话；第三方固定错误原文允许保留，但应补充中文解释。

11. Go 测试落点必须满足统一禁放规则。
说明：`test/` 之外不得出现 `*_test.go`；若存在白盒诉求，先补可外部测试 seam（导出接口 / 依赖注入 / 测试钩子），再按测试规则落到 `test/<时间戳>/` ASCII 镜像路径。

12. Go 服务实现默认按业务子目录落位。
说明：本轮改动中不得将业务实现直接新增/修改为 `internal/service/*.go` 根目录文件；应落到 `internal/service/<domain>/` 下。

13. 多步骤函数/方法的编号注释必须逐步就近落位。
说明：当代码存在 2 个及以上连续步骤时，`1.` `2.` `3.` 注释必须紧贴对应步骤代码入口；若在函数头部集中编号但后续步骤无对应编号注释，判定为不通过。

14. 本轮改动必须完成 `code-comment-rules` 注释补齐闸门。
说明：新增或修改代码后，若未覆盖改动位点注释、方法最近修改时间、`[参数]` / `[返回]` 或多步骤 `1/2/3` 就近注释，判定为不通过。

15. 后端 HTTP API 改动必须同步 Swagger/OpenAPI。
说明：如果本轮改动新增或修改了接口路径、请求 DTO、响应 DTO、错误响应、tag、摘要、鉴权要求或调试入口，但没有同步 Swagger/OpenAPI 文档、注解或暴露策略，判定为不通过。

## 默认执行流程

1. 默认先读 `references/review-scope.md`，确认本轮自审覆盖哪些实现层问题。
2. 如果涉及与语法检查、格式清理、代码归位或测试验证的边界，再读 `references/review-boundaries.md`。
3. 如果需要对照通过 / 驳回样例，再读 `references/review-examples.md`。
4. 对 Go 项目补充执行禁放扫描并记录命中路径：`rg --files -g "*_test.go" | rg -v "^test/"`。
5. 对 Go 项目补充执行服务层落点扫描并记录命中路径：`git diff --name-only HEAD -- | rg "^internal/service/[^/]+\\.go$"`。
6. 核验本轮改动位点注释补齐是否完整（`code-comment-rules` 四项：改动位点注释、最近修改时间、`[参数]` / `[返回]`、`1/2/3` 就近注释）。
7. 对本轮重点文件补充检查行数与职责密度，识别是否命中“500+ 行持续膨胀”风险。
8. 如果本轮涉及后端 HTTP API，补充检查 Swagger/OpenAPI 注解、文档定义或暴露配置是否同步。
9. 输出通过项、驳回项、风险项和需要回改的实现问题。
10. 自审未通过时，不进入测试域。

## 权责边界与不负责事项

- 只负责实现层质量自审，不代替 `syntax-check-review-rules` 做完整语法和类型校验。
- 可以要求清理未使用变量、未使用导入、空分支和明显冗余逻辑，但不代替专门的格式化工具执行机械整理。
- 不代替 `code-placement-review-rules` 做目录归位和依赖方向纠偏。
- 不把功能正确性验证当成自己的职责，那属于测试域。
- 不在实现目标、需求边界尚未明确时，假装通过“实现自审”掩盖需求问题。
- 不代替 `api-swagger-rules` 设计 Swagger/OpenAPI 方案本身，但可以把“未同步 Swagger/OpenAPI”判定为自审不通过。

## 需要暂停并确认的条件

- 当前实现已经明显偏离最小改动原则。
- 注释、命名、结构问题已经严重到需要回到编码阶段重做。
- 当前实现存在多处质量争议，且不同修法会影响后续测试或评审判断。
- 当前方案为了“优化”而引入过多抽象、过多依赖或明显维护负担，是否回退需要确认。
- Go 项目命中 `test/` 外 `*_test.go`，但是否改 seam 方案尚未达成一致。
- Go 项目命中 `internal/service/[^/]+.go` 根目录落点，但业务子目录拆分方案尚未达成一致。
- 本轮涉及后端 HTTP API，但 Swagger/OpenAPI 是否必须同步、该以哪份注解或文档定义为准仍未确认。
- 500+ 行文件拆分会引发跨模块接口调整，且本轮范围是否允许拆分尚未确认。

## 执行通过 / 驳回标准

- 通过：实现目标达成，且没有明显违反可读性优先、单一职责、命名语义化、常量抽离、依赖治理、错误处理、日志可追溯、方法注释更新时间、UTF-8 与中文注释（除标准协议字段/第三方固定错误原文外）、编号注释就近落位等要求的重大问题；并且本轮改动已完成 `code-comment-rules` 的注释补齐闸门；对 500 行及以上且持续膨胀的文件已完成拆分或给出本轮可执行拆分方案；Go 场景下 `test/` 外不存在 `*_test.go`，且本轮改动未命中 `internal/service/[^/]+.go` 根目录落点；涉及后端 HTTP API 时，Swagger/OpenAPI 已同步。
- 驳回：虽然功能写完，但实现层质量明显不达标，或本轮改动未完成 `code-comment-rules` 注释补齐，或 500 行及以上文件继续堆功能却没有拆分动作/方案，或 Go 场景命中 `test/` 外 `*_test.go` 禁放项，或本轮改动命中 `internal/service/[^/]+.go` 根目录落点；或本轮涉及后端 HTTP API 但 Swagger/OpenAPI 未同步；继续进测试只会放大返工成本。
- 驳回：发现函数头部集中编号注释、后续步骤没有对应编号注释就近落位。

## 执行结果归档要求

- 将自审结论、驳回项和回改建议记录到 `review/`。
- 归档内容至少包含检查范围、通过项、未通过项和下一步动作。
- 如果只是轻量实现且自审无争议，可以简化记录，但应保留结论。
