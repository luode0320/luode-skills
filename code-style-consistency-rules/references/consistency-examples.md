# 风格一致性正反例

## 正例

### 正例 1：延续同目录写法

- 同目录都使用早返回和简洁函数组织。
- 新增函数继续沿用同样方式。

### 正例 2：局部跟随，不扩大全局

- 当前文件使用一种稳定测试写法。
- 本次只在当前文件范围内跟随，不顺手改其他文件。

### 正例 3：Go 路由注册区段使用代码块

- 同一函数步骤中有多条 `POST` 注册语句。
- 使用代码块 `{ ... }` 包裹注册区段，区段内仅保留路由注释与注册语句。
- 结果是步骤边界清晰，视觉结构稳定。

### 正例 4：Go 局部变量逐行声明

- 在函数内需要预声明多个变量时，使用逐行 `var` 声明。
- 示例：`var order *model.AdminTenantOrder`、`var err error`。
- 结果是变量语义更直观，后续增删变量时改动更聚焦。

### 正例 5：Go 多参数函数签名单行或参数结构体

- 参数数量可控时，函数签名保持单行，例如 `func (w *ExecutorWorker) handleStageFailure(order *model.AdminTenantOrder, stageFailedStatus, failCode int, failMsg string, retryable bool) error`。
- 参数进一步增多时，提炼 `HandleStageFailureParams` 结构体并传参，避免直接拆成多行参数列表。
- 结果是入口更直观，调用方也更容易理解参数语义。

## 反例

### 反例 1：带入个人格式偏好

- 仓库普遍使用一种结构。
- 本次改动突然改成另一套个人习惯写法。

### 反例 2：借局部改动做全目录风格清洗

- 当前只改一个函数。
- 却把整个目录的空行、缩进、组织方式全部重排。

### 反例 3：借简化之名引入外部风格

- 项目原本偏显式、步骤化表达。
- 本次改动突然改成高密度链式调用、嵌套三元或陌生抽象风格。
- 结果是局部看似“更短”，但与项目整体写法割裂。

### 反例 4：Go 路由注册区段直接散落

- 同一阶段有多条路由注册语句，却没有使用代码块 `{ ... }` 收口。
- 结果导致步骤注释与注册语句边界模糊，不符合项目约定。

### 反例 5：Go 局部变量使用 var 分组声明

- 在函数或方法内部写 `var ( ... )` 声明多个局部变量。
- 例如把 `order` 和 `err` 放进同一个分组块。
- 结果不符合当前项目约定，应改为逐行单独 `var` 声明。

### 反例 6：Go 多参数函数签名按参数换行

- 仅因为参数较多，就把函数签名改成“每个参数一行”的多行参数列表。
- 没有评估是否可保持单行，或是否应提炼参数结构体。
- 结果是声明视觉长度增加，读者需要跨多行拼接函数语义。
