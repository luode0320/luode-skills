# 用户风格反馈捕获与学习流程

只在用户以文字或截图指出“某种写法不对 / 这个风格不对 / 不能这么写 / 这样写不行”时使用本流程。
目标：把用户不喜欢的写法沉淀为一条反例、把用户喜欢的写法记为对应正例，经用户确认后写入全局反例库 `user-style-feedback-library.md`，此后跨项目不再产出该写法。

## 触发信号

- 文字信号（语义等价即命中，不要求逐字匹配）：`不能这么写`、`这个风格不对`、`这个写法不对`、`这样写不行`、`别这样写`、`这写法我不喜欢`、`改掉这种写法`、`以后不要这么写`。
- 截图信号：用户发送含代码的截图并表达否定，由 `image-redbox-focus-rules` 聚焦红框后路由到本流程；读图理解沿用模型原生多模态能力，本流程不做 OCR。
- 命中触发后先按本流程处理，再回到常规一致性检查。

## 处理步骤

1. 定位被否定写法：从文字或截图中确认用户具体反对的代码片段与场景（语言、所在结构、触发点）。
2. 提取正例：确认用户期望的写法。若用户已给出期望写法，直接采用；若未给出，向用户追问期望写法，不凭空生成正例，不写入库。
3. 归一化与去重键：按 `style-case-template.md` 生成 `语言|场景|反例签名` 三元组去重键，先规范化大小写与空白。
4. 组装 candidate：按 `style-case-template.md` 字段组装一条 `status: candidate` 条目（反例代码块 + 正例代码块 + 一句话规则）。
5. 回显确认：把 candidate 回显给用户，请用户确认或修正。candidate 只在当轮对话展示，不落盘。
6. 写入 active：用户确认后，把条目改为 `status: active` 写入 `user-style-feedback-library.md`；用户否决则丢弃 candidate，不留磁盘痕迹。
7. 去重合并：写入前用去重键比对；命中同键只更新原条目的确认时间与出现次数，不新增正文；正例有更新则更新原条目正例。
8. 收口联动：因本次写入改动了本 skill 的 `description` 或 `##` 标题时，收口前重跑 `python skill-dictionary/generate_dictionary.py` 刷新字典产物，并走 `skill-compliance-gate-rules`。

## 与写码前加载的衔接

- 写入 active 后，`code-generation-style-rules` 在下一次写码前加载本库，把该反例并入本轮风格契约的“禁用写法”。
- 写后闸门命中该反例时驳回并给出正例改写。

## 边界

- 本流程只处理“用户对代码写法的否定反馈”；需求、Bug、功能问题不走本流程。
- 全局通用偏好写入本库；项目专属一次性约定走 `PROJECT_STYLE.md` 与 `project-style-rules`。
- 无法脱敏、无法给出可复用正例、或只适用于一次性上下文的反馈，不写入库，只在当轮说明。
