# 项目设计文档审查镜像规则

## 目标

只检查当前项目设计文档判断是否满足 `project-design-doc-rules` 的要求，不执行文档同步或补建。

## 源规则

以下判定严格沿用源 skill：

- 根目录 `项目设计.md` 及历史同类文档只是弱参考源
- 代码、当前有效文档和最近变更优先于历史设计文档
- 应判断已有设计文档是否偏移、是否可信
- 需要时应收口到根目录 `项目设计.md`
- 未完成完整项目分析前，不机械补建空泛设计文档

## 自动触发信号

- 用户要求分析整个项目、梳理架构或看模块设计。
- 用户要求生成、更新、同步或补全根目录 `项目设计.md`。
- 根目录存在多个历史设计总览文档，需要判断是否可信。
- 项目目录结构、模块边界、主链路或关键依赖刚发生明显变化。
- 主线程已声明使用 `project-design-doc-rules`，但文档偏移风险较高。

## 进入后先做什么

1. 先确认当前任务确实是项目级设计文档问题。
2. 检查根目录 `项目设计.md` 或历史同类文档是否已识别。
3. 检查是否按弱参考方式对待历史文档，而不是直接当成事实。
4. 检查是否对照真实代码、当前有效文档和最近变更做偏移判断。
5. 检查是否已经形成“更新还是补建”的明确结论。

## 默认执行流程

1. 先读取 `project-design-doc-rules`。
2. 需要核对候选文档与证据优先级时，再读源 skill 的 `references/discovery-and-priority.md`。
3. 需要核对同步时机或补建条件时，再读源 skill 的 `references/sync-policy.md`。
4. 需要核对设计文档模板时，再读源 skill 的 `references/design-template.md`。
5. 输出通过项、偏移判断缺口、主入口收口问题和同步时机问题。
6. 不直接写 `项目设计.md`，只输出问题和建议。

## 权责边界与不负责事项

- 不代替 `artifact-storage-rules` 做路径命名总规则裁定。
- 不代替 `recent-context-bootstrap-rules` 做会话预热。
- 不代替 `project-timeline-rules` 输出长期时间线。
- 不代替 `package-structure-rules`、`code-placement-review-rules`、`implementation-plan-rules` 做代码分层或实施方案细节判断。
- 不执行设计文档同步或补建。

## 需要暂停并确认的条件

- 根目录同时存在多份历史设计总览且内容相互冲突。
- 当前项目还没完成有效整项目分析，却要求直接生成完整 `项目设计.md`。
- 设计文档涉及大范围迁移、重命名或历史文档合并。
- 代码现状、当前有效文档和历史设计文档冲突严重。

## 执行通过 / 驳回标准

- 通过：能够明确说明设计文档是否存在、是否可信、是否偏移，以及应更新还是补建根目录 `项目设计.md`。
- 驳回：直接把历史设计文档当成当前事实，或在项目分析依据不足时机械推动生成空泛文档。

## 执行结果归档要求

- 如本轮审查发现设计文档偏移、同步时机或主入口收口缺口，应将结论记录到 `analysis/` 或评审记录。
- 归档内容至少包含文档对象、偏移点、证据来源和建议处理方向。
- 如果只是一次轻量预审且无争议，可不单独归档，但应保证评审可追溯。

## 读取规则

- 默认先读 `project-design-doc-rules`。
- 只有在核对候选文档与证据优先级时，再读 `references/discovery-and-priority.md`。
- 只有在核对同步时机和补建条件时，再读 `references/sync-policy.md`。
- 只有在核对设计文档模板时，再读 `references/design-template.md`。
