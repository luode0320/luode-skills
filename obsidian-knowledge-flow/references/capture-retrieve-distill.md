# 捕获、检索、沉淀

本 skill 的默认节奏是“会话开始先检索、会话总结再沉淀”。所有 vault 读写都走 Obsidian CLI；CLI 不可用时阻断。

## 会话开始检索流程

1. 先判断当前请求是否依赖历史知识。出现以下任一信号时触发检索：
   - “上次”“之前”“我们约定”“项目规则”“当时怎么说的”“从知识库找”“Obsidian 里”。
   - 重复出现的人、项目、仓库、工具、术语、产品、组织或长期偏好。
   - 问题涉及历史决策、既有流程、项目事实、旧来源、已沉淀 prompt 或调试经验。
2. 解析并校验 vault 根目录，检查 Obsidian CLI 可用性。
3. 构造检索词集合：
   - 用户原始用词。
   - 别名和缩写。
   - 可能的中英文变体。
   - 标签、MOC 名称和实体名。
4. 先查导航入口，再查具体内容：
   - `INDEX.md` 和 `30-MOCs/`。
   - 精确文件名、标题和别名。
   - 标签和 frontmatter。
   - wikilinks 和 backlinks。
   - 全文和上下文行。
5. 回答前读取最匹配的笔记。一次典型检索读取 3-7 篇笔记；只有证据不足时才扩大范围。
6. 回答时明确区分“检索到的事实”“当前推断”和“缺失证据”。
7. 检索笔记对答案有实质支撑时，引用本地笔记路径。

如果当前请求不依赖历史知识，不要为了形式自动读取 vault。

## 总结阶段捕获流程

会话总结、阶段收口或最终回复前，必须做一次轻量捕获判断。

1. 先列出本轮可能值得复用的信息：
   - 稳定事实、项目口径、环境路径、工具命令。
   - 用户偏好、工作流约束、命名规则、输出格式。
   - 决策、取舍、长期流程、定义、术语和别名。
   - 来源链接、仓库路径、调试经验、失败原因和恢复路径。
2. 排除不应捕获的信息：
   - 纯闲聊、情绪性表达、一次性过程语句。
   - 未确认猜测、临时中间方案、已被推翻的旧说法。
   - secret、API key、密码、私有 token 或凭据原值。
3. 先通过 CLI 检索现有笔记，按标题、别名、标签和 wikilink 判断是否已有承接位置。
4. 对每条可长期保存的信息分类：
   - 会话上下文 -> `10-Sessions/`
   - 稳定概念、规则、决策、流程、偏好 -> `20-Knowledge/`
   - 人、项目、仓库、工具、术语、产品、组织 -> `40-Entities/`
   - 网页、视频、仓库、论文、数据集 -> `50-Sources/`
   - 不确定片段 -> `00-Inbox/`
5. 主题已存在时，优先更新现有笔记。
6. 只有信息足够稳定，或新增笔记能提升检索效率时，才创建新笔记。
7. 当笔记变成有用导航入口时，补充 backlinks，并更新相关 MOC 或 `INDEX.md`。

不要捕获每一条消息。没有可复用信息时，本阶段直接跳过写入。

## 沉淀流程

沉淀是把会话材料转成长期知识笔记。

1. 读取会话笔记，以及它链接的来源笔记和实体笔记。
2. 只抽取稳定信息：事实、决策、定义、流程、偏好、约束或可复用经验。
3. 尽量合并进已有知识笔记或实体笔记。
4. 通过 `source_sessions` 保留会话笔记作为证据。
5. 把长期知识笔记反向链接到相关实体、MOC 和来源笔记。
6. 未解决问题保留在会话笔记或 `00-Inbox/` 中，不提升为事实。

## 排序启发式

多个笔记同时命中时，用下面的轻量评分辅助排序：

- 精确标题或别名命中：+5
- 命中启用中的 MOC：+4
- 命中标签或实体：+3
- backlink 邻域命中：+2
- 近期会话命中：+1
- `active` 状态：+1
- `stale`、`deprecated` 或 `retired`：-3
- `conflicted`：-5，除非用户询问冲突本身
