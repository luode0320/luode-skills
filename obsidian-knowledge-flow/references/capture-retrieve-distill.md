# 捕获、检索、沉淀

本 skill 的默认节奏是“先加载项目本地四件套，再做 Obsidian 选择性判断，需要时检索，收口时按价值沉淀”。项目本地 Markdown 与 vault 是两条独立链路：前者使用标准文件工具，后者所有读写都走公开 bridge；bridge doctor 失败时阻断。

## 项目本地四件套先行

1. 读取项目目录父目录的当前平台规则文件。
2. 在项目根目录检测 `PROJECT_CURRENT.md`、`PROJECT_MEMORY.md` 和 `PROJECT_HISTORY.md`；缺失时创建最小 UTF-8 模板。
3. 按“`PROJECT_CURRENT.md` -> `PROJECT_MEMORY.md`”读取当前状态与稳定规则。
4. 普通启动不读 `PROJECT_HISTORY.md`；只有历史追问、当前状态不足或真实卡点时，按日期、关键词或事件类型窄读。
5. 项目状态覆盖写入 `PROJECT_CURRENT.md` 前检查 UTF-8 字节数，不得超过 51,200；超限先压缩并阻断，不静默截断。稳定规则写入 `PROJECT_MEMORY.md`，历史事件写入 `PROJECT_HISTORY.md` 时只追加。

项目四件套已经能回答当前项目问题时，保持 `Obsidian:不适用`，不为形式调用 bridge。只有跨项目历史、vault 笔记或长期知识依赖时才进入下面的检索流程。

## 选择性默认判定流程

每轮仓库任务先完成轻量判断，并输出 `Obsidian:<检索/沉淀/不适用/阻断>`。

1. 先看是否存在历史知识依赖：
   - 有历史决策、知识库内容、长期偏好、重复实体、项目事实、旧来源或“上次 / 之前 / 我们约定 / 当时怎么说 / Obsidian 里 / 知识库里”等信号，判定为 `检索`。
2. 再看是否存在收口沉淀价值：
   - 本轮形成未来会复用的事实、决策、流程、定义、偏好、来源、调试经验、规则口径或稳定命令，判定为 `沉淀`。
3. 两者都没有时，判定为 `不适用`：
   - 普通实现、普通文档、一次性过程说明或临时中间动作，不调用 bridge，不读 vault，也不写笔记。
4. 本应检索或沉淀但 bridge doctor / vault 前置条件失败时，判定为 `阻断`：
   - 说明命令、vault 根目录和恢复动作，不得改用 `rg`、`Get-Content` 或直接文件写入替代。
5. 路径已经固定到 `D:\obsidian_data` + `知识库/` 之后，判断过程只负责决定是否需要检索、沉淀或阻断，不再把 root probing、环境变量回查或候选 vault 比较当成独立步骤。

所有 bridge 写入都必须在同一请求内串行完成分块、目标操作和 CLI readback；应用恢复最多隐藏启动一次并重试一次，不能把批处理失败变成无限重试。`create`、`append` 或 `INDEX` 更新没有 `verified=true` readback 时，批次状态只能记为失败或待人工恢复。

## 会话开始检索流程

1. 先判断当前请求是否依赖历史知识。出现以下任一信号时触发检索：
   - “上次”“之前”“我们约定”“项目规则”“当时怎么说的”“从知识库找”“Obsidian 里”。
   - 重复出现的人、项目、仓库、工具、术语、产品、组织或长期偏好。
   - 问题涉及历史决策、既有流程、项目事实、旧来源、已沉淀 prompt 或调试经验。
2. 直接使用固定映射 `D:\obsidian_data` + `知识库/`，并运行 bridge doctor；WSL 自动由 bridge 使用 Windows interop。
3. 构造检索词集合：
   - 用户原始用词。
   - 别名和缩写。
   - 可能的中英文变体。
   - 标签、MOC 名称和实体名。
4. 先查导航入口，再查具体内容：
   - `知识库/INDEX.md` 和 `知识库/30-MOCs/`。
   - 精确文件名、标题和别名。
   - 标签和 frontmatter。
   - wikilinks 和 backlinks。
   - 全文和上下文行。
5. 回答前读取最匹配的笔记。一次典型检索读取 3-7 篇笔记；只有证据不足时才扩大范围。
6. 回答时明确区分“检索到的事实”“当前推断”和“缺失证据”。
7. 检索笔记对答案有实质支撑时，引用本地笔记路径。

如果当前请求不依赖历史知识，不要为了形式自动读取 vault；此时应保持 `Obsidian:不适用`。

## 总结阶段捕获流程

会话总结、阶段收口或最终回复前，必须做一次轻量捕获判断；只有判定为 `沉淀` 时才进入 bridge 检索和写入准备。

1. 先列出本轮可能值得复用的信息：
   - 稳定事实、项目口径、环境路径、工具命令。
   - 用户偏好、工作流约束、命名规则、输出格式。
   - 决策、取舍、长期流程、定义、术语和别名。
   - 来源链接、仓库路径、调试经验、失败原因和恢复路径。
2. 排除不应捕获的信息：
   - 纯闲聊、情绪性表达、一次性过程语句。
   - 未确认猜测、临时中间方案、已被推翻的旧说法。
   - secret、API key、密码、私有 token 或凭据原值。
3. 先通过 bridge 检索现有笔记，按标题、别名、标签和 wikilink 判断是否已有承接位置。
4. 对每条可长期保存的信息分类：
   - 会话上下文 -> `知识库/10-Sessions/`
   - 稳定概念、规则、决策、流程、偏好 -> `知识库/20-Knowledge/`
   - 人、项目、仓库、工具、术语、产品、组织 -> `知识库/40-Entities/`
   - 网页、视频、仓库、论文、数据集 -> `知识库/50-Sources/`
   - 不确定片段 -> `知识库/00-Inbox/`
5. 主题已存在时，优先更新现有笔记。
6. 只有信息足够稳定，或新增笔记能提升检索效率时，才创建新笔记。
7. 当笔记变成有用导航入口时，补充 backlinks，并更新相关 MOC 或 `知识库/INDEX.md`。

不要捕获每一条消息。没有可复用信息时，本阶段直接跳过写入，并保持 `Obsidian:不适用`。

## 沉淀流程

沉淀是把会话材料转成长期知识笔记。

1. 读取会话笔记，以及它链接的来源笔记和实体笔记。
2. 只抽取稳定信息：事实、决策、定义、流程、偏好、约束或可复用经验。
3. 尽量合并进已有知识笔记或实体笔记。
4. 通过 `source_sessions` 保留会话笔记作为证据。
5. 把长期知识笔记反向链接到相关实体、MOC 和来源笔记。
6. 未解决问题保留在会话笔记或 `知识库/00-Inbox/` 中，不提升为事实。

全量 vault 或目录沉淀时，必须把“已读取并归纳每篇源文档”作为完成定义，而不是只创建来源地图或目录索引。

- 开始前统计来源 Markdown 总数，并按顶层目录或稳定主题拆批次。
- 每篇源文档至少沉淀一条记录：源路径、分类/子类、摘要线索、关键小节或关键词、状态；读取失败必须写明失败原因。
- 每个批次生成一篇逐篇沉淀笔记，记录文档数、成功读取数、失败数、敏感主题数、子类统计和逐篇摘要表。
- 生成一篇全量总览，链接所有批次，汇总应处理数、已读取数、失败数和目标笔记入口。
- 全量总览、批次笔记和 `知识库/INDEX.md` 入口都必须通过 bridge 创建、读取或追加，并保留每次写入的 `verified=true` readback 证据。
- 完成验证必须能回证覆盖率：批次表格数据行数等于对应来源 Markdown 数，所有批次数量之和等于来源总数。
- 高敏目录或主题只沉淀通用线索、标题、小节和可复用经验，不摘录密钥、账号、token、私钥、助记词、服务器地址、连接串或其他凭据原值。
- 长任务要支持可恢复的批次执行；某批失败时保留已完成批次证据，并从失败批次重新运行，不把半成品声明为全量完成。
- 自动恢复只允许 bridge 已定义的一次有限策略；同一批次仍失败时保留脱敏错误和已完成批次，转人工恢复，不得无限重试。

## 排序启发式

多个笔记同时命中时，用下面的轻量评分辅助排序：

- 精确标题或别名命中：+5
- 命中启用中的 MOC：+4
- 命中标签或实体：+3
- backlink 邻域命中：+2
- 近期会话命中：+1
- `active` 状态：+1
- `stale`、`deprecated` 或 `retired`：-3
- `conflicted`：-5，除非用户询问冲突本身
