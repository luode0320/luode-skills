# 冲突与过期

## 安全规则

- 不要用更弱或更旧的来源静默覆盖长期事实。
- 除非用户明确要求且笔记已经脱敏或限定访问范围，否则不要捕获 secret、API key、密码、token 或私有凭据。
- 外部来源摘要默认只是摘要，不是权威事实；只有一手来源或用户标记为权威时才提升权重。
- 保留用户手写笔记和评论。优先追加证据或增加带日期的更新章节。

## 状态含义

- `active`: 当前最可信事实或仍有用的笔记。
- `stale`: 可能过期；检索时需谨慎。
- `conflicted`: 两个或多个来源互相矛盾，且没有权威来源裁决。
- `deprecated`: 仍有历史价值，但已被新规则、新路径或新决策替代。
- `retired`: 不再使用，只保留历史。
- `archived`: 已移出活跃检索范围。
- `candidate`: 已完成根因确认和同输入 local 复验，但尚未满足 active 晋级门禁；只可作为诊断线索。
- `superseded`: 已被新的执行案例替代；保留历史证据，不参与自动复用。
- `rejected`: 方案被验证为错误或不安全；不得自动恢复为 active。

执行案例的当前状态以正文最后一条追加式状态事件为准，frontmatter 的 `status` 只作创建时缓存。允许转换和 active 门禁见 [execution-case-notes.md](execution-case-notes.md)。

## 冲突流程

1. 检索同一实体、标题、别名或决策的已有笔记。
2. 如果新信息与 `active` 笔记冲突，比较来源优先级：
   - 用户当前明确指令
   - 当前项目文件或本地事实源
   - 近期已验证笔记
   - 更旧笔记
   - 外部摘要
3. 如果新来源明显更强，更新笔记，设置 `status: active`，并在带日期的证据/更新章节中记录被替换说法。
4. 如果没有来源明显胜出，设置 `status: conflicted`，保留两种说法，并写清未解决问题。
5. 如果旧笔记不再活跃但仍有历史价值，标记为 `deprecated` 或 `retired`，不要删除。

## 过期触发条件

出现以下情况时，把笔记标记为 `stale`：

- 它描述的软件、包版本、法律、价格、日程或 API 行为可能已经变化；
- 它来自未经验证的会话说法；
- 后续检索发现更新但尚不完整的证据；
- 用户表示该笔记可能已经过时。

## 脱敏规则

- 用 `[REDACTED]` 替换敏感值。
- 不要把完整凭据写入别名、标签、文件名或 backlinks。
- 如果敏感项对上下文确有必要，只记录用途和用户可从哪里找到它，不记录值本身。
