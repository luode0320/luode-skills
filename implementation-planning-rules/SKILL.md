# 实施规划规则

只在来源对象（需求或 Bug）已经基本讲清、前置验收标准已明确、当前准备进入编码，但实现路径仍然复杂，需要先把执行步骤拆成清晰实施方案时使用这个 skill。
如果当前问题还是需求信息不全、边界不清、验收未定或任务还没拆出当前优先闭环，请先转交相邻需求域 skill；但若用户本轮明确在索要计划，也必须先给出“受限计划 / 阻断计划”收口，不能表现成计划规则没有触发。

## Skill 作用与适用场景

- 若当前上下文处于 `Plan Mode`，本 skill 负责接住第一层计划路由，不允许把问题直接绕过去交给需求、Bug、测试或交付域；若当前问题仍存在需求前置缺口，应把它们显式 handoff 给相邻计划前置 skill，而不是直接跳到实施步骤。
- 若当前上下文处于 `Plan Mode`，且运行环境要求最终计划放进 `<proposed_plan>` 或其他专用计划包裹中，包裹层不改变本 skill 的正文格式要求；外层是渲染协议，内层仍必须按本 skill 模板组织。
- 把已经确认的来源对象（需求或 Bug）或当前优先子项，转成编码前可执行的实施规划。
- 当新项目启动或项目初期同时存在多份需求、验收标准、实施总览或实施周期时，先建立“需求与实施计划全量顺序实施方案”，统一排列需求 -> 验收标准 -> 实施总览 -> 实施周期 -> 周期内最小任务的全量执行顺序。
- “需求与实施计划全量顺序实施方案”是跨需求 / 跨来源对象的总调度表；它不替代单个来源对象的实施总览、实施周期和最小任务文档，只负责把这些文档串成一个可执行的项目级顺序。
- 实施规划阶段默认采用只读计划模式：禁止写代码、禁止直接改实现、禁止边计划边试做；只允许读仓库、定依赖、列风险、拆任务、写实施文档。
- 明确本轮要改哪些目录、文件、模块和职责边界，避免一边写代码一边临时漂移。
- 若计划包含新增文件，强制使用目录树（`text` 代码块）列出新增文件，并在树节点后标注用途说明。
- 若计划包含数据库新表或字段变更，强制给出可执行 SQL（`CREATE TABLE` / `ALTER TABLE`），禁止只写“需要改表”。
- 给出实施周期拆分、最小任务清单、任务顺序、依赖关系、最小闭环、真实测试安排（建议独立成章）和每一步的验证点。
- 实施计划正文开头必须先写“当前计划最终方案的简要说明”，用 1-3 句先交代推荐方案、主落点和为什么这么做；随后再写“agent 对当前问题的理解”，至少包含：agent 理解的问题 / 目标、本轮计划范围、不在范围、当前优先闭环、关键假设或待确认点；不得直接跳到阶段、任务或测试安排。
- 实施周期是项目第一期、第二期、第三期等大进度单位，也是实施顺序边界；实施文档必须显式写出周期序号、期次定位、周期目标、阶段划分、阶段产出、周期内最小任务清单、进入条件和收口条件。周期决定先做哪一期，最小任务决定当前周期内怎么执行。
- 执行顺序必须先按实施周期推进：当前周期未完成全部最小任务闭环前，不得跳到后续周期；同一周期内必须按最小任务逐个完成“实现 -> 真实测试 -> 审查 -> 验收”。禁止先连续实现多个最小任务后统一测试、统一审查或统一验收。
- 实施计划必须先拆到最小可独立闭环任务粒度；每个最小任务都要单独写清产出、验证点、真实测试要求、任务完成条件、任务停止 / 结束条件、阻断条件和与前后任务的依赖关系。
- 最小任务默认优先按“依赖图 + 垂直切片”拆分，不按前端 / 后端 / 数据库等水平分层批量拆分；每个任务都应尽量形成端到端最小闭环。
- 单任务尽量单次专注完成，且默认应控制在约 5 个文件以内；若明显超过该粒度，优先继续拆小任务，而不是放宽任务边界。
- 实施计划必须显式写出任务完成条件、任务停止 / 结束条件和最大推进边界；长文本 / agent 执行模式下，缺少这些边界就不得进入实现或连续执行。
- 只要计划涉及生成代码、修改代码、重构代码或其他会改变可执行行为的实现变更，就必须显式写出真实测试安排（独立章节），至少包含测试入口、依赖环境、样本 / 数据来源和通过标准，并默认把“真实测试通过”写入完成定义。
- `build`、`lint`、静态检查、代码审查、人工阅读或口头说明都不算真实测试；若当前无法执行真实测试，必须在计划中记为阻断项或受限交接项，不得写成“后面再测”。
- 只有纯文档、纯注释、纯排版、纯静态资源改名 / 搬运、不会影响运行结果的配置说明整理等不改变可执行行为的场景，才允许显式标记“无需真实测试”并写明理由。
- 开发周期应尽量独立可切分；如果当前任务可以拆成多个更小的子任务周期，就不要把多个功能实现硬塞进同一个周期。
- 每个阶段默认只做一件事，只承载一个清晰目标；禁止把多个并列能力揉进同一步骤或同一阶段。
- 实施步骤必须显式归属到具体开发阶段，保证“阶段 -> 步骤 -> 验证点”链路清楚。
- 每个最小任务都必须单独闭环：实现完成后，必须先完成该任务自己的真实测试、审查、验收，再允许推进下一个最小任务；禁止先连续实现多个任务再统一检查。
- 对复杂执行路径优先补图（流程图、时序图或步骤泳道表），避免仅靠长段文字描述实施步骤。
- 在技术路径存在多种合理实现时，先收敛 2-3 个方案，再锁定推荐路线。
- 保证计划中的每一步都可执行、可验证、可交接，不写空泛占位词。
- 将实施方案单独沉淀到 `artifact-storage-rules` 约定的实施总览/实施周期文档位置，作为后续编码、审查、测试和最终验收的阶段性依据。
- 当用户要求“补充需求、实施文档”或“完善实施文档”时，如果当前来源对象已经存在实施周期拆分，必须同步更新受影响的实施周期文档；不得只修改实施总览。若当前变更已经形成新的独立闭环但尚无对应周期文档，必须补建或补齐该周期文档。
- 降低“需求虽然清楚，但真正动代码时落点混乱、顺序错乱、验证遗漏”的风险。
- 对于“原需求已实现、当前只是原需求上的新增改动”的场景，必须在同一份实施总览下新增本次改动对应的实施周期，或在必要时补建新的实施总览；旧实施记录只能作为历史依据保留，不得直接伪装成当前闭环。
## 自动触发信号

- 当前上下文处于 `Plan Mode`。
- 来源对象、边界和验收标准已基本稳定，但任务仍涉及多个文件、多个模块、多个接口、多个步骤或多个验证点。
- `requirement-splitting-rules` 已拆出当前优先子项，但编码前还需要继续细化实施顺序和文件落点。
- 新项目开始、项目初期或大项目规划阶段已经出现多份需求文档、多份实施总览、多个实施周期，执行前需要先确定总执行顺序。
- 用户要求“实施顺序总表”“全量顺序实施方案”“需求与实施计划全量顺序实施方案”“新项目实施总顺序”“项目第一期/第二期整体排序”等跨需求执行索引。
- 用户明确要求先写开发计划、实施方案、落地步骤、任务清单或编码前执行方案。
- 用户明确提出计划型问题，例如“这件事怎么做”“这个需求怎么落地”“先给我一个计划”“先出方案和步骤”“这个怎么改最合适”。
- 用户明确表示“按建议执行 / 按方案执行 / 就按刚才那个建议做 / 按你说的做”，但当前只有聊天建议或方案结论，还没有正式实施计划。
- 用户要求“补充实施文档 / 补充需求、实施文档 / 完善需求与实施文档”，且目标是已有实施总览或实施周期文档。
- 当前实现路径存在多个合理技术方案，需要先比较后收口。
- 不先写计划就容易造成改动范围失控、顺序颠倒、验证遗漏或多人协作断层。
- 当前环境提供了专用计划输出协议（如 `<proposed_plan>`），但仓库仍要求计划正文遵守项目内 skill 模板，而不是退化成通用简版计划。
