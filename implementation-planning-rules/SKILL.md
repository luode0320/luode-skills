# 实施规划规则

只在需求已经基本讲清、前置验收标准已明确、当前准备进入编码，但实现路径仍然复杂，需要先把执行步骤拆成清晰实施方案时使用这个 skill。
如果当前问题还是需求信息不全、边界不清、验收未定或任务还没拆出当前优先闭环，请先转交相邻需求域 skill。

## Skill 作用与适用场景

- 把已经确认的需求或当前优先子项，转成编码前可执行的实施规划。
- 明确本轮要改哪些目录、文件、模块和职责边界，避免一边写代码一边临时漂移。
- 若计划包含新增文件，强制使用目录树（`text` 代码块）列出新增文件，并在树节点后标注用途说明。
- 若计划包含数据库新表或字段变更，强制给出可执行 SQL（`CREATE TABLE` / `ALTER TABLE`），禁止只写“需要改表”。
- 给出实施周期拆分、任务顺序、依赖关系、最小闭环和每一步的验证点。
- 实施周期是实施文档必填项，必须显式写出阶段划分、阶段目标、阶段产出和阶段验证门槛。
- 开发周期应尽量独立可切分；如果当前任务可以拆成多个更小的子任务周期，就不要把多个功能实现硬塞进同一个周期。
- 每个阶段默认只做一件事，只承载一个清晰目标；禁止把多个并列能力揉进同一步骤或同一阶段。
- 实施步骤必须显式归属到具体开发阶段，保证“阶段 -> 步骤 -> 验证点”链路清楚。
- 对复杂执行路径优先补图（流程图、时序图或步骤泳道表），避免仅靠长段文字描述实施步骤。
- 在技术路径存在多种合理实现时，先收敛 2-3 个方案，再锁定推荐路线。
- 保证计划中的每一步都可执行、可验证、可交接，不写空泛占位词。
- 将实施方案单独沉淀到 `artifact-storage-rules` 约定的实施总览/实施周期文档位置，作为后续编码、审查、测试和最终验收的阶段性依据。
- 降低“需求虽然清楚，但真正动代码时落点混乱、顺序错乱、验证遗漏”的风险。
- 对于“原需求已实现、当前只是原需求上的新增改动”的场景，必须在同一份实施总览下新增本次改动对应的实施周期，或在必要时补建新的实施总览；旧实施记录只能作为历史依据保留，不得直接伪装成当前闭环。

## 自动触发信号

- 需求、边界和验收标准已基本稳定，但任务仍涉及多个文件、多个模块、多个接口、多个步骤或多个验证点。
- `requirement-splitting-rules` 已拆出当前优先子项，但编码前还需要继续细化实施顺序和文件落点。
- 用户明确要求先写开发计划、实施方案、落地步骤、任务清单或编码前执行方案。
- 当前实现路径存在多个合理技术方案，需要先比较后收口。
- 不先写计划就容易造成改动范围失控、顺序颠倒、验证遗漏或多人协作断层。

## 进入后先做什么

1. 先确认需求文档、边界结论和验收标准已经基本稳定。
2. 确认当前要规划的是整个需求，还是拆分后当前优先闭环子项。
3. 先看仓库现状，找到相关目录、模块、已有实现和可能的复用点。
4. 列出本轮必须落地的文件 / 模块落点，不先展开代码细节。
5. 先拆开发周期与阶段边界，再拆任务顺序、依赖关系、最小闭环和验证点。
6. 如果实现路径存在多种合理方向，先做 2-3 个方案比选并给出推荐。
7. 找到当前需求对应的实施总览文档；如果还没有，就按 `artifact-storage-rules` 的命名模板初始化独立实施总览文档，并判断是否需要同步拆出实施周期文档。

## 默认执行流程

1. 默认先读 `references/plan-entry-checklist.md`，确认进入实施规划阶段前的前置条件是否满足。
2. 再读 `../artifact-storage-rules/references/path-map.yaml` 与 `../artifact-storage-rules/references/update-policy.md`，确认实施总览/实施周期文档路径、命名模板和同文档更新策略。
3. 再读 `references/plan-structure-template.md`，按统一结构生成或更新实施总览文档，并在任务较大时拆出实施周期文档。
4. 如果任务粒度、依赖顺序或最小闭环不好判断，再读 `references/task-granularity-and-order.md`。
5. 如需判断与相邻 skill 的边界、或对照正反例，再读 `references/plan-boundaries-and-examples.md`。
6. 输出实施范围、实施周期、阶段划分、文件 / 模块落点、任务顺序、验证步骤、风险与阻断项，并更新到当前需求对应的实施总览/实施周期文档。
7. 若涉及新增文件，实施文档中必须补齐“新增文件清单目录树（`text` 代码块 + 用途注释）”；普通条目描述视为计划未收口。
8. 若涉及数据库新表或字段变更，实施文档中必须补齐 SQL（`CREATE TABLE` / `ALTER TABLE`）；缺 SQL 视为计划未收口。
9. 如果还没有实施总览文档，则按 `artifact-storage-rules` 约定的命名规则创建；如同一天存在多个近似标题，在不改变中文语义前提下补充更具体描述以避免重名。
10. 如果当前任务体量较大、闭环较多，必须先维护一份实施总览，再按 `实施周期NN_中文主题` 拆出周期文档；每个周期只承载一个清晰目标。
11. 如果当前需求属于“已实现需求上的新增改动”，必须在现有需求主文档已更新后，新建或增补本次改动对应的实施周期文档；不要沿用已完成的旧实施周期正文作为本次执行计划。
12. 若当前计划仍把多个并列功能塞进同一实施周期，或单个阶段承载多个目标，必须继续拆成更小的独立子任务周期；未拆分视为计划未收口。
13. 计划写完后，必须再读 `references/plan-review-checklist.md` 做一次自审，重点检查实施周期、阶段职责单一性、占位词、可执行性和验证闭环。
14. 对复杂任务补齐图形化步骤表达（至少一种：流程图/时序图/泳道表），并校验图文一致。
15. 自审通过后，请用户先 review 当前实施文档；如用户提出调整，继续更新同一份实施总览或对应周期文档，不新建无关平行记录。
16. 计划确认后，仍必须等待用户明确说“开始实施 / 开始执行 / 按实施计划执行”才能进入编码基线域和相应代码位点域；仅有“计划写完了”“需求确认了”不构成开工授权。
17. 若用户已明确“开始实施 / 开始执行”，则按 `autonomous-execution-rules` 默认连续推进“当前实施周期编码实现 -> 当前周期编码审查 -> 当前周期测试策略与验证 -> 当前周期审核/验收收口 -> 下一实施周期”；除阻断级节点外不在中间重复征求下一步确认。

## 权责边界与不负责事项

- 只负责把已确认需求转成实施方案，不代替 `requirement-splitting-rules` 做顶层需求拆分。
- 不代替 `acceptance-criteria-rules` 编写验收标准；验收标准应已基本存在。
- 不代替实际编码，不在本 skill 内直接展开实现代码。
- 不代替测试域制定完整测试策略，但必须把当前计划对应的最小验证点写清。
- 不接受 `TBD`、`TODO`、`后续再补`、`实现时再看` 这类空泛占位计划。
- 不接受“一个开发周期覆盖很多功能点、很多阶段目标”的笼统计划；能拆小周期就必须拆小周期。
- 不接受“一个阶段同时做多件事”的混合步骤设计；阶段目标必须单一。
- 不为同一个需求重复新建多份无关实施总览；若总览文档已存在，应优先更新原文档并按需补充实施周期。
- 不得把“实施计划已经完成”误判成“已经得到开始实施授权”；开工授权必须来自用户在当前任务中的明确表态。
- 如果技术分歧的根源其实是业务目标不清、范围有争议或验收没定，应回流相邻需求域或验收域 skill，而不是假装用实施规划掩盖前置问题。
- 如果技术分歧出现在“已实现需求上的新增改动”，应优先把本次改动单独落到新的实施周期文档中，而不是沿用旧周期继续补丁式覆盖。

## 需要暂停并确认的条件

- 需求、边界或验收标准仍在明显变化，当前计划随时会失效。
- 当前任务还没拆出可独立推进的优先闭环，却试图一次规划全部大需求。
- 当前任务明明可以拆成多个独立子任务周期，却仍被写成一个大周期。
- 实现路径存在重大分歧，但尚未形成 2-3 个可比方案和推荐结论。
- 无法确定主要文件落点、模块职责或依赖顺序，说明仍需先回看仓库现状或回流需求域。
- 计划中仍存在明显占位词、跳步描述、阶段目标不单一或无法验证的任务项。
- 用户尚未明确“开始实施 / 开始执行”，但流程准备从实施文档切换到正式编码。

## 执行通过 / 驳回标准

- 通过：能够形成一份可直接指导后续编码的实施总览与必要的实施周期文档，至少说明本轮范围、实施周期、阶段划分、文件 / 模块落点、任务顺序、依赖关系、最小闭环、验证点、风险和阻断项，并完成自审与用户确认门槛；且每个阶段只做一件事，并明确“是否已获得开始实施授权”。
- 驳回：只是把需求原文换一种说法重写，没有落实到实施周期、文件 / 模块和执行步骤；或计划里仍充满占位词、顺序不清、阶段混杂、无法验证；或本应先回流需求拆分 / 边界 / 验收，却被硬塞成实施规划；或未等用户明确开工就直接切进编码。

## 执行结果归档要求

- 将实施范围、实施周期、阶段划分、文件落点、任务顺序、验证点、风险和阻断项记录到 `artifact-storage-rules` 约定的实施总览/实施周期文档中，不与需求文档、验收标准文档或最终验收文档混写。
- 文档文件名、根目录和同文档更新策略统一遵循 `../artifact-storage-rules/references/path-map.yaml` 与 `../artifact-storage-rules/references/update-policy.md`。
- 实施总览文件名必须复用对应需求文档同前缀并在末尾追加“实施总览”，例如：`2026-04-16_141518_多业务复用Swap适配控制需求实施总览.md`。
- 实施周期文件名必须复用对应需求文档同前缀并在末尾追加“实施周期NN_中文主题”，例如：`2026-04-16_141518_多业务复用Swap适配控制需求实施周期01_接口与配置落点.md`。
- 同一个需求的实施总览应持续更新同一份总览文档；若拆周期，也应持续更新对应周期文档，而不是重复新建多份平行记录。
- 归档内容至少包含对应需求来源、当前优先闭环、实施总览、阶段清单、目录 / 文件落点、任务清单、推荐顺序、每步验证点、自审结论、用户确认状态和“是否已获得开始实施授权”状态。
- 对高复杂度计划，归档建议补充图形化执行路径（流程图/时序图/泳道表）以降低沟通成本。
- 若涉及新增文件，归档必须包含“新增文件清单目录树（`text` 代码块）”，并在树节点后标注用途；普通条目描述不合格。
- 若涉及数据库新表或字段变更，归档必须包含对应 SQL（`CREATE TABLE` / `ALTER TABLE`），不能只写“会改数据库”。
- 如果后续需求变更、边界变化或验收标准调整推翻原方案，应在同一份实施总览或对应周期文档中更新新旧差异和重新确认状态。
- 如果后续发生的是已实现需求上的新增改动，应创建或增补一份新的实施周期文档记录本次改动的独立闭环，旧周期保留为历史记录。
- 如果一个需求被拆成多个独立子任务周期，归档中必须明确每个周期只承载一个清晰目标，并标明周期之间的衔接关系。

- 默认先读 `references/plan-entry-checklist.md`。
- 在决定实施总览/实施周期文档的根目录、命名模板和同文档更新策略时，先读 `../artifact-storage-rules/references/path-map.yaml` 与 `../artifact-storage-rules/references/update-policy.md`。
- 只有在 输出或更新结构化实施文档 时，再读 `references/plan-structure-template.md`。
- 只有在 判断任务颗粒度、依赖顺序和最小闭环 时，再读 `references/task-granularity-and-order.md`。
- 只有在 做计划自审 时，再读 `references/plan-review-checklist.md`。
- 只有在 判断与相邻 skill 的边界或对照正反例 时，再读 `references/plan-boundaries-and-examples.md`。
