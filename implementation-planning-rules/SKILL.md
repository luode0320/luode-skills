# 实施规划规则

只在来源对象（需求或 Bug）已经基本讲清、前置验收标准已明确、当前准备进入编码，但实现路径仍然复杂，需要先把执行步骤拆成清晰实施方案时使用这个 skill。
如果当前问题还是需求信息不全、边界不清、验收未定或任务还没拆出当前优先闭环，请先转交相邻需求域 skill；但若用户本轮明确在索要计划，也必须先给出“受限计划 / 阻断计划”收口，不能表现成计划规则没有触发。

## Skill 作用与适用场景

- 若当前上下文处于 `Plan Mode`，本 skill 负责接住第一层计划路由，不允许把问题直接绕过去交给需求、Bug、测试或交付域；若当前问题仍存在需求前置缺口，应把它们显式 handoff 给相邻计划前置 skill，而不是直接跳到实施步骤。
- 若当前上下文处于 `Plan Mode`，且运行环境要求最终计划放进 `<proposed_plan>` 或其他专用计划包裹中，包裹层不改变本 skill 的正文格式要求；外层是渲染协议，内层仍必须按本 skill 模板组织。
- 把已经确认的来源对象（需求或 Bug）或当前优先子项，转成编码前可执行的实施规划。
- 实施规划阶段默认采用只读计划模式：禁止写代码、禁止直接改实现、禁止边计划边试做；只允许读仓库、定依赖、列风险、拆任务、写实施文档。
- 明确本轮要改哪些目录、文件、模块和职责边界，避免一边写代码一边临时漂移。
- 若计划包含新增文件，强制使用目录树（`text` 代码块）列出新增文件，并在树节点后标注用途说明。
- 若计划包含数据库新表或字段变更，强制给出可执行 SQL（`CREATE TABLE` / `ALTER TABLE`），禁止只写“需要改表”。
- 给出实施周期拆分、最小任务清单、任务顺序、依赖关系、最小闭环、真实测试安排和每一步的验证点。
- 实施计划正文开头必须先写“agent 对当前问题的理解”，至少包含：agent 理解的问题 / 目标、本轮计划范围、不在范围、当前优先闭环、关键假设或待确认点；不得直接跳到阶段、任务或测试安排。
- 实施周期是实施文档必填项，必须显式写出阶段划分、阶段目标、阶段产出和阶段验证门槛；但实施周期只是管理容器，不是默认执行粒度。
- 实施计划必须先拆到最小可独立闭环任务粒度；每个最小任务都要单独写清产出、验证点、真实测试要求、任务完成条件、任务停止 / 结束条件、阻断条件和与前后任务的依赖关系。
- 最小任务默认优先按“依赖图 + 垂直切片”拆分，不按前端 / 后端 / 数据库等水平分层批量拆分；每个任务都应尽量形成端到端最小闭环。
- 单任务尽量单次专注完成，且默认应控制在约 5 个文件以内；若明显超过该粒度，优先继续拆小任务，而不是放宽任务边界。
- 实施计划必须显式写出任务完成条件、任务停止 / 结束条件和最大推进边界；长文本 / agent 执行模式下，缺少这些边界就不得进入实现或连续执行。
- 只要计划涉及生成代码、修改代码、重构代码或其他会改变可执行行为的实现变更，就必须显式写出真实测试安排，至少包含测试入口、依赖环境、样本 / 数据来源和通过标准，并默认把“真实测试通过”写入完成定义。
- `build`、`lint`、静态检查、代码审查、人工阅读或口头说明都不算真实测试；若当前无法执行真实测试，必须在计划中记为阻断项或受限交接项，不得写成“后面再测”。
- 只有纯文档、纯注释、纯排版、纯静态资源改名 / 搬运、不会影响运行结果的配置说明整理等不改变可执行行为的场景，才允许显式标记“无需真实测试”并写明理由。
- 开发周期应尽量独立可切分；如果当前任务可以拆成多个更小的子任务周期，就不要把多个功能实现硬塞进同一个周期。
- 每个阶段默认只做一件事，只承载一个清晰目标；禁止把多个并列能力揉进同一步骤或同一阶段。
- 实施步骤必须显式归属到具体开发阶段，保证“阶段 -> 步骤 -> 验证点”链路清楚。
- 每个最小任务都必须单独闭环：实现完成后，必须先完成该任务自己的真实测试、审查、验收，再允许推进下一个最小任务；禁止先连续实现多个任务再统一检查。
- 对复杂执行路径优先补图（流程图、时序图或步骤泳道表），避免仅靠长段文字描述实施步骤。
- 在技术路径存在多种合理实现时，先收敛 2-3 个方案，再锁定推荐路线。
- 保证计划中的每一步都可执行、可验证、可交接，不写空泛占位词。
- 将实施方案单独沉淀到 `artifact-storage-rules` 约定的实施总览/实施周期文档位置，作为后续编码、审查、测试和最终验收的阶段性依据。
- 降低“需求虽然清楚，但真正动代码时落点混乱、顺序错乱、验证遗漏”的风险。
- 对于“原需求已实现、当前只是原需求上的新增改动”的场景，必须在同一份实施总览下新增本次改动对应的实施周期，或在必要时补建新的实施总览；旧实施记录只能作为历史依据保留，不得直接伪装成当前闭环。

## 自动触发信号

- 当前上下文处于 `Plan Mode`。
- 来源对象、边界和验收标准已基本稳定，但任务仍涉及多个文件、多个模块、多个接口、多个步骤或多个验证点。
- `requirement-splitting-rules` 已拆出当前优先子项，但编码前还需要继续细化实施顺序和文件落点。
- 用户明确要求先写开发计划、实施方案、落地步骤、任务清单或编码前执行方案。
- 用户明确提出计划型问题，例如“这件事怎么做”“这个需求怎么落地”“先给我一个计划”“先出方案和步骤”“这个怎么改最合适”。
- 用户明确表示“按建议执行 / 按方案执行 / 就按刚才那个建议做 / 按你说的做”，但当前只有聊天建议或方案结论，还没有正式实施计划。
- 当前实现路径存在多个合理技术方案，需要先比较后收口。
- 不先写计划就容易造成改动范围失控、顺序颠倒、验证遗漏或多人协作断层。
- 当前环境提供了专用计划输出协议（如 `<proposed_plan>`），但仓库仍要求计划正文遵守项目内 skill 模板，而不是退化成通用简版计划。

## 进入后先做什么

1. 先确认来源对象文档、边界结论和验收标准已经基本稳定。
2. 先把“agent 对当前问题的理解”写清：当前理解的问题 / 目标是什么，本轮准备覆盖什么，不覆盖什么，当前优先闭环是什么，哪些是假设或仍待确认。
2.1 若本轮是“用户先问计划怎么做”，先判断当前能否直接进入正式实施规划；若不能，也必须先整理一版“受限计划 / 阻断计划”，明确前置缺口、建议流转和后续计划落点。
2.2 若当前环境要求用 `<proposed_plan>` 之类的专用包裹输出计划，先把本 skill 要求的计划正文结构整理完整，再在最终输出时做外层包裹；不得先按外部简版格式压缩，导致正文缺字段。
3. 确认当前要规划的是整个需求，还是拆分后当前优先闭环子项。
4. 先看仓库现状，找到相关目录、模块、已有实现和可能的复用点。
5. 列出本轮必须落地的文件 / 模块落点，不先展开代码细节。
6. 先拆开发周期与阶段边界，再拆到最小任务顺序、依赖关系、最小闭环、真实测试安排和验证点。
7. 先画依赖关系，再按垂直切片识别最小任务，避免直接按技术层分堆。
8. 对每个最小任务估算文件落点；若单任务预计会触达超过约 5 个文件，优先继续拆小。
9. 如果实现路径存在多种合理方向，先做 2-3 个方案比选并给出推荐。
10. 找到当前来源对象对应的实施总览文档；如果还没有，就按 `artifact-storage-rules` 的命名模板初始化独立实施总览文档，并判断是否需要同步拆出实施周期文档。
11. 若当前开工意图来自“采纳建议/采纳方案”，先把建议内容改写为正式计划结构，再判断是否允许进入实现。

## 默认执行流程

1. 默认先读 `references/plan-entry-checklist.md`，确认进入实施规划阶段前的前置条件是否满足。
2. 再读 `../artifact-storage-rules/references/path-map.yaml` 与 `../artifact-storage-rules/references/update-policy.md`，确认实施总览/实施周期文档路径、命名模板和同文档更新策略。
3. 再读 `references/plan-structure-template.md`，按统一结构生成或更新实施总览文档，并在任务较大时拆出实施周期文档。
4. 如果任务粒度、依赖顺序或最小闭环不好判断，再读 `references/task-granularity-and-order.md`。
5. 如需判断与相邻 skill 的边界、或对照正反例，再读 `references/plan-boundaries-and-examples.md`。
6. 输出问题理解、本轮范围、非范围、当前优先闭环、关键假设 / 待确认点、实施周期、阶段划分、最小任务清单、文件 / 模块落点、任务顺序、真实测试安排、验证步骤、任务完成条件、任务停止 / 结束条件、最大推进边界、风险与阻断项，并更新到当前来源对象对应的实施总览/实施周期文档。
6.0 若本轮是用户显式索要计划，但前置条件仍未齐备，则至少输出“问题理解 + 当前不可直接形成正式实施计划的原因 + 前置缺口 + 建议流转顺序 + 计划成文后的预期结构”；不得直接结束为“先去补需求”，也不得表现成计划规则没有触发。
6.1 在任务清单中显式标注每个最小任务的依赖关系、垂直切片目标、预计触达文件数，以及是否超过“约 5 个文件”警戒线。
6.2 若某任务偏向水平分层拆法（例如单独一批前端、一批后端、一批数据库），必须继续改写为按端到端闭环组织的垂直切片任务，或明确记录为何当前场景无法采用垂直切片。
6.3 若最终回复使用 `<proposed_plan>` 之类专用计划包裹，包裹内正文至少仍需完整承载本 skill 要求的问题理解、本轮范围、非范围、当前优先闭环、关键假设 / 待确认点、实施周期、阶段划分、最小任务清单、文件 / 模块落点、真实测试安排、任务完成条件、任务停止 / 结束条件和最大推进边界；允许合并同类小节，不允许丢字段。
7. 若涉及新增文件，实施文档中必须补齐“新增文件清单目录树（`text` 代码块 + 用途注释）”；普通条目描述视为计划未收口。
8. 若涉及数据库新表或字段变更，实施文档中必须补齐 SQL（`CREATE TABLE` / `ALTER TABLE`）；缺 SQL 视为计划未收口。
9. 如果还没有实施总览文档，则按 `artifact-storage-rules` 约定的命名规则创建；如同一天存在多个近似标题，在不改变中文语义前提下补充更具体描述以避免重名。
10. 如果当前任务体量较大、闭环较多，必须先维护一份实施总览，再按 `来源对象标识_实施周期NN_中文主题` 拆出周期文档；每个周期只承载一个清晰目标，但周期内仍必须继续拆到最小任务清单。
11. 如果当前需求属于“已实现需求上的新增改动”，必须在现有需求主文档已更新后，新建或增补本次改动对应的实施周期文档；不要沿用已完成的旧实施周期正文作为本次执行计划。
12. 若当前计划仍把多个并列功能塞进同一实施周期，或单个阶段承载多个目标，必须继续拆到更小的最小任务粒度；必要时再补拆新的实施周期文档。只要仍存在“一个任务项里混着多个实现目标 / 多个验证目标 / 多个验收目标”，都视为计划未收口。
12.1 若当前最小任务预计触达文件数明显超过约 5 个，或无法单次专注完成，也视为计划未收口，应继续拆分或重组。
13. 计划写完后，必须再读 `references/plan-review-checklist.md` 做一次自审，重点检查实施周期、阶段职责单一性、真实测试安排、占位词、可执行性和验证闭环。
14. 对复杂任务补齐图形化步骤表达（至少一种：流程图/时序图/泳道表），并校验图文一致。
15. 自审通过后，请用户先 review 当前实施文档；如用户提出调整，继续更新同一份实施总览或对应周期文档，不新建无关平行记录。
16. 计划确认后，仍必须等待用户明确说“开始实施 / 开始执行 / 按实施计划执行”才能进入编码基线域和相应代码位点域；仅有“计划写完了”“需求确认了”不构成开工授权。
16.1 若用户直接说“开始实施 / 开始实现 / 开始执行 / 直接做 / 继续做完 / 按文档实现”，但当前没有可执行计划、任务完成条件、任务停止 / 结束条件和最大推进边界，必须先补本轮受限计划并停在计划收口处，不得直接编码。
16.2 若用户直接说“按建议执行 / 按方案执行 / 就按你刚才说的做 / 按你说的做”，但当前只有建议、方案比选或修复思路，没有正式实施计划，也必须先补齐实施计划并停在计划收口处，不得直接编码。
16.3 受限计划不得作为实施授权；用户即使明确采纳，agent 也只能先补齐缺失前置条件并将其升级为正式执行计划，未升级前禁止进入编码、改码、重构、测试实施或其他执行动作。
17. 若用户已明确“开始实施 / 开始执行”，则按 `autonomous-execution-rules` 默认连续推进“当前实施周期内最小任务A实现 -> 最小任务A真实测试 -> 最小任务A审查 -> 最小任务A验收 -> 最小任务B…… -> 当前实施周期收口 -> 下一实施周期”；每个最小任务都必须先完成自己的真实测试、审查、验收闭环，才允许进入下一个任务；除阻断级节点外不在中间重复征求下一步确认。

## 权责边界与不负责事项

- 只负责把已确认来源对象转成实施方案，不代替 `requirement-splitting-rules` 做顶层需求拆分，也不代替 Bug 域做根因定位。
- 不代替 `acceptance-criteria-rules` 编写验收标准；验收标准应已基本存在。
- 不代替实际编码，不在本 skill 内直接展开实现代码。
- 不代替测试域制定完整测试策略，但必须把当前计划对应的真实测试安排、最小验证点和免测理由写清。
- 不允许在用户明确索要计划时静默回退成“这还不能计划”；即使只能给受限计划，也要把计划前置缺口和受限边界写出来。
- 不允许因为运行环境提供了 `<proposed_plan>` 或其他专用计划包裹，就把正文退化成通用摘要、产品 PRD 摘要或只剩 3-5 条 bullet 的泛化计划；外层协议不能覆盖项目内计划格式。
- 不允许在本 skill 内边写计划边改代码，或通过“试做一点”来代替计划判断。
- 不接受 `TBD`、`TODO`、`后续再补`、`实现时再看` 这类空泛占位计划。
- 不接受“一个开发周期覆盖很多功能点、很多阶段目标”的笼统计划；能拆小周期就必须拆小周期，能继续拆成更小最小任务就不得停在粗粒度步骤。
- 不接受“一个阶段同时做多件事”的混合步骤设计；阶段目标必须单一。
- 不接受按前端 / 后端 / 数据库水平分层堆计划、却缺少端到端最小闭环的任务拆法；若无充分理由，默认视为计划未收口。
- 不接受明显超过约 5 个文件、且无法单次专注完成的最小任务定义；能拆就必须继续拆。
- 不为同一个来源对象重复新建多份无关实施总览；若总览文档已存在，应优先更新原文档并按需补充实施周期。
- 不得把“实施计划已经完成”误判成“已经得到开始实施授权”；开工授权必须来自用户在当前任务中的明确表态。
- 不得把“用户采纳了建议/方案”误判成“已经有正式执行计划”；建议只有转写为计划后才能进入实现。
- 不得把“受限计划已输出”误判成“已经拿到实施授权”；受限计划只有在前置条件补齐并升级为正式执行计划后，才允许进入执行。
- 如果技术分歧的根源其实是业务目标不清、范围有争议或验收没定，应回流相邻需求域或验收域 skill，而不是假装用实施规划掩盖前置问题。
- 如果技术分歧出现在“已实现需求上的新增改动”，应优先把本次改动单独落到新的实施周期文档中，而不是沿用旧周期继续补丁式覆盖。

## 需要暂停并确认的条件

- 来源对象、边界或验收标准仍在明显变化，当前计划随时会失效。
- 当前任务还没拆出可独立推进的优先闭环，却试图一次规划全部大需求。
- 当前任务明明可以拆成多个独立子任务周期，却仍被写成一个大周期。
- 当前任务虽然拆成了最小任务，但仍明显按水平分层组织，缺少端到端垂直切片闭环。
- 当前最小任务预计触达文件数明显超过约 5 个，或无法单次专注完成。
- 计划涉及代码变更，但没有为对应最小任务写出真实测试入口、依赖环境、样本 / 数据来源或通过标准。
- 计划把 `build`、`lint`、静态检查、代码审查或人工阅读误写成真实测试。
- 实现路径存在重大分歧，但尚未形成 2-3 个可比方案和推荐结论。
- 无法确定主要文件落点、模块职责或依赖顺序，说明仍需先回看仓库现状或回流需求域。
- 计划中仍存在明显占位词、跳步描述、阶段目标不单一或无法验证的任务项。
- 用户尚未明确“开始实施 / 开始执行”，但流程准备从实施文档切换到正式编码。
- 用户虽已明确“按建议执行 / 按方案执行”，但建议尚未被收口成正式实施计划。
- 用户明确索要计划，但当前仍没有来源对象文档、前置验收口径或可参考现状；此时必须停在受限计划 / 阻断计划，而不是假装已经拿到正式实施条件。
- 当前环境要求专用计划包裹，但正文仍未满足本 skill 模板必填项；此时应继续补正文结构，而不是把包裹当作已收口。
- 用户即使明确说“就按这个受限计划实施”，但受限计划仍未升级为正式执行计划；此时必须继续补前置条件，不得进入编码、改码、重构、测试实施或其他执行动作。

## 执行通过 / 驳回标准

- 通过：能够形成一份可直接指导后续编码的实施总览与必要的实施周期文档，或在前置条件未齐时形成一份结构完整的受限计划 / 阻断计划；至少说明 agent 当前理解的问题 / 目标、本轮范围、非范围、当前优先闭环、关键假设或待确认点、实施周期、阶段划分、最小任务清单、文件 / 模块落点、代码生成 / 修改 / 重构类任务的“代码落点目录树”、任务顺序、依赖关系、最小闭环、真实测试安排、任务完成条件、任务停止 / 结束条件、最大推进边界、风险和阻断项，并完成自审与用户确认门槛；若当前只能给受限计划，也必须明确前置缺口、建议流转顺序和何时升级为正式实施规划。且实施规划阶段保持只读、每个阶段只做一件事、每个最小任务只承载一个可独立闭环目标、任务默认按垂直切片组织、单任务默认控制在约 5 个文件以内；代码变更类任务默认把“真实测试通过”纳入任务完成条件，免测场景已显式写明理由，并明确“是否已获得开始实施授权”；若开工意图来自采纳建议，也已先把建议收口成正式计划；受限计划不得直接作为实施授权，只有升级为正式执行计划后才允许进入执行。
- 驳回：用户明确索要计划，但没有命中本 skill 或没有输出任何受限计划 / 阻断计划；或只是把来源对象原文换一种说法重写，没有先收口 agent 对问题的理解，也没有落实到实施周期、文件 / 模块、代码落点目录树和执行步骤；或计划里仍充满占位词、顺序不清、阶段混杂、无法验证；或缺少任务完成条件、任务停止 / 结束条件或最大推进边界；或代码变更类任务缺少真实测试安排、把 `build` / `lint` / 静态检查冒充真实测试、无法真实测试却未记为阻断项；或本应先回流需求拆分 / Bug 定位 / 边界 / 验收，却被硬塞成实施规划；或未等用户明确开工就直接切进编码；或在计划阶段直接改代码；或最小任务仍按水平分层组织、超出约 5 个文件且未继续拆分；或用户只是采纳了建议却未先落计划就直接进入实现；或代码变更类计划只给文件名清单，没有给出代码落点目录树；或因为使用 `<proposed_plan>` 等专用包裹而丢失本 skill 模板要求的关键结构与字段。

## 执行结果归档要求

- 将实施范围、实施周期、阶段划分、文件落点、代码落点目录树、任务顺序、真实测试安排、验证点、风险和阻断项记录到 `artifact-storage-rules` 约定的实施总览/实施周期文档中，不与需求文档、Bug 文档、验收标准文档或最终验收文档混写。
- 文档文件名、根目录和同文档更新策略统一遵循 `../artifact-storage-rules/references/path-map.yaml` 与 `../artifact-storage-rules/references/update-policy.md`。
- 实施总览文件名必须使用 `YYYY-MM-DD_HHmmss` 时间前缀，并保留来源对象标识（需求或 Bug）后追加“_实施总览”，例如：`2026-04-16_141518_多业务复用Swap适配控制需求_实施总览.md`。
- 实施周期文件名必须使用 `YYYY-MM-DD_HHmmss` 时间前缀，并保留来源对象标识（需求或 Bug）后追加“_实施周期NN_中文主题”，例如：`2026-04-16_141518_多业务复用Swap适配控制需求_实施周期01_接口与配置落点.md`；不得只写“时间_阶段_说明”。
- 同一个来源对象的实施总览应持续更新同一份总览文档；若拆周期，也应持续更新对应周期文档，而不是重复新建多份平行记录。
- 归档内容至少包含对应来源对象（需求或 Bug）、agent 当前理解的问题 / 目标、本轮范围、非范围、当前优先闭环、关键假设或待确认点、实施总览、阶段清单、目录 / 文件落点、代码落点目录树、任务清单、推荐顺序、真实测试安排、任务完成条件、任务停止 / 结束条件、最大推进边界、每步验证点、自审结论、用户确认状态和“是否已获得开始实施授权”状态。
- 对每个最小任务，归档中必须体现依赖关系、垂直切片目标、预计触达文件数，以及“实现 -> 真实测试 -> 审查 -> 验收”闭环顺序；若无需真实测试，必须同时保留明确理由。
- 对高复杂度计划，归档建议补充图形化执行路径（流程图/时序图/泳道表）以降低沟通成本。
- 若涉及代码生成、修改或重构，归档中的“现状与落点”必须包含“代码落点目录树（`text` 代码块）”，并在树节点后标注用途；若同时涉及新增文件，也必须在这棵树里一并体现，普通文件名清单或普通条目描述都不合格。
- 若涉及数据库新表或字段变更，归档必须包含对应 SQL（`CREATE TABLE` / `ALTER TABLE`），不能只写“会改数据库”。
- 如果后续来源对象变更、边界变化或验收标准调整推翻原方案，应在同一份实施总览或对应周期文档中更新新旧差异和重新确认状态。
- 如果后续发生的是已实现需求上的新增改动，应创建或增补一份新的实施周期文档记录本次改动的独立闭环，旧周期保留为历史记录。
- 如果一个需求被拆成多个独立子任务周期，归档中必须明确每个周期只承载一个清晰目标，并标明周期之间的衔接关系。
