# Plan Mode 输出闸门

## 适用场景

在以下任一场景中，最终输出前必须执行本闸门：

- 当前上下文处于 Plan Mode。
- 最终回复准备使用 `<proposed_plan>`。
- 用户明确要求“给计划 / 怎么做 / 方案步骤 / 按计划执行 / 按建议执行”。
- 当前回复声称自己是一份正式实施计划、受限计划或阻断计划。

外层 `<proposed_plan>` 只负责渲染协议，不改变仓库级计划正文要求。

## 计划类型判定

先把当前输出归入以下一种类型，再按对应矩阵检查：

| 计划类型 | 判定条件 | 输出目标 |
| --- | --- | --- |
| 正式实施计划 | 来源对象、边界和验收标准足以指导后续实现 | 可直接交给后续 agent 作为实施依据 |
| 受限计划 | 用户正在索要计划，但来源对象、验收、边界或落点仍缺关键前置条件 | 明确缺口、禁止实施、说明升级条件 |
| 阻断计划 | 当前缺口已经阻止正确规划或继续执行 | 说明阻断证据、恢复路径和停止边界 |

不允许把“建议摘要”“方案概览”“实现思路”伪装成上述任一计划。

## 正式实施计划必填字段

正式实施计划缺少任一项即判定不通过，必须补齐或重写：

| 字段 | 合格标准 |
| --- | --- |
| 当前计划最终方案简要说明 | 必须位于正文最开头，在“基本信息”之前；用 1-3 句或等价短 bullet 交代推荐方案、主落点和选择原因。 |
| 对应来源对象 | 写明需求 / Bug / 当前来源对象标识；尚未落盘时写明拟落盘路径。 |
| agent 理解的问题 / 目标 | 同时说明用户原始目标和 agent 当前理解。 |
| 当前计划范围 | 明确本轮覆盖内容。 |
| 明确不在范围 | 明确不会做什么，避免实施外溢。 |
| 当前优先闭环 | 写清本次先闭环哪一件事。 |
| 关键假设 / 待确认点 | 写出当前仍依赖的前提。 |
| 是否已获得开始实施授权 | 明确是 / 否；不能用计划确认替代开工授权。 |
| 需求与实施计划全量顺序实施方案 | 新项目、项目初期或多来源对象场景必填；必须写明总表路径、项目或来源集合标识、总执行顺序和本实施总览在总表中的位置。 |
| 实施周期总览 | 至少说明周期数、周期顺序、第一期 / 第二期 / 第三期等期次定位、周期目标、进入条件、收口条件、完成标志和衔接关系。 |
| 阶段计划 | 每个阶段只做一件事，并写清输入、输出和验证门槛。 |
| 最小任务清单 | 每个任务只承载一个目标，且必须归属到唯一实施周期。 |
| 周期内最小任务执行顺序 | 每个周期都要列出本周期最小任务顺序，先当前周期逐个闭环，再进入下一周期。 |
| 最小任务所属周期与周期内顺序 | 每个最小任务都要写所属周期和周期内顺序；不得跨多个周期复用同一最小任务。 |
| 垂直切片目标 | 每个最小任务都要写。 |
| 预计触达文件数 | 每个最小任务都要写；超过约 5 个文件需继续拆分或说明原因。 |
| 文件 / 模块落点 | 明确到目录、模块或文件。 |
| 代码落点目录树 | 涉及代码生成、修改或重构时必填。 |
| 新增文件目录树 | 涉及新增文件时必填，树节点后必须标注用途。 |
| 数据库 SQL | 涉及新表或字段变更时必填 `CREATE TABLE` / `ALTER TABLE`。 |
| 实施步骤 | 每步标明所属周期、阶段、最小任务和本步只做什么。 |
| 每步验证点（对应真实测试安排） | 能指导后续编码审查、功能验证、回归判断和验收准备，并应落在真实测试安排章节中。 |
| 真实测试安排（独立章节） | 涉及行为变更时必须单独成章，写明测试入口、依赖环境、样本 / 数据来源和通过标准。 |
| 免测理由 | 仅纯文档、纯注释、纯排版、纯静态资源改名 / 搬运等不影响运行结果的场景可免测。 |
| 任务完成条件 | 每个最小任务都要写。 |
| 任务停止 / 结束条件 | 每个最小任务都要写。 |
| 最大推进边界 | 写明本次 agent 最多推进到哪里。 |
| 风险与阻断项 | 写真实风险和阻断，不写空话。 |
| 自审结论 | 覆盖字段完整性、阶段单一目标、最小任务闭环、可执行性和图文一致性。 |

## 受限计划必填字段

受限计划不能作为实施授权。缺少任一项即判定不通过：

| 字段 | 合格标准 |
| --- | --- |
| agent 理解的问题 / 目标 | 写清当前想规划什么。 |
| 当前不可形成正式实施计划的原因 | 明确缺的是来源对象、边界、验收、落点、测试条件还是用户决策。 |
| 已完成侦察证据 | 列出已读文件、命令、资料或当前证据。 |
| 前置缺口 | 只列无法通过继续侦察解决的关键缺口。 |
| 建议流转顺序 | 写明应先转需求、缺口、边界、验收还是继续侦察。 |
| 计划成文后的预期结构 | 说明补齐后会按正式计划哪些章节落地。 |
| 禁止实施说明 | 明确补齐前不得编码、改码、重构、测试实施或连续执行。 |
| 升级条件 | 写明满足哪些条件后可升级为正式实施计划。 |

## 阻断计划必填字段

阻断计划用于停止错误推进。缺少任一项即判定不通过：

| 字段 | 合格标准 |
| --- | --- |
| 阻断对象 | 写明被阻断的是需求、计划、实现、测试还是交付。 |
| 阻断证据 | 引用真实文件、命令、对话确认或缺失证据。 |
| 阻断原因 | 写清为什么继续会导致错误实现、错误测试或错误归档。 |
| 恢复路径 | 写明恢复到哪个 skill / 哪个阶段 / 哪个文件。 |
| 不得执行事项 | 写明在恢复前禁止哪些动作。 |
| 重新进入条件 | 写明恢复后从哪里重启。 |

## 硬失败结构

出现以下任一情况，必须判定当前计划不合格并重写：

- 正文主结构是 `Summary`、`Key Changes`、`Implementation Changes`、`Public Interfaces`、`Test Plan`、`Assumptions` 等通用工程计划壳。
- 只有摘要、变更点、测试摘要，没有阶段计划和最小任务清单。
- 正文从“基本信息”或“agent 理解的问题 / 目标”直接开头，缺少位于最前面的“当前计划最终方案简要说明”。
- 实施周期只写成概述，没有周期顺序、期次定位、进入条件、收口条件或周期内最小任务列表。
- 新项目、项目初期或多来源对象场景缺少“需求与实施计划全量顺序实施方案”，或总表无法回指需求文档、验收标准、实施总览和实施周期。
- 最小任务没有按周期分组和排序，或只列成一张无所属周期的全局清单。
- 执行步骤跨过当前未收口周期，提前进入下一周期。
- 一个最小任务同时归属多个实施周期。
- 计划把多个最小任务先连续实现，再在周期末统一测试、统一审查或统一验收。
- 有最小任务，但每个任务缺少完成条件或停止 / 结束条件。
- 有测试段落，但没有独立的真实测试安排章节，或其中没有真实测试入口、依赖环境、样本 / 数据来源和通过标准。
- 把 `build`、`lint`、静态检查、代码审查或人工阅读写成真实测试。
- 涉及新增文件但没有目录树。
- 涉及数据库变更但没有 SQL。
- 使用 `<proposed_plan>` 后省略仓库模板字段。
- 计划中保留 `TBD`、`TODO`、`后续再补`、`实现时再看` 等占位词。

## 输出前动作

1. 先判定计划类型。
2. 按对应必填字段矩阵逐项检查。
3. 若缺字段，先在内部补齐或重写计划正文，不输出半成品计划。
4. 若当前只能受限或阻断，明确写出禁止实施说明。
5. 只有矩阵通过后，才允许输出 `<proposed_plan>` 或声明计划已完成。

## 回归样例要求

当本 skill 或本闸门发生修改时，至少用一个历史不合格计划样例做回归检查。样例应能证明：

- 通用工程计划壳会被拒绝。
- 缺少阶段计划、真实测试安排（独立章节）或自审结论会被拒绝。
- 受限计划不会被误判成正式实施授权。
- 缺少周期顺序、周期内最小任务顺序，或把测试 / 审查 / 验收集中到周期末的计划会被拒绝。
