# plan-mode-and-cycle-contracts

> 归属 owner：`implementation-planning-rules`。本文件是条件路由的细则承载，不产生独立自动触发入口；所有原有规则、用户习惯、安全边界、授权边界、暂停条件和归档要求在迁移后继续有效。

## 进入后先做什么

1. 先确认来源对象文档、边界结论和验收标准已经基本稳定。
2. 先写“当前计划最终方案的简要说明”：用 1-3 句交代推荐方案、主落点和为什么这么做；紧接着再把“agent 对当前问题的理解”写清：当前理解的问题 / 目标是什么，本轮准备覆盖什么，不覆盖什么，当前优先闭环是什么，哪些是假设或仍待确认。
2.1 若本轮是“用户先问计划怎么做”，先判断当前能否直接进入正式实施规划；若不能，也必须先整理一版“受限计划 / 阻断计划”，明确前置缺口、建议流转和后续计划落点。
2.2 若当前环境要求用 `<proposed_plan>` 之类的专用包裹输出计划，先把本 skill 要求的计划正文结构整理完整，再在最终输出时做外层包裹；不得先按外部简版格式压缩，导致正文缺字段。
3. 确认当前要规划的是整个需求，还是拆分后当前优先闭环子项。
3.1 若当前是新项目 / 多需求 / 多实施文档场景，先判断是否需要创建或更新“需求与实施计划全量顺序实施方案”；需要时先按全量顺序总表确定跨需求顺序，再回到单个来源对象的实施总览与实施周期拆分。
4. 先看仓库现状，找到相关目录、模块、已有实现和可能的复用点。
5. 列出本轮必须落地的文件 / 模块落点，不先展开代码细节。
6. 先确认实施周期顺序和每个周期的大进度定位（第一期、第二期、第三期等），再拆周期内阶段边界、最小任务顺序、依赖关系、最小闭环、真实测试安排和验证点。
7. 先画依赖关系，再按垂直切片识别最小任务，避免直接按技术层分堆。
8. 对每个最小任务估算文件落点；若单任务预计会触达超过约 5 个文件，优先继续拆小。
9. 如果实现路径存在多种合理方向，先做 2-3 个方案比选并给出推荐。
10. 找到当前来源对象对应的实施总览文档；如果还没有，就按 `artifact-storage-rules` 的命名模板初始化独立实施总览文档，并判断是否需要同步拆出实施周期文档。
10.1 若本轮是在补充已有实施文档，先盘点当前来源对象已有的实施总览和实施周期文档，判断哪些周期受影响、哪些周期保持不变，以及是否需要为新增闭环补建周期文档。
11. 若当前开工意图来自“采纳建议/采纳方案”，先把建议内容改写为正式计划结构，再判断是否允许进入实现。
## 默认执行流程

1. 默认先读 `references/plan-entry-checklist.md`，确认进入实施规划阶段前的前置条件是否满足。
2. 再读 `../artifact-storage-rules/references/path-map.yaml` 与 `../artifact-storage-rules/references/update-policy.md`，确认实施总览/实施周期文档路径、命名模板和同文档更新策略。
3. 若当前是新项目 / 多需求 / 多实施文档场景，先读 `references/full-sequence-master-plan.md`，按 `artifact-storage-rules` 命名模板生成或更新“需求与实施计划全量顺序实施方案”，再进入单来源对象实施总览 / 实施周期。
4. 再读 `references/plan-structure-template.md`，按统一结构生成或更新实施总览文档，并在任务较大时拆出实施周期文档。
5. 如果任务粒度、依赖顺序或最小闭环不好判断，再读 `references/task-granularity-and-order.md`。
6. 如需判断与相邻 skill 的边界、或对照正反例，再读 `references/plan-boundaries-and-examples.md`。
7. 输出当前计划最终方案简要说明、问题理解、本轮范围、非范围、当前优先闭环、关键假设 / 待确认点、实施周期、周期顺序、阶段划分、最小任务清单、周期内最小任务顺序、文件 / 模块落点、任务顺序、真实测试安排、验证步骤、任务完成条件、任务停止 / 结束条件、最大推进边界、风险与阻断项，并更新到当前来源对象对应的实施总览/实施周期文档；若存在全量顺序总表，还必须同步回写本来源对象在总表中的顺序、依赖、当前状态和下一执行入口。
7.1 若本轮是用户显式索要计划，但前置条件仍未齐备，则至少输出“当前计划最终方案简要说明 + 问题理解 + 当前不可直接形成正式实施计划的原因 + 前置缺口 + 建议流转顺序 + 计划成文后的预期结构”；不得直接结束为“先去补需求”，也不得表现成计划规则没有触发。
7.2 在任务清单中显式标注每个最小任务的所属周期、周期内顺序、依赖关系、垂直切片目标、预计触达文件数，以及是否超过“约 5 个文件”警戒线。
7.3 若某任务偏向水平分层拆法（例如单独一批前端、一批后端、一批数据库），必须继续改写为按端到端闭环组织的垂直切片任务，或明确记录为何当前场景无法采用垂直切片。
7.4 若最终回复使用 `<proposed_plan>` 之类专用计划包裹，必须先读 `references/plan-output-gate.md` 并按正式计划 / 受限计划 / 阻断计划字段矩阵完成输出前检查；缺任一必填字段时不得输出半成品计划，必须先补齐或重写。
7.5 `<proposed_plan>` 包裹内正文至少仍需完整承载本 skill 要求的当前计划最终方案简要说明、问题理解、本轮范围、非范围、当前优先闭环、关键假设 / 待确认点、实施周期、阶段划分、最小任务清单、文件 / 模块落点、真实测试安排、任务完成条件、任务停止 / 结束条件和最大推进边界；允许合并同类小节，不允许丢字段。
8. 若涉及新增文件，实施文档中必须补齐“新增文件清单目录树（`text` 代码块 + 用途注释）”；普通条目描述视为计划未收口。
9. 若涉及数据库新表或字段变更，实施文档中必须补齐 SQL（`CREATE TABLE` / `ALTER TABLE`）；缺 SQL 视为计划未收口。
10. 如果还没有实施总览文档，则按 `artifact-storage-rules` 约定的命名规则创建；如同一天存在多个近似标题，在不改变中文语义前提下补充更具体描述以避免重名。
11. 如果当前任务体量较大、闭环较多，必须先维护一份实施总览，再按 `来源对象标识_实施周期NN_中文主题` 拆出周期文档；每个周期只承载一个清晰目标，并写清与前后周期的顺序关系，周期内仍必须继续拆到最小任务清单。
11.1 新项目或多来源对象场景必须先维护一份“需求与实施计划全量顺序实施方案”：总表按项目级顺序列出每个需求 / Bug 来源对象、验收标准、实施总览、实施周期、周期内最小任务、前置依赖、阻断项和当前状态；单个来源对象的实施总览只写本来源对象，不承担跨需求排序。
11.2 若当前是在补充已有需求对应的实施文档，则更新实施总览时必须同步处理受影响的周期子文档；已有周期拆分却只更新总览、未更新受影响周期，视为实施文档未收口。
12. 如果当前需求属于“已实现需求上的新增改动”，必须在现有需求主文档已更新后，新建或增补本次改动对应的实施周期文档；不要沿用已完成的旧实施周期正文作为本次执行计划。
13. 若当前计划仍把多个并列功能塞进同一实施周期，或单个阶段承载多个目标，必须继续拆到更小的最小任务粒度；必要时再补拆新的实施周期文档。只要仍存在“一个任务项里混着多个实现目标 / 多个验证目标 / 多个验收目标”，都视为计划未收口。
13.1 若当前最小任务预计触达文件数明显超过约 5 个，或无法单次专注完成，也视为计划未收口，应继续拆分或重组。
13.2 若当前计划只列出一组全局最小任务，没有归属到具体实施周期和周期内顺序，或把多个最小任务的测试、审查、验收集中到周期末统一处理，也视为计划未收口，必须改为“周期 -> 周期内最小任务逐个全流程闭环 -> 周期收口 -> 下一周期”的顺序。
14. 计划写完后，必须再读 `references/plan-review-checklist.md` 做一次自审，并再次按 `references/plan-output-gate.md` 做输出前闸门检查，重点检查实施周期、阶段职责单一性、真实测试安排、占位词、可执行性和验证闭环。
15. 对复杂任务补齐图形化步骤表达（至少一种：流程图/时序图/泳道表），并校验图文一致。
16. 自审通过后，请用户先 review 当前实施文档；如用户提出调整，继续更新同一份实施总览或对应周期文档，不新建无关平行记录。
17. 计划确认后，仍必须等待用户明确说“开始实施 / 开始执行 / 按实施计划执行”才能进入编码基线域和相应代码位点域；仅有“计划写完了”“需求确认了”不构成开工授权。
17.1 若用户直接说“开始实施 / 开始实现 / 开始执行 / 直接做 / 继续做完 / 按文档实现”，但当前没有可执行计划、任务完成条件、任务停止 / 结束条件和最大推进边界，必须先补本轮受限计划并停在计划收口处，不得直接编码。
17.2 若用户直接说“按建议执行 / 按方案执行 / 就按你刚才说的做 / 按你说的做”，但当前只有建议、方案比选或修复思路，没有正式实施计划，也必须先补齐实施计划并停在计划收口处，不得直接编码。
17.3 受限计划不得作为实施授权；用户即使明确采纳，agent 也只能先补齐缺失前置条件并将其升级为正式执行计划，未升级前禁止进入编码、改码、重构、测试实施或其他执行动作。
18. 若用户已明确“开始实施 / 开始执行”，则按 `autonomous-execution-rules` 默认连续推进“当前实施周期内最小任务A实现 -> 最小任务A真实测试 -> 最小任务A审查 -> 最小任务A验收 -> 最小任务B…… -> 当前实施周期收口 -> 下一实施周期”；每个最小任务都必须先完成自己的真实测试、审查、验收闭环，才允许进入下一个任务；除阻断级节点外不在中间重复征求下一步确认。
## 权责边界与不负责事项

- 只负责把已确认来源对象转成实施方案，不代替 `requirement-splitting-rules` 做顶层需求拆分，也不代替 Bug 域做根因定位。
- 不代替 `acceptance-criteria-rules` 编写验收标准；验收标准应已基本存在。
- 不代替实际编码，不在本 skill 内直接展开实现代码。
- 不代替测试域制定完整测试策略，但必须把当前计划对应的真实测试安排、最小验证点和免测理由写清。
- 不允许在用户明确索要计划时静默回退成“这还不能计划”；即使只能给受限计划，也要把计划前置缺口和受限边界写出来。
- 不允许因为运行环境提供了 `<proposed_plan>` 或其他专用计划包裹，就把正文退化成通用摘要、产品 PRD 摘要或只剩 3-5 条 bullet 的泛化计划；外层协议不能覆盖项目内计划格式。
- 不允许把 `Summary`、`Key Changes`、`Public Interfaces`、`Test Plan`、`Assumptions` 等通用工程计划壳作为 Plan Mode 主结构；一旦输出中出现这类主结构，必须判定为本 skill 未正确执行，立即按 `references/plan-structure-template.md` 重写计划正文。
- 不允许在本 skill 内边写计划边改代码，或通过“试做一点”来代替计划判断。
- 不接受 `TBD`、`TODO`、`后续再补`、`实现时再看` 这类空泛占位计划。
- 不接受“一个开发周期覆盖很多功能点、很多阶段目标”的笼统计划；能拆小周期就必须拆小周期，能继续拆成更小最小任务就不得停在粗粒度步骤。
- 不接受“一个阶段同时做多件事”的混合步骤设计；阶段目标必须单一。
- 不接受按前端 / 后端 / 数据库水平分层堆计划、却缺少端到端最小闭环的任务拆法；若无充分理由，默认视为计划未收口。
- 不接受明显超过约 5 个文件、且无法单次专注完成的最小任务定义；能拆就必须继续拆。
- 不为同一个来源对象重复新建多份无关实施总览；若总览文档已存在，应优先更新原文档并按需补充实施周期。
- 不把项目级“需求与实施计划全量顺序实施方案”写成某个单需求的实施总览；总表用于跨来源对象排序，实施总览用于单来源对象落地，两者职责必须分开。
- 不因为已经有某个来源对象的实施总览，就跳过新项目 / 多需求场景下的全量顺序总表；只要存在多个需求或多份实施计划需要统一排序，总表就是进入实施执行前的上层入口。
- 不得把“实施计划已经完成”误判成“已经得到开始实施授权”；开工授权必须来自用户在当前任务中的明确表态。
- 不得把“用户采纳了建议/方案”误判成“已经有正式执行计划”；建议只有转写为计划后才能进入实现。
- 不得把“受限计划已输出”误判成“已经拿到实施授权”；受限计划只有在前置条件补齐并升级为正式执行计划后，才允许进入执行。
- 如果技术分歧的根源其实是业务目标不清、范围有争议或验收没定，应回流相邻需求域或验收域 skill，而不是假装用实施规划掩盖前置问题。
- 如果技术分歧出现在“已实现需求上的新增改动”，应优先把本次改动单独落到新的实施周期文档中，而不是沿用旧周期继续补丁式覆盖。
## 需要暂停并确认的条件

- 来源对象、边界或验收标准仍在明显变化，当前计划随时会失效。
- 当前任务还没拆出可独立推进的优先闭环，却试图一次规划全部大需求。
- 当前任务明明可以拆成多个独立子任务周期，却仍被写成一个大周期。
- 当前任务虽然拆成了最小任务，但仍明显按水平分层组织，缺少端到端垂直切片闭环。
- 当前最小任务预计触达文件数明显超过约 5 个，或无法单次专注完成。
- 计划涉及代码变更，但没有为对应最小任务写出真实测试入口、依赖环境、样本 / 数据来源或通过标准。
- 计划把 `build`、`lint`、静态检查、代码审查或人工阅读误写成真实测试。
- 实现路径存在重大分歧，但尚未形成 2-3 个可比方案和推荐结论。
- 无法确定主要文件落点、模块职责或依赖顺序，说明仍需先回看仓库现状或回流需求域。
- 计划中仍存在明显占位词、跳步描述、阶段目标不单一或无法验证的任务项。
- 用户尚未明确“开始实施 / 开始执行”，但流程准备从实施文档切换到正式编码。
- 用户虽已明确“按建议执行 / 按方案执行”，但建议尚未被收口成正式实施计划。
- 用户明确索要计划，但当前仍没有来源对象文档、前置验收口径或可参考现状；此时必须停在受限计划 / 阻断计划，而不是假装已经拿到正式实施条件。
- 新项目或多需求场景下已出现多份需求 / 实施文档，但无法判断跨需求依赖、第一期入口或总执行顺序；此时应先停在全量顺序总表补齐，不得直接进入任一局部实施周期。
- 当前环境要求专用计划包裹，但正文仍未满足本 skill 模板必填项；此时应继续补正文结构，而不是把包裹当作已收口。
- 用户即使明确说“就按这个受限计划实施”，但受限计划仍未升级为正式执行计划；此时必须继续补前置条件，不得进入编码、改码、重构、测试实施或其他执行动作。
## 执行通过 / 驳回标准

- 通过：能够形成一份可直接指导后续编码的实施总览与必要的实施周期文档，或在前置条件未齐时形成一份结构完整的受限计划 / 阻断计划；至少说明当前计划最终方案简要说明、agent 当前理解的问题 / 目标、本轮范围、非范围、当前优先闭环、关键假设或待确认点、实施周期、周期顺序、期次定位、阶段划分、最小任务清单、周期内最小任务顺序、文件 / 模块落点、代码生成 / 修改 / 重构类任务的“代码落点目录树”、任务顺序、依赖关系、最小闭环、真实测试安排（独立章节）、任务完成条件、任务停止 / 结束条件、最大推进边界、风险和阻断项，并完成自审与用户确认门槛；若当前只能给受限计划，也必须明确前置缺口、建议流转顺序和何时升级为正式实施规划。且实施规划阶段保持只读、每个阶段只做一件事、每个最小任务只承载一个可独立闭环目标、任务默认按垂直切片组织、单任务默认控制在约 5 个文件以内；代码变更类任务默认把“真实测试通过”纳入任务完成条件，免测场景已显式写明理由，并明确“是否已获得开始实施授权”；若开工意图来自采纳建议，也已先把建议收口成正式计划；受限计划不得直接作为实施授权，只有升级为正式执行计划后才允许进入执行。
- 驳回：用户明确索要计划，但没有命中本 skill 或没有输出任何受限计划 / 阻断计划；或缺少位于正文最开头的“当前计划最终方案简要说明”，或只是把来源对象原文换一种说法重写，没有先收口 agent 对问题的理解，也没有落实到实施周期、周期顺序、周期内最小任务顺序、文件 / 模块、代码落点目录树和执行步骤；或计划里仍充满占位词、顺序不清、阶段混杂、无法验证；或缺少任务完成条件、任务停止 / 结束条件或最大推进边界；或代码变更类任务缺少真实测试安排（独立章节）、把 `build` / `lint` / 静态检查冒充真实测试、无法真实测试却未记为阻断项；或本应先回流需求拆分 / Bug 定位 / 边界 / 验收，却被硬塞成实施规划；或未等用户明确开工就直接切进编码；或在计划阶段直接改代码；或最小任务仍按水平分层组织、超出约 5 个文件且未继续拆分；或最小任务未归属具体周期、跨多个周期复用、跳过当前周期进入后续周期，或把多个最小任务统一测试/审查/验收；或用户只是采纳了建议却未先落计划就直接进入实现；或代码变更类计划只给文件名清单，没有给出代码落点目录树；或因为使用 `<proposed_plan>` 等专用包裹而丢失本 skill 模板要求的关键结构与字段；或 Plan Mode 输出仍以 `Summary`、`Key Changes`、`Public Interfaces`、`Test Plan`、`Assumptions` 等通用工程计划小节作为主结构。
## 执行结果归档要求

- 将实施范围、实施周期、周期顺序、期次定位、阶段划分、文件落点、代码落点目录树、任务顺序、周期内最小任务顺序、真实测试安排、验证点、风险和阻断项记录到 `artifact-storage-rules` 约定的实施总览/实施周期文档中，不与需求文档、Bug 文档、验收标准文档或最终验收文档混写。
- 若当前存在多个来源对象或多份实施文档，必须将跨需求总顺序、来源对象清单、验收标准链接、实施总览链接、实施周期链接、周期内最小任务摘要、依赖关系、阻断项和执行状态记录到 `artifact-storage-rules` 约定的“需求与实施计划全量顺序实施方案”中。
- 文档文件名、根目录和同文档更新策略统一遵循 `../artifact-storage-rules/references/path-map.yaml` 与 `../artifact-storage-rules/references/update-policy.md`。
- 实施总览文件名必须使用 `YYYY-MM-DD_HHmmss` 时间前缀，并保留来源对象标识（需求或 Bug）后追加“_实施总览”，例如：`2026-04-16_141518_多业务复用Swap适配控制需求_实施总览.md`。
- 实施周期文件名必须使用 `YYYY-MM-DD_HHmmss` 时间前缀，并保留来源对象标识（需求或 Bug）后追加“_实施周期NN_中文主题”，例如：`2026-04-16_141518_多业务复用Swap适配控制需求_实施周期01_接口与配置落点.md`；不得只写“时间_阶段_说明”。
- 同一个来源对象的实施总览应持续更新同一份总览文档；若拆周期，也应持续更新对应周期文档，而不是重复新建多份平行记录。
- 若用户本轮要求同时补充需求文档与实施文档，实施域归档必须同时说明：实施总览更新范围、受影响周期文档清单、未受影响周期清单，以及新增周期文档的补建原因。
- 归档内容至少包含对应来源对象（需求或 Bug）、agent 当前理解的问题 / 目标、本轮范围、非范围、当前优先闭环、关键假设或待确认点、实施总览、实施周期顺序、每个周期的期次定位、阶段清单、目录 / 文件落点、代码落点目录树、任务清单、周期内推荐顺序、真实测试安排、任务完成条件、任务停止 / 结束条件、最大推进边界、每步验证点、自审结论、用户确认状态和“是否已获得开始实施授权”状态。
- 对每个最小任务，归档中必须体现所属周期、周期内顺序、依赖关系、垂直切片目标、预计触达文件数，以及“实现 -> 真实测试 -> 审查 -> 验收”闭环顺序；若无需真实测试，必须同时保留明确理由。
- 对高复杂度计划，归档建议补充图形化执行路径（流程图/时序图/泳道表）以降低沟通成本。
- 若涉及代码生成、修改或重构，归档中的“现状与落点”必须包含“代码落点目录树（`text` 代码块）”，并在树节点后标注用途；若同时涉及新增文件，也必须在这棵树里一并体现，普通文件名清单或普通条目描述都不合格。
- 若涉及数据库新表或字段变更，归档必须包含对应 SQL（`CREATE TABLE` / `ALTER TABLE`），不能只写“会改数据库”。
- 如果后续来源对象变更、边界变化或验收标准调整推翻原方案，应在同一份实施总览或对应周期文档中更新新旧差异和重新确认状态。
- 如果后续发生的是已实现需求上的新增改动，应创建或增补一份新的实施周期文档记录本次改动的独立闭环，旧周期保留为历史记录。
- 如果一个需求被拆成多个独立子任务周期，归档中必须明确每个周期只承载一个清晰目标，并标明周期之间的衔接关系。
## 极致完整性与零决策执行硬闸门

- 实施总览、实施周期和最小任务卡必须把强模型已经做出的业务/技术/测试决策冻结下来；普通模型只执行当前周期的当前 `TASK-*`，不得自行补默认值、异常处理、文件落点、回滚或测试断言。
- 行为变更默认必须建立实施总览和至少一个实施周期；每个周期必须有完整周期模板，每个周期内必须有可独立闭环的最小任务。旧周期不能伪装承载已实现需求上的新改动。
- 每个 `TASK-*` 必须明确唯一目标、前置条件、允许文件、目标符号/区段、操作类型、禁止触碰区、步骤、真实测试、断言、失败预期、清理、回滚、完成条件、停止条件和最大推进边界；预计触达文件超过 5 个时必须继续拆分或给出不可拆分证据。
- 每个任务必须按“实现 -> 真实测试 -> 审查 -> 验收”逐个闭环；禁止先连续实现多个任务后统一测试、统一审查或统一验收。
- 追踪必须覆盖 `REQ/RULE -> AC -> CYCLE -> TASK -> 文件/符号 -> TEST -> EVIDENCE`，并能从实施任务反向回到需求和验收；孤立 ID、重复 ID、单向映射或未决 P0/P1 均阻断。
- 真实测试必须写精确命令、local 环境、样本/fixture、断言、失败预期、清理和证据路径；`build`、`lint`、静态检查、人工阅读和代码审查不能代替真实测试。免测只能用于纯文档/纯注释/纯排版/不改变运行结果的静态资源整理，并写 `N/A + 原因 + 证据`。
- 总览至少包含边界图、周期依赖图和端到端流程/时序图；周期至少包含任务依赖图和一张领域匹配图。每张 Mermaid 图必须写目的和关联 ID，图文术语一致并通过语法校验。
- 发现计划落点与代码基线不一致、符号不存在、命令失败、断言失败、依赖缺失或出现安全/数据损坏风险时，立即停止并记录 `GAP-*`，不得猜测替代实现路径。
## Markdown 图片资产实施规则

- 实施总览、实施周期和最小任务卡必须显式写出图片资产决策：`需要`，或 `N/A + 原因 + 证据`；禁止“按需”“以后补图”等占位语。
- 图片必需场景包括 UI/原型、截图证据、视觉对比、真实产物、空间布局、外观基线及 Mermaid 无法准确表达的视觉内容。流程、时序、状态、依赖和数据关系仍必须使用 Mermaid，图片不得替代既有图形门禁。
- 真实图片生成必须调用 `imagegen`，将输出显式写入项目根 `doc/data/images/`；命名采用 `<document_stem>.<asset-slug>-v<number>.<ext>`，返修递增版本，生成失败即记录阻断并停止，不得使用程序绘图、占位图、Base64 或外部热链冒充。
- Markdown 引用必须从当前文档位置计算 `/` 分隔相对路径（需求/实施文档通常为 `../data/images/<file>`），alt 非空且包含 `IMG-*`；禁止绝对路径、反斜杠、`file://`、远程图片、HTML `<img>`、越界路径和直接引用 `doc/data/<file>`。
- 图片资产清单必须登记 `IMG-*`、用途/生成输入、来源、相对路径、版本、关联 `REQ/RULE`、`AC`、`CYCLE`、`TASK`、引用章节、敏感状态和版权状态；共享图片可多文档引用，未被任何 Markdown 引用的图片必须作为孤儿阻断。
- 每个涉及图片的最小任务必须给出目标资产路径、生成命令、引用章节、格式/签名检查、清理或回滚步骤，并在“实现 -> 真实测试 -> 审查 -> 验收”闭环中验证；imagegen 通道不可用时不得降级为伪图。
- 输出实施计划、总览、周期或任务卡前，必须读取 `../artifact-delivery-gate-rules/references/plain-language-document-contract.md`。正文先说推荐方案、业务影响和完成结果；文件树、符号、任务 ID、命令、测试和追踪矩阵统一放到执行或追踪附录。
- 同时必须读取 `../artifact-delivery-gate-rules/references/review-acceptance-gate-contract.md`，把第三方、浏览器和专项验收写成条件化门禁，不因不适用条件阻断无关实施准备。
