# 进入实施规划前检查清单

## 前置条件

- 已经存在对应需求文档，且目标、背景、输入输出基本成立。
- 需求缺口已经补到可继续推进，不再停留在一句话想法层。
- 当前优先闭环已经明确；如果任务过大，已先经过需求拆分。
- 边界结论已经基本明确，知道哪些属于本次、哪些不属于本次。
- 验收标准已经具备最小可参考口径，知道做到什么算完成。
- 当前没有未收敛的 gap / boundary / splitting / change 阻断项。

## 计划型提问补充

- 若用户本轮明确在问“怎么做 / 怎么改 / 先给计划 / 先出方案 / 先列步骤”，即使前置条件未全部满足，也必须先命中实施规划规则。
- 当前置条件未满足时，不是“不触发”，而是输出受限计划 / 阻断计划。
- 受限计划至少要写：当前计划最终方案简要说明、问题理解、当前范围、为什么还不能形成正式实施规划、缺哪些前置条件、建议先走哪条需求域或 Bug 域流转、正式计划补齐后会包含哪些结构。
- 受限计划不得作为实施授权；即使用户明确说“按这个受限计划做”，也只能继续补前置条件并升级为正式执行计划，未升级前不得进入编码、改码、重构、测试实施或其他执行动作。
- Plan Mode 下必须按 `references/plan-question-coverage.md` 逐维度识别实现层不确定决策点，并用 `request_user_input` 向用户弹窗多选（推荐项置首、禁用 `autoResolutionMs`）。
- 已弹窗但用户未选择的决策点，保持开放并标为“未决(待用户选择)”，禁止采纳推荐项或默认推进；只要存在未决决策点，正式计划一律不放行。

## 进入前先确认

- 当前要规划的是整个需求，还是某个优先子项。
- 当前仓库里涉及哪些目录、模块、接口、页面、表或脚本。
- 哪些现有实现可以复用，哪些必须新增。
- 是否存在 2-3 条都合理、但影响明显不同的技术路径。
- 是否已经知道计划完成后要如何验证第一闭环。
- 是否已经能写出本轮完成定义、停止条件和最大推进边界，避免开工后长文本模式无限续做。
- 是否已经知道计划完成后要如何验证第一闭环，且真实测试安排可以单独成章，而不是只散落在最小任务字段里。
- 是否已经明确实施文档应落到 `doc/3-实施/`，以及最终验收将单独落到 `doc/7-验收/`。
- 若本轮是用户显式索要计划但前置条件不全，是否已经准备输出受限计划，而不是直接回一句“先补需求再说”。
- 若当前只能输出受限计划，是否已经明确写出“这不是实施授权”，以及升级为正式执行计划前禁止进入执行。

## 不应进入本 skill 的信号

- 需求本身还缺字段、缺流程、缺前提、缺业务规则。
- “这是不是本次范围”还说不清。
- 验收口径完全没有，连完成标准都无法判断。
- 当前其实是在修历史缺陷，应转 Bug 域。
- 当前已经开始改代码，但没有先补齐前置争议，应先回总控层或相邻需求域 skill 纠偏。
