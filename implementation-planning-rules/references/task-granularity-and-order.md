# 任务颗粒度与顺序规则

## 颗粒度原则

- 实施周期是项目第一期、第二期、第三期等大进度单位，负责表达阶段性目标、进入条件、收口条件和与前后周期的衔接关系。
- 新项目、项目初期或多来源对象场景下，`需求与实施计划全量顺序实施方案` 是实施周期之上的总顺序入口，负责先定位当前应推进哪个来源对象和哪个实施周期。
- 执行顺序先按实施周期推进：当前周期未完成前，不得跳到后续周期；周期内部再按最小任务顺序逐个执行。
- 开发周期优先按“一个周期只承载一个清晰目标”拆分，而不是按“尽量多装几个功能”拼装。
- 如果一个任务可以拆成多个独立推进、独立验证、独立交接的子任务周期，就必须拆开。
- 每个阶段只做一件事；若一个阶段里出现多个并列目标，应继续拆成多个阶段，必要时拆成多个周期。
- 真正执行粒度是“最小任务”；一个最小任务应只对应一个明确目标，而不是把多个目标揉成一句话。
- 一个最小任务完成后，应能独立完成“实现 -> 真实测试 -> 审查 -> 验收”闭环，再进入下一个最小任务。
- 禁止先连续实现多个最小任务，再统一测试、统一审查或统一验收；这种写法说明最小任务没有真正闭环。
- 最小任务默认优先按“依赖图 + 垂直切片”组织，先追求端到端最小闭环，而不是先按前端 / 后端 / 数据库水平分层摊开。
- 单任务默认尽量控制在约 5 个文件以内；若明显超过，优先继续拆分或重组，而不是把大任务强行保留。
- 一个步骤应对应一个明确目标；若一个步骤跨越多个最小任务，说明粒度仍然过粗。
- 一个步骤最好能落在一组强相关文件 / 模块上，便于后续最小改动实施。
- 一个步骤完成后，最好能得到一个可检查的中间结论。
- 不要把步骤拆到“改一个变量名”这种过细颗粒度。
- 也不要写成“完成后端所有改动”“完成前端所有改动”这种过粗颗粒度。

## 最小任务定义

- 最小任务是当前实施计划中的最小可独立闭环单位，而不是浏览性分组标题。
- 一个最小任务至少要同时回答清楚：
  - 做什么
  - 改哪里
  - 如何测试
  - 如何审查
  - 如何验收
  - 什么算完成
- 若一个任务必须等多个并列能力都一起完成才能验证，通常说明它还可以继续拆。
- 若一个任务完成后连第一轮验证结果都拿不到，通常说明它仍然过粗。
- 若一个任务小到只剩机械性单行修改，通常应并入其直接业务闭环任务，不单独立项。
- 若一个任务只是“后端先做完 / 前端先做完 / 数据库先做完”，通常说明它仍停留在水平分层，而不是最小闭环任务。
- 若一个任务预计会同时触达很多弱相关文件，通常说明还没顺着依赖图拆到合适粒度。

## 顺序原则

- 先落基础约束，再落业务主链路，最后补外围适配。
- 先处理会影响后续多步判断的结构性改动，再处理局部实现。
- 先完成最小闭环，再扩展次级能力，不要一开始同时铺开所有边角。
- 先画依赖关系，再决定最小任务顺序；不要先按技术层列一串模块清单，再事后补闭环解释。
- 先划清开发周期边界，再安排周期内阶段和最小任务顺序；不要先写一长串步骤，最后再倒推阶段。
- 多来源对象场景先维护全量顺序总表，再进入单来源对象实施总览；不要在多个需求之间凭口头印象跳转。
- 先确认周期序号与期次定位（第一期、第二期、第三期等），再列周期内最小任务；不得把所有最小任务平铺成一张无周期归属的清单。
- 当前周期所有最小任务都通过“实现 -> 真实测试 -> 审查 -> 验收”后，才允许进入下一周期。
- 如果某一步依赖上一步的结论，必须显式写出依赖关系。
- 如果需要跨前后端或跨模块联动，先明确接口和数据边界，再分别推进实现。

## 最小闭环判断

- 当前计划是否能在一轮编码后形成“可验证的第一结果”。
- 如果第一闭环失败，是否容易回退或重新收口。
- 当前闭环是否过大，导致一轮改动无法稳定验证。
- 当前闭环是否过小，小到不足以证明方案有效。
- 当前闭环是否能作为一个独立开发周期单独交接。

## 正反例

- 正例：新增参数校验并返回明确错误
  - 实现：补请求参数校验与错误映射
  - 测试：补非法参数输入验证
  - 审查：确认未影响原合法请求链路
  - 验收：接口在非法输入下返回预期错误
- 正例：补齐列表页空状态展示
  - 实现：增加空数据态组件与文案
  - 测试：验证有数据/无数据两种渲染
  - 审查：确认未破坏现有布局与交互
  - 验收：页面空列表时正确显示空状态
- 反例：完成后端所有改动
  - 过粗，无法形成单一闭环
- 反例：顺手把接口、前端、文档一起做完
  - 混入多个目标，无法独立审查和验收
- 反例：先把后端模块全做完，再统一补前端
  - 属于水平分层推进，缺少端到端最小闭环
- 反例：把字段名从 A 改成 B
  - 过细，缺少独立业务闭环，应并入所属最小任务

## 从周期到最小任务的拆分示例

- 实施周期 01（第一期）：用户筛选入口闭环
  - 阶段 1：请求入口收口
  - 最小任务 1：补筛选参数解析与校验闭环
  - 阶段 2：查询链路落地
  - 最小任务 2：补查询条件拼装与结果验证闭环
- 实施周期 02（第二期）：用户筛选界面反馈闭环
  - 阶段 3：界面反馈收口
  - 最小任务 3：补页面筛选交互与空状态反馈闭环
- 正确执行顺序：
  - 周期 01 / 最小任务 1 实现 -> 最小任务 1 真实测试 -> 最小任务 1 审查 -> 最小任务 1 验收
  - 周期 01 / 最小任务 2 实现 -> 最小任务 2 真实测试 -> 最小任务 2 审查 -> 最小任务 2 验收
  - 周期 01 收口
  - 周期 02 / 最小任务 3 实现 -> 最小任务 3 真实测试 -> 最小任务 3 审查 -> 最小任务 3 验收
  - 周期 02 收口

## 常见坏味道

- 计划中出现 `TBD`、`TODO`、`后续再说`、`实现时决定`。
- 任务顺序不依赖真实代码落点，只是按口头习惯排列。
- 把需求拆分、实施规划、测试策略、发布方案混在一份计划里。
- 一个步骤跨多个不相干系统，导致无法独立验证。
- 一个开发周期同时承载多个功能目标，导致上下文难以衔接。
- 最小任务没有写所属周期和周期内顺序，导致执行时不知道先做第一期还是第二期。
- 当前周期还没完成全部最小任务验收，就把步骤跳到下一周期。
- 新项目或多需求场景没有全量顺序总表，就直接挑某个局部实施周期开工。
- 一个阶段描述里同时出现“先做 A，再补 B，顺手处理 C”。
- 一个最小任务同时写了多个实现目标、多个测试目标或多个验收目标。
- 多个最小任务先统一实现，再在周期末统一测试、审查、验收。
- 一个最小任务预计触达超过约 5 个文件，但计划里没有继续拆分说明。
- 任务顺序按前端 / 后端 / 数据库分堆罗列，却没有解释每个垂直切片如何独立闭环。
