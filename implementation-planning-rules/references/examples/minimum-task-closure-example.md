# 最小任务闭环示例

## 示例目标

演示如何把“一个实施周期”继续拆成“多个最小任务”，并按“实现 -> 测试 -> 审查 -> 验收”逐个闭环推进。

## 示例结构

- 实施周期：筛选能力改造
  - 周期目标：让用户可以按状态筛选列表结果
  - 周期完成标志：接口、查询链路和页面交互都支持状态筛选，且逐任务完成闭环

## 阶段划分

- 阶段 1：请求入口收口
  - 只做这一件事：补齐状态筛选参数入口
- 阶段 2：查询链路落地
  - 只做这一件事：让状态筛选真正参与查询
- 阶段 3：页面反馈收口
  - 只做这一件事：让用户看见并使用筛选结果

## 最小任务清单

### 最小任务 1：补状态筛选参数解析

- 所属周期：筛选能力改造
- 所属阶段：请求入口收口
- 本任务只做这一件事：补请求参数 `status` 的解析与校验
- 输入条件：接口已有列表查询入口
- 实现产出：请求模型、参数校验、错误映射补齐
- 测试点：合法状态值通过，非法状态值返回预期错误
- 审查点：未破坏无筛选条件下的原查询行为
- 验收点：调用方传入 `status` 时接口能稳定识别
- 完成定义：参数入口闭环成立，错误返回口径明确
- 阻断条件：状态枚举口径未冻结
- 下一任务依赖：只有本任务验收通过后，才能进入“查询条件拼装”

### 最小任务 2：补查询条件拼装

- 所属周期：筛选能力改造
- 所属阶段：查询链路落地
- 本任务只做这一件事：把 `status` 条件接入仓储查询
- 输入条件：最小任务 1 已验收通过
- 实现产出：查询条件拼装、仓储过滤逻辑补齐
- 测试点：不同状态值返回对应结果集
- 审查点：未影响原排序、分页与其他筛选条件
- 验收点：接口返回结果与筛选条件一致
- 完成定义：查询链路闭环成立
- 阻断条件：底层数据表没有可用状态字段
- 下一任务依赖：只有本任务验收通过后，才能进入“页面筛选交互”

### 最小任务 3：补页面筛选交互

- 所属周期：筛选能力改造
- 所属阶段：页面反馈收口
- 本任务只做这一件事：让页面支持状态筛选选择与结果展示
- 输入条件：最小任务 2 已验收通过
- 实现产出：筛选控件、请求透传、空状态反馈补齐
- 测试点：选择不同状态时页面结果同步变化
- 审查点：未破坏原列表加载、重置与空状态布局
- 验收点：用户可在页面上真实完成筛选
- 完成定义：页面交互闭环成立
- 阻断条件：前后端筛选枚举定义不一致
- 下一任务依赖：无；本任务完成后当前实施周期可收口

## 执行顺序示例

1. 最小任务 1 实现
2. 最小任务 1 测试
3. 最小任务 1 审查
4. 最小任务 1 验收
5. 最小任务 2 实现
6. 最小任务 2 测试
7. 最小任务 2 审查
8. 最小任务 2 验收
9. 最小任务 3 实现
10. 最小任务 3 测试
11. 最小任务 3 审查
12. 最小任务 3 验收

## 关键约束

- 未完成当前最小任务的验收前，不得跳到下一个最小任务。
- “实施周期”负责管理视图，“最小任务”负责真实执行。
- 如果某个最小任务同时包含多个并列目标，必须继续拆分。
