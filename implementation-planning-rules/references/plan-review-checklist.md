# 实施规划自审清单

## 覆盖度检查

- 若当前处于 Plan Mode、最终回复使用 `<proposed_plan>`、或输出正式计划 / 受限计划 / 阻断计划，是否已执行 `references/plan-output-gate.md`。
- 当前计划类型是否已明确归类为正式实施计划、受限计划或阻断计划。
- 当前计划类型对应的必填字段矩阵是否全部通过。
- 正文开头是否先给出“当前计划最终方案的简要说明”，而不是直接进入基本信息。
- 当前计划最终方案简要说明是否已经讲清推荐方案、主落点和选择原因。
- 当前优先闭环是否清楚。
- 是否已经先写出 agent 当前理解的问题 / 目标，而不是直接跳到阶段或任务。
- 是否已经写出本轮计划范围、明确不在范围和关键假设 / 待确认点。
- 规划阶段是否保持只读，没有把“先试做一点”混进计划过程。
- 是否已写出实施周期总览。
- 是否已明确当前计划为何可以独立成一个实施周期。
- 若任务还能拆成多个更小的子任务周期，是否已继续拆分，而不是硬并到一个周期里。
- 是否已写出阶段清单，且每个阶段只有一个清晰目标。
- 是否已写出最小任务清单。
- 每个最小任务是否只承载一个独立目标。
- 最小任务是否优先按“依赖图 + 垂直切片”组织，而不是按前端 / 后端 / 数据库水平分层堆叠。
- 每个最小任务是否已经标出预计触达文件数；若明显超过约 5 个文件，是否已继续拆分。
- 目录、模块、文件落点是否清楚。
- 若涉及新增文件，是否已使用目录树（`text` 代码块）列出新增文件，并在树节点后标注用途。
- 若涉及数据库新表或字段变更，是否已给出对应 SQL（`CREATE TABLE` / `ALTER TABLE`）。
- 是否已经写出任务顺序、依赖关系、周期/阶段归属和验证点。
- 是否已经为每个最小任务写出实现产出、测试点、审查点、验收点。
- 若涉及代码生成、修改或重构，“现状与落点”是否已经给出代码落点目录树，而不是只列文件名、模块名或普通条目。
- 涉及代码变更的最小任务，是否已经写出真实测试是否必需、测试入口、依赖环境、样本 / 数据来源和通过标准。
- 若标记“无需真实测试”，是否明确属于纯文档、纯注释、纯排版、纯静态资源改名 / 搬运或不会影响运行结果的说明整理，并已写明理由。
- 是否已经分别写出任务完成条件、任务停止 / 结束条件和最大推进边界，而不是混成一句空话。
- 是否已经说明主要风险、阻断项和待确认点。
- 是否已明确对应最终验收仍需等待测试与审查完成。
- 若当前是受限计划 / 阻断计划，是否已经明确写出“受限计划不得作为实施授权”，以及升级为正式执行计划前禁止进入编码、改码、重构、测试实施或其他执行动作。

## 占位词检查

- 不允许保留 `TBD`、`TODO`、`后续补充`、`实现时再看`。
- 不允许用“若干文件”“相关逻辑”“对应模块”等模糊说法代替具体落点。
- 不允许用“优化一下”“处理一下”代替实际执行动作。
- 不允许用“本阶段顺手一起做”“这个周期把几个功能都做完”这类混合目标表述。
- 不允许用“这个最小任务顺手再做一点”“一起补掉”把多个目标混进同一个最小任务。
- 不允许出现“后端先全做完”“前端统一补”“数据库最后一起改”这类水平分层推进表述，除非已明确说明无法切成垂直闭环的原因。

## 可执行性检查

- 是否存在 `Summary`、`Key Changes`、`Implementation Changes`、`Public Interfaces`、`Test Plan`、`Assumptions` 等通用工程计划壳；若存在，是否已判定为未通过并重写。
- 是否存在“基本信息先开头、没有当前计划最终方案简要说明”的旧结构；若存在，是否已判定为未通过并重写。
- 若当前是受限计划，是否明确写出禁止实施说明和升级为正式实施计划的条件。
- 若当前是阻断计划，是否明确写出阻断证据、恢复路径、不得执行事项和重新进入条件。
- 每一步是否能直接指导后续编码。
- 每一步完成后是否能形成可检查结果。
- 计划开头的问题理解是否足以约束后续阶段、任务和测试，不会出现“后面写得很细但前面理解错题”。
- 是否把 `build`、`lint`、静态检查、代码审查或人工阅读误当成真实测试。
- 每个实施周期结束后是否有明确的任务停止 / 结束条件，而不是依赖“下一步建议”继续扩散。
- 顺序是否符合依赖关系，而不是纯口头顺序。
- 是否存在一上来就把改动面铺太大的问题。
- 每个阶段是否真的只做一件事。
- 每个步骤是否已经标明所属开发周期与所属阶段。
- 每个步骤是否已经标明对应最小任务。
- 是否已经说明每个最小任务为什么能形成端到端最小闭环，而不是模块清单。
- 当前最小任务未完成实现、真实测试、审查、验收前，是否明确禁止进入下一最小任务。
- 每个最小任务完成后是否都有可独立检查的闭环结果。
- 代码变更类计划是否已经把落点明确到目录树层级，而不是只写“改这里”“修一下”。
- 周期与阶段之间是否能独立交接，而不是必须依赖大段口头上下文。
- 命中文件新增或数据库变更场景时，缺目录树或缺 SQL 必须判定为未通过；若仍使用普通条目描述目录信息同样判定未通过。

## 交接性检查

- 其他人接手时，是否能根据这份计划知道先改什么、改哪里、怎么验。
- 其他人接手时，是否能明确当前做到哪个周期、哪个阶段、哪个最小任务、下一步承接什么。
- 后续编码审查、测试和交付是否能引用这份计划收口。
- 如计划被需求变更推翻，是否容易在同一份文档里更新差异。
