# 实施规划自审清单

## 覆盖度检查

- 当前优先闭环是否清楚。
- 是否已写出实施周期总览。
- 是否已明确当前计划为何可以独立成一个实施周期。
- 若任务还能拆成多个更小的子任务周期，是否已继续拆分，而不是硬并到一个周期里。
- 是否已写出阶段清单，且每个阶段只有一个清晰目标。
- 目录、模块、文件落点是否清楚。
- 若涉及新增文件，是否已使用目录树（`text` 代码块）列出新增文件，并在树节点后标注用途。
- 若涉及数据库新表或字段变更，是否已给出对应 SQL（`CREATE TABLE` / `ALTER TABLE`）。
- 是否已经写出任务顺序、依赖关系、周期/阶段归属和验证点。
- 是否已经写出完成定义、停止条件和最大推进边界。
- 是否已经说明主要风险、阻断项和待确认点。
- 是否已明确对应最终验收仍需等待测试与审核完成。

## 占位词检查

- 不允许保留 `TBD`、`TODO`、`后续补充`、`实现时再看`。
- 不允许用“若干文件”“相关逻辑”“对应模块”等模糊说法代替具体落点。
- 不允许用“优化一下”“处理一下”代替实际执行动作。
- 不允许用“本阶段顺手一起做”“这个周期把几个功能都做完”这类混合目标表述。

## 可执行性检查

- 每一步是否能直接指导后续编码。
- 每一步完成后是否能形成可检查结果。
- 每个实施周期结束后是否有明确停止条件，而不是依赖“下一步建议”继续扩散。
- 顺序是否符合依赖关系，而不是纯口头顺序。
- 是否存在一上来就把改动面铺太大的问题。
- 每个阶段是否真的只做一件事。
- 每个步骤是否已经标明所属开发周期与所属阶段。
- 周期与阶段之间是否能独立交接，而不是必须依赖大段口头上下文。
- 命中文件新增或数据库变更场景时，缺目录树或缺 SQL 必须判定为未通过；若仍使用普通条目描述目录信息同样判定未通过。

## 交接性检查

- 其他人接手时，是否能根据这份计划知道先改什么、改哪里、怎么验。
- 其他人接手时，是否能明确当前做到哪个周期、哪个阶段、下一步承接什么。
- 后续编码审查、测试和交付是否能引用这份计划收口。
- 如计划被需求变更推翻，是否容易在同一份文档里更新差异。
