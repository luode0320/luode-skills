# 需求与实施计划全量顺序实施方案

结论：本文件统一多来源对象的实施先后顺序；影响：项目成员可以据此判断当前该做什么以及不能跳过什么；范围：覆盖需求、验收、实施总览、实施周期和最小任务的总排序；非范围：不替代单个需求、周期或任务的具体执行说明；变化：先说明排序带来的业务结果，再在附录保留执行和追踪细节；完成标准：每个当前入口都能回指其依赖、收口条件和证据；术语说明：无技术术语需要解释；验证状态：计划阶段按本地实际执行结果确认。

## 方案说明

一、触发场景

- 新项目启动时已经存在多份需求文档、多个需求子项或多份实施计划。
- 项目初期同时创建了多份实施总览 / 实施周期，后续执行需要先决定第一期、第二期、第三期等总顺序。
- 用户要求“实施顺序总表”“需求与实施计划全量顺序实施方案”“全量顺序实施方案”“项目整体实施顺序”。
- 需求变更、拆分或新增来源对象影响跨需求依赖或整体实施顺序。

二、职责边界

- 全量顺序实施方案负责跨来源对象排序，不替代需求主文档、验收标准、实施总览、实施周期或最终验收。
- 实施总览负责单个来源对象的实施周期和最小任务，不负责项目级多个来源对象的总排序。
- 实施周期负责第一期 / 第二期 / 第三期等大进度边界，最小任务负责真实执行闭环。
- 当前总表中的任何最小任务摘要，都必须能回指到具体实施总览或实施周期；不能只在总表里孤立存在。

三、建议字段

- 基本信息：项目 / 来源集合标识、总表命名主干、适用项目阶段、维护状态。
- 来源对象清单：需求或 Bug 标识、需求文档、验收标准、实施总览、相关实施周期。
- 全量执行顺序：总序号、来源对象、所属项目期次、对应实施周期、周期内最小任务摘要、前置依赖、进入条件、收口条件。
- 当前执行入口：当前应执行的周期、最小任务、下一步只做什么、不得跳过的前置闭环。
- 状态与证据：未开始 / 进行中 / 已完成 / 阻断，已完成项需记录实现、真实测试、审查、验收证据入口。
- 变更记录：新增需求、顺序调整、依赖变化、阻断解除和重新排序原因。

四、执行约束

- 执行必须先按全量顺序总表定位当前来源对象和实施周期，再进入对应实施总览 / 实施周期。
- 当前总表所指向的实施周期未收口前，不得跳到后续来源对象或后续周期。
- 周期内仍按最小任务逐个完成“实现 -> 真实测试 -> 审查 -> 验收”，不得把多个任务集中到周期末统一检查。
- 若总表与单个实施总览 / 实施周期冲突，以最新确认的需求、验收标准和当前代码事实为准，并同步回开总表与受影响实施文档。

## 执行附录

- local 环境、总表维护步骤、精确命令、样本、日志、SQL 和接口报文：填写调整总顺序或验证当前入口时的执行细节。
- 回滚与清理：记录顺序调整失败后的恢复步骤，以及恢复后需要重新核对的项目。

## 追踪附录

- 稳定 ID、来源对象、证据、跨文档追踪矩阵和机器可读定位：记录总表与需求、验收、实施总览、实施周期及最小任务之间的回指。
- 附录维护规则：执行附录与追踪附录按此顺序连续置于文档末尾，禁止在其后追加正文。
