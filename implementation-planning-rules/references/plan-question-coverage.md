# Plan Mode 决策维度提问与无选择处理协议

> 归属 owner：`implementation-planning-rules`。本文件是条件路由细则，不产生独立自动触发入口。用于 Plan Mode 下按维度识别实现层不确定决策点、向用户多选提问，并在用户未选择时保持未决而绝不默认推进。

## 适用场景

- 当前处于 Plan Mode，且已进入实施规划阶段（需求、边界、验收前置已基本稳定）。
- 需要在正式 `<proposed_plan>` 之前，把实现层所有真实不确定的技术决策点交给用户选择。
- 与需求 / 边界 / 验收前置缺口无关、纯属实现层的技术取舍，集中在本协议处理。

## 决策维度清单

按语义遍历以下维度，判断每个维度当前是否存在“需要用户拍板”的取舍（维度不限于此表，可按任务补充）：

1. 架构 / 技术路线
2. 代码落点（目录 / 包 / 文件位置）
3. 实现方式（存在 2-3 个都合理的方案时的比选）
4. 命名（变量 / 函数 / 类型 / 接口 / 字段口径）
5. 注释（粒度 / 语言 / 风格）
6. 日志（埋点位置 / 级别 / 字段 / 格式）
7. 错误处理与异常策略
8. 数据模型 / 表结构 / 字段
9. 接口契约（请求参数 / 响应结构 / 分页 / 错误码）
10. 依赖与库（复用现有 vs 新增第三方）
11. 测试策略 / 范围 / 样本来源
12. 配置 / 开关 / 环境差异
13. 兼容性 / 迁移 / 版本
14. 性能 / 并发取舍
15. 安全 / 权限
16. 回滚 / 灰度 / 发布方式

## 三态分类

对每个维度先做只读探索，再归入一种状态：

- `已确定`：仓库已有明确约定、既有实现或需求已冻结，可直接采用；必须写出依据（文件 / 约定 / 需求条目），不向用户提问。
- `真实不确定且影响大`：存在多个都合理但影响明显不同的取舍，且无法由探索得出唯一答案；必须向用户弹窗多选。
- `不适用`：本任务不涉及该维度；记 `N/A + 原因`。

禁止就“可由探索得到的事实”提问（如某结构在哪、用了什么框架）；这类先探索，不占用用户决策名额。

## 多选提问协议

- 一律使用 `request_user_input`，禁止在正文里写多选题让用户“回文本选”。
- 每个问题给 2-4 个互斥选项，推荐项置于首位并在标签后标注“(推荐)”；禁止填充明显错误或无意义的凑数选项。
- 决策型提问一律阻塞：禁止使用 `autoResolutionMs`（自动放行 / 到时按判断继续）；`autoResolutionMs` 仅可用于纯信息性、非决策的可选补充。
- 每次调用最多 3 问；允许多个决策点同时处于挂起状态，用户可按自身节奏先处理其它问题、稍后回来选。
- 多轮迭代提问，直到所有 `真实不确定` 维度都被用户明确选定。
- 广度策略为“维度清单驱动”：凡真实不确定的维度都要逐一弹窗，不设硬性数量上限，也不为凑数而问。

## 无选择 = 未决协议（核心，强制）

当已就某决策点向用户弹窗、但用户未做出选择时：

- 保持该弹窗 / 决策点开放，不关闭、不撤回、不重复骚扰式追问。
- 严禁采纳推荐项，严禁把推荐项写成“默认假设”后继续，严禁以“用户没选就按最优先项推进”为由推进。
- 将该决策点标记为 `未决(待用户选择)`，并列入“待用户选择清单”。
- 挂起所有依赖该决策的计划分支与正式 `<proposed_plan>`：只要存在未决决策点，就不得输出正式实施计划，只能输出受限 / 挂起说明。
- 允许继续做不依赖该决策的只读探索，或就其它维度继续提问；用户完成其它问题后可随时回来选择本项。
- 仅当用户明确表达“你来定 / 按推荐 / 随便”等授权时，才可采用推荐项，并记录该授权来源；缺少此明确授权不得自行默认。

## 决策维度覆盖表 + 待用户选择清单

在计划正文中维护一张覆盖表，逐维度登记状态：

| 维度 | 状态 | 结论 / 依据 |
| --- | --- | --- |
| 架构 | 已确定 / 已弹窗且用户已选 / 未决待选 / N/A | 写依据、用户所选项，或未决原因 |
| ...（其余相关维度） | ... | ... |

- `已确定`：写明依据来源。
- `已弹窗且用户已选`：写明用户所选项。
- `未决待选`：同时登记到“待用户选择清单”，并说明它挂起了哪些下游内容。
- `N/A`：写原因。

“待用户选择清单”单独成节，列出所有 `未决待选` 决策点、各自选项、以及被挂起的下游内容；只要该清单非空，正式计划一律不放行。

## 相邻域边界

- 需求语义缺口、业务规则、输入输出定义 -> `requirement-intake-rules` / `requirement-change-rules`。
- 是否属于本次范围、兼容边界 -> `requirement-boundary-rules`。
- 拆分优先级 -> `requirement-splitting-rules`。
- 完成标准口径 -> `acceptance-criteria-rules`。
- 本协议只覆盖“进入实施规划后、实现层面的技术决策点”，不替代上述需求 / 边界 / 验收前置域。
