# 实施文档结构模板

## 建议结构

如果当前运行环境要求把最终计划放进 `<proposed_plan>` 或其他专用计划包裹中，处理方式如下：

- 外层包裹只负责渲染或协议，不改变本模板的正文结构要求。
- 可以在最终回复里对小节做合并压缩，但不能省略本模板要求的关键字段。
- 优先保证“当前计划最终方案的简要说明、agent 对当前问题的理解、本轮范围、非范围、当前优先闭环、关键假设 / 待确认点、实施周期、最小任务、真实测试安排、任务完成条件、任务停止 / 结束条件、最大推进边界”这些信息完整。
- 输出前必须同时执行 [plan-output-gate.md](plan-output-gate.md) 的字段矩阵；矩阵未通过时不得输出 `<proposed_plan>`。
- 硬失败判定：若正文主结构是 `Summary`、`Key Changes`、`Public Interfaces`、`Test Plan`、`Assumptions`，或其他只描述变更概要 / 接口 / 测试摘要的通用计划壳，视为没有使用本模板，必须立即重写为下方结构。

## 结构要求

- 正文开头必须先写“当前计划最终方案的简要说明”，用 1-3 句或等价短 bullet 先交代推荐方案、主落点和为什么这么做；随后再写清 agent 当前理解的问题 / 目标、本轮范围、非范围、当前优先闭环和关键假设 / 待确认点，再展开实施周期与任务拆分；不要把整张大路线图塞成一份笼统计划。
- Plan Mode 或 `<proposed_plan>` 只能作为外层协议；正文缺少“当前计划最终方案的简要说明、agent 当前理解、范围、非范围、当前优先闭环、关键假设 / 待确认点、实施周期、阶段计划、最小任务、真实测试、完成条件、停止 / 结束条件、最大推进边界”任一核心字段时，判定模板未正确使用，必须补齐后再输出。
- 正式计划、受限计划和阻断计划分别按 [plan-output-gate.md](plan-output-gate.md) 的必填字段矩阵判定；不能用受限计划冒充正式实施计划。
- “实施周期总览”是必填项，必须说明每个周期作为第一期、第二期、第三期等大进度单位的定位、周期顺序、进入条件、收口条件，以及为什么当前计划可以独立成一个周期；若还能继续拆小，就继续拆到更清晰的子任务周期。
- “阶段计划”是必填项，每个阶段只能承载一个清晰目标，并写清输入、输出和验证门槛。
- “最小任务清单”是强制章节；周期是大进度与顺序边界，真正执行单位是周期内的“最小任务”，不是把整个实施周期或阶段当成一次执行。
- 执行顺序固定为“周期 01 -> 周期内最小任务逐个完成实现、真实测试、审查、验收 -> 周期收口 -> 周期 02”；当前周期未收口前，不得进入后续周期。
- 每个最小任务必须且只能归属一个实施周期，并写清周期内顺序；禁止把最小任务列成全局平铺清单，或让一个最小任务同时跨多个周期。
- 每个最小任务都必须具备实现、真实测试、审查、验收四类闭环信息，以及显式的任务完成条件与任务停止 / 结束条件；缺其中任一项都视为计划未收口；若无需真实测试，必须显式写明理由。
- 禁止把多个最小任务先连续实现，再在周期末统一测试、统一审查或统一验收；这种写法视为没有按最小任务闭环。
- 每个最小任务只承载一个目标；若一个任务同时包含多个并列产出、多个验证目标或多个验收目标，必须继续拆分。
- 每个最小任务都应写清“垂直切片目标”和“预计触达文件数”；若明显超过约 5 个文件，默认视为仍需继续拆分。
- 涉及生成代码、修改代码、重构代码或其他可执行行为变更的最小任务，必须额外写清真实测试入口、依赖环境、样本 / 数据来源和通过标准；`build`、`lint`、静态检查、代码审查都不能代替真实测试。
- 当前最小任务未完成实现、真实测试、审查、验收并达到任务完成条件前，不得进入下一个最小任务。
- “现状与落点”必须明确到目录、模块或文件层级；若涉及代码生成、修改或重构，还必须给出代码落点目录树，不能只写“后端处理一下”“前端改一下”。
- 若涉及新增文件，必须使用目录树（`text` 代码块）列出新增文件，并在树节点后用 `#` 标注用途；禁止普通条目描述。
- 若涉及数据库新表或字段变更，必须补齐可执行 SQL（`CREATE TABLE` / `ALTER TABLE`），不能只写自然语言。
- “方案选择”只在确实存在明显不同路线时出现；没有分歧时可省略冗余方案节。
- “实施步骤”按周期顺序和周期内最小任务顺序写，不要按想到什么写什么；每一步都必须标明所属周期、所属阶段和对应最小任务。
- 一个步骤只做一件事；若一步骤同时描述多个并列动作，必须继续拆分。
- 若任务顺序仍按前端 / 后端 / 数据库分层罗列，而没有映射到垂直切片最小任务，视为模板未正确使用。
- “真实测试安排”必须能指导后续编码审查、功能验证、回归判断与最终验收准备。
- “真实测试安排”是必填项，必须独立成章，不得只散落在最小任务字段、验证点或备注里。
- 对复杂任务建议补图，图中步骤命名应与“实施步骤”保持一致，避免图文错位。
- “风险与阻断项”必须写真实阻断，不要写空话。

## 周期子文档补充

当任务已拆出 `来源对象标识_实施周期NN_中文主题` 子文档时，周期文档至少包含以下内容：

- 对应需求文档
- 来源对象标识（需求或 Bug）
- 对应实施总览
- 周期文档命名主干（必须包含来源对象标识）
- 当前周期序号 / 大进度定位（第一期 / 第二期 / 第三期等）
- 当前周期目标
- 当前周期只做这一件事
- 当前周期进入条件
- 当前周期收口条件
- 当前周期最小任务清单
- 当前周期内最小任务执行顺序
- 每个最小任务的闭环状态或计划闭环顺序（实现 / 真实测试 / 审查 / 验收）
- 当前周期步骤顺序
- 当前周期验证点
- 当前周期阻断项
- 若本轮是在“补充需求、实施文档”或“补充已有实施文档”，必须明确写出本周期为何受本次变更影响；未受影响的周期不改正文，但应在实施总览中标明“本轮未变更”。
- 不允许只在实施总览补一段概述就跳过受影响的周期子文档；只要周期目标、步骤、验证点、阻断项或闭环范围发生变化，就必须同步更新该周期文档。
- 若当前变更已经形成新的独立闭环、而现有周期文档无法承载，必须新增对应的周期子文档，并在实施总览里补齐索引与衔接关系。
