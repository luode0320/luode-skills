## 结构要求

- 必须先写清 agent 当前理解的问题 / 目标、本轮范围、非范围、当前优先闭环和关键假设 / 待确认点，再展开实施周期与任务拆分；不要把整张大路线图塞成一份笼统计划。
- “实施周期总览”是必填项，必须说明为什么当前计划可以独立成一个周期；若还能继续拆小，就继续拆到更清晰的子任务周期。
- “阶段计划”是必填项，每个阶段只能承载一个清晰目标，并写清输入、输出和验证门槛。
- “最小任务清单”是强制章节；真正执行单位是“最小任务”，不是“实施周期”或“阶段”。
- 每个最小任务都必须具备实现、真实测试、审查、验收四类闭环信息，以及显式的任务完成条件与任务停止 / 结束条件；缺其中任一项都视为计划未收口；若无需真实测试，必须显式写明理由。
- 每个最小任务只承载一个目标；若一个任务同时包含多个并列产出、多个验证目标或多个验收目标，必须继续拆分。
- 每个最小任务都应写清“垂直切片目标”和“预计触达文件数”；若明显超过约 5 个文件，默认视为仍需继续拆分。
- 涉及生成代码、修改代码、重构代码或其他可执行行为变更的最小任务，必须额外写清真实测试入口、依赖环境、样本 / 数据来源和通过标准；`build`、`lint`、静态检查、代码审查都不能代替真实测试。
- 当前最小任务未完成实现、真实测试、审查、验收并达到任务完成条件前，不得进入下一个最小任务。
- “现状与落点”必须明确到目录、模块或文件层级；若涉及代码生成、修改或重构，还必须给出代码落点目录树，不能只写“后端处理一下”“前端改一下”。
- 若涉及新增文件，必须使用目录树（`text` 代码块）列出新增文件，并在树节点后用 `#` 标注用途；禁止普通条目描述。
- 若涉及数据库新表或字段变更，必须补齐可执行 SQL（`CREATE TABLE` / `ALTER TABLE`），不能只写自然语言。
- “方案选择”只在确实存在明显不同路线时出现；没有分歧时可省略冗余方案节。
- “实施步骤”按执行顺序写，不要按想到什么写什么；每一步都必须标明所属周期、所属阶段和对应最小任务。
- 一个步骤只做一件事；若一步骤同时描述多个并列动作，必须继续拆分。
- 若任务顺序仍按前端 / 后端 / 数据库分层罗列，而没有映射到垂直切片最小任务，视为模板未正确使用。
- “每步验证点”必须能指导后续编码审查、功能验证、回归判断与最终验收准备。
- 对复杂任务建议补图，图中步骤命名应与“实施步骤”保持一致，避免图文错位。
- “风险与阻断项”必须写真实阻断，不要写空话。

## 周期子文档补充

当任务已拆出 `来源对象标识_实施周期NN_中文主题` 子文档时，周期文档至少包含以下内容：

- 对应需求文档
- 来源对象标识（需求或 Bug）
- 对应实施总览
- 周期文档命名主干（必须包含来源对象标识）
- 当前周期目标
- 当前周期只做这一件事
- 当前周期最小任务清单
- 当前周期步骤顺序
- 当前周期验证点
- 当前周期阻断项
