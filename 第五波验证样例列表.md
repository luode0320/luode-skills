# 第五波验证样例列表

## 用途

用于验证第五波已落地的编码审查域与测试域 skill 是否能够按预期分流，不发生职责混淆。

## 样例 1：实现质量不达标，必须拦在编码审查入口

- 场景：功能代码已写完，但命名混乱、注释缺失、明显偏离最小改动原则。
- 预期命中：`implementation-review-rules`
- 预期流转路径：编码完成 -> `implementation-review-rules` -> 驳回 -> 回流编码阶段
- 验证重点：不能因为“功能看起来能跑”就直接进入测试域。

## 样例 2：语法与格式问题必须在编码审查域内部收口

- 场景：代码实现目标基本正确，但存在语法错误、引用缺失、未使用导入、调试残留、多余换行。
- 预期命中：`syntax-check-review-rules`、`cleanup-format-review-rules`
- 预期流转路径：`implementation-review-rules` 通过 -> `syntax-check-review-rules` / `cleanup-format-review-rules` -> 修正后再进入测试域
- 验证重点：语法和格式问题不能漂移到功能验证或回归验证阶段。

## 样例 3：测试资源位置与命名问题必须由测试资源管理链接管

- 场景：新增测试目录、测试文件、测试脚本、测试数据目录，但目录落点错误或命名混乱。
- 预期命中：`test-location-rules`、`test-naming-rules`
- 预期流转路径：编码审查通过 -> `test-location-rules` -> `test-naming-rules` -> 进入 `test-program-rules` / `test-doc-rules`
- 验证重点：资源组织问题不应被错误判定成“功能验证失败”。

## 样例 4：当前需求功能正确性问题应命中功能验证

- 场景：当前接口返回字段不符合需求、页面交互结果与验收标准不一致、输入输出结果错误。
- 预期命中：`functional-validation-rules`
- 预期流转路径：测试资源准备完成 -> `functional-validation-rules` -> 驳回 -> 回流编码域或需求域
- 验证重点：只回答“这次改动是否做对了”，不扩大成回归范围。

## 样例 5：Bug 修复后的兼容性问题应命中回归验证

- 场景：某个 Bug 已修复，当前修复路径验证通过，但担心共享逻辑、旧入口、关联链路被带坏。
- 预期命中：`test-regression-rules`
- 预期流转路径：`functional-validation-rules` 通过 -> `test-regression-rules` -> 通过 / 驳回 -> 回流 Bug 域或编码域
- 验证重点：回归验证只回答“有没有带坏别的地方”。

## 样例 6：编码审查通过但功能验证失败，应回流实现或 Bug 修复

- 场景：代码风格、语法、格式、目录归位全部通过，但运行后发现当前需求逻辑仍不正确。
- 预期命中：`functional-validation-rules`
- 预期流转路径：编码审查域通过 -> 测试资源管理链通过 -> `functional-validation-rules` 驳回 -> 回流编码域或 Bug 域
- 验证重点：不能停留在测试资源管理链内“假通过”。

## 总体验证结论

- 第五波应形成明确链路：编码完成 -> 编码审查域 -> 测试资源管理链 -> 功能验证 -> 回归验证。
- 只要上述 6 类场景都能稳定分流，第五波核心职责边界就算跑通。
