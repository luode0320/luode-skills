# 项目级工具入口约定

## 目标

当 `mcp-installation-rules` 已经判断“当前项目后续应由哪条 CLI 工具链接管”后，不再默认补齐项目级 Codex MCP 配置。

当前默认策略是：

- 浏览器任务走 `agent-browser`
- Godot 任务走本地 `godot4` / `godot` 或项目自带启动脚本
- 旧 MCP 配置只做兼容读取，不再作为默认补齐目标

## 检查路径顺序

只有在以下场景，才需要查看项目级配置文件：

- 用户明确要求兼容旧 MCP 方案
- 仓库里已经存在 `./codex/config.toml` 或 `./.codex/config.toml`，需要确认是否还残留旧接管配置
- 需要解释“为什么当前不再建议继续补齐 MCP”

可检查路径：

1. `./codex/config.toml`
2. `./.codex/config.toml`

处理原则：

- 默认不创建新的项目级 MCP 配置文件。
- 默认不因为历史存在旧 MCP 配置，就继续扩写或补齐它。
- 若仓库里没有这些配置文件，这本身不是问题，不需要为了 CLI 路线额外创建。

## 默认动作

当项目已命中某类工具接管标记时：

- 命中前端项目标记：直接声明浏览器任务由 `agent-browser` 接管。
- 命中 Godot 项目标记：直接声明 Godot 任务由本地 Godot CLI / 编辑器命令行接管。
- 两类都命中：分别声明，不互相覆盖。

## 历史配置处理要求

- 若项目级配置中已经存在旧 MCP 配置，可记录“历史兼容存在”，但不要自动删除。
- 若已存在旧配置，也不得重复追加第二份同类配置。
- 若用户没有明确要求继续维护旧 MCP，不要再修改这些配置。
- 补齐动作默认不再执行；当前标准行为是停止新增 MCP 配置，而不是继续口头建议或自动落盘。

## 阻断与回退

- 配置缺失不再是阻断项，因为当前默认路线不依赖项目级 MCP 配置。
- 若 `agent-browser` 或本地 Godot CLI 缺失，阻断点应写在工具可执行入口层，而不是转回“补一个 MCP 配置试试”。
- 若无法安全判断现有 TOML 结构，只需说明“发现旧配置但本轮不再扩写”，不要做破坏性覆盖。

## 输出要求

最终结论至少要明确：

- 识别到了哪类工具接管需求
- 是否检查了旧项目级配置路径
- 是否发现历史 MCP 配置
- 本轮是否保持只读兼容
- 后续由哪条 CLI 工具链优先接管
