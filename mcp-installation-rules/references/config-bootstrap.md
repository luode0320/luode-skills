# Codex MCP 配置补齐规则

## 目标

当 `mcp-installation-rules` 已经判断“当前项目需要某个 MCP”后，不只停留在口头建议，还要继续检查项目级 Codex 配置文件是否已声明对应 MCP；若缺失，则默认补齐。

## 检查路径顺序

按以下顺序检查项目级配置文件：

1. `./codex/config.toml`
2. `./.codex/config.toml`

处理原则：

- 若第 1 个存在，优先在该文件内检查和补齐。
- 若第 1 个不存在但第 2 个存在，则在第 2 个文件内检查和补齐。
- 若两个都不存在，默认创建 `./.codex/config.toml` 作为项目级 Codex 配置入口。
- 不要因为用户主目录里已有全局 `~/.codex/config.toml`，就跳过项目级检查；项目级配置仍应独立补齐。

## 默认补齐动作

当项目已命中某类 MCP 标记时：

- 命中前端项目标记：检查是否已存在 `Chrome DevTools MCP` 对应配置；缺失则默认补齐。
- 命中 Godot 项目标记：检查是否已存在 `Godot AI MCP` 对应配置；缺失则默认补齐。
- 两类都命中：分别检查，缺哪个补哪个。

## 配置补齐要求

- 优先增量补齐，不覆盖用户已有配置。
- 已存在同名 MCP 配置时，不得重复追加第二份。
- 若配置文件中已有其他 `mcp_servers.*` 段落，应保持现有顺序和风格，仅追加缺失段。
- 若用户项目中已经采用不同但可明确映射到同一 MCP 的命名，先做名称归一，再决定是否真的缺失。
- 补齐动作默认不需要用户确认；这是 `mcp-installation-rules` 的标准执行行为。

## 阻断与回退

- 配置缺失本身不再只是“提醒项”，而是默认执行补齐项。
- 若配置已补齐，但本机运行环境仍未安装对应 MCP 可执行入口，才进入“安装未完成”的阻断路径。
- 若无法安全判断现有 TOML 结构，或用户已经有复杂的自定义注释 / 模板 / include 机制，先保守停下并说明原因，不做破坏性覆盖。

## 输出要求

最终结论至少要明确：

- 识别到了哪个 MCP 需求
- 检查了哪个项目级配置路径
- 配置原本是否缺失
- 是否已默认补齐
- 补齐后后续由哪个 MCP 优先接管

## Claude Code 配置探测（待确认，新增）

以上全部路径（`./codex/config.toml`、`./.codex/config.toml`）与 TOML `mcp_servers.*` schema 均为 Codex CLI 专属机制，不适用于 Claude Code。

当前运行环境为 Claude Code 时：

- 项目级 MCP 配置的具体机制（是否为根目录 `.mcp.json`、是否存在等价的补齐命令）**本次未实测验证**，不得直接照搬本文件描述的 `./codex/config.toml` 检查顺序或 TOML 字段结构。
- 在核实具体机制前，遇到 Claude Code 环境下的 MCP 缺口判断，只能：
  1. 明确告知用户"当前项目可能缺少 <MCP 名称> 配置，但 Claude Code 的具体配置文件/命令尚未核实"；
  2. 引导用户查阅当前 Claude Code 版本的官方文档或帮助信息确认配置方式；
  3. 待核实后再按"检测缺失 → 增量补齐 → 不覆盖已有配置"的思路（与 Codex 分支同一原则）执行，但具体文件路径与字段名称需以核实结果为准。
- 不得在未核实的情况下默认创建任何猜测出的 Claude Code 配置文件，避免写入无效或错误位置。
