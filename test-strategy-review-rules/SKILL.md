# 测试策略审查镜像规则

## 目标

只检查当前测试策略是否满足 `test-strategy-rules` 的要求，不执行策略落文和测试分流。

## 源规则

以下判定严格沿用源 skill：

- 选择合适的测试类型和优先级
- 平衡功能验证、联调验证、回归验证和资源限制
- 明确必须覆盖路径与待补测风险
- 策略应与时间戳测试目录结构对齐
- 禁止为了测试目的污染生产代码
- 前后端联调阶段应使用 `agent-browser`

## 自动触发信号

- 编码审查通过后准备进入测试，但测试重点还没收口。
- 变更涉及多链路、多风险层级或多测试方式。
- 时间、环境或资源有限，需要先核对测试优先级。
- 团队对功能验证、联调验证和回归验证优先存在争议。
- 主线程已声明使用 `test-strategy-rules`，但策略收口风险较高。

## 进入后先做什么

1. 先确认任务类型、改动范围和主要风险来源是否已说明。
2. 检查策略是否已明确哪些测试类型必须出现。
3. 检查哪些路径必须覆盖、哪些只能待补测。
4. 检查是否已给出时间戳测试根目录拆分方案。
5. 检查后续分流目标是否明确。

## 默认执行流程

1. 先读取 `test-strategy-rules`。
2. 需要核对测试策略维度时，再读源 skill 的 `references/strategy-dimensions.md`。
3. 需要核对优先级模型时，再读源 skill 的 `references/priority-model.md`。
4. 需要核对策略模板或正反例时，再读源 skill 的 `references/strategy-template.md`。
5. 输出策略通过项、覆盖遗漏、优先级问题、目录落点缺口和测试污染风险。
6. 不直接写 README，只输出问题和建议。

## 权责边界与不负责事项

- 不代替 `functional-validation-rules`、`test-regression-rules` 或 `agent-browser` 执行验证。
- 不代替测试资源类 skill 管理目录、命名、测试程序或测试文档。
- 不把所有路径都判成必须全测。
- 不在需求或修复目标尚未收敛时强做策略结论。
- 不执行测试任务主说明回写。

## 需要暂停并确认的条件

- 需求目标、修复目标或影响范围还没稳定。
- 关键测试环境、数据或上下游条件不具备。
- 团队对主风险来源判断分歧过大。
- 当前策略试图覆盖全量测试却没有匹配资源。

## 执行通过 / 驳回标准

- 通过：测试类型、优先级、覆盖范围、待补测风险和测试目录拆分方案完整，且未引入测试污染思路。
- 驳回：策略仍停留在“都测一下”，没有优先级、没有范围收口、没有时间戳目录对应，或包含通过污染生产代码降低测试成本的做法。

## 执行结果归档要求

- 如本轮审查发现策略覆盖、优先级或目录落点缺口，应将结论记录到 `analysis/` 或评审记录。
- 归档内容至少包含策略对象、发现的问题、优先级争议和建议流转方向。
- 如果只是一次轻量预审且无争议，可不单独归档，但应保证评审可追溯。

## 读取规则

- 默认先读 `test-strategy-rules`。
- 只有在核对策略维度时，再读 `references/strategy-dimensions.md`。
- 只有在核对测试优先级和覆盖收口时，再读 `references/priority-model.md`。
- 只有在核对模板和正反例时，再读 `references/strategy-template.md`。
