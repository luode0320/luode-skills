# 工作报告汇总规则

只在需要按项目汇总 Git 提交，并在必要时补充未提交进行中事项，输出年报/月报/周报/日报时使用这个 skill。
如果当前任务不是“按时间范围汇总提交”，不要触发。

## Skill 作用与适用场景

- 基于预配置项目清单批量统计提交记录，避免逐仓库手工整理。
- 从每个项目当前工作区的代码 diff 中补充“正在进行中的任务”，优先总结真实变更内容和所属模块；设计文件、设计目录和文档文件不得作为任务语义来源。
- 输出统一格式的日报、周报、月报、年报。
- 报告默认按项目分组，按提交事实输出“报告内容点”。
- 未提交事项必须显式标注为 `进行中`，不能冒充已提交完成项。
- 标题或时间范围必须带“日期 + 星期”。

## 自动触发信号

- 用户明确说“生成日报/周报/月报/年报”。
- 用户明确说“汇总今天/本周/本月/今年做了什么”。
- 用户要求“根据多个项目 git 提交生成阶段性工作总结”。

## 进入后先做什么

1. 识别报告类型：`daily` / `weekly` / `monthly` / `yearly`。
2. 确认基准日期：
   - 用户给日期就按用户日期。
   - 用户未给日期，默认使用系统当天日期。
3. 读取 `references/projects.json`，确认项目路径、项目名、作者过滤配置、低价值提交过滤配置、未提交工作区扫描配置、设计/文档排除配置和报告保存路径配置。
4. 运行 `scripts/generate_git_report.py` 生成报告草稿。
5. 输出给用户前做一次清洗：补充空项目说明，检查日期与星期格式，并确认未提交事项都已明确标注为 `进行中`。

## 默认执行流程

1. 默认先读 `references/projects.json`。
2. 执行脚本生成报告：
   - 日报：`python3 scripts/generate_git_report.py --period daily --date 2026-04-03`
   - 周报：`python3 scripts/generate_git_report.py --period weekly --date 2026-04-03`
   - 月报：`python3 scripts/generate_git_report.py --period monthly --date 2026-04-03`
   - 年报：`python3 scripts/generate_git_report.py --period yearly --date 2026-04-03`
3. 如需按特定作者过滤，可加 `--author`（可重复）覆盖配置文件中的作者列表；若配置和参数都为空，脚本会自动用全局 `git config user.name/user.email` 识别“我方提交”。
4. 若配置开启未提交工作区扫描，脚本必须读取各项目 `git status --short` 和 `git diff HEAD`；先分析代码文件的 diff 内容，再把真实存在的未提交事项补充到项目明细中。
5. 将脚本输出直接作为报告正文，并确认报告已保存到目标目录。

## 报告输出要求

- 顶部必须包含时间范围，使用“日期 + 星期”格式。
- 报告来源默认包含两类：
  - 已提交事项：来自指定时间范围内、作者精确匹配的 Git 提交。
  - 进行中事项：来自项目当前工作区真实存在的未提交改动，必须显式标注为 `进行中`。
- 周报/月报/年报：时间范围之后必须先输出 `总体归纳统计`，格式为：
  - `当前开发待发布版本总体进度(98%):`
  - `项目名: 未完成事项 -> 项目完成度: 98%`
  再输出 `各项目明细`。
- 日报：不输出 `总体归纳统计`，直接输出 `各项目明细`。
- `总体归纳统计` 必须是待发布进度口径，不要输出“完成X项/修复X项”这类数量统计口径。
- 无提交记录的项目不需要出现在 `总体归纳统计` 与 `各项目明细` 中。
- 每个项目固定结构：
  - `项目名:`
  - `报告内容点:`
- `报告内容点` 仅输出提交内容摘要，不再附带提交时间。
- 未提交事项必须在 `报告内容点` 中以 `进行中:` 前缀输出，避免与已提交事项混淆。
- 未提交事项必须优先根据代码 diff 的新增/删除内容、代码模块和改动文件总结“当前正在做什么”；不能只根据文件名、设计稿、需求文档或审查文档命名推断任务。
- 设计文件、设计目录、文档文件和纯设计/文档改动不参与进行中任务分析；若项目只有这些改动，不输出进行中事项。
- 只有代码 diff 无法提取可靠语义时，才回退到代码路径与状态摘要。
- `完成/修复` 只是示例口径，不做强制分类，核心要求是覆盖该时间范围内的我方提交任务点。
- 默认过滤低价值提交：重命名、回滚、构建、文档、测试（可通过 `references/projects.json` 覆盖）。
- 默认过滤低价值未提交改动：仅文档、测试、构建产物、重命名痕迹或其他无业务语义的工作区改动，不应强行写入周报。
- 作者过滤必须严格限定为“本人提交”：仅匹配作者姓名或邮箱的精确值，不做模糊包含匹配。
- 若某项目在统计周期内无提交，但存在未提交进行中事项，仍应出现在 `各项目明细` 中。
- 若本周期所有项目都无提交记录且无未提交进行中事项，报告中必须明确写“本周期无提交记录”。
- 不得把未在 Git 提交记录或 Git 工作区证据中出现的事项写入报告。
- 报告文本必须使用中文输出，编码使用 UTF-8。
- 报告必须自动保存到 `/home/luode/code`，文件名格式为 `?报-YYYYMMDDHHMMSS`（例如 `周报-20260403114521`）。
- `总体归纳统计` 可从 `references/projects.json` 读取以下可选字段：
  - 顶层 `release_overall_progress_percent`
  - 项目级 `release_pending_summary`、`release_progress_percent`
- 若项目未配置 `release_pending_summary` 或 `release_progress_percent`，默认输出 `待补充未完成事项 -> 项目完成度: 未配置`。
- 未提交工作区补充可从 `references/projects.json` 读取以下可选字段：
  - 顶层 `include_uncommitted_changes`
  - 顶层 `uncommitted_excluded_keywords`
  - 顶层 `uncommitted_excluded_paths`
  - 顶层 `uncommitted_max_paths`
  - 顶层 `uncommitted_analysis_excluded_paths`
  - 顶层 `uncommitted_analysis_excluded_extensions`

## 权责边界与不负责事项

- 只负责“基于 Git 提交 + Git 工作区未提交改动”的工作汇总，不负责替代发布总结或项目复盘。
- 不自动推断业务影响级别，只做提交事实汇总与轻量分类。
- 不把 Issue、需求单、口头进展当成已完成事项。
- 不把未提交改动写成“已完成”；未提交事项只能写成 `进行中`。
- 不使用设计文件、设计稿、需求文档、实施文档或审查文档作为进行中任务的语义依据。

## 需要暂停并确认的条件

- `references/projects.json` 中项目路径不存在或无法访问。
- 用户要求的时间范围与报告类型冲突（例如日报却要求跨多天）。
- 用户明确要求纳入非 Git 来源信息（需要单独确认口径）。
- 用户明确要求排除未提交事项，但当前配置默认开启工作区扫描（需要先确认以用户口径为准）。

## 执行通过 / 驳回标准

- 通过：报告覆盖配置中的项目，时间范围正确，格式完整；已提交事项可追溯到提交记录，进行中事项可追溯到代码 diff 或代码路径，且设计/文档文件未作为任务依据。
- 驳回：日期范围错误、项目遗漏、内容与 Git 证据不一致，或已提交 / 未提交事项混写不清。

## references 读取规则

- 默认先读 `references/projects.json`。
- 需要调整输出样式时，再读 `references/report-format.md`。
- 需要确认未提交工作区事项如何纳入周报时，再读 `references/uncommitted-worktree.md`。
- 输出工作报告前，必须读取 `../artifact-delivery-gate-rules/references/plain-language-document-contract.md`；正文先说明工作结果和影响，提交明细、时间口径和证据进入附录。
- 涉及审查或验收结果汇总时，同时读取 `../artifact-delivery-gate-rules/references/review-acceptance-gate-contract.md`，不把不适用验证写成阻断。
