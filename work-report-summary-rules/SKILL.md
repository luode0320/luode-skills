# 工作报告汇总规则

只在需要按项目汇总 Git 提交并输出年报/月报/周报/日报时使用这个 skill。
如果当前任务不是“按时间范围汇总提交”，不要触发。

## Skill 作用与适用场景

- 基于预配置项目清单批量统计提交记录，避免逐仓库手工整理。
- 输出统一格式的日报、周报、月报、年报。
- 报告默认按项目分组，按提交事实输出“报告内容点”。
- 标题或时间范围必须带“日期 + 星期”。

## 自动触发信号

- 用户明确说“生成日报/周报/月报/年报”。
- 用户明确说“汇总今天/本周/本月/今年做了什么”。
- 用户要求“根据多个项目 git 提交生成阶段性工作总结”。

## 进入后先做什么

1. 识别报告类型：`daily` / `weekly` / `monthly` / `yearly`。
2. 确认基准日期：
   - 用户给日期就按用户日期。
   - 用户未给日期，默认使用系统当天日期。
3. 读取 `references/projects.json`，确认项目路径、项目名、作者过滤配置、低价值提交过滤配置和报告保存路径配置。
4. 运行 `scripts/generate_git_report.py` 生成报告草稿。
5. 输出给用户前做一次清洗：补充空项目说明、检查日期与星期格式。

## 默认执行流程

1. 默认先读 `references/projects.json`。
2. 执行脚本生成报告：
   - 日报：`python3 scripts/generate_git_report.py --period daily --date 2026-04-03`
   - 周报：`python3 scripts/generate_git_report.py --period weekly --date 2026-04-03`
   - 月报：`python3 scripts/generate_git_report.py --period monthly --date 2026-04-03`
   - 年报：`python3 scripts/generate_git_report.py --period yearly --date 2026-04-03`
3. 如需按特定作者过滤，可加 `--author`（可重复）覆盖配置文件中的作者列表；若配置和参数都为空，脚本会自动用全局 `git config user.name/user.email` 识别“我方提交”。
4. 将脚本输出直接作为报告正文，并确认报告已保存到目标目录。

## 报告输出要求

- 顶部必须包含时间范围，使用“日期 + 星期”格式。
- 周报/月报/年报：时间范围之后必须先输出 `总体归纳统计`（按项目给出本周期任务大纲），再输出 `各项目明细`。
- 日报：不输出 `总体归纳统计`，直接输出 `各项目明细`。
- `总体归纳统计` 必须是任务大纲说明，不要输出“完成X项/修复X项”这类数量统计口径。
- 无提交记录的项目不需要出现在 `总体归纳统计` 与 `各项目明细` 中。
- 每个项目固定结构：
  - `项目名:`
  - `1. 报告内容点:`
- 每条任务点时间必须精确到秒，例如：`2026-04-03 11:12:54: 功能: xxx`。
- 时间口径必须统一为北京时间（`Asia/Shanghai`）。
- `完成/修复` 只是示例口径，不做强制分类，核心要求是覆盖该时间范围内的我方提交任务点。
- 默认过滤低价值提交：重命名、回滚、构建、文档、测试（可通过 `references/projects.json` 覆盖）。
- 作者过滤必须严格限定为“本人提交”：仅匹配作者姓名或邮箱的精确值，不做模糊包含匹配。
- 若本周期所有项目都无提交记录，报告中必须明确写“本周期无提交记录”。
- 不得把未在 Git 记录中出现的事项写入报告。
- 报告文本必须使用中文输出，编码使用 UTF-8。
- 报告必须自动保存到 `/home/luode/code`，文件名格式为 `?报-YYYYMMDDHHMMSS`（例如 `周报-20260403114521`）。

## 权责边界与不负责事项

- 只负责“基于 Git 提交的工作汇总”，不负责替代发布总结或项目复盘。
- 不自动推断业务影响级别，只做提交事实汇总与轻量分类。
- 不把 Issue、需求单、口头进展当成已完成事项。

## 需要暂停并确认的条件

- `references/projects.json` 中项目路径不存在或无法访问。
- 用户要求的时间范围与报告类型冲突（例如日报却要求跨多天）。
- 用户明确要求纳入非 Git 来源信息（需要单独确认口径）。

## 执行通过 / 驳回标准

- 通过：报告覆盖配置中的项目，时间范围正确，格式完整，内容可追溯到提交记录。
- 驳回：日期范围错误、项目遗漏、内容与 Git 不一致，或提交任务点明显缺失。

## references 读取规则

- 默认先读 `references/projects.json`。
- 需要调整输出样式时，再读 `references/report-format.md`。
