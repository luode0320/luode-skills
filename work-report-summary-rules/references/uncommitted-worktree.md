# 未提交工作区事项补充规则

## 目标

把“项目当前未提交改动”稳定纳入工作报告，但只用代码变更证据总结真实进行中任务，不得把设计材料或文档名称当成任务结论。

## 事实来源

- 必须先读取真实 Git 工作区状态：`git status --short`。
- 对已跟踪文件，必须结合 `git diff HEAD --unified=0` 分析新增和删除的代码内容。
- 对未跟踪文件，仅允许读取可识别的文本代码文件，且必须限制文件大小；二进制文件直接跳过。
- 不允许把口头描述、Issue 标题、需求标题或 AI 猜测直接写入周报。

## 写入口径

1. 未提交事项只能作为“进行中事项”补充，不能和已提交事项混写成“完成/修复”。
2. 输出时统一使用 `进行中:` 前缀。
3. 代码 diff 存在可靠业务语义时，摘要必须优先说明模块和变更内容，例如“正在推进兑换相关开发，代码变更集中在免 gas、交易量过滤”。
4. 设计文件、设计目录、需求/实施/审查文档以及纯文档改动不得参与任务语义分析；只有这些改动时不输出进行中事项。
5. 代码 diff 无法可靠提取语义时，才回退到代码路径和改动状态摘要。
6. 若项目本周期无提交，但代码工作区有真实未提交改动，该项目仍应显示在“各项目明细”中。

## 设计与文档排除

默认排除以下目录前缀：

- `doc/`
- `docs/`
- `design/`
- `designs/`
- `figma/`
- `mockup/`
- `prototype/`
- `.codegraph/`

默认排除以下扩展名：

- `.md`、`.markdown`、`.txt`
- `.pdf`、`.doc`、`.docx`
- `.ppt`、`.pptx`、`.xls`、`.xlsx`
- `.fig`、`.sketch`、`.xd`、`.drawio`
- `.png`、`.jpg`、`.jpeg`、`.gif`、`.webp`、`.pid`、`.lock`、`.log`

可通过 `references/projects.json` 覆盖：

- `uncommitted_analysis_excluded_paths`
- `uncommitted_analysis_excluded_extensions`

## 通过标准

- 报告中的未提交事项都能回溯到真实工作区状态。
- 进行中摘要优先来自代码 diff 内容和代码模块路径。
- 所有未提交事项都显式标注为 `进行中`。
- 没有把设计文件、文档文件名或文档内容误写成任务结论。
- 无法归纳时才使用代码路径兜底，不输出设计/文档路径。
