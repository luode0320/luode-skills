# 未提交工作区事项补充规则

## 目标

把“项目当前未提交改动”稳定纳入工作报告，但严格限制为 Git 工作区可验证的进行中事项，不得冒充已完成结果。

## 事实来源

- 只允许使用真实 Git 工作区证据：
  - `git status --short`
  - 必要时结合 `git diff --stat`
- 不允许把口头描述、Issue 标题、需求标题或 AI 猜测直接写入周报。

## 写入口径

1. 未提交事项只能作为“进行中事项”补充，不能和已提交事项混写成“完成/修复”。
2. 输出时统一使用 `进行中:` 前缀。
3. 若项目本周期无提交，但工作区有真实未提交改动，该项目仍应显示在 `各项目明细` 中。
4. 若工作区改动仅包含低价值内容（如文档、测试、构建产物、纯重命名），默认不写入周报。
5. 周报、月报、年报、日报都可复用同一补充逻辑；周报只是最常见场景。

## 推荐摘要方式

- 优先从未提交改动里的中文文件名、需求/实施/审查文档名中提取“当前正在做什么”的中文简要任务描述。
- 任务描述只需要一句中文简要说明，不需要展开路径、文件数或技术细节。
- 推荐格式：
  - `进行中: 完善订单风控校验联调`
  - `进行中: 补充周报 skill 的未提交任务描述`
- 若无法可靠提取中文任务名，再回退到路径摘要：
  - `进行中: 工作区仍有未提交改动（修改 2 项 / 新增 1 项），涉及 internal/order、doc/2-需求、scripts/report`
  - `进行中: 工作区仍有未提交改动（新增 3 项），涉及 app/exchange、internal/router`

## 配置项

可在 `references/projects.json` 顶层配置：

- `include_uncommitted_changes`
- `uncommitted_excluded_keywords`
- `uncommitted_excluded_paths`
- `uncommitted_max_paths`

## 通过标准

- 报告中的未提交事项都能回溯到真实工作区状态。
- 所有未提交事项都显式标注为 `进行中`。
- 没有把未提交事项误写成已完成结果。
- 能提取任务名时，优先输出中文简要任务描述；提取失败时才允许回退到路径摘要。
