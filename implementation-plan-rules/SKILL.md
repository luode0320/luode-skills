# 实施计划拆解规则

只在需求已经基本讲清、当前准备进入编码，但实现路径仍然复杂，需要先把执行步骤拆成清晰计划时使用这个 skill。
如果当前问题还是需求信息不全、边界不清、验收未定或任务还没拆出当前优先闭环，请先转交相邻需求域 skill。

## Skill 作用与适用场景

- 把已经确认的需求或当前优先子项，转成编码前可执行的实施计划。
- 明确本轮要改哪些目录、文件、模块和职责边界，避免一边写代码一边临时漂移。
- 给出任务顺序、依赖关系、最小闭环和每一步的验证点。
- 在技术路径存在多种合理实现时，先收敛 2-3 个方案，再锁定推荐路线。
- 保证计划中的每一步都可执行、可验证、可交接，不写空泛占位词。
- 将实施计划单独沉淀到 `artifact-storage-rules` 约定的实施计划文档位置，作为后续编码、审查、测试和交付的阶段性依据。
- 降低“需求虽然清楚，但真正动代码时落点混乱、顺序错乱、验证遗漏”的风险。

## 自动触发信号

- 需求、边界和验收标准已基本稳定，但任务仍涉及多个文件、多个模块、多个接口、多个步骤或多个验证点。
- `requirement-splitting-rules` 已拆出当前优先子项，但编码前还需要继续细化实施顺序和文件落点。
- 用户明确要求先写开发计划、实施计划、落地步骤、任务清单或编码前执行方案。
- 当前实现路径存在多个合理技术方案，需要先比较后收口。
- 不先写计划就容易造成改动范围失控、顺序颠倒、验证遗漏或多人协作断层。

## 进入后先做什么

1. 先确认需求文档、边界结论和验收标准已经基本稳定。
2. 确认当前要规划的是整个需求，还是拆分后当前优先闭环子项。
3. 先看仓库现状，找到相关目录、模块、已有实现和可能的复用点。
4. 列出本轮必须落地的文件 / 模块落点，不先展开代码细节。
5. 再拆任务顺序、依赖关系、最小闭环和验证点。
6. 如果实现路径存在多种合理方向，先做 2-3 个方案比选并给出推荐。
7. 找到当前需求对应的实施计划文档；如果还没有，就按 `artifact-storage-rules` 的命名模板初始化单独文档。

## 默认执行流程

1. 默认先读 `references/plan-entry-checklist.md`，确认进入实施计划阶段前的前置条件是否满足。
2. 再读 `../artifact-storage-rules/references/path-map.yaml` 与 `../artifact-storage-rules/references/update-policy.md`，确认实施计划文档路径、命名模板和同文档更新策略。
3. 再读 `references/plan-structure-template.md`，按统一结构生成或更新实施计划文档。
4. 如果任务粒度、依赖顺序或最小闭环不好判断，再读 `references/task-granularity-and-order.md`。
5. 如需判断与相邻 skill 的边界、或对照正反例，再读 `references/plan-boundaries-and-examples.md`。
6. 输出实施范围、文件 / 模块落点、任务顺序、验证步骤、风险与阻断项，并更新到当前需求对应的实施计划文档。
7. 如果还没有实施计划文档，则按 `artifact-storage-rules` 约定的命名规则创建；如同一天存在多个近似标题，在不改变中文语义前提下补充更具体描述以避免重名。
8. 计划写完后，必须再读 `references/plan-review-checklist.md` 做一次自审，重点检查覆盖度、占位词、可执行性和验证闭环。
9. 自审通过后，请用户先 review 当前实施计划文档；如用户提出调整，继续更新同一份计划文档，不新建平行记录。
10. 计划确认后，才进入编码基线域和相应代码位点域执行实际实现。

## 权责边界与不负责事项

- 只负责把已确认需求转成实施计划，不代替 `requirement-splitting-rules` 做顶层需求拆分。
- 不代替 `acceptance-criteria-rules` 编写验收标准；验收标准应已基本存在。
- 不代替实际编码，不在本 skill 内直接展开实现代码。
- 不代替测试域制定完整测试策略，但必须把当前计划对应的最小验证点写清。
- 不接受 `TBD`、`TODO`、`后续再补`、`实现时再看` 这类空泛占位计划。
- 不为同一个需求重复新建多份实施计划文档；若计划文档已存在，应直接更新原文档。
- 如果技术分歧的根源其实是业务目标不清、范围有争议或验收没定，应回流相邻需求域 skill，而不是假装用实施计划掩盖前置问题。

## 需要暂停并确认的条件

- 需求、边界或验收标准仍在明显变化，当前计划随时会失效。
- 当前任务还没拆出可独立推进的优先闭环，却试图一次规划全部大需求。
- 实现路径存在重大分歧，但尚未形成 2-3 个可比方案和推荐结论。
- 无法确定主要文件落点、模块职责或依赖顺序，说明仍需先回看仓库现状或回流需求域。
- 计划中仍存在明显占位词、跳步描述或无法验证的任务项。

## 执行通过 / 驳回标准

- 通过：能够形成一份可直接指导后续编码的实施计划，至少说明本轮范围、文件 / 模块落点、任务顺序、依赖关系、最小闭环、验证点、风险和阻断项，并完成自审与用户确认门槛。
- 驳回：只是把需求原文换一种说法重写，没有落实到文件 / 模块和执行步骤；或计划里仍充满占位词、顺序不清、无法验证；或本应先回流需求拆分 / 边界 / 验收，却被硬塞成实施计划。

## 执行结果归档要求

- 将实施范围、文件落点、任务顺序、验证点、风险和阻断项记录到 `artifact-storage-rules` 约定的实施计划文档中，不与需求文档、验收文档混写。
- 文档文件名、根目录和同文档更新策略统一遵循 `../artifact-storage-rules/references/path-map.yaml` 与 `../artifact-storage-rules/references/update-policy.md`。
- 同一个需求的实施计划应持续更新同一份计划文档，而不是重复新建多份平行计划记录。
- 归档内容至少包含对应需求来源、当前优先闭环、目录 / 文件落点、任务清单、推荐顺序、每步验证点、自审结论和用户确认状态。
- 如果后续需求变更、边界变化或验收调整推翻原计划，应在同一份计划文档中更新新旧差异和重新确认状态。
