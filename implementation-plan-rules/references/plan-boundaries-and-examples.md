# 实施计划边界与样例

## 与相邻 skill 的边界

### 1. 不是需求拆分

- `requirement-splitting-rules` 负责把大需求拆成子项、波次或子系统。
- `implementation-plan-rules` 负责把“当前要先做的那个子项”继续拆成编码前执行步骤。

### 2. 不是验收标准

- `acceptance-criteria-rules` 回答“做到什么算完成”。
- `implementation-plan-rules` 回答“为了做到它，当前编码前先按什么顺序落地”。

### 3. 不是测试策略

- 测试域负责完整功能验证、联调、回归和测试资源规范。
- 本 skill 只要求把“每一步怎么做最小验证”写清，避免计划与验证彻底脱节。

### 4. 不是实际编码

- 本 skill 到“计划确认”就结束。
- 一旦开始改代码，应转入编码基线域和相应代码位点域。

## 正例

- 需求、边界、验收都清楚，但当前要同时改配置、接口、数据库访问和前端展示，先写实施计划再编码。
- 大需求已经拆成三个子项，本轮只做“消息列表分页接口”，先把文件落点、接口顺序和验证点拆清。
- 存在“沿用旧模块扩展”和“新建独立模块”两种实现方向，先对比后锁定推荐路线，再开始写代码。

## 反例

- 连字段和业务规则都没讲清，就直接开始写实施计划。
- 明明是“聊天 + 文件 + 计费 + 分析”四个独立能力，却不先做需求拆分。
- 验收口径完全没定，就想靠实施计划兜底。
- 计划里直接写大段实现代码，混淆了计划和编码阶段。
