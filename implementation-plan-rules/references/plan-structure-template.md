- 涉及文件 / 模块：

## 结构要求

- 必须先写清当前优先闭环，不要把整张大路线图塞成一份笼统计划。
- “现状与落点”必须明确到目录、模块或文件层级，不能只写“后端处理一下”“前端改一下”。
- 若涉及新增文件，必须给出每个新文件的“目标目录 + 文件名 + 用途”；禁止“新增若干文件”这类模糊表述。
- 若涉及数据库新表或字段变更，必须补齐可执行 SQL（`CREATE TABLE` / `ALTER TABLE`），不能只写自然语言。
- “方案选择”只在确实存在明显不同路线时出现；没有分歧时可省略冗余方案节。
- “实施步骤”按执行顺序写，不要按想到什么写什么。
- “每步验证点”必须能指导后续编码审查、功能验证或回归判断。
- “风险与阻断项”必须写真实阻断，不要写空话。
