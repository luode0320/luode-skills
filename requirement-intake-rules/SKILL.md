# 需求接入规则

只在任务刚进入研发、需要先把需求讲清楚并整理成可继续推进的输入时使用这个 skill。
如果当前争议已经变成需求缺口、需求边界、需求变更或验收标准问题，请转交相邻需求域 skill。

## Skill 作用与适用场景

- 把口头需求、零散指令或会议结论整理成统一的研发入口。
- 抽取目标、背景、上下游、输入输出、依赖和交付物形态。
- 帮助判断下一步应进入缺口识别、边界确认、需求拆分还是直接实现。
- 避免还没讲清需求就直接进入编码或误走 Bug 流程。

## 自动触发信号

- 用户刚提出新功能、新页面、新接口、新模块、新脚本或新自动化任务。
- 需求描述还停留在一句话、聊天片段或零散文档摘录。
- 当前还没有明确目标用户、业务目的、输入输出和上下游依赖。
- 总控层需要先把任务落到需求域入口，而不是直接分配代码位点。

## 进入后先做什么

1. 先抽取这次需求到底要解决什么问题、服务谁、产出什么。
2. 再梳理输入、输出、调用方、被调用方和关键依赖。
3. 列出已明确项、默认前提和待确认项，不提前脑补业务规则。
4. 判断下一步应该转入哪个需求域子 skill。

## 默认执行流程

1. 默认先读 `references/intake-checklist.md`，先确认需求入口最少要补齐哪些信息。
2. 如需继续展开，再读 `references/requirement-structure-template.md`，需要把零散需求整理成统一结构模板。
3. 需要对照边界或正反例时，再读 `references/intake-boundaries-and-examples.md`，需要判断需求接入与相邻 skill 的边界。
4. 输出 标准化需求接入摘要、待确认项列表和下一步分流建议。
5. 若基础信息仍不足，转交 `requirement-gap-rules`；若范围和旧逻辑影响不清，转交 `requirement-boundary-rules`；若需求体量过大，转交 `requirement-splitting-rules`。

## 权责边界与不负责事项

- 只负责需求接入整理，不替代 `requirement-gap-rules` 做缺口识别。
- 不替代 `requirement-boundary-rules` 做影响面和旧逻辑改动边界判断。
- 不负责把大需求拆成实施波次，那属于 `requirement-splitting-rules`。
- 不把历史故障、线上异常或旧行为错误误判成新需求，疑似缺陷应转 Bug 域。

## 需要暂停并确认的条件

- 连目标和交付物都无法说清，无法形成最小接入结论。
- 当前任务同时像需求又像历史 Bug，归属无法判定。
- 关键输入输出、上下游依赖或目标角色完全未知。
- 用户一次性提出多项不相干诉求，明显需要先拆任务。

## 执行通过 / 驳回标准

- 通过：能够形成一份可供后续 skill 继续工作的需求入口摘要，至少说明目标、输入输出、上下游、约束和待确认项。
- 驳回：仍停留在原始口头描述层，没有形成结构化入口，或把需求接入与缺口、边界、Bug 归属混成一件事。

## 执行结果归档要求

- 将需求接入摘要、待确认项和分流结论记录到 `analysis/`。
- 归档内容至少包含需求来源、目标、输入输出、上下游依赖和下一步建议。
- 如果只是极小需求且信息完整，可简化记录，但应保留接入结论。

## references 读取规则

- 默认先读 `references/intake-checklist.md`。
- 只有在 需要输出结构化需求摘要模板 时，再读 `references/requirement-structure-template.md`。
- 只有在 判断是否应转交相邻 skill 或对照正反例 时，再读 `references/intake-boundaries-and-examples.md`。
