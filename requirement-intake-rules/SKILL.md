# 需求接入规则

只在任务刚进入研发、需要先把需求讲清楚并整理成可继续推进的输入时使用这个 skill。
如果当前争议已经变成需求缺口、需求边界、需求变更或验收标准问题，请转交相邻需求域 skill。

## Skill 作用与适用场景

- 把需求 URL、口头描述、聊天片段、零散资料、截图、物料和会议结论整理成统一的研发入口。
- 优先根据需求 URL 和现有资料补齐目标、背景、上下游、输入输出、依赖和交付物形态。
- 在资料或数据资源本身存在逻辑冲突时，先停止继续给实现建议，转入逐步完善需求。
- 帮助判断下一步应进入缺口识别、边界确认、需求拆分还是直接实现。
- 在需求达到可开发状态后，生成结构化需求文档并保存到 `ment/` 目录。
- 避免还没讲清需求就直接进入编码或误走 Bug 流程。

## 自动触发信号

- 用户刚提出新功能、新页面、新接口、新模块、新脚本或新自动化任务。
- 需求描述还停留在一句话、聊天片段、需求 URL、零散文档摘录或杂项物料。
- 当前还没有明确目标用户、业务目的、输入输出和上下游依赖。
- 用户同时给出需求链接、资料包、字段说明、流程图、截图或数据样例，需要统一整理。
- 资料之间存在明显逻辑冲突、字段对不上、流程不闭环或数据资源本身有问题。
- 总控层需要先把任务落到需求域入口，而不是直接分配代码位点。

## 进入后先做什么

1. 先收集并排序需求来源，优先阅读需求 URL，再补读聊天描述、附件、物料和数据样例。
2. 抽取这次需求到底要解决什么问题、服务谁、产出什么。
3. 再梳理输入、输出、调用方、被调用方、关键依赖和可用数据资源。
4. 列出已明确项、默认前提和待确认项，不提前脑补业务规则。
5. 检查资料、字段、流程、物料和数据资源之间是否自洽；一旦发现逻辑冲突，先停止继续给实现建议。
6. 判断下一步应该转入哪个需求域子 skill。

## 默认执行流程

1. 默认先读 `references/intake-checklist.md`，确认需求 URL、资料、数据资源和入口信息最少要补齐哪些内容。
2. 再读 `references/requirement-structure-template.md`，把零散需求整理成统一文档结构，并准备产出到 `ment/`。
3. 需要对照边界或正反例时，再读 `references/intake-boundaries-and-examples.md`，判断当前问题该留在接入层，还是转交相邻需求 skill。
4. 输出标准化需求接入摘要、资料逻辑检查结论、待确认项列表和下一步分流建议。
5. 若资料、字段、流程、业务规则、前提或数据资源不完整，转交 `requirement-gap-rules`。
6. 若范围、兼容性、旧逻辑影响或“这是不是本次需求”不清，转交 `requirement-boundary-rules`。
7. 若需求已经成立但体量过大，任务项拆分由 `requirement-splitting-rules` 负责，不在本 skill 内直接拆任务。
8. 当需求已补齐到可开发状态时，生成需求文档并保存到 `ment/YYYY-MM-DD_需求中文说明.md`；如同一天存在多个近似标题，在不改变中文语义前提下补充更具体描述以避免重名。

## 权责边界与不负责事项

- 只负责需求接入整理、资料归并、基础逻辑核对和文档落地，不替代 `requirement-gap-rules` 做缺口识别。
- 不替代 `requirement-boundary-rules` 做影响面和旧逻辑改动边界判断。
- 不负责把大需求拆成实施波次或子任务项，那属于 `requirement-splitting-rules`。
- 不把历史故障、线上异常或旧行为错误误判成新需求，疑似缺陷应转 Bug 域。
- 不在需求资料已发生逻辑冲突时继续输出实现建议，而是先停下来给用户补齐和修正建议。

## 需要暂停并确认的条件

- 连目标和交付物都无法说清，无法形成最小接入结论。
- 需求 URL、资料、字段说明、流程图或数据样例之间出现明显逻辑冲突。
- 数据资源本身不成立、口径互相矛盾、关键字段对不上或流程无法闭环。
- 当前任务同时像需求又像历史 Bug，归属无法判定。
- 关键输入输出、上下游依赖或目标角色完全未知。
- 用户一次性提出多项不相干诉求，明显需要先拆任务。

## 执行通过 / 驳回标准

- 通过：能够基于需求 URL、资料和补充说明形成一份可供后续代码开发继续使用的结构化需求文档，至少说明目标、输入输出、上下游、约束、资料来源、待确认项和下一步流向。
- 驳回：仍停留在原始口头描述层，没有形成结构化入口；或资料本身有逻辑问题却继续给实现建议；或把需求接入与缺口、边界、Bug 归属混成一件事。

## 执行结果归档要求

- 将需求接入摘要、资料来源、待确认项和分流结论记录到 `ment/`。
- 文档文件名统一使用 `ment/YYYY-MM-DD_需求中文说明.md` 格式，例如 `ment/2026-03-28_需求中文将要说明.md`。
- 归档内容至少包含需求来源、需求 URL、资料列表、目标、输入输出、上下游依赖、约束、资料逻辑检查结论、待确认项和下一步建议。
- 如果需求暂时不能进入开发，也要在文档中明确写出阻断原因和待补齐项，而不是不落文档。

## references 读取规则

- 默认先读 `references/intake-checklist.md`。
- 只有在需要输出结构化需求文档模板和 `ment/` 文档格式时，再读 `references/requirement-structure-template.md`。
- 只有在判断是否应转交相邻 skill 或对照正反例时，再读 `references/intake-boundaries-and-examples.md`。
