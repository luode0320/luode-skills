# 需求接入检查清单

## 项目上下文

- 是否已经查看当前项目中相关目录、现有模块、已有文档或相似实现。
- 是否已经了解当前仓库已有约束、命名、流程或历史做法，避免脱离现状整理需求。
- 如果这是在旧系统上继续扩展，是否已识别已有能力和现有边界。

## 资料来源

- 是否给出了需求 URL。
- 是否给出了聊天描述、截图、原型、字段说明、流程图、数据样例或其他物料。
- 各资料之间是否能相互对应，是否存在冲突。

## 目标

- 当前需求解决什么问题。
- 目标用户、调用方或业务角色是谁。
- 最终交付物是页面、接口、脚本、配置、流程能力还是多者组合。
- 当前成功标准是否已经至少能用一句话说清。

## 基础信息

- 输入是什么，来源在哪里。
- 输出是什么，交付给谁。
- 上下游依赖、前置条件、约束条件是什么。
- 数据资源是否可用，字段、口径、流程是否自洽。
- 已知不在本次范围内的事项有哪些。

## 澄清节奏

- 是否遵守“一次只推进一个关键问题”，而不是一次抛出大量混合问题。
- 当前问题是否来自真实缺口，而不是 agent 预填的业务猜测。
- 当前优先澄清的是目的、约束、成功标准，还是字段、流程、依赖中的哪一项。
- 是否存在多个合理方向，需要先给 2-3 个方案和推荐结论，而不是直接拍板。

## 范围校验

- 当前需求是否一次打包了多个独立子系统、多个产品子域或多条相对独立主线。
- 如果是，是否应先交给 `requirement-splitting-rules`，而不是继续细化单点细节。

## 逻辑检查

- URL、资料、物料和数据样例之间是否存在逻辑冲突。
- 关键字段是否对得上。
- 关键流程是否闭环。
- 是否存在“资料本身有问题，不能继续实现”的阻断点。

## 防偷跑检查

- 正式需求主文档未落盘前，是否已经明确禁止进入实施规划或实现。
- 是否存在“先做一版看看”“先做了再补需求”“用户应该就是这个意思”这类偷跑倾向。
- 如果用户已表达后续想直接开工，是否仍保持“先文档、后验收标准、后实施规划、再等待明确开工授权”的顺序。

## 文档质量门槛

- 是否按 SRS 主结构组织文档（引言/产品概览/需求定义/验证/附录），而非零散段落堆叠。
- 是否包含文档信息与修订记录（版本、状态、变更原因）。
- 文档中是否仍存在 `TBD`、`TODO`、`待补`、`后续确认` 这类未收口占位词。
- 文档内部是否存在前后矛盾或两种解释都说得通的描述。
- 当前需求是否已经小到可以进入后续实现或继续拆分，而不是仍然过大失控。
- 功能需求是否按条目编号表达（如 `REQ-FUNC-001`），并具备“输入/处理/输出/异常/验收映射”最小字段。
- 是否包含验证章节：每类核心需求是否明确验证方法（测试/分析/检查/演示）与通过标准。
- 需求主文档是否包含 Mermaid 流程图（`flowchart`）与 Mermaid 时序图（`sequenceDiagram`）代码块；复杂流程不得只写文字。
- 图形化内容与正文、表格中的规则是否一致，是否存在图文冲突。
- 需求文档是否采用“正文摘要 + 附录详解”：正文只保留最小必要定义，详细指标口径、术语解释和核心判定逻辑已按需沉到附录。
- 若正文出现核心指标、关键术语或核心逻辑，附录是否已补齐对应详解，且正文能够回指到附录位置。
- 关键规则是否用表格或矩阵表达（如角色-职责、状态-动作、模块-边界），而非全部线性段落。
- 是否已在正文写明“图片资产决策：需要”，或写 `N/A + 原因 + 证据`。
- 若用户提供需求截图或需生成外部图片，是否已统一保存到唯一目录 `doc/data/images/`，而非 `doc/data/` 根或历史需求域图片目录。
- 需求主文档是否已使用非空 alt 的 Markdown 图片语法按 `../data/images/<file>` 相对路径引用，并登记 `IMG-*`、来源、用途、版本、关联 ID、敏感状态和版权状态。
- 图片文件扩展名、签名和版本命名是否符合中央契约；是否拒绝绝对路径、反斜杠、`file://`、Base64/data URI、HTML `<img>`、外部热链、越界路径及旧路径。
- 声明需要图片时是否使用真实 `imagegen` 生成或归档；生成失败是否阻断，是否存在未被引用的孤儿图片。
- Mermaid 流程、时序、状态、依赖和数据关系是否仍完整，图片是否仅作为视觉证据而未替代 Mermaid。
- 若涉及新增文件，是否已使用目录树（`text` 代码块）列出新增文件，并在树节点后标注用途。
- 若涉及数据库新表或字段变更，是否已给出对应 SQL（`CREATE TABLE` / `ALTER TABLE`）。
- 文档落地后是否已请用户 review，再进入后续实现或计划阶段。

## 结论要求

- 形成“已明确 / 待确认 / 阻断项 / 下一步流向”四段式摘要。
- 如果存在多个合理方向，应补一段“候选方案 / 推荐方案 / 推荐理由”。
- 不要在接入阶段直接跳到实现动作。
- 当需求达到可开发状态时，输出或更新 `doc/2-需求/YYYY-MM-DD_HHmmss_<需求中文简介标题>.md` 文档。
- 后续需求子 skill 必须继续更新这同一份文档，不额外新建平行记录。
- 命中文件新增或数据库变更场景时，缺目录树或缺 SQL 必须判定为未通过；若目录信息仍是普通条目描述同样判定未通过。
