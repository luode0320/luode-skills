# 需求接入检查清单

## 资料来源

- 是否给出了需求 URL。
- 是否给出了聊天描述、截图、原型、字段说明、流程图、数据样例或其他物料。
- 各资料之间是否能相互对应，是否存在冲突。

## 目标

- 当前需求解决什么问题。
- 目标用户、调用方或业务角色是谁。
- 最终交付物是页面、接口、脚本、配置、流程能力还是多者组合。

## 基础信息

- 输入是什么，来源在哪里。
- 输出是什么，交付给谁。
- 上下游依赖、前置条件、约束条件是什么。
- 数据资源是否可用，字段、口径、流程是否自洽。
- 已知不在本次范围内的事项有哪些。

## 逻辑检查

- URL、资料、物料和数据样例之间是否存在逻辑冲突。
- 关键字段是否对得上。
- 关键流程是否闭环。
- 是否存在“资料本身有问题，不能继续实现”的阻断点。

## 结论要求

- 形成“已明确 / 待确认 / 阻断项 / 下一步流向”四段式摘要。
- 不要在接入阶段提前承诺实现方案。
- 当需求达到可开发状态时，输出 `ment/YYYY-MM-DD_需求中文说明.md` 文档。
