# 需求结构模板（SRS 映射版）

结论：{用一句普通中文说明本次要达成的业务结果}；影响：{说明受影响的用户、角色或业务环节}；范围：{列出本次会做的事情}；非范围：{列出明确不做的事情}；变化：{说明用户或业务可感知的变化}；完成标准：{说明可观察的完成结果}；术语说明：无技术术语需要解释，或 {术语：它在本次业务目标中的作用}；验证状态：{说明当前验证进展和依据}。

本模板基于 `jam01/SRS-Template` 的核心章节映射到当前项目需求文档写法，并融合本项目的图形化、归档和可验收约束。

## 白话正文与附录

新建文档先按 [白话文档与附录分层契约](../../artifact-delivery-gate-rules/references/plain-language-document-contract.md) 在 H1 后写一段包含“结论、影响、范围、非范围、变化、完成标准、术语说明、验证状态”的普通中文开场；保留当前文档的标题、层级和顺序。本模板其余字段、表格、图形 ID 和追踪内容放在原有技术章节或文末连续附录。涉及审查、验收、浏览器或第三方条件时，读取 [审查与验收条件化门禁契约](../../artifact-delivery-gate-rules/references/review-acceptance-gate-contract.md)。

## 建议结构

### 0. 文档信息

1. 需求标题：中文标题，适合用于 `doc/2-需求/YYYY-MM-DD_HHmmss_<需求中文简介标题>.md` 文件名。
2. 版本与状态：`v0.x / 草稿 / 已确认 / 变更中`。
3. 需求来源：需求 URL、聊天记录、附件、截图、物料列表。
4. 修订记录：记录版本、日期、变更原因、责任人。

### 1. 引言（SRS 1. Introduction）

1.1 文档目的（Document Purpose）：
- 本文档用于定义“做什么”和“做到什么算完成”，不描述低层实现细节。
- 说明主要读者（产品/研发/测试/运维/安全）及使用方式。

1.2 产品范围（Product Scope）：
- 需求目标：一句话说明要完成什么。
- 背景动因：为什么现在要做。
- 本次纳入范围 / 明确排除范围（Out of Scope）。

1.3 术语与缩写（Definitions, Acronyms）：
- 正文只保留当前阅读必需的最小术语定义；更细的别名、易混点和上下文差异统一下沉到附录 5.2。

1.4 参考资料（References）：
- 需求链接、设计稿、接口文档、历史方案、相关规则文档。

1.5 文档导读（Document Overview）：
- 说明第 2~5 章分别承载什么内容，便于评审定位。

### 2. 产品概览（SRS 2. Product Overview）

2.1 产品视角（Product Perspective）：
- 项目上下文：当前仓库相关模块、已有能力、现有约束或相似实现。
- 上下游依赖：接口、页面、服务、配置、数据源。

2.2 产品功能摘要（Product Functions）：
- 列出 5-10 条高层功能点，不展开细节。

2.3 约束条件（Product Constraints）：
- 时限、兼容、权限、环境、性能、安全、合规约束。

2.4 用户与角色特征（User Characteristics）：
- 角色、权限、使用频率、关键诉求、可访问性要求。

2.5 假设与依赖（Assumptions and Dependencies）：
- 外部系统、团队协作、基础数据、配置可用性等假设。

2.6 需求分配与波次（Apportioning of Requirements）：
- 对大需求可按子系统、模块、阶段做拆分映射，注明当前优先闭环。

### 3. 需求定义（SRS 3. Requirements）

3.1 外部接口需求（External Interfaces）：
- 用户界面交互、软件接口、数据交换、异常返回约定。

3.2 功能需求（Functional Requirements）：
- 建议逐条编号（如 `REQ-FUNC-001`），每条至少包含：
  - `ID`
  - `标题`
  - `需求陈述（shall）`
  - `触发条件/输入`
  - `处理规则`
  - `输出/结果`
  - `异常分支`
  - `验收标准链接`

3.3 质量需求（Quality of Service）：
- 性能、安全、可靠性、可用性、可维护性、可观测性等可量化指标。

3.4 合规与边界（Compliance / Boundary）：
- 合规要求、审计要求、数据边界、跨境、隐私等。

3.5 数据与实现约束（Design & Implementation Constraints）：
- 若涉及数据库新表或字段变更，必须附 SQL：
  - 建表 SQL（`CREATE TABLE`）
  - 字段变更 SQL（`ALTER TABLE`）
- 若涉及新增文件，必须附目录树（`text` 代码块）并标注用途：

```text
app/
└── exchange/
    ├── bizpolicy/
    │   └── context.go                     # 业务上下文与过滤规则定义
    ├── repository/
    │   ├── exchangeBusinessControl.go     # 配置查询仓储
    │   └── model/
    │       └── exchangeBusinessControl.go # 配置实体模型
    └── service/
        └── business_filter.go             # 过滤服务（链/币/渠道）

scripts/
└── 20260416_exchange_business_control.sql # 数据库变更脚本
```

3.6 图形化需求表达（项目增强，强制）：
- 流程图（必填，Mermaid `flowchart` 代码块）。
- 时序图（必填，Mermaid `sequenceDiagram` 代码块）。
- 关键映射表（建议）：角色-职责、状态-动作、模块-边界矩阵。
- 对核心指标、关键术语和核心逻辑，正文只保留最小必要摘要；若需要解释口径、公式、边界、优先级、例外或易混点，应在附录 5.1~5.3 展开，并在正文中明确回指。
- 图片资产决策（必填）：写“图片资产决策：需要”，或写 `N/A + 原因 + 证据`。
- 若用户提供需求截图或需外部生成图片素材：
  - 统一保存到 `doc/data/images/`，不得直接写入 `doc/data/`。
  - 使用从当前需求文档位置计算的相对路径，在主文档直接引用展示；需求文档统一使用 `../data/images/<file>`。
  - 示例：`![IMG-REQ-001 登录页原型](../data/images/YYYY-MM-DD_HHmmss_<需求中文简介标题>.login-state-v1.png)`。
  - 真实生成必须调用 `imagegen`；失败时将需求状态置为 `blocked`，不得用程序绘图或占位图替代。

### 4. 验证（SRS 4. Verification）

4.1 验证策略：
- 每类需求采用何种验证方式：测试、分析、检查、演示。

4.2 可验收标准映射：
- 将核心需求条目映射到可验收条件，注明验证入口与预期结果。

4.3 风险与阻断项：
- 当前阻断、潜在风险、缓解动作、责任人、截止时间。

4.4 待确认项与下一步：
- 待确认项（必须可执行）。
- 必要后续动作（仅需求尚未闭环、仍需转缺口/边界/拆分/验收/实施规划，或用户明确要求建议时填写；已闭环时不写）：转缺口、边界、拆分、验收标准、实施规划或直接实现。

## 使用要求

- 文档路径统一为 `doc/2-需求/YYYY-MM-DD_HHmmss_<需求中文简介标题>.md`。
- 同一个需求只保留一份 `doc/2-需求/` 文档；后续缺口、边界、拆分、变更都继续更新这份文档。
- 复杂流程不得仅用长段文字，必须使用 Mermaid 流程图与时序图，且两者应直接写在主文档正文内。
- 图、表、正文三者必须口径一致；出现冲突时按“先修文档再推进实现”处理。
- 需求文档默认采用“正文摘要 + 附录详解”：正文保留阅读当前需求所需的最小定义和主结论，涉及核心指标、关键术语或核心逻辑时，再按需在附录 5.1~5.3 补齐详解。
- 若正文出现业务指标、术语或核心判定逻辑，但读者仅靠正文无法理解口径、边界、公式、优先级或例外，视为附录未收口。
- 若有截图或外部图片，必须落 `doc/data/images/`，使用非空 alt 的 `../data/images/<file>` 相对路径在 md 内直接引用展示，并登记 `IMG-*`。
- 图片引用禁止绝对路径、反斜杠、`file://`、Base64/data URI、HTML `<img>`、外部热链、越界路径和直接引用 `doc/data/` 根文件；Mermaid 流程/时序/状态/依赖图仍必须保留。
- 若涉及数据库变更，缺 SQL 视为文档未收口。
- 若涉及新增文件，缺目录树视为文档未收口。
- 文档完成后先做轻量自审（占位词、矛盾、歧义、图文一致性），再请用户确认。

## 极致完整性字段契约（强制）

需求主文档交付给普通模型执行前，以下字段必须逐项出现；不适用项必须写成 `N/A + 原因 + 证据`，不得留空、写“后续补充”或使用未授权默认值。

| 字段组 | 必填字段 | 通过条件 |
| --- | --- | --- |
| 来源与决策 | `SRC-*`、`DEC-*`、来源类型、原文摘录、责任人、决策状态、决策时间、证据路径 | 每个关键结论都能回指来源与决策；未决结论标记 `unresolved_decisions` |
| 需求条目 | `REQ-*`、`RULE-*`、目的、角色、触发、输入、处理、输出、异常、权限、兼容、可观测性、回滚 | 每条需求可独立执行、验证，并回指至少一个 `AC-*` |
| 数据与接口 | 字段类型/单位/精度/空值、接口方法/路径/请求/响应/错误、数据来源/保留期/脱敏 | 新增或修改数据时附 SQL；接口变更有兼容策略 |
| 范围与拆分 | In Scope、Out of Scope、影响矩阵、垂直切片唯一归属、依赖与顺序 | 不存在重复归属、隐含范围或未裁决冲突 |
| 验收与证据 | `AC-*`、测试入口、local 样本、断言、失败预期、清理、回滚、证据编号 | `REQ/RULE -> AC -> TEST -> EVIDENCE` 双向可追踪 |
| 图形与附录 | 流程图、时序图、状态/依赖图（按复杂度）、图例、术语、指标、公式、例外 | 图前写目的和关联 ID；图、表、正文术语一致 |
| 图片资产 | 图片资产决策、`IMG-*`、来源、用途、版本、`doc/data/images/` 相对路径、引用章节、敏感/版权状态 | 需要图片时文件真实存在、非空 alt、路径可解析且无孤儿；不需要时写 `N/A + 原因 + 证据` |

### 零决策交付闸门

1. `P0/P1` 未决项、权限/兼容/数据/异常/回滚任一关键决策缺失时，文档状态必须为 `blocked`，不得进入验收、实施计划或编码。
2. `P2` 未决项必须有默认值授权人、有效期、替代方案、风险、复核时间和证据；没有授权不得由执行模型猜测。
3. 每个章节必须给出责任人、输入、输出、前置依赖、完成标准、失败处理和回滚边界；不适用项仍需按 `N/A + 原因 + 证据` 填写。
4. 文档必须在落盘后通过 UTF-8、front matter、稳定 ID、占位词、链接、Mermaid 语法和双向追踪检查；机器校验失败即阻断交付。

### 5. 执行附录（SRS 5. Appendixes）

5.1 核心指标说明（按需）：
- 建议字段：指标名、定义/业务意义、统计口径或公式、数据来源、适用范围、注意事项。

5.2 核心术语与缩写补充（按需）：
- 建议字段：术语/缩写、别名或英文名、定义、使用上下文、易混点。

5.3 核心逻辑与判定规则（按需）：
- 建议字段：逻辑主题、触发条件、输入、判定规则、输出、异常/例外、关联需求条目。

5.4 正文图示索引：
- 流程图：第 3.6 节中的 Mermaid `flowchart`。
- 时序图：第 3.6 节中的 Mermaid `sequenceDiagram`。

5.5 执行细节（按需）：
- 记录 local 环境、执行步骤、命令、样本、预期失败、清理和回滚；正文只回指本节，不重复机器细节。

### 6. 追踪附录

6.1 来源、决策与证据追踪：
- 汇总 `SRC -> DEC -> REQ/RULE -> AC -> CYCLE -> TASK -> TEST -> EVIDENCE` 的双向链接和证据定位。

6.2 图片素材清单（按需）：
- 存放目录：`doc/data/images/`（图片唯一目录；`doc/data/` 根不允许直接存图）。
- 每项列出 `IMG-*`、文件名、来源、用途、版本、相对路径、关联 `SRC/DEC/REQ/RULE/AC`、引用位置（章节号）、敏感状态和版权状态。
- 同一图片可被多个文档引用；清单、正文引用和实际文件必须一一对应，未被任何 Markdown 引用的文件视为孤儿。

6.3 变更追踪：
- 记录关键需求变更、边界变更、验收变更与影响面。
