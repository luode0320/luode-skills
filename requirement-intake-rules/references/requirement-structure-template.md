# 需求结构模板（SRS 映射版）

本模板基于 `jam01/SRS-Template` 的核心章节映射到当前项目需求文档写法，并融合本项目的图形化、归档和可验收约束。

## 建议结构

### 0. 文档信息

1. 需求标题：中文标题，适合用于 `doc/2-需求/YYYY-MM-DD_HHmmss_<需求中文简介标题>.md` 文件名。
2. 版本与状态：`v0.x / 草稿 / 已确认 / 变更中`。
3. 需求来源：需求 URL、聊天记录、附件、截图、物料列表。
4. 修订记录：记录版本、日期、变更原因、责任人。

### 1. 引言（SRS 1. Introduction）

1.1 文档目的（Document Purpose）：
- 本文档用于定义“做什么”和“做到什么算完成”，不描述低层实现细节。
- 说明主要读者（产品/研发/测试/运维/安全）及使用方式。

1.2 产品范围（Product Scope）：
- 需求目标：一句话说明要完成什么。
- 背景动因：为什么现在要做。
- 本次纳入范围 / 明确排除范围（Out of Scope）。

1.3 术语与缩写（Definitions, Acronyms）：
- 关键业务术语、缩写、状态名、角色名统一定义（建议表格）。

1.4 参考资料（References）：
- 需求链接、设计稿、接口文档、历史方案、相关规则文档。

1.5 文档导读（Document Overview）：
- 说明第 2~5 章分别承载什么内容，便于评审定位。

### 2. 产品概览（SRS 2. Product Overview）

2.1 产品视角（Product Perspective）：
- 项目上下文：当前仓库相关模块、已有能力、现有约束或相似实现。
- 上下游依赖：接口、页面、服务、配置、数据源。

2.2 产品功能摘要（Product Functions）：
- 列出 5-10 条高层功能点，不展开细节。

2.3 约束条件（Product Constraints）：
- 时限、兼容、权限、环境、性能、安全、合规约束。

2.4 用户与角色特征（User Characteristics）：
- 角色、权限、使用频率、关键诉求、可访问性要求。

2.5 假设与依赖（Assumptions and Dependencies）：
- 外部系统、团队协作、基础数据、配置可用性等假设。

2.6 需求分配与波次（Apportioning of Requirements）：
- 对大需求可按子系统、模块、阶段做拆分映射，注明当前优先闭环。

### 3. 需求定义（SRS 3. Requirements）

3.1 外部接口需求（External Interfaces）：
- 用户界面交互、软件接口、数据交换、异常返回约定。

3.2 功能需求（Functional Requirements）：
- 建议逐条编号（如 `REQ-FUNC-001`），每条至少包含：
  - `ID`
  - `标题`
  - `需求陈述（shall）`
  - `触发条件/输入`
  - `处理规则`
  - `输出/结果`
  - `异常分支`
  - `验收标准链接`

3.3 质量需求（Quality of Service）：
- 性能、安全、可靠性、可用性、可维护性、可观测性等可量化指标。

3.4 合规与边界（Compliance / Boundary）：
- 合规要求、审计要求、数据边界、跨境、隐私等。

3.5 数据与实现约束（Design & Implementation Constraints）：
- 若涉及数据库新表或字段变更，必须附 SQL：
  - 建表 SQL（`CREATE TABLE`）
  - 字段变更 SQL（`ALTER TABLE`）
- 若涉及新增文件，必须附目录树（`text` 代码块）并标注用途：

```text
app/
└── exchange/
    ├── bizpolicy/
    │   └── context.go                     # 业务上下文与过滤规则定义
    ├── repository/
    │   ├── exchangeBusinessControl.go     # 配置查询仓储
    │   └── model/
    │       └── exchangeBusinessControl.go # 配置实体模型
    └── service/
        └── business_filter.go             # 过滤服务（链/币/渠道）

scripts/
└── 20260416_exchange_business_control.sql # 数据库变更脚本
```

3.6 图形化需求表达（项目增强，强制）：
- 流程图（必填，Mermaid `flowchart` 代码块）。
- 时序图（必填，Mermaid `sequenceDiagram` 代码块）。
- 关键映射表（建议）：角色-职责、状态-动作、模块-边界矩阵。
- 若用户提供需求截图或需外部生成图片素材：
  - 统一保存到 `doc/2-需求/data/`
  - 在主文档直接引用展示（建议相对路径）
  - 示例：`![登录页原型](data/YYYY-MM-DD_HHmmss_<需求中文简介标题>.req-shot-login.png)`

### 4. 验证（SRS 4. Verification）

4.1 验证策略：
- 每类需求采用何种验证方式：测试、分析、检查、演示。

4.2 可验收标准映射：
- 将核心需求条目映射到可验收条件，注明验证入口与预期结果。

4.3 风险与阻断项：
- 当前阻断、潜在风险、缓解动作、责任人、截止时间。

4.4 待确认项与下一步：
- 待确认项（必须可执行）。
- 必要后续动作（仅需求尚未闭环、仍需转缺口/边界/拆分/验收/实施规划，或用户明确要求建议时填写；已闭环时不写）：转缺口、边界、拆分、验收标准、实施规划或直接实现。

### 5. 附录（SRS 5. Appendixes）

5.1 SVG 图清单：
- 流程图：`YYYY-MM-DD_HHmmss_<需求中文简介标题>.flow.svg`
- 时序图：`YYYY-MM-DD_HHmmss_<需求中文简介标题>.sequence.svg`
- 两张 SVG 必须显式白底（建议全画布白色 `rect`，`fill="#FFFFFF"`）。

5.2 图片素材清单（按需）：
- 存放目录：`doc/2-需求/data/`
- 列出文件名、来源、用途、引用位置（章节号）。

5.3 变更追踪：
- 记录关键需求变更、边界变更、验收变更与影响面。

## 使用要求

- 文档路径统一为 `doc/2-需求/YYYY-MM-DD_HHmmss_<需求中文简介标题>.md`。
- 同一个需求只保留一份 `doc/2-需求/` 文档；后续缺口、边界、拆分、变更都继续更新这份文档。
- 复杂流程不得仅用长段文字，必须使用 Mermaid 流程图与时序图。
- 图、表、正文三者必须口径一致；出现冲突时按“先修文档再推进实现”处理。
- 若有截图或外部图片，必须落 `doc/2-需求/data/` 且在 md 内可直接引用展示。
- 若涉及数据库变更，缺 SQL 视为文档未收口。
- 若涉及新增文件，缺目录树视为文档未收口。
- 文档完成后先做轻量自审（占位词、矛盾、歧义、图文一致性），再请用户确认。
