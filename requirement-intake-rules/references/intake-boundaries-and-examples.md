# 需求接入边界与样例

## 应继续留在需求接入的情况

- 只是入口信息零散，需要先整理目标、输入输出、背景和资料来源。
- 任务刚进入研发，还没开始讨论实现边界和验收标准。
- 用户给了需求 URL、零散附件和物料，需要先统一成一份结构化需求文档。
- 当前需求还只是粗略想法，需要先结合项目上下文逐步澄清，而不是直接进入实现。
- 当前存在多个合理方向，需要先做方案对比和推荐结论，再进入后续细化。
- 需要先判断资料是否自洽，再决定能否进入开发。

## 应转交相邻 skill 的情况

- 信息不全且缺字段、缺流程、缺规则：转 `requirement-gap-rules`。
- 范围、兼容性、旧逻辑影响不清：转 `requirement-boundary-rules`。
- 需求过大、跨多个模块或多个实施阶段：转 `requirement-splitting-rules`。
- 需求一次打包多个独立子系统或多个相对独立主线：先转 `requirement-splitting-rules`，再分别细化。
- 编码中出现新条件或优先级变化：转 `requirement-change-rules`。
- 需要把“做到什么算完成”写成可验证标准：转 `acceptance-criteria-rules`。
- 如果本质是历史行为错误或故障：转 Bug 域。

## 应先暂停的情况

- 需求 URL 和附件里的字段定义互相冲突。
- 数据样例、流程说明或物料本身逻辑不成立。
- 一个需求同时覆盖多个独立子系统，但还没先做顶层拆分。
- 关键数据资源明显有问题，继续给实现建议只会放大误导。

## 正例

- 先阅读需求 URL，再把聊天补充、字段说明和截图整理进同一份需求文档。
- 先看当前项目相关模块和已有文档，再一次只问一个关键问题，逐步澄清目的、约束和成功标准。
- 当存在多个合理方向时，先给 2-3 个方案、权衡和推荐结论，再写入同一份需求文档。
- 发现资料中字段口径不一致，先停止实现建议，列出差异并要求补齐。
- 需求整理完成后，将文档落到 `ment/YYYY-MM-DD_需求中文说明.md`，先做占位词/矛盾/歧义/范围自审，再交给用户 review，确认后才进入拆分、边界或实现阶段，且后续 skill 只能继续更新这一份文档。

## 反例

- 看到需求 URL 就直接开始写代码，不整理资料来源和逻辑冲突。
- 一次抛出很多混合问题，用户还没答清前一个问题就继续堆叠假设。
- 明明存在多个合理方向，却不做对比，直接把个人偏好写成需求定论。
- 数据资源明显对不上，仍继续假设字段含义并给出实现方案。
- 需求明明已经过大，却在接入层直接机械拆任务，而不交给 `requirement-splitting-rules`。
- 同一个需求在接入、补齐、拆分、变更阶段分别新建多份文档，导致信息分散、无法追溯。
