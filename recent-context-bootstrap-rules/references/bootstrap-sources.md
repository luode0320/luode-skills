# 近 3 天上下文扫描来源

## 默认时间范围

- 默认只看最近 3 天。
- 优先理解为“从当前时刻向前 72 小时内”的材料。
- 如果用户明确给出日期范围，服从用户范围。

## 默认扫描来源

默认扫描顺序遵循 `artifact-storage-rules/references/path-map.yaml`，当前顺序为：

1. `ment/`
2. `test/`
3. `bug/`
4. `doc/`
5. 最近 3 天的 git 提交

## Skill 目录补充来源

在完成默认扫描来源后，补充加载 skill 索引来源：

1. 当前项目根目录下 `./.skills`
2. 系统的所有 skills

skill 目录的读取粒度默认如下：

- 优先读取各子目录 `SKILL.md` frontmatter（`name`、`description`）和 `agents/openai.yaml`。
- 不默认深读所有 skill body；只有当前任务确实需要时才展开对应 skill 的正文和 references。
- 目录不存在或无权限访问时直接跳过，并在结果中注明“未命中来源”。

## 条件补充来源

如果当前任务明显是在做整项目分析、项目架构梳理、模块总览或启动前快速认识项目，可额外读取根目录项目设计类文档：

- `项目设计.md`
- `架构设计.md`
- `项目大纲.md`
- `系统设计.md`
- `项目说明.md`

这些文件属于弱参考源：

- 不保证是最新版本。
- 优先级低于真实代码、当前有效研发文档和最近 Git 变更。
- 只能帮助快速建立项目总览，不能直接当成当前实现真相。

## 扫描原则

- 目录不存在时直接跳过，不报假错误。
- 优先看最近修改时间更近、文件名更像当前任务的材料。
- 不要一上来深读所有文件，先找最近、最相关的候选。
- git 先看提交标题；只有标题不足以判断时，再看必要的改动线索。
- 如果最近 3 天材料很多，优先保留和当前问题最相关的 3 到 10 条。
- skill 目录优先保留和当前任务直接相关的候选，不要把所有 skill 全量塞进摘要。
- 根目录项目设计类文档只在当前任务需要整项目认知时补充读取，不默认抢占近 3 天扫描结果。

## 结果要求

- 必须明确写出扫描了哪些来源，哪些来源不存在，哪些来源没有发现相关内容。
- 不要把“目录不存在”写成“项目没有这类历史”。
- 不要把旧材料混进最近 3 天摘要里。
- 如果引用了根目录项目设计类文档，必须明确标注“弱参考”。
