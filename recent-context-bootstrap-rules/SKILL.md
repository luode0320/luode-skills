# 新会话上下文预热规则

只在**新会话刚开始**、当前轮缺少历史上下文、但任务明显依赖最近项目动态时使用这个 skill。
如果用户只是闲聊、泛问知识，或者问题与当前项目无关，不要触发它。

## Skill 作用与适用场景

- 在新会话开始时，为当前任务补一层“最近 3 天发生了什么”的前置上下文。
- 优先压缩最近需求、最近 Bug、最近测试、最近文档和最近提交，而不是立刻做深度历史回忆。
- 帮后续需求域、Bug 域、编码域、测试域更快进入正确上下文。
- 在没有明确历史追问的情况下，降低“新会话开局完全失忆”的问题。
- 只做轻量预热，不做长期历史报告，不做完整记忆检索。

## 自动触发信号

- 当前会话刚开始，用户直接问当前项目里的需求、Bug、代码修改、测试或交付问题。
- 用户没有补充足够前情，但当前任务看起来明显依赖最近几天的项目动态。
- 用户说“继续”“接着上次做”“看看最近改了什么再做”，但本轮没有足够上下文。
- 当前任务和仓库本地材料强相关，且先做最近上下文压缩会显著降低误判。

## 进入后先做什么

1. 先确认这是新会话场景，而不是当前会话里已经有足够上下文。
2. 先确认用户问题与当前项目直接相关，不要把泛问题误拉进项目预热。
3. 把扫描范围固定在最近 3 天，避免一上来读太多旧材料。
4. 优先扫描 `artifact-storage-rules` 定义的最近上下文根目录中的材料；目录不存在时直接跳过。
5. 同步查看最近 3 天的 git 提交标题和必要的变更线索。
6. 先输出压缩后的前置上下文，再转交真正的主域 skill 继续处理。

## 默认执行流程

1. 默认先读 `../artifact-storage-rules/references/path-map.yaml`，确认最近上下文扫描根目录。
2. 再读 `references/bootstrap-sources.md`，确认最近 3 天扫描范围、目录优先级和缺省处理方式。
3. 再读 `references/boundary-rules.md`，确认它和历史回忆、项目时间线、需求 / Bug 主流程的边界。
4. 需要统一输出结构时，再读 `references/output-format.md`。
5. 扫描近 3 天的中央约定目录和 git 提交。
6. 输出压缩后的上下文摘要，明确来源、时间范围、相关程度和仍然缺失的信息。
7. 预热完成后，立即转交对应主域 skill，不停留在记忆域里空转。

## 权责边界与不负责事项

- 只负责新会话的最近上下文预热，不负责完整历史回忆，那属于 `history-recall-rules`。
- 不负责长期项目演进报告，那属于 `project-timeline-rules`。
- 不代替当前需求澄清、当前 Bug 定位、当前编码实现和当前测试验证。
- 不把“最近 3 天没有查到明显记录”硬解释成“当前任务没有前置上下文”。
- 不因为做了上下文预热，就跳过后续主域 skill 的正常分析。

## 需要暂停并确认的条件

- 当前任务和本地项目明显无关，继续扫描最近材料只会引入噪音。
- 用户本轮已经给出了足够完整上下文，再做预热反而重复。
- 最近 3 天几乎没有可靠材料，且 git 提交也不足以支撑有价值摘要。
- 最近材料之间明显冲突，无法判断哪条才是当前有效前置结论。

## 执行通过 / 驳回标准

- 通过：能在不拉长上下文的前提下，给出最近 3 天与当前任务最相关的前置信息、来源路径、时间范围和缺失点。
- 驳回：把无关材料硬拼成背景，或把最近活动摘要误当成当前任务已经确认的结论。

## 执行结果归档要求

- 默认不单独新建持久化文档；它本身是基于已有持久化材料做的启动压缩。
- 如果预热结论明显影响当前需求、当前 Bug 或当前测试路径，应写回当前主任务文档，而不是散落在口头说明里。
- 归档内容至少包含：扫描时间范围、主要来源、压缩结论、当前仍缺的信息。

## references 读取规则

- 默认先读 `../artifact-storage-rules/references/path-map.yaml`。
- 再读 `references/bootstrap-sources.md`。
- 只有在判断边界和让路关系时，再读 `references/boundary-rules.md`。
- 只有在需要统一摘要格式时，再读 `references/output-format.md`。
