# 日志追踪检查镜像规则

## 目标

只检查当前日志与链路追踪是否满足 `logging-trace-rules` 的标准，不执行修改。

## 源规则

以下判定严格沿用源 skill：

- 后端日志必须使用项目日志框架，不能用控制台打印替代
- 业务日志默认中文
- 日志初始化必须先 LoadConfig，再 InitLogger，且只初始化一次
- 禁止空配置预初始化
- 业务日志必须带 `[业务简要说明]` 前缀
- trace / span / 审计 / 脱敏边界必须清晰

## 自动触发信号

- 新增或修改 logger、日志字段、日志模板。
- 新增或修改日志初始化、日志配置项、日志目录或日志输出目标。
- 新增或修改 trace-id、span-id、链路上下文透传。
- 新增或修改审计日志、排障日志、debug 日志。

## 进入后先做什么

1. 先判断当前记录点属于业务过程、调试排障还是审计留痕。
2. 确认当前服务使用的日志框架和项目封装。
3. 确认日志配置和初始化顺序。
4. 确认不存在控制台打印替代正式日志。

## 默认执行流程

1. 先读取 `logging-trace-rules`。
2. 如需判断框架和配置，再读源 skill 的对应 references。
3. 如需判断记录点和正反例，再读源 skill 的对应 references。
4. 输出日志框架、配置来源、初始化顺序、日志类型、字段和透传边界。
5. 不直接改代码，只输出缺口。

## 权责边界与不负责事项

- 不代替 `error-handling-rules` 设计异常处理路径。
- 不代替监控、告警、报表平台设计。
- 不允许以控制台打印替代正式日志框架。
- 不把所有调试打印都合理化成正式业务日志。

## 需要暂停并确认的条件

- 日志内容包含敏感信息、隐私信息或高风险字段。
- trace 透传会影响上下游约定或旧链路兼容。
- 当前日志方案无法说明配置来源，或日志仍只输出到控制台。

## 执行通过 / 驳回标准

- 通过：能明确日志框架、配置来源、初始化顺序、日志类型、关键字段、记录点、trace 透传方式和脱敏边界。
- 驳回：控制台打印替代正式 logger、缺少配置化管理、配置加载前初始化 logger、重复初始化、日志类型混乱或敏感信息裸打。

## 执行结果归档要求

- 如果本次定义了日志字段规范、trace 透传方案或审计边界，将结论记录到 `analysis/` 或 `review/`。
- 归档内容至少包含日志类型、关键字段、透传方案、脱敏说明和兼容影响。
- 如果只是沿用既有清晰模式且无争议，可以不单独归档。

## 读取规则

- 默认先读 `logging-trace-rules`。
- 只有在判断框架和配置时，再读源 skill 的对应 references。
- 只有在判断记录点和正反例时，再读源 skill 的对应 references。
