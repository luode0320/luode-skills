# 证据与记忆回写规则

## 证据分级

- 强证据：当前项目代码、数据库 schema、迁移脚本、配置文件、官方接口文档、官网或自有站点内容、真实命令输出、用户明确确认。
- 中证据：项目设计文档、历史需求文档、测试记录、审查记录、同仓库相似实现。
- 弱证据：历史说明、命名相似、注释暗示、公共 GitHub 仓库、社区文章、第三方公开文档、用户未验证的外部线索。

输出方案时必须标注证据等级。弱证据只能支撑候选方向，不能写成最终事实。

## 用户补充线索处理

- 本地路径：先确认路径存在，再只读扫描目录结构和入口文档。
- 其他项目：先查项目说明、配置、路由、模型、接口和 README，再判断是否与当前 idea 相关。
- URL：必须真实打开或请求；失败时记录失败原因，不凭 URL 名称猜内容。
- GitHub：优先区分自有仓库与公共仓库；公共仓库默认只作补充证据，不能单独定案。
- 数据库：默认只查连接线索和 schema；生产数据查询、写入或高成本查询必须先确认。
- 第三方服务：优先查项目中的 SDK / client / 配置和官方文档；付费或生产调用必须先确认。

## 必须回写长期记忆的线索

- 已验证可复用的本地项目路径。
- 已验证可复用的 URL、接口文档或第三方文档入口。
- 已验证相关的数据库、表、字段、索引或 schema 位置。
- 已验证相关的服务名、配置项、SDK、上下游调用入口。
- 本次侦察证明有用的搜索关键词、目录入口或代码入口。
- 被证明无效但容易重复误查的路径或资料源，可记录为废弃或不适用线索。

## 不得回写的内容

- 未访问成功的 URL 内容猜测。
- 未读取真实文件的路径描述。
- 未确认的数据库字段含义。
- 只服务本轮一次性输出、后续不可复用的临时推理。
- 敏感凭据、token、密码、生产连接串原文。

## 回写格式建议

交给 `project-memory-rules` 时，按以下信息组织：

- `别名`: 便于后续搜索的中文名和代码名。
- `类型`: 资料入口、数据库线索、服务入口、项目路径、URL、搜索经验、废弃线索。
- `定义`: 说明这个线索能帮助解决哪类需求。
- `来源`: 文件路径、URL、命令或用户确认来源。
- `适用范围`: 当前项目、某业务域、某类需求或跨项目。
- `状态`: 启用、待验证、废弃。
