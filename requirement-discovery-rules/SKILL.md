# 需求主动侦察规则

只在“用户给了想法，但资料还没有整理好”时使用这个 skill。
目标是让用户像老板一样只给方向，agent 负责先找资料、找数据、看代码、查上下游、比对方案，再把可确认的需求设计交给用户。

## 核心原则

- 先侦察，再追问；不能因为 idea 粗略就立刻把大量细节问题抛回给用户。
- 只读优先；读取代码、文档、配置、schema、URL 和本地项目可以主动做，写数据库、调用生产接口或执行高风险动作必须暂停确认。
- 证据优先；每个关键判断都要能追溯到真实文件、命令输出、数据库结构、接口文档、URL 内容或用户明确补充。
- 不脑补；没有证据的内容只能写成候选方案、待验证假设或需用户拍板的问题。
- 可复用线索要沉淀；找过且有价值的路径、数据库、URL、项目、服务、表、接口和侦察经验，必须转交 `project-memory-rules` 更新到 `PROJECT_MEMORY.md`。

## 自动触发信号

- 用户只说一个想法，例如“想做用户活跃分析”“做一个订单异常预警”“后台加一个运营看板”。
- 用户明确说“不想先写完整需求，希望 agent 自己找资料 / 找数据 / 看代码 / 查上下游”。
- 用户补充“某个路径还有项目”“某个 URL 里有资料”“某个数据库可以查”“第三方其实也是我们的项目”。
- 当前需求信息缺失，但缺失项可能通过当前项目、数据库、文档、历史需求或其他本地项目主动获得。
- `requirement-intake-rules` 或 `requirement-gap-rules` 准备向用户追问大量上下文前，应先转入本 skill 做侦察。

## 进入后先做什么

1. 用一句话复述原始 idea 和当前理解，不扩写成确定需求。
2. 列出本轮可用侦察入口：当前仓库、`PROJECT_MEMORY.md`、`PROJECT_STYLE.md`、`doc/需求/`、`doc/架构/`、`doc/阶段/`、代码目录、配置文件、数据库连接线索、第三方服务线索、用户补充路径或 URL。
3. 先读长期记忆，检查是否已有相关数据库、路径、URL、服务或历史需求线索。
4. 按只读方式侦察项目事实，优先找相似功能、相关表、相关接口、相关页面、相关配置和上下游调用。
5. 把证据分成“已确认事实”“可复用素材”“候选方案”“仍需用户拍板的问题”。
6. 形成 2-3 个可行方案；若只有一个明显方案，也要说明为什么其他方向不合适。
7. 输出推荐方案、依据、风险和下一步应转交的需求域 skill。

## 默认执行流程

1. 默认先读 `references/discovery-checklist.md`，确定本轮侦察范围。
2. 需要判断证据可信度、用户补充路径 / URL 如何使用、以及哪些线索要写入长期记忆时，再读 `references/evidence-and-memory.md`。
3. 需要判断需求域相邻 skill 的承接顺序、是否应该合并或让路时，再读 `references/requirement-domain-routing.md`。
4. 需要输出需求设计草案时，再读 `references/output-template.md`。
5. 先从当前项目和长期记忆开始，不要求用户先提供完整材料。
6. 如果用户提供额外路径、URL、数据库名、服务名或项目名，把它们作为新增侦察入口；无法访问时记录阻断原因。
7. 对每个重要结论标注来源，不允许用“应该”“大概”“看起来”替代证据。
8. 完成侦察后，把可复用线索交给 `project-memory-rules`，按原条目回写或新增长期记忆。
9. 若已经形成可开发需求，转交 `requirement-intake-rules` 生成或更新同一份需求主文档。
10. 若发现资料缺口仍阻断，转交 `requirement-gap-rules`，但只保留真正无法通过侦察解决的问题。
11. 若多个方案的业务取舍影响明显，输出最少拍板问题，而不是继续替用户猜。

## 权责边界与不负责事项

- 只负责 idea 到需求设计前的主动侦察、资料收集、方案比较和证据整理。
- 不直接替代 `requirement-intake-rules` 落正式需求主文档。
- 不替代 `requirement-boundary-rules` 裁决是否改旧逻辑或纳入历史问题。
- 不替代 `acceptance-criteria-rules` 写验收标准。
- 不替代 `implementation-plan-rules` 写文件落点、阶段和编码步骤。
- 不把无证据推断写入 `PROJECT_MEMORY.md`；只有验证过、后续可复用的线索才回写长期记忆。
- 不把生产数据库查询、生产接口调用、外部付费 API 调用当作默认侦察动作；需要时先确认安全边界。

## 需要暂停并确认的条件

- idea 本身连目标对象都无法判断，无法确定侦察方向。
- 需要访问用户未授权的私有仓库、生产数据库、付费接口、登录态页面或敏感凭据。
- 多个可行方案都成立，但业务目标、成本、风险或优先级差异明显，需要用户拍板。
- 侦察到的资料互相冲突，无法通过当前项目证据判断哪条是权威口径。
- 用户补充路径或 URL 无法访问，且当前项目内没有替代资料。

## 执行通过 / 驳回标准

- 通过：已经主动侦察当前项目和用户提供线索，输出有证据来源的资料清单、项目现状、可用数据 / 代码 / 上下游、方案对比、推荐方案、风险和最少待确认问题，并转交长期记忆回写可复用线索。
- 驳回：用户只给粗略 idea 时直接要求用户补完整需求；未查项目就列大段猜测；关键结论没有来源；找过的有价值路径、数据库、URL 或项目线索没有进入长期记忆回写流程。

## 执行结果归档要求

- 侦察结论应回流到同一份需求主文档；如果需求主文档还不存在，转交 `requirement-intake-rules` 创建。
- 可复用侦察线索必须通过 `project-memory-rules` 写入 `PROJECT_MEMORY.md`，字段至少包含别名、类型、定义、来源、适用范围、更新时间和状态。
- 不单独新建“侦察文档”作为平行入口，除非 `artifact-storage-rules` 后续定义了专门产物类型。

## references 读取规则

- 默认先读 `references/discovery-checklist.md`。
- 只有在处理证据可信度、用户补充路径 / URL、长期记忆回写时，再读 `references/evidence-and-memory.md`。
- 只有在判断需求域相邻 skill 的承接顺序、合并边界或下一步流转时，再读 `references/requirement-domain-routing.md`。
- 只有在组织输出结构时，再读 `references/output-template.md`。
