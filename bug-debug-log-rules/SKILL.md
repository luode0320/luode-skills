# Bug 调试日志规则

只在需要通过临时日志补充运行期证据，并且已有明确观察目标时使用这个 skill。
如果当前还没有明确观察目标，请先转 `bug-runtime-debug-rules`；如果需要正式日志治理，请转交 `logging-trace-rules`。

## Skill 作用与适用场景

- 用最小可回收的临时日志补足运行期证据。
- 约束日志放在哪里、打什么、打到什么粒度就够。
- 防止把诊断日志写成长期噪音或泄露敏感信息。
- 为后续清理、降级或保留提供明确收口要求。

## 自动触发信号

- 需要观察关键变量、条件分支、上下游请求响应或时间线。
- 断点不方便或线上问题需要通过日志补证。
- 需要区分多个根因假设，但现有日志不足。
- 团队准备临时加打印，但尚未明确哪些日志是必须的。

## 进入后先做什么

1. 先明确本轮日志要验证什么假设、观察哪个状态。
2. 选择最靠近证据点的最小日志落位。
3. 控制日志粒度、输出量、敏感字段和生命周期。
4. 提前约定日志使用后如何删除、降级或转正式观测点。

## 默认执行流程

1. 默认先读 `references/debug-log-placement.md`，先判断日志应落在哪些关键证据点。
2. 如需继续展开，再读 `references/debug-log-cleanup.md`，需要规划日志生命周期和回收方式。
3. 需要对照边界或正反例时，再读 `references/debug-log-examples.md`，需要对照日志诊断正反例。
4. 输出 调试日志方案、落点说明和清理要求。
5. 日志加完后继续回到 `bug-runtime-debug-rules` 或 `bug-root-cause-rules` 使用证据定位；诊断完成后必须交给 `cleanup-format-review-rules` 检查清理状态。

## 权责边界与不负责事项

- 只负责临时诊断日志，不替代正式日志和 trace 设计。
- 不在没有明确假设时大面积撒日志。
- 不输出敏感信息、全量大对象或高频无界日志。
- 不把临时日志直接留在交付代码中而不说明去留。

## 需要暂停并确认的条件

- 当前没有明确观察目标，日志只是为了“看看会不会发现什么”。
- 准备输出敏感信息、账号标识或高风险数据。
- 日志落点过多、粒度过细，明显会制造噪音或性能风险。
- 诊断结束后没有清理方案。

## 执行通过 / 驳回标准

- 通过：能够说明为什么要加这些日志、放在哪里、预计观察什么、如何控制风险，以及诊断后如何回收。
- 驳回：为了图方便随手加打印，没有明确目标、没有清理计划，或日志量和敏感性失控。

## 执行结果归档要求

- 将日志方案、落点、观察目标和清理状态记录到 `analysis/`。
- 归档内容至少包含日志目的、位置、输出字段、使用结果和去留结论。
- 如果有日志被保留为正式观测点，应写明理由并转交正式日志规则复核。

## references 读取规则

- 默认先读 `references/debug-log-placement.md`。
- 只有在 确定日志清理、降级或保留策略 时，再读 `references/debug-log-cleanup.md`。
- 只有在 对照样例或判断日志是否过量 时，再读 `references/debug-log-examples.md`。
