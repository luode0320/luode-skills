# 命名基本原则

- 名字应直接表达职责、对象和范围。
- 内部代码标识符默认以驼峰命名族为主，不混用 `snake_case`、`kebab-case` 和驼峰。
- 变量、函数、方法、字段、事件、任务、测试标识符优先使用小驼峰。
- 类、接口、类型、枚举、DTO 等类型名优先使用大驼峰。
- 文件名、目录名和包名优先服从语言生态或项目既有结构规则，不因为“偏好驼峰”而反向破坏现有约定。
- 优先用稳定业务术语，不随意发明近义词。
- 缩写只在团队已有稳定约定时使用。
- 外部接口字段、数据库字段、消息体键名、配置协议等如果已有固定格式，沿用外部约束，在映射层与内部驼峰命名做隔离。
- 避免 `data`、`info`、`util`、`handler2` 这类泛化命名。
