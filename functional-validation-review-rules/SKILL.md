# 功能验证审查镜像规则

## 目标

只检查当前功能验证方案或结论是否满足 `functional-validation-rules` 的要求，不执行验证和结论回写。

## 源规则

以下判定严格沿用源 skill：

- 只验证当前需求、当前修改和当前验收标准
- 不主动扩张到全量历史回归
- 验证范围、验证步骤、通过 / 驳回标准和留痕方式应明确
- 结论应写入中央约定 README，详细证据应落到 ASCII 镜像路径
- 禁止为了测试目的污染生产代码
- 前后端联调阶段应使用 `agent-browser`

## 自动触发信号

- 需要确认某个接口、页面、交互链路、输入输出或异常分支是否满足当前预期。
- 代码已完成并通过基础审查，准备进入功能验证。
- 功能验证失败后，需要先判断结论是否来自正确的验证范围和标准。
- 功能验证留痕准备散落到错误目录。
- 主线程已声明使用 `functional-validation-rules`，但验证收口风险较高。

## 进入后先做什么

1. 先确认当前需求、当前修改和验收标准是否已经明确。
2. 检查本轮是否只验证当前变更本身，没有扩张到历史回归。
3. 检查验证对象、验证步骤和通过 / 驳回标准是否清楚。
4. 检查 README 结论与 ASCII 镜像路径证据落点是否明确。
5. 检查是否存在联调遗漏或测试污染风险。

## 默认执行流程

1. 先读取 `functional-validation-rules`。
2. 需要核对验证范围时，再读源 skill 的 `references/validation-scope.md`。
3. 需要核对功能验证、回归验证和联调边界时，再读源 skill 的 `references/validation-boundaries.md`。
4. 需要核对模板和样例时，再读源 skill 的 `references/validation-template-and-examples.md`。
5. 输出通过项、范围扩张问题、判定标准缺口、留痕缺口、联调遗漏和测试污染风险。
6. 不直接执行验证，不直接写 README，只输出问题和建议。

## 权责边界与不负责事项

- 不代替 `test-strategy-rules` 做测试优先级设计。
- 不代替 `test-regression-rules` 做旧能力回归验证。
- 不代替测试资源类 skill 管理目录、命名和测试程序。
- 不代替 `agent-browser` 执行浏览器联调。
- 不执行功能验证结论回写。

## 需要暂停并确认的条件

- 当前需求、验收标准或预期结果本身不明确。
- 当前失败现象更像联调问题或上下游环境问题。
- 当前验证范围已经扩张成回归检查或全链路稳定性检查。
- 当前测试资源或前置环境尚未准备好。

## 执行通过 / 驳回标准

- 通过：验证范围、验证步骤、通过 / 驳回标准、留痕方式和联调要求完整，且没有发现测试污染或范围误扩张。
- 驳回：验证结论无法回溯到明确范围和证据，留痕落点错误，或为了验证通过引入测试专用生产代码。

## 执行结果归档要求

- 如本轮审查发现验证范围、联调要求、留痕或判定标准缺口，应将结论记录到 `analysis/` 或评审记录。
- 归档内容至少包含验证对象、发现的问题、建议流转方向和证据缺口。
- 如果只是一次轻量预审且无争议，可不单独归档，但应保证评审可追溯。

## 读取规则

- 默认先读 `functional-validation-rules`。
- 只有在核对验证范围时，再读 `references/validation-scope.md`。
- 只有在核对职责边界和问题归属时，再读 `references/validation-boundaries.md`。
- 只有在核对结论模板和样例时，再读 `references/validation-template-and-examples.md`。
