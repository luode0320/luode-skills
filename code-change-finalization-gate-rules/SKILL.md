# 代码变更最终收口闸门规则

只在“本轮存在代码或测试改动，且准备最终回复”时使用这个 skill。
目标不是重复实现业务，而是在收口前确认代码改动相关的终检闸门确实被执行。

本 skill 与 `skill-execution-compliance-gate-rules` 是同源拆分：本 skill 只负责代码/测试改动的最终收口闸门，其他已命中 skill 的整体执行完整性检查、非预期工具/命令/API/模型执行失败的分类、Skill 执行证据与运行时状态真收口判断由 `skill-execution-compliance-gate-rules` 负责，两者互不重复检查。

## Skill 作用与适用场景

- 当本轮发生代码新增或修改且准备最终回复时，作为默认收口闸门触发，防止遗漏终检。
- 在最终回复前补一条注释终检：凡本轮改动位点都要核验 `comment-placement-granularity-rules` 与 `comment-completion-gate-rules` 是否已执行，既检查注释必要性 / 放置位置 / 字段注释 / 过期注释治理，也检查步骤流程至少 `1`（多步骤 `1/2/3`）编号、方法更新时间、更新时间后的改动原因与 `[参数]` / `[返回]`。
- 在最终回复前注释终检必须产出“函数注释核对清单”；若本轮存在补丁位点，还必须产出“补丁注释核对清单”，没有清单视为注释闸门未执行完成。
- 函数注释核对清单 / 补丁注释核对清单属于注释闸门的内部执行证据，可并入最终回复的「Skill 执行证据」呈现；按 `reasoning-summary-structure-rules` 的总结视觉规范，不再要求在最终总结中单列“函数注释核对”展示节点（清单本身仍必须产出，不得省略）。
- 在最终回复前补一条基础实现终检：凡本轮存在代码新增或修改，都要核验是否已执行 `implementation-review-rules` 的最低测试前收口（至少覆盖基础格式、语法/类型/引用、目录归位/依赖方向）。
- 当用户请求“补充注释/只补注释/注释完善”时，强制检查注释是否优先覆盖“未提交且有改动”的代码位点，并核验函数头注释与方法块注释完整性。
- 在最终回复前强制增加测试目录闸门：本次新增测试文件路径必须全部位于同一个“当天时间戳目录”，否则判定 skill 未完成并禁止进入提交阶段。
- 在最终回复前强制增加“真实运行验证闸门”：凡本轮存在功能代码新增或修改，必须核验核心接口、页面、导出、查询、提交或任务入口是否完成真实运行验证；若只完成 `build`、lint、静态搜索、代码审查或实现自审，只能标记为“仅静态验证”，不得给“功能 / 接口已验证可用”的完成结论。
- 若真实运行验证因本地环境、授权、样本数据或依赖服务不足无法执行，最终回复必须明确列出“未完成真实验证”的路径、阻断原因和人工验证步骤；核心路径未运行且无人工交接时，判定为阻断级缺口。
- 在最终回复前强制增加“用户手改保护闸门”：若本轮出现“用户手动修改代码且与 AI 记忆/旧上下文冲突”的迹象，必须确认已执行 `code-context-resync-rules`，且最终 diff 未回退用户手改内容，否则判定未通过。

## 自动触发信号

- 本轮发生代码改动，且需要确认是否漏触发 `comment-placement-granularity-rules` 与 `comment-completion-gate-rules`。
- 本轮发生代码新增或修改，且进入最终回复前收口。

## 进入后先做什么

1. 对测试相关改动执行当天时间戳目录一致性检查：新增测试文件是否全部位于同一个“当天时间戳目录”。
2. 对代码改动执行注释闸门检查：先核验 `comment-placement-granularity-rules` 是否已覆盖注释必要性、放置位置、字段注释与过期注释治理，再核验 `comment-completion-gate-rules` 的改动位点注释、步骤流程至少顶层 `1` 就近注释（多步骤补齐 `1/2/3`，复杂步骤可含 `1.1/1.2` 细分）、`[参数]` / `[返回]`、最近修改时间、最近修改时间后的改动原因是否齐全；若方法体存在步骤流程却无顶层 `1` 编号，直接判定未通过。
2.1 注释闸门检查完成后，必须输出“函数注释核对清单”；若本轮改动不涉及函数/方法，必须明确写出“函数位点 0 个”。
2.2 若本轮涉及原有方法补丁分支、兜底或兼容逻辑，必须额外输出“补丁注释核对清单”；若无补丁位点，明确写出“补丁位点 0 个”。
3. 若用户请求补注释，额外执行优先级闸门检查：未提交改动位点是否已优先补齐；函数、方法是否同时具备函数头注释与方法块注释。
4. 对代码改动执行基础实现闸门检查：核验 `implementation-review-rules` 是否已完成最低测试前收口（至少覆盖基础格式、语法/类型/引用、目录归位/依赖方向）；未执行则判定未通过。
5. 对功能代码改动执行真实运行验证闸门：核验是否已有 `functional-validation-rules` 或等价测试证据证明核心路径真实运行；若证据只覆盖构建 / 静态检查，必须降级为“仅静态验证”并检查是否输出人工验证交接。
6. 对用户手改保护执行闸门检查：若出现上下文冲突，必须核验是否已重读基线并保留用户手改位点；若缺失则判定未通过。

## 默认执行流程

1. 先判定本轮是否发生代码新增或修改；若是且准备最终回复，立即进入本闸门流程，不得跳过。
2. 若本次改动包含 `internal/router`，执行路由风格与约束闸门检查。
3. 若本次涉及新增测试文件，强制检查这些文件路径是否全部位于同一个“当天时间戳目录”；若发现跨天目录写入，直接判定 skill 未完成，禁止进入提交阶段。
4. 最终回复前必须确认：本轮改动位点已完成注释双 skill 收口，即 `comment-placement-granularity-rules` 的必要性 / 放置位置 / 字段注释 / 过期注释治理检查与 `comment-completion-gate-rules` 的步骤流程至少顶层 `1`（多步骤 `1/2/3`）就近注释、`[参数]` / `[返回]`、最近修改时间、最近修改时间后的改动原因、函数注释顺序检查、补丁位点“做了什么 + 为什么要加”注释检查均已完成，且已输出函数注释核对清单（及需要时的补丁注释核对清单）。
5. 最终回复前必须确认：本轮代码改动已完成 `implementation-review-rules` 的最低测试前收口（至少基础格式、语法/类型/引用、目录归位/依赖方向）。
6. 最终回复前必须确认：本轮功能代码改动的核心路径已完成真实运行验证；若仅完成构建、静态分析、代码审查或实现自审，最终结论必须写成“仅静态验证，未完成真实运行验证”，并列出人工验证交接，不得写成“接口可用 / 功能可用 / 已完整验证”。
7. 若本轮为补注释请求，最终回复前必须确认：未提交改动位点优先补注释已完成，且相关函数、方法同时满足“函数头注释 + 方法块注释”。
8. 最终回复前必须确认：若本轮存在用户手改与 AI 上下文冲突，`code-context-resync-rules` 已执行完成，且无用户手改被回退。

## 提交前闸门检查（internal/router）

- 若本次改动包含 `internal/router`，必须检查：
- 不存在 `.GET(`、`.PUT(`、`.PATCH(`、`.DELETE(`。
- 不存在 `:id` 或 `{id}` 路径参数。
- 不存在 `registerPostRoutes` 或 route 列表驱动注册模式。
- 路由 path 未通过本地常量引用（允许全局根前缀常量）。
- 任一不满足则阻断交付，先回改再继续。

## 阻断判定与处理

属于阻断级：

- 新增测试文件出现跨天目录写入，或未全部位于同一个“当天时间戳目录”。
- 本轮代码改动未完成 `comment-placement-granularity-rules` 与 `comment-completion-gate-rules` 的注释双 skill 闸门。
- 本轮代码改动未输出函数注释核对清单（或未声明函数位点 0 个）。
- 本轮存在补丁位点但未输出补丁注释核对清单（或未声明补丁位点 0 个）。
- 本轮代码改动未完成 `implementation-review-rules` 的最低测试前收口闸门。
- 本轮功能代码改动的核心接口、页面、导出、查询、提交或任务入口未完成真实运行验证，且最终回复未明确降级为“仅静态验证 / 未完成真实验证”并给出人工验证交接。
- 本轮仅执行 `build`、lint、静态搜索、代码审查或实现自审，却在最终结论中声称“接口可用 / 功能可用 / 已完整验证”。
- 补注释请求中，未优先覆盖未提交改动位点，或函数、方法缺函数头注释/缺方法块注释。
- 本轮已出现“用户手改与上下文冲突”但未执行 `code-context-resync-rules` 重同步，或最终 diff 回退了用户手改内容。

属于非阻断级：

- 缺口不影响当前结论正确性，但应在后续迭代补齐。
- 可先完成当前任务收口；只有用户明确要求建议/backlog，才把这类内容作为后续内容输出。

## 权责边界与不负责事项

- 只负责代码/测试改动的最终收口闸门，不代替具体业务实现。
- 不强制要求一次性补齐所有历史遗留规则，优先处理当前阶段可执行缺口。
- 不把“当前阶段不适用”的规则误报为未执行缺口。
- 不输出冗长核对报告，除非用户明确要求展开明细。
- 不因为执行本 skill 而丢弃其他来源的主要建议。
- 其他已命中 skill 的整体执行完整性检查、非预期工具/命令/API/模型执行失败的分类、Skill 执行证据与运行时状态真收口判断由 `skill-execution-compliance-gate-rules` 负责，本 skill 不重复检查。

## 执行通过 / 驳回标准

- 通过：已完成 `comment-placement-granularity-rules` 与 `comment-completion-gate-rules` 注释终检、函数注释核对清单输出、补丁位点场景下的补丁注释核对清单输出、`implementation-review-rules` 最低测试前收口终检、功能代码改动的真实运行验证状态核验；若存在原执行计划内未完成必需项、阻断项，或用户明确要求后续建议/backlog，则输出合并后的 1-3 条后续内容；若不存在三类合法后续，则强制无下一步并直接结束。
- 驳回：发现新增测试文件跨天目录写入仍继续推进提交阶段；或本轮代码改动未完成注释补齐、未输出函数注释核对清单、补丁位点缺补丁注释核对清单、最低测试前收口未完成、核心路径真实运行验证状态缺失，仍给出“可继续 / 已完成 / 已验证可用”结论；或本轮发生代码新增/修改且已到最终回复前仍未触发本闸门。
- 驳回：补注释请求中未先处理未提交改动，或函数、方法注释缺函数头/方法块任一项，仍给出“可继续”结论。
- 驳回：已识别用户手改冲突，但未执行 `code-context-resync-rules` 或执行后仍回退用户手改。

## 回到主流程的重启点

- 若为阻断级：先执行“后续内容”中的高优先级动作，再从当前阶段入口重新开始。
- 若为非阻断级：当前任务可收口；若用户未显式要求建议/backlog，则直接结束，不默认追加后续区块。

## 输出要求（简化版）

本 skill 不拥有面向用户的“后续内容”或“任务阻断收口”模板。真实阻断时只输出合规 PASS/FAIL 与共享契约缺失项；最终用户区块由 `reasoning-summary-structure-rules` 唯一渲染。

约束：

- 上述结构是普通 Markdown 示例，不得包进 ` ```text ` 或其他代码围栏；只有真实代码、命令、配置、日志、JSON/YAML 等片段才使用代码围栏。
- 合规结论必须可追溯到原执行计划、完成定义、验收标准或共享阻断契约，不给泛化空话。
- `limited/not_applicable` 不得被本 skill 升级为真实阻断；若主任务无阻断级缺口和计划内未完成必需项，且用户未显式要求建议/backlog，默认直接结束。
- 可选优化、额外整理、未来迭代、体验提升、文档再润色等内容，若不属于原计划必需项，不得作为默认后续内容输出。
- 禁止为了占位输出“等待用户新指令”“无需继续动作”“当前无需补充动作”“无下一步”等文案；无下一步时直接结束。
