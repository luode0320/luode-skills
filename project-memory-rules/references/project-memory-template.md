# 项目记忆模板

把这份模板作为项目根目录唯一长期记忆主文件 `PROJECT_MEMORY.md` 的基线结构。该文件固定采用“人类阅读区 + 底部机器索引区”的单文件双区模型。

## 目标

- 只保留一份权威长期记忆主文件。
- 只写明确、可直接复用的事实。
- 让后续对话、代码阅读和其他 skill 共享同一套长期记忆。
- 让机器索引区负责结构化检索、人类阅读区负责稳定表达。

## 人类阅读区模板

建议按以下章节组织正文区：

1. `核心记忆`
2. `术语表`
3. `指标与口径`
4. `表与字段映射`
5. `缓存、变量与常量映射`
6. `方法与调用映射`
7. `计算规则`
8. `业务约束`
9. `废弃术语`
10. `变更记录`

正文区条目仍建议使用中文字段：

```md
### 活跃用户
- 别名: 登录用户, active user
- 类型: 指标
- 定义: 按登录行为统计的活跃用户数
- 来源: `UserService.CountActiveUsers()`
- 适用范围: 日报
- 更新时间: 2026-07-03
- 状态: 启用
```

字段规则：

- 标题、章节名、条目标题和字段名必须使用中文。
- 英文只允许作为代码符号、文件名、数据库字段名、原始接口字段名或别名保留。
- 人类阅读区负责总结稳定事实，不强制展开全部机器字段。

## 机器索引区模板

`PROJECT_MEMORY.md` 底部固定追加以下受管区块：

````md
## 机器索引区

```yaml
version: 1
entities: []
relations: []
evidence: []
contexts: []
lifecycle:
  active: []
  deprecated: []
  stale: []
  conflicted: []
  retired: []
retrieval_hints:
  aliases: {}
  scopes: {}
  sources: {}
extensions:
  external_refs: []
  retrieval_provider: ""
  vector_doc_id: ""
  graph_node_id: ""
```
````

要求：

- 标题固定为 `## 机器索引区`，不得改名。
- 使用 `yaml` fenced block，便于后续 agent 稳定解析。
- 若区块缺失，先补骨架；若区块已存在，只做幂等更新。
- 机器索引区属于主文件内嵌受管区，不是第二个主文件。

## 双区同步规则

1. 先更新机器索引区，再同步人类阅读区。
2. 机器索引区负责实体、关系、证据、上下文、生命周期和检索提示。
3. 人类阅读区负责主题归纳、中文可读性和长期浏览体验。
4. 同一事实重复出现时，优先更新原实体；人类阅读区也优先回写原词条。
5. 机器索引区存在 `conflicted`、`stale`、`deprecated` 或 `retired` 状态时，人类阅读区必须同步其业务含义。
6. 只有在更新会影响后续理解时，才追加 `变更记录`。

## 增量更新清单

- 先读整个 `PROJECT_MEMORY.md`，不要只读单条词条。
- 先定位 `## 机器索引区`；缺失则补骨架。
- 先用 `entity_id + 类型 + 作用域 + 别名 + 来源` 定位原实体。
- 机器层完成更新后，再同步正文区。
- 证据不足、身份不明或来源冲突时，不得直接写成人类正文真相。
- 旧正文可保留并逐步迁移，不要求一次性全量结构化。

## 什么算长期记忆

- 反复被提到的指标定义。
- 从代码中读出的字段含义。
- 具有稳定业务意义的缓存键、变量或常量。
- 计算公式、业务规则、状态流转。
- 值得长期记住职责的方法、脚本、模块和流程节点。

## 什么不该写进来

- 未确认的猜测。
- 临时讨论笔记。
- 重复的原始聊天记录。
- 与 `PROJECT_MEMORY.md` 并行存在的第二份长期记忆根文件。
