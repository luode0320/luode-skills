# 抽取与回写流程

## 目标

统一长期记忆的抽取、定位、合并和回写顺序，避免“直接改正文词条”的旧心智。

## 标准流程

1. 读取 `PROJECT_CURRENT.md`，确认当前任务、边界、待办和阻断。
2. 读取整个 `PROJECT_MEMORY.md`。
3. 定位底部 `## 机器索引区`。
4. 若机器索引区缺失，补齐最小骨架。
5. 从代码、项目文档和当前对话中提取候选稳定事实。
6. 为候选事实补齐类型、作用域、证据、上下文和生命周期判断。
7. 按 `entity_id + 类型 + 作用域 + 别名 + 来源` 查找现有实体。
8. 优先更新原实体；只有未命中时才新增实体。
9. 同步补齐关系、证据和上下文索引。
10. 机器索引区更新完成后，再同步人类阅读区。
11. 当前任务状态覆盖写入 `PROJECT_CURRENT.md`，检查 UTF-8 字节数不超过 51,200。
12. 关键历史事件追加到 `PROJECT_HISTORY.md`，不得覆盖旧记录。
13. 若本次更新会影响后续理解，再更新 `变更记录`。

## 合并优先级

1. 当前项目代码
2. 最近明确对话
3. 已有项目文档
4. 旧记忆内容

## 阻断条件

- 事实来源不明确
- 证据缺失
- 身份无法稳定定位
- 两个高优先级来源互相冲突且无法裁决

出现阻断条件时，不得直接写入“启用事实”。

## 文件职责冲突处理

- 当前状态、待办、阻断和下一执行点只能进入 `PROJECT_CURRENT.md`。
- 稳定规则、关键决策和长期事实进入 `PROJECT_MEMORY.md`，并按机器索引区规则合并。
- 重要历史事件进入 `PROJECT_HISTORY.md`，只追加；不得用历史流水污染当前状态或稳定记忆。
- `PROJECT_CURRENT.md` 超过 51,200 字节时阻断写入并先压缩，禁止静默截断。
