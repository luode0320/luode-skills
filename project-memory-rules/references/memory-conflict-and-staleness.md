# 冲突与失效处理

## 目标

定义长期记忆中状态变化、证据冲突和旧口径失效时的处理方式。

## 生命周期状态

| 状态 | 含义 | 处理方式 |
| --- | --- | --- |
| `active` | 当前有效且证据明确 | 可同步到人类阅读区 |
| `deprecated` | 旧口径仍可参考，但不再推荐 | 在正文区标注废弃或旧口径 |
| `stale` | 证据过期、样本失真或近期验证失败 | 不能继续当作当前真相使用 |
| `conflicted` | 多个高优先级来源冲突且未裁决 | 必须保留冲突标记，不得直接覆盖 |
| `retired` | 已退役、仅保留历史记录 | 只保留历史背景，不再参与当前推理 |

## 典型场景

### 口径更新

- 旧规则被新规则替代
- 处理：原实体更新为新定义，旧定义进入 `deprecated` 或 `retired`

### 测试再次失败

- 之前可复用的数据、参数或事实不再成立
- 处理：状态改为 `stale`，并补新证据

### 证据冲突

- 代码与文档、代码与对话、文档与对话互相冲突
- 处理：优先按来源优先级裁决；无法裁决时标记 `conflicted`

### 历史兼容

- 旧正文需要暂时保留，但新结构已上线
- 处理：保留正文区，机器区逐步补齐，避免一次性硬迁移

## 原则

1. 冲突不等于删除；冲突首先要被看见。
2. 过期不等于丢弃；失效事实仍可保留历史上下文。
3. 状态变化应优先体现在机器索引区，再同步到正文区。
