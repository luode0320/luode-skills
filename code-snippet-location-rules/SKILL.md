# 代码片段位置定位规则

只在“用户给了代码内容，但没给代码位置”的前置定位场景使用。
目标是先找准要改的文件和代码位点，再进入后续需求、Bug 或编码流程。

## Skill 作用与适用场景

- 处理用户只说“这段代码”“这里”“当前这个逻辑”但没有给路径的请求。
- 优先利用当前活动编辑器、当前打开文件、当前选区等客户端上下文。
- 在 VS Code / Cursor / JetBrains 等编辑器场景下，把用户正在看的文件视为强线索。
- 避免因为仓库中存在相似函数、相同变量名或重复片段，就误改其他文件。
- 在无法唯一定位时先输出候选和差异点，必要时向用户确认。
- 定位完成后，把确认的文件路径交给 `code-context-resync-rules` 或后续编码 / Bug skill。

## 自动触发信号

- 用户粘贴代码块、函数片段、报错上下文片段或局部 diff，但没有写文件路径。
- 用户说“这里改一下”“这段代码需要调整”“这个函数有问题”，但“这里”只能从上下文推断。
- 用户没有指定模块名、包名、文件名、类名或唯一符号。
- 仓库搜索能找到多个相似片段，且其中一个候选可能就是用户当前打开的文件。
- 当前客户端可能提供活动文件、打开标签页、选区或 IDE 上下文。

## 进入后先做什么

1. 先判断用户是否已经给出明确路径、文件名、符号全名或唯一模块位置；若已给出，不使用推断覆盖用户明示位置。
2. 提取代码片段中的稳定锚点：函数名、类型名、关键字段、连续 3-8 行原文、报错行附近语句。
3. 检查当前对话或工具上下文是否暴露活动编辑器、当前打开文件、当前选区或 IDE 选中文件。
4. 如果存在活动编辑器上下文，先在该文件中验证代码片段或稳定锚点是否匹配。
5. 如果活动文件不可见或不匹配，再用 `rg` 做仓库内精确匹配。
6. 如果多个候选匹配，优先选择活动文件 / 打开文件 / 当前选区命中的候选；仍不唯一时暂停确认。
7. 定位成功后，明确记录“目标文件路径 + 定位依据 + 是否存在其他候选”。

## 默认执行流程

1. 默认先读 `references/source-priority.md`，按定位来源优先级排序。
2. 先使用用户明示路径；没有明示路径时，读取可用的活动编辑器 / 当前打开文件 / 当前选区上下文。
3. 对活动文件做片段验证：能匹配原文、函数名或稳定锚点时，优先把它作为目标文件。
4. 活动文件不可用或验证失败时，用 `rg` 搜索代码片段中的稳定锚点。
5. 候选唯一时继续；候选多个时按活动编辑器、打开文件、最近修改文件、当前工作目录相关性排序。
6. 仍然无法唯一判断时，不写代码，先向用户确认目标文件。
7. 目标文件确认后，转入 `code-context-resync-rules` 重新读取当前文件，再继续具体修改。

## 权责边界与不负责事项

- 只负责“没有路径时先找准代码位置”，不负责具体代码怎么改。
- 不替代 `code-context-resync-rules`；一旦目标文件确定，仍要以当前文件内容为真实基线。
- 不把活动编辑器上下文当成绝对真相；必须验证片段或锚点。
- 不因为搜索结果靠前就直接认定目标文件。
- 不根据模型记忆、历史项目结构或文件名猜测覆盖真实上下文。

## 需要暂停并确认的条件

- 用户未给路径，活动编辑器上下文不可见，仓库搜索又出现多个合理候选。
- 活动文件与用户粘贴片段不匹配，但仓库其他文件匹配。
- 用户粘贴的代码像是经过格式化、简化或改写，无法可靠精确匹配。
- 多个候选文件都包含同名函数或高度相似逻辑，且改错文件会有明显风险。
- 用户说“当前打开的文件”但当前运行环境没有暴露活动文件信息。

## 执行通过 / 驳回标准

- 通过：能明确给出目标文件、定位依据、是否存在其他候选，并在继续修改前完成目标文件重读。
- 通过：活动编辑器上下文只在片段或锚点验证通过后才被采用。
- 驳回：用户未指定路径时直接改第一个搜索结果。
- 驳回：活动编辑器不可见时仍声称“用户当前打开的是某文件”。
- 驳回：存在多个候选但没有提示歧义，也没有向用户确认。

## 执行结果归档要求

- 默认不单独落文档。
- 若本次定位影响后续需求、Bug 或测试记录，应在对应主记录中写明“目标文件来源于活动编辑器 / 片段匹配 / 用户确认”。
- 最终回复中应简短说明目标文件定位依据，避免用户误以为是随意猜测。

## references 读取规则

- 默认先读 `references/source-priority.md`。
