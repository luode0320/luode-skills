# 代码片段定位来源优先级

## 总原则

用户未给文件路径时，不能直接把仓库搜索结果当成目标位置。
必须先判断是否存在更贴近用户当前意图的上下文来源。

## 优先级

1. 用户明确路径、文件名、符号全名或模块位置。
2. 当前活动编辑器、当前打开文件、当前选区、IDE 暴露的活动文件信息。
3. 用户本轮消息中的附件路径、截图来源、终端输出路径或报错堆栈路径。
4. 活动文件 / 打开文件中与代码片段的原文或稳定锚点匹配。
5. 仓库内 `rg` 精确匹配，且候选唯一。
6. 仓库内多个候选时，结合活动文件、打开文件、最近修改时间、当前工作目录相关性排序。
7. 仍不唯一时，向用户确认目标文件。

## 活动编辑器使用规则

- 只有当前运行环境明确提供活动文件、打开文件或选区信息时，才能引用它。
- 活动文件必须通过片段原文、函数名、类型名、关键字段或连续语句验证。
- 活动文件和仓库搜索冲突时，先说明冲突，不要直接改任一方。
- 如果活动编辑器不可见，应明确说“当前环境没有暴露活动文件信息”，再使用搜索和确认流程。

## 搜索建议

- 优先搜索连续 3-8 行稳定原文。
- 原文可能被用户格式化时，退回搜索函数名、类型名、字段名、错误文本或唯一字符串。
- 多语言仓库中优先限定扩展名或目录，但不能凭目录猜测覆盖证据。

## 典型输出

```text
我会先把这段代码定位到具体文件。当前没有用户明示路径；如果活动编辑器上下文可用，我会先验证当前打开文件是否包含这段片段。若不可用或不匹配，再用仓库搜索候选；候选不唯一时先请你确认。
```
