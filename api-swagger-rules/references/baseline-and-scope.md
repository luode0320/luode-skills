# Swagger / OpenAPI 基线与适用范围

## 什么时候默认要求接入

- 项目存在后端 HTTP API，且开发、测试、前后端联调或接口排查需要可视化调试入口时，默认要求接入统一 Swagger/OpenAPI 方案。
- 只要 HTTP API 已经成为稳定交付面，就不建议长期依赖纯口头说明或零散文档。

## 什么时候不强制

- 纯定时任务项目。
- 纯消息消费 / 队列处理项目。
- 纯工具库、SDK、CLI 项目。
- 没有 HTTP API 暴露面的内部模块。

## 单一主方案原则

- 同一项目只保留一套主 Swagger/OpenAPI 方案。
- 允许存在不同 UI 皮肤或包装层，但底层契约源应唯一。
- 不允许同一项目同时长期维护两份彼此独立的接口文档定义。

## 每个接口的最小文档项

- 接口摘要。
- 所属 tag / 分组。
- 请求方法与路径。
- 请求体模型。
- 成功响应模型。
- 错误响应模型。
- 鉴权要求。
- 至少一个可用于调试的示例。

## 基本判断原则

- Swagger/OpenAPI 是接口契约和调试入口，不是普通业务注释。
- 接口代码与文档冲突时，优先以“当前实际实现 + 当前有效规则”校正 Swagger/OpenAPI 文档。
- 如果接口仍在快速试错，可暂时简化描述，但不能完全缺失文档入口。
