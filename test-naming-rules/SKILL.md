# 测试命名规则

只在"测试资产名称应该怎么起"这个问题上使用这个 skill。
如果当前争议是测试资源放在哪里，请转交 `test-location-rules`；如果是测试程序具体怎么写，请转交 `test-program-rules`。

**重要：本 skill 必须以 `test-location-rules` 为基础，所有测试资产必须统一放在 `test/` 根目录下，任务目录命名必须遵循 `yyyy-MM-DD_HHmmss_测试任务中文简介` 格式。**

## Skill 作用与适用场景

- 作为测试资源管理链的第二道规则，在落点确定后统一命名。
- 统一测试任务目录、测试文件、测试脚本、测试数据目录、fixture 和 mock 目录的命名方式。
- 提升测试资产的可读性、可检索性和任务对应关系，减少"找不到、看不懂、分不清"的问题。
- 防止使用临时、含糊、重复或风格混乱的测试命名。
- 确保任务目录命名符合 `yyyy-MM-DD_HHmmss_测试任务中文简介` 格式。

## 自动触发信号

- 新增或修改测试任务目录、测试文件、测试脚本、测试数据目录、fixture 目录、mock 目录。
- 需要为当前需求、当前 Bug、当前接口、当前页面交互创建新的测试资产名称。
- 发现测试目录或测试文件名称存在 `temp`、`new`、`final`、`test1` 等临时命名痕迹。
- 同类测试资产已经出现多种命名风格并存，需要统一时。
- 测试资产没有放在 `test/` 根目录下，需要纠正时。
- 任务目录命名不符合 `yyyy-MM-DD_HHmmss_测试任务中文简介` 格式时。

## 进入后先做什么

1. 先确认测试资产落点已经稳定，且在 `test/` 根目录下，不在当前 skill 内重新决定目录位置。
2. 确认任务目录命名符合 `yyyy-MM-DD_HHmmss_测试任务中文简介` 格式。
3. 区分当前名称对象是任务目录、测试文件、测试脚本还是测试数据目录。
4. 提取命名必须表达的核心信息：业务对象、测试层级、场景目标、是否为公共资源。
5. 优先复用项目现有命名模式，不随意引入新的前后缀和分隔符。

## 默认执行流程

1. 先确认遵循 `test-location-rules` 的约定：所有测试资产在 `test/` 根目录下，任务目录使用 `yyyy-MM-DD_HHmmss_测试任务中文简介` 格式。
2. 默认先读 `references/naming-baseline.md`，确认命名维度、顺序和禁用写法。
3. 如果命名争议涉及落点、职责或需求含义不清，再读 `references/naming-boundaries.md`。
4. 如果需要判断当前名称是否合格，再读 `references/naming-examples.md` 对照正反例。
5. 输出推荐名称、命名问题清单、是否需要统一重命名，以及重命名时的最小改动建议。
6. 确认每个测试任务子目录都包含 README.md。
7. 名称未稳定前，不进入 `test-program-rules` 和 `test-doc-rules`。

## 权责边界与不负责事项

- 只负责命名规则，不负责测试目录放置方案，那属于 `test-location-rules`。
- 必须遵循 `test-location-rules` 的约定：统一使用 `test/` 根目录和 `yyyy-MM-DD_HHmmss_测试任务中文简介` 的任务目录命名格式。
- 不负责测试程序实现、断言方式和脚本内容，那属于 `test-program-rules`。
- 不负责测试说明文档章节和记录格式，那属于 `test-doc-rules`。
- 不负责功能是否正确、覆盖是否充分，那属于 `functional-validation-rules` 和 `test-regression-rules`。
- 如果名称无法确定是因为需求目标、场景边界或 Bug 定义本身不清，应回流 `requirement-gap-rules`、`requirement-boundary-rules` 或 Bug 域相关 skill。

## 需要暂停并确认的条件

- 项目尚无稳定测试命名模式，且当前命名选择会影响整个测试域后续扩展。
- 同一个测试资产同时承载多个业务目标，导致名称无法聚焦。
- 为统一命名需要批量改动大量历史测试文件，已经超出当前最小改动范围。
- 名称变更会影响脚本引用、构建配置或外部约定，但影响面尚未确认。
- 测试资产没有放在 `test/` 根目录下，需要确认迁移方案。

## 执行通过 / 驳回标准

- 通过：当前测试目录和文件名称能够稳定表达业务对象、测试层级和场景目标，且与项目现有命名模式一致，没有临时命名和歧义命名问题；所有测试资产在 `test/` 根目录下；任务目录命名符合 `yyyy-MM-DD_HHmmss_测试任务中文简介` 格式；每个任务目录都包含 README.md。
- 驳回：当前名称含糊、重复、临时化或风格混乱，继续进入后续测试编写和验证会显著增加维护成本；或测试资产不在 `test/` 根目录下；或任务目录命名不符合规范格式。

## 执行结果归档要求

- 将命名结论记录到当前任务测试目录下的 README.md 中。
- 归档内容至少包含命名对象、命名理由、命名模式、禁用名称和待统一项。
- 如果发生重命名，应在记录中说明影响范围和引用同步要求。
- 提醒用户旧的测试目录以每周日为时间间隔，归档到 `test/` 目录下（可创建 `archive/` 子目录存放归档内容）。

## references 读取规则

- 默认先读 `references/naming-baseline.md`。
- 只有在职责边界或需求含义不清时，再读 `references/naming-boundaries.md`。
- 只有在需要对照正反例时，再读 `references/naming-examples.md`。
