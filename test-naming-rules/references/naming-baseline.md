# 测试命名基线

## 用途

用于统一测试目录、测试文件、测试脚本和测试数据目录的命名维度与表达顺序。

**重要：本规则必须以 `test-location-rules` 为基础，所有测试资产必须统一放在 `test/` 根目录下。**

## 铁律：test/ 根目录与任务目录命名

- **所有测试资产必须统一放在 `test/` 根目录下**，不得在业务代码目录或其他位置存放测试资产。
- **任务目录命名必须遵循格式：`yyyy-MM-DD_HHmmss_测试任务中文简介`**。
  - 示例：`2026-03-28_143022_用户登录功能验证`
  - 示例：`2026-03-28_154511_订单创建接口测试`
- **每个测试任务子目录必须包含 README.md**，说明测试目的、执行方式、依赖数据、覆盖范围和验证结论。
- **同一需求的多个测试需要分别创建独立目录**，不要混在一个目录中。

## 命名原则

- 名称先表达业务对象或场景目标，再表达测试层级或用途。
- 同一类测试资产保持相同的分隔方式、前后缀和复数规则。
- 复用项目现有命名风格，不额外引入新的缩写体系。
- 名称应可检索、可推断、可和需求或 Bug 对应，不用临时词。

## 名称中应优先体现的信息

- 业务对象或模块名
- 场景目标或动作
- 测试层级或资源用途
- 是否为共享资源或任务私有资源

## 禁用写法

- `temp`、`tmp`、`new`、`final`、`test1`、`demo` 之类的临时名称
- 与业务无关的模糊名称，如 `case-a`、`data-x`、`script2`
- 同一目录中既有下划线又有短横线，且无现有项目约定支撑
- 一个名称里同时塞入多个无主次的场景目标
- 使用 `tests/`、`qa/`、`verify/`、`tmp-test/` 等非 `test/` 名称作为测试根目录
- 任务目录不使用 `yyyy-MM-DD_HHmmss_测试任务中文简介` 格式

## 检查清单

- 名称是否能让人一眼看出测试对象是什么。
- 名称是否与所在目录层级和文件类型匹配。
- 同类名称是否沿用现有项目模式。
- 名称是否避免了临时化、歧义化和过长堆词问题。
- 所有测试资产是否都在 `test/` 根目录下。
- 任务目录命名是否符合 `yyyy-MM-DD_HHmmss_测试任务中文简介` 格式。
- 每个任务目录是否都包含 README.md。
