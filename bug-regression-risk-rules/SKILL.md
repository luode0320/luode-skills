# Bug 回归风险规则

只在修复方案已经成形，需要先判断“这次修复会不会顺手带坏别处”时使用这个 skill。
如果当前还没有形成修复方向，请先转 `bug-fix-proposal-rules`；如果已经进入实际回归验证，请转 `test-regression-rules` 或 `bug-validation-rules`。

## Skill 作用与适用场景

- 识别修复动作可能波及的旧能力、共享依赖和兼容路径。
- 给风险点排序，帮助决定验证优先级。
- 区分局部风险、链路风险和公共能力风险。
- 将回归风险结论统一沉淀到当前 Bug 根目录。
- 避免“修好了当前问题，却没看见副作用”。

## 自动触发信号

- 修复涉及公共方法、复用组件、共享配置、基础库或中间件。
- 修复会改变原有接口返回、数据写入、缓存逻辑或异常处理。
- 团队担心某个修复会带来兼容性或历史行为回退。
- 需要在修复实施前先决定回归重点。

## 进入后先做什么

1. 先列出修复直接改动点和它们的复用范围。
2. 为当前 Bug 确认统一的根目录，继续沿用项目根目录 `bug/` 下的当前 Bug 目录。
3. 识别会被牵动的旧流程、旧接口、共享逻辑和数据行为。
4. 按影响范围和破坏成本给风险分级。
5. 形成验证优先级和需要重点回看的链路。

## 默认执行流程

1. 默认先读 `references/risk-dimensions.md`，先按统一维度识别风险点。
2. 如需继续展开，再读 `references/risk-ranking-and-scope.md`，需要做风险分级和验证优先级排序。
3. 需要对照边界或正反例时，再读 `references/risk-examples.md`，需要对照风险识别样例。
4. 输出 回归风险清单、风险等级和验证优先级。
5. 风险识别完成后回流 `bug-fix-proposal-rules` 或交给 `test-regression-rules`、`bug-validation-rules` 落实验证。

## 权责边界与不负责事项

- 只负责识别风险，不等于已经完成回归验证。
- 不代替 `test-strategy-rules` 制定整体测试资源优先级。
- 不把所有可能性都列成高风险，必须区分主次。
- 不直接决定修复方案是否采纳，但应指出高风险代价。

## 需要暂停并确认的条件

- 修复方案本身还没定，风险无法可靠评估。
- 影响范围和复用范围证据不足，暂时无法分级。
- 高风险路径没有任何验证条件或环境支持。
- 团队试图跳过风险识别直接合入高影响改动。

## 执行通过 / 驳回标准

- 通过：能够列出主要风险点、解释影响范围，并明确哪些路径必须优先验证、哪些风险可记录后续补看。
- 驳回：泛泛而谈“可能有风险”，没有具体风险对象、分级和验证重点。

## 执行结果归档要求

- 将回归风险清单和分级结论统一记录到项目根目录 `bug/` 下的当前 Bug 根目录中。
- 当前 Bug 根目录必须包含 `README.md`，至少写明改动点、影响对象、风险等级和建议验证路径。
- 如果后续落实了回归测试，应在当前 Bug 根目录中补充对应测试目录或验证记录路径。
- 如最终接受已知风险，应保留理由和补救安排。

## references 读取规则

- 默认先读 `references/risk-dimensions.md`。
- 只有在 对风险做分级和确定优先级 时，再读 `references/risk-ranking-and-scope.md`。
- 只有在 对照样例或判断是否高估 / 低估风险 时，再读 `references/risk-examples.md`。
