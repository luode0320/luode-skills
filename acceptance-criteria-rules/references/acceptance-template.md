# 验收标准模板

## 白话正文与附录

先用普通中文说明验收要确认什么、对谁有影响、怎样算完成，以及本次验收是否适用。不适用时写原因和依据但不阻断；必须完成却缺条件时才阻断。精确场景数据、命令、ID 和证据统一放入附录，详见 [白话文档与附录分层契约](../../artifact-delivery-gate-rules/references/plain-language-document-contract.md)。

## 适用定位

- 本模板只用于前置“验收标准”，发生在实施前。
- 文档路径统一为 `doc/7-验收/YYYY-MM-DD_HHmmss_<来源对象标识>_验收标准.md`。
- 验收标准文档与来源对象文档（需求或 Bug）、实施文档、最终验收文档分开维护。

## 推荐结构

1. 场景名称：在哪个业务场景下验证。
2. 前置条件：依赖数据、角色、环境、权限、配置。
3. 输入动作：用户操作、接口调用、数据变化。
4. 预期结果：页面表现、返回结果、状态变化、日志或副作用。
5. 异常分支：失败时应如何表现。
6. 边界条件：空值、极值、重复提交、并发、历史数据。
7. 范围外说明：本次不负责什么。
8. 验收流程图（建议，复杂场景必填）：使用 Mermaid `flowchart` 表达主路径与分支。
9. 验收时序图（跨角色/跨服务必填）：使用 Mermaid `sequenceDiagram` 表达参与者、调用顺序、返回结果与异常分支；图前写明绘图目的和关联 `AC-*`，单角色场景不适用时填写 `N/A + 原因 + 证据`。
10. 验收决策表（建议）：用表格表达条件组合与预期结果，避免纯文字歧义。

## 归档要求

- 验收标准文档路径统一为 `doc/7-验收/YYYY-MM-DD_HHmmss_<来源对象标识>_验收标准.md`。
- 验收标准文档与来源对象文档分开维护，不要混写到 `doc/2-需求/YYYY-MM-DD_HHmmss_<需求中文简介标题>.md` 或 `doc/4-bugs/YYYY-MM-DD_HHmmss_问题中文简介/README.md`。
- 同一个来源对象只保留一份验收标准文档，后续补充和变更继续更新同一份文档。
- 涉及多个分支或条件组合的场景，建议补图或表（流程图/决策表）提升可读性与一致性。
- 最终放行结论不写在这里；后置放行应交给 `final-acceptance-rules` 写入最终验收文档。

## 极致完整性字段契约（强制）

每条验收标准必须使用稳定 `AC-*`，并完整填写下表字段；任何不适用字段都写 `N/A + 原因 + 证据`。

| 字段 | 必填内容 |
| --- | --- |
| 追踪 | `REQ-*`/`RULE-*` 来源、`AC-*`、实施周期/任务、测试用例、证据编号 |
| 场景 | 角色、前置条件、环境、权限、样本来源、初始状态、触发动作 |
| 断言 | 可观察输出、状态/错误码/日志/指标、数值阈值、时间窗口、允许误差 |
| 分支 | 主流程、异常、边界、并发/重复、权限拒绝、兼容旧数据、回滚/清理 |
| 交付 | local 执行命令、通过标准、失败预期、证据路径、审查人、验收人、验收时间 |

### 二值化与阻断规则

1. 每条 `AC-*` 只能产生 `PASS` 或 `FAIL`；不得使用“基本通过”“人工确认”“体验良好”等不可复核结论。
2. 主流程、异常流程、边界流程、权限/兼容流程至少各有一个可执行场景；确实不存在时必须给出 `N/A + 原因 + 证据`。
3. 对来源明确要求且当前必须完成的适用验收，缺少测试命令、local 样本、断言、失败预期、清理或回滚任一项，验收标准状态为 `blocked`；不适用项写原因和依据，受限项写替代验证、人工补验和通过标准。
4. L2 及以上场景必须包含 Mermaid 流程图；跨角色或跨服务场景必须包含 Mermaid 时序图；图前说明目的和关联 `AC-*`。
