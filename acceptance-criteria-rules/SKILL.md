# 验收标准细化规则

只在需要把需求目标翻译成可检查的完成标准时使用这个 skill。
如果当前争议是需求信息不全、边界不清或实现已经有问题，请先转交相邻需求域或 Bug 域 skill。

## Skill 作用与适用场景

- 把“要做什么”落成“做到什么算完成”的可验证标准。
- 覆盖主路径、异常路径、边界情况和兼容要求。
- 对关键验收路径优先提供图形化表达（流程图/决策表/状态表），避免纯文字导致验收口径理解偏差。
- 为实现、功能验证、回归验证和交付说明提供统一判定口径。
- 参考 `artifact-storage-rules` 的目录与命名约定，将验收标准落到统一的前置验收文档中，但与需求文档、Bug 文档分开保存。
- 减少实现阶段和测试阶段再临时补验收口径的返工。

## 自动触发信号

- 需求准备进入实现，但完成标准还停留在抽象口号层。
- 测试前需要明确主流程、异常流程和边界条件的通过口径。
- 交付前需要确认哪些结果必须验证，哪些属于范围外。
- 当前需求、Bug 或修复任务与测试域之间缺少可直接对接的验收描述。

## 专属阶段边界

- 本入口只把稳定的需求或 Bug 目标转成实施前可验证的验收标准，不执行功能测试、回归测试或最终放行。
- 标准必须与来源对象分离归档；实现路径复杂时转 `implementation-planning-rules`，功能与回归验证分别转测试域。
- 来源目标、输入输出、主/异常路径或兼容要求尚未稳定时必须暂停，不能以测试结果倒推前置验收标准。

## 共享证据和专属契约

- 进入执行、创建或更新验收标准、形成通过/驳回结论、归档或判定阻断前，必须读取 `references/shared-evidence-and-specialized-contracts.md`。
- 该引用保留可验证性、图文一致、来源追踪、归档、暂停、通过/驳回与相邻 route 的完整细则；本文件保留自动触发与前置验收阶段职责。

## references 读取规则

- 读取顺序遵循「共享证据和专属契约」：先读 `references/shared-evidence-and-specialized-contracts.md`，再按其中路由读取模板、可测试性清单、边界说明和 artifact-storage-rules 的路径/更新策略。
