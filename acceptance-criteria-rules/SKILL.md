# 验收标准细化规则

只在需要把需求目标翻译成可检查的完成标准时使用这个 skill。
如果当前争议是需求信息不全、边界不清或实现已经有问题，请先转交相邻需求域或 Bug 域 skill。

## Skill 作用与适用场景

- 把“要做什么”落成“做到什么算完成”的可验证标准。
- 覆盖主路径、异常路径、边界情况和兼容要求。
- 为实现、功能验证、回归验证和交付说明提供统一判定口径。
- 参考 `requirement-intake-rules` 的目录归档方式，将验收标准落到 `ment/` 下，但与需求文档分开保存。
- 减少实现阶段和测试阶段再临时补验收口径的返工。

## 自动触发信号

- 需求准备进入实现，但完成标准还停留在抽象口号层。
- 测试前需要明确主流程、异常流程和边界条件的通过口径。
- 交付前需要确认哪些结果必须验证，哪些属于范围外。
- 当前需求与测试域之间缺少可直接对接的验收描述。

## 进入后先做什么

1. 先确认需求目标和边界已经基本稳定。
2. 把用户期望拆成可观察、可判断、可复核的结果描述。
3. 补齐成功条件、失败条件、异常处理、边界限制和范围外说明。
4. 找到当前需求对应的验收文档；如果还没有，就按本 skill 的命名规则在 `ment/` 下初始化单独的验收文档。
5. 检查这些标准是否真的能被实现和测试直接使用，并准备回写到该验收文档。

## 默认执行流程

1. 默认先读 `references/acceptance-template.md`，先按统一结构写验收标准。
2. 如需继续展开，再读 `references/testable-criteria-checklist.md`，需要检查验收标准是否真的可测可判。
3. 需要对照边界或正反例时，再读 `references/acceptance-boundaries.md`，需要判断验收标准与需求边界、测试执行的边界。
4. 输出结构化验收标准、边界口径和验证关注点，并更新到当前需求对应的验收文档。
5. 如果该需求还没有验收文档，则按 `ment/YYYY-MM-DD_验收中文说明.md` 的命名规则创建；如同一天存在多个近似标题，在不改变中文语义前提下补充更具体描述以避免重名。
6. 验收标准明确后，需求可进入实现；功能正确性验证转交 `functional-validation-rules`，兼容性和旧能力回归转交 `test-regression-rules`。

## 权责边界与不负责事项

- 只负责写清验收标准，不直接执行测试，那属于测试域。
- 不代替 `requirement-gap-rules` 补基础信息；前提没补齐时不要硬写验收。
- 不代替 `requirement-boundary-rules` 决定是否允许修改旧逻辑，但要把范围外写清。
- 不把实现方案、技术细节和验收标准混写在一起。
- 不把验收标准直接混写进需求文档；验收标准必须单独维护在 `ment/` 下的验收文档中。

## 需要暂停并确认的条件

- 需求目标和边界还不稳定，写出来的验收标准随时会失效。
- 连输入输出和关键场景都不清，无法写成可验证条件。
- 主路径、异常路径或兼容要求存在明显争议。
- 当前团队想用测试执行结果反推验收标准，而不是先定义标准。

## 执行通过 / 驳回标准

- 通过：能够形成可验证、可测试、可复核的验收口径，至少覆盖主流程、异常流程、边界条件、范围外事项和通过标准，并回写到当前需求对应的验收文档。
- 驳回：仍停留在“体验更好”“支持一下”“处理正确”这类抽象描述，无法指导实现与验证。

## 执行结果归档要求

- 将验收标准记录到 `ment/` 下单独的验收文档中，不与需求文档混写。
- 文档文件名统一使用 `ment/YYYY-MM-DD_验收中文说明.md` 格式，例如 `ment/2026-03-28_验收中文将要说明.md`。
- 同一个需求的验收标准应持续更新同一份验收文档，而不是在 `ment/` 下重复新建多份平行验收记录。
- 归档内容至少包含场景、输入、预期结果、异常分支和范围外说明。
- 如后续验收标准发生变化，应保留版本差异，避免实现和测试口径错位。

## references 读取规则

- 默认先读 `references/acceptance-template.md`。
- 只有在 检查验收标准是否可验证 时，再读 `references/testable-criteria-checklist.md`。
- 只有在 判断与相邻 skill 的边界或对照样例 时，再读 `references/acceptance-boundaries.md`。
