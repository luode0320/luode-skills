# 会话标题规则

## 目标

让项目会话列表始终能从标题直接看出当前问题、需求、Bug、实施、审查或测试主题，减少“提交 git”“继续做”“新会话”这类泛化标题带来的检索成本。

本 skill 只负责会话元数据命名，不改变项目文件、不替代研发流程，也不作为提交、测试或审查授权。

## 自动触发条件

满足以下任一情况时自动触发，不等待用户说“帮我重命名会话”：

1. 当前用户提问、请求或补充信息已经能稳定归纳成一个中文任务主题，例如明确的需求、Bug、实施、审查、测试、提交或规则更新任务。
2. 当前会话标题为空、默认标题、泛称、动作词过强或明显过时，例如“新会话”“提交 git”“继续做”“开始实施”“分析项目”。
3. 任务中途发生实质转向，旧标题已经无法代表当前主任务。
4. 用户粘贴截图、URL、文档、报错或提问后，当前轮已经能稳定归纳出中文任务简要。
5. 当前进入 goal 创建、goal 恢复、上下文压缩后继续、长任务阶段切换、执行阶段主题稳定或阶段性进度更新节点，且标题仍为空泛、过时或不匹配当前主任务。

## 跳过条件

出现以下任一情况时不改名，并在中间进度中说明跳过原因：

1. 当前还只是闲聊、开放式问答、探索或需求未稳定，无法形成可靠任务简要。
2. 当前标题已经准确表达当前任务。
3. 用户明确说不要重命名、不要改标题或保持当前标题。
4. 当前环境没有真实线程重命名工具，或无法可靠确定当前会话 ID。
5. 需要通过伪指令、正文模拟工具调用或猜测工具结果才能完成重命名。
6. 只是最小任务内部的小步骤推进，任务对象和阶段主题没有实质变化，继续沿用当前准确标题更稳定。

## 标题生成规则

标题使用中文简要，优先 8-18 字，最长不超过 24 字。

1. 写“任务对象 + 动作 / 症状 / 阶段”，避免只写动作。
2. 不默认加项目名；项目名通常已由工作区或侧边栏分组承担。
3. 不使用泛化标题，例如“提交 git”“开始实施”“继续做”“修复 bug”“更新文档”。
4. 不加入无关状态词，例如“已完成”“待处理”“紧急”，除非这是任务主题的一部分。
5. 标题应稳定到足以复用，不为每个小步骤频繁改名。
6. 如果输入是提问，也要改写成结论式主题，不直接写成问句。
7. 对 goal 或长任务过程中的改名，优先表达当前主目标或阶段主题，避免写成“goal执行”“继续执行”“自动续做”这类泛化状态。

## 平台能力矩阵

在执行改名之前，必须先按当前运行平台判断“是否真的存在可调用的线程改名能力”；规则文件存在与否，不等同于平台支持改名。

| 平台 | 默认能力判断 | 执行动作 |
| --- | --- | --- |
| Codex | 支持真实线程改名工具；若当前工具列表未直接暴露，先用 `tool_search` 发现线程工具 | 优先使用 `set_thread_title` |
| Claude Code | 仅在宿主环境或插件实际暴露真实线程改名工具时支持 | 有工具则执行，无工具则显式跳过 |
| Claude Desktop | 默认视为未暴露真实自动改名工具 | 显式跳过，说明平台不支持 |
| 其他 agent | 仅在真实工具存在时支持 | 无工具则显式跳过 |

补充约束：

1. 不得把 `AGENTS.md` / `CLAUDE.md` 已存在，误判成“当前平台支持自动改名”。
2. 不得把用户可以手工改标题，误判成“agent 可以自动改名”。
3. Claude Desktop 若无真实改名工具，必须把结果写成跳过证据，不能静默失败。

示例：

| 场景 | 推荐标题 | 避免标题 |
| --- | --- | --- |
| 需求 | `超大单 sheet 导出` | `新需求` |
| Bug | `稳定币费率异常` | `修复 bug` |
| 审查 | `审查-超大单导出` | `代码审查` |
| 提交 | `提交-费率修复` | `提交 git` |
| 规则更新 | `会话自动重命名规则` | `更新 skill` |

## 执行流程

1. 先判断当前任务主题是否稳定，以及当前标题是否需要更新。
2. 若需要更新，基于用户提问、请求、goal 目标、恢复上下文或当前阶段主题提炼标题，生成符合标题规则的中文简要。
3. 判断当前平台属于 Codex、Claude Code、Claude Desktop 还是其他 agent。
4. 探测当前平台是否存在真实线程重命名工具：
   - Codex：若当前工具列表还没有 `set_thread_title` 或 `list_threads`，先用 `tool_search` 搜索并加载相关线程工具；拿到工具后优先使用 `set_thread_title`，不要因为首屏未暴露就直接判定“工具不可用”。
   - Claude Code：只有宿主环境或插件真实暴露改名工具时才执行；否则跳过。
   - Claude Desktop：默认视为无真实自动改名工具，直接跳过。
   - 其他 agent：只有能可靠识别当前会话且存在真实工具时才执行。
5. 若需要确认当前线程 ID，优先使用真实线程列表工具读取最近线程，并结合 `cwd`、最近更新时间、当前轮 preview 或当前任务主题交叉识别当前会话；只有能可靠识别当前线程时才调用重命名。
6. goal 或长任务过程中，只要进入新的稳定阶段且标题不匹配，就在阶段开始或恢复后尽早尝试改名；不得等到最终总结才首次处理标题。
7. 重命名成功后，在中间进度输出：`会话标题: 已重命名为「标题」`。
8. 跳过时输出：`会话标题: 跳过（原因）`。

## 工具与证据约束

- 必须通过真实工具调用完成会话重命名，禁止在回复正文中伪造 `<invoke>`、`<result>` 或 raw directive。
- 在 Codex 中，`set_thread_title`、`list_threads` 等线程工具可能通过 `tool_search` 延迟暴露；未先做工具发现就直接跳过，不算“工具不可用”。
- 只有完成工具发现后仍拿不到真实线程工具时，才能记录“工具不可用，未重命名”，不得声称已经改名。
- Claude Desktop 默认没有真实自动改名工具时，只能记录“平台无真实自动改名工具，未重命名”，不得把无结果当成成功。
- 若重命名失败，保留失败原因并继续主任务；标题失败通常不阻断研发任务本身。
- 本 skill 不写入仓库文件；若为了沉淀本规则而修改 `SKILL.md`、`AGENTS.md` 或长期记忆，则改动本身应由 `skill-evolution-rules`、`project-rule-file-bootstrap-rules`、`project-memory-rules`、`project-style-rules` 和 `skill-execution-compliance-gate-rules` 负责收口。

## 通过标准

- 应重命名时，已使用真实工具把当前会话标题更新为中文简要。
- 不应重命名或无法重命名时，已给出明确跳过原因。
- Claude Desktop 场景下，若不存在真实自动改名工具，跳过原因必须显式可见，不能静默无结果。
- 标题符合“对象 + 动作 / 症状 / 阶段”的中文简要规则。
- 没有把会话标题规则当作需求分析、Bug 定位、实施规划、测试验证或 Git 提交授权。
