# 会话标题规则

## 目标

让项目会话列表始终能从标题直接看出当前问题、需求、Bug、实施、审查或测试主题，减少“提交 git”“继续做”“新会话”这类泛化标题带来的检索成本。

本 skill 只负责会话元数据命名，不改变项目文件、不替代研发流程，也不作为提交、测试或审查授权。

## 自动触发条件

满足以下任一情况时自动触发，不等待用户说“帮我重命名会话”：

1. 当前用户提问、请求或补充信息已经能稳定归纳成一个中文任务主题，例如明确的需求、Bug、实施、审查、测试、提交或规则更新任务。
2. 当前会话标题为空、默认标题、泛称、动作词过强或明显过时，例如“新会话”“提交 git”“继续做”“开始实施”“分析项目”。
3. 任务中途发生实质转向，旧标题已经无法代表当前主任务。
4. 用户粘贴截图、URL、文档、报错或提问后，当前轮已经能稳定归纳出中文任务简要。
5. 当前进入 goal 创建、goal 恢复、上下文压缩后继续、长任务阶段切换、执行阶段主题稳定或阶段性进度更新节点，且标题仍为空泛、过时或不匹配当前主任务。

## 跳过条件

出现以下任一情况时不改名，并在中间进度中说明跳过原因：

1. 当前还只是闲聊、开放式问答、探索或需求未稳定，无法形成可靠任务简要。
2. 当前标题已经准确表达当前任务。
3. 用户明确说不要重命名、不要改标题或保持当前标题。
4. 当前环境既未暴露 `rename_current_thread`，也未暴露可直接作用于当前会话的 `set_thread_title`；或者 MCP 返回当前线程上下文缺失，且没有安全的原生回退能力。
5. 需要通过伪指令、正文模拟工具调用或猜测工具结果才能完成重命名。
6. 只是最小任务内部的小步骤推进，任务对象和阶段主题没有实质变化，继续沿用当前准确标题更稳定。
7. `rename_current_thread` 要求模型传入 `threadId`、需要通过线程列表猜测当前会话，或工具结果不能证明真实改名。

## 标题生成规则

标题使用中文简要，优先 8-18 字，最长不超过 24 字。

1. 写“任务对象 + 动作 / 症状 / 阶段”，避免只写动作。
2. 不默认加项目名；项目名通常已由工作区或侧边栏分组承担。
3. 不使用泛化标题，例如“提交 git”“开始实施”“继续做”“修复 bug”“更新文档”。
4. 不加入无关状态词，例如“已完成”“待处理”“紧急”，除非这是任务主题的一部分。
5. 标题应稳定到足以复用，不为每个小步骤频繁改名。
6. 如果输入是提问，也要改写成结论式主题，不直接写成问句。
7. 对 goal 或长任务过程中的改名，优先表达当前主目标或阶段主题，避免写成“goal执行”“继续执行”“自动续做”这类泛化状态。

## 平台能力矩阵

在执行改名之前，必须按当前真实工具发现结果判断能力；模型名称、规则文件存在与否或用户可以手工改标题，都不能证明 agent 已具备自动改名能力。

| 当前真实能力 | 执行动作 |
| --- | --- |
| 已暴露 `rename_current_thread` | 首选调用，模型只传 `title` |
| 两个工具都存在 | 先调用 MCP；MCP 返回 `RENAMED` 后立即停止，不得再调用原生工具 |
| MCP 未暴露或首次调用返回非 `INVALID_TITLE` 失败，同时存在 `set_thread_title` | 最多执行一次原生回退 |
| 只有 `set_thread_title` | 使用一次原生回退 |
| 两者均不存在 | 显式跳过 |
| MCP 缺少当前线程上下文 | 禁止通过 `list_threads`、路径、时间或标题相似度猜测；只有原生工具可直接作用于当前会话时才回退 |
| 模型或宿主完全不支持工具调用 | 显式跳过，不宣称跨模型支持 |

平台边界：

1. Codex App 第一版使用 `rename_current_thread` MCP 适配器，`set_thread_title` 只作兼容回退。
2. Claude Code、Claude Desktop 和其他 agent 仅按真实工具发现结果执行，不因 GPT、Claude 或其他模型名称推断能力。
3. 第一版 MCP 适配器只声明支持 Codex App；模型无关不等于宿主无关。

## 执行流程

1. 判断当前任务主题是否稳定，以及当前标题是否需要更新。
2. 若需要更新，基于用户提问、请求、goal 目标、恢复上下文或当前阶段主题提炼符合标题规则的中文简要。
3. 发现当前宿主是否真实暴露 `rename_current_thread`；存在时只传 `title`，不得传入或猜测 `threadId`。
4. MCP 返回 `RENAMED` 时立即停止工具路由，并输出：`会话标题: 已通过 rename_current_thread 重命名为「标题」`。
5. MCP 未暴露，或返回 `THREAD_CONTEXT_MISSING`、`APP_SERVER_UNAVAILABLE`、`APP_SERVER_REJECTED`、`TIMEOUT`、未知错误或格式错误结果时，仅在真实存在可直接作用于当前会话的 `set_thread_title` 时执行一次原生回退。
6. MCP 首次返回 `INVALID_TITLE` 时，按标题规则修正后最多重试 MCP 一次；第二次调用无论返回何种失败码都直接跳过，不再调用原生工具。
7. 原生回退成功后输出：`会话标题: 已通过 set_thread_title 回退重命名为「标题」`。
8. 两条路径均不可用或失败时输出明确跳过原因；当前线程上下文缺失时禁止使用 `list_threads`、最近更新时间、`cwd`、preview 或标题相似度猜测当前会话。
9. goal 或长任务过程中，只要进入新的稳定阶段且标题不匹配，就在阶段开始或恢复后尽早尝试改名；不得等到最终总结才首次处理标题。

## 工具与证据约束

- 统一工具输入、返回码、元数据来源和失败处理以 [rename-tool-contract.md](references/rename-tool-contract.md) 为准。
- 必须通过真实工具调用完成会话重命名，禁止在回复正文中伪造 `<invoke>`、`<result>` 或 raw directive。
- MCP 成功后不得重复调用 `set_thread_title`；正常标题路径只调用一次 MCP。仅首次 `INVALID_TITLE` 允许修正标题后再调用一次 MCP，第二次失败后直接跳过；只有首次调用的非 `INVALID_TITLE` 失败最多再执行一次原生回退。
- `THREAD_CONTEXT_MISSING` 不得降级为线程列表猜测；当前会话身份只能来自可信请求元数据或可直接作用于当前会话的原生工具。
- `TIMEOUT` 不做同输入、同工具的无变化重试；可以对同一标题执行一次原生回退。
- 只有真实工具结果证明改名成功时才能写“已重命名”；未知或格式错误结果按失败处理。
- 若重命名失败，保留失败原因并继续主任务；标题失败通常不阻断研发任务本身。
- 本 skill 不写入仓库文件；若为了沉淀本规则而修改 `SKILL.md`、MCP 资源、`AGENTS.md` 或长期记忆，则改动本身由相应 Skill 合规、演进、审计、项目记忆和风格规则负责收口。

## 通过标准

- 应改名且 MCP 可用时，已优先调用 `rename_current_thread`，并且模型只传标题。
- MCP 成功后没有重复调用原生工具。
- MCP 未暴露或失败时，只有真实存在 `set_thread_title` 才执行一次回退。
- 两个工具均不可用时，已给出明确跳过原因。
- 当前线程上下文缺失时，没有使用线程列表、路径、时间或标题相似度猜测当前会话。
- 标题符合 8-24 字、“对象 + 动作 / 症状 / 阶段”和稳定不频繁改名的规则。
- 用户禁止、标题已准确、主题不稳定和最小步骤推进的跳过语义仍然有效。
- 没有把会话标题规则当作需求分析、Bug 定位、实施规划、测试验证或 Git 提交授权。
