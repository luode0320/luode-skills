# Git 协作规则

只在需要处理 Git 协作动作，而不是代码内容本身时使用这个 skill。
如果当前问题是代码质量或行为风险，请先转交实现自审；如果已经进入上线动作，请转回团队发布流程处理。

## Skill 作用与适用场景

- 规范提交粒度、提交说明、分支同步和 PR 整理。
- 强制分析未提交的代码，如果可以分成不同的业务修改，就用多次不同的 git 提交。
- 当用户明确请求“提交git”时，强制以“清空当前未提交改动”为完成目标，不得只提交一部分后结束。
- 当用户明确请求“提交git”时，允许按业务拆分多次提交，但必须持续执行直到工作区无剩余未提交改动。
- 强制 git 提交消息遵循 feat/fix 等格式，包含标题和说明。
- 强制标题统一使用 `<type>: [中文简要说明] 标题说明`，`[]` 内直接写中文简要描述。
- 强制禁止 `type(scope): 标题` 这类括号 scope 写法（如 `feat(reqsend): ...`），必须改为 `type: [scope] 标题`（如 `feat: [reqsend] ...`）。
- 强制 `[]` 内禁止加功能/业务/接口等分类前缀，且禁止直接写 API URL（如 `POST /api/order/create`）。
- 强制多行提交说明使用真实换行，结构必须是“标题行 + 空行 + 说明正文”，禁止在任意 shell 把 `\n` 当字面文本写入提交消息。
- 强制在提交前检查“用户输入的提交文案”是否包含字面量 `\n`；命中时必须先反转义为真实换行，再执行 git 提交。
- 强制提交后执行一次 commit message 校验：最新提交正文中不得出现字面量 `\n`，命中即视为提交不合格并立即修正。
- 强制 git 提交前先在项目根目录 `README.md` 追加本次改动日志，日志标题使用本次 git 提交标题。
- 强制项目根目录 `README.md` 改动日志按时间正序排列，不能把较晚时间的记录写在较早时间之前。
- 强制 Go 项目在 git 提交前执行 `*_test.go` 禁放扫描，命中则不得提交。
- 强制 Go 项目在 git 提交前检查 staged 文件中是否存在 `internal/service/*.go` 根目录直落实现，命中则不得提交。
- 减少把多个无关目标挤进一次提交的情况。
- 帮助在多人协作下保持变更边界清晰。
- 让交付域真正覆盖到合并前的协作动作。

## 自动触发信号

- 准备提交代码、整理 commit、发起 PR 或同步分支。
- 用户明确输入“提交git”或同义指令，要求直接完成提交流程。
- 变更中混入多个目标，需要收缩提交边界。
- 多人同时修改相邻模块，需要明确协作边界。
- 需要写清 commit message、PR 描述或同步策略。
- 提交代码时未分析未提交的代码，可能可以拆分成多个业务修改。
- 任意 shell 下准备写多行提交说明，但换行写成 `\n` 字面量，或标题与说明之间缺少空行。
- 用户直接粘贴多行提交文案，但正文里含有 `\n` 字面量。
- 提交代码前，项目根目录 `README.md` 尚未记录本次改动日志。
- Go 项目提交前尚未检查 `test/` 外 `*_test.go`。
- Go 项目提交前尚未检查 staged 文件是否命中 `internal/service/[^/]+.go`。

## 进入后先做什么

1. 先确认本次变更目标和边界是否已经收敛。
2. 先盘点当前全部未提交改动（staged、unstaged、untracked）。
3. **分析未提交的代码，如果可以分成不同的业务修改，就用多次不同的 git 提交**。
4. 把提交粒度对齐到可审查、可回滚、可追溯的最小单元。
5. 检查是否混入冲突残留或未整理文件。
6. **确认 git 提交消息遵循 feat/fix 等格式，标题为 `<type>: [中文简要说明] 标题说明`，且结构为“标题行 + 空行 + 说明正文”；禁止 `type(scope): ...` 括号 scope 写法。**
7. **如果用户给的提交文案含 `\n` 字面量，先反转义为真实换行，再进入提交步骤。**
8. **确认提交说明是“真实换行”而不是 `\n` 字符串；优先使用 `git commit -F -` 或多 `-m` 参数。**
9. 每次提交前都在项目根目录 `README.md` 追加对应提交日志，格式为 `yyyy-MM-DD HH:mm:ss 提交标题`，并保持 README 改动日志按时间正序排列。
10. 对 Go 项目执行提交前禁放扫描：`rg --files -g "*_test.go" | rg -v "^test/"`；命中即阻断提交。
11. 对 Go 项目执行 staged 服务层落点扫描：`git diff --cached --name-only | rg "^internal/service/[^/]+\\.go$"`；命中即阻断提交。
12. 每次提交后执行校验：`git log -1 --pretty=%B | rg '\\\\n'`；若命中则立刻 `git commit --amend` 修正为真实换行。
13. 若用户请求为“提交git”，重复执行“分组 -> 提交 -> 校验”循环，直到 `git status --short` 为空再结束。
14. 整理 commit、分支同步方式和必要的 PR 摘要；如果没有 PR，就不额外生成说明记录。

## 默认执行流程

1. 默认先读 `references/branch-and-commit.md`，先确认提交和分支应如何收口。
2. 如需继续展开，再读 `references/sync-and-pr-scope.md`，需要处理同步、合并和 PR 范围。
3. 需要对照边界或正反例时，再读 `references/collaboration-examples.md`，需要对照 Git 协作正反例。
4. 输出 提交整理建议、协作边界、README 改动日志要求和 PR 准备结论。
5. **特别强调：分析未提交的代码，如果可以分成不同的业务修改，就用多次不同的 git 提交**。
6. **正式 git 提交前，必须确认提交消息结构为“标题行 + 空行 + 说明正文”，且标题符合 `<type>: [中文简要说明] 标题说明`；任意 shell 下不得出现字面量 `\n`。**
7. **如果提交文案来自用户输入，且包含字面量 `\n`，必须先做反转义归一化（转成真实换行）再提交。**
8. **正式 git 提交前，必须确认项目根目录 `README.md` 已追加本次改动日志，日志标题与本次提交标题一致，且日志顺序为时间正序。**
9. Go 项目正式提交前，必须确认 `test/` 外不存在 `*_test.go`。
10. Go 项目正式提交前，必须确认 staged 文件中不存在 `internal/service/[^/]+.go`。
11. 提交后必须执行一次 `git log -1 --pretty=%B` 结果检查，禁止出现字面量 `\n`。
12. 当用户请求“提交git”时，必须循环执行拆分提交，直到 `git status --short` 为空；除阻断级规则外，不允许以“还有无关改动”为由提前结束。
13. Git 协作整理完成后，再进入正式代码评审、发布或交付总结流程；不额外要求写入 `delivery/` 之类的持久化说明目录。

## 权责边界与不负责事项

- 只负责 Git 协作，不代替代码评审做质量结论。
- 不把分支操作问题和上线发布问题混写。
- 不为了凑提交而隐藏真实风险或遗漏必要说明。
- 不鼓励把大量不相干修改压进一次提交。
- **必须分析未提交的代码，拆分成不同的业务修改多次提交**。
- **当用户请求“提交git”时，必须把当前未提交改动全部纳入提交闭环；可以多次提交，但不能遗漏。**
- **不允许在“提交git”请求下以“无关改动”名义留存未提交文件并直接结束。**
- **任意 shell 下都必须保证提交说明为真实换行，不允许把 `\n` 作为普通字符写入 git 提交消息。**
- **提交标题必须统一为 `<type>: [中文简要说明] 标题说明`。**
- **不允许使用 Conventional Commits 的括号 scope 标题写法（如 `feat(reqsend): ...`）；必须写成 `feat: [reqsend] ...`。**
- **`中文简要说明` 必须是直接描述，不允许出现功能/业务/接口等分类前缀，也禁止直接出现 API URL 或方法+路径。**
- **如果用户输入的提交文案含 `\n`，必须先归一化成真实换行，不能直接原样提交。**
- **提交后必须执行 commit message 校验；如果命中字面量 `\n`，必须当场修正，不允许带错继续推进。**
- **必须在 git 提交前更新项目根目录 `README.md` 改动日志，不允许遗漏。**
- **必须保证项目根目录 `README.md` 改动日志按时间正序排列；如果本次补记时间早于文件末尾已有日志，必须插入到正确位置，不能机械追加到最后。**
- **Go 项目命中 `test/` 外 `*_test.go` 时必须阻断提交，不允许“先提后改”。**
- **Go 项目命中 staged `internal/service/[^/]+.go` 时必须阻断提交，不允许“先提后改”。**

## 需要暂停并确认的条件

- 当前改动边界仍不清，无法整理成稳定提交。
- 存在明显无关改动、冲突残留或脏文件未区分。
- 多人改动边界冲突，但责任和合并策略未确认。
- PR 准备阶段缺少需求背景、测试结论或风险说明。
- 用户明确要求“只提交指定文件/指定目录”，与“提交全部未提交改动”目标冲突，且范围尚未确认。
- 提交说明换行方式不明确，无法保证不会把 `\n` 字面量写进提交消息。
- 用户提供了带 `\n` 的提交文案，但未确认是否先反转义归一化。
- 提交后校验命中字面量 `\n`，但尚未确定是否允许 amend 修正。
- 准备提交代码，但项目根目录 `README.md` 还没有补充本次改动日志。
- 项目根目录 `README.md` 已补改动日志，但日志顺序出现时间倒挂。
- Go 项目扫描命中 `test/` 外 `*_test.go`，但改 seam 与迁移方案尚未确认。
- Go 项目扫描命中 staged `internal/service/[^/]+.go`，但业务子目录拆分方案尚未确认。

## 执行通过 / 驳回标准

- 通过：提交粒度清晰、说明完整、协作边界明确，并且后续评审和回滚都能基于这些提交稳定开展；提交代码时分析了未提交的代码，按业务修改拆分成了多次提交；git 提交消息遵循 feat/fix 等格式，标题符合 `<type>: [中文简要说明] 标题说明`，结构符合“标题行 + 空行 + 说明正文”；任意 shell 下未出现 `\n` 字面量误写且提交后校验通过；git 提交前已在项目根目录 `README.md` 追加本次改动日志，格式为 `yyyy-MM-DD HH:mm:ss 提交标题`，且日志顺序保持时间正序；Go 项目中 `test/` 外不存在 `*_test.go`，且 staged 文件未命中 `internal/service/[^/]+.go`；当请求为“提交git”时，最终 `git status --short` 为空。
- 驳回：提交杂糅无关内容、说明空泛、协作边界混乱，导致评审和回滚成本显著上升；未分析未提交的代码就一次提交所有业务修改；git 提交消息格式不符合规范（缺少 `[中文简要说明]`、或使用 `type(scope): ...` 括号 scope 写法、或 `[]` 内仍写功能/业务/接口等前缀、或直接写 API URL），或在任意 shell 下把 `\n` 字面量写进提交说明且提交后未修正；git 提交前未更新项目根目录 `README.md` 改动日志，或 README 改动日志出现时间倒挂；或 Go 项目提交前扫描命中 `test/` 外 `*_test.go`，或 staged 文件命中 `internal/service/[^/]+.go`；或用户请求“提交git”后仍遗留未提交改动。

## 执行结果归档要求

- 默认不要求额外写入 `delivery/`、评审记录或其他持久化说明文件。
- 默认以 git 提交消息、提交拆分结果和项目根目录 `README.md` 改动日志作为持久化记录。
- 如果确实存在 PR，则将本次协作摘要写入 PR 描述；如果没有 PR，不额外补说明文档。
- 如果进行了多次拆分提交，应确保每个提交自己的标题和说明已经能独立表达范围，不再额外复制到别处。
- 项目根目录 `README.md` 必须保留本次改动日志，日志格式统一为 `yyyy-MM-DD HH:mm:ss 提交标题`，例如：`2026-03-29 15:30:00 fix: 修复订单金额计算错误`。
- 项目根目录 `README.md` 的改动日志必须保持按时间正序排列，即旧记录在前、新记录在后；如果是补记历史提交，必须插入到对应时间位置。
