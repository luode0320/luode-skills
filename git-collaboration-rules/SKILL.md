# Git 协作规则

只在需要处理 Git 协作动作，而不是代码内容本身时使用这个 skill。
如果当前问题是代码质量或行为风险，请先转交实现自审；如果已经进入上线动作，请转回团队发布流程处理。

## Skill 作用与适用场景

- 规范提交粒度、提交说明、分支同步和 PR 整理。
- 强制分析未提交的代码，如果可以分成不同的业务修改，就用多次不同的 git 提交。
- 强制 git 提交消息遵循 feat/fix 等格式，包含标题和说明。
- 减少把多个无关目标挤进一次提交的情况。
- 帮助在多人协作下保持变更边界清晰。
- 让交付域真正覆盖到合并前的协作动作。

## 自动触发信号

- 准备提交代码、整理 commit、发起 PR 或同步分支。
- 变更中混入多个目标，需要收缩提交边界。
- 多人同时修改相邻模块，需要明确协作边界。
- 需要写清 commit message、PR 描述或同步策略。
- 提交代码时未分析未提交的代码，可能可以拆分成多个业务修改。

## 进入后先做什么

1. 先确认本次变更目标和边界是否已经收敛。
2. **分析未提交的代码，如果可以分成不同的业务修改，就用多次不同的 git 提交**。
3. 把提交粒度对齐到可审查、可回滚、可追溯的最小单元。
4. 检查是否混入无关改动、冲突残留或未整理文件。
5. **确认 git 提交消息遵循 feat/fix 等格式，包含标题和说明**。
6. 整理 commit、分支同步方式和 PR 摘要。

## 默认执行流程

1. 默认先读 `references/branch-and-commit.md`，先确认提交和分支应如何收口。
2. 如需继续展开，再读 `references/sync-and-pr-scope.md`，需要处理同步、合并和 PR 范围。
3. 需要对照边界或正反例时，再读 `references/collaboration-examples.md`，需要对照 Git 协作正反例。
4. 输出 提交整理建议、协作边界和 PR 准备结论。
5. **特别强调：分析未提交的代码，如果可以分成不同的业务修改，就用多次不同的 git 提交**。
6. Git 协作整理完成后，再进入正式代码评审、发布或交付总结流程。

## 权责边界与不负责事项

- 只负责 Git 协作，不代替代码评审做质量结论。
- 不把分支操作问题和上线发布问题混写。
- 不为了凑提交而隐藏真实风险或遗漏必要说明。
- 不鼓励把大量不相干修改压进一次提交。
- **必须分析未提交的代码，拆分成不同的业务修改多次提交**。

## 需要暂停并确认的条件

- 当前改动边界仍不清，无法整理成稳定提交。
- 存在明显无关改动、冲突残留或脏文件未区分。
- 多人改动边界冲突，但责任和合并策略未确认。
- PR 准备阶段缺少需求背景、测试结论或风险说明。
- 未提交的代码可以拆分成多个业务修改，但用户要求一次提交所有。

## 执行通过 / 驳回标准

- 通过：提交粒度清晰、说明完整、协作边界明确，并且后续评审和回滚都能基于这些提交稳定开展；提交代码时分析了未提交的代码，按业务修改拆分成了多次提交；git 提交消息遵循 feat/fix 等格式。
- 驳回：提交杂糅无关内容、说明空泛、协作边界混乱，导致评审和回滚成本显著上升；未分析未提交的代码就一次提交所有业务修改；git 提交消息格式不符合规范。

## 执行结果归档要求

- 将 PR 摘要、提交整理说明和协作边界记录到 PR 描述、`delivery/` 或评审记录。
- 归档内容至少包含变更目标、主要提交、测试结论和待关注风险。
- 涉及多人协作冲突处理时，应保留处理依据和结果。
- 如果进行了多次拆分提交，应记录拆分的理由和每个提交的范围。

## references 读取规则

- 默认先读 `references/branch-and-commit.md`。
- 只有在确定同步策略和 PR 范围说明时，再读 `references/sync-and-pr-scope.md`。
- 只有在对照样例或判断提交流程是否合格时，再读 `references/collaboration-examples.md`。
