# Git 协作规则（最小闭环版）

## -1.4 极简硬闸门（新增，强制）

- 识别到 Git 协作意图后，首条中间进度第一行必须输出：
  - `命中检查:通过; Git规则:通过`
- 首条中间进度第二行必须输出：
  - `命中技能:skill-hit-check-rules,git-collaboration-rules`
- 若未输出该行，仅允许执行盘点命令：
  - `git status --short`
  - `git diff --cached --stat`
- 执行其它 git 命令视为违规，判定 `阻断`。

## -1.5 首条输出收敛（新增，强制）

- 首条中间进度仅强制要求上述两行命中信息。
- 不再强制输出 `前置自检`、`Git联动`、`自恢复`、首条证据三要素文案。

## -1.6 违规自恢复（新增，强制）

- 若发生“未完成首条命中检查就执行 git 命令”，必须立即进入恢复模式：

1. 立即停止非盘点 git 命令。
2. 重走首条两行命中输出，完成后才允许继续。

## -1. 触发确认（强制）

当用户请求包含 `git/commit/提交/push/pull/status/diff/log` 任一关键词时，必须命中本 skill；未命中不得执行 git 命令。

## -1.8 当前轮次授权边界（新增，强制）

- Git 协作授权只来自“当前这轮用户消息”，不得继承、沿用或默认延续之前轮次的 `提交git`、`commit一下`、`推送代码` 等表述。
- 若当前这轮用户消息没有明确出现 Git 协作意图，即使更早轮次出现过“提交git”，本轮也一律不得执行 `git commit`、`git push`、`git pull` 等 Git 协作命令。
- “上一轮用户说过要提交”“上文已经授权过提交”“之前已经进入 Git 模式”都不能作为本轮继续提交的依据。
- 若当前轮 Git 协作动作同步形成可复用事实、决策、流程、定义、偏好、来源或调试经验，先按 `obsidian-knowledge-flow` 做 `Obsidian:沉淀` 判定，再继续 Git 协作收口；该判定只负责知识捕获，不构成 `git commit` / `git push` 的授权依据。
- 若代理准备执行 Git 协作命令时，发现当前这轮用户消息并未重新给出 Git 意图，必须立即停止并按未命中处理。

## -1.0 新会话首轮保障（强制）

- 新会话第一条消息只要出现 Git 协作意图，必须立刻命中本 skill，不得等待后续轮次确认。
- 即使用户仅输入极短指令（如 `提交git`、`push`、`看下状态`），也必须按命中处理。
- 命中后第一条中间进度先输出证据模板，再执行除盘点外的 Git 命令。

## -1.1 语义触发扩展（强制）

以下表达即使不出现关键词，也必须命中本 skill：

- `提交git`、`帮我提交`、`commit一下`、`把改动提交到分支`
- `推送代码`、`给我推上去`、`同步到远端`
- `看下git状态`、`看下改动`、`看看最近提交`

判定原则：只要用户意图是让代理执行或检查 Git 协作动作，即视为命中，不得以“缺少明确 git 关键词”回避。

## -1.2 联动前置（强制）

- 本 skill 一旦命中，必须同时命中 `skill-hit-check-rules`。
- 若首条中间进度缺少两行命中信息，则 `阻断`，除盘点命令外禁止继续执行 Git 命令。

## -1.3 口语与缩写兜底（强制）

以下输入默认视为 Git 协作，不得漏判：

- 单词/短句：`提交`、`提交代码`、`提一下`、`交一下`、`推一下`、`拉一下`、`同步代码`
- 中英混写：`提交git`、`git提交`、`commit下`、`push下`
- 省略主语命令式：`帮我弄到远端`、`把这次改动交了`

## -1.7 用户显式放行未由代理改动文件（新增，强制）

- 当工作区存在“未由代理产生”的改动时，默认应先停下并征求确认。
- 若用户在当前轮次明确表达“这些也要提交/一起提交/都提交”，视为显式放行。
- 命中显式放行后：可将这些文件纳入本次提交，不再重复阻断询问。
- 最终回执必须注明：哪些文件属于用户显式放行后一并提交。

## 0. 首条中间进度（强制）

必须先输出两行：

- `命中检查:通过; Git规则:通过`
- `命中技能:skill-hit-check-rules,git-collaboration-rules`

## 1. 执行短清单（强制按顺序）

1. 盘点：`git status --short`、`git diff --cached --stat`
2. 业务拆分：可按改动主题拆分为多次提交（不要求一次提交完成）；默认提交域隔离如下：
   - `doc/2-需求/`、`doc/3-实施/`、`doc/4-bugs/`、`doc/5-tests/`、`doc/6-审查/`、`doc/7-验收/` 各自单独提交
   - 测试文件（至少包含 `doc/5-tests/**`、`*_test.*`、`*.spec.*`、`*.test.*`）归入测试提交，不与代码实现混提
   - 代码实现 / 运行配置单独提交，不与上述流程文档域或测试文件混提
3. 提交前基础代码核查与基础验收：每次 `git commit` 均以当前 staged 改动为准执行，不读取、不要求、也不以 `doc/6-审查/` 或 `doc/7-验收/` 的最近文档作为前置条件；不默认联动 `project-change-review-rules` 的完整审查流程或 `final-acceptance-rules` 的正式最终验收。
   - 格式：先执行 `git diff --cached --check`；若 staged 包含代码、脚本、配置或测试，再复用项目已定义的格式化检查命令并记录结果。
   - 注释：核对本次变更中的非直观业务规则、并发约束、错误处理和边界行为是否已有准确且未过期的必要注释。
   - 安全性：核对新增或调整的输入、权限、敏感信息、日志、文件/网络访问及注入面没有明显误放行、泄露或不安全默认值。
   - 并发安全性：核对共享可变状态、锁、goroutine/线程、取消、资源关闭和异步回调不存在竞态、死锁、泄漏或重复执行风险。
    - 系统崩溃风险：核对空值、panic/异常、数组或切片越界、类型断言、资源耗尽及错误传播不会让进程或关键链路意外中断。
    - 边界条件：核对空值、零值、范围/溢出、超时、分页/批量上限、状态顺序与外部依赖失败路径。
    - 基础验收：按当前 staged 改动的实际风险确认项目已有的测试、功能验证或等价验证是否已覆盖；没有适用验证时明确记录“不适用 + 原因”。
    - 纯文档提交也必须完成本项核查；其中没有关联源码的维度明确记为“不适用 + 原因”，不得要求额外审查文档。
    - 本步骤只在 Git 提交证据中记录审查与验收结论，禁止仅因提交而自动生成 `doc/6-审查/` 或 `doc/7-验收/` 文档；只有用户显式发起独立总审查或正式最终验收时，才回到对应 skill 的归档流程。
    任一维度发现 P0/P1、关键注释缺失、安全/并发/崩溃风险、边界缺陷、验证缺口或明确阻断项时，必须先修复或补齐验证再提交。
4. 写 README 日志：根目录 `README.md` 在改动日志最后一条提交日志后追加一行 `<yyyy-MM-dd HH:mm:ss> <提交标题>`，时间戳使用当前北京时间
5. 设定标题：`<type>: [中文简要说明] 标题说明`
6. 提交前闸门：优先执行本 skill 自带 `scripts/pre_commit_gate.sh "<title>"`；若不可用再查找仓库根目录 `scripts/pre_commit_gate.sh "<title>"`；均不可用才执行回退检查，失败即阻断。闸门负责 `git diff --cached --check`、标题格式、README 日志匹配、提交域隔离、Go 测试落点和服务层落点等机械检查；第 3 步的基础代码核查仍须由当前 staged 代码本身完成，不能被脚本或审查文档替代。
7. 执行提交：`git commit`（正文必须真实换行，禁止字面量 `\n`）
8. 提交后闸门：优先执行本 skill 自带 `scripts/post_commit_gate.sh`；若不可用再查找仓库根目录 `scripts/post_commit_gate.sh`；均不可用才执行回退检查，失败即 `git commit --amend` 修正
9. 若用户请求“提交git”：重复 1-8，直到 `git status --short` 为空
10. 最终回执：按时间顺序列出本次执行产生的全部 commit（hash + 标题），并逐项给出本轮基础代码核查与基础验收的覆盖范围、通过/不适用结论与阻断项（如有）；不生成审查或验收文档。

## 1.1 脚本查找与缺失回退（强制）

脚本优先级：

1. 本 skill 目录自带脚本：`<git-collaboration-rules>/scripts/pre_commit_gate.sh`、`<git-collaboration-rules>/scripts/post_commit_gate.sh`
2. 仓库根目录脚本：`scripts/pre_commit_gate.sh`、`scripts/post_commit_gate.sh`
3. 两处都不存在或不可执行时，执行等价回退检查，并在最终证据中标注 `回退`

不得仅因仓库根目录缺少 `scripts/pre_commit_gate.sh` 或 `scripts/post_commit_gate.sh` 判定阻断；系统级 skill 自带脚本存在且可执行时，应直接使用系统级脚本。

- pre 回退检查：
  - 标题匹配 `<type>: [中文简要说明] 标题说明`
  - `README.md` 的改动日志末尾一行（剥除 `yyyy-MM-dd HH:mm:ss ` 时间戳前缀后）等于本次提交标题
  - `doc/2-需求/`、`doc/3-实施/`、`doc/4-bugs/`、`doc/5-tests/`、`doc/6-审查/`、`doc/7-验收/` 六类流程产物域不得在同一次提交中混提
  - staged 若同时包含代码实现 / 运行配置与流程产物域或测试文件（至少 `doc/5-tests/**`、`*_test.*`、`*.spec.*`、`*.test.*`）则阻断
  - 本次提交涉及的新增/修改 Go 测试文件不得位于 `doc/5-tests/` 目录外；历史已存在、未纳入本次提交的 `doc/5-tests/` 外 `*_test.go` 不作为阻断项
  - staged 不命中 `internal/service/[^/]+.go`
  - `git diff --cached --check` 通过；格式化、注释、安全性、并发安全性、系统崩溃风险、边界条件和基础验收仍必须按第 3 步逐项完成，不能由回退检查或审查/验收文档省略
- post 回退检查：
  - `git log -1 --pretty=%B` 不包含字面量 `\\n`

缺任一回退检查视为未完成，不得给“已完成”。

## 2. 阻断条件（脚本化/回退等价）

以下任一命中必须停止提交并回报阻断：

- 当前这轮用户消息未明确给出 Git 协作意图，却试图沿用历史轮次授权继续提交 / 推送
- 标题不匹配 `<type>: [中文简要说明] 标题说明`
- `README.md` 改动日志末尾一行剥除时间戳前缀后不等于本次提交标题
- `doc/2-需求/`、`doc/3-实施/`、`doc/4-bugs/`、`doc/5-tests/`、`doc/6-审查/`、`doc/7-验收/` 中命中多个流程产物域仍尝试同一次提交
- staged 同时命中代码实现 / 运行配置与流程产物域或测试文件（至少 `doc/5-tests/**`、`*_test.*`、`*.spec.*`、`*.test.*`）
- 本次提交新增/修改的 Go 测试文件位于 `doc/5-tests/` 目录外
- staged 命中 `internal/service/[^/]+.go`
- 当前 staged 改动的基础代码核查或基础验收发现 P0/P1、格式错误、关键注释缺失、安全/并发/崩溃风险、边界缺陷、验证缺口、明显潜在 bug，或仍有未完成的明确阻断项
- 最新 commit 正文含字面量 `\\n`

## 3. 统一证据输出（强制）

最终回复必须贴出并填写：

- `templates/evidence-template.md` 中的 **Skill 执行证据清单** 区块（首条证据区块不再强制）。
- 且必须追加固定校验行：
  - `命中检查:通过`
  - `Git规则:通过`

## 4. 通过标准

- 闸门脚本存在时均为通过；若脚本缺失，回退检查全部通过 且证据标注 `回退`
- 每次提交均已对当前 staged 改动完成基础代码核查与基础验收；格式、注释、安全性、并发安全性、系统崩溃风险、边界条件和验证适用性均已通过或被明确标注为不适用，且未发现阻断项
- Git 提交未因基础核查与基础验收自动生成 `doc/6-审查/` 或 `doc/7-验收/` 文档
- 提交标题与正文格式合规
- 每个 commit 只覆盖一个流程产物域或一个代码实现域，测试文件不与代码实现混提
- 用户请求“提交git”时工作区清空（`git status --short` 为空），且允许多次提交完成清空
- 证据模板已完整填写
- 固定校验行完整输出（`命中检查:通过`、`Git规则:通过`）
