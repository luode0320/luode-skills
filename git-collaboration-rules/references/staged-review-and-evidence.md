# Staged 基础核查与证据

每次 `git commit` 前均以当前 staged 改动为准执行本文件；最近的审查或验收文档不能替代，也不能被当作提交前置依赖。

### 基础代码核查

- 格式：运行 `git diff --cached --check`；若包含代码、脚本、配置或测试，再复用项目已有格式检查命令。
- 注释：非直观业务规则、并发约束、错误处理和边界行为具有准确且未过期的必要注释。
- 安全性：输入、权限、敏感信息、日志、文件/网络访问和注入面没有明显误放行、泄露或不安全默认值。
- 并发安全性：共享状态、锁、线程/goroutine、取消、资源关闭和异步回调没有明显竞态、死锁、泄漏或重复执行风险。
- 系统崩溃风险：空值、panic/异常、越界、类型断言、资源耗尽和错误传播不会让进程或关键链路意外中断。
- 边界条件：空值、零值、范围/溢出、超时、分页/批量上限、状态顺序与外部依赖失败路径已覆盖。

发现 P0/P1、关键注释缺失、安全/并发/崩溃风险、边界缺陷或明显潜在 Bug 时，必须先修复或补齐验证再提交。

### 基础验收

按 staged 改动的实际风险确认已有真实测试、功能验证或等价验证是否覆盖。纯文档也必须核查；无关联源码的维度写明“不适用 + 原因”。不得因普通提交自动生成 `doc/6-审查/` 或 `doc/7-验收/` 文档，只有用户显式发起独立总审查或正式最终验收时才进入对应归档流程。

### 最终证据

使用 `templates/evidence-template.md`，按时间顺序列出：

- 本轮产生的全部 commit hash 与标题。
- 每个 commit 的提交域和文件范围。
- 格式、注释、安全、并发、崩溃、边界和基础验收结论；不适用项附原因。
- pre/post gate 使用的脚本或回退路径及结果。
- 用户显式放行后一并提交的文件。
- 剩余阻断项和未清空文件；没有则明确为无。

最终固定校验行仍按 `skill-hit-check-rules` 与当前 Git 场景要求输出，不在此复制首条格式。
