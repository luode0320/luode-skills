# 同步与 PR 范围

## 多次提交原则

- **提交代码时必须分析未提交的代码，如果可以分成不同的业务修改，就用多次不同的 git 提交**。
- 不要一次提交所有导致业务没有区分清楚。
- 当用户明确请求“提交git”时，必须覆盖当前全部未提交改动，允许多次提交但不得遗漏。
- 当用户明确请求“提交git”时，结束前必须确认 `git status --short` 为空。
- PR 可以包含多个提交，但每个提交应该有独立的业务目标。
- `doc/2-需求/`、`doc/3-实施/`、`doc/4-bugs/`、`doc/5-tests/`、`doc/6-审查/`、`doc/7-验收/` 六类流程产物默认各自单独提交。
- 测试文件（至少 `doc/5-tests/**`、`*_test.*`、`*.spec.*`、`*.test.*`）归入测试提交；代码实现 / 运行配置单独提交，不与这些文档域或测试文件混提。
- PR 描述要覆盖目标、主要改动、测试结论和风险点。
- 如果没有 PR，就不额外生成持久化说明文件；默认依赖 git 提交消息和根目录 `README.md` 改动日志留痕。

## PR 描述内容

PR 描述应包含：
- 需求背景和目标
- 主要改动内容（按提交列表，并保持需求 / 实施 / Bug / 测试 / 审查 / 验收 / 代码实现的提交域边界）
- 测试结论
- 风险点和注意事项
- 验证方式

## 同步原则

- 同步分支前先确认本地改动边界，避免把不相关更新一起带入。
- 多人协作时要明确谁负责哪个模块和冲突处理原则。
- PR 范围过大时，应先回到需求或提交整理阶段收缩。
- 每次同步前确保本地提交已经按业务拆分完成。
- 每次同步前确认没有把代码实现和测试 / 审查 / 验收 / 需求 / 实施 / Bug 文档混在同一个 commit。
- 若同一轮工作同时产出需求、实施、测试、Bug、审查、验收和代码实现，优先整理成多个 commit 再同步，而不是靠 PR 描述硬解释混提原因。
- Go 项目每次同步或提交前都应先确认 `doc/5-tests/` 外不存在 `*_test.go`。
- Go 项目每次同步或提交前都应先确认 staged 文件中不存在 `internal/service/[^/]+.go`。
