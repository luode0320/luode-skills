# 同步与 PR 范围

## 多次提交原则

- **提交代码时必须分析未提交的代码，如果可以分成不同的业务修改，就用多次不同的 git 提交**。
- 不要一次提交所有导致业务没有区分清楚。
- PR 可以包含多个提交，但每个提交应该有独立的业务目标。
- PR 描述要覆盖目标、主要改动、测试结论和风险点。

## PR 描述内容

PR 描述应包含：
- 需求背景和目标
- 主要改动内容（按提交列表）
- 测试结论
- 风险点和注意事项
- 验证方式

## 同步原则

- 同步分支前先确认本地改动边界，避免把不相关更新一起带入。
- 多人协作时要明确谁负责哪个模块和冲突处理原则。
- PR 范围过大时，应先回到需求或提交整理阶段收缩。
- 每次同步前确保本地提交已经按业务拆分完成。
