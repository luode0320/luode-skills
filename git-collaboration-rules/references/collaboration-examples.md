# 协作样例

## 更好

### 正例 1：按业务拆分成多次提交

假设同时修改了用户登录和订单创建两个功能，应该分成两次提交：

**第一次提交：**
```
feat: 新增用户登录功能

说明：
- 支持手机号验证码登录
- 支持密码登录
- 登录成功后返回用户基本信息和 token
```

**第二次提交：**
```
feat: 新增订单创建功能

说明：
- 支持创建普通订单
- 支持创建预约订单
- 订单创建成功后返回订单号
```

### 正例 2：提交消息格式正确

```
feat: 新增用户登录功能

说明：
- 支持手机号验证码登录
- 支持密码登录
- 登录成功后返回用户基本信息和 token
```

```
fix: 修复订单创建时金额计算错误

说明：
- 问题：订单创建时优惠券折扣计算错误
- 原因：折扣计算逻辑中缺少对满减条件的判断
- 影响：已创建的订单金额可能不正确
```

### 正例 3：先拆出功能提交，再单独补测试和文档说明

```
feat: 新增用户登录功能

说明：
- 支持手机号验证码登录
- 支持密码登录
```

```
test: 补充用户登录功能测试用例

说明：
- 添加正常登录测试
- 添加异常登录测试
- 添加验证码过期测试
```

```
docs: 更新用户登录功能文档

说明：
- 更新 API 接口文档
- 添加使用示例
- 添加注意事项
```

### 正例 4：PR 描述写清需求背景、风险点和验证方式

```
## 需求背景
用户反馈登录流程复杂，需要优化。

## 主要改动
1. 新增手机号验证码登录
2. 优化密码登录体验
3. 调整登录成功后的返回数据

## 测试结论
- 所有测试用例通过
- 性能测试达标
- 兼容性测试通过

## 风险点
- 验证码接口可能有并发问题
- 需要关注第三方短信服务稳定性

## 验证方式
1. 使用测试账号登录验证
2. 检查日志无异常
3. 验证 token 有效期正确
```

## 较差

### 反例 1：一个提交里同时包含功能、重构、格式化和调试残留

```
fix: 修复一些问题

说明：修复了一些 bug
```

❌ 问题：一次提交包含用户登录、订单创建、代码格式化、调试代码等多个不相关的修改，应该按业务拆分成多次提交。

### 反例 2：提交消息格式不正确，只有标题没有说明

```
feat: 新增功能
```

❌ 问题：没有说明，不知道具体做了什么。

```
fix: 修复 bug
```

❌ 问题：没有说明，不知道修复了什么 bug、原因是什么、影响范围是什么。

### 反例 3：PR 只写"fix bug"或"update code"

```
fix bug
```

❌ 问题：PR 描述过于简单，没有需求背景、主要改动、测试结论、风险点等信息。

### 反例 4：一次提交所有业务修改

假设同时修改了用户登录、订单创建、商品管理三个功能，却只做了一次提交：

```
feat: 新增一些功能

说明：
- 用户登录
- 订单创建
- 商品管理
```

❌ 问题：应该按业务拆分成三次独立的提交，而不是一次提交所有。
