# 协作样例

## 更好

### 正例 1：按业务拆分成多次提交

假设同时修改了用户登录和订单创建两个功能，应该分成两次提交：

**第一次提交：**
```
feat: [用户登录] 新增密码与验证码双登录能力

说明：
- 支持手机号验证码登录
- 支持密码登录
- 登录成功后返回用户基本信息和 token
```

**第二次提交：**
```
feat: [订单创建] 新增普通与预约订单创建能力

说明：
- 支持创建普通订单
- 支持创建预约订单
- 订单创建成功后返回订单号
```

### 正例 2：提交消息格式正确

```
feat: [用户登录] 新增密码与验证码双登录能力

说明：
- 支持手机号验证码登录
- 支持密码登录
- 登录成功后返回用户基本信息和 token
```

```
fix: [订单结算] 修复订单创建时金额计算错误

说明：
- 问题：订单创建时优惠券折扣计算错误
- 原因：折扣计算逻辑中缺少对满减条件的判断
- 影响：已创建的订单金额可能不正确
```

### 正例 3：先拆出功能提交，再单独补测试和文档说明

```
feat: [用户登录] 新增密码与验证码双登录能力

说明：
- 支持手机号验证码登录
- 支持密码登录
```

```
test: [用户登录] 补充登录流程测试用例

说明：

```

### 正例 4：PR 描述写清需求背景、风险点和验证方式

```
## 需求背景
用户反馈登录流程复杂，需要优化。

## 主要改动
1. 新增手机号验证码登录
2. 优化密码登录体验
3. 调整登录成功后的返回数据

## 测试结论
- 所有测试用例通过
- 性能测试达标
- 兼容性测试通过

## 风险点
- 验证码接口可能有并发问题
- 需要关注第三方短信服务稳定性

## 验证方式
1. 使用测试账号登录验证
2. 检查日志无异常
3. 验证 token 有效期正确
```

### 正例 5：提交前先补项目根目录 README 改动日志

假设本次准备提交的标题为：

```
fix: [订单结算] 修复订单创建时金额计算错误
```

那么提交前，应先在项目根目录 `README.md` 中追加：

```
2026-07-01 14:25:30 fix: [订单结算] 修复订单创建时金额计算错误
```

如果这次拆成两次提交，就应追加两条日志，且每条日志都和对应提交标题一致。

如果 README 中连续记录两次提交，正确写法是始终往末尾追加：

```text
2026-06-28 01:12:00 feat: [平台安全] 增加内网来源 IP 白名单中间件
2026-06-28 04:17:15 refactor: [租户会话与订单路由] 拆分注册函数
```

而不应把新日志插到最后一条旧日志前面：

```text
2026-06-28 04:17:15 refactor: [租户会话与订单路由] 拆分注册函数
2026-06-28 01:12:00 feat: [平台安全] 增加内网来源 IP 白名单中间件
```

✅ 结果：README 改动日志总是从末尾继续追加，执行动作稳定，不需要为了时间顺序回插历史位置。

### 正例 8：Windows/PowerShell 下多行提交说明换行正确

```powershell
git commit -m "fix: [订单结算] 修复订单创建时金额计算错误" -m @"
说明：
- 问题：订单创建时优惠券折扣计算错误
- 原因：折扣计算逻辑中缺少对满减条件的判断
- 影响：已创建的订单金额可能不正确
"@
```

✅ 结果：标题与说明之间由 git 自动分段，说明正文是可读的真实换行，没有 `\n` 字面量残留。

### 正例 9：收到“提交git”后分两次提交并清空工作区

假设当前有两组未提交改动：A（注释规则）和 B（前端规则），应执行：

1. 先提交 A，写入对应 README 改动日志。
2. 再提交 B，写入对应 README 改动日志。
3. 最后检查 `git status --short` 为空再结束。

✅ 结果：既按业务拆分提交，又没有遗留未提交改动。


### 正例 10：提交前有通过的审查文档

提交前检查 `doc/6-审查/` 下最近一份审查文档：

```text
审查结论: 通过
审查范围: git-collaboration-rules/SKILL.md, git-collaboration-rules/scripts/pre_commit_gate.sh
是否允许提交: 是
阻断问题: 无
```

确认审查结论为通过、允许提交为是、无未解决 P0/P1，再继续提交。

✅ 结果：审查文档覆盖本次改动范围，且结论为通过。

### 正例 11：需求、实施、测试、审查、验收和代码实现分开提交

同一轮工作同时产出需求文档、实现代码、测试文件、审查文档和验收文档时，可以这样拆：

```text
docs: [稳定币费率需求] 补充清算需求说明
feat: [费率计算] 新增稳定币清算规则
docs: [费率实现方案] 归档实施说明
test: [费率计算] 补充结算回归用例
docs: [费率审查] 归档本次改动审查结论
docs: [费率验收] 归档最终验收结果
```

其中：

- `doc/2-需求/` 单独一笔
- 代码实现 / 运行配置单独一笔
- `doc/3-实施/` 单独一笔
- `doc/5-tests/`、`*_test.*`、`*.spec.*`、`*.test.*` 统一归入测试提交
- `doc/6-审查/` 单独一笔
- `doc/7-验收/` 单独一笔

✅ 结果：人工 review 时可以直接按提交域检查，不需要在一个 commit 里来回切换代码、文档和测试。

### 反例 10：提交前没有通过的审查文档

提交前检查 `doc/6-审查/` 下最近一份审查文档：

```text
审查结论: 阻断
审查范围: git-collaboration-rules/SKILL.md
是否允许提交: 否
阻断问题: [P1] SKILL.md:91 - 闸门描述不完整
```

仍然继续提交。

❌ 问题：审查文档结论为阻断、不允许提交，必须先修复阻断问题并重新审查。

### 反例 11：审查文档范围不覆盖当前待提交改动

当前 staged 文件包含 `implementation-review-rules/SKILL.md`，但最近一份审查文档的审查范围只覆盖 `git-collaboration-rules/SKILL.md`。

仍然继续提交。

❌ 问题：审查范围未覆盖当前待提交文件，应先补齐覆盖该文件的审查文档再提交。
## 较差

### 反例 1：一个提交里同时包含功能、重构、格式化和调试残留

```
fix: 修复一些问题

说明：修复了一些 bug
```

❌ 问题：一次提交包含用户登录、订单创建、代码格式化、调试代码等多个不相关的修改，应该按业务拆分成多次提交。

### 反例 2：提交消息格式不正确，只有标题没有说明

```
feat: 新增功能
```

❌ 问题：没有说明，不知道具体做了什么。

```
fix: 修复 bug
```

❌ 问题：没有说明，不知道修复了什么 bug、原因是什么、影响范围是什么。

### 反例 2.1：错误使用 `type(scope):` 括号 scope 格式

```text
feat(reqsend): 转账详细统计支持多Sheet导出错误明细
```

❌ 问题：标题格式不符合本规则。禁止 `type(scope): ...` 写法，必须改成 `type: [scope] ...`。

正确写法：

```text
feat: [reqsend] 转账详细统计支持多Sheet导出错误明细
```

### 反例 3：PR 只写"fix bug"或"update code"

```
fix bug
```

❌ 问题：PR 描述过于简单，没有需求背景、主要改动、测试结论、风险点等信息。

### 反例 4：一次提交所有业务修改

假设同时修改了用户登录、订单创建、商品管理三个功能，却只做了一次提交：

```
feat: 新增一些功能

说明：
- 用户登录
- 订单创建
- 商品管理
```

❌ 问题：应该按业务拆分成三次独立的提交，而不是一次提交所有。

### 反例 5：提交前没有更新项目根目录 README 改动日志

准备提交：

```
fix: [订单结算] 修复订单创建时金额计算错误
```

但项目根目录 `README.md` 没有追加任何本次改动记录，或者只写了下面这种模糊内容：

```
今天修了一个 bug
更新了一些内容
```

❌ 问题：日志没有直接复用本次提交标题，后续无法按 README 记录快速追溯每次提交的变更。

### 反例 5.1：README 改动日志没有追加到末尾

```text
2026-06-28 04:17:15 refactor: [租户会话与订单路由] 拆分注册函数
2026-06-28 01:12:00 feat: [平台安全] 增加内网来源 IP 白名单中间件
```

❌ 问题：新日志没有接在现有改动日志末尾，说明执行时没有遵守“始终最后追加”的规则。

### 反例 6：Go 项目命中禁放仍继续提交

扫描结果命中：

```text
internal/service/order_service_test.go
```

仍然执行提交。

❌ 问题：违反测试落点基线，应先改 seam 并迁移到 `doc/5-tests/` 根目录体系后再提交。

### 反例 7：Go 项目命中服务层根目录直落仍继续提交

扫描结果命中：

```text
internal/service/create_order.go
```

仍然执行提交。

❌ 问题：违反服务层子目录化规则，应先迁移到 `internal/service/<domain>/` 后再提交。

### 反例 8：Windows/PowerShell 下把 `\n` 当换行写进提交消息

```powershell
git commit -m "fix: [订单结算] 修复订单创建时金额计算错误\n\n说明：\n- 问题：订单创建时优惠券折扣计算错误"
```

❌ 问题：在 PowerShell 中 `\n` 不是换行转义，通常会被当成普通字符写进提交消息，导致说明格式错乱、审查可读性下降。

### 反例 9：收到“提交git”后只提交一部分就结束

先提交了改动 A，但仍有改动 B 未提交，并直接回复“已提交完成”。

❌ 问题：完成标准不达标。对于“提交git”请求，必须继续提交剩余改动（可拆分多次）直到 `git status --short` 为空。

### 反例 12：代码实现和审查 / 验收文档混在同一次提交

当前 staged 文件同时包含：

```text
internal/service/fee_rule.go
doc/6-审查/2026-07-08_101500_稳定币费率修复_当前改动总审查.md
doc/7-验收/2026-07-08_101900_稳定币费率修复_最终验收.md
```

仍然尝试一次提交。

❌ 问题：代码实现必须单独提交，审查和验收文档也各自属于独立提交域；混提会让人工检查时同时穿梭代码与归档文档，边界不清楚。

### 反例 13：代码实现和测试文件混在同一次提交

当前 staged 文件同时包含：

```text
src/modules/order/create-order.ts
src/modules/order/create-order.spec.ts
doc/5-tests/2026-07-08_102300_订单创建回归/测试记录.md
```

仍然尝试一次提交。

❌ 问题：`*.spec.*`、`*.test.*`、`*_test.*` 和 `doc/5-tests/` 都应归入测试提交；代码实现应单独提交，不能和测试文件混提。
