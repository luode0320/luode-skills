# 分支与提交

## 铁律：拆分业务修改，多次提交

**提交代码时必须分析未提交的代码，如果可以分成不同的业务修改，就用多次不同的 git 提交。**

- 不要一次提交所有导致业务没有区分清楚。
- 按业务功能、业务模块、业务目标来拆分提交。
- 每个提交只承载一个清晰的业务目标。

## 提交前必须补 README 改动日志

在正式执行 git 提交之前，必须先更新**项目根目录** `README.md`，追加一条本次改动日志。

日志格式固定为：

```text
yyyy-MM-DD HH:mm:ss 提交标题
```

要求如下：

- 时间使用本次准备提交时的实际时间。
- 标题直接使用本次 git 提交标题，保持一致，不要另外改写一套说法。
- 一次 git 提交对应一条 README 改动日志。
- 如果拆成多次 git 提交，就要按提交次数追加多条日志。
- README 改动日志必须按时间正序排列，即时间早的在前，时间晚的在后。
- 如果本次是补记历史提交，且时间早于当前最后一条日志，必须插入到正确位置，不能机械追加到文件最后。

示例：

```text
2026-03-29 15:30:00 fix: 修复订单创建时金额计算错误
2026-03-29 15:45:00 feat: 新增用户登录验证码校验
```

正确示例：

```text
2026-03-30 10:05:00 feat: 增加平台级内网来源IP白名单中间件
2026-03-30 10:12:27 refactor: 拆分租户会话与订单路由注册函数
```

错误示例：

```text
2026-03-30 10:12:27 refactor: 拆分租户会话与订单路由注册函数
2026-03-30 10:05:00 feat: 增加平台级内网来源IP白名单中间件
```

上面的错误示例属于时间倒挂，后续按 README 回看提交历史时会误导阅读顺序，不允许这样记录。

不要写成模糊描述，例如：

```text
今天修了一个 bug
更新了一些内容
```

## 提交消息格式

提交消息必须遵循以下格式：

### 格式一：feat（新增功能）

```
feat: 新增标题

说明：详细说明本次新增的功能内容、影响范围等。
```

示例：

```
feat: 新增用户登录功能

说明：
- 支持手机号验证码登录
- 支持密码登录
- 登录成功后返回用户基本信息和 token
```

### 格式二：fix（修复 bug）

```
fix: 修复xxbug

说明：详细说明修复的问题、原因、影响范围等。
```

示例：

```
fix: 修复订单创建时金额计算错误

说明：
- 问题：订单创建时优惠券折扣计算错误
- 原因：折扣计算逻辑中缺少对满减条件的判断
- 影响：已创建的订单金额可能不正确
```

### 格式三：其他类型

根据需要也可以使用以下类型：

- `refactor:` - 重构代码
- `style:` - 代码格式调整
- `docs:` - 文档更新
- `test:` - 测试相关
- `chore:` - 构建/工具相关

## Windows/PowerShell 提交说明格式（重点：换行）

Windows/PowerShell 场景下，提交消息格式必须满足：

```text
<type>: <标题>

说明：
- 第一条
- 第二条
```

重点要求：

- 标题行和说明正文之间必须有且仅有一空行。
- 说明正文必须是“真实换行”，不允许出现字面量 `\n`。
- 正文建议使用短句或列表，避免一整段大长句。

推荐写法一（`-m` + here-string，适合常规提交）：

```powershell
git commit -m "fix: 修复订单创建时金额计算错误" -m @"
说明：
- 问题：订单创建时优惠券折扣计算错误
- 原因：折扣计算逻辑中缺少对满减条件的判断
- 影响：已创建的订单金额可能不正确
"@
```

推荐写法二（`-F -`，适合更长说明）：

```powershell
@"
fix: 修复订单创建时金额计算错误

说明：
- 问题：订单创建时优惠券折扣计算错误
- 原因：折扣计算逻辑中缺少对满减条件的判断
- 影响：已创建的订单金额可能不正确
"@ | git commit -F -
```

错误写法（PowerShell 下会把 `\n` 当普通字符）：

```powershell
git commit -m "fix: 修复订单创建时金额计算错误\n\n说明：\n- 问题：订单创建时优惠券折扣计算错误"
```

## 分支原则

- 一次提交最好只承载一个清晰目标。
- 提交说明应能看出做了什么、为什么做。
- 分支要尽量围绕单任务或单主题，避免长期混杂演进。
- 无关格式化、试验代码和调试残留不要混进正式提交。
- 对这个 skill 来说，默认以 git 提交本身和项目根目录 `README.md` 改动日志作为持久化记录，不再额外复制到 `delivery/` 之类目录。
- 项目根目录 `README.md` 改动日志不仅要补充完整，还要维持时间正序，不能出现较晚时间排在较早时间前面的情况。

## Go 提交前禁放扫描

Go 项目在执行 `git commit` 前，必须执行：

```powershell
rg --files -g "*_test.go" | rg -v "^test/"
```

判定规则：

- 命中为空：通过，可继续提交。
- 命中非空：阻断提交，先改 seam 并迁移到 `test/<时间戳>/` 后再提交。

## Go 服务层提交前扫描

Go 项目在执行 `git commit` 前，还必须检查 staged 文件中是否出现服务层根目录直落实现：

```powershell
git diff --cached --name-only | rg "^internal/service/[^/]+\.go$"
```

判定规则：

- 命中为空：通过，可继续提交。
- 命中非空：阻断提交，先迁移到 `internal/service/<domain>/` 后再提交。
