# 分支与提交

## 铁律：拆分业务修改，多次提交

**提交代码时必须分析未提交的代码，如果可以分成不同的业务修改，就用多次不同的 git 提交。**

- 不要一次提交所有导致业务没有区分清楚。
- 按业务功能、业务模块、业务目标来拆分提交。
- 每个提交只承载一个清晰的业务目标。

## 铁律：`提交git` 请求必须全量收口

当用户明确请求“提交git”时，完成标准不是“提交一部分”，而是：

- 允许按业务拆分成多次提交。
- 但必须持续提交直到当前未提交改动清空。
- 最终必须满足 `git status --short` 为空。
- 除阻断级规则（如 Go 禁放扫描失败）外，不允许以“无关改动”名义提前结束。

同义短指令也按相同规则处理，包括但不限于：

- `提交 git`
- `git提交`
- `commit一下`
- `帮我提交代码`

以上短指令都视为“执行提交动作”，不要求用户先补充“测试已完成”或“准备交付”的阶段描述。

## 提交前必须补 README 改动日志

在正式执行 git 提交之前，必须先更新**项目根目录** `README.md`，在“改动日志”区域最后一条提交日志后追加一行本次改动日志。

日志内容固定为：

```text
yyyy-MM-dd HH:mm:ss 提交标题
```

要求如下：

- 标题直接使用本次 git 提交标题，保持一致，不要另外改写一套说法。
- 时间戳使用当前北京时间，格式为 `yyyy-MM-dd HH:mm:ss`。
- 一次 git 提交对应一条 README 改动日志。
- 如果拆成多次 git 提交，就要按提交次数追加多条日志。
- 必须始终追加到当前“改动日志”最后一条提交日志后面，不插到更前面，也不追加到更靠前的旧日志上方。
- 默认不处理“补记历史提交并按时间回插”的场景；本 skill 只要求本次提交日志始终落在改动日志末尾。

示例：

```text
2026-07-01 14:25:30 fix: [订单创建接口] 修复订单金额计算错误
2026-07-01 14:30:00 feat: [用户登录] 新增验证码校验能力
```

正确示例：

```text
2026-06-28 01:12:00 feat: [平台安全] 增加内网来源 IP 白名单中间件
2026-06-28 04:17:15 refactor: [租户会话与订单路由] 拆分注册函数
```

错误示例：

```text
2026-06-28 04:17:15 refactor: [租户会话与订单路由] 拆分注册函数
2026-06-28 01:12:00 feat: [平台安全] 增加内网来源 IP 白名单中间件
```

上面的错误示例里，较新的提交如果被插到已有末尾日志前面，就会破坏“始终从末尾继续追加”的规则，不允许这样记录。

不要写成模糊描述，例如：

```text
今天修了一个 bug
更新了一些内容
```

## 提交消息格式

提交消息必须遵循以下格式：

### 标题模板（必选）

```text
<type>: [中文简要说明] 标题说明
```

标题规则：

- `<type>` 使用 `feat`、`fix`、`refactor`、`style`、`docs`、`test`、`chore` 等规范类型。
- `中文简要说明` 直接描述本次改动对象，例如“用户登录”“订单结算”“订单创建接口”。
- 禁止使用 `type(scope): 标题` 的括号 scope 写法（例如 `feat(reqsend): 转账详细统计支持多Sheet导出错误明细`）。
- 遇到 scope 场景必须改写为 `type: [scope] 标题`（例如 `feat: [reqsend] 转账详细统计支持多Sheet导出错误明细`）。
- `中文简要说明` 不允许加“功能/”“业务/”“接口/”等分类前缀。
- `中文简要说明` 不允许直接写 API URL（如 `POST /api/order/create`）。
- `标题说明` 用动宾短句说明动作，例如“新增短信验证码登录能力”“修复优惠券重复抵扣”。

正误对照（必须遵循）：

```text
错误：feat(reqsend): 转账详细统计支持多Sheet导出错误明细
正确：feat: [reqsend] 转账详细统计支持多Sheet导出错误明细
```

### 格式一：feat（新增功能）

```
feat: [用户登录] 新增短信验证码登录能力

说明：详细说明本次新增的功能内容、影响范围等。
```

示例：

```
feat: [用户登录] 新增密码与验证码双登录能力

说明：
- 支持手机号验证码登录
- 支持密码登录
- 登录成功后返回用户基本信息和 token
```

### 格式二：fix（修复 bug）

```
fix: [订单创建接口] 修复优惠券重复抵扣问题

说明：详细说明修复的问题、原因、影响范围等。
```

示例：

```
fix: [订单结算] 修复订单创建时金额计算错误

说明：
- 问题：订单创建时优惠券折扣计算错误
- 原因：折扣计算逻辑中缺少对满减条件的判断
- 影响：已创建的订单金额可能不正确
```

### 格式三：其他类型

根据需要也可以使用以下类型：

- `refactor:` - 重构代码
- `style:` - 代码格式调整
- `docs:` - 文档更新
- `test:` - 测试相关
- `chore:` - 构建/工具相关

## 多行说明写法（跨 shell 统一）

推荐优先使用以下两种方式，避免把 `\n` 写进提交消息正文：

### 写法一：多 `-m`（跨 shell 稳定）

```bash
git commit \
  -m "fix: [财务配置] 适配 financeConfig 新字段与动态开关规则" \
  -m "1. 新增 financeConfig 的 version/limit/enable 字段映射。
2. 根据 gVersion 与 ipCountry 动态计算 enable。
3. 补充关键结构与函数中文注释。"
```

### 写法二：`-F -`（长正文推荐）

```bash
cat <<'MSG' | git commit -F -
fix: [财务配置] 适配 financeConfig 新字段与动态开关规则

1. 新增 financeConfig 的 version/limit/enable 字段映射。
2. 根据 gVersion 与 ipCountry 动态计算 enable。
3. 补充关键结构与函数中文注释。
MSG
```

## 提交后强制校验（新增硬规则）

提交完成后必须执行一次：

```bash
git log -1 --pretty=%B | rg '\\n'
```

判定规则：

- 命中为空：通过。
- 命中非空：说明提交正文含字面量 `\n`，必须立即 `git commit --amend` 修正为真实换行。

## Windows/PowerShell 提交说明格式（重点：换行）

Windows/PowerShell 场景下，提交消息格式必须满足：

```text
<type>: [中文简要说明] 标题说明

说明：
- 第一条
- 第二条
```

重点要求：

- 标题行和说明正文之间必须有且仅有一空行。
- 说明正文必须是“真实换行”，不允许出现字面量 `\n`。
- 正文建议使用短句或列表，避免一整段大长句。
- 提交后仍需执行 `git log -1 --pretty=%B | rg '\\n'` 校验，确保没有字面量 `\n` 遗留。

推荐写法一（`-m` + here-string，适合常规提交）：

```powershell
git commit -m "fix: [订单结算] 修复订单创建时金额计算错误" -m @"
说明：
- 问题：订单创建时优惠券折扣计算错误
- 原因：折扣计算逻辑中缺少对满减条件的判断
- 影响：已创建的订单金额可能不正确
"@
```

推荐写法二（`-F -`，适合更长说明）：

```powershell
@"
fix: [订单结算] 修复订单创建时金额计算错误

说明：
- 问题：订单创建时优惠券折扣计算错误
- 原因：折扣计算逻辑中缺少对满减条件的判断
- 影响：已创建的订单金额可能不正确
"@ | git commit -F -
```

错误写法（PowerShell 下会把 `\n` 当普通字符）：

```powershell
git commit -m "fix: [订单结算] 修复订单创建时金额计算错误\n\n说明：\n- 问题：订单创建时优惠券折扣计算错误"
```

## 分支原则

- 一次提交最好只承载一个清晰目标。
- 提交说明应能看出做了什么、为什么做。
- 分支要尽量围绕单任务或单主题，避免长期混杂演进。
- 无关格式化、试验代码和调试残留不要混进正式提交。
- 对这个 skill 来说，默认以 git 提交本身和项目根目录 `README.md` 改动日志作为持久化记录，不再额外复制到 `delivery/` 之类目录。
- 项目根目录 `README.md` 改动日志默认只要求“每次从末尾继续追加”，不再要求为了时间排序去回插旧位置。

## Go 提交前禁放扫描

Go 项目在执行 `git commit` 前，必须执行：

```powershell
rg --files -g "*_test.go" | rg -v "^doc/5-tests/"
```

判定规则：

- 命中为空：通过，可继续提交。
- 命中非空：阻断提交，先改 seam 并迁移到 `doc/5-tests/<时间戳>/` 后再提交。


## 提交前必须有通过的审查文档

在执行 `git commit` 之前，必须确认 `doc/6-审查/` 下存在最近一份覆盖当前待提交改动范围的审查文档，且文档中包含以下统一判定字段：

- `审查结论: 通过`
- `审查范围: <文件列表或等价范围说明>`
- `是否允许提交: 是`
- `阻断问题: 无` 或不含未解决 P0/P1

如果审查文档不存在、审查未通过或审查范围不覆盖当前改动，必须先补齐审查文档再提交。

此闸门由 `pre_commit_gate.sh` 第 5 步自动校验，回退检查时同样适用。
## Go 服务层提交前扫描

Go 项目在执行 `git commit` 前，还必须检查 staged 文件中是否出现服务层根目录直落实现：

```powershell
git diff --cached --name-only | rg "^internal/service/[^/]+\.go$"
```

判定规则：

- 命中为空：通过，可继续提交。
- 命中非空：阻断提交，先迁移到 `internal/service/<domain>/` 后再提交。
