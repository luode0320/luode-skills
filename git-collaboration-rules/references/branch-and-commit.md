# 分支与提交

## 铁律：拆分业务修改，多次提交

**提交代码时必须分析未提交的代码，如果可以分成不同的业务修改，就用多次不同的 git 提交。**

- 不要一次提交所有导致业务没有区分清楚。
- 按业务功能、业务模块、业务目标来拆分提交。
- 每个提交只承载一个清晰的业务目标。

## 提交消息格式

提交消息必须遵循以下格式：

### 格式一：feat（新增功能）

```
feat: 新增标题

说明：详细说明本次新增的功能内容、影响范围等。
```

示例：
```
feat: 新增用户登录功能

说明：
- 支持手机号验证码登录
- 支持密码登录
- 登录成功后返回用户基本信息和 token
```

### 格式二：fix（修复 bug）

```
fix: 修复xxbug

说明：详细说明修复的问题、原因、影响范围等。
```

示例：
```
fix: 修复订单创建时金额计算错误

说明：
- 问题：订单创建时优惠券折扣计算错误
- 原因：折扣计算逻辑中缺少对满减条件的判断
- 影响：已创建的订单金额可能不正确
```

### 格式三：其他类型

根据需要也可以使用以下类型：

- `refactor:` - 重构代码
- `style:` - 代码格式调整
- `docs:` - 文档更新
- `test:` - 测试相关
- `chore:` - 构建/工具相关

## 分支原则

- 一次提交最好只承载一个清晰目标。
- 提交说明应能看出做了什么、为什么做。
- 分支要尽量围绕单任务或单主题，避免长期混杂演进。
- 无关格式化、试验代码和调试残留不要混进正式提交。
