# 项目设计文档规则

只在“需要读取、判断、同步或补建项目级设计 / 架构 / 大纲文档”这个问题上使用这个 skill。
如果当前只是决定文档放在哪里、叫什么、是否复用入口文件，请先转交 `artifact-storage-rules`；如果只是新会话快速预热，请转交 `recent-context-bootstrap-rules`。

## Skill 作用与适用场景

- 为整项目分析、架构梳理和模块总览提供统一的项目设计文档处理规则。
- 把根目录 `项目设计.md` 及其历史同类文档视为弱参考源，而不是绝对真相。
- 在项目架构、目录、大纲和核心链路发生变化后，推动同步更新项目设计文档。
- 当项目缺少根目录 `项目设计.md`，且已经完成一次全项目分析时，补建统一主入口。
- 减少“代码已经演进，设计文档长期失真”或“同时维护多份平行设计总览”的问题。

## 自动触发信号

- 用户要求“分析整个项目”“先梳理项目架构”“看一下模块设计再继续”。
- 用户要求生成、更新、同步、补全根目录 `项目设计.md`。
- 根目录已经存在 `项目设计.md`、`架构设计.md`、`项目大纲.md`、`系统设计.md`、`项目说明.md` 等同类文档，需要判断是否继续可信。
- 项目目录结构、核心模块边界、主链路、关键依赖或总体设计刚发生明显变化。
- 需要在完成全项目分析后，为没有设计总览文档的项目补建根目录 `项目设计.md`。

## 进入后先做什么

1. 先确认当前任务真的是“项目级设计文档处理”，而不是单一需求、单一 Bug 或单一代码位点问题。
2. 默认先读 `../artifact-storage-rules/references/path-map.yaml`，确认根目录主入口 `项目设计.md` 与历史别名规则。
3. 查找根目录是否存在 `项目设计.md` 或历史同类文档。
4. 如果存在，先按弱参考方式阅读，不默认相信其内容已经最新。
5. 再对照真实代码、当前有效需求 / 实施 / Bug / 测试文档和最近 Git 变更判断是否偏移。
6. 如果没有根目录 `项目设计.md`，先确认当前是否已经具备“完整项目分析”前提，再决定是否补建。

## 默认执行流程

1. 默认先读 `references/discovery-and-priority.md`，确认根目录候选文档与证据优先级。
2. 如果当前重点是“什么时候该更新、什么时候该补建”，再读 `references/sync-policy.md`。
3. 如果需要输出或补建 `项目设计.md`，再读 `references/design-template.md`。
4. 输出当前结论：已有设计文档是否存在、可信度如何、与当前项目是否偏移、应更新还是补建。
5. 如果需要同步，优先把结果收口到根目录 `项目设计.md`，避免继续维护多个平行主入口。

## 权责边界与不负责事项

- 只负责项目设计文档的读取、偏移判断、同步和补建，不代替 `artifact-storage-rules` 做路径命名总规则裁定。
- 不代替 `recent-context-bootstrap-rules` 做新会话最近 3 天轻量预热。
- 不代替 `project-timeline-rules` 输出长期项目演进历史。
- 不代替 `package-structure-rules`、`code-placement-review-rules`、`implementation-plan-rules` 去定义代码分层、文件落位或当前实施方案细节。
- 不允许用过期的项目设计文档反压真实代码、当前有效文档和最近 Git 变更。

## 需要暂停并确认的条件

- 根目录同时存在多份历史设计总览，且内容相互冲突，无法直接判断主入口。
- 当前项目还没完成有效的整项目分析，就要求直接生成完整 `项目设计.md`。
- 设计文档明显涉及大范围迁移、重命名或历史文档合并，超出当前最小改动范围。
- 代码现状、当前有效文档和设计文档冲突严重，需要先确认到底以哪个真实状态为准。

## 执行通过 / 驳回标准

- 通过：能够明确说明根目录项目设计类文档是否存在、当前是否可信、与真实项目是否偏移，以及应继续更新还是补建根目录 `项目设计.md`。
- 驳回：直接把历史设计文档当成当前事实，或在没有足够项目分析依据时机械生成一份空泛的 `项目设计.md`。

## 执行结果归档要求

- 项目级设计总览统一归档到仓库根目录 `项目设计.md`。
- 如果存在 `架构设计.md`、`项目大纲.md`、`系统设计.md`、`项目说明.md` 等历史同类文档，可在当前轮先读取，但后续应尽量收口到 `项目设计.md`。
- `项目设计.md` 至少应包含：文档说明、最近同步时间、当前可信度说明、技术栈、目录与模块总览、核心链路、关键依赖、当前约束和待同步项。
- 如果本轮只是判断偏移而未实际更新，也应明确说明“不更新的原因”和“后续补同步条件”。

## references 读取规则

- 默认先读 `references/discovery-and-priority.md`。
- 只有在判断同步时机和补建条件时，再读 `references/sync-policy.md`。
- 只有在需要生成或重写 `项目设计.md` 时，再读 `references/design-template.md`。
