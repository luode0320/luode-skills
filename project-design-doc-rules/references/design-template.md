# 项目设计文档模板

结论：{用一句普通中文说明项目当前的设计结论}；影响：{说明受影响的用户、角色或业务环节}；范围：{说明本文档覆盖的项目内容}；非范围：{说明本文档明确不覆盖的内容}；变化：{说明设计更新带来的业务或协作变化}；完成标准：{说明项目设计说明达到可用的条件}；术语说明：无技术术语需要解释，或 {术语：它在本次项目设计中的作用}；验证状态：{说明事实核对、链接或图示验证进展和依据}。

## 白话正文与附录

正文先讲项目目标、服务对象、范围、非范围、关键变化、完成标准、术语含义和验证进展；H1 后单段必须完整包含八个固定字段。技术栈版本、目录树、依赖明细和来源证据放入文末连续附录。

根目录 `项目设计.md` 建议至少包含以下章节：

## 1. 文档说明

- 文档用途
- 当前可信度说明
- 最近同步时间：`yyyy-MM-DD HH:mm:ss`
- 主要依据：代码 / 当前文档 / 最近 Git 变更 / 历史设计文档

## 2. 项目目标与范围

- 项目主要用途
- 当前主要服务对象或业务目标
- 当前不覆盖的范围

## 3. 技术栈与运行形态

- 后端 / 前端 / 数据库 / 中间件 / 外部依赖
- 关键框架与语言
- 部署或运行形态的概要说明

## 4. 目录与模块总览

- 根目录结构概要
- 核心模块及职责
- 关键公共层与共享能力

## 5. 核心链路

- 主要业务主流程
- 关键上下游交互
- 关键数据流或调用链

## 6. 当前约束与风险

- 设计上的当前限制
- 已知偏移点
- 待同步或待补齐项

## 写作要求

- 文档默认使用 UTF-8。
- 说明文字默认使用中文。
- 不追求面面俱到，优先写“当前仍然成立的总览事实”。
- 如果某部分仍不确定，应明确写“待确认”，不要伪造确定性。

## 执行附录

- 技术栈版本、目录树、依赖明细、生成或核对步骤和当前运行形态的技术细节。

## 追踪附录

- 当前代码、文档和历史设计来源的证据定位；稳定标识、链接、图示和同步记录。
