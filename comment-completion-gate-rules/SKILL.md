# 注释补齐闸门规则

只在需要判断“本轮改动位点注释是否补齐、能不能通过收口”时使用本 skill。
如果当前争议是注释该不该写、写在哪、写多细，请转交 `comment-placement-granularity-rules`；如果主要争议是中文表达，请转交 `chinese-comment-rules`；如果是 Swagger/OpenAPI 接口文档注解，请转交 `api-swagger-rules`。

## Skill 作用与适用场景

- 当用户请求“补充注释 / 只补注释 / 注释完善”时，优先对 `git` 已改动且未提交的代码位点补注释。
- 强制本轮新增或修改的函数、方法补齐函数头 `[参数]` / `[返回]` / `最近修改时间`。
- 强制本轮新增或修改的函数、方法补齐方法体步骤或意图注释。
- 当方法体存在 1 个及以上步骤时，强制至少使用顶层 `1.` 步骤编号；若存在多个连续步骤，继续使用 `1.` `2.` `3.`。
- 任一关键注释缺失，都不得给出“通过”或“可提交”结论。
- 强制在本轮首次代码改动后即触发一次“中间阶段注释预检”，不得把注释检查全部延后到最终收口。

## 强制规则：补注释优先范围

- 当用户请求补注释时，优先范围固定为：`git` 已改动且未提交的代码位点（包含 staged 与 unstaged）。
- 在未提交改动位点未补齐前，不得先扩散到未改动历史代码。
- 若未提交改动中存在函数、方法改动，必须优先完成这些函数、方法的函数头注释与方法体注释。

## 强制门禁：改动位点注释补齐

- 本轮新增或修改的函数、方法、关键步骤入口和字段处理位点，必须逐一检查是否需要补齐或更新注释。
- 未改动的周边历史代码可以不补注释，但不能以此为由跳过本轮改动位点。
- 如果未完成改动位点注释补齐，本 skill 判定为不通过，不得进入“实现自审通过”或“可提交”结论。
- 即使用户只要求“补充注释”，也不能只补函数头，仍需检查方法体步骤注释是否完整。

## 强制门禁：前端必注释位点

- 当本轮改动涉及 `.vue`、`.tsx`、`.jsx`、`.css`、`.scss` 文件时，以下位点必须补中文注释（就近落位）：
  - 复杂样式块（如 `overflow`、`position`、动画、媒体查询、主题 token）
  - 条件渲染逻辑
  - 事件处理函数
  - 异步数据加载与分页逻辑
- 前端位点注释必须说明“意图/原因”，不能只重复代码表面含义。

## 强制规则：步骤编号

- 当函数、方法存在 1 个及以上步骤时，必须至少使用顶层 `1.` 编号步骤注释；若存在多个连续步骤，继续使用 `1.` `2.` `3.`。
- 当顶层步骤内部仍有 2 个及以上子步骤时，允许并推荐继续使用 `1.1`、`1.2`、`2.1` 这类分层编号。
- 每个编号注释必须紧贴其对应步骤的第一行代码上方。
- 步骤代码若是 `if/for/switch/select`，编号注释放在该结构入口上方。
- 禁止把多个步骤编号集中写在函数开头作为总览后不就近落位。
- 禁止只出现子步骤编号而缺失对应顶层编号。
- 当方法体已经出现流程型普通注释（例如“查询订单”“分批处理”“更新状态”）时，必须改写为编号步骤注释；即使只有一个步骤也至少补 `1.`。
- 函数头注释完整，也不能抵消方法体步骤编号缺失。

## 强制规则：函数头元信息

- 每个新增或修改的函数、方法都必须补齐函数头注释，不允许仅补方法体内注释。
- 函数、方法注释中的元信息顺序必须固定为：先 `[参数]`，再 `[返回]`，最后 `最近修改时间`。
- 当无参数或无返回时，仍按固定顺序保留对应行并写 `无`。
- `最近修改时间` 必须作为函数、方法注释块的最后一行，格式为北京时间：`yyyy-MM-DD HH:mm:ss`。
- `最近修改时间` 行必须包含本次改动原因简述，要求简短、具体、可定位。

## 自动触发信号

- 任何新增或修改代码都必须触发一次“改动位点注释补齐检查”。
- 本轮首次代码写入完成后，必须立即触发一次中间阶段注释预检。
- 用户要求“补充注释”“只补注释”“注释完善”。
- 新增或修改函数、方法，需要补齐或更新函数头元信息。
- 方法体中已经出现步骤流程（含单步骤），或已经写出多个流程型普通注释。
- 需要判断流程型普通注释是否必须改写为编号步骤注释（至少 `1.`，多步骤补齐 `1.` `2.` `3.`）。
- 准备进入实现自审、测试前验证或可提交收口，需要确认注释是否达到闸门标准。
- 本轮执行链路较长并需要中间阶段总结时，必须再次触发注释补齐复检。

## 进入后先做什么

1. 先定位本轮新增或修改的代码位点。
2. 本轮首次代码改动后，先执行一次中间阶段注释预检并输出缺口清单。
3. 若本轮是补注释请求，先定位 `git` 未提交改动（staged + unstaged）并锁定优先处理清单。
4. 对每个新增或修改的函数、方法，先补齐函数头 `[参数]` / `[返回]` / `最近修改时间`。
5. 对每个新增或修改的函数、方法，检查方法体是否已有就近步骤或意图注释。
6. 只要方法体存在步骤流程，就先补齐顶层 `1.`；若存在多个连续步骤，再补齐 `2.`、`3.`，必要时补 `1.1`、`1.2` 分层编号。
7. 对照本轮改动位点做最终闸门复核，任一关键项缺失都判定未完成。

## 默认执行流程

1. 默认先读 `references/comment-completion-priority.md`，确认补注释优先范围和改动位点闸门。
2. 需要核对函数头元信息时，再读 `references/comment-function-metadata.md`。
3. 需要核对方法体步骤编号时，再读 `references/comment-step-numbering-gate.md`。
4. 需要对照正反例时，再读 `references/comment-examples.md`。
5. 首次代码改动后先输出一次中间阶段注释预检结果（命中状态 + 缺失项）。
6. 最终收口前输出本轮改动位点注释补齐结论、缺失项清单和阻断原因。
7. 若当前争议其实是注释位置或颗粒度不清，应转交 `comment-placement-granularity-rules`；若争议是中文表达，再转交 `chinese-comment-rules`。

## 权责边界与不负责事项

- 只负责改动位点补齐闸门，不代替 `comment-placement-granularity-rules` 判断一般性的注释必要性和放置位置。
- 不代替 `chinese-comment-rules` 处理中文表达。
- 不代替 `api-swagger-rules` 处理 Swagger/OpenAPI 接口文档注解。
- 不允许用“函数头已经补了”来掩盖方法体步骤编号缺失。
- 不要求为了过闸门而给未改动历史代码无差别补注释。

## 需要暂停并确认的条件

- 代码本身可以通过重命名或小重构解决理解问题。
- 现有注释和实现冲突，但代码行为还未确认。
- 为了通过评审而临时补大量无效注释。
- 方法体流程混乱到需要先做结构调整，再决定如何编号。

## 执行通过 / 驳回标准

- 通过：本轮改动位点都完成注释补齐；新增或修改的函数、方法均有 `[参数]` / `[返回]` / `最近修改时间` 且顺序正确；方法体存在步骤流程时已至少使用 `1.` 并就近落位；若存在多个连续步骤，已补齐 `1.` `2.` `3.`；若顶层步骤存在子步骤，已使用 `1.1`、`1.2` 等分层编号；若是补注释请求，未提交改动代码优先补齐已完成；每个本轮改动的前端文件（`.vue`/`.tsx`/`.jsx`/`.css`/`.scss`）至少包含 1 条与改动直接相关的中文意图注释；首次代码改动后的中间阶段预检已执行。
- 驳回：本轮任一改动位点未补齐注释，或存在步骤流程却没有编号步骤注释，或只写子编号不写顶层编号，或把多个步骤编号集中写在函数开头，或多个流程型普通注释未升级为编号步骤注释，或新增 / 修改的函数、方法缺少 `[参数]` / `[返回]` / `最近修改时间`，或元信息顺序不符合“`[参数]`、`[返回]`、`最近修改时间`（最后一行）”，或任一本轮改动前端文件缺少与改动直接相关的中文意图注释，或首次代码改动后未进行中间阶段注释预检。
