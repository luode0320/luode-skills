# 注释补齐闸门规则

只在需要判断“本轮改动位点注释是否补齐、能不能通过收口”时使用本 skill。
如果当前争议是注释该不该写、写在哪、写多细，请转交 `comment-placement-granularity-rules`；如果主要争议是中文表达，请转交 `chinese-comment-rules`；如果是 Swagger/OpenAPI 接口文档注解，请转交 `api-swagger-rules`。

## 注释模式（新增）

- `default`（默认最小注释模式）：本轮代码改动默认启用。只强制关键复杂位点注释，不要求所有改动函数都补函数头元信息。
- `strict`（严格注释模式）：仅在用户明确提出“补充注释/只补注释/注释完善/严格注释”时启用。此模式下启用函数头元信息 + 步骤编号的完整门禁。
- 未明确要求 strict 时，一律按 default 判定，不得误判为注释缺失阻断。

## Skill 作用与适用场景

- 当用户请求“补充注释 / 只补注释 / 注释完善”时，优先对 `git` 已改动且未提交的代码位点补注释。
- 统一本轮改动位点的注释补齐标准，避免出现“命中了注释 skill 但没有执行”。
- 在 strict 模式下，强制函数头元信息与方法体步骤编号。
- 任一关键注释缺失，都不得给出“通过”或“可提交”结论。
- 强制在本轮首次代码改动后即触发一次“中间阶段注释预检”，不得把注释检查全部延后到最终收口。

## 强制规则：补注释优先范围

- 当用户请求补注释时，优先范围固定为：`git` 已改动且未提交的代码位点（包含 staged 与 unstaged）。
- 在未提交改动位点未补齐前，不得先扩散到未改动历史代码。

## 强制门禁：改动位点注释补齐

- 本轮新增或修改的函数、方法、关键步骤入口和字段处理位点，必须逐一检查是否需要补齐或更新注释。
- 未改动的周边历史代码可以不补注释，但不能以此为由跳过本轮改动位点。
- 如果未完成改动位点注释补齐，本 skill 判定为不通过，不得进入“实现自审通过”或“可提交”结论。

## default 模式门禁（新增）

- 仅以下位点缺注释时判定不通过：
  - 并发控制（锁、协程并发聚合、重试状态机）
  - 风控/黑白名单/安全校验分支
  - 外部渠道兼容分支或异常降级分支
  - 金额/精度/汇率边界修正
- 简单流程、直观代码不因缺“函数头模板注释”而阻断。

## strict 模式门禁（新增）

- 每个新增或修改的函数、方法都必须补齐函数头注释，不允许仅补方法体内注释。
- 函数、方法注释中的元信息顺序必须固定为：先 `[参数]`，再 `[返回]`，最后 `最近修改时间`。
- 当无参数或无返回时，仍按固定顺序保留对应行并写 `无`。
- `最近修改时间` 必须作为函数、方法注释块的最后一行，格式为北京时间：`yyyy-MM-DD HH:mm:ss`。
- `最近修改时间` 行必须包含本次改动原因简述，要求简短、具体、可定位。
- 当函数、方法存在 1 个及以上步骤时，必须至少使用顶层 `1.` 编号步骤注释；若存在多个连续步骤，继续使用 `1.` `2.` `3.`。
- 每个编号注释必须紧贴其对应步骤的第一行代码上方。
- 函数头注释完整，也不能抵消方法体步骤编号缺失。

## 自动触发信号

- 任何新增或修改代码都必须触发一次“改动位点注释补齐检查”。
- 本轮首次代码写入完成后，必须立即触发一次中间阶段注释预检。
- 用户要求“补充注释”“只补注释”“注释完善”“严格注释”。
- 准备进入实现自审、测试前验证或可提交收口，需要确认注释是否达到闸门标准。

## 进入后先做什么

1. 先定位本轮新增或修改的代码位点。
2. 先判定当前模式：`default` 或 `strict`。
3. 本轮首次代码改动后，先执行一次中间阶段注释预检并输出缺口清单。
4. 若本轮是补注释请求，先定位 `git` 未提交改动（staged + unstaged）并锁定优先处理清单。
5. 按当前模式执行对应门禁，并输出通过/阻断结论。

## 默认执行流程

1. 默认先读 `references/comment-completion-priority.md`，确认补注释优先范围和改动位点闸门。
2. strict 模式需要核对函数头元信息时，再读 `references/comment-function-metadata.md`。
3. strict 模式需要核对方法体步骤编号时，再读 `references/comment-step-numbering-gate.md`。
4. 需要对照正反例时，再读 `references/comment-examples.md`。
5. 首次代码改动后先输出一次中间阶段注释预检结果（命中状态 + 缺失项）。
6. 最终收口前输出本轮改动位点注释补齐结论、缺失项清单和阻断原因。
7. 若当前争议其实是注释位置或颗粒度不清，应转交 `comment-placement-granularity-rules`；若争议是中文表达，再转交 `chinese-comment-rules`。

## 权责边界与不负责事项

- 只负责改动位点补齐闸门，不代替 `comment-placement-granularity-rules` 判断一般性的注释必要性和放置位置。
- 不代替 `chinese-comment-rules` 处理中文表达。
- 不代替 `api-swagger-rules` 处理 Swagger/OpenAPI 接口文档注解。
- 不要求为了过闸门而给未改动历史代码无差别补注释。

## 需要暂停并确认的条件

- 代码本身可以通过重命名或小重构解决理解问题。
- 现有注释和实现冲突，但代码行为还未确认。
- 为了通过评审而临时补大量无效注释。
- 方法体流程混乱到需要先做结构调整，再决定如何编号。

## 执行通过 / 驳回标准

- 通过：本轮改动位点完成了当前模式要求的注释补齐；若为补注释请求，未提交改动代码优先补齐已完成；首次代码改动后的中间阶段预检已执行。
- 驳回：本轮任一改动位点未满足当前模式门禁，或首次代码改动后未进行中间阶段注释预检。

## references 读取规则

- 默认先读 `references/comment-completion-priority.md`。
- 只有在 strict 模式核对函数头元信息时，再读 `references/comment-function-metadata.md`。
- 只有在 strict 模式核对方法体步骤编号时，再读 `references/comment-step-numbering-gate.md`。
- 只有在对照正反例或补齐缺失项时，再读 `references/comment-examples.md`。
