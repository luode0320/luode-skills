# 补注释优先范围

- 当用户请求补注释时，优先范围固定为 `git` 已改动且未提交的代码位点，包含 staged 与 unstaged。
- 在未提交改动位点未补齐前，不得先扩散到未改动历史代码。
- 若未提交改动中存在函数、方法改动，必须优先完成这些函数、方法的函数头注释与方法体注释。
- 本轮新增或修改的关键步骤入口、字段处理位点和风险分支，也必须纳入改动位点检查清单。
- 前端代码与后端代码一样进入注释闸门；`.vue`、`.tsx`、`.jsx`、`.ts`、`.js`、`.html`、`.css`、`.scss`、`.less` 中新增或修改的函数、事件处理、异步加载、条件渲染、复杂样式和兜底逻辑都必须优先纳入检查。
- 纯 import 排序、空行整理、无语义格式化可不新增业务注释，但仍需在函数注释核对清单或补丁注释核对清单中说明函数位点/补丁位点为 0。
- 若历史旧注释与当前实现冲突，应先确认行为，再决定是否同步修正；不能因历史冲突而跳过本轮改动位点闸门。
