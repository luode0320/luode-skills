# 注释检查规则

## 目标

只检查本轮改动位点是否已经满足注释闸门，不执行注释补写。

## 源规则

以下判定严格沿用源 skill：

- 函数、方法注释的 `[参数]` / `[返回]` / `最近修改时间`
- 方法体步骤编号，至少顶层 `1.`，多步骤时继续 `1.` `2.` `3.`
- 补丁逻辑的“做了什么 + 为什么要加”
- 结构体字面量字段注释
- 关键配置对象字段注释
- 前端改动位点的中文意图注释
- 注释就近落位
- 注释与实现一致
- 过期、错位、误导性旧注释治理

## 审查原则

1. 只读审查，不补写注释、不改文件。
2. 只检查是否满足源 skill 的闸门，不发明新的注释写法。
3. 若争议属于“该不该写、写在哪、写多细”，按源 skill 的边界判断；不在这里自造颗粒度标准。
4. 输出必须能回流到 `comment-completion-gate-rules` 的补齐动作。

## 输入

- 当前 diff
- 目标文件列表
- 现有注释规范或项目约定
- 源 skill 的闸门要求

## 输出

- 注释缺口清单
- 优先级
- 应交给 `comment-completion-gate-rules` 的补充建议
- 需要回流的位点
- 需要继续确认的边界

## 边界

- 不替代 `comment-completion-gate-rules`
- 不执行写入型补注释
- 不处理中文表达细节

## 自动触发信号

- 任何新增或修改代码都需要触发一次注释检查。
- 用户要求“补充注释”“只补注释”“注释完善”“补下注释”“加注释”。
- 新增或修改函数、方法、关键步骤入口和字段处理位点。
- 在原有方法或新增补丁逻辑中出现兜底、兼容、空值保护、重试、短路返回或条件绕行。
- 方法体中已经出现步骤流程，或已经写出多个流程型普通注释。

## 进入后先做什么

1. 先定位本轮新增或修改的代码位点。
2. 本轮首次代码改动后，先执行一次中间阶段注释预检并输出缺口清单。
3. 若本轮是补注释请求，先定位 `git` 未提交改动并锁定优先处理清单。
4. 对每个新增或修改的函数、方法，先检查是否满足源 skill 的元信息、步骤编号和补丁注释门禁。

## 默认执行流程

1. 先读取 `comment-completion-gate-rules`。
2. 需要判断注释该不该写、写在哪、写多细时，再读 `comment-placement-granularity-rules`。
3. 需要判断中文表达时，再交给 `chinese-comment-rules`。
4. 输出函数注释核对清单、字段/结构体字面量注释核对清单和补丁注释核对清单的缺口。
5. 若任一关键项缺失，不得给“通过”结论。

## 需要暂停并确认的条件

- 代码本身可以通过重命名或小重构解决理解问题。
- 现有注释和实现冲突，但代码行为还未确认。
- 为了通过评审而临时补大量无效注释。
- 方法体流程混乱到需要先做结构调整，再决定如何编号。

## 执行通过 / 驳回标准

- 通过：本轮改动位点都完成注释补齐；新增或修改的函数、方法均有 `[参数]` / `[返回]` / `最近修改时间` 且顺序正确；`最近修改时间` 与本轮改动一致且包含改动原因；方法体存在步骤流程时已至少使用 `1.` 并就近落位；若存在多个连续步骤，已补齐 `1.` `2.` `3.`；若顶层步骤存在子步骤，已使用 `1.1`、`1.2` 等分层编号；结构体字面量字段映射和关键配置字段已逐字段或逐字段组补中文注释；修改原有方法或补丁逻辑位点已补“做了什么 + 为什么要加”的就近注释；若是补注释请求，所有未提交改动代码位点都已完成注释规则，不允许只完成其中一部分；每个本轮存在语义改动的前端文件至少包含 1 条与改动直接相关的中文意图注释；首次代码改动后的中间阶段预检已执行；最终已输出函数注释核对清单、字段/结构体字面量注释核对清单与补丁注释核对清单且无缺口。
- 驳回：本轮任一改动位点未补齐注释，或存在步骤流程却没有编号步骤注释，或只写子编号不写顶层编号，或把多个步骤编号集中写在函数开头，或多个流程型普通注释未升级为编号步骤注释，或同一文件内已经存在的同类注释因本轮改动未统一顺序/格式而保留不一致，或新增/修改的函数、方法缺少 `[参数]` / `[返回]` / `最近修改时间`，或元信息顺序不符合要求，或 `最近修改时间` 未更新/无改动原因，或结构体字面量字段映射、关键配置字段仍裸露无注释，或补丁位点缺少“做了什么 + 为什么要加”的就近注释，或任一本轮存在语义改动的前端文件缺少与改动直接相关的中文意图注释，或首次代码改动后未进行中间阶段注释预检，或未输出函数注释核对清单/字段结构体字面量注释核对清单/补丁注释核对清单。

## 执行结果归档要求

- 将本轮注释缺口、已确认可通过项和未完成项记录到 `review/`。
- 归档内容至少包含改动位点、缺失项和下一步动作。
- 如果只是轻量注释核对且无争议，可以简化记录，但应保留结论。

## 输出格式

- 写出缺失注释的文件和位点。
- 说明缺的是函数头、步骤编号、字段注释、补丁说明还是前端意图注释。
- 明确建议交给 `comment-completion-gate-rules` 补齐。

## 读取规则

- 默认先读 `comment-completion-gate-rules`。
- 若需要判断注释该不该写、写在哪、写多细，再读 `comment-placement-granularity-rules`。
- 若需要判断中文表达，再交给 `chinese-comment-rules`。
