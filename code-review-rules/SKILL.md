# 代码评审规则

只在需要做正式代码评审，而不是作者自查时使用这个 skill。
如果当前还处于作者自审阶段，请先转 `implementation-review-rules`；如果只是提交整理问题，请转交 `git-collaboration-rules`。

## Skill 作用与适用场景

- 规范正式代码评审的关注点和阻断标准。
- 强调优先找真实问题，而不是做风格表演。
- 约束评审评论写法，让结论可执行、可追溯。
- 把“实现自审”和“正式评审”拆成两个独立环节。

## 自动触发信号

- 代码已经完成，准备提 PR、走合并前评审或请他人复核。
- 需要输出评审发现、风险、阻断项和结论。
- 评审对象涉及公共模块、高风险逻辑或大范围变更。
- 用户明确要求做 code review。

## 进入后先做什么

1. 先确认作者自审、语法检查和基础清理已经完成。
2. 聚焦行为回归、风险、边界和缺失测试，而不是先看格式末节。
3. 把发现按严重程度排序，并说明影响。
4. 形成通过、需修改或阻断结论。

## 默认执行流程

1. 默认先读 `references/review-focus.md`，先确认正式评审应优先看什么。
2. 如需继续展开，再读 `references/review-comments.md`，需要组织评审意见和评论写法。
3. 需要对照边界或正反例时，再读 `references/review-examples.md`，需要对照正式评审正反例。
4. 输出 评审发现、阻断项和评审结论。
5. 评审未通过时回流编码域修正；评审通过后再进入测试、发布或合并流程。

## 权责边界与不负责事项

- 只负责正式评审，不代替作者自己做实现自审。
- 不把语法、格式和目录归位等基础问题当成全部评审内容。
- 不在没有依据时给出主观性大而不可执行的评论。
- 不因为评审而随意扩大任务范围。

## 需要暂停并确认的条件

- 作者自审和基础清理尚未完成。
- 改动范围或需求背景不清，难以判断行为正确性。
- 关键测试、复现或风险说明缺失。
- 评审意见与任务边界、需求口径明显冲突。

## 执行通过 / 驳回标准

- 通过：能够输出按严重程度排序的真实问题、说明其影响，并给出明确的评审结论和后续动作。
- 驳回：评审只停留在样式挑刺、泛泛主观评论，或没有把重要风险和阻断项说清。

## 执行结果归档要求

- 将评审结论和阻断项记录到 `review/`、PR 评论或评审记录。
- 归档内容至少包含发现列表、严重程度、影响说明和结论。
- 若无发现，也应记录无发现结论和剩余风险 / 测试缺口。

## references 读取规则

- 默认先读 `references/review-focus.md`。
- 只有在 组织评审评论和结论 时，再读 `references/review-comments.md`。
- 只有在 对照评审样例或判断问题严重度 时，再读 `references/review-examples.md`。
