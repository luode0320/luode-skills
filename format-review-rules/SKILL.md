# 格式检查规则

## 目标

只检查是否存在需要交给 `cleanup-format-review-rules` 处理的格式噪音，不执行格式修正。

## 源规则

以下判定严格沿用 `cleanup-format-review-rules`：

- 未使用导入
- 未使用变量
- 未使用方法引用
- 死代码
- 调试残留
- 临时代码
- 无意义打印
- 多余换行
- 格式不一致
- 缩进、对齐、空行、基础排版
- 前端 import 排序/分组、模板 / script / style 区块排版

## 审查原则

1. 只读审查，不改文件、不写文件、不做收口。
2. 只输出“需要交给清理 skill 处理”的位点。
3. 不把语义问题包装成格式问题。
4. 不发明新的格式规范，跟随源 skill 的执行标准。

## 输入

- 当前 diff
- 目标文件列表
- 现有格式规范或项目约定

## 输出

- 格式问题清单
- 风险等级
- 应交给 `cleanup-format-review-rules` 的修正建议
- 需要清理的位点
- 不需要清理但需要留意的边界

## 边界

- 不替代 `cleanup-format-review-rules`
- 不执行写入型修正
- 不做语义重构

## 自动触发信号

- 新增代码文件。
- 修改已有文件中的实现风格。
- 补充同目录下的新函数、新方法或新测试。
- 调整局部写法时需要判断是否与项目现有风格冲突。

## 进入后先做什么

1. 先观察当前目录和相邻文件的既有写法。
2. 识别本项目已经稳定存在的格式约定。
3. 区分“项目既有格式”与“个人理想格式”。
4. 先输出需要交给 `cleanup-format-review-rules` 的位点，再决定是否继续分发。

## 默认执行流程

1. 先读取 `cleanup-format-review-rules`。
2. 若需要判断是可直接清理还是会越界到功能修改，再按源 skill 的默认流程继续读对应 references。
3. 输出可直接清理项、需回到编码阶段的问题和禁止混入的功能修改。
4. 清理未完成前，不进入测试域。

## 需要暂停并确认的条件

- 某项“清理”会改变行为、分支或数据流。
- 当前目录存在大量历史噪音，但本次任务只涉及很小改动范围。
- 调试残留是否需要降级为正式日志仍不清楚。
- 清理动作已经扩散成大面积无关格式化。

## 执行通过 / 驳回标准

- 通过：未使用项、调试残留、死代码和明显格式噪音已清理；新增/修改代码满足最低格式达标（缩进、对齐、空行可读）；且未混入功能修改。
- 驳回：借清理之名扩大改动范围，或把调试残留、临时代码和明显排版混乱代码继续带入测试；前端代码存在明显 import 顺序/分组混乱、未使用 import、模板/script/style 区块排版混乱、缩进或连续空行问题却仍声明完成。

## 执行结果归档要求

- 将清理结论、未处理项和需回流的功能性问题记录到 `review/`。
- 归档内容至少包含已清理项、未清理原因和下一步动作。
- 如果只是轻量整理且无争议，可以简化记录，但应保留结论。

## 输出格式

- 写出文件和位点。
- 说明对应的是未使用项、调试残留、死代码还是基础格式问题。
- 标明是否建议交给 `cleanup-format-review-rules` 处理。

## 读取规则

- 默认先读 `cleanup-format-review-rules`。
- 若需要判定某个问题是否属于“可直接清理”还是“会越界到功能修改”，以源 skill 的通过 / 驳回标准为准。
