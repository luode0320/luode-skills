# 测试文档最小要求

结论：测试文档应让业务读者先了解测试是否支持继续推进；影响：相关人员能判断受影响流程与风险；范围：覆盖测试对象、通过标准、留痕和归档；非范围：不把未执行的测试或环境问题写成功能结论；变化：测试结果、执行细节和追踪依据按层分开；完成标准：主 README 能说明结果并回指详细资料；术语说明：无技术术语需要解释；验证状态：执行时以 local 环境的真实结果为准。

## 白话正文

测试任务的中文 README 先写给业务读者，至少包含：

1. 先说结论：测试是否通过、是否可以继续。
2. 为什么要测：测试对象和会影响的流程。
3. 本次覆盖和未覆盖什么。
4. 怎样算完成：业务可观察的通过标准。
5. 审查和验收是否适用：不适用时写原因和依据；必须验证但条件缺失且无替代方式时才阻断。

## 归档要求

- 主 README 固定放在 `doc/5-tests/YYYY-MM-DD_HHmmss/<测试任务中文主题>/README.md`。
- 详细说明、检查单、案例表和截图解读放在同一时间戳根目录的 ASCII 真实代码路径镜像目录。
- 主 README 必须链接这些详细资料，避免产生孤立附件。
- 不得将测试文档归档到 `testing/`、`analysis/`、仓库根目录或业务代码目录。

完整规则见 [白话文档与附录分层契约](../../artifact-delivery-gate-rules/references/plain-language-document-contract.md)。

## 执行附录

- 对应代码路径或测试对象、local 环境、依赖数据、运行方式、精确命令和清理方式。
- 覆盖案例、失败项、未覆盖项、人工验证步骤，以及同一时间戳根目录的 ASCII 真实代码路径镜像资料入口。

## 追踪附录

- 测试 ID、来源对象、执行时间、执行人、原始日志、产物路径、机器校验、审查结论和证据链接。
- 附录维护规则：执行附录在前、追踪附录在后，且两者连续位于文档末尾。
