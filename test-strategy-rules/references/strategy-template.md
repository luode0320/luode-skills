# 测试策略模板

结论：本模板用于把测试安排收敛为可执行的优先顺序；影响：相关人员能了解先测什么、为何要测和风险在哪里；范围：覆盖目标、风险、类型、环境、数据和待补测项；非范围：不直接替代具体测试执行或功能结论；变化：先给业务可读的测试安排，再后置执行和追踪材料；完成标准：每项测试都能确定优先级、前提和通过依据；术语说明：无技术术语需要解释；验证状态：策略阶段在 local 环境执行后确认。

## 主摘要

主摘要写入 `doc/5-tests/yyyy-MM-DD_HHmmss/<测试任务中文主题>/README.md`，建议至少包含：

1. 任务目标与改动范围。
2. 当前时间戳测试任务目录，或建议拆分出的多个时间戳任务目录。
3. 主要风险来源。
4. 计划采用的测试类型。
5. 优先级排序。
6. 环境与数据前提。
7. 待补测项与原因。
8. 必要测试流向（仅原始测试目标未闭环、仍需进入具体测试 skill 或用户明确要求建议时填写；测试策略已闭环且无必需动作时不写）。

## 反例提示

- 把策略摘要写到 `testing/`、`analysis/` 或仓库根目录。
- 在中文说明目录里继续创建多个策略 markdown，而不是直接把摘要放进 README。
- 已经明显是多轮独立测试，却不拆任务目录，只在一个 README 中持续追加。

## 执行附录

- local 环境、数据准备、测试类型、优先级、执行顺序、精确命令、样本、日志和清理方式。
- 必要测试流向的触发条件和恢复方式，避免把策略摘要当作执行指令。

## 追踪附录

- 测试 ID、来源对象、风险依据、证据、测试任务目录和机器可读定位。
- 附录维护规则：执行附录与追踪附录连续位于文档末尾，追踪附录之后不得再写正文。
