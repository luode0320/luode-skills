# Bug 修复验证规则

只在修复动作已经完成，需要回答“现在能不能确认这个 Bug 已关闭”时使用这个 skill。
如果修复还没完成，请先回到编码或修复方案；如果当前在做旧能力兼容性验证，请同时结合 `test-regression-rules`。

## Skill 作用与适用场景

- 确认原问题现象是否消失，修复目标是否达成。
- 检查修复后是否引入新的可见副作用。
- 决定是否需要补测试、补样例或保留待补测说明。
- 将验证结论、剩余风险和关闭建议统一沉淀到当前 Bug 根目录。
- 形成 Bug 关闭前的验证结论和留痕。

## 自动触发信号

- Bug 修复代码已经提交，准备验证闭环。
- 需要判断这个问题是否能标记为已修复。
- 团队要求说明修复依据、验证方式和剩余风险。
- 修复涉及多个条件分支，需要确认主现象和关联现象都被覆盖。
- 需要把验证结论与前序根因、修复建议记录继续写入同一个 `bug/` 根目录。

## 进入后先做什么

1. 先对照原始 Bug 现象、复现路径和修复目标。
2. 为当前 Bug 确认统一的根目录，并确认关联测试任务目录是否已按 `test-location-rules` 建立。
3. 确认修复后主现象是否消失、异常路径是否符合预期。
4. 检查是否出现新的明显副作用或未覆盖分支。
5. 形成通过、驳回或待补测结论。

## 默认执行流程

1. 默认先读 `references/validation-checklist.md`，先确认修复验证至少应覆盖哪些内容。
2. 如需继续展开，再读 `references/validation-boundaries.md`，需要区分修复验证与功能验证 / 回归验证边界。
3. 需要对照边界或正反例时，再读 `references/validation-template.md`，需要输出统一修复验证结论模板。
4. 输出修复验证结论、未覆盖说明、关联测试目录和关闭建议。
5. 将本轮验证结论写回当前 Bug 根目录。
6. 若发现兼容性和旧能力风险，继续转 `test-regression-rules`；若验证未通过，则回流编码域或 `bug-fix-proposal-rules`。

## 权责边界与不负责事项

- 只负责 Bug 修复闭环，不替代完整测试策略。
- 不代替 `functional-validation-rules` 验证整个新需求是否正确。
- 不把“主现象没了”直接等同于“所有副作用都不存在”。
- 不在未验证关键路径时轻易宣告问题关闭。
- 实际测试脚本、测试数据和测试说明仍应放在 `test/` 目录；本 skill 只统一 Bug 结论记录落点。

## 需要暂停并确认的条件

- 原始 Bug 复现路径和修复目标对不上。
- 修复后主现象虽然消失，但出现新的明显异常。
- 关键验证条件、环境或样本暂时不具备。
- 需要关闭问题，但当前只能给出部分验证结论。
- 当前 Bug 还没有建立统一的 `bug/` 根目录，或验证记录准备再次散落到其他目录。

## 执行通过 / 驳回标准

- 通过：能够说明原问题是否已消失、验证覆盖了哪些关键路径、是否存在副作用和未覆盖项，以及是否具备关闭条件；相关结论统一落在当前 Bug 根目录下。
- 驳回：只说“本地看起来好了”，没有对照原问题、没有记录验证路径和剩余风险，或没有把验证结论继续写回当前 Bug 根目录。

## 执行结果归档要求

- 将修复验证结论统一记录到项目根目录 `bug/` 下的当前 Bug 根目录中，例如 `bug/2026-03-28_171500_订单退款状态错乱排查/`。
- 当前 Bug 根目录必须包含 `README.md`，至少写明验证环境、验证路径、通过项、未通过项、未覆盖说明和关闭建议。
- 如果本轮验证依赖测试脚本、测试数据或测试报告，应继续放在 `test/` 任务目录中，并在当前 Bug 根目录中记录对应测试目录路径。
- 如果结论为待补测或带风险关闭，必须写明缺口和后续动作。

## references 读取规则

- 默认先读 `references/validation-checklist.md`。
- 只有在 判断与相邻测试 skill 的边界 时，再读 `references/validation-boundaries.md`。
- 只有在 套用验证结论模板或对照样例 时，再读 `references/validation-template.md`。
