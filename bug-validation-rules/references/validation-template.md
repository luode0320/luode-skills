# 修复验证模板

结论：{用一句普通中文说明修复是否有效}；影响：{说明受影响的用户、角色或业务环节}；范围：{列出本次会验证的修复场景}；非范围：{列出明确不验证的场景}；变化：{说明修复后用户或业务可感知的变化}；完成标准：{说明可放行或驳回的可观察条件}；术语说明：无技术术语需要解释，或 {术语：它在本次问题中的作用}；验证状态：{说明当前验证进展、覆盖范围和依据}。

## 白话正文与附录

正文先给出修复是否有效、是否可放行、范围和非范围、用户可感知变化、术语含义和未覆盖影响；H1 后单段必须完整包含八个固定字段。环境、样本、命令、逐步结果和证据统一写进文末连续附录。

1. 缺陷名称与原始现象。
2. 修复目标。
3. 当前 Bug 根目录与关联测试目录。
4. 验证环境与样本。
5. 验证步骤。
6. 当前结论：通过 / 驳回 / 待补测。
7. 副作用与未覆盖项。
8. 必要后续动作（仅验证未通过、待补测、需回流修复或用户明确要求建议时填写；验证闭环且无必需动作时不写）。

## 任务阻断收口

- 仅真实阻断时使用本节。
- 当关闭 Bug 所必需的验证无法继续且无替代验证，或修复后出现必须回改的问题时，按 `artifact-delivery-gate-rules/references/task-blocker-closure-contract.md` 完整填写 `BLK-*`、任务状态、阻断阶段、阻断依据与证据、已尝试与停止边界、影响、最多三步解决计划、恢复后重入点和去重键。
- 每一步解决计划必须明确责任方、前置条件、完成判据和 Bug 重验入口；同根因已有 `BLK-*` 时引用已有记录。
- `limited`、`not_applicable`、普通待补测和 P2/P3 风险不使用本节。

## 执行附录

- local 验证环境、样本、逐步操作、执行命令、观察结果、失败预期、清理和回滚。
- 受限验证的人工补验方式、执行人、时间和通过标准。

## 追踪附录

- Bug、修复任务、测试和证据的双向追踪；来源、稳定标识、审查与验收依据、证据定位和结论变更。
