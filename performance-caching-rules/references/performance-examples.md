# 性能优化与过度优化样例

## 正例

- 查询慢先确认索引、筛选条件和分页方式，再决定是否缓存。
- 列表渲染卡顿时先限制数据量、分页或虚拟滚动，而不是盲目重构整页状态。
- 接口耗时高时先找串行依赖、重复计算和无谓 I/O，再决定是否拆异步或加缓存。
- 公共缓存 key 明确区分租户、用户和权限态，避免串数据。

## 反例

- 没有瓶颈证据就引入复杂缓存层。
- 把业务逻辑改写成多层抽象，只为了“以后可能更快”。
- 用缓存掩盖错误的查询或错误的 schema 设计。
- 把前端性能问题一律归因于框架，不看实际渲染热点和数据量。

## 判定提示

- 如果优化收益说不清，成本却明显增加，通常就是过度优化。
- 如果缓存策略讲不清 key、TTL、失效和权限边界，通常还不该上线。
- 如果问题本质是联调环境、网络链路或上游依赖，不要硬塞进本 skill。
