# 性能与缓存规则

只在“性能瓶颈在哪里、缓存值不值得加、缓存怎么失效、渲染热点怎么控”这个问题上使用这个 skill。
如果当前问题是 schema 结构、查询实现或前端组件职责本身，请优先转交相邻 skill。

## Skill 作用与适用场景

- 统一 SQL 性能、接口耗时、缓存读取写入、缓存 key、缓存失效和前端渲染热点的边界。
- 区分性能问题、结构问题、查询问题、缓存问题和业务逻辑问题。
- 约束缓存引入条件、失效策略、观察点和一致性风险。
- 防止把过早优化、过度抽象或危险缓存副作用包装成“性能优化”。

## 自动触发信号

- 新增或修改 SQL 性能优化、接口耗时控制、缓存读取写入、缓存 key、缓存失效、大列表、图表、虚拟滚动。
- 需要判断某个热点是否应靠索引、查询改写、缓存、分页、批量处理或渲染优化解决时。
- 发现接口慢、查询慢、缓存击穿、缓存雪崩、缓存脏读或前端列表卡顿时。
- 团队计划以“优化”为名引入大范围代码重构、过度抽象或额外状态层时。

## 进入后先做什么

1. 先确认当前问题是性能瓶颈，还是功能错误、结构失衡或环境问题伪装成的性能问题。
2. 识别热点位于数据库访问、接口服务、缓存层、前端渲染还是网络链路。
3. 判断当前优化是否有明确瓶颈依据，不凭感觉添加缓存或复杂抽象。
4. 明确当前变更是否会引入一致性、失效、兼容性或可维护性成本。

## 默认执行流程

1. 默认先读 `references/performance-boundaries.md`，判断性能问题该落在哪一层解决。
2. 如果涉及缓存策略、key 设计和失效规则，再读 `references/cache-strategy-rules.md`。
3. 如果需要对照正反例或判断是否过度优化，再读 `references/performance-examples.md`。
4. 输出瓶颈归因、优化手段、缓存必要性、失效策略、观测指标和禁止改法。
5. 如果问题本质应回流 schema、query 或前端结构层，停止停留在性能层并转交相邻 skill。

## 权责边界与不负责事项

- 只负责性能与缓存边界，不负责表结构、索引和迁移，那属于 `database-schema-rules`。
- 不负责 SQL / Repository / 事务 / 锁的实现细节，那属于 `database-query-rules`。
- 不负责前端组件结构和页面职责，那属于前端组件类 skill。
- 不把功能逻辑变化、数据口径变化或权限策略变化伪装成性能优化。
- 不默认“加缓存”就是第一选择，也不默认“先上虚拟滚动”能解决所有前端性能问题。

## 需要暂停并确认的条件

- 当前优化没有明确瓶颈证据，只是主观猜测。
- 当前缓存会影响一致性、幂等性、权限态或用户态隔离，但边界未说明。
- 当前所谓优化需要重写大量无关代码，已经偏离最小改动原则。
- 当前性能问题更像环境差异、联调链路或上游依赖问题，而非代码热点。

## 执行通过 / 驳回标准

- 通过：瓶颈归因清楚，优化手段与问题层级匹配，缓存策略和失效规则可解释，且没有借优化名义扩大无关改动。
- 驳回：没有瓶颈依据就加复杂缓存、把结构问题硬留在性能层、引入明显一致性风险，或用大规模重构伪装成优化。

## 执行结果归档要求

- 将性能优化结论记录到任务记录、评审记录、性能说明或交付说明中。
- 归档内容至少包含瓶颈点、优化目标、优化手段、缓存策略、失效规则、观测指标和风险说明。
- 如果新增缓存，必须记录 key 组成、TTL、失效触发点、绕过条件和不适用场景。

## references 读取规则

- 默认先读 `references/performance-boundaries.md`。
- 只有在涉及缓存读写、key 和失效时，再读 `references/cache-strategy-rules.md`。
- 只有在需要对照正反例和过度优化样例时，再读 `references/performance-examples.md`。
