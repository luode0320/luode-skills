# 代码上下文重同步规则

只在“要继续改代码，但发现代码与记忆不一致”时使用。
目标是避免把用户手动修改或最新代码误覆盖回旧版本。

## 核心原则

- 当前文件内容永远是基线，记忆内容只能做参考。
- 先重读再改，禁止基于旧快照直接写入。
- 以“增量合并”为默认策略，不做“回滚还原式覆盖”。
- 如果不确定某段变更意图，先暂停并向用户确认。

## 自动触发信号

- 准备改某文件时，发现内容与自己记忆明显不一致。
- `apply_patch`/补丁因上下文不匹配失败。
- 同一文件在本轮内被用户或其他流程改动过。
- 代码结构、函数签名、变量命名与上次读取结果不一致。
- 发现“自己想改的行”已被手动调整为另一种实现。

## 进入后先做什么

1. 立刻停止基于旧记忆的写入动作。
2. 重新读取目标文件（必要时连同关联文件一起读取）。
3. 提取“用户已手动修改的区域”和“本次需求仍需完成的区域”。
4. 设计最小改动合并方案：保留用户修改，只补本次需求缺口。
5. 写入后复查 diff，确认没有把用户改动回退掉。

## 默认执行流程

1. **重读基线**：重新读取将要修改的文件最新内容。
2. **对齐差异**：把“记忆中的计划修改点”映射到最新代码位置。
3. **保留用户改动**：用户已改内容默认保留，不可默认覆盖。
4. **最小增量落地**：仅修改本次需求相关代码，避免顺手重构。
5. **提交前自检**：检查 diff 中是否出现“与本次需求无关的还原/删除”。
6. **异常即阻断**：若出现大段异常回退，撤销本次思路并重读后再改。

## 强制禁止项

- 禁止按旧记忆整段覆盖当前文件。
- 禁止因为“和记忆不同”就把代码还原成上次生成版本。
- 禁止在未重读当前文件前继续沿用旧 patch。
- 禁止把用户刚改的逻辑当成“脏改动”自动清理。

## 需要暂停并确认的条件

- 无法判断某段手动修改是临时调试还是正式实现。
- 用户改动与本次需求目标发生逻辑冲突，无法自动兼容。
- 继续改动将影响多个模块，且意图存在多解。
- 发现潜在破坏性回退风险（大量删除/回滚非目标代码）。

## 通过 / 驳回标准

- 通过：以当前文件为基线完成需求，且保留了用户手动修改；diff 仅包含本次任务相关改动。
- 驳回：出现基于旧记忆的覆盖还原、误删用户手动改动、或引入与任务无关的大段回退。

## 与其他规则的关系

- 本规则优先解决“上下文漂移与覆盖风险”，不替代业务实现规则。
- 若任务同时涉及提交动作，应再触发 `git-collaboration-rules`。
- 若发现不一致来自真实缺陷，应转入 Bug 相关流程继续定位。
