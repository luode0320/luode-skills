# 语法校验规则

只在“代码已经写完，先确认它至少能过基础语法 / 类型 / 引用这一关”时使用这个 skill。
如果当前问题是实现质量、注释风格或未使用导入清理，请转给相邻审查 skill。

## Skill 作用与适用场景

- 在进入测试前检查语法、类型和依赖引用是否存在明显问题。
- 识别容易导致构建失败的基础错误。
- 让测试阶段不被低级语法和引用问题浪费时间。
- 作为实现自审之后的第二道静态质量关口。
- 对前端代码改动，必须覆盖 `.vue`、`.tsx`、`.jsx`、`.ts`、`.js` 的 import/export、组件语法、模板绑定、类型引用和基础构建风险。

## 自动触发信号

- 新增或修改代码后准备进入测试。
- 变更涉及类型系统、依赖导入、模块引用或构建入口。
- 不确定当前改动是否存在明显编译失败风险。
- 新增或修改 `.vue`、`.tsx`、`.jsx`、`.ts`、`.js` 前端代码后准备声明完成或进入测试。

## 进入后先做什么

1. 先确认本次改动涉及哪些语言和构建单元。
2. 检查最容易出错的语法、类型、导入和引用位置。
3. 判断是否存在明显构建失败风险。
4. 如果存在基础错误，先拦下，不进入测试域。
5. 若涉及前端代码，优先执行项目已有的 `vue-tsc`、`tsc --noEmit`、ESLint、npm/yarn/pnpm lint/build 等本地基础校验；如果项目没有命令或依赖不可用，至少人工核对 import/export、模板绑定、组件注册、路径别名和类型引用。

## 默认执行流程

1. 默认先读 `references/check-scope.md`，确认本轮语法校验覆盖哪些问题。
2. 如果涉及类型和引用，再读 `references/type-and-reference-risks.md`。
3. 如果需要对照常见错误样例，再读 `references/check-examples.md`。
4. 输出通过项、语法 / 类型 / 引用问题和是否允许进入测试。
5. 语法未过时，不进入后续测试和交付流程。

## 权责边界与不负责事项

- 只负责语法、类型、引用和明显构建失败风险，不代替 `implementation-review-rules` 检查实现质量。
- 不代替 `cleanup-format-review-rules` 清理未使用导入、调试残留和格式噪音。
- 不代替测试域验证功能是否正确。
- 不要求在这里处理所有构建生态细枝末节，只拦明显基础问题。

## 需要暂停并确认的条件

- 当前项目缺少必要的基础校验上下文，无法判断语法 / 类型 / 构建状态。
- 不同语言或构建单元之间存在耦合，单点结论不可靠。
- 继续推进会让低级错误直接带进测试阶段。

## 执行通过 / 驳回标准

- 通过：无明显语法错误、类型错误、引用缺失和基础构建失败风险；前端改动已完成可用的 lint/typecheck/build 或人工基础语法/引用核对。
- 驳回：存在会直接导致编译失败、类型失配、引用缺失或运行前就可确认的基础错误；前端 import/export、模板绑定、路径别名、组件注册或类型引用明显异常却仍声明完成。

## 执行结果归档要求

- 将语法校验结论、未通过项和修正建议记录到 `review/`。
- 归档内容至少包含检查范围、问题类型、影响文件和下一步动作。
- 如果本次改动很小且校验无争议，可以简化记录，但应保留通过结论。

## references 读取规则

- 默认先读 `references/check-scope.md`。
- 只有在判断类型和引用风险时，再读 `references/type-and-reference-risks.md`。
- 只有在对照常见错误样例时，再读 `references/check-examples.md`。
