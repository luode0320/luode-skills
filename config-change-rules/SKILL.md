# 配置变更规则

只在“配置应该怎么分层、怎么注入、默认值该不该给、敏感项能不能进仓库”这个问题上使用这个 skill。
如果当前争议是认证鉴权、安全策略或请求头协议，请转交对应安全与接口类 skill。

## Skill 作用与适用场景

- 作为第六波入口，先稳定配置与环境差异边界。
- 统一配置来源、配置分层、优先级、默认值策略、密钥占位和特性开关规则。
- 防止把业务逻辑、危险默认值、真实密钥、客户端不可公开配置混进配置层。
- 为后续数据库、接口、安全、性能和部署类 skill 提供稳定配置基线。

## 自动触发信号

- 新增或修改 `.env`、`.env.example`、`application.yml`、`application.yaml`、`settings.py`、`config.ts`、`appsettings.json`、`values.yaml` 等配置文件。
- 新增或修改环境变量读取、默认值、配置注入、配置分层、特性开关、密钥占位逻辑。
- 需要区分服务端私有配置、客户端可公开配置、环境差异配置和运行时注入配置时。
- 发现真实凭证、数据库连接串、签名密钥、第三方密钥被写入仓库、日志或客户端配置时。

## 进入后先做什么

1. 先识别当前变更涉及的配置类型：公开配置、私有配置、密钥、默认值、环境覆盖、特性开关。
2. 判断配置消费端属于服务端、客户端、构建时还是运行时。
3. 明确当前仓库已有配置来源和优先级，不随意叠加新层级。
4. 先确认当前改动是真配置变更，还是把业务逻辑伪装成配置。

## 默认执行流程

1. 默认先读 `references/config-layering-and-sources.md`，确定配置来源、分层和优先级。
2. 如果涉及默认值、密钥、敏感配置或客户端暴露问题，再读 `references/defaults-and-secrets.md`。
3. 如果需要判断边界、反例或是否混入业务逻辑，再读 `references/config-boundaries-and-examples.md`。
4. 输出配置来源、分层方案、默认值策略、敏感级别、失败策略和回滚注意项。
5. 配置边界未稳定前，不继续扩展数据库、安全、接口和发布类变更。

## 权责边界与不负责事项

- 只负责配置来源、分层、默认值和密钥占位边界，不负责认证鉴权策略本身，那属于 `auth-security-rules`。
- 不负责请求头协议和透传规则，那属于 `request-header-rules`。
- 不负责具体发布流程、灰度策略和回滚执行，那属于发布类 skill。
- 不负责把配置改动包装成业务逻辑切换或运行时判断，这种情况应回流对应代码位点 skill。
- 如果配置问题本质上是联调环境差异或线上排障问题，应在后续由 `integration-debugging-rules` 或交付域 skill 接手。

## 需要暂停并确认的条件

- 当前 diff 中出现真实密钥、密码、私钥、带凭证连接串或可直接使用的第三方令牌。
- 当前配置既可能进入客户端，也可能只应停留在服务端，暴露边界不清。
- 当前配置修改实际改变了业务规则、权限模型或核心流程，但没有明确业务确认。
- 当前仓库已有多个配置来源，优先级和覆盖关系不清，贸然新增会扩大环境差异。

## 执行通过 / 驳回标准

- 通过：配置来源和分层清楚，默认值安全可解释，敏感配置外置且未泄露，客户端与服务端暴露边界明确，配置缺失时处理策略合理。
- 驳回：真实密钥进入仓库、危险默认值被静默启用、客户端暴露了私有配置，或配置变更实质上混入了未确认的业务逻辑。

## 执行结果归档要求

- 将配置变更结论记录到配置变更说明、部署说明或任务记录中。
- 归档内容至少包含配置项、配置来源、适用环境、默认值策略、敏感级别、缺失处理方式和回滚注意项。
- 如果涉及新增环境变量或密钥占位，必须同步记录示例文件、占位规则和注入方式。

## references 读取规则

- 默认先读 `references/config-layering-and-sources.md`。
- 只有在涉及默认值、密钥、公开配置和敏感配置时，再读 `references/defaults-and-secrets.md`。
- 只有在需要边界判定和正反例时，再读 `references/config-boundaries-and-examples.md`。
