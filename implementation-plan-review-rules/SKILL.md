# 实施计划审查镜像规则

## 目标

只检查当前实施计划是否满足 `implementation-plan-rules` 的要求，不执行计划落文和后续编码推进。

## 源规则

以下判定严格沿用源 skill：

- 需求、边界和验收标准应已基本稳定
- 明确文件落点、模块职责、开发周期和阶段目标
- 每个阶段默认只做一件事
- 任务顺序、依赖关系、最小闭环和验证点清楚
- 新增文件必须给目录树，用途要标注
- 数据库变更必须给 SQL
- 禁止 `TBD`、`TODO`、`后续再补` 这类占位计划

## 自动触发信号

- 当前任务准备进入编码，但计划涉及多个文件、模块、接口、步骤或验证点。
- 已有实施计划，但担心顺序、阶段边界或收口质量有遗漏。
- 用户要求先写实施计划、任务清单或编码前执行方案。
- 主线程已声明使用 `implementation-plan-rules`，但计划复杂度较高。

## 进入后先做什么

1. 先确认需求文档、边界结论和验收标准是否已基本稳定。
2. 检查当前计划是否只覆盖当前优先闭环，而不是把多个并列目标硬塞在一起。
3. 检查文件 / 模块落点是否明确。
4. 检查阶段划分、依赖顺序和验证点是否完整。
5. 检查是否存在新增文件目录树、数据库 SQL 或图形化步骤缺口。

## 默认执行流程

1. 先读取 `implementation-plan-rules`。
2. 需要核对进入条件时，再读源 skill 的 `references/plan-entry-checklist.md`。
3. 需要核对计划结构时，再读源 skill 的 `references/plan-structure-template.md`。
4. 需要核对粒度、顺序或最小闭环时，再读源 skill 的 `references/task-granularity-and-order.md`。
5. 需要核对计划自审项时，再读源 skill 的 `references/plan-review-checklist.md`。
6. 输出通过项、阶段混杂问题、依赖顺序问题、验证闭环缺口和归档缺口。
7. 不直接写计划文档，只输出问题和建议。

## 权责边界与不负责事项

- 不代替 `requirement-splitting-rules` 做需求拆分。
- 不代替 `acceptance-criteria-rules` 编写验收标准。
- 不代替实际编码或测试执行。
- 不执行实施计划文档回写。
- 不接受以计划名义掩盖前置需求问题。

## 需要暂停并确认的条件

- 需求、边界或验收标准仍在明显变化。
- 当前任务还没拆出可独立推进的优先闭环。
- 明明可以拆成多个子任务周期，却仍被写成一个大周期。
- 主要文件落点、模块职责或依赖顺序无法稳定。
- 计划中仍存在明显占位词或无法验证的任务项。

## 执行通过 / 驳回标准

- 通过：开发周期、阶段划分、文件落点、依赖关系、最小闭环、验证点、风险和阻断项完整，且每个阶段目标单一，没有明显计划收口缺口。
- 驳回：只是重述需求，没有落实到周期、阶段、文件和执行步骤；或仍充满占位词、阶段混杂、顺序不清、无法验证。

## 执行结果归档要求

- 如本轮审查发现计划结构、阶段边界或验证闭环缺口，应将结论记录到 `analysis/` 或评审记录。
- 归档内容至少包含计划对象、发现的问题、建议调整方向和影响阶段。
- 如果只是一次轻量预审且无争议，可不单独归档，但应保证评审可追溯。

## 读取规则

- 默认先读 `implementation-plan-rules`。
- 只有在核对进入条件时，再读 `references/plan-entry-checklist.md`。
- 只有在核对计划结构时，再读 `references/plan-structure-template.md`。
- 只有在核对粒度、顺序和最小闭环时，再读 `references/task-granularity-and-order.md`。
- 只有在核对计划自审项时，再读 `references/plan-review-checklist.md`。
