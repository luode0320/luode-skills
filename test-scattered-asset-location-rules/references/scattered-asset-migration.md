# 散落测试资产识别与迁移

## 1. 默认判定

出现以下任一情况时，默认按散落测试资产处理：

- 文件名或内容明显属于测试脚本、测试程序、验证代码。
- 文件是 mock、fixture、测试输入输出样例、测试日志、测试截图或测试报告。
- 文件是测试说明、验证记录或测试结论，但未与对应测试任务目录关联。
- 文件只为当前测试目的服务，却落在生产目录、仓库根目录或非测试目录中。

## 2. 明确禁止的位置

- 仓库根目录
- 业务代码目录，如 `src/`、`app/`、`lib/`、`internal/`、`common/`
- `docs/`、`wiki/` 等业务文档目录
- `scripts/`、`tools/` 等非测试目录
- 中文说明目录内部

## 3. 迁移原则

- 优先把当前任务直接相关的散落资产迁移到当前测试时间戳根目录。
- 迁移时保持最小改动，不顺手清理无关历史残留。
- 如果现有文件兼具生产和测试职责，先拆分为两个文件，再分别归位。
- 对说明性文件，迁移后要保证中文说明目录或 `README.md` 仍可追踪到真实测试资产路径。

## 4. 需要升级处理的场景

出现以下情况时，不只按“散落资产迁移”处理，还要转交 `go-test-compile-path-rules`：

- 散落文件位于 Go 源码目录，且会进入编译链路。
- 出现源码目录 `*_test.go`。
- 中文路径进入 Go 可编译测试包目录。
- 白盒同包诉求被当成落地测试文件的理由。

## 5. 最小检查清单

- 当前文件是否真的属于测试资产，而不是生产资产。
- 当前文件是否散落在 `test/` 根目录之外。
- 迁移目标是否已存在正确的时间戳根目录。
- 是否误把真实测试资产放进中文说明目录。
- 是否遗漏了需要升级到 Go 编译路径规则的问题。
