# 交付总结规则

只在需要形成一份可交接、可复盘、可追溯的交付说明时使用这个 skill。
如果当前还没有稳定实现结论、测试结论或发布结论，请先完成相邻环节；如果只是做上线动作，请转 `release-rollout-rules`。

## Skill 作用与适用场景

- 把本次交付的变更、验证、风险和遗留事项收成一份可读说明。
- 降低后续交接、复盘和维护成本。
- 让交付域不只停留在发布动作，还能形成信息闭环。
- 避免“代码已经合了，但没人知道到底改了什么”。

## 自动触发信号

- 任务已经准备交付，需要说明变更内容、验证结论和风险。
- 用户明确要求给出交付说明、更新摘要或后续建议。
- 需要把实施结果交给测试、产品、运维或下一位维护者。
- 需要留下一份可供后续快速定位的交付记录。

## 进入后先做什么

1. 先确认需求、实现、测试和发布结论已经基本收口。
2. 提炼本次交付的目标、主要改动和影响范围。
3. 汇总已验证项、未验证项、风险点和遗留事项。
4. 形成结构化交付摘要和后续建议。

## 默认执行流程

1. 默认先读 `references/delivery-template.md`，先按统一结构输出交付说明。
2. 如需继续展开，再读 `references/delivery-risk-and-debt.md`，需要整理风险、遗留和待补项。
3. 需要对照边界或正反例时，再读 `references/delivery-examples.md`，需要对照交付总结正反例。
4. 输出 交付总结、验证摘要、风险与遗留说明。
5. 交付总结完成后，可用于提交需求方、测试方、发布方或后续维护者；若总结中暴露新缺口，应回流相邻阶段修正。

## 权责边界与不负责事项

- 只负责交付说明，不代替代码评审、测试执行或上线动作。
- 不把所有实现细节写成冗长日志，重点是让接收方知道要点。
- 不隐去已知风险和未覆盖项。
- 不在结论尚未收敛时伪造“已完全完成”的交付口径。

## 需要暂停并确认的条件

- 实现、测试或发布结论仍明显不稳定。
- 关键验证结果、遗留问题或风险项还没汇总。
- 交付对象和需要的说明深度不明确。
- 团队试图用交付说明掩盖实际未完成项。

## 执行通过 / 驳回标准

- 通过：交付说明能清楚说明目标、主要改动、验证结论、风险、遗留事项和后续建议，并适合接收方快速使用。
- 驳回：交付说明只写空泛总结，缺少验证、风险和遗留说明，无法支持后续交接。

## 执行结果归档要求

- 将交付总结记录到 `delivery/`、任务记录、发布说明或 `analysis/`。
- 归档内容至少包含变更摘要、验证结论、风险项、遗留事项和建议动作。
- 如果后续又有补交付或补验证，应保留增量更新记录。

## references 读取规则

- 默认先读 `references/delivery-template.md`。
- 只有在 梳理交付中的风险与遗留事项 时，再读 `references/delivery-risk-and-debt.md`。
- 只有在 对照样例或判断总结是否足够可交接 时，再读 `references/delivery-examples.md`。
