# Header 边界与样例

## 正例

- 使用认证头承载访问凭证，而不是把令牌放进 query 参数。
- 使用 trace-id / span-id 传递链路上下文，并在日志层单独定义记录规则。
- 使用幂等键控制重试去重，并明确由谁生成、保存和校验。
- 对租户头明确只能由可信入口注入，下游不直接信任任意客户端值。

## 反例

- 把业务筛选条件、分页参数、复杂 JSON 上下文塞进 Header。
- 既让客户端写 `X-Forwarded-For`，又直接把它当真实 IP 使用。
- 把 trace 头当作日志字段规范本身，导致 Header 规则和日志规则纠缠。
- 把对象级权限、角色判定规则直接依赖某个可伪造 Header，而没有可信来源控制。

## 判定提示

- 如果值需要参与业务查询、分页、筛选，通常更像请求参数而不是 Header。
- 如果问题核心是“记录什么”和“在哪里记录”，通常更像日志规则而不是 Header 规则。
- 如果问题核心是“谁有权访问”，通常更像认证鉴权规则而不是 Header 规则。
