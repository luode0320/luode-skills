# 功能验证结论模板与样例

结论：本模板用于判断当前功能是否满足已确认的业务要求；影响：相关人员能了解验证结果、风险和是否可以继续；范围：覆盖验证对象、范围、步骤、结果和证据；非范围：不把环境或归档问题直接归因为功能失败；变化：先给业务可读结论，再把执行和追踪材料置于附录；完成标准：验证范围、实际结果和未通过项均有可复核依据；术语说明：无技术术语需要解释；验证状态：功能执行后以 local 环境的真实结果确认。

## 结论模板

主结论写入 `doc/5-tests/yyyy-MM-DD_HHmmss/<测试任务中文主题>/README.md`，至少包含：

- 验证对象：
- 关联需求 / 变更：
- 验证范围：
- 执行环境：
- 验证步骤摘要：
- 实际结果：
- 未通过项 / 风险项：
- 详细证据路径：
- 结论：通过 / 驳回 / 待确认
- 下一步流转建议：

## 任务阻断收口

- 仅真实阻断时使用本节。
- 当明确必需的功能验证无法继续且无替代验证，或测试污染导致本次结论无效时，按 `artifact-delivery-gate-rules/references/task-blocker-closure-contract.md` 完整填写 `BLK-*`、任务状态、阻断阶段、阻断依据与证据、已尝试与停止边界、影响、最多三步解决计划、恢复后重入点和去重键。
- 解决计划必须写明责任方、前置条件、完成判据和功能重验入口；同根因已有 `BLK-*` 时引用已有记录。
- `limited`、`not_applicable`、仅静态验证或已有人工验证交接的待确认状态不使用本节。

## 正例

- 对新增接口验证请求参数、返回结果和错误分支，并在 README 中写明详细记录位于 `doc/5-tests/2026-03-28_180000/internal/api/user_create_validation.md`。
- 对页面交互验证关键操作路径、提示文案和状态变化，并把截图、日志和补充步骤放到对应的 ASCII 镜像路径目录中。
- 对修改后的业务逻辑验证核心输入输出和关键边界条件，并记录未通过项和回改建议。

## 反例

- 只写“已测通过”，没有验证范围、步骤和结论依据。
- 本应验证当前需求，却扩成全量系统回归。
- 把联调失败直接判成当前功能不通过，但没有区分环境问题和实现问题。
- 把测试资源目录问题、文档缺失问题当成功能失败结论。
- 把功能验证结论写到 `testing/`、交付说明或中文说明目录之外的其他位置，导致测试留痕脱离当前时间戳任务目录。

## 执行附录

- local 环境、验证步骤、精确命令、样本、截图、日志、接口报文、失败项和清理方式。
- 人工补验与环境问题的记录方式，确保无法执行时不会伪报功能结论。

## 追踪附录

- 关联需求或变更、测试 ID、详细证据路径、风险项、审查结论和机器可读定位。
- 附录维护规则：执行附录在前、追踪附录在后，且两者是文档最后的连续章节。
