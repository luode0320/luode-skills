# 回归验证审查镜像规则

## 目标

只检查当前回归验证方案或结论是否满足 `test-regression-rules` 的要求，不执行回归验证和结论回写。

## 源规则

以下判定严格沿用源 skill：

- 只验证旧能力、关联链路、共享依赖和兼容行为是否被带坏
- 不代替当前功能验证
- 回归范围、用例选取、未覆盖项和遗留风险应明确
- 结论应写入中央约定 README，详细证据应落到 ASCII 镜像路径
- 禁止为了测试目的污染生产代码
- 回归范围未稳定前，不直接宣告任务具备交付条件

## 自动触发信号

- Bug 修复、功能迭代、公共模块修改、共享逻辑调整或接口兼容变化后准备做回归验证。
- 当前功能验证已完成，需要继续确认旧逻辑和兼容行为是否被破坏。
- 变更涉及公共方法、公共组件、共享配置、基础库或复用链路。
- 用户明确要求确认“会不会影响别的地方”。
- 主线程已声明使用 `test-regression-rules`，但回归收口风险较高。

## 进入后先做什么

1. 先确认当前需求或 Bug 的功能验证已经基本完成。
2. 检查回归范围、影响面和主要兼容风险是否已说明。
3. 检查用例选取、未覆盖项和遗留风险是否清楚。
4. 检查 README 结论与 ASCII 镜像路径证据落点是否明确。
5. 检查是否存在测试污染或把功能验证和回归验证混写的问题。

## 默认执行流程

1. 先读取 `test-regression-rules`。
2. 需要核对回归范围和用例选择时，再读源 skill 的 `references/regression-scope-selection.md`。
3. 需要核对功能验证、联调验证和回归验证边界时，再读源 skill 的 `references/regression-boundaries.md`。
4. 需要核对模板和样例时，再读源 skill 的 `references/regression-template-and-examples.md`。
5. 输出通过项、范围缺口、用例遗漏、兼容风险遗漏、留痕缺口和测试污染风险。
6. 不直接执行回归，不直接写 README，只输出问题和建议。

## 权责边界与不负责事项

- 不代替 `functional-validation-rules` 验证当前需求本身是否正确。
- 不代替 `test-strategy-rules` 做测试优先级和资源收口。
- 不代替联调域处理环境和链路打通问题。
- 不代替测试资源类 skill 管理目录、命名和测试程序。
- 不执行回归结论回写。

## 需要暂停并确认的条件

- 当前功能验证尚未通过。
- 本次改动影响面不清，难以稳定界定回归范围。
- 当前回归环境、历史数据或关键链路依赖不具备。
- 回归范围已经扩张到全量测试计划。

## 执行通过 / 驳回标准

- 通过：回归范围、用例选取、兼容性影响、未覆盖项和留痕方式完整，且没有发现测试污染或范围误判。
- 驳回：无法证明旧能力未被带坏，关键回归范围未验证，留痕落点错误，或为了测试通过引入测试专用生产代码。

## 执行结果归档要求

- 如本轮审查发现回归范围、兼容性影响、用例选取或留痕缺口，应将结论记录到 `analysis/` 或评审记录。
- 归档内容至少包含回归对象、发现的问题、建议补测方向和风险缺口。
- 如果只是一次轻量预审且无争议，可不单独归档，但应保证评审可追溯。

## 读取规则

- 默认先读 `test-regression-rules`。
- 只有在核对回归范围和用例选取时，再读 `references/regression-scope-selection.md`。
- 只有在核对验证边界时，再读 `references/regression-boundaries.md`。
- 只有在核对结论模板和样例时，再读 `references/regression-template-and-examples.md`。
