# 实现自审镜像规则

## 目标

只检查本轮实现是否满足 `implementation-review-rules` 的收口标准，不执行修改。

## 源规则

以下判定严格沿用源 skill：

- 可读性优先、单一职责、命名语义化
- 注释完整、错误处理明确、日志可追溯
- 依赖使用审慎、魔法值治理、冗余逻辑清理
- 500 行及以上且持续膨胀的文件必须拆分评估
- 必须核验 `comment-placement-granularity-rules` 与 `comment-completion-gate-rules`
- 必须检查可复用公共工具是否被重复封装
- 必须检查最近修改超过 7 天的高复用通用代码是否被直接改旧行为
- Go 项目必须检查 `test/` 外 `*_test.go`
- Go 项目必须检查 `internal/service` 根目录落点、结构体散落、`var (...)`、多行签名、`map[string]any` 硬编码解析
- 后端 HTTP API 改动必须同步 Swagger/OpenAPI

## 自动触发信号

- 功能代码、Bug 修复代码或重构代码刚完成。
- 准备进入测试、联调或回归验证前。
- 需要确认当前实现是否已经达到可交给测试域的最低质量。
- 代码评审前希望先做一轮作者自审。
- Go 代码改动涉及测试、服务层、结构体落点、局部变量分组、多参数签名或第三方响应解析。
- 后端 HTTP API 已新增或修改。
- 代码已改但尚未确认注释双 skill 是否完成。
- 本轮目标文件已达 500 行及以上且仍在继续膨胀。
- 本轮代码虽然可构建，但存在明显排版混乱风险。

## 进入后先做什么

1. 先确认实现目标已经落地，不在目标未收敛时机械自审。
2. 再检查是否偏离最小改动原则。
3. 优先检查表达层冗余、嵌套三元、过密一行逻辑和无价值包装。
4. 按可读性、结构、命名、常量抽离、依赖选择、错误处理、日志、注释、编码规范逐项检查。
5. 核验本轮改动是否已执行注释双 skill。
6. 检查存在步骤流程的函数/方法是否编号注释就近落位。
7. 判断当前实现是否具备进入语法检查和测试阶段的基础条件。
8. 对 Go 项目执行禁放扫描、服务层落点扫描、结构体扫描、局部变量分组扫描、多参数签名扫描和第三方响应解析扫描。
9. 如果本轮改动涉及后端 HTTP API，检查 Swagger/OpenAPI 是否已同步。
10. 检查 500+ 行文件、工具复用和 7 天冻结问题。

## 默认执行流程

1. 先读取 `implementation-review-rules`。
2. 如果涉及边界、正反例或审查范围，再按源 skill 的默认流程继续读对应 references。
3. 输出通过项、驳回项、风险项和需要回改的实现问题。
4. 自审未通过时，不进入测试域。

## 需要暂停并确认的条件

- 当前实现已经明显偏离最小改动原则。
- 注释、命名、结构问题已经严重到需要回到编码阶段重做。
- 当前实现存在多处质量争议，且不同修法会影响后续测试或评审判断。
- 当前方案为了“优化”而引入过多抽象、过多依赖或明显维护负担。
- Go 项目命中 `test/` 外 `*_test.go`，但是否改 seam 方案尚未达成一致。
- Go 项目命中 `internal/service/[^/]+.go` 根目录落点，但业务子目录拆分方案尚未达成一致。
- Go 项目发现 `internal/service` 内请求/响应/第三方结果结构体散落，但迁移边界尚未达成一致。
- 第三方 API 响应结构既无文档也无法通过探测脚本确认。
- 本轮涉及后端 HTTP API，但 Swagger/OpenAPI 是否必须同步仍未确认。
- 500+ 行文件拆分会引发跨模块调用改造，且本轮范围是否允许拆分尚未确认。
- 已检索到可复用公共工具，但是否允许替换调用路径尚未达成一致。
- 命中通用代码 7 天冻结，但是否允许直接修改旧行为尚未达成一致。

## 执行通过 / 驳回标准

- 通过：实现目标达成，且没有明显违反可读性优先、单一职责、命名语义化、常量抽离、依赖治理、错误处理、日志可追溯、注释更新时间、UTF-8 与中文注释、编号注释就近落位等要求的重大问题；并且本轮改动已完成注释双 skill，`cleanup-format-review-rules` 与 `syntax-check-review-rules` 的基础收口已完成；对 500 行及以上且持续膨胀的文件已完成拆分或给出本轮可执行拆分方案；已确认可复用公共工具未被重复封装，且命中 7 天冻结时未直接修改旧行为；Go 场景下 `test/` 外不存在 `*_test.go`，本轮改动未命中 `internal/service/[^/]+.go` 根目录落点，`internal/service` 未散落请求/响应/第三方结果结构体，函数/方法内未使用 `var (...)` 分组声明局部变量，多参数函数签名未直接采用多行参数列表，第三方响应解析未使用 map + key 硬编码；涉及后端 HTTP API 时，Swagger/OpenAPI 已同步。
- 驳回：实现层质量明显不达标，或本轮改动未完成注释双 skill 收口，或未完成基础格式/语法收口，或 500 行及以上文件继续堆功能却没有拆分动作/方案，或存在已可复用公共工具却仍重复封装同语义实现，或命中 7 天冻结仍直接修改高复用通用代码旧行为，或 Go 场景命中禁放项、落点项、结构体散落项、`var (...)`、多行签名、map 硬编码解析，或本轮涉及后端 HTTP API 但 Swagger/OpenAPI 未同步。
- 驳回：发现函数头部集中编号注释、后续步骤没有对应编号注释就近落位。

## 执行结果归档要求

- 将自审结论、驳回项和回改建议记录到 `review/`。
- 归档内容至少包含检查范围、通过项、未通过项和下一步动作。
- 如果只是轻量实现且自审无争议，可以简化记录，但应保留结论。

## 输出格式

- 写出检查范围、通过项、驳回项和下一步建议。

## 读取规则

- 默认先读 `implementation-review-rules`。
- 需要边界或正反例时，再按源 skill 的默认流程继续读对应 references。
