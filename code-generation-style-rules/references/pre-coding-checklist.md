# 编码前检查清单

## 必读来源

开始写代码前按顺序确认：

1. 用户本轮明确要求和限制。
2. 目标文件最新内容；不存在时读取拟落点同目录或同模块样例。
3. 根目录 `PROJECT_STYLE.md` 中与当前任务相关且状态启用的条目。
4. 全局用户风格反例库 `code-style-consistency-rules/references/user-style-feedback-library.md` 中状态为 active 的条目，作为本轮必须规避的用户禁用写法。
5. 本轮已经命中的编码类 skill 及其当前阶段要求。
6. 项目内已有 helper / util / common / 依赖声明，确认是否可复用。
7. 当前文件、同目录或同模块是否已经形成高度统一的代码结构；若是，提取一个可直接复用的局部模板。
8. 如果本轮实现已有接口，查找最近且稳定的既有接口实现，记录文件、符号和需要沿用的结构。

## 判断问题

- 这次改动是否真需要写代码，还是可以不写。
- 当前文件已有哪种命名、结构、注释、日志、错误处理和排版习惯。
- 同目录样例是否与当前文件一致；若不一致，优先当前文件。
- 当前上下文是否存在连续、同构或重复出现的代码模式；若存在，新增代码只允许做目标名称和必要业务内容的替换。
- 局部统一风格是否要求保持调用链、注册顺序、注释位置、空行、字段顺序或错误处理方式不变。
- 如果实现接口，是否已经找到同文件、同目录、同模块或同语言项目的既有实现；找到后是否记录参考差异。
- 如果找不到接口实现或候选样例冲突，是否已经明确降级依据或停止原因。
- `PROJECT_STYLE.md` 是否提供可复用片段，而不是抽象口号。
- 本轮是否有明确禁用写法，例如不新增抽象、不装依赖、不批量格式化。
- 当前目标是否可以通过复制既有局部模板并替换最少内容完成；如果可以，不得额外引入 helper、变量、循环、日志、校验或抽象。
- 是否命中全局用户风格反例库中的 active 反例；命中则按其对应正例改写。

## 允许简化的场景

- 纯一行配置或常量调整时，契约可以只记录命名、排版和禁用无关格式化。
- 纯测试支撑代码时，仍需确认测试命名、fixture 位置、辅助函数风格和数据构造方式。
- 纯脚本修改时，仍需确认编码、错误退出、日志输出和参数解析风格。

## 不允许简化的场景

- 涉及公共方法、接口、数据库、日志、错误处理、安全边界、金额或时间计算。
- 新增文件、新增工具函数、新增测试框架辅助、跨目录迁移。
- 当前文件和 `PROJECT_STYLE.md` 存在明显冲突。
