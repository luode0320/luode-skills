# 风格优先级

## 默认优先级

当多个风格来源发生冲突时，按以下顺序裁决：

1. 用户本轮明确要求，前提是不违反安全、兼容、测试和仓库级硬规则。
2. 当前正在修改文件中的稳定写法。
3. 同目录、同模块、同语言的稳定样例。
4. `PROJECT_STYLE.md` 中状态为启用且适用范围命中的风格样例。
5. 通用 `code-*`、命名、注释、日志、错误处理等 skill 的默认规则。

## 冲突处理

- 当前文件风格与 `PROJECT_STYLE.md` 冲突时，优先跟随当前文件，并在收口时记录是否需要用 `project-style-rules` 更新长期风格。
- 用户要求与当前文件风格冲突时，先判断是否只是局部要求；若会改变团队长期写法，暂停确认是否回写 `PROJECT_STYLE.md`。
- 通用规则与局部稳定写法冲突时，优先保证当前改动不产生局部割裂；若局部写法明显存在安全或正确性问题，转交对应专业 skill 处理。
- 为了统一风格需要大面积改动时，不在当前任务中顺手做；除非用户明确要求风格治理任务。

## 禁止降级

- 不得把“没有读到风格”解释为可以自由发挥。
- 不得只读取 `PROJECT_STYLE.md` 而跳过当前文件或同目录样例。
- 不得用个人偏好覆盖项目稳定写法。
- 不得把一次性需求写入长期风格，除非它已经被确认会反复适用。
