# 风格优先级

## 默认优先级

当多个风格来源发生冲突时，按以下顺序裁决：

1. 用户本轮明确要求，前提是不违反安全、兼容、测试和仓库级硬规则。
2. 当前正在修改文件中的稳定写法。
3. 同目录、同模块、同语言的稳定样例。
4. `PROJECT_STYLE.md` 中状态为启用且适用范围命中的风格样例。
5. 通用 `code-*`、命名、注释、日志、错误处理等 skill 的默认规则。

## 局部上下文与接口实现专项裁决

- 当前文件存在连续、同构或重复出现的稳定代码模式时，优先把该模式作为本轮局部模板；新增代码只做完成目标所需的最小替换。
- 同目录或同模块存在多个相似实现时，优先采用距离当前改动最近、重复出现次数最多且没有已知正确性问题的写法。
- 实现已有接口时，接口参考实现的优先级高于外部模板和个人习惯；参考顺序为同文件、同目录/同模块、同语言项目样例。
- 接口没有可参考实现时，必须在风格契约中记录“无既有实现可参考”，再降级使用当前文件和同目录稳定风格；不得静默发明新的实现模板。
- 多个参考实现无法收敛时，停止编码并记录冲突，不得为了推进而任选一种个人偏好。
- 局部风格不得覆盖安全、正确性、兼容、错误处理和接口契约等硬性要求；必要的例外代码必须在契约中说明原因。

## 冲突处理

- 当前文件风格与 `PROJECT_STYLE.md` 冲突时，优先跟随当前文件，并在收口时记录是否需要用 `project-style-rules` 更新长期风格。
- 用户要求与当前文件风格冲突时，先判断是否只是局部要求；若会改变团队长期写法，暂停确认是否回写 `PROJECT_STYLE.md`。
- 通用规则与局部稳定写法冲突时，优先保证当前改动不产生局部割裂；若局部写法明显存在安全或正确性问题，转交对应专业 skill 处理。
- 为了统一风格需要大面积改动时，不在当前任务中顺手做；除非用户明确要求风格治理任务。

## 禁止降级

- 不得把“没有读到风格”解释为可以自由发挥。
- 不得只读取 `PROJECT_STYLE.md` 而跳过当前文件或同目录样例。
- 不得用个人偏好覆盖项目稳定写法。
- 不得把一次性需求写入长期风格，除非它已经被确认会反复适用。
- 不得在已有统一代码区段中额外加入 helper、wrapper、factory、manager、临时变量、循环、日志、校验、抽象或其他无需求代码。
- 不得实现接口时跳过既有实现检索，直接套用外部模板或个人习惯。
