# 代码风格契约模板

## 内部契约字段

本轮开始写代码前，至少整理以下字段。最终回复可按任务复杂度简述，不要求机械展开全部字段。

| 字段 | 写法 |
| --- | --- |
| 适用范围 | 本轮要新增、修改或重构的文件 / 目录 / 语言 |
| 风格来源 | 用户要求、当前文件、同目录样例、`PROJECT_STYLE.md`、已命中编码类 skill |
| 局部风格证据 | 当前文件 / 同目录 / 同模块中可直接复用的稳定片段、文件位置和适用范围；没有时写 `N/A + 原因 + 证据` |
| 接口参考实现 | 接口名称、参考文件/符号、沿用的结构；没有时写 `N/A + 原因 + 证据` |
| 命名 | 本轮命名偏好、禁用命名、领域词选择 |
| 结构 | 函数组织、参数模型、分支顺序、目录归属、复用方式 |
| 注释 | 中文注释、函数头、步骤编号、补丁说明、字段注释 |
| 日志 | 日志等级、字段、模板、上下文、脱敏 |
| 错误处理 | 返回、异常、兜底、重试、降级、错误映射 |
| 复用 | 项目内工具、标准库、已装依赖、禁止重复实现 |
| 排版 | 缩进、空行、换行、导入顺序、声明写法 |
| 最小新增内容 | 为完成当前目标允许新增的代码、字段、分支、注释或错误处理 |
| 禁用写法 | 本轮不得引入的抽象、模板风格、无意义防御、批量格式化 |
| 用户反例规避 | 全局用户风格反例库中命中的 active 反例 id 与对应正例改写 |
| 冲突裁决 | 若来源冲突，写清跟随哪一侧和原因 |
| 回写判断 | 是否需要联动 `project-style-rules` 更新 `PROJECT_STYLE.md` |

## 最小输出示例

```markdown
代码风格契约:
- 适用范围: internal/order/*.go
- 风格来源: 当前文件 + 同目录 service 样例 + PROJECT_STYLE.md
- 局部风格证据: 当前文件已有同构注册区段，新增项只替换业务名称和必要中文说明
- 接口参考实现: N/A + 本轮不实现接口；证据：目标仅修改已有调用区段
- 命名: 沿用 orderID / tenantID，不新增缩写
- 结构: 保持单函数单阶段处理，不新增接口抽象
- 注释: 复杂分支补中文步骤注释，普通赋值不注释
- 日志: 复用 applog.Infof/Errorf，字段包含 order_id 和 stage
- 错误处理: 沿用当前函数返回 error 的包装方式
- 复用: 优先复用已有 helper，不新增第三方依赖
- 排版: gofmt；多参数日志按当前文件两段式换行
- 禁用写法: 不做全文件格式化，不引入全局配置工厂
- 最小新增内容: 仅新增当前目标所需的一行注册调用
- 用户反例规避: 命中 STYLE-CASE-GO-002 禁用 iota，改为显式常量赋值
- 回写判断: 本轮仅跟随旧风格，不更新 PROJECT_STYLE.md
```

## 使用约束

- 契约是写代码前的约束，不是写完后的解释。
- 简单一行修改可以形成简短契约，但不能完全跳过来源判断。
- 契约不得覆盖安全、测试、兼容和仓库级硬规则。
- 如果存在局部统一模板，契约必须明确“只做必要替换”和“不得新增内容”；如果实现接口，必须明确参考实现或无参考实现的降级依据。
