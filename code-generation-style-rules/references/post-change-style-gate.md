# 写完后的风格闸门

## 核对顺序

1. 对照本轮代码风格契约，确认每项约束都有落实或有明确不适用原因。
2. 对照当前文件与同目录样例，确认没有引入局部风格跳变。
3. 如果存在局部统一代码区段，逐项比较新增代码的调用结构、顺序、空行、注释、排版和错误处理，确认只做必要替换。
4. 如果本轮实现接口，确认契约中记录了既有参考实现、沿用内容和必要差异；没有参考实现时确认已记录降级依据。
5. 对照 `PROJECT_STYLE.md`，确认启用样例没有被误用或忽略。
6. 对照已命中的编码类 skill，确认最小改动、可读性、命名、注释、日志、错误处理等专业规则仍成立。
7. 判断是否形成新的稳定团队偏好；若是，联动 `project-style-rules` 回写。

## 必须驳回的情况

- 未读当前文件或同目录样例就声称“按项目风格写”。
- 只根据 `PROJECT_STYLE.md` 改写局部文件，导致当前目录出现割裂。
- 借风格统一做了无关格式化、重排 import、批量改名或大面积重构。
- 新增代码使用外部模板风格，和项目已有写法明显不一致。
- 在高度统一的局部区段中新增了 helper、wrapper、factory、manager、临时变量、循环、日志、校验、抽象或其他无需求代码。
- 实现接口时未检索既有实现，或没有记录参考实现/降级依据。
- 新增代码改变了既有实现的调用链、注册顺序、注释位置、排版或错误处理方式，且没有正确性、安全、兼容或接口契约证据。
- 已形成新的稳定偏好，却没有说明是否需要回写 `PROJECT_STYLE.md`。

## 收口说明口径

最终回复中不需要展开完整契约，但应能回答：

- 本轮主要跟随了哪类风格来源。
- 是否有风格冲突，以及如何裁决。
- 是否更新了 `PROJECT_STYLE.md`。
- 若只做静态规则资产修改，应说明没有业务运行路径需要验证。
