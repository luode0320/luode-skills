# 代码风格检查规则

## 目标

只检查当前改动是否偏离 `code-style-consistency-rules` 的约定，不执行风格修正。

## 源规则

以下判定严格沿用源 skill：

- Go 路由单行注册格式
- 路由说明注释放在调用上一行
- 同一函数步骤内连续路由注册使用代码块包裹
- 函数/方法内局部变量禁止 `var (...)`
- 函数/方法签名默认单行参数
- 参数过多时优先参数结构体而不是多行签名
- 多参数函数调用与格式化日志优先单行
- 日志必须保持“模板一行、参数下一行”的两段式
- `strings.TrimSpace` 仅在业务规则明确需要时使用
- `iota` 禁止用于常量和枚举

## 审查原则

1. 只读审查，不改文件、不写文件。
2. 只检查是否跟随项目现有写法，不引入外部模板。
3. 若风格与语义纠缠，交还给可读性或最小改动规则，不在这里自创解决方案。
4. 输出必须能回流到 `code-style-consistency-rules` 的修正动作。

## 输入

- 当前 diff
- 目标文件列表
- 项目风格约定
- 源 skill 的风格约定

## 输出

- 风格偏离清单
- 优先级
- 应交给 `code-style-consistency-rules` 的修正建议
- 需要回流的位点

## 边界

- 不替代 `code-style-consistency-rules`
- 不执行写入型修正
- 不处理业务语义变化

## 自动触发信号

- 新增代码文件。
- 修改已有文件中的实现风格。
- 补充同目录下的新函数、新方法或新测试。
- 调整局部写法时需要判断是否与项目现有风格冲突。

## 进入后先做什么

1. 先观察当前目录和相邻文件的既有写法。
2. 识别本项目已经稳定存在的风格约定。
3. 区分“项目既有风格”与“个人理想风格”。
4. 在当前必要改动范围内跟随已有写法，而不是强推新风格。

## 默认执行流程

1. 先读取 `code-style-consistency-rules`。
2. Go 代码改动默认补读 `references/go-coding-rules.md`。
3. 优先跟随当前模块、当前目录、当前语言在仓库内的既有写法。
4. 如果仓库内存在明显冲突风格，暂停并确认应跟随哪一侧，而不是现场自定标准。

## 需要暂停并确认的条件

- 同一项目存在两套明显冲突的既有风格，无法判断该跟随哪一套。
- 为了保持一致必须扩大到大范围无关改动。
- 用户明确要求引入新风格，但仓库中没有对应先例。
- 当前目录的历史风格本身质量很差，继续跟随会明显降低可维护性。

## 执行通过 / 驳回标准

- 通过：当前改动与项目现有写法一致，没有引入新的局部风格跳变。
- 驳回：把个人偏好、外部模板风格或新的局部规则带进现有项目，导致同层代码出现明显割裂感。

## 执行结果归档要求

- 如果仓库内存在冲突风格并做了跟随选择，将结论记录到 `analysis/` 或 `review/`。
- 归档内容至少包含跟随依据、冲突位置和暂定处理结论。
- 如果本次只是常规跟随现有风格且无争议，可以不单独归档。

## 输出格式

- 写出偏离风格的文件和位点。
- 说明偏离的是路由、局部变量、函数签名、日志调用、字符串裁剪还是常量写法。
- 明确建议交给 `code-style-consistency-rules` 修正。

## 读取规则

- 默认先读 `code-style-consistency-rules`。
- Go 代码改动默认补读 `references/go-coding-rules.md`。
