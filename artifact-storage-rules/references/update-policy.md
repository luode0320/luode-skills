# 复用与更新策略

## 架构域

- `1-总架构.md`、`2-目录树.md`、`3-模块职责.md`、`4-主要业务链路.md` 是稳定主入口，后续变化直接更新原文档。
- 同一业务目标、入口和职责边界下的链路变化，持续更新原编号业务链路文档。
- 新业务链路具备独立业务目标、入口或责任模块时，使用当前最大业务链路编号加一创建 `<递增序号>-业务链路-<业务中文名称>.md`；首条业务链路从 `5` 开始。
- 新建链路后，在 `4-主要业务链路.md` 增加索引、摘要和状态。
- 链路重命名时保留原编号，只更新中文业务名称、索引及所有引用。
- 链路废弃或删除后不复用其编号，也不重排后续链路编号。
- 业务链路改变模块边界、目录职责或项目顶层理解时，同步更新 `3-模块职责.md`、`2-目录树.md`、`1-总架构.md`，必要时再同步根目录 `项目设计.md`。

## 需求域

- 同一个需求只保留一份需求入口文档，流程和时序表达随主文档正文一并维护，不再拆分独立图文件。
- 需求、实施、验收和通用 Markdown 文档的位图统一复用 `doc/data/images/`；同一图片允许多文档引用，任何删除动作必须先检索全部 Markdown 引用并确认替代版本已落盘。
- 图片返修沿用同一资产主干并递增 `-v<number>`；旧版本只有在无任何引用、资产清单已更新且回滚证据保留时才可删除。
- `requirement-intake-rules` 的 `gap-routing`、`requirement-boundary-rules`、`requirement-splitting-rules`、`requirement-change-rules` 默认围绕这同一份需求文档推进。
- `requirement-intake-rules` 的 `gap-routing` 在关键缺口阻断时允许创建一份临时缺口文档，命名复用对应需求文档主干并追加 `.gap.md`；该文档仅用于 gap 未关闭阶段，用户确认并回填主需求文档后必须删除。
- `requirement-boundary-rules`、`requirement-splitting-rules`、`requirement-change-rules` 的稳定结论仍直接更新主需求文档，不额外新建平行长期入口。
- 验收标准、最终验收和实施文档都与来源对象文档分开维护；来源对象可以是需求或 Bug，同一来源对象各自只保留一份主文档；实施域允许在一份实施总览下按需拆实施周期文档。
- 新项目、项目初期或多来源对象需要统一排序时，应在 `doc/3-实施/` 下维护一份“需求与实施计划全量顺序实施方案”；同一项目或同一来源集合只保留一份总顺序文档，后续新增需求、优先级调整、依赖变化或周期顺序变化都更新原文档。
- 全量顺序实施方案只负责跨需求 / 跨来源对象的总执行顺序，不替代单个来源对象的实施总览；实施总览和实施周期仍按来源对象持续维护。
- 当用户要求“补充需求、实施文档”时，需求域仍持续更新同一份需求主文档；实施域则必须同时回开同一份实施总览与所有受影响的实施周期文档，不得只更新总览。
- 若本次补充已形成新的独立实施闭环，而原实施总览下尚无可承载的周期文档，应在同一实施总览下新增对应实施周期文档；旧周期正文只作为历史记录保留。
- 来源对象未稳定前，不进入验收标准文档和实施文档；若需求 gap 未关闭、Bug 根因未定位、边界未裁清、拆分未完成或变更未重算，必须阻断进入这两类下游文档。
- 没有验收标准，不得进入实施规划和正式编码。
- 没有完成测试和审核，不得进入最终验收文档。
- 若来源对象变更影响范围、默认值、交互、交付物、修复口径或实施顺序，必须回开验收标准与实施文档，并使原最终验收失效待重验。
- 若来源对象变更会影响跨需求依赖、项目级期次排序或全量执行入口，还必须同步回开对应的“需求与实施计划全量顺序实施方案”。

## Bug 域

- 同一个 Bug 从录入到验证，默认沿用同一个 Bug 根目录。
- 后续 skill 应在同一个根目录下补充记录，而不是为同一 Bug 再新建并行目录。

## 测试域

- 同一轮测试、同一批验证证据、同一波回归，默认沿用同一个测试时间戳根目录。
- 如果是新的独立测试轮次、不同目标或不同验证波次，应新建新的时间戳根目录。

## 最近上下文预热

- 新会话近期上下文预热只读取已有持久化材料，不单独创建新目录。
- 如果预热结果影响当前需求、Bug 或测试路径，应回写到对应主域文档，而不是单独散落。

## 项目设计文档

- 同一个项目的总体设计、架构、大纲说明默认只保留一份根目录主入口 `项目设计.md`。
- 若根目录存在 `架构设计.md`、`项目大纲.md`、`系统设计.md`、`项目说明.md` 等历史同类文档，可先作为弱参考读取，但后续维护应尽量收口到 `项目设计.md`。
- 当项目架构、核心模块边界、目录结构、总体链路、关键依赖或设计大纲发生明显变化时，应同步更新 `项目设计.md`。
- 如果当前项目尚无根目录 `项目设计.md`，但已经完成一次全项目分析或全项目梳理，可补建该文件作为后续统一主入口。

## 项目专属 skill

- 同一个项目的专属规则应持续沉淀在根目录 `skill/`，避免散落到根目录或通用文档目录。
- 已有主题 skill 优先增量更新，不重复新建同义目录。
- 新增独立主题时再新建子目录，保持“一主题一 skill”的可检索结构。
