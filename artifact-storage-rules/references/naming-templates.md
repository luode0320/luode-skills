# 命名模板

本文件定义默认命名模板。模板调整时，应先修改 `path-map.yaml`，这里负责补充说明。

## 需求域

- 需求文档：`doc/2-需求/YYYY-MM-DD_HHmmss_<需求中文简介标题>.md`
- 临时缺口文档：`doc/2-需求/<需求文档同名主干>.gap.md`
- 需求正文图示：直接写入需求主文档正文，不另建独立流程图或时序图文件
- 需求图片素材（截图/外部图片）：`doc/2-需求/data/<需求文档同名主干>.<素材标识>.<扩展名>`
  - 示例：`doc/2-需求/data/2026-05-25_103000_支付路由改造.req-shot-login.png`
- 验收标准：`doc/7-验收/YYYY-MM-DD_HHmmss_<来源对象标识>_验收标准.md`
- 最终验收：`doc/7-验收/YYYY-MM-DD_HHmmss_<来源对象标识>_最终验收.md`
- 需求与实施计划全量顺序实施方案：`doc/3-实施/YYYY-MM-DD_HHmmss_<项目或来源集合标识>_需求与实施计划全量顺序实施方案.md`
- 实施总览：`doc/3-实施/YYYY-MM-DD_HHmmss_<来源对象标识>_实施总览.md`
- 实施周期：`doc/3-实施/YYYY-MM-DD_HHmmss_<来源对象标识>_实施周期NN_<周期中文主题>.md`
  - 示例：需求文档 `doc/2-需求/2026-04-16_141518_多业务复用Swap适配控制需求.md`
  - 对应临时缺口文档：`doc/2-需求/2026-04-16_141518_多业务复用Swap适配控制需求.gap.md`
  - 对应验收标准：`doc/7-验收/2026-04-16_141518_多业务复用Swap适配控制需求_验收标准.md`
  - 对应实施总览：`doc/3-实施/2026-04-16_141518_多业务复用Swap适配控制需求_实施总览.md`
  - 对应实施周期：`doc/3-实施/2026-04-16_141518_多业务复用Swap适配控制需求_实施周期01_接口与配置落点.md`
  - 对应最终验收：`doc/7-验收/2026-04-16_141518_多业务复用Swap适配控制需求_最终验收.md`

补充要求：

- `<需求中文简介标题>` 必须使用本次需求的中文简介标题，禁止固定写成“需求中文说明”。
- 需求、实施、Bug、测试、审查、验收等活动文档或任务目录的时间前缀统一使用 `YYYY-MM-DD_HHmmss`，不得退化成只有日期的 `YYYY-MM-DD`。
- 来源对象标识用于回指进入实施、审查或验收阶段的上游对象，可以来自需求文档，也可以来自 Bug 根目录；该标识不重复文件名前置时间戳，优先使用来源中文主干或短 ID，必要时加 `需求-` 或 `Bug-` 类型前缀，命名中必须能看出来源是哪个需求或哪个 Bug。
- 验收标准与最终验收文件名必须使用 `YYYY-MM-DD_HHmmss` 时间前缀，并在时间后保留来源对象标识，分别追加“_验收标准”“_最终验收”。
- 需求与实施计划全量顺序实施方案必须使用 `YYYY-MM-DD_HHmmss` 时间前缀，并在时间后保留项目或来源集合标识，再追加“_需求与实施计划全量顺序实施方案”；该文档只在新项目、项目初期或多来源对象统一排序时使用，不替代单来源对象实施总览。
- 实施总览文件名必须使用 `YYYY-MM-DD_HHmmss` 时间前缀，并在时间后保留来源对象标识，再追加“_实施总览”。
- 实施周期文件名必须使用 `YYYY-MM-DD_HHmmss` 时间前缀，并在时间后保留来源对象标识，再追加“_实施周期NN_中文主题”；不得只写“时间_阶段_说明”，必须能从文件名看出来自哪个需求或 Bug。
- 同一天存在多个近似标题时，在不改变中文语义前提下补充更具体描述，避免重名。
- 同一需求的补齐、边界、拆分、变更应持续更新同一份需求文档，不新建平行入口。
- 临时缺口文档只在 gap 阶段存在；用户确认并回填主需求文档后必须删除，不保留为长期平行入口。
- 需求文档落地时应在正文中同步补齐 Mermaid 流程图与时序图，不再另建配套图文件。
- 需求正文中的图示应与正文、表格和口径保持一致。
- 用户提供的需求截图与外部生成图片统一存放到 `doc/2-需求/data/`，并在需求主文档内使用 Markdown 图片语法直接引用展示。
- 素材标识建议使用 ASCII：如 `req-shot-login`、`state-diagram-v2`，避免空格和中文标点。

## 架构域

- 总架构入口：`doc/1-架构/1-总架构.md`
- 目录树入口：`doc/1-架构/2-目录树.md`
- 模块职责入口：`doc/1-架构/3-模块职责.md`
- 主要业务链路索引：`doc/1-架构/4-主要业务链路.md`
- 第一条业务链路：`doc/1-架构/5-业务链路-<业务中文名称>.md`
- 后续业务链路：`doc/1-架构/<递增序号>-业务链路-<业务中文名称>.md`
- 其他架构专题附录：`doc/1-架构/附录-<架构中文主题>.md`
- 架构配套图：`doc/1-架构/<架构文档同名主干>.<图类型>.svg`

补充要求：

- 四个主入口必须使用上述 `1-4` 稳定中文文件名，不创建无编号或同义平行入口。
- `1-总架构.md` 负责架构总览并链接目录树、模块职责和主要业务链路，不替代根目录 `项目设计.md` 的项目级总览职责。
- `4-主要业务链路.md` 是业务链路索引与摘要入口，不承载所有链路的完整步骤。
- 单条业务链路从序号 `5` 开始；同一链路持续更新原编号文档，新增独立链路取当前最大业务链路编号加一。
- 业务链路废弃、删除或重命名后，不复用其历史编号，也不重排后续文件，避免已有引用整体漂移。
- 新增、重命名或废弃业务链路时，同步更新 `4-主要业务链路.md`。
- 其他长期架构专题使用 `附录-<架构中文主题>.md`，不占用业务链路递增序号，也不插入 `1-4` 主入口序列。
- `<业务中文名称>` 使用稳定、可检索的中文业务动作或结果，例如“用户登录”“订单支付”“内容发布”，禁止使用“新流程”“链路一”一类临时名称。
- `<架构中文主题>` 应明确体现专题，例如“配置目录规范”“后台系统结构方案”。
- 架构域用于配置目录、后台结构、运行时读取、字段规范、编码规范等长期专题，不与需求域混写。

## 实施域

- 需求与实施计划全量顺序实施方案：`doc/3-实施/YYYY-MM-DD_HHmmss_<项目或来源集合标识>_需求与实施计划全量顺序实施方案.md`
- 实施总览：`doc/3-实施/YYYY-MM-DD_HHmmss_<来源对象标识>_实施总览.md`
- 实施周期：`doc/3-实施/YYYY-MM-DD_HHmmss_<来源对象标识>_实施周期NN_<周期中文主题>.md`

补充要求：

- `<项目或来源集合标识>` 用于表达新项目或多来源对象集合，例如“交易后台重构”“支付链路一期”；它不是单个需求文档的来源对象标识。
- 需求与实施计划全量顺序实施方案是跨需求 / 跨来源对象的实施顺序总表，负责链接需求主文档、验收标准、实施总览、实施周期和周期内最小任务，并记录总执行顺序、依赖、状态和阻断项。
- 同一个项目或同一来源集合默认只维护一份全量顺序实施方案；新增需求、调整优先级或改变实施周期顺序时，继续更新原总表。
- `<周期中文主题>` 应明确体现当前周期唯一目标，例如“接口与配置落点”“页面联调闭环”“回归与收口”。
- 实施文档文件名必须保留 `<来源对象标识>`，来源可以是需求或 Bug，格式形如 `YYYY-MM-DD_HHmmss_来源对象标识_实施周期NN_周期说明.md`，禁止只保留 `YYYY-MM-DD_HHmmss_阶段_说明.md`。
- 来源对象标识不重复文件名前置时间戳，优先使用来源中文主干或短 ID；若仅看中文主干无法区分需求和 Bug，应写成 `需求-xxx` 或 `Bug-xxx`。
- 同一来源对象先维护一份实施总览；只有任务体量确实较大时才拆实施周期文档。
- 每个实施周期只承载一个清晰目标，禁止一个周期混装多组并列能力。
- 需求与实施计划全量顺序实施方案、实施总览与实施周期文档统一放在 `doc/3-实施/`，不要散落到 `doc/1-架构/`、`doc/2-需求/` 或 `doc/`。

## 验收域

- 验收标准：`doc/7-验收/YYYY-MM-DD_HHmmss_<来源对象标识>_验收标准.md`
- 最终验收：`doc/7-验收/YYYY-MM-DD_HHmmss_<来源对象标识>_最终验收.md`

补充要求：

- 验收标准是前置验收口径，发生在实施前。
- 最终验收是后置放行结论，发生在测试和审核之后。
- 同一来源对象只保留一份验收标准文档和一份最终验收文档，后续补充和变更继续更新同一份文档。

## Bug 域

- Bug 根目录：`doc/4-bugs/YYYY-MM-DD_HHmmss_问题中文简介/`
- Bug 主入口文件：`doc/4-bugs/YYYY-MM-DD_HHmmss_问题中文简介/README.md`

补充要求：

- 同一个 Bug 的后续复现、定位、诊断、方案和验证都沿用同一根目录。
- `README.md` 作为唯一主入口文件，流程图与时序图统一用 Mermaid 语法直接写入正文，不另建独立 `flow.svg`/`sequence.svg` 等图文件，与需求域"正文内嵌 Mermaid 图示、不另建配套图文件"的约定保持一致。
- 其他附件和细节记录围绕该根目录组织。

## 测试域

- 测试任务根目录：`doc/5-tests/YYYY-MM-DD_HHmmss/`
- 测试中文主说明：`doc/5-tests/YYYY-MM-DD_HHmmss/<测试任务中文主题>/README.md`
- 真实测试资产：放在同一时间戳根目录下的 ASCII 真实代码路径镜像目录中

补充要求：

- `<测试任务中文主题>` 为动态目录名，应与该 `README.md` 将要描述的测试任务主题一致，禁止固定写成“测试任务中文简介”。
- 会被 Go 工具链扫描或编译的路径必须保持 ASCII。
- 中文说明默认只放在中文主说明目录，不进入编译路径。

## 审查域

- 审查主文档：`doc/6-审查/YYYY-MM-DD_HHmmss_<来源对象标识>_<审查中文主题>.md`
- 审查汇总入口：`doc/6-审查/README.md`

补充要求：

- 审查域用于结构归位、实现审查、格式清理、语法检查、注释核对和规则核对结论。
- 审查文档必须包含时分秒前缀，并在时间后保留来源对象标识（需求或 Bug），禁止使用 `YYYY-MM-DD_<审查中文主题>.md` 或 `YYYY-MM-DD_HHmmss_<审查中文主题>.md` 这类缺少来源对象标识的命名。
- 来源对象标识不重复文件名前置时间戳，优先使用来源中文主干或短 ID；若仅看中文主干无法区分需求和 Bug，应写成 `需求-xxx` 或 `Bug-xxx`。
- 若项目需要长期维护审查索引，优先收口到 `doc/6-审查/README.md`。

## doc 域

- 通用项目文档默认放在 `doc/` 下。
- 具体文件名按文档主题命名，但应避免使用 `temp.md`、`result.txt` 这类无意义名称。

## skill 域

- 项目专属 skill 默认放在 `skill/` 下。
- 子 skill 建议目录：`skill/<topic>/SKILL.md`。
- `<topic>` 建议使用 ASCII 主题名并统一 `-rules` 后缀，便于跨平台和工具扫描。

## 根目录项目设计主入口

- 项目设计主入口：`项目设计.md`

补充要求：

- 允许兼容读取 `架构设计.md`、`项目大纲.md`、`系统设计.md`、`项目说明.md` 等历史同类文件。
- 如果当前仓库已经存在上述历史别名，后续分析和同步时应尽量收口到根目录 `项目设计.md`。
- `项目设计.md` 作为项目级总览入口；更细的专题设计、附录和长期说明可继续放在 `doc/` 下。
