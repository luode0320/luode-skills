# 根目录职责

本文件定义当前项目研发产物的默认根目录职责边界。目录名如果未来调整，以 `path-map.yaml` 为最终准。

## 当前默认根目录

- `doc/1-架构/`
  用于项目架构、目录树、目录职责、主要业务 / 功能设计架构、模块关系、关键链路、配置目录、后台结构、字段规范、运行时读取、编码规范等长期架构专题文档。
  默认维护 `1-总架构.md`、`2-目录树.md`、`3-模块职责.md`、`4-主要业务链路.md` 四个有序中文主入口；独立业务链路从 `5-业务链路-<业务中文名称>.md` 开始依次下推，并由 `4-主要业务链路.md` 统一索引。
- `doc/2-需求/`
  用于需求文档、需求补齐、边界结论和需求变更等需求域持久化产物。
  其中允许 `requirement-gap-rules` 在 gap 未关闭时创建临时缺口文档；该文档补齐并回填主需求文档后必须删除，不作为长期平行入口保留。
  其中 `doc/2-需求/data/` 用于需求截图、外部生成图片和需求文档引用的图片素材。
- `doc/3-实施/`
  用于实施总览、实施周期拆分、阶段推进总览和各周期实施文档。
- `doc/4-bugs/`
  用于 Bug 录入、复现、根因、运行时诊断、修复建议、风险评估和修复验证等 Bug 域持久化产物。
  规范目录名固定为 `doc/4-bugs/`，`buy/` 不是合法 Bug 根目录。
- `doc/5-tests/`
  用于测试任务目录、测试说明、测试脚本、模拟数据、验证证据、测试截图、测试日志和回归结果。
- `doc/6-审查/`
  用于结构归位、实现自审、格式清理、语法检查、注释核对和规则审查等审查类持久化结论。
  审查类结果应统一进入 `doc/` 体系，不再单独维护根级 `review/`、`ment/` 或 `analysis/`。
- `doc/7-验收/`
  用于前置验收标准和后置最终验收文档。
- `doc/`
  用于不直接归属单一需求、单一 Bug、单一测试任务或单一审查结论的通用项目文档、专题说明和长期说明材料。
- `skill/`
  用于当前项目专属 skill 资产（项目私有编码规则、工具包用法、框架约定等）的统一目录入口。

## 根目录特殊主入口

- 根目录 `项目设计.md`
  作为项目级设计、架构、大纲和总体说明的统一主入口文件。
  它是根目录特殊主入口，不放入 `doc/` 子目录。
  允许兼容读取 `架构设计.md`、`项目大纲.md`、`系统设计.md`、`项目说明.md` 等历史同类文档，但后续主入口应尽量收口到 `项目设计.md`。

## 目录职责边界

- 需求域默认写入 `doc/2-需求/`，不要把需求文档散落到 `doc/`，也不要继续在需求域下混写实施和验收文档。
- 需求截图、需求原型截图和外部生成图片默认写入 `doc/2-需求/data/`，不要散落到仓库根目录或 `doc/`。
- 架构域默认写入 `doc/1-架构/`，不要把配置规范、后台结构方案和运行时设计混写到 `doc/2-需求/` 或 `doc/`。
- 架构域四个主入口固定使用 `1-4` 编号；业务链路从 `5` 开始使用递增编号。同一业务链路持续更新原编号文档，新增独立链路取当前最大业务链路编号加一，历史编号不复用、不重排。
- 实施域默认写入 `doc/3-实施/`，不要把实施总览、实施周期和阶段推进文档散落到 `doc/1-架构/`、`doc/2-需求/` 或 `doc/`。
- 验收域默认写入 `doc/7-验收/`，不要把验收标准或最终验收混写到 `doc/2-需求/`、`doc/5-tests/` 或 `doc/`。
- Bug 域默认写入 `doc/4-bugs/`，不要把 Bug 主记录混写到 `doc/5-tests/` 或 `doc/`。
- 测试域默认写入 `doc/5-tests/`，不要把测试证据散落到业务代码目录、仓库根目录或 `doc/`。
- 审查域默认写入 `doc/6-审查/`，不要把实现自审、归位结论、格式清理结果和语法校验结论散落到根级 `review/`、`analysis/`、仓库根目录或其他临时目录。
- 通用项目说明才进入 `doc/`；如果文档明显服务于单一需求、单一 Bug、单一测试任务或单一审查任务，应优先归回对应根目录。
- 项目专属 skill 统一放在 `skill/`，不要混放到根目录或 `doc/`。
- 项目级总体设计、架构总览、模块大纲优先写入根目录 `项目设计.md`，不要同时在根目录和 `doc/` 下各维护一份平行主入口。
