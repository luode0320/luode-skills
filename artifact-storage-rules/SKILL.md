# 研发产物存储规则

只在“研发过程产生的持久化产物应该放到哪里、叫什么、是否复用旧文档”这类问题上使用这个 skill。
如果当前争议是需求内容、Bug 根因、测试策略、生产代码目录归属或流程分流，请转交对应主域 skill。

## Skill 作用与适用场景

- 作为项目内研发产物目录和命名约定的单一真相源。
- 统一 `doc/1-架构/`、`doc/2-需求/`、`doc/3-实施/`、`doc/4-bugs/`、`doc/5-tests/`、`doc/6-审查/`、`doc/7-验收/`、`doc/`、`skill/` 和根目录 `项目设计.md` 等产物入口的职责边界。
- 统一架构域 `1-4` 四个有序中文主入口，以及从序号 `5` 开始依次下推的业务链路文档命名、索引和增量更新规则。
- 统一需求文档、验收标准、最终验收、实施总览、实施周期、Bug 根目录、测试任务目录、项目设计主入口和主入口 `README.md` 的命名模板。
- 统一活动产物时间前缀为 `YYYY-MM-DD_HHmmss`；需求、实施、Bug、测试、审查、验收均不得只使用 `YYYY-MM-DD` 日期前缀。
- 统一实施、审查、验收等下游产物必须保留来源对象标识：来源可以是需求，也可以是 Bug；文件名必须使用 `YYYY-MM-DD_HHmmss_来源对象标识_阶段或主题说明.md` 结构，且来源对象标识不重复前置时间戳，避免只看见阶段或审查主题而看不出来源对象。
- 统一需求域与 Bug 域的主入口产物完整性要求：需求入口与 Bug 入口都是单一 `md` 主文档（Bug 为 `README.md`），流程图与时序图统一用 Mermaid 语法直接写入正文，不另建独立图文件。
- 统一 Markdown 图片资产规则：所有需求、实施、验收及通用文档使用的位图统一落在 `doc/data/images/`，并按当前 Markdown 文件位置使用 `/` 分隔的相对路径引用；`doc/data/` 只作为数据总目录，不直接存放图片。
- 统一实现自审、目录归位、格式清理、语法检查和注释核对等审查结论的默认根目录为 `doc/6-审查/`；除非项目另有明确约定，不再把根级 `review/` 或 `analysis/` 当正式长期目录。
- 统一“同一需求是否持续更新同一文档”“同一 Bug 是否持续沿用同一根目录”“测试任务何时新建时间戳根目录”的规则。
- 统一最近上下文预热、历史回溯、需求域、Bug 域、测试域和交付域对这些目录的读取与引用方式。
- 减少路径规则散落在多个 skill 中导致的重复修改和遗漏风险。
- 固化“默认新建优先于复用旧目录”的决策优先级，并明确复用旧目录所需任务标识一致性证据。
- 当仓库级目录约定发生迁移时，强制同步项目入口文档、长期记忆、长期风格和仓库规则，避免只移动文件不更新引用。
- 当目录迁移已完成且用户未要求保留兼容入口时，默认删除旧目录、旧占位文件和旧跳转文档，不保留并行旧包。

## Markdown 图片资产契约

`doc/data/images/` 是项目内 Markdown 位图的唯一资产根目录。首次产生图片时创建 `doc/data/` 与 `doc/data/images/`；没有图片时不提交空目录。需求截图、UI/原型、视觉对比、真实产物和外观基线等需要真实视觉证据的场景必须生成或归档图片；流程、时序、状态、依赖和数据关系仍使用 Mermaid，图片不得替代 Mermaid 门禁。

- 生成链路必须调用真实 `imagegen`，并将输出目标显式指定为 `doc/data/images/<file>`；不得用程序绘图、占位图或 Base64 冒充生成结果。
- 文件名采用 `<document_stem>.<asset-slug>-v<number>.<ext>`；`asset-slug` 使用 ASCII kebab-case，返修递增版本号。
- 允许 `.png`、`.jpg`、`.jpeg`；`.webp` 仅在渲染兼容性已验证时使用。PNG 用于 UI、截图、文字密集图和信息图，JPEG 用于照片。
- Markdown 引用必须从当前文档位置计算相对路径并统一使用 `/`，例如 `![IMG-DOC-001 登录状态对比](../data/images/2026-07-12_040216_登录状态.login-state-v1.png)`。
- 相对路径映射固定为：`doc/2-需求/*.md`、`doc/3-实施/*.md`、`doc/7-验收/*.md` 使用 `../data/images/<file>`；`doc/*.md` 使用 `data/images/<file>`；仓库根目录 Markdown 使用 `doc/data/images/<file>`。
- 禁止绝对路径、反斜杠、`file://`、Base64/data URI、HTML `<img>`、外部热链、路径越界以及直接引用 `doc/data/<file>`。
- 目标文档必须声明“图片资产决策：需要”，或写 `N/A + 原因 + 证据`；声明需要但缺文件、缺引用、缺非空 alt 或缺 `IMG-*` 资产 ID 时不得交付。
- 资产清单登记 `IMG-*`、用途、来源、相对路径、版本、关联 `SRC/DEC/REQ/RULE/AC/CYCLE/TASK`、敏感状态和版权状态。同一图片可被多个文档引用；`doc/data/images/` 中没有任何 Markdown 引用的文件视为孤儿。
- 图片生成失败时，若场景标记为必需，任务必须阻断；不得静默降级为伪图。未要求兼容旧路径时，不保留历史需求域图片目录或 `doc/data/<file>` 入口。

## 测试目录复用优先级（写死边界）

- 默认新建目录优先级高于复用旧目录。
- 旧目录仅用于追踪同一任务的后续迭代，且需要明确任务标识一致。
- 没有任务标识一致性证据时，一律新建。

可以直接使用以下规则原文（保持一致）：

```text
默认新测试任务必须创建当天时间戳目录；禁止将新文件写入历史时间戳目录。
若检测到目标目录日期早于当前日期，必须中止并向用户确认是否复用历史目录。
未获明确确认前，禁止复用历史目录。
允许读取历史目录作为参考，但新增与修改应落在当前时间戳目录。
最终交付前必须校验新增测试文件路径全部位于当前任务时间戳目录。
```

## 自动触发信号

- 需要决定需求文档、Bug 记录、测试资源、项目说明文档、项目专属 skill 资产或根目录项目设计文档应该保存到哪个根目录或主入口。
- 需要调整 `doc/1-架构/`、`doc/2-需求/`、`doc/3-实施/`、`doc/4-bugs/`、`doc/5-tests/`、`doc/6-审查/`、`doc/7-验收/`、`doc/`、`skill/`、根目录 `项目设计.md` 等目录名称、层级或命名模板。
- 需要判断同一任务是否应继续更新原文档，还是新建新的任务目录。
- 需要给其他 skill 提供统一的目录、文件名和入口文件引用基准。
- 发现多个 skill 对相同产物的存放位置、命名格式或复用策略描述不一致。
- 需要为新 skill 建立产物落点规则，避免它再次在自己的 `SKILL.md` 里硬编码一套新路径。

## 进入后先做什么

1. 先确认当前问题属于“研发产物存储约定”，而不是需求内容、Bug 根因或代码目录归属问题。
2. 识别当前产物类型：需求、验收标准、最终验收、实施总览、实施周期、Bug、测试、项目设计、通用文档还是交付记录。
3. 默认先读 `references/path-map.yaml`，确认当前有效的根目录、命名模板和入口文件。
4. 再读 `references/root-directories.md`，确认每个根目录的职责边界。
5. 需要判断是否复用原文档、是否该新建目录时，再读 `references/update-policy.md`，并优先应用“默认新建 > 复用旧目录”。
6. 若目标是测试目录且检测到历史日期目录，先阻断并确认是否存在“同一任务标识一致 + 用户明确复用许可”；缺任一项则必须新建当天目录。
7. 需要给其他 skill 建立引用方式时，再读 `references/skill-integration.md`。
8. 若本轮属于目录迁移，继续检查仓库级 `AGENTS.md`、`README.md`、根目录 `项目设计.md`、`PROJECT_MEMORY.md`、`PROJECT_STYLE.md` 是否已同步新路径；未同步前不得判定迁移完成。

## 默认执行流程

1. 默认先读 `references/path-map.yaml`。
2. 再读 `references/root-directories.md`，确认目录职责和默认扫描范围。
3. 如果当前争议在命名模板、入口文件或目录结构，再读 `references/naming-templates.md`。
4. 如果当前争议在“更新原记录还是新建新记录”，再读 `references/update-policy.md`，优先执行“默认新建 > 复用旧目录”。
5. 对测试资产执行复用边界检查：若无任务标识一致性证据，一律新建；若目录日期早于当天，未获明确复用确认前禁止写入。
6. 如果当前争议在“其他 skill 应该怎么引用这里的规则”，再读 `references/skill-integration.md`。
7. 若本轮涉及仓库内目录迁移，必须同步更新仓库级 `AGENTS.md`、`README.md`、根目录 `项目设计.md`、`PROJECT_MEMORY.md`、`PROJECT_STYLE.md` 和受影响的主入口文档，确保读取入口与真实目录一致。
8. 若迁移后的新目录已稳定且用户未要求保留兼容层，删除旧目录、旧入口文件和旧跳转文档；不要保留新旧并行目录继续共存。
9. 输出统一的根目录、命名模板、入口文件和复用策略结论；架构域同时说明四个中文主入口与业务链路新增 / 更新规则。
10. 如果全局约定发生变化，优先修改本 skill 的 references，而不是在其他 skill 中四处复制新模板。

## 权责边界与不负责事项

- 只负责研发产物的存放约定，不负责生产代码的包结构、模块归属和依赖方向，那属于 `package-structure-rules` 或 `implementation-review-rules` 的测试前归位检查。
- 不负责需求本身是否完整、Bug 是否定位、测试是否通过、交付是否可上线。
- 不代替 `team-development-rules` 做流程分流或冲突裁决。
- 不允许其他 skill 再自行定义与这里冲突的根目录、命名模板或复用策略。
- 可以给其他 skill 提供引用基准，但不直接承担这些 skill 的主流程分析。

## 需要暂停并确认的条件

- 仓库本身已经存在明确且稳定的目录约定，与当前 skill 默认值冲突。
- 调整目录名称或层级会导致大范围迁移，已经超出当前任务的最小改动范围。
- 同一类产物已经在多个历史目录长期共存，无法直接判断哪个才是当前主入口。
- 不同团队或不同子项目确实需要不同约定，但还没有先划清作用范围。
- 测试资产准备写入历史时间戳目录，但无法证明“同一任务标识一致”或未获用户明确复用许可。
- 目录已经迁移，但仓库级 `AGENTS.md`、`README.md`、`项目设计.md`、`PROJECT_MEMORY.md`、`PROJECT_STYLE.md` 仍引用旧路径。

## 执行通过 / 驳回标准

- 通过：能给出某类产物唯一清晰的根目录、命名模板、主入口文件和复用策略，并且其他 skill 可以直接引用，不需要再各自定义一套；测试目录复用决策满足“默认新建优先、复用需任务标识一致且经明确确认”；若发生目录迁移，仓库级入口与长期记忆已同步到新路径，且旧目录已按用户要求清理。
- 驳回：仍然让多个 skill 分别维护相同路径规则，或同一类产物存在多个互相冲突的模板，没有唯一基准。

## 执行结果归档要求

- 日常执行不要求额外新建持久化文档；本 skill 自身的 references 就是正式规则载体。
- 一旦全局目录、命名或复用策略发生变化，必须优先更新 `references/path-map.yaml`，并同步更新相关说明文件。
- 如果其他 skill 需要配合调整，应优先把它们改成“引用本 skill”，而不是把新模板继续复制进去。
- 归档结果至少包含：当前有效根目录、命名模板、入口文件、复用策略和跨域引用规则。
- 若发生仓库级目录迁移，归档结果还必须包含：受影响主入口清单、已删除的旧目录清单，以及“是否保留兼容层”的明确结论。

## references 读取规则

- 默认先读 `references/path-map.yaml`。
- 只有在解释根目录职责时，再读 `references/root-directories.md`。
- 只有在解释命名模板时，再读 `references/naming-templates.md`。
- 只有在解释复用策略时，再读 `references/update-policy.md`。
- 只有在需要指导其他 skill 如何引用时，再读 `references/skill-integration.md`。
