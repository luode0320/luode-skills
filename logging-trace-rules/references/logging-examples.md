# 日志与追踪正反例

## 正例

### 正例 1：业务日志清楚

- 记录模块、动作、对象、结果。
- 出错时能快速知道是哪一步失败。
- 使用项目日志框架而不是控制台打印。
- 例如 Go 项目通过 `utils/log/` 封装调用：`log.Infof("########################### 创建订单开始: [%s]", req.OrderId)`。

### 正例 2：trace 能串起链路

- 入口请求和下游调用共享同一条 trace 主线。
- 关键失败点能关联到对应 trace。

### 正例 3：日志配置完整

- 日志级别、日志路径、文件滚动和输出目标来自配置文件。
- `enableLog2Stdout` 在后端服务默认配置为 `false`。

## 反例

### 反例 1：无意义日志

- “开始执行”“执行完成”“进入方法”这类没有上下文的信息大量刷屏。

### 反例 2：调试打印长期残留

- 临时变量和调试内容直接保留在正式日志里。
- 使用 `fmt.Println`、`println` 或 `console.log` 代替项目 logger。

### 反例 3：敏感信息裸打

- 用户隐私、密钥、完整证件号或 token 直接出现在日志中。

### 反例 4：日志配置缺失

- 在代码里硬编码日志级别和输出路径，未通过配置文件统一管理。
