# Bug 修复建议规则

只在根因已经足够明确、准备从“定位”进入“修改决策”时使用这个 skill。
如果当前根因还不稳定，先回到 `bug-root-cause-rules` 或 `bug-runtime-debug-rules`，不要提前谈修复方案。

## Skill 作用与适用场景

- 把定位结果整理成可执行的修改建议。
- 评估修改范围、风险点、回归影响和备选方案。
- 判断当前问题是否必须先等用户确认，还是可以直接进入编码。
- 作为 Bug 域到编码域的稳定交接层。

## 自动触发信号

- 根因已经基本明确，准备讨论如何改。
- 修改方案可能影响公共模块、共享逻辑、数据库行为或兼容性。
- 需要比较多个修复方向的风险和代价。
- 需要明确哪些情况应先给建议、等确认后再编码。

## 进入后先做什么

1. 先确认根因结论是否足够稳定。
2. 列出至少一个主修复方案和必要的影响面。
3. 如果存在明显替代路线，也列出备选方案。
4. 判断当前方案是否必须先经用户确认。

## 默认执行流程

1. 默认先读 `references/fix-proposal-template.md`，按统一模板组织修复建议。
2. 如果需要判断风险和影响范围，再读 `references/risk-assessment-checklist.md`。
3. 如果需要判断何时必须等确认、何时可以直接编码，再读 `references/confirm-before-coding.md`。
4. 输出主方案、备选方案、风险点、影响范围、验证方式和确认建议。
5. 在确认条件未满足前，不直接进入编码实施。

## 权责边界与不负责事项

- 只负责修复建议与交接，不负责重新定位根因。
- 不代替编码类 skill 直接实施代码修改。
- 不把“想到一个修复方式”直接等同于“方案已通过”。
- 不跳过风险说明和验证方式，直接给出模糊修复建议。

## 需要暂停并确认的条件

- 根因仍存在多个竞争假设。
- 修改范围会明显扩大到公共模块、共享接口或兼容行为。
- 方案会影响数据库结构、接口契约或已有流程。
- 存在两个以上方案且代价差异明显，需要用户决策。

## 执行通过 / 驳回标准

- 通过：能明确给出主方案、影响范围、风险点、验证方式，以及是否需要确认。
- 驳回：方案仍停留在口号层，如“改一下这里看看”，或完全不说明风险、范围和验证方法。

## 执行结果归档要求

- 将修复建议、影响范围、风险评估、确认结论记录到 `analysis/` 或 `review/`。
- 归档内容至少包含根因摘要、主方案、备选方案、风险点、验证方式和下一步动作。
- 如果明确结论是“可直接进入编码”，也要保留该判断依据。

## references 读取规则

- 默认先读 `references/fix-proposal-template.md`。
- 只有在评估影响范围和风险时，再读 `references/risk-assessment-checklist.md`。
- 只有在判断是否必须先确认时，再读 `references/confirm-before-coding.md`。
