# Go 后端安全规则（通用版）

## 适用范围

适用于使用 Go、标准库 `net/http` 或同类 HTTP 服务栈开发的后端服务。

这个参考文件用于两类场景：

- 写 Go 后端代码时，默认采用更安全的实现
- 审查 Go 后端代码时，优先扫描高影响安全问题

## 默认工作方式

- 所有请求输入都视为不可信，包括路径参数、查询参数、Header、Cookie、请求体、上传文件和来自外部系统的数据
- 审计结论必须基于证据，引用具体文件、配置和代码
- 如果某项安全能力可能由网关、反向代理或云平台提供，但代码里看不到，要明确标记为“代码中不可见，需运行时确认”

## 生产安全基线

- Go 版本和关键依赖保持更新
- 对外 HTTP 服务必须设置超时和 `MaxHeaderBytes`
- 敏感配置通过环境变量或外部配置注入，不能写死在代码里
- 诊断接口如 `pprof`、`expvar`、内部 metrics 不能直接暴露到公网
- 反向代理和转发头的信任关系必须显式配置
- 会话 Cookie 在 HTTPS 生产环境下必须带安全属性

## 核心规则

### GO-CONFIG-001：密钥必须外置，且不能进入日志

- 不要把 API Key、数据库密码、JWT 密钥、私钥硬编码进仓库
- 不要把完整凭证写进日志、报错或调试输出
- 配置缺失时应快速失败，不要默默回落到危险默认值

### GO-HTTP-001：HTTP 服务必须设置超时和头大小限制

- `ReadTimeout`、`WriteTimeout`、`IdleTimeout` 不能省略
- 面向公网的服务必须设置 `MaxHeaderBytes`
- 长连接、慢请求、大 Header 都要视为 DoS 面

### GO-HTTP-002：请求体和 multipart 解析必须做大小限制

- 不能无上限读取 Body
- 上传接口要限制单文件大小、字段数量和总大小
- 大文件或流式上传要配合网关限流和存储策略

### GO-PROXY-001：反向代理信任关系必须显式

- 只有在明确受信代理存在时才读取 `X-Forwarded-*`
- 不要默认信任客户端自己带来的转发头
- 真实 IP、协议和 Host 的获取逻辑要统一

### GO-COOKIE-001：生产 Cookie 必须带安全属性

- 生产 HTTPS 环境应启用 `Secure`
- 会话 Cookie 应启用 `HttpOnly`
- `SameSite` 应有明确策略，默认优先 `Lax`

### GO-CSRF-001：Cookie 认证的写操作必须防 CSRF

- 仅当浏览器会自动携带凭证时，CSRF 才是核心风险
- 使用 Cookie 认证的 POST、PUT、PATCH、DELETE 都要有 CSRF 防护
- 如果系统只使用 `Authorization` Header，则一般不需要额外 CSRF 机制

### GO-CORS-001：CORS 必须显式且最小授权

- 不要把 CORS 当成认证机制
- 不要对带凭证请求配置全开放 Origin
- 允许的域名、方法和 Header 都要最小化

### GO-TEMPLATE-001：模板必须保持自动转义

- HTML 输出优先使用安全模板能力
- 不能把不可信字符串直接拼成 HTML 返回
- 不要从用户输入动态加载或执行模板

### GO-FILE-001：文件读取和下载必须防路径穿越

- 不要把用户输入直接当文件路径
- 文件下载优先使用服务端映射 ID，而不是直接路径
- 用户上传文件不要作为活动内容直接内联暴露

### GO-UPLOAD-001：上传文件必须校验、隔离和安全提供

- 校验扩展名不足够，还要校验 MIME 或内容特征
- 服务端生成文件名，不信任用户原始文件名
- HTML、SVG、JS 等活动内容默认按附件下载处理

### GO-INJECT-001：数据库访问必须参数化

- 禁止用字符串拼接 SQL
- 即使用 ORM，也要检查是否绕开了参数化能力
- 对排序字段、筛选字段等动态输入要走白名单

### GO-CMD-001：避免命令注入

- 能用库就不用 shell
- 使用 `exec.Command` 时参数分离，不要拼接整条命令
- 不要把用户输入传给 shell、PowerShell、`sh -c`、`bash -c`

### GO-SSRF-001：外部请求必须防 SSRF

- 用户可控 URL 一律高风险
- 只能访问允许的协议和目标域
- 禁止访问本机、内网地址、链路本地地址和云元数据地址
- HTTP 客户端必须设置超时，并谨慎跟随重定向

### GO-REDIRECT-001：跳转地址必须校验

- `next`、`redirect`、`return_to` 等参数只能跳相对路径或白名单域名
- 校验失败时应回退到安全默认页

### GO-CRYPTO-001：随机数和密码存储必须使用安全实现

- 安全随机数使用 `crypto/rand`
- 密码存储使用 bcrypt 或 argon2id
- 比较敏感值时优先使用安全比较函数

### GO-DIAG-001：调试和诊断接口不能对公网开放

- `pprof`、`expvar`、调试端口、开发辅助接口仅限内网或受保护访问
- 诊断接口暴露出来通常意味着系统细节泄漏

## 审计重点

- 是否存在无超时 HTTP 服务或 HTTP 客户端
- 是否存在字符串拼接 SQL、命令执行、危险模板或任意文件路径
- 是否记录了密钥、令牌或完整用户敏感数据
- 是否存在对外暴露的 `pprof`、metrics、调试接口
- 是否把用户可控 URL 直接交给 `http.Client`
- 是否错误信任转发头、Cookie 或 Host
