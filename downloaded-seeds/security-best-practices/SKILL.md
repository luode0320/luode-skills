# 安全最佳实践种子 Skill

## 概览

这个 skill 用于两类工作：

1. 在新代码生成时默认采用更安全的实现方式。
2. 在已有代码中识别高风险安全问题，并在需要时输出审计结论或修复建议。

## 使用流程

第一步先识别当前仓库或任务中涉及的全部语言和核心框架。

- 如果是 Web 项目，要同时识别前端和后端技术栈。
- 如果仓库里同时存在 React 和 FastAPI，就要同时读取对应的前端和后端安全参考。
- 如果只确定语言，不确定框架，也要先读取对应语言的一般安全规则文件。

## 参考文件选择规则

参考文件命名格式为：

- `<language>-<framework>-<stack>-security.md`
- `<language>-general-<stack>-security.md`

选择原则：

- 有明确框架时，优先读取框架专用文件。
- 同时存在前后端时，两边都要读。
- 没有完全匹配的文件时，再退回到通用语言常识，不要凭空假设项目具备某些安全设施。

## 工作模式

### 1. 安全默认生成模式

当任务是写新代码或修改已有代码时：

- 必须优先选择安全默认实现。
- 必须避免引入新的高风险危险点，例如命令执行、危险模板渲染、不安全文件处理、过度开放的跨域、把密钥放进客户端等。
- 如果项目已有明确约束与最佳实践冲突，应说明风险后遵循项目约束，不要无条件硬改。

### 2. 被动审查模式

当你在仓库中工作，但用户没有明确要求做安全扫描时：

- 仍然要对接触到的代码保持基本安全警觉。
- 只主动指出高影响或明显违反安全基线的问题。
- 给出简短原因和安全修复方向即可，不必每次都出完整报告。

### 3. 主动审计模式

当用户明确要求“安全扫描”“安全审计”“生成安全报告”时：

- 系统性搜索代码库中的高风险问题。
- 输出结构化 findings，按严重程度排序。
- 每条 finding 必须给出规则编号、文件位置、证据、影响和修复建议。

## 报告要求

生成完整报告时：

- 报告文件默认写入 `security_best_practices_report.md`
- 报告开头要有简短摘要
- 按严重程度分组
- 每个问题都要带编号，便于后续跟踪
- 引用代码时必须给出文件路径和行号

## 修复原则

修复安全问题时：

- 一次优先修一个问题，不要把无关问题混到一个提交里
- 优先做最小安全改动，避免破坏项目行为
- 修复后必须结合项目原有测试或补充验证
- 如果修复会影响现有逻辑，先明确告知用户风险和兼容影响

## 通用安全原则

- 不要请求、输出、记录、提交密钥、密码、私钥、签名密钥、带凭证的数据库连接串
- 不要用“关闭保护”来假装修安全问题
- 不要把 CORS 当成认证
- 使用 Cookie 认证时，要考虑 CSRF
- 前端永远不能作为最终授权边界
- 不要在对外 ID 中使用容易遍历的自增 ID
- 生产环境中有关 TLS、Secure Cookie、HSTS 等配置要结合真实部署条件判断，不要脱离环境生搬硬套
