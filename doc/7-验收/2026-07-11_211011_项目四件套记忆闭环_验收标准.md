# 验收标准：项目四件套记忆闭环

## 1. 验收目标

- 新项目或新线程按“父目录公共规则 -> current -> memory”启动。
- 缺失项目记忆文件时能创建最小 UTF-8 模板。
- current 覆盖式维护且不超过 51,200 字节。
- memory 只保留稳定规则和关键决策。
- history 只追加，普通启动不读取。
- Obsidian vault 仍由 CLI 限定在 `D:\obsidian_data`，不使用文件系统 fallback。

## 2. 验收场景

| 场景 | 通过标准 |
| --- | --- |
| 新项目初始化 | 三个项目记忆文件自动创建，规则文件包含启动顺序 |
| 幂等复跑 | 已有内容不被破坏，重复执行不产生重复区块 |
| 当前状态 | 覆盖式更新；51,200 字节通过，超限阻断 |
| 稳定记忆 | 只更新规则、决策和长期事实，不混入流水 |
| 历史流水 | 旧事件保持不变，新事件追加，默认读取路径不包含 history |
| Obsidian | `version`、vault 定位、search/read 通过；CLI 失败时阻断 |
| 生成物 | 字典由生成脚本刷新，frontmatter 和 UTF-8 检查通过 |

## 3. 失败条件

- 启动时直接读取 history 或跳过 current/memory。
- 超限时静默截断 current。
- 用 vault CLI 代替项目本地文件读写，或反过来用文件系统读写 vault。
- 规则、skill、模板和 bootstrap 对四件套职责描述不一致。
