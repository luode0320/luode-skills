# Bug 回归风险镜像规则

## 目标

只检查本轮 Bug 修复是否满足 `bug-regression-risk-rules` 的风险识别标准，不执行修复。

## 源规则

以下判定严格沿用源 skill：

- 识别修复动作可能波及的旧能力、共享依赖和兼容路径
- 给风险点排序，帮助决定验证优先级
- 区分局部风险、链路风险和公共能力风险
- 将回归风险结论沉淀到当前 Bug 根目录
- 修复涉及公共方法、复用组件、共享配置、基础库或中间件时必须检查
- 修复会改变原有接口返回、数据写入、缓存逻辑或异常处理时必须检查

## 自动触发信号

- 修复涉及公共方法、复用组件、共享配置、基础库或中间件。
- 修复会改变原有接口返回、数据写入、缓存逻辑或异常处理。
- 团队担心某个修复会带来兼容性或历史行为回退。
- 需要在修复实施前先决定回归重点。

## 进入后先做什么

1. 先列出修复直接改动点和它们的复用范围。
2. 为当前 Bug 确认统一的根目录。
3. 识别会被牵动的旧流程、旧接口、共享逻辑和数据行为。
4. 按影响范围和破坏成本给风险分级。
5. 形成验证优先级和需要重点回看的链路。

## 默认执行流程

1. 先读取 `bug-regression-risk-rules`。
2. 再按源 skill 的默认流程继续读对应 references。
3. 输出回归风险清单、风险等级和验证优先级，并更新到当前 Bug 根目录下的 `README.md`。
4. 风险识别完成后回流 `bug-fix-proposal-rules` 或交给 `test-regression-rules`、`bug-validation-rules` 落实验证。

## 需要暂停并确认的条件

- 修复方案本身还没定，风险无法可靠评估。
- 影响范围和复用范围证据不足，暂时无法分级。
- 高风险路径没有任何验证条件或环境支持。
- 团队试图跳过风险识别直接合入高影响改动。

## 执行通过 / 驳回标准

- 通过：能够列出主要风险点、解释影响范围，并明确哪些路径必须优先验证、哪些风险可记录后续补看。
- 驳回：泛泛而谈“可能有风险”，没有具体风险对象、分级和验证重点。

## 执行结果归档要求

- 将回归风险清单和分级结论统一记录到当前 Bug 根目录中。
- 当前 Bug 根目录必须包含 `README.md`，至少写明改动点、影响对象、风险等级和建议验证路径。
- 如果后续落实了回归测试，应在当前 Bug 根目录中补充对应测试目录或验证记录路径。
- 如最终接受已知风险，应保留理由和补救安排。

## 输出格式

- 写出回归风险点、风险等级和验证优先级。

## 读取规则

- 默认先读 `bug-regression-risk-rules`。
- 需要做风险分级、根目录或正反例判断时，再按源 skill 的默认流程继续读对应 references。
