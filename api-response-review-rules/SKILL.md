# 接口响应检查镜像规则

## 目标

只检查当前响应模型是否满足 `api-response-rules` 的响应结构标准，不执行修改。

## 源规则

以下判定严格沿用源 skill：

- 成功响应和错误响应都必须包含状态码、状态、消息、数据四个字段
- 响应解析必须使用明确 DTO
- 禁止长期使用 map / any / RawMessage 作为响应入口
- 兼容字段和版本字段边界要清晰

## 自动触发信号

- 新增或修改返回体、统一响应包装器。
- 新增或修改错误响应结构、分页结构、兼容字段或版本字段。
- 准备使用 `map[string]interface{}`、`any`、`json.RawMessage` 直接解析响应体。

## 进入后先做什么

1. 先确认接口入口和请求结构已经稳定。
2. 再判断当前返回是成功结果、错误结果还是分页结果。
3. 明确业务数据、元信息、错误信息和兼容字段的层次。
4. 确认响应 DTO 结构并完成字段映射。

## 默认执行流程

1. 先读取 `api-response-rules`。
2. 如需判断响应边界或正反例，再读源 skill 的对应 references。
3. 输出成功响应、错误响应、分页结构和兼容字段放置结论。
4. 不直接改代码，只输出缺口。

## 权责边界与不负责事项

- 不代替 `error-handling-rules` 设计异常处理流程。
- 不代替 `api-endpoint-rules` 处理入口职责和 HTTP 语义。
- 不代替 `api-request-rules` 设计请求模型。
- 不代替 `api-swagger-rules` 维护成功响应、错误响应、分页响应和示例。

## 需要暂停并确认的条件

- 当前接口是否需要统一包装仍不清楚。
- 兼容字段和版本字段会影响既有客户端解析。
- 错误响应结构调整会波及大量旧调用方。
- 响应层已经无法单独收口，问题实质变成了错误处理机制设计。

## 执行通过 / 驳回标准

- 通过：能明确成功 / 错误 / 分页响应结构及兼容字段放置方式，且边界稳定。
- 驳回：返回结构层次混乱、错误信息散落、分页元信息无统一位置，或响应解析仍为 map/any/RawMessage。

## 执行结果归档要求

- 如果本次定义了统一响应结构、修改了错误响应或兼容字段策略，将结论记录到 `analysis/` 或 `review/`。
- 归档内容至少包含响应目标、结构结论、成功响应字段、错误响应字段、兼容影响和调用方风险。
- 如果只是沿用现有清晰结构且无争议，可以不单独归档。

## 读取规则

- 默认先读 `api-response-rules`。
- 只有在处理错误、分页和兼容字段时，再读源 skill 的对应 references。
