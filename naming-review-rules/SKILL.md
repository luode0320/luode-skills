# 命名检查镜像规则

## 目标

只检查当前命名是否满足 `naming-rules` 的命名标准，不执行重命名。

## 源规则

以下判定严格沿用源 skill：

- 业务术语和仓库现有叫法是否一致
- 默认使用驼峰命名族
- 缩写边界、语义粒度和职责准确性
- 避免同一概念多套叫法并存
- 避免历史别名、歧义命名和误导性名字

## 自动触发信号

- 新增或修改变量、函数、类、模块、接口、字段、事件、任务名、测试名或配置项。
- 同一概念在仓库里出现多个叫法。
- 实现自审或代码评审指出命名不清晰或不一致。
- 不确定当前名字该用什么命名形态。

## 进入后先做什么

1. 先确认这个名字指向的对象、职责和生命周期。
2. 查已有代码和文档里同一概念是否已有稳定叫法。
3. 判断名称粒度是否和职责匹配。
4. 判断是否需要同步调整相邻命名。

## 默认执行流程

1. 先读取 `naming-rules`。
2. 需要对齐业务术语或跨层命名口径时，再读源 skill 的对应 references。
3. 需要对照命名样例或判断歧义时，再读源 skill 的对应 references。
4. 输出命名建议、术语对齐结论和需要同步调整的命名项。
5. 不直接改代码，只输出命名缺口和建议。

## 权责边界与不负责事项

- 不代替 `code-readability-rules` 做整体结构可读性判断。
- 不代替 `code-style-consistency-rules` 处理格式和风格细节。
- 不替代源 skill 的实际重命名工作。
- 不为了统一命名而扩大改动范围。

## 需要暂停并确认的条件

- 对象职责本身不清，命名无法稳定。
- 仓库中已有多个历史术语且暂时无法统一。
- 命名调整会引发大范围无关重构。
- 新增名字与产品、业务、数据库或接口文档术语明显冲突。

## 执行通过 / 驳回标准

- 通过：命名能够准确表达对象职责，与现有术语系统一致，并且不会引入新的歧义或历史别名冲突。
- 驳回：命名仍然抽象、模糊、缩写过度，或同一概念出现多套叫法并存且没有给出统一方案。

## 执行结果归档要求

- 如命名调整涉及公共术语或跨文件同步，应将结论记录到 `analysis/` 或评审记录。
- 归档内容至少包含旧名、新名、调整原因和影响范围。
- 如果只是局部一次性命名选择，可不单独归档，但应保证评审可追溯。

## 读取规则

- 默认先读 `naming-rules`。
- 需要对齐业务术语或跨层命名口径时，再读源 skill 的对应 references。
- 需要对照命名样例或判断歧义时，再读源 skill 的对应 references。
