# 需求边界审查镜像规则

## 目标

只检查当前边界判断是否满足 `requirement-boundary-rules` 的要求，不执行需求主文档回写。

## 源规则

以下判定严格沿用源 skill：

- 区分当前需求、需求变更和历史问题
- 明确本次允许改动的层、模块和旧逻辑
- 识别兼容性影响面和上下游影响面
- 边界结论应回到同一份需求主文档
- 边界变化应同步图形化表达，避免图文口径漂移
- 不把历史遗留缺陷混入当前需求一起推进

## 自动触发信号

- 用户说“顺手一起改一下旧逻辑”。
- 实施中暴露历史问题，不确定是否并入本次需求。
- 验收不通过，但不确定是理解偏差还是实现缺陷。
- 当前需求可能影响旧接口、旧页面、旧流程或兼容行为。
- 主线程已声明使用 `requirement-boundary-rules`，但边界收口风险较高。

## 进入后先做什么

1. 先确认当前任务是否已经列出本次明确目标。
2. 再确认是否列出顺带发现的问题、旧逻辑或兼容风险。
3. 检查这些项是否被正确归类为当前需求、需求变更或历史缺陷。
4. 检查边界结论是否准备回到同一份需求主文档。
5. 检查是否存在图文不同步风险。

## 默认执行流程

1. 先读取 `requirement-boundary-rules`。
2. 需要核对边界检查项时，再读源 skill 默认要求的 `references/boundary-checklist.md`。
3. 需要核对历史问题与需求变更区分时，再读源 skill 的 `references/history-vs-change.md`。
4. 需要核对验收偏差归因时，再读源 skill 的 `references/acceptance-routing-examples.md`。
5. 输出边界通过项、归类错误、兼容性遗漏和回写缺口。
6. 不直接改文档，只输出问题和建议。

## 权责边界与不负责事项

- 不代替 `requirement-gap-rules` 补齐缺失信息。
- 不代替 Bug 域做根因定位。
- 不代替 `acceptance-criteria-rules` 制定完整验收标准。
- 不执行需求主文档回写。
- 不因为可一起修改就放宽边界标准。

## 需要暂停并确认的条件

- 当前问题既像需求变更又像历史缺陷，无法稳定归类。
- 兼容性调整会明显扩大改动范围。
- 用户要求同时处理多个历史问题，但是否并入当前任务尚未确认。
- 验收不通过，但需求理解偏差和实现缺陷都无法排除。

## 执行通过 / 驳回标准

- 通过：能够明确区分当前需求、需求变更和历史问题，边界、排除项、兼容性影响面和流转路径完整，且没有发现回写口径偏差。
- 驳回：边界说不清，历史问题与当前需求混推，或把需求偏差直接误判成 Bug。

## 执行结果归档要求

- 如本轮审查发现边界归类、兼容性影响面或文档收口缺口，应将结论记录到 `analysis/` 或评审记录。
- 归档内容至少包含边界对象、发现的问题、建议流转方向和影响面。
- 如果只是一次轻量预审且无争议，可不单独归档，但应保证评审可追溯。

## 读取规则

- 默认先读 `requirement-boundary-rules`。
- 只有在核对边界检查项时，再读 `references/boundary-checklist.md`。
- 只有在区分历史问题和需求变更时，再读 `references/history-vs-change.md`。
- 只有在核对验收偏差归因时，再读 `references/acceptance-routing-examples.md`。
