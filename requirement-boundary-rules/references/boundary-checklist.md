# 需求边界检查清单

## 用途

用于明确当前需求到底覆盖哪些内容，不覆盖哪些内容。

## 检查项

- 当前需求直接目标是什么。
- 本次允许改动哪些模块、接口、页面或流程。
- 本次不允许顺手处理哪些历史问题。
- 本次是否涉及兼容性调整、上下游联动或旧数据影响。

## 判断原则

- 直接服务于当前目标的，优先纳入。
- 只是顺带暴露出来、且不属于当前目标的，优先拆出。
- 只要会明显扩大改动范围，就需要单独确认。
- 边界判定结论必须更新到当前需求对应的同一份 `doc/2-需求/` 文档中。

## 极致完整性边界矩阵（强制）

每个边界结论必须使用唯一 `BOUND-*`，并填写：对象/模块、In Scope、Out of Scope、允许修改层、禁止修改层、输入输出、上游下游、权限、兼容策略、数据影响、性能/安全影响、责任人、证据、回滚路径和验收回指。

| 结论 | 必须同时记录 | 未满足时 |
| --- | --- | --- |
| 纳入本次需求 | 目标、理由、`REQ-*`、实现落点、`AC-*`、测试证据 | 不得进入实施任务 |
| 排除本次需求 | 排除理由、替代入口、后续 owner、风险与复核时间 | 标记为 `unresolved_decisions` |
| 转 Bug/历史问题 | 复现证据、历史版本、Bug ID、与当前需求的隔离边界 | 不得混入需求实现 |
| 兼容性影响 | 旧行为、新行为、迁移/回滚、版本窗口、验证样本 | `P0/P1` 阻断 |
| 图片资产边界 | 是否属于当前需求、`doc/data/images/` 资产根、`IMG-*` 关联、共享引用、敏感/版权状态、删除条件 | 缺路径、消费者或授权即 `blocked` |

### 边界收口闸门

1. 每个功能点只能有一个归属：当前需求、明确排除项、独立变更、Bug 或历史问题；重复归属和空白归属均阻断。
2. 边界变化必须同步更新正文、范围表、Mermaid 流程/时序图、拆分矩阵、验收映射和实施周期；只改段落不改图表视为未收口。
3. 未得到授权的默认兼容、权限、数据迁移或回滚方案不得写成确定结论；按 `N/A + 原因 + 证据` 或 `blocked` 处理。
4. 图片只允许进入 `doc/data/images/`，Markdown 必须使用当前文档位置计算的 `../data/images/<file>` 相对路径；绝对/远程/Base64/HTML/反斜杠/越界路径以及旧 `doc/data/<file>`、历史需求域图片目录均不属于允许范围。
