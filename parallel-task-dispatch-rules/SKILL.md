# 并行任务分发规则

## 目标

在正式执行前，将当前工作明确划分为“可并行”“条件并行”“必须串行”三类，并优先按文件集、模块边界、职责边界、独立问题或独立证据来源拆分。
并行判定不是终点；只要判定结果允许并行且没有阻断条件，就必须继续推进到真实子线程启动。

## 核心判定规则

1. 当任务只读，或每个线程都能拥有完全不重叠的写入文件集时，优先并行。
2. 当任务彼此相关，但能按子目录、模块或审查面稳定切开时，判定为条件并行。
3. 当多个任务必须修改同一批文件、边界不清晰，或后一步强依赖前一步产出时，必须串行。
4. 不允许两个并行线程同时写同一个文件。
5. 先做边界定义，再做线程分发；不要把定边界工作直接拆并行。
6. 审查类 skill 默认优先并行，除非其输出会直接写入同一批文件或依赖前一步产出。
7. 若判定为可并行或条件并行，且当前环境支持 subagent / multi-agent / thread 能力，必须立即联动 `subagent-dispatch-rules` 发起真实并行执行；只输出“并行技能”但未实际启动，不算并行已生效。
8. 若原计划可并行但最终未真实启动，必须明确回退为串行或本地执行，并写清回退原因；不得把“计划并行”伪装成“已经并行”。
9. 并行判定不以 `existing-skill-mapping.md` 为白名单；映射表只提供常见示例，主 agent 必须基于当前任务结构自行识别新的可并行子任务。
10. 项目分析、找 Bug、需求完善侦察、资料/日志/调用链收集等任务，只要能拆成独立问题或独立证据源，默认先尝试并行侦察；主 agent 保留最终判断、归纳和冲突裁决职责。
11. 单一根因、接口契约、schema、架构决策等需要统一裁决的主路径必须串行；但可委派子 agent 并行收集调用方、日志、复现样例、文档证据或影响面清单。

## 典型分类

- 可并行：
  - 多目录 / 多模块项目结构盘点
  - 独立资料源或官方文档检索
  - 只读代码库侦察
  - 代码规范检查
  - 注释补充
  - 格式清理
  - import 清理
  - 按独立文件集拆分的 lint 修复
  - 按独立模块拆分的单元测试补充
  - 按独立区域拆分的文档更新
- 条件并行：
  - 项目分析中的模块职责、入口链路、配置/部署面并行侦察
  - 需求完善中的代码现状、文档现状、外部资料、竞品/API 资料并行侦察
  - Bug 分诊中的日志分析、复现路径、近期改动、影响面并行收集
  - 按模块拆分的功能开发
  - Bug 分诊与定向验证
  - 文件归属清晰前提下的 review 与 cleanup
  - 接口已冻结前提下的前后端并行开发
- 必须串行：
  - 单一问题的根因裁决
  - 下游实现前的接口或 schema 变更
  - 重写共享抽象的重构任务
  - 任何最终会汇聚到同一批文件的任务
  - 需求边界、契约、schema 或架构方向尚未冻结前的最终决策

## 分发流程

1. 先识别任务类型和可能涉及的文件集。
2. 再按独立输出面、独立问题或独立证据来源分组。
3. 为每个线程标明文件归属。
4. 若判定结果为可并行或条件并行，先生成线程拆分方案，再立即联动 `subagent-dispatch-rules` 发起真实并行执行。
5. 若本轮确实启动了并行线程，必须在中间进度显式打印本次“并行技能”列表，明确哪些 skill 已按并行方式启动。
6. 只有边界明确后才允许启动并行线程。
7. 启动后必须核对“计划线程数”和“实际启动线程数”是否一致；不一致时立即回退或补齐说明。
8. 最终由一个主线程统一收口并处理冲突。

## 输出要求

分发时必须明确写出：

- 并行判定结果（可并行 / 条件并行 / 必须串行）
- 哪些任务可以并行
- 哪些任务必须串行
- 每个线程负责的文件或目录
- 最终由哪个线程收口合并
- 若存在并行 skill，必须追加一行：
  - `并行技能:<skill1,skill2,...>`
- 若本轮命中了 `parallel-task-dispatch-rules` 但判定最终不启动任何并行 skill，也必须明确写：
  - `并行技能:无`
- 若判定可并行或条件并行，还必须补充：
  - `子线程启动:<已启动N个/计划N个/未启动>`
  - `回退原因:<仅在未全部启动时填写>`

## 与 subagent-dispatch-rules 的联动闸门

- `parallel-task-dispatch-rules` 负责回答“能不能并行、怎么拆”。
- `subagent-dispatch-rules` 负责回答“是否真的启动了子线程、启动了几个、分别做什么”。
- 只要本 skill 判定结果不是“必须串行”，且不存在高风险阻断，就必须继续联动 `subagent-dispatch-rules`。
- 若当前环境没有真实 subagent / multi-agent / thread 能力，本 skill 必须显式写明“环境不支持真实并行执行”，并把结果回退为串行或条件并行未启动；不得仅靠文本模拟启动。

## 执行通过 / 驳回标准

- 通过：已明确并行判定、线程边界、文件归属、收口线程；若允许并行且环境支持，已真实联动 `subagent-dispatch-rules` 启动子线程，并给出计划线程数与实际启动数。
- 驳回：只输出“并行技能”或线程分配文案，但没有真实启动；或明明允许并行且环境支持，却没有继续联动 `subagent-dispatch-rules`；或计划线程数与实际启动数不一致却没有回退说明。
