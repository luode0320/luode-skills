# 并行任务分发规则

## 目标

在正式执行前，将当前工作明确划分为“可并行”“条件并行”“必须串行”三类，并优先按文件集、模块边界或职责边界拆分。

## 核心判定规则

1. 当任务只读，或每个线程都能拥有完全不重叠的写入文件集时，优先并行。
2. 当任务彼此相关，但能按子目录、模块或审查面稳定切开时，判定为条件并行。
3. 当多个任务必须修改同一批文件、边界不清晰，或后一步强依赖前一步产出时，必须串行。
4. 不允许两个并行线程同时写同一个文件。
5. 先做边界定义，再做线程分发；不要把定边界工作直接拆并行。
6. 审查类 skill 默认优先并行，除非其输出会直接写入同一批文件或依赖前一步产出。

## 典型分类

- 可并行：
  - 代码规范检查
  - 注释补充
  - 格式清理
  - import 清理
  - 按独立文件集拆分的 lint 修复
  - 按独立模块拆分的单元测试补充
  - 按独立区域拆分的文档更新
- 条件并行：
  - 按模块拆分的功能开发
  - Bug 分诊与定向验证
  - 文件归属清晰前提下的 review 与 cleanup
  - 接口已冻结前提下的前后端并行开发
- 必须串行：
  - 单一问题的根因分析
  - 下游实现前的接口或 schema 变更
  - 重写共享抽象的重构任务
  - 任何最终会汇聚到同一批文件的任务

## 分发流程

1. 先识别任务类型和可能涉及的文件集。
2. 再按独立输出面分组。
3. 为每个线程标明文件归属。
4. 只有边界明确后才允许启动并行线程。
5. 最终由一个主线程统一收口并处理冲突。

## 输出要求

分发时必须明确写出：

- 哪些任务可以并行
- 哪些任务必须串行
- 每个线程负责的文件或目录
- 最终由哪个线程收口合并
