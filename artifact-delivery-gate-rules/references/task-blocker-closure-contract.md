# 任务阻断收口契约

## 1. 目的

本契约规定任务无法继续推进时的唯一交接格式。它让最终回复、审查报告、最终验收和验证结果共享同一条 `BLK-*` 事实，而不是只写“任务被阻断”或给出彼此矛盾的后续建议。

本契约只在真实任务阻断时启用：文档 `status: blocked`、审查结论为“阻断”，或最终验收结论为“不通过/待重验”。`limited`、`not_applicable` 和正常通过仍使用各自既有语义，不得误报为任务阻断。

## 2. 唯一记录与去重

- 每个根因只能创建一个 `BLK-*` 记录，格式为 `BLK-<来源或领域>-<序号>`。
- 去重键固定为“根因错误码 + 关联证据”；同一根因在审查、验收或验证阶段重复出现时引用已有记录，不重复编写解决计划。
- 任务状态只能是 `blocked` 或 `manual_handoff`。两者在最终回复中都必须明确说明“任务已阻断”。

## 3. 正文模板

真实阻断文档必须包含以下章节和字段。字段内容必须落在当前文档，不得以聊天记录、链接标题或“见上文”替代。

```markdown
## 任务阻断收口

- 阻断记录 ID: `BLK-<领域>-<序号>`
- 任务状态: `blocked` 或 `manual_handoff`
- 阻断阶段: 实现自审、总审查、最终验收、功能验证、Bug 验证、执行失败或运行时恢复
- 阻断依据与证据: 明确根因、稳定错误码，以及命令、文档或证据 ID
- 已尝试与停止边界: 已执行的恢复动作与停止原因；没有可安全执行的动作时写 `N/A + 原因: ... + 证据: ...`
- 影响: 明确不能宣布完成、不能放行或不能进入的阶段
- 解决计划: 1. 责任方: ...；前置条件: ...；动作: ...；完成判据: ...；验证入口: ...
- 恢复后重入点: 原测试、复审、重验或恢复后的最小健康检查
- 去重键: 根因错误码 + 关联证据
```

解决计划最多三步；每步必须可由当前 owner Skill 或人工角色执行，且要写清完成判据和验证入口。需要权限、外部依赖、不可逆操作或超出原任务授权时，计划必须停止在 `manual_handoff`，不能假装已经恢复。

## 4. 机器校验与最终回复

- `validate_engineering_docs.py` 在 `status: blocked`、审查结论“阻断”、最终验收结论“不通过/待重验”时强制检查“任务阻断收口”的全部字段。
- 缺少章节或字段时输出 `blocker.closure_missing` 或 `blocker.closure_invalid`，并阻断文档放行。
- 最终回复由 `reasoning-summary-structure-rules` 消费当前 `BLK-*` 记录；该 Skill 负责展示，不重新定义字段或生成第二套解决计划。
