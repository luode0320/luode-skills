# 白话文档与附录分层契约

## 1. 适用范围

本契约适用于需求、验收、实施计划、实施记录、Bug、测试、审查、最终验收、架构、项目设计、交付和工作报告。它不减少原有的决策、追踪、测试和证据要求，只改变信息出现的位置和写法。

## 2. 给人读的正文

正文默认写给不熟悉研发术语的业务读者。读完正文后，读者应能回答：为什么做、影响谁、这次做什么和不做什么、结果会怎样、怎样算完成、审查和验收是否适用。

正文保留当前文档在 `HEAD` 中的标题、层级和顺序。白话开场放在文档 H1 后、首个 H2 前，用一段短文先说清结论、影响和是否可以继续。文档原有的技术章节继续保留原位置，不因为白话化被强行改名或搬走。

白话开场使用短句和常用中文。必要的技术词第一次出现时必须紧跟一句解释。开场不得出现稳定 ID、机器状态、命令、日志、SQL、接口报文、文件行号、追踪矩阵或代码块。技术章节可以保留这些精确内容。

## 3. 两类附录

已有 `执行附录`、`追踪附录` 时保留原名，不改名、不合并、不强制补建。它们分别继续承载执行细节和机器追踪信息。原结构确实没有合适的技术承载位置时，才允许在文档末尾新增唯一的 `附录`；`附录` 不能出现在文档中间，也不能重复出现。

## 4. 审查与验收适用性

正文可以说明本次审查和验收的结论；具体适用性和放行判定以 [审查与验收条件化门禁契约](review-acceptance-gate-contract.md) 的 `review_acceptance_gates` 为准。不适用必须写原因和依据，受限必须写替代验证、人工补验方式和通过标准。

## 5. 新模板元数据

新建或迁移到本契约的文档在 YAML front matter 中增加：

```yaml
reader_level: business_general
writing_style: plain_chinese
appendix_policy: preserve_existing_or_one_terminal_appendix
review_acceptance_gates: []
```

历史文档不因缺少这些字段被要求批量重写；它们在再次修改时迁移。

## 6. 自检问题

1. 不看附录，普通业务读者能否说清变化和完成标准？
2. 执行人员进入附录后，能否不靠猜测完成操作和验证？
3. 不适用的审查或验收是否已经说明原因和依据，而没有被写成阻断？
4. 白话开场是否独立可读，且没有把机器字段、命令或日志塞进去？
5. `review_acceptance_gates` 是否说明了不适用、受限和正式放行的真实边界？
