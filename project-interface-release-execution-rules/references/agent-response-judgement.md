# Agent 接口响应判定规则

本文件定义agent如何独立判断接口响应是否符合预期，不需要人工参与，所有判定必须有明确理由。常规接口结论为通过 / 不通过 / 待确认；存在参数依赖或环境阻断时，必须追加结构化阻断分类，避免把前置失败误判成目标接口失败。

## 判定前置要求
1. 接口请求参数必须输出为JSON字符串，所有敏感字段必须脱敏（如userId、token、手机号、身份证号、银行卡号用***代替）。
2. 接口简要响应必须输出为JSON字符串，只保留参与判定的关键字段，完整响应落盘到证据文件中，不输出到主报告。
3. agent必须完全基于接口语义、业务预期、响应结构、状态变化来判断，不能依赖固定响应值。
4. 判定必须独立给出，不能参考人工判断结果。
5. 测试数据必须按 `test-data-construction-rules.md` 的统一参数来源优先级构造，确保参数来自可复用样本、上游接口、local 数据、fixture 或明确规则；不得用不存在的数据测试查询接口并接受失败为通过。
6. 目标接口依赖前置参数时，必须先读取 `dependency-trace.json`；没有参数来源证据时不得判定目标接口通过或不通过。

## 判定依据优先级
1. **优先看HTTP状态码**：2xx为成功，非2xx按实际语义判断（3xx重定向、4xx客户端错误、5xx服务端错误）。
2. **其次看业务返回码**：根据接口基线中定义的业务成功/失败判定规则判断，没有基线的参考同模块其他接口的业务码规则。
3. **再看响应结构完整性**：必填字段是否存在、字段类型是否正确、是否有异常报错信息、是否有乱码、是否符合预期结构。
4. **最后看业务逻辑合理性**：响应内容是否符合本次请求的语义、数据是否符合业务规则、状态是否符合预期、关联字段是否匹配。

## 判定规则细则
### 判定为「通过」的条件
同时满足以下所有条件：
1. HTTP状态码符合预期（成功类接口2xx，重定向类接口3xx，预期失败的接口4xx/5xx）。
2. 业务返回码符合业务成功规则。
3. 响应结构完整，所有必填字段存在且类型正确。
4. 响应数据符合业务逻辑：
   - 创建类接口返回有效ID、状态正确。
   - 查询类接口返回数据和请求参数匹配。
   - 修改类接口返回修改后的结果和预期一致。
   - 删除类接口返回成功状态。
   - 状态流转符合业务规则。
5. 无异常报错信息、无堆栈、无敏感信息泄露。
6. 响应内容和请求语义匹配，没有逻辑矛盾。
7. 查询类接口必须基于真实存在的数据验证正常返回，不得仅用不存在的数据验证错误处理后就判定为通过。

### 判定为「不通过」的条件
满足以下任一条件即为不通过：
1. HTTP状态码不符合预期（预期2xx返回5xx、4xx等）。
2. 业务返回码符合业务失败规则。
3. 响应结构缺失必填字段、字段类型错误、存在异常报错、堆栈、乱码。
4. 响应数据不符合业务逻辑：
   - 创建类接口返回无效ID、状态错误。
   - 查询类接口返回数据和请求不匹配、空数据。
   - 修改类接口返回修改后结果和预期不一致。
   - 删除类接口返回失败状态。
   - 状态流转不符合业务规则。
5. 存在敏感信息泄露（明文密码、密钥、用户隐私未脱敏）。
6. 响应内容和请求语义矛盾，逻辑明显错误。
7. 用不存在的数据测试查询接口得到错误响应（如"record not found"、"空列表"），却判定为通过。必须先用真实数据验证正常路径。

### 判定为「待确认」的条件
满足以下任一条件即为待确认，必须说明缺少的信息或需要确认的点：
1. 接口基线缺失，没有明确的业务成功/失败规则，无法判断。
2. 响应内容符合结构，但业务含义不明确，无法判断是否正确。
3. 依赖的前置数据不确定，无法确认响应是否正确。
4. 响应部分符合预期，但部分字段含义不明确，无法判断。
5. 接口返回业务错误码，但不确定是预期内的正常错误还是异常错误。
6. 其他agent无法独立判断的情况，必须明确列出需要确认的点。
7. 接口返回外部服务超时或连接失败错误，但未排查代理配置（如 `httpConfig.proxy`）是否正确启用，无法区分是网络环境问题还是接口代码问题。

## 参数依赖与环境阻断判定（强制）

以下分类不是目标接口自身失败，必须在报告中单独展示：

| 分类 | 适用条件 | 门禁含义 |
| --- | --- | --- |
| `BLOCKED_BY_DEPENDENCY` | provider 接口失败、未执行或无可用响应，导致 consumer 无法构造参数 | consumer 未执行，不得算作目标接口 FAIL；provider 按自身结果判定 |
| `PARAM_UNRESOLVED` | provider 成功或 local 数据可访问，但必填参数无法按 selector / extract 提取 | 参数规则或数据分布缺口，不得伪造参数继续测试 |
| `ENV_BLOCKED` | local 服务、数据库、缓存、代理、鉴权或本地依赖不可用 | 环境阻断，不得连接 test / prod 兜底 |
| `BASELINE_STALE` | 接口 schema、参数来源或历史样本已漂移，尚未完成复验 | 基线需更新或参数需复验后再执行 |

处理要求：

1. `BLOCKED_BY_DEPENDENCY` 必须写明失败 provider、失败响应文件、失败原因和受影响 consumer 列表。
2. `PARAM_UNRESOLVED` 必须写明参数名、来源类型、提取路径、selector 和未命中原因。
3. `ENV_BLOCKED` 必须写明本地阻断项，不得输出为目标接口“不通过”。
4. `BASELINE_STALE` 必须写明受影响字段和需要复验的 `reusable-params.yaml` 条目。
5. 上述分类必须同步写入 `dependency-trace.json` 与接口明细报告。

## 判定理由撰写要求
每个接口的判定理由必须包含以下内容：
1. 说明判断的主要依据：如“HTTP状态码200，业务code=0，返回orderId有效，状态为pending，符合创建订单预期”。
2. 如果是不通过，说明具体不通过的原因：如“HTTP状态码500，返回内部错误，堆栈显示数据库连接异常”。
3. 如果是待确认，说明需要确认的具体内容：如“业务返回code=1001，含义未知，需要确认是正常业务错误还是异常”。
4. 理由必须明确、具体，不能用模糊描述如“看起来正常”、“应该没问题”。

## 特殊情况处理
- 对于响应结果是动态值的情况（如ID、时间、随机数、余额、分页数据、动态状态），只判断语义正确性，不判断具体值是否正确，只要结构合法、类型正确、语义合理即为通过。
- 对于响应内容过大的情况，只提取关键字段用于判断，完整响应落盘到证据文件，说明“响应内容过大，仅校验关键字段，完整响应见附件”。
- 对于敏感字段，必须脱敏后再输出，判断时忽略真实值，只判断类型和格式是否正确。
- 对于测试数据真实性要求：查询类接口必须优先使用可追踪的真实参数来源验证正常返回路径，不得仅用不存在的数据测试并接受失败为通过。写入类接口必须用真实参数执行实际调用，不得仅做代码审查就判定为通过。若所有参数来源都无法提供可用数据，标记为待确认或参数未解析，不得直接判定为通过。

## 写接口专用判定分类（与 `test-data-construction-rules.md` 联动）

> 本节是所有产生业务副作用的写入口专用判定分类，是对上文三类判定的强化，不是替代。具体业务错误白名单必须由项目 adapter 声明。

### 一、为什么写接口需要扩展判定分类

- 写入口的请求受外部依赖和业务状态影响，历史参数在当前时点不保证仍然成功。
- 只用"通过/不通过"两类判定，会把"业务允许的失败"误判为"接口不通过"，导致"测试任务这个接口不通过"被错误放大，掩盖真实接口问题。
- 用 4 类判定（`PASS` / `EXPECTED_FAIL` / `UNEXPECTED_FAIL` / `PENDING`）后，业务允许的失败可以走 PARTIAL 门禁，真异常（5xx、空 msg、堆栈）才能被准确识别。

### 二、4 类判定定义

- `PASS`：HTTP 2xx 且业务码符合创建/修改类成功语义，资源/订单/记录创建成功，状态正确。
- `EXPECTED_FAIL`：HTTP 4xx 或业务失败码落在业务允许集合内，且错误信息明确属于以下任一类：
  - 项目基线声明的业务拒绝码、错误类型或状态组合
  - 上游服务明确返回且已在 adapter 白名单中的业务级拒绝
- `UNEXPECTED_FAIL`：5xx、空 msg、堆栈、字段缺失、状态不符、敏感信息泄露等真异常。
- `PENDING`：缺少判定依据（基线缺失、样本矩阵不全、外部依赖无法评估等），需要人工或下一轮测试补齐。

### 三、判定映射与历史结论保留

- 历史报告中的"通过"等价于 `PASS`；"不通过"在写接口场景下需要细分为 `EXPECTED_FAIL` 或 `UNEXPECTED_FAIL`，不得混用。
- 历史报告中的"待确认"在写接口场景下等价于 `PENDING`。
- 重新出报告时，写接口明细必须按 4 类展示，混用 3 类判定视为报告不合格。

### 四、判定理由撰写补充（写接口专用）

- 写接口判定理由必须显式标注 EXPECTED_FAIL 的子类（手续费不足 / 维护中 / 黑名单 / 超范围 / 上游业务级拒绝），并给出 `status` 值或错误信息关键词。
- 写接口 UNEXPECTED_FAIL 的判定理由必须包含 HTTP 状态码、错误信息原文片段（截断到 80 字符）、堆栈首行（如有）。
- 写接口 PASS 的判定理由必须包含资源/订单/记录 ID（脱敏后）、创建后状态。
