## 2. 项目目标与范围

### 2.1 项目目标

本项目是一个面向团队研发协作的 skill 仓库。核心目标是让 AI 在处理需求、Bug、编码、审查、测试和交付任务时，能够按任务阶段与代码位点自动命中对应规则，而不是依赖人工临时拼接提示词。

### 2.2 主要服务对象

- 需要长期维护 AI 协作规则的研发团队
- 使用 Codex 或类似代理参与研发流程的开发者
- 希望把团队方法论沉淀成可复用规则资产的维护者

### 2.3 当前不覆盖的范围

- 不提供业务系统本身的后端服务、前端应用或数据库能力
- 不作为独立运行的在线服务发布
- 不替代具体业务项目中的真实需求判断、Bug 根因分析和测试执行，只提供规则与流程约束

## 3. 技术栈与运行形态

| 维度 | 当前形态 |
| --- | --- |
| 内容主体 | Markdown、YAML、SVG、少量 CSV/JS |
| 工具脚本 | Python、Shell |

补充说明：

- 仓库没有业务应用常见的构建配置、服务进程入口或 CI 流水线定义。
- 仓库价值主要在“规则资产组织 + 生成可检索索引 + 供代理运行时调用”。
- 少量模块具备程序化能力，例如：
  - `skill-dictionary/generate_dictionary.py`：生成字典与统计数据

### 4.2 skill 目录骨架

当前仓库共有约 93 个一级目录，其中约 85 个目录包含 `SKILL.md`，形成 skill 主体。常见骨架如下：

- `SKILL.md`：定义触发条件、边界、流程和通过标准
- `references/`：承载拆分规则、模板、示例和边界细则
- `agents/openai.yaml`：代理侧元信息
- `scripts/`：具备可执行能力的少量 skill 才会包含
- `templates/`、`assets/`、`data/`：只在特定 seed 或工具型 skill 中出现

这说明仓库维护重点不是业务代码，而是结构化规则资产。

## 5. 核心链路

### 5.1 研发任务执行主链路

本项目的核心不是函数调用链，而是“任务阶段驱动的规则路由链”：

1. 用户提出任务
2. 先经过 `skill-hit-check-rules` 做本轮命中检查
3. 新会话且缺上下文时，进入 `recent-context-bootstrap-rules`
4. 阶段不清或多 skill 冲突时，进入 `team-development-rules`
5. 涉及产物落点或项目总览时，进入 `artifact-storage-rules` / `project-design-doc-rules`
6. 根据任务性质进入需求域或 Bug 域
7. 一旦开始改代码，默认叠加编码基线域，再按改动位点叠加细粒度规则
8. 编码完成后进入编码审查域
9. 通过后进入测试域
10. 测试完成后进入交付域
11. 最终由 `skill-compliance-gate-rules` 和总结类规则收口

### 5.2 仓库维护与字典生成链路

仓库内部还有一条规则资产维护链路：

1. 新增或调整某个 skill
2. 更新对应目录中的 `SKILL.md`、`references/`、`agents/openai.yaml`，必要时更新 `scripts/`
3. 运行 `python3 skill-dictionary/generate_dictionary.py`
4. 刷新 `字典.md` 与 `skill-dictionary/data.js`
5. 由 `index.html` + `skill-dictionary/app.js` 消费数据，提供静态检索界面

## 6. 关键支撑模块

### 6.1 字典系统

字典系统是仓库中最明确的程序化能力：

- `skill-dictionary/generate_dictionary.py`
  - 扫描规划文档与 skill 目录
  - 汇总 domain、skill、references、agents 和统计信息
  - 输出 `字典.md` 与 `skill-dictionary/data.js`
- `index.html`
  - 本地静态入口页
- `skill-dictionary/app.js`
  - 用于展示、筛选与检索 skill 字典

### 6.2 仓库级自举与协作脚本

- `project-agents-bootstrap/scripts/bootstrap_agents.sh`
  - 确保仓库根目录存在并补齐 `AGENTS.md`
- `git-collaboration-rules/scripts/pre_commit_gate.sh`
  - 提交前 gate
- `git-collaboration-rules/scripts/post_commit_gate.sh`
  - 提交后 gate

### 6.3 工具型 seed skill

以下模块不只是说明文档，还提供实际辅助能力：

- `doc`
- `pdf`
- `spreadsheet`
- `frontend-ui-visual-rules`
- `agent-browser`
- `work-report-summary-rules`

这些模块说明本仓库是“规则优先、少量工具增强”的混合形态，而不是纯静态文档集合。

### 6.4 代码图谱探索

- `codegraph-analysis-rules`
  - 任何仓库默认先尝试初始化 CodeGraph，再用于分析项目、调用链、符号关系和影响面
  - 适合项目总览、跨文件定位、重构前影响评估和 Bug 根因分析
  - 不可用时再回退到本地搜索与阅读

## 7. 当前约束与风险

### 7.1 字典产物存在时间滞后风险

- 当前 `skill-dictionary/data.js` 的生成时间是 `2026-05-25 16:18:34`。
- 最近 Git 提交是 `2026-06-11`，说明后续文档级调整未必每次都同步刷新字典。
- 一旦新增、删除或重命名 skill 后未刷新字典，`README.md`、`字典.md`、静态页面与真实目录会再次偏移。

### 7.2 `.system/` 与正式仓库主线需要继续区分

- `.system/` 目录当前可见，但它属于运行环境注入能力，不是仓库正式跟踪的主线资产。
- 做项目分析、字典统计或规则维护时，应优先聚焦仓库正式目录，不把 `.system/` 误判为本仓库业务模块。

### 7.3 `doc/` 目录存在语义重叠

- 在通用产物规则里，`doc/` 常被视为项目文档根目录。
- 但本仓库中，`doc/` 已经是一个独立 skill 目录。
- 这会让“项目普通文档目录”和“doc skill”在命名上产生语义重叠。当前通过根目录 `项目设计.md` 作为主入口规避了一部分冲突，但该约束需要持续保持。

### 7.4 研发产物目录尚未形成丰富真实样本

- `doc/需求/`、`doc/bugs/`、`doc/tests/` 目录目前只有少量示例或占位内容；`doc/架构/`、`doc/阶段/`、`doc/审查/` 已作为正式规则口径保留，但当前样本仍较少。
- 这些目录更多承担“规则体系自证”和“流程归档规范样本”的作用，而不是大规模业务项目沉淀结果。

### 7.5 缺少自动化校验流水线

- 当前没有看到 CI、自动测试流水线或自动字典刷新机制。
- 一致性主要依赖维护者手动同步规则、文档和生成产物。

## 8. 维护建议

### 8.1 日常维护顺序

每次新增、删除或明显调整 skill 后，建议至少同步以下内容：

1. 对应 skill 的 `SKILL.md`
2. 对应的 `references/`、`agents/openai.yaml`、必要脚本或模板
3. `README.md` 中的总览或统计描述
4. 根目录 `项目设计.md`
5. 运行 `python3 skill-dictionary/generate_dictionary.py` 刷新字典

### 8.2 文档收口建议

- 根目录 `项目设计.md` 继续作为唯一项目总览入口。
- 如果后续再出现架构总览类文档，应视为过渡材料，最终收口回 `项目设计.md`。

### 8.3 规则资产演进建议

- 继续保持“小 skill + references 拆分”的结构，不把规则重新合并成单体大文档。
- 当某个 skill 出现职责膨胀时，优先通过 `skill-split-preserve-rules` 拆分，而不是继续堆厚单文件。

## 9. 当前结论

当前仓库已经形成一套较完整的团队研发 AI 协作规则体系。它的核心价值不在提供业务功能，而在把研发过程拆成一组可以自动命中的细粒度 skill，并通过统一骨架、字典生成链路和仓库级规则保持可维护性。

从当前真实仓库状态看，项目具备以下特征：

- 域划分已经稳定
- 已落地 skill 数量完整，覆盖需求、Bug、编码、测试、交付全流程
- 仓库以 Markdown 规则资产为主，以少量脚本增强可执行能力
- 根目录说明、主规划文档、字典系统和项目设计文档已形成较清晰的四层总览

后续最重要的不是继续快速堆数量，而是保持以下几项同步：

1. 真实目录与 skill 定义
2. `README.md` 与 `项目设计.md`
3. `字典.md` 与 `skill-dictionary/data.js`
