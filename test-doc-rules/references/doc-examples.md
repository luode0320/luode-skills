# 测试文档正反例

## 正例

- `test/2026-03-28_180000/用户登录接口参数校验/README.md` 先说明测试目的、执行方式、覆盖范围和结论。
- `test/2026-03-28_180000/internal/service/login_service_validation.md` 记录详细案例和执行结果，并在 README 中写清路径。
- README 写清“真实测试代码位于 `test/2026-03-28_180000/internal/service/login_service_test.go`”，接手人可以直接定位。
- 覆盖说明明确指出本次覆盖范围与未覆盖部分，不制造“已经全测过”的错觉。

## 反例

- `test/2026-03-28_180000_用户登录接口参数校验/README.md` 把中文直接拼进时间戳根目录名，重新引入旧结构。
- 在 `test/2026-03-28_180000/用户登录接口参数校验/` 里继续放 `report.md`、`steps.md`、截图和附件，破坏“仅 README”约束。
- 把测试说明单独放到 `testing/`、仓库根目录或 `analysis/`，与当前时间戳任务目录脱离。
- README 只写“已测通过”，没有测试对象、入口路径、步骤、环境和结论依据。

## 判定提示

- 如果接手人看完 README 仍不知道真实测试代码和证据放在哪，说明文档不达标。
- 如果中文说明目录中除了 README 之外还有一堆附加文件，说明结构已经跑偏。
- 如果详细说明文件没有指向真实代码路径镜像目录，而是落在随意命名的目录里，说明归档方式不达标。
