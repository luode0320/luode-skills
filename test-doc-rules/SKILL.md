# 测试文档规则

只在“测试说明文档应该怎么写、怎么组织、怎么留痕”这个问题上使用这个 skill。
如果当前争议是测试程序怎么写，请转交 `test-program-rules`；如果是功能到底算不算通过，请转交 `functional-validation-rules`。

## Skill 作用与适用场景

- 作为测试资源管理链的第四道规则，在 `test-location-rules` 基础上统一测试文档最小结构和归档要求。
- 约束测试 README、验证说明、测试报告、覆盖说明、测试执行记录的字段与章节边界。
- 保证测试资产、验证过程、结果结论和后续复盘信息可追溯。
- 防止只写口头说明、不留记录，或把功能验收结论和文档组织问题混在一起。

## 自动触发信号

- 新增或修改测试 `README.md`、验证说明、测试报告、覆盖说明、测试执行记录。
- 为当前需求、当前 Bug 或当前联调任务补充测试过程说明和结果留痕。
- 发现测试已执行，但缺少最小说明、缺少结果记录或缺少覆盖范围说明。
- 发现测试文档章节混乱，无法定位测试对象、执行步骤、环境依赖和结论时。
- 发现测试文档准备放到 `testing/`、`analysis/`、根目录或其他非 `test/` 任务目录位置时。

## 进入后先做什么

1. 先确认测试资源落点、命名和测试程序结构已经基本稳定，并且已遵循 `test-location-rules`。
2. 确认当前文档位于 `test/` 根目录下的当前任务目录中；默认以该任务目录的 `README.md` 作为主文档入口。
3. 区分当前文档属于测试 README、验证说明、测试报告、覆盖说明还是执行记录。
4. 明确当前文档服务的是资源说明、执行过程、结果记录还是覆盖范围说明。
5. 检查当前文档是否已经能支撑复现、交接、复盘和回归引用。

## 默认执行流程

1. 默认先确认遵循 `test-location-rules`：所有测试文档统一放在 `test/` 根目录下的当前任务目录中，每个任务目录必须包含 `README.md`。
2. 默认再读 `references/doc-minimums.md`，确认不同文档类型的最小章节和字段。
3. 如果当前文档与功能验证、回归验证、程序实现职责混淆，再读 `references/doc-boundaries.md`。
4. 如果需要判断当前文档是否达标，再读 `references/doc-examples.md` 对照正反例。
5. 输出缺失章节、补充建议、通过 / 驳回结论和归档建议。
6. 测试文档未稳定前，不进入正式测试交付和回归留档结论。

## 权责边界与不负责事项

- 只负责测试文档结构、字段和文档组织，不负责决定测试资源目录落点，那属于 `test-location-rules`；但本 skill 必须服从其根目录与任务目录规则。
- 不负责测试程序实现与脚手架拆分，那属于 `test-program-rules`。
- 不负责功能是否满足需求和验收标准，那属于 `functional-validation-rules`。
- 不负责历史能力是否受损和回归范围是否充分，那属于 `test-regression-rules`。
- 如果文档内容无法写清，是因为需求、Bug、联调问题本身尚未厘清，应回流对应领域 skill。

## 需要暂停并确认的条件

- 当前文档既想充当测试资源说明，又想充当功能验收结论，职责混杂。
- 当前项目尚无稳定测试文档载体，新增模板会影响整个仓库后续沉淀方式。
- 为补全文档需要先补齐大量缺失的测试事实、执行结果或环境信息。
- 文档中涉及的测试结论与实际执行结果矛盾，无法直接归档。
- 当前测试文档准备脱离当前测试任务目录单独放到 `testing/`、`analysis/` 或其他位置。

## 执行通过 / 驳回标准

- 通过：当前测试文档已经清楚说明测试对象、执行方式、依赖条件、结果结论和归档位置，且统一位于 `test/` 根目录下的当前任务目录中，后续人员可据此复现和追溯。
- 驳回：当前测试文档缺少关键章节、结果留痕或覆盖说明；或文档脱离 `test/` 任务目录散落存放，无法支撑复现、交接、复盘和后续回归引用。

## 执行结果归档要求

- 将测试文档统一归档到 `test/` 根目录下的当前测试任务目录中，不得再使用 `testing/`、`analysis/` 或其他并行目录。
- 当前测试任务目录必须包含 `README.md`，并默认作为该任务的主文档入口；如需拆分更多说明文件，也只能放在同一任务目录中。
- 归档内容至少包含文档类型、适用范围、执行日期、执行人、结果结论和关联测试资产位置。
- 如果同一需求存在多轮独立验证或多个测试程序，应分别放在各自的测试任务目录中，各自维护自己的 `README.md` 和说明文件。
- 旧测试目录归档继续遵循 `test-location-rules`，统一放在 `test/` 目录下处理。

## references 读取规则

- 默认先读 `references/doc-minimums.md`。
- 只有在职责边界混淆或结论归属不清时，再读 `references/doc-boundaries.md`。
- 只有在需要正反例和模板对照时，再读 `references/doc-examples.md`。
