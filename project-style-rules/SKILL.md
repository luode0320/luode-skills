# 项目风格规则

## 核心目标

维护项目根目录下唯一的代码风格记忆文档 `PROJECT_STYLE.md`，把项目里反复出现、值得长期复用的代码写法统一沉淀到这一份主文档里。

本 skill 只维护长期风格记忆，不负责在每次编码前生成本轮风格契约；代码生成、修改或重构前的风格契约由 `code-generation-style-rules` 负责生成和收口。

## 适用场景

- 记录项目中的方法、注释、类、结构体、变量、异步写法、日志写法、错误处理、接口写法、工具调用和循环写法等风格样例。
- 通过阅读代码归纳项目偏好的写法，并把结论写回风格文档。
- 对话中出现新的明确风格样例后，立即补入主文档。
- 用户后续调整某个写法、注释风格、命名偏好或接口写法时，立即回写原样例。
- 后续写代码时，由 `code-generation-style-rules` 读取这份风格文档、当前文件和同目录样例，再形成本轮可执行的代码风格契约。
- 用户给出“根据 skill 补充更新 md / 补充更新 md”等聚合指令时，作为四个核心 md 之一被统一编排：检测根目录 `PROJECT_STYLE.md`，缺失则按主文档模板创建，已存在则按合并规则增量回写。

## 默认流程

1. 先读取当前的 `PROJECT_STYLE.md`，再开始分析项目风格。
2. 再读取当前对话、相关代码和已有项目文档。
3. 只抽取明确、可直接复用的风格样例。
3.1 优先提取可落地的代码片段、配置片段、路径片段或命令片段，避免只写抽象口号。
4. 先用条目标题、别名、来源、适用范围定位是否已有同一主样例。
5. 若已有主样例且写法已被新规则替换，直接更新原样例，不新增平行冲突样例。
6. 规范化名称，合并别名，去除重复项。
7. 必要时在 `变更记录` 追加一条简短更新说明。
8. 直接重写主风格文档，让新样例立刻可用。
9. 后续对话优先使用更新后的主文档。

## 写入规则

- 只保留一个长期风格文件：`PROJECT_STYLE.md`。
- 不创建 `PROJECT_STYLE.log.md` 或其他临时风格文件。
- 不保留待确认区、草稿区或临时区。
- `PROJECT_STYLE.md` 的标题、章节名、条目标题和字段名必须使用中文；英文只允许作为代码符号、文件名、原始字段名或别名保留。
- 不写猜测内容，风格不明确就不写入。
- 两条样例描述有重叠时，合并成一个主样例。
- `示例` 字段必须优先写成代码片段、配置片段、命令片段、路径片段或真实使用片段；只有确实不存在可引用片段时，才允许退化成简短文字示例。
- 若风格来自代码实现，优先保留最小可复用片段，而不是只写“中文优先”“保持一致”这类抽象结论。
- 同一写法如果只是项目偏好更新，必须回写原样例，不得额外新增一个同义冲突样例。
- 旧写法过时了，就直接替换为当前权威写法，并同步刷新 `更新时间`。
- 新旧写法确实需要并存时，必须按 `适用范围` 拆开；若旧写法已失效，则直接标记为 `状态: 废弃`。
- 发生风格切换、命名偏好调整或注释规范修订时，若会影响后续实现，必须在 `变更记录` 中追加一条简短原因说明。
- 某条写法已废弃时，直接在主文档里标记为 `状态: 废弃`，不要迁移到别的文件。

## 来源优先级

- 优先当前项目代码。
- 其次优先最近一次明确的对话确认。
- 再次优先已有项目文档。
- 最后才参考旧风格内容，而且只能在没有更新来源时使用。

## 风格条目结构

每条内容尽量短、稳定、便于检索。必须使用中文字段名：

- `别名`
- `类型`
- `示例`
- `说明`
- `来源`
- `适用范围`
- `更新时间`
- `状态`

补充要求：

- `示例` 默认使用反引号包裹最小片段；如果一行放不下，可使用 fenced code block。
- `示例` 应能让后续实现者直接模仿，不应只停留在抽象描述。
- 同一条目若同时需要“推荐写法”和“禁用写法”，优先在 `示例` 中给推荐片段，在 `说明` 中补禁用边界。

参考 [project-style-template.md](references/project-style-template.md) 获取可直接复制的主文档模板和合并检查清单。
