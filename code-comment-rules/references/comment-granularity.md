# 注释颗粒度

- 能靠好命名解决的，不靠注释重复解释。
- 颗粒度控制以“改动位点必须补齐”为前提，先保证改动位点注释完整，再谈是否精简。
- 注释优先解释为什么、边界是什么、风险在哪里。
- 逐行注释通常意味着代码或抽象本身有问题，但字段清单和对象初始化的逐项字段注释属于必须保留的例外。
- 只要代码块存在大于等于 2 个步骤，就必须写中文步骤注释，并使用顶层 `1.`、`2.`、`3.` 连续编号。
- 步骤注释应直接写在对应步骤前，描述该步骤要做什么，不要只写”处理数据””继续执行”这类空话。
- 步骤编号与代码步骤必须一一对应，不能只补 `1.` 不补后续 `2.`、`3.`。
- 若顶层步骤内部存在 2 个及以上子步骤，推荐继续使用 `1.1`、`1.2`、`2.1` 分层编号；但不允许只写子编号而缺失顶层编号。
- 若方法体已经有两个及以上流程型普通注释，必须升级为编号步骤注释；保留普通注释不编号视为不通过。
- 类、结构体或 DTO 的字段注释不能只写在定义处；在初始化、字段映射、出参组装或字段使用这些字段时，也要重复标注字段含义。
- 每个函数、方法的修改都必须补充最近修改时间，格式为北京时间：`yyyy-MM-DD HH:mm:ss`。修改时间应放在函数、方法注释的显眼位置。
- 最近修改时间行必须追加本次改动原因简述，要求简短且可定位，不得只写“优化”或“调整”这类空泛词。
- 每个函数、方法都必须补充 `[参数]` / `[返回]` 简单注释，建议每项一句话说明核心语义；无参数或无返回时写 `无`，避免留空。
- 函数、方法注释元信息顺序固定为：`[参数]`、`[返回]`、`最近修改时间`；其中“最近修改时间”必须在注释块最后一行。
- 即便用户只要求“补充注释”，也不能只补方法头，仍需检查方法体是否满足 `1.` `2.` `3.` 步骤注释规则。
