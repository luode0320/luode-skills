# 代码注释规则

只在需要决定“这里该不该写注释、写哪种注释、写多细”时使用这个 skill。
如果主要争议是注释用中文还是英文，请转交 `chinese-comment-rules`；如果代码本身结构已经难以理解，应优先回到可读性和结构问题。

## Skill 作用与适用场景

- 判断注释是否必要，避免无效注释泛滥。
- 规范块注释、步骤注释、边界注释和风险注释的放置方式。
- 强制注释语言必须中文（除标准协议字段/第三方固定错误原文外）。
- 强制多步骤代码块使用中文编号步骤注释。
- 强制类、结构体字段及其初始化、使用位置补充字段注释。
- 强制每个函数、方法都补充 `[参数]` / `[返回]` 简单注释（无参数或无返回时明确写 `无`）。
- 强制每个函数方法的修改都必须补充最近修改时间（北京时间：yyyy-MM-DD HH:mm:ss）。
- 约束注释颗粒度，防止把每行代码都重新写一遍。
- 治理过期注释、错位注释和误导性注释。

## 强制规则：步骤注释位置

- 当函数/方法存在 2 个及以上连续步骤时，必须使用中文编号注释 `1.` `2.` `3.`。
- 每个编号注释必须紧贴其对应步骤的第一行代码上方。
- 禁止把多个步骤编号集中写在函数开头作为总览后不就近落位。
- 步骤编号与代码块必须一一对应；一个编号只描述一个步骤，不跨步骤复用。
- 步骤代码若是 `if/for/switch/select`，编号注释放在该结构入口上方。

## 强制规则：注释语言

- 注释必须使用中文表达。
- 标准协议字段、标准关键字、第三方固定错误原文可保持原文。
- 出现第三方固定错误原文时，建议补一行中文解释。
- 除上述例外外，不允许英文注释主句。

## 自动触发信号

- 实现中出现复杂分支、非直观约束、历史兼容逻辑或风险处理。
- 一个代码块中已经存在大于等于 2 个明确步骤。
- 新增或修改类、结构体、DTO、配置对象、请求体、响应体等字段列表。
- 初始化对象、结构体或使用多个字段时，不确定是否需要重复标注字段含义。
- 新增或修改函数、方法时，需要补充或更新最近修改时间。
- 新增或修改函数、方法时，需要补充或更新 `[参数]` / `[返回]` 简单注释。
- 代码评审指出缺少关键说明或存在误导性注释。
- 准备补注释，但不确定应该写在哪里、写到多细。
- 需要清理与当前实现不一致的旧注释。

## 进入后先做什么

1. 先判断代码是否已经足够自解释。
2. 检查是否新增或修改了函数、方法，如果是，必须补充或更新最近修改时间（北京时间：yyyy-MM-DD HH:mm:ss）。
3. 对每个新增或修改的函数、方法，补充 `[参数]` / `[返回]` 简单注释；无参数或无返回时写 `无`。
4. 统计当前代码块是否已经形成大于等于 2 个步骤；如果是，直接规划中文 `1.`、`2.`、`3.` 步骤注释。
5. 确认当前代码是否存在类、结构体或对象字段清单，并定位字段定义、初始化和使用位置。
6. 选择最能帮助理解的位置和最小必要颗粒度。
7. 检查注释是否和当前实现保持同步。

## 默认执行流程

1. 默认先读 `references/comment-placement.md`，先判断注释应放在哪里最有效。
2. 如需继续展开，再读 `references/comment-granularity.md`，需要控制注释写多细、写多少。
3. 需要对照边界或正反例时，再读 `references/comment-examples.md`，需要对照注释结构正反例。
4. 输出注释必要性判断、步骤注释方案、字段注释方案、函数方法 `[参数]` / `[返回]` 注释方案和待清理注释清单。
5. 注释结构确定后，再交给 `chinese-comment-rules` 优化语言表述；最终在 `implementation-review-rules` 中复核整体质量。

## 权责边界与不负责事项

- 只负责注释结构与必要性，不代替 `chinese-comment-rules` 处理语言风格。
- 不把糟糕命名、复杂结构和错误抽象都靠注释硬解释。
- 不鼓励逐行复述代码行为，但“多步骤编号注释”和“字段重复注释”属于明确要求的例外场景。
- 不要求把 `[参数]` / `[返回]` 写成大段文档，保持简单、可读、与签名一致即可。
- 不保留已经与实现脱节的注释。

## 需要暂停并确认的条件

- 代码本身可以通过重命名或小重构解决理解问题。
- 注释打算解释一大段明显应拆函数的流程。
- 现有注释和实现冲突，但代码行为还未确认。
- 为了通过评审而临时补大量无效注释。

## 执行通过 / 驳回标准

- 通过：只要代码块存在大于等于 2 个步骤，就使用中文 `1.`、`2.`、`3.` 步骤注释；注释语言满足“必须中文（除标准协议字段/第三方固定错误原文外）”；类和结构体字段有注释，字段初始化和使用位置也有重复字段注释；所有新增或修改的函数、方法都有最近修改时间标注（北京时间：yyyy-MM-DD HH:mm:ss）以及 `[参数]` / `[返回]` 简单注释；其余注释与实现一致。
- 驳回：多步骤代码块没有编号步骤注释，或注释语言不满足“必须中文（除标准协议字段/第三方固定错误原文外）”，或字段定义/初始化/使用位置缺少注释，新增或修改的函数、方法缺少最近修改时间标注或缺少 `[参数]` / `[返回]` 注释，或注释泛滥、位置错乱、明显过期，反而增加理解成本。
- 驳回：出现“`1/2/3` 注释集中写在一起，但未在各自代码位置就近注释”的情况。

## 执行结果归档要求

- 一般不要求单独归档；如清理了关键过期注释或新增重要边界注释，可在评审记录中说明。
- 如果注释承载重要设计边界，应确保相关文档或评审结论可追溯。

## references 读取规则

- 默认先读 `references/comment-placement.md`。
- 只有在 控制注释颗粒度和必要性 时，再读 `references/comment-granularity.md`。
- 只有在 对照注释样例或判断是否过量 时，再读 `references/comment-examples.md`。
