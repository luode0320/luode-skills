# 审查报告模板

## 白话正文与附录

正文先给出是否放行、主要原因和会带来的影响。问题优先级、文件行号、命令、覆盖范围和证据只放入执行附录或追踪附录。

```text
发现:
- [P1] path/to/file.ext:123 - 简短标题
  证据: 指出当前 diff 或源码行为。
  风险: 说明可能造成的缺陷、安全、回归或规则违规。
  建议: 给出最小可执行修复方向。
  关联 skill: <主 skill 名称；若仅为总审查发现可留空>

未发现阻断项:
- 未发现 P0/P1 时写本段；若存在阻断项则省略。

统一判定:
- 审查结论: 通过/阻断
- 审查范围: <文件列表或等价范围说明>
- 是否允许提交: 是/否
- 阻断问题: <P0/P1 摘要，没有则写无>

已覆盖:
- Diff 范围:
- 已读规则:
- 已联动 skill:
- 已执行命令:

未覆盖/剩余风险:
- 未运行测试:
- 未展开文件:
- 仍需确认:

必要后续动作（仅存在 P0/P1/P2、未覆盖阻断或用户明确要求建议时输出）:
1. 优先处理 P0/P1。
2. 补齐验证或专门 skill 审查。
3. 复审修复后的 diff。
```

## 写法约束

- findings 永远放最前。
- 不用“整体不错”开头。
- 没有发现时直接写“未发现阻断项”，同时列出未覆盖风险。
- 必要后续动作必须能落地到文件、测试、skill 或复审动作；若未发现阻断项且原始目标已完成，不输出本区块，也不写等待类或无需动作类占位文案。
