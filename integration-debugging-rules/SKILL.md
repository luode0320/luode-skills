# 联调排障规则

只在“单点代码看起来没问题，但跨系统、跨环境、跨链路跑不通”这个问题上使用这个 skill。
如果当前问题只是当前需求功能本身不正确，请转交 `functional-validation-rules`；如果只是单模块运行时定位 Bug，请转交 `bug-runtime-debug-rules`。

## Skill 作用与适用场景

- 统一上下游系统联调、环境差异排查、字段对齐、链路不通和 trace 断链场景的处理方式。
- 约束联调证据采集、分流判断、环境记录和排查结论留痕。
- 区分功能验证失败、联调失败和单模块运行时诊断失败的边界。
- 防止把跨系统问题误当成功能编码问题，或把单模块问题硬抬到联调层。

## 自动触发信号

- 联调时出现接口不通、字段不一致、环境差异、链路不通、trace 断链、测试环境问题。
- 当前功能本身单点验证看起来没问题，但上下游串起来失败。
- 需要判断问题出在调用方、被调方、网关、Header 透传、日志追踪还是环境配置差异时。
- 需要保留联调证据、环境信息、请求响应样本和链路观测结论时。

## 进入后先做什么

1. 先确认当前问题确实跨越多个系统、多个服务或多个环境，而不是单点功能错误。
2. 画清调用链：谁发起、谁转发、谁消费、在哪一跳失败。
3. 记录环境信息、版本信息、配置差异、Header 样本、请求响应样本和 trace 线索。
4. 先判断是协议不一致、Header 断链、环境差异、权限差异还是依赖未准备好。

## 默认执行流程

1. 默认先读 `references/integration-scope-and-evidence.md`，确认联调场景和最小证据集合。
2. 如果涉及环境差异、字段不一致、链路断裂，再读 `references/env-diff-and-chain-diagnostics.md`。
3. 如果需要判断应回流功能验证、Header 规则、日志 trace 规则还是 Bug 运行时诊断，再读 `references/integration-boundaries-and-handoff.md`。
4. 输出链路位置、环境差异、证据样本、责任分流和下一步动作。
5. 联调结论未稳定前，不直接宣告功能失败或回归失败。

## 权责边界与不负责事项

- 只负责跨系统、跨环境、跨链路联调排障，不负责当前需求功能正确性结论，那属于 `functional-validation-rules`。
- 不负责单模块运行时断点、单步、变量观察的 Bug 诊断，那属于 `bug-runtime-debug-rules`。
- 不负责 Header 契约定义本身，那属于 `request-header-rules`，这里只判断 Header 是否按约定透传。
- 不负责日志字段设计，那属于 `logging-trace-rules`，这里只使用既有日志 / trace 证据判断链路是否断裂。
- 不把所有调用失败都归因于“环境问题”，也不把所有联调失败都粗暴回流给编码域。

## 需要暂停并确认的条件

- 当前证据不足，连失败发生在哪一跳都无法说明。
- 当前联调问题涉及多个环境版本、多个网关或多个上下游团队，但基础信息未统一。
- 当前排障已经明显需要修改产品需求、协议契约或大范围配置，却未先固定责任边界。
- 当前现象更像单模块代码 Bug 或纯功能失败，继续停留在联调层只会扩大噪音。

## 执行通过 / 驳回标准

- 通过：能说明失败链路、环境差异、协议样本、trace / Header 线索和责任分流，且结论足以指导下一步修复或回流。
- 驳回：联调排障停留在模糊描述，没有链路证据、没有环境信息、没有分流结论，或把不属于联调层的问题硬留在联调层。

## 执行结果归档要求

- 将联调排障结论记录到 `testing/`、联调说明、任务记录或交付说明中。
- 归档内容至少包含调用链、环境信息、关键 Header / 请求响应样本、trace 线索、失败位置、责任分流和下一步动作。
- 如果存在环境特有问题，必须单独记录差异项、复现条件和临时绕过方式。

## references 读取规则

- 默认先读 `references/integration-scope-and-evidence.md`。
- 只有在涉及环境差异、字段不一致和链路断裂时，再读 `references/env-diff-and-chain-diagnostics.md`。
- 只有在需要回流其他 skill 或判断职责边界时，再读 `references/integration-boundaries-and-handoff.md`。
