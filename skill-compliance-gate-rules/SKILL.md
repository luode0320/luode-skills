# Skill 执行完整性闸门规则

只在“skill 已触发但执行闭环可能不完整”时使用这个 skill。
目标不是重复实现业务，而是在收口前补齐“可执行规则未落实时的可行动反馈”。

## Skill 作用与适用场景

- 对本轮已触发的 skill 做末端合规检查，确认是否存在可执行但未执行的规则。
- 当无法一次性补齐全部规则时，不强行“假完成”，而是输出明确的下一步行动建议。
- 把“静默跳过规则”变成“可见的后续计划”，避免用户误以为已全部收口。
- 在不扩大输出复杂度的前提下，只强制补充“下一步建议”。
- 合并主任务建议与 skill 补齐建议，避免只输出 skill 建议导致主任务信息丢失。
- 当本轮发生代码新增或修改且准备最终回复时，作为默认收口闸门触发，防止遗漏终检。
- 在最终回复前补一条注释终检：凡本轮改动位点都要核验 `comment-placement-granularity-rules` 与 `comment-completion-gate-rules` 是否已执行，并先判定注释模式（`default`/`strict`）；`default` 仅检查关键复杂位点意图注释，`strict` 再额外检查步骤流程编号、方法更新时间、更新时间后的改动原因与 `[参数]` / `[返回]`。
- 在最终回复前补一条基础格式终检：凡本轮存在代码新增或修改，都要核验是否已执行 `cleanup-format-review-rules` 的最低格式达标检查（至少缩进、对齐、空行可读）。
- 当用户请求“补充注释/只补注释/注释完善”时，强制检查注释是否优先覆盖“未提交且有改动”的代码位点，并核验函数头注释与方法块注释完整性。
- 在最终回复前强制增加测试目录闸门：本次新增测试文件路径必须全部位于同一个“当天时间戳目录”，否则判定 skill 未完成并禁止进入提交阶段。

## 自动触发信号

- 本轮任务已命中某个 skill，且出现“只做了部分规则”的反馈或风险。
- 用户明确提出“检查是否按 skill 全部执行”或“为什么只按部分规则实现”。
- 任务准备结束，但当前输出无法说明剩余可执行项的下一步动作。
- 多个 skill 同时命中，可能出现边界冲突导致部分规则被跳过。
- 本轮发生代码改动，且需要确认是否漏触发 `comment-placement-granularity-rules` 与 `comment-completion-gate-rules`。
- 本轮发生代码新增或修改，且进入最终回复前收口。

## 进入后先做什么

1. 列出本轮实际命中的 skill 清单，不要臆造未命中的 skill。
2. 为每个 skill 判断“当前阶段应执行的规则”，只检查当前阶段可落地项。
3. 按 `references/applicability-and-gap-check.md` 识别“可执行但未执行”的规则缺口。
4. 同时收集主任务建议候选（实现、验证、风险收口、文档回写等）。
5. 判断缺口是否阻断当前任务正确性。
6. 对测试相关改动执行当天时间戳目录一致性检查：新增测试文件是否全部位于同一个“当天时间戳目录”。
7. 对代码改动执行注释闸门检查：先核验 `comment-placement-granularity-rules` 是否已覆盖注释必要性、放置位置、字段注释与过期注释治理，再按模式核验 `comment-completion-gate-rules`：`default` 仅核验关键复杂位点注释；`strict` 再核验步骤流程至少顶层 `1` 就近注释（多步骤补齐 `1/2/3`，复杂步骤可含 `1.1/1.2` 细分）、`[参数]` / `[返回]`、最近修改时间、最近修改时间后的改动原因；strict 下若方法体存在步骤流程却无顶层 `1` 编号，直接判定未通过。
8. 若用户请求补注释，额外执行优先级闸门检查：未提交改动位点是否已优先补齐；函数、方法是否同时具备函数头注释与方法块注释。
9. 对代码改动执行基础格式闸门检查：核验 `cleanup-format-review-rules` 是否已完成最低格式达标（缩进、对齐、空行）；未执行则判定未通过。
10. 合并建议并输出精简的“下一步建议 1/2/3”。

## 默认执行流程

1. 收集证据：本轮编辑、测试、日志、结论说明。
2. 先判定本轮是否发生代码新增或修改；若是且准备最终回复，立即进入本闸门流程，不得跳过。
3. 识别规则：从目标 skill 中提取当前阶段应执行规则（尤其是含“必须/默认/先做/通过标准/阻断条件”的规则）。
4. 逐条核对：判断每条规则是“已执行 / 未执行但可执行 / 当前阶段不适用”。
5. 对测试相关任务执行禁放扫描：检查禁放目录是否存在 `*_test.go`（默认禁放目录为 `test/` 之外路径）。
6. 若本次改动包含 `internal/router`，执行路由风格与约束闸门检查。
7. 若本次涉及新增测试文件，强制检查这些文件路径是否全部位于同一个“当天时间戳目录”；若发现跨天目录写入，直接判定 skill 未完成，禁止进入提交阶段。
8. 形成候选：分别生成“主任务建议候选”与“skill 补齐建议候选”。
9. 最终回复前必须确认：本轮改动位点已完成注释双 skill 收口，即 `comment-placement-granularity-rules` 的必要性 / 放置位置 / 字段注释 / 过期注释治理检查已完成，且 `comment-completion-gate-rules` 已按当前模式执行：`default` 完成关键复杂位点注释检查，`strict` 完成步骤流程至少顶层 `1`（多步骤 `1/2/3`）就近注释、`[参数]` / `[返回]`、最近修改时间、最近修改时间后的改动原因、函数注释顺序检查。
10. 最终回复前必须确认：本轮代码改动已完成 `cleanup-format-review-rules` 的最低格式收口（至少缩进、对齐、空行）。
11. 若本轮为补注释请求，最终回复前必须确认：未提交改动位点优先补注释已完成，且相关函数、方法同时满足“函数头注释 + 方法块注释”。
12. 合并排序：主任务建议优先，skill 补齐建议次优先；去重后保留 1-3 条。
13. 输出给用户：仅强制补充“下一步建议”，不强制输出完整核对明细。

## 提交前闸门检查（internal/router）

- 若本次改动包含 `internal/router`，必须检查：
- 不存在 `.GET(`、`.PUT(`、`.PATCH(`、`.DELETE(`。
- 不存在 `:id` 或 `{id}` 路径参数。
- 不存在 `registerPostRoutes` 或 route 列表驱动注册模式。
- 路由 path 未通过本地常量引用（允许全局根前缀常量）。
- 任一不满足则阻断交付，先回改再继续。

## 输出要求（简化版）

必须包含如下格式：

```text
下一步建议：
1. ...
2. ...
3. ...
```

约束：

- 建议来源必须可追溯到本轮上下文，不给泛化空话。
- 主任务建议优先于 skill 补齐建议，不得因 skill 检查而覆盖主任务核心建议。
- 建议必须是用户或代理下一步可执行动作，不写背景复述。
- 建议条数 1-3 条，按优先级排序。
- 若主任务与 skill 均无缺口，可输出“下一步建议：1. 当前无需补充动作，可进入下一阶段。”

## 阻断判定与处理

属于阻断级：

- 缺口会直接导致实现错误、测试结论失真或归档位置错误。
- 不补该规则就无法继续当前阶段的正确推进。
- 新增测试文件出现跨天目录写入，或未全部位于同一个“当天时间戳目录”。
- 本轮代码改动未完成 `comment-placement-granularity-rules` 与 `comment-completion-gate-rules` 的注释双 skill 闸门（含模式判定错误）。
- 本轮代码改动未完成 `cleanup-format-review-rules` 的最低格式闸门（缩进、对齐、空行仍明显混乱）。
- 补注释请求中，未优先覆盖未提交改动位点，或 strict 模式下函数、方法缺函数头注释/缺方法块注释。

属于非阻断级：

- 缺口不影响当前结论正确性，但应在后续迭代补齐。
- 可先完成当前任务收口，但必须给出下一步建议。

## 权责边界与不负责事项

- 只负责执行完整性闸门与下一步建议，不代替具体业务实现。
- 不强制要求一次性补齐所有历史遗留规则，优先处理当前阶段可执行缺口。
- 不把“当前阶段不适用”的规则误报为未执行缺口。
- 不输出冗长核对报告，除非用户明确要求展开明细。
- 不因为执行本 skill 而丢弃其他来源的主要建议。

## 执行通过 / 驳回标准

- 通过：已完成闸门检查（含 `comment-placement-granularity-rules` 与 `comment-completion-gate-rules` 注释终检，以及 `cleanup-format-review-rules` 最低格式终检），并给出合并后的 1-3 条下一步建议（主任务优先，skill 补齐次优先）。
- 驳回：只说“还有规则没做完”，但没有给出可执行的下一步动作；或发现新增测试文件跨天目录写入仍继续推进提交阶段；或本轮代码改动未完成注释补齐或最低格式收口仍给出“可继续”结论；或本轮发生代码新增/修改且已到最终回复前仍未触发本闸门。
- 驳回：补注释请求中未先处理未提交改动，或函数、方法注释缺函数头/方法块任一项，仍给出“可继续”结论。

## references 读取规则

- 默认先读 `references/applicability-and-gap-check.md`。
- 需要组织最终输出文案时再读 `references/next-step-suggestion-template.md`。

## 回到主流程的重启点

- 若为阻断级：先执行“下一步建议”中的高优先级项，再从当前阶段入口重新开始。
- 若为非阻断级：当前任务可收口，但下个迭代进入同域任务前优先执行建议 1。
