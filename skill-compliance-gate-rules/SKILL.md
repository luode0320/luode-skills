# Skill 执行完整性闸门规则

只在“skill 已触发但执行闭环可能不完整”时使用这个 skill。
目标不是重复实现业务，而是在收口前补齐“可执行规则未落实时的可行动反馈”。

## Skill 作用与适用场景

- 对本轮已触发的 skill 做末端合规检查，确认是否存在可执行但未执行的规则。
- 当无法一次性补齐原执行计划内必需规则或阻断级缺口时，不强行“假完成”，而是记录并校验可交接的事实。
- 把“静默跳过必需规则”变成“可校验的计划内缺口或共享阻断事实”，避免用户误以为已全部收口。
- 真实 `blocked/manual_handoff` 时，只校验 `artifact-delivery-gate-rules/references/task-blocker-closure-contract.md` 的完整性、去重与唯一渲染条件；面向用户的阻断区块和解决计划仅由 `reasoning-summary-structure-rules` 渲染。
- 非阻断时，不得因本 skill 生成任何扩散性后续。
- 收到用户明确结束指令（如“结束”“停止”“到此为止”“不要继续”“不要下一步建议”“不要扩散”）时，结束指令优先于默认建议模板；没有阻断级缺口时直接最小收口，且不得输出下一步区块或等待类占位文案。
- 当本轮发生代码新增或修改且准备最终回复时，作为默认收口闸门触发，防止遗漏终检。
- 在最终回复前补一条注释终检：凡本轮改动位点都要核验 `comment-placement-granularity-rules` 与 `comment-completion-gate-rules` 是否已执行，既检查注释必要性 / 放置位置 / 字段注释 / 过期注释治理，也检查步骤流程至少 `1`（多步骤 `1/2/3`）编号、方法更新时间、更新时间后的改动原因与 `[参数]` / `[返回]`。
- 在最终回复前注释终检必须产出“函数注释核对清单”；若本轮存在补丁位点，还必须产出“补丁注释核对清单”，没有清单视为注释闸门未执行完成。
- 函数注释核对清单 / 补丁注释核对清单属于注释闸门的内部执行证据，可并入最终回复的「Skill 执行证据」呈现；按 `reasoning-summary-structure-rules` 的总结视觉规范，不再要求在最终总结中单列“函数注释核对”展示节点（清单本身仍必须产出，不得省略）。
- 在最终回复前补一条基础实现终检：凡本轮存在代码新增或修改，都要核验是否已执行 `implementation-review-rules` 的最低测试前收口（至少覆盖基础格式、语法/类型/引用、目录归位/依赖方向）。
- 当用户请求“补充注释/只补注释/注释完善”时，强制检查注释是否优先覆盖“未提交且有改动”的代码位点，并核验函数头注释与方法块注释完整性。
- 在最终回复前强制增加测试目录闸门：本次新增测试文件路径必须全部位于同一个“当天时间戳目录”，否则判定 skill 未完成并禁止进入提交阶段。
- 在最终回复前强制增加“真实运行验证闸门”：凡本轮存在功能代码新增或修改，必须核验核心接口、页面、导出、查询、提交或任务入口是否完成真实运行验证；若只完成 `build`、lint、静态搜索、代码审查或实现自审，只能标记为“仅静态验证”，不得给“功能 / 接口已验证可用”的完成结论。
- 若真实运行验证因本地环境、授权、样本数据或依赖服务不足无法执行，最终回复必须明确列出“未完成真实验证”的路径、阻断原因和人工验证步骤；核心路径未运行且无人工交接时，判定为阻断级缺口。
- 在最终回复前强制增加“用户手改保护闸门”：若本轮出现“用户手动修改代码且与 AI 记忆/旧上下文冲突”的迹象，必须确认已执行 `code-context-resync-rules`，且最终 diff 未回退用户手改内容，否则判定未通过。
- 在最终回复前强制增加“运行时状态真收口闸门”：若当前运行环境存在 goal / plan / task 等显式状态收口机制，且主任务已满足原始用户目标或已满足该机制的阻断条件，必须确认已执行对应真实收口动作；未执行时不得给“已完成 / 等待用户新指令”结论。Codex goal 的 `get_goal` / `update_goal` 仅是其中一种特例。

## 自动触发信号

- 本轮任务已命中某个 skill，且出现“只做了部分规则”的反馈或风险。
- 用户明确提出“检查是否按 skill 全部执行”或“为什么只按部分规则实现”。
- 任务准备结束，但仍存在原执行计划内未完成必需项或阻断项；若已经没有三类合法后续，则不得触发后续内容输出。
- 多个 skill 同时命中，可能出现边界冲突导致部分规则被跳过。
- 本轮发生代码改动，且需要确认是否漏触发 `comment-placement-granularity-rules` 与 `comment-completion-gate-rules`。
- 本轮发生代码新增或修改，且进入最终回复前收口。
- 本轮发生任一工具调用，且最终回复需要给出可核验的 skill 执行证据。
- 本轮发生非预期执行失败、退出成功但产物不可信，或对已知失败进行了恢复尝试。

## 进入后先做什么

1. 列出本轮实际命中的 skill 清单，不要臆造未命中的 skill。
2. 为每个 skill 判断“当前阶段应执行的规则”，只检查当前阶段可落地项。
3. 按 `references/applicability-and-gap-check.md` 识别“可执行但未执行”的规则缺口。
4. 识别计划内未完成必需项与真实阻断候选，判断缺口是否阻断当前任务正确性。
5. 若已真实收口为 `blocked/manual_handoff`，校验共享契约的状态、阶段、依据与证据、已尝试与停止条件、影响、1-3 条解决计划、恢复验证 / 重入点；不得在本 skill 内另行生成用户文案。
6. 对测试相关改动执行当天时间戳目录一致性检查：新增测试文件是否全部位于同一个“当天时间戳目录”。
7. 对代码改动执行注释闸门检查：先核验 `comment-placement-granularity-rules` 是否已覆盖注释必要性、放置位置、字段注释与过期注释治理，再核验 `comment-completion-gate-rules` 的改动位点注释、步骤流程至少顶层 `1` 就近注释（多步骤补齐 `1/2/3`，复杂步骤可含 `1.1/1.2` 细分）、`[参数]` / `[返回]`、最近修改时间、最近修改时间后的改动原因是否齐全；若方法体存在步骤流程却无顶层 `1` 编号，直接判定未通过。
7.1 注释闸门检查完成后，必须输出“函数注释核对清单”；若本轮改动不涉及函数/方法，必须明确写出“函数位点 0 个”。
7.2 若本轮涉及原有方法补丁分支、兜底或兼容逻辑，必须额外输出“补丁注释核对清单”；若无补丁位点，明确写出“补丁位点 0 个”。
8. 若用户请求补注释，额外执行优先级闸门检查：未提交改动位点是否已优先补齐；函数、方法是否同时具备函数头注释与方法块注释。
9. 对代码改动执行基础实现闸门检查：核验 `implementation-review-rules` 是否已完成最低测试前收口（至少覆盖基础格式、语法/类型/引用、目录归位/依赖方向）；未执行则判定未通过。
10. 对功能代码改动执行真实运行验证闸门：核验是否已有 `functional-validation-rules` 或等价测试证据证明核心路径真实运行；若证据只覆盖构建 / 静态检查，必须降级为“仅静态验证”并检查是否输出人工验证交接。
11. 合并主任务状态与缺口状态；真实阻断只交由 `reasoning-summary-structure-rules` 按共享契约渲染，非阻断时不存在原执行计划内未完成必需项或用户显式建议请求则强制无下一步。
12. 对用户手改保护执行闸门检查：若出现上下文冲突，必须核验是否已重读基线并保留用户手改位点；若缺失则判定未通过。
13. 若当前运行环境存在显式状态收口机制，额外核验是否已经完成真实的状态收口动作；若没有，则本轮仍不得视为完成。

## 默认执行流程

1. 收集证据：本轮编辑、测试、日志、结论说明。
2. 先判定本轮是否发生代码新增或修改；若是且准备最终回复，立即进入本闸门流程，不得跳过。
3. 识别规则：从目标 skill 中提取当前阶段应执行规则（尤其是含“必须/默认/先做/通过标准/阻断条件”的规则）。
4. 逐条核对：判断每条规则是“已执行 / 未执行但可执行 / 当前阶段不适用”。
4.1 若出现非预期执行失败，核对 `execution-failure-learning-rules` 是否完成分类、active 案例检索、有限重试、同输入复验、candidate/active 状态判断和业务 Bug/Skill gap 分流。
5. 对测试相关任务执行禁放扫描：检查禁放目录是否存在 `*_test.go`（默认禁放目录为 `doc/5-tests/` 之外路径）。
6. 若本次改动包含 `internal/router`，执行路由风格与约束闸门检查。
7. 若本次涉及新增测试文件，强制检查这些文件路径是否全部位于同一个“当天时间戳目录”；若发现跨天目录写入，直接判定 skill 未完成，禁止进入提交阶段。
8. 形成校验结论：分别标记“计划内必需项”“真实阻断候选”与“用户显式要求的建议”；不得在本 skill 生成用户可见的恢复计划。
9. 最终回复前必须确认：本轮改动位点已完成注释双 skill 收口，即 `comment-placement-granularity-rules` 的必要性 / 放置位置 / 字段注释 / 过期注释治理检查与 `comment-completion-gate-rules` 的步骤流程至少顶层 `1`（多步骤 `1/2/3`）就近注释、`[参数]` / `[返回]`、最近修改时间、最近修改时间后的改动原因、函数注释顺序检查、补丁位点“做了什么 + 为什么要加”注释检查均已完成，且已输出函数注释核对清单（及需要时的补丁注释核对清单）。
10. 最终回复前必须确认：本轮代码改动已完成 `implementation-review-rules` 的最低测试前收口（至少基础格式、语法/类型/引用、目录归位/依赖方向）。
11. 最终回复前必须确认：本轮功能代码改动的核心路径已完成真实运行验证；若仅完成构建、静态分析、代码审查或实现自审，最终结论必须写成“仅静态验证，未完成真实运行验证”，并列出人工验证交接，不得写成“接口可用 / 功能可用 / 已完整验证”。
12. 若本轮为补注释请求，最终回复前必须确认：未提交改动位点优先补注释已完成，且相关函数、方法同时满足“函数头注释 + 方法块注释”。
13. 若真实状态为 `blocked/manual_handoff`，校验共享阻断事实满足契约且本轮最终回复只由 `reasoning-summary-structure-rules` 输出一个阻断区块；同根因不得重复计划。
14. 输出给用户：本 skill 只报告 PASS/FAIL 与缺失字段；默认不补充“后续内容”、阻断区块或解决计划。不存在计划内未完成必需项或用户显式建议请求时禁止输出“等待用户新指令”“无需继续动作”等占位文案。
15. 最终回复前必须确认：若本轮存在用户手改与 AI 上下文冲突，`code-context-resync-rules` 已执行完成，且无用户手改被回退。
16. 若本轮发生任一工具调用，最终回复必须包含 `Skill 执行证据`（命中项、执行动作、对应产物/命令）；缺失则判定失败并触发一次自动补救重试（先补证据后结束）。
17. 若当前运行环境存在显式状态收口机制，则最终回复前必须确认：不是只写了“后续内容”或完成文案，而是已经真实执行了对应的状态收口动作。

## 提交前闸门检查（internal/router）

- 若本次改动包含 `internal/router`，必须检查：
- 不存在 `.GET(`、`.PUT(`、`.PATCH(`、`.DELETE(`。
- 不存在 `:id` 或 `{id}` 路径参数。
- 不存在 `registerPostRoutes` 或 route 列表驱动注册模式。
- 路由 path 未通过本地常量引用（允许全局根前缀常量）。
- 任一不满足则阻断交付，先回改再继续。

## 输出要求（简化版）

本 skill 不拥有面向用户的“后续内容”或“任务阻断收口”模板。真实阻断时只输出合规 PASS/FAIL 与共享契约缺失项；最终用户区块由 `reasoning-summary-structure-rules` 唯一渲染。

约束：

- 上述结构是普通 Markdown 示例，不得包进 ` ```text ` 或其他代码围栏；只有真实代码、命令、配置、日志、JSON/YAML 等片段才使用代码围栏。
- 合规结论必须可追溯到原执行计划、完成定义、验收标准或共享阻断契约，不给泛化空话。
- 缺失状态、阶段、依据与证据、已尝试与停止条件、影响、1-3 条解决计划、恢复验证 / 重入点中任一字段，或渲染 owner 不是 `reasoning-summary-structure-rules`，均为 FAIL。
- `limited/not_applicable` 不得被本 skill 升级为真实阻断；若主任务无阻断级缺口和计划内未完成必需项，且用户未显式要求建议/backlog，默认直接结束。
- 可选优化、额外整理、未来迭代、体验提升、文档再润色等内容，若不属于原计划必需项，不得作为默认后续内容输出。
- 禁止为了占位输出“等待用户新指令”“无需继续动作”“当前无需补充动作”“无下一步”等文案；无下一步时直接结束。

## 阻断判定与处理

属于阻断级：

- 缺口会直接导致实现错误、测试结论失真或归档位置错误。
- 不补该规则就无法继续当前阶段的正确推进。
- 新增测试文件出现跨天目录写入，或未全部位于同一个“当天时间戳目录”。
- 本轮代码改动未完成 `comment-placement-granularity-rules` 与 `comment-completion-gate-rules` 的注释双 skill 闸门。
- 本轮代码改动未输出函数注释核对清单（或未声明函数位点 0 个）。
- 本轮存在补丁位点但未输出补丁注释核对清单（或未声明补丁位点 0 个）。
- 本轮代码改动未完成 `implementation-review-rules` 的最低测试前收口闸门。
- 本轮功能代码改动的核心接口、页面、导出、查询、提交或任务入口未完成真实运行验证，且最终回复未明确降级为“仅静态验证 / 未完成真实验证”并给出人工验证交接。
- 本轮仅执行 `build`、lint、静态搜索、代码审查或实现自审，却在最终结论中声称“接口可用 / 功能可用 / 已完整验证”。
- 补注释请求中，未优先覆盖未提交改动位点，或函数、方法缺函数头注释/缺方法块注释。
- 本轮已出现“用户手改与上下文冲突”但未执行 `code-context-resync-rules` 重同步，或最终 diff 回退了用户手改内容。
- 本轮发生工具调用但最终回复缺少 `Skill 执行证据`，且未完成一次自动补救重试。
- 当前运行环境存在显式状态收口机制，但原始用户目标已完成后未真实执行对应完成收口，或满足 blocked / failed / cancelled 条件后未真实执行对应状态收口。

属于非阻断级：

- 缺口不影响当前结论正确性，但应在后续迭代补齐。
- 可先完成当前任务收口；只有用户明确要求建议/backlog，才把这类内容作为后续内容输出。

## 权责边界与不负责事项

- 只负责执行完整性闸门与合法后续内容判定，不代替具体业务实现。
- 不强制要求一次性补齐所有历史遗留规则，优先处理当前阶段可执行缺口。
- 不把“当前阶段不适用”的规则误报为未执行缺口。
- 不输出冗长核对报告，除非用户明确要求展开明细。
- 不因为执行本 skill 而丢弃其他来源的主要建议。

## 执行通过 / 驳回标准

- 通过：已完成闸门检查（含 `comment-placement-granularity-rules` 与 `comment-completion-gate-rules` 注释终检、函数注释核对清单输出、补丁位点场景下的补丁注释核对清单输出、`implementation-review-rules` 最低测试前收口终检、功能代码改动的真实运行验证状态核验，以及需要时的运行时状态真收口检查）；若存在原执行计划内未完成必需项、阻断项，或用户明确要求后续建议/backlog，则输出合并后的 1-3 条后续内容；若不存在三类合法后续，则强制无下一步并直接结束。
- 驳回：只说“还有规则没做完”，但没有给出可执行的计划内必需动作或阻断恢复动作；或发现新增测试文件跨天目录写入仍继续推进提交阶段；或本轮代码改动未完成注释补齐、未输出函数注释核对清单、补丁位点缺补丁注释核对清单、最低测试前收口未完成、核心路径真实运行验证状态缺失，仍给出“可继续 / 已完成 / 已验证可用”结论；或本轮发生代码新增/修改且已到最终回复前仍未触发本闸门。
- 驳回：补注释请求中未先处理未提交改动，或函数、方法注释缺函数头/方法块任一项，仍给出“可继续”结论。
- 驳回：已识别用户手改冲突，但未执行 `code-context-resync-rules` 或执行后仍回退用户手改。
- 驳回：当前运行环境存在显式状态收口机制，原始用户目标已完成却只输出“等待用户新指令 / 任务已完成”文案，没有真实执行对应状态收口动作。
- 驳回：原始用户目标已完成，或用户明确要求结束，却仍默认追加“下一步状态 / 下一步建议 / 等待用户新指令 / 任务已完成，无需继续动作 / 当前无需补充动作”等占位或扩散性后续。
