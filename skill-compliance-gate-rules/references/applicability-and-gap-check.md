# 可执行性与缺口判定

## 目标

在不增加用户输出负担的前提下，稳定识别“可执行但未执行”的规则缺口：该结束就结束；没有下一步时强制不输出下一步区块或等待类占位，避免循环 loop 会话把占位文案误触发成新任务。

## 判定流程

1. 收集本轮命中的 skill 清单。
2. 对每个 skill 读取 `SKILL.md` 的主流程、阻断条件、通过标准、references 读取规则。
3. 抽取当前阶段规则（编码/审查/测试/交付）。
4. 按以下三类标记每条规则：
   - `已执行`：已有代码、测试、文档或结论证据。
   - `未执行但可执行`：当前信息与环境足够，理论上可以做但没做。
   - `当前阶段不适用`：规则应在其他阶段执行，或前置条件不成立。
5. 若本轮涉及测试相关 skill，执行禁放扫描：
   - 扫描命令示例：`rg --files -g \"*_test.go\"`
   - 判定命令示例（PowerShell）：`rg --files -g \"*_test.go\" | rg -v \"^doc/5-tests/\"`
   - 若命中结果非空，判定为“测试落点规则未完整执行”。
6. 形成两组候选建议：
   - `主任务建议候选`：来自当前任务的实现、验证、风险或交付收口动作。
   - `skill 补齐建议候选`：来自 `未执行但可执行` 的 skill 规则缺口。
7. 先判断收口状态：
   - 若原始用户目标已满足，或用户明确要求结束，且不存在阻断级缺口，默认直接结束，不输出任何下一步区块、等待类文案或扩散性后续。
   - 不得把可选优化、未来迭代、额外整理动作升级为当前必须继续执行的下一步。
8. 合并排序：
   - 先放主任务建议候选。
   - 再放 skill 补齐建议候选。
   - 仅在原始目标未完成且仍有必需动作，或用户明确要求后续建议时，去重后截取前 1-3 条作为最终“下一步状态”中的行动建议部分。

## 规则抽取优先级

优先抽取以下表达对应的规则：

- 包含“必须”“默认”“先做”“通过标准”“阻断”“不得”“至少”等词的条目。
- 位于“默认执行流程”“通过 / 驳回标准”“暂停条件”“归档要求”中的条目。

次级抽取：

- 包含“建议”“优先”“必要时”的条目（仅在当前任务相关时纳入）。

## 阻断级判定

判定为阻断级缺口：

- 缺口会直接导致实现不正确。
- 缺口会使测试结论无效或误导。
- 缺口会导致任务产物落错位置或缺关键留痕。

判定为非阻断级缺口：

- 缺口不影响当前结论正确性，但会影响后续维护效率或一致性。

## 结果约束

- 不把“当前阶段不适用”误报为缺口。
- 不要求一次性列出所有历史欠账，仅关注本轮可落地的缺口。
- 不因为 skill 缺口检查而丢弃主任务的关键建议。
- 若主任务与 skill 均无可执行缺口，且原始用户目标已满足或用户明确要求结束，应直接结束，而不是输出任何默认终止文案、“等待用户新指令”“无需继续动作”或“进入下一阶段”。
- 测试相关任务中，若 `doc/5-tests/` 外存在 `*_test.go`，不得判定为“无缺口”。
