# 下一步建议模板（简化版）

## 固定输出格式

```text
下一步建议：
1. ...
2. ...
3. ...
```

## 生成规则

1. 只输出 1-3 条，按优先级排序。
2. 每条建议必须是可执行动作，避免抽象口号。
3. 每条建议尽量包含动作对象与完成标志。
4. 建议应合并两类来源：主任务建议与 skill 补齐建议。
5. 排序时主任务建议优先，skill 补齐建议次优先。
6. 若主任务建议已占满 3 条，允许本轮不出现 skill 补齐建议。

## 文案要求

- 用动词开头，例如“补齐…/新增…/运行…/回写…/重新验证…”。
- 单条建议尽量控制在一行内，避免长段解释。
- 不强制输出“已完成规则/未完成规则/未完成原因”。
- 不因输出格式简化而丢失主任务关键建议。

## 示例

```text
下一步建议：
1. 先补齐订单超时场景的核心业务分支并补对应单测，确保主流程可正确关闭。
2. 重新执行本次变更的核心验证路径并记录结论，确认修复未引入回归。
3. 补齐 `api-response-rules` 要求的统一错误响应字段，保证接口输出一致性。
```

## 测试落点缺口示例

```text
下一步建议：
1. 先扫描并迁移 `test/` 之外的 `*_test.go`，统一放入中央测试根目录的 ASCII 镜像路径。
2. 对白盒诉求补齐可外部测试 seam（导出接口或测试钩子），不要在源码目录同包落地测试文件。
3. 迁移后重新执行测试并在任务 README 回写落点与验证结论。
```

## 注释闸门未通过示例

```text
下一步建议：
1. 先补齐本轮改动函数的 `[参数]` / `[返回]` 与最近修改时间（含改动原因），确保函数注释结构完整。
2. 将方法体中的流程型普通注释改为就近编号步骤注释（至少补 `1.`，多步骤补 `1.` `2.` `3.`），必要时补 `1.1` `1.2` 子步骤。
3. 补齐后重新执行 `comment-placement-granularity-rules` 与 `comment-completion-gate-rules` 闸门复核，再进入最终回复收口。
```

## 代码改动收口通过示例

```text
下一步建议：
1. 当前代码改动的注释补齐与 skill 闸门检查均已通过，可进入下一阶段。
```

## 无缺口示例

```text
下一步建议：
1. 当前阶段无需补充动作，可进入下一阶段。
```
