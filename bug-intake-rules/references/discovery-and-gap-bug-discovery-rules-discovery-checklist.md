# Bug 主动侦察清单

## 侦察入口

- 现象与报错：用户原话、报错文案、异常堆栈、HTTP 状态码、接口返回体、页面提示。
- 截图与资料：用户提供的页面截图、报错截图、控制台截图、网络请求截图、数据截图。
- 长期记忆：`PROJECT_MEMORY.md`（历史同类问题、相关表、相关代码入口、本地配置位置）、`PROJECT_STYLE.md`。
- 历史记录：`doc/4-bugs/`（同类 Bug 根目录）、`doc/2-需求/`、`doc/3-实施/`、`doc/6-审查/`。
- 代码入口：路由、controller、service、repository、model、frontend 页面、任务脚本、中间件、错误处理与日志埋点。
- 调用链：联动 `codegraph-analysis-rules` 追上游调用方、下游被调用方、可疑分支与状态流转。
- 日志与 trace：现有运行日志、错误日志、链路 trace、关键埋点输出。
- 数据入口：数据库连接配置（本地）、表名、字段、索引、ORM schema、相关 SQL、迁移脚本。
- 配置入口：本地 / 开发配置文件、特性开关、环境变量、依赖版本。
- 上下游入口：HTTP client、SDK、第三方服务配置、webhook、队列、定时任务、RPC 调用。

## 侦察顺序

1. 先读现象、报错与截图，提取最具体的错误信号（文案、堆栈行、状态码、错误数据值）。
2. 再查长期记忆与历史 Bug，避免重复定位已经沉淀过的同类问题。
3. 再静态看代码：从报错信号反查代码位点，联动 CodeGraph 追调用链，定位可疑逻辑、分支和状态转换。
4. 再判断是否数据问题：需要时按「本地环境配置发现与连接」只读连本地库，查真实数据与期望值对比。
5. 再查配置、依赖与上下游，确认是否环境 / 版本 / 外部服务导致。
6. 最后用截图与日志二次印证根因候选，排除“看着像但证据不足”的假设。

## 连库取证最小动作（只读）

- 先从本地配置文件拿到本地库连接信息（不外泄明文密钥）。
- 用 `SELECT` / `SHOW` / `EXPLAIN` / `DESCRIBE` 查目标表结构与相关行数据。
- 把“期望数据”与“实际数据”逐字段对比，定位是数据错、写入错还是读取错。
- 大表加条件 / `LIMIT`，记录执行过的只读 SQL 与结论（脱敏）。
- 需要改数据才能验证时，停手，改为产出 SQL 脚本交用户手动执行。

## 最小成果

- 现象复述与 agent 当前理解。
- 已侦察来源清单（代码、日志、数据、截图、配置）。
- 根因候选列表，每条带证据与证据等级。
- 连库只读查询与数据对比结论（脱敏）。
- 仍需用户确认的最少问题。
- 下一步应转交的 Bug 域 skill 与进入条件。
- 需要回写长期记忆的可复用线索。
