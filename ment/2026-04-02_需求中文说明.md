## 3. 项目上下文

- 当前仓库是一个团队研发协作 skill 仓库。
- `skills拆分.md` 不是一个已落地的 skill 目录，而是一份“Skill 拆分 Skill 设计（功能不删减版）”的设计草案。
- 这份文档的目标，是设计一个建议名为 `skill-split-preserve-rules` 的独立 skill，用来把已有 skill 在“功能不删减”的前提下拆分成多个独立 skill。
- 文档当前已经具备较完整的结构，包括：
  - 目标
  - 触发条件
  - 核心约束
  - 分类二分拆分策略
  - 特例规则保护
  - 输入输出
  - 标准流程
  - 功能覆盖映射表
  - 命名规则
  - 验收标准
  - 阻断条件
  - 回滚策略
  - 推荐描述草案

## 4. 当前需求目标

分析 `skills拆分.md` 在当前状态下还能如何继续优化，重点明确：

- 这份设计文档目前已经做得好的地方
- 目前最关键的缺口和含糊点是什么
- 如果继续优化，应优先优化哪些章节
- 哪些优化属于“可直接补文档”，哪些优化要等真正开始实现该 skill 时再落地

## 5. 背景动因

继续优化这份文档的意义，主要在于解决以下风险：

- 文档虽然结构完整，但部分规则仍偏“原则正确、执行难落地”
- “分类二分”虽然是核心策略，但缺少更细的判定方法，实际执行时容易出现主观分组
- 触发条件目前偏宽，可能导致本 skill 在普通小修场景中过度触发
- 功能覆盖映射表和特例保护机制已经提出，但还缺少更具体的口径和模板
- 旧 skill 删除前若承接检查不充分，容易出现“旧 skill 已删，但新 skill 承接不完整”的断层

## 6. 当前文档的优点

从设计完整度看，`skills拆分.md` 已经有几个明显优点：

### 6.1 原则很稳

- 明确把“功能不删减”和“原规则零丢失”放在最高优先级。
- 对“看起来奇怪的低频规则”给出了强保留态度，能有效防止拆分时把历史兼容规则静默吞掉。

### 6.2 主策略清晰

- 明确提出“分类二分拆分优先”，避免一上来过细拆分。
- 给了一个可理解的 `test-location` 示例，说明不是只停留在抽象口号。

### 6.3 过程意识较完整

- 已经覆盖基线采集、规则原子化、拆分设计、独立化落地、覆盖校验、收口策略六个阶段。
- 已经意识到 references / scripts / assets 不能丢，这是很多拆分设计文档容易忽略的点。

### 6.4 风险控制意识较强

- 已包含验收标准、阻断条件、回滚策略。
- 已要求 100% 覆盖映射，这是很重要的防漏机制。

## 7. 当前最关键的缺口

当前继续优化时，最值得优先处理的不是“再补更多原则”，而是把几个执行口径补得更可落地。

### 7.1 触发边界仍偏宽

当前触发条件写的是：

- 对某个现有 skill 做“新增规则/补充规则/优化规则”
- 发现某个 skill 的职责混合、触发歧义、正文过长，准备拆分
- 需要把一个 skill 的多类职责拆成多个平级独立 skill

问题：

- 第一条太宽，几乎所有有效规则更新都可能命中。
- 缺少“什么程度算值得启动拆分”的门槛。

优化方向：

- 增加拆分触发门槛，例如至少满足以下之一才进入本 skill：
  - 同一 skill 已出现两个以上可独立命中的并列职责组
  - `SKILL.md` 与 references 总量已超过既定阈值
  - description 已明显覆盖多个互不相同的触发对象
  - 本轮新增规则已无法在原边界内自然收纳

### 7.2 “分类”识别方式不够具体

当前文档要求优先识别“并列类别集合”，但没有定义：

- 类别如何识别
- 如何证明某两组真的是并列类别而不是强耦合步骤
- 如何避免人为主观拆分

优化方向：

- 在文档中增加“分类识别清单”，例如：
  - 触发对象是否能自然并列罗列
  - 每个类别是否都能独立写出“当...时触发”
  - 类别之间是否共享同一主流程但处理对象不同
  - 拆开后是否仍能分别独立验证

### 7.3 缺少“何时不该继续拆”的硬约束

现在文档强调“可拆就继续分类二分”，但还缺一个反向约束：

- 什么情况下已经不该继续拆

优化方向：

- 增加停止条件，例如：
  - 某一组已无法形成新的并列类别
  - 再拆会让新 skill 只剩微小语义差异
  - 拆后平均命中 skill 数将稳定超过 3
  - 拆后必须依赖父 skill 上下文才能理解职责

### 7.4 规则原子化口径不够细

文档要求“把原规则按触发对象拆成最小职责单元”，但没有明确：

- 一条规则的最小粒度是什么
- 一段话中多个“必须/默认/阻断”是否要拆成多条
- 示例、说明性约束、兼容说明如何计数

优化方向：

- 增加规则原子化口径，例如：
  - 每个“必须/默认/阻断/通过标准”句默认单独成条
  - 一条规则若同时包含触发、动作、例外，应拆成主规则 + 例外规则
  - 示例默认不算主规则，但若承载兼容语义，则必须单独入表

### 7.5 覆盖映射表仍缺执行模板

当前已经有映射表字段设计，但还缺更可操作的要求：

- 原规则 ID 如何生成
- 一个原规则可否映射到多个新 skill
- 哪些情况算“等价改写”
- 哪些情况必须写“增强补充说明”

优化方向：

- 增加统一口径，例如：
  - 规则 ID 按章节顺序生成，如 `R-3.1-001`
  - 一条原规则允许映射到多个新 skill，但必须指定主落点
  - 只要触发条件或约束强度变化，就必须写“等价性说明”
  - 若新 skill 通过多条规则共同覆盖原规则，也必须给出组合覆盖说明

### 7.6 删除收口策略需要改写

上一轮文档写到：

- 旧 skill 进入兼容期时，只保留迁移说明，不再新增规则
- 兼容期结束后下线旧 skill

问题：

- 这与最新要求“拆分完成后旧 skill 会被删除”冲突
- 原先的兼容状态机和兼容期命中路由模板不再是最终策略
- 现在真正需要的是：删除前如何确认新 skill 已完整承接

优化方向：

- 改成删除收口策略，例如：
  - `active`：原 skill 正常使用
  - `splitting`：正在拆分，禁止继续新增无关规则
  - `deleted`：覆盖校验通过后删除旧 skill
- 同时要求删除前必须有“删除后由谁承接”的检查模板

## 8. 可行优化方案

### 8.1 方案 A：只补文案定义

内容：

- 只补触发边界、分类定义、停止条件和映射口径

优点：

- 成本低
- 可以快速提升文档可读性

风险：

- 对未来真正实现 skill 的帮助仍有限

### 8.2 方案 B：补文案定义 + 增加执行模板

内容：

- 在现有文档基础上，补：
  - 触发门槛
  - 分类识别清单
  - 停止拆分条件
  - 规则原子化口径
  - 映射表填写规则
  - 删除状态机
  - 删除前承接检查模板

优点：

- 仍属于文档层优化
- 但已经能显著降低未来落地时的主观性

风险：

- 文档会进一步变长，需要注意不要压回“大而全”状态

### 8.3 方案 C：直接推进为正式 skill 规格

内容：

- 不只是补文档，而是直接开始规划 `SKILL.md`、`references/`、模板文件和可能的脚本

优点：

- 最接近可执行落地

风险：

- 当前还缺很多细粒度口径，直接落地容易边做边改

### 8.4 推荐结论

当前更推荐方案 B。

原因：

- 目前最缺的是“如何执行”的中间口径，而不是新的大原则
- 先把文档从“方向正确”补到“执行可判定”，再考虑是否正式落 skill，更稳
