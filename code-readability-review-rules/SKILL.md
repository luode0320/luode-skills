# 可读性检查规则

## 目标

只检查当前实现是否需要交给 `code-readability-rules` 处理，不执行可读性改写。

## 源规则

以下判定严格沿用源 skill：

- 主流程清晰、关键分支可追、职责相对单一
- 500 行及以上且继续扩展时必须触发拆分评估
- 业务方法禁止直接用 JSON 字符串 key 取字段
- Go 第三方 API 默认使用结构体解析响应，禁止长期使用 map + key 硬编码
- 已有公共工具优先复用，禁止同语义重复封装
- 简单场景禁止无脑新增接口层、接口实现对或单实现接口包装
- Go 函数签名默认单行参数，过多时优先参数结构体
- 格式化日志优先单行，必要时才按模板一行、参数下一行收口

## 审查原则

1. 只读审查，不改文件、不写文件。
2. 只复核当前实现是否满足源 skill 的可读性标准，不发明新的“更优雅”规则。
3. 如果问题已经超出可读性，转交最小改动、风格一致性或包结构规则，不在这里自造重构方案。
4. 输出必须能回流到 `code-readability-rules` 的改写动作。

## 输入

- 当前 diff
- 目标文件列表
- 现有可读性约定
- 源 skill 的可读性约定

## 输出

- 可读性问题清单
- 优先级
- 应交给 `code-readability-rules` 的改进建议
- 需要回流的位点

## 边界

- 不替代 `code-readability-rules`
- 不执行写入型改写
- 不处理风格统一本身

## 自动触发信号

- 新增业务函数、服务方法、控制器逻辑、工具函数。
- 修改已有复杂逻辑。
- 修复 Bug 后需要重排逻辑顺序或拆小函数。
- 编写脚本或任务流程代码时需要控制步骤可读性。
- 同一文件超过 500 行且本轮还在新增方法、分支或流程步骤。
- Go 接入第三方 API 且文档仅有 curl 示例、缺少响应字段说明。

## 进入后先做什么

1. 先识别当前实现的主流程。
2. 确认读代码的人能否快速看出输入、关键分支和输出。
3. 找出过深嵌套、职责混杂、命名模糊、控制流跳跃、嵌套三元和过密表达的位置。
4. 判断哪些地方是真正影响理解的冗余抽象或包装。
5. 在不扩大改动范围、且功能完全不变的前提下优化结构表达。
6. 若本轮目标文件已达 500 行及以上，先按“功能域/流程阶段/上下游边界”给出最小拆分方案。

## 默认执行流程

1. 先读取 `code-readability-rules`。
2. 如果涉及函数拆分、步骤组织和控制流安排，再读取对应 references。
3. 如果需要判断某种写法是否虽然可运行但可读性很差，再读取 `readability-examples`。
4. 先保证主流程直观，再处理局部细节；清晰优先于短小，显式优先于炫技。
5. 如果改善可读性会明显扩大改动范围，暂停并交还给 `code-minimal-change-rules` 判断。
6. 若目标文件已达 500 行及以上且仍持续膨胀，优先拆分到职责更清晰的文件；业务流程明显增多时，联动 `package-structure-rules` 评估子目录/子包拆分。

## 需要暂停并确认的条件

- 为了提升可读性必须做大范围重构。
- 当前实现牵涉多个职责，拆分后会明显改变模块边界。
- 为了让代码更清晰，必须同步调整接口、数据结构或目录归属。
- 用户更关心快速止血，不接受当前阶段做结构梳理。
- 目标文件已超过 500 行，但拆分将引发跨模块接口重排或发布风险，需要先确认拆分边界。

## 执行通过 / 驳回标准

- 通过：主流程清晰、关键分支可追、函数职责相对单一、读者能快速理解执行顺序，且没有用更短但更难读的写法替代清楚表达；对 500 行及以上且继续扩展的文件已完成拆分或给出本轮可执行拆分方案；改动位点注释补齐已联动完成；同语义能力优先复用既有公共工具，无重复封装。
- 驳回：虽然代码能跑，但逻辑跳跃、嵌套过深、职责混杂、阅读成本过高，或为了“更优雅”而引入更难懂的抽象、嵌套三元和过密一行表达；或在简单场景里无真实多实现需求却强行新增接口层、接口实现对或多一跳包装；或 500 行及以上文件仍持续堆功能却未做拆分评估；或结构改动后未联动注释补齐；或已存在公共工具却重复封装同语义函数。

## 执行结果归档要求

- 如果为了控制可读性而主动放弃某些重构点，将结论记录到 `analysis/` 或 `review/`。
- 归档内容至少包含当前可读性问题、保留原因和后续建议。
- 如果本次只做常规可读性约束且无争议，可以不单独归档。

## 输出格式

- 写出影响可读性的文件和位点。
- 说明是结构、顺序、条件层级、复杂度、JSON key 直取、第三方响应解析还是接口抽象问题。
- 明确建议交给 `code-readability-rules` 改写。

## 读取规则

- 默认先读 `code-readability-rules`。
- 若本轮牵涉到 500 行以上文件、第三方 API 结构体解析或接口抽象，再按源 skill 的默认流程继续读对应 references。
