# 测试目录落点规则

只在"测试资产应该放到哪里"这个问题上使用这个 skill。
如果当前争议是测试名称怎么起，请转交 `test-naming-rules`；如果是测试程序怎么写，请转交 `test-program-rules`；如果是测试说明怎么组织，请转交 `test-doc-rules`。

## 铁律：所有测试资产必须统一在 test/ 目录下

**绝对禁止任何例外！** 以下所有与测试相关的内容，无论形式如何、用途如何、临时与否，**必须统一放在 `test/` 目录下**：

- 测试脚本、测试程序、验证代码
- Mock 数据、Fixture 数据、测试输入输出样例
- 测试说明文档、验证记录、测试结论
- 临时测试文件、临时验证脚本
- 测试用例定义、测试场景描述
- 仅供测试使用的常量、辅助函数、伪造实现
- 测试截图、测试日志、测试报告
- 任何其他与测试相关的数据文件或说明文件

**任何测试相关的文件，只要发现不在 `test/` 目录下，立即迁移到正确位置。**

## Skill 作用与适用场景

- 作为测试资源管理链的第一道规则，先稳定测试资产落点。
- 统一测试资源根目录、任务目录的使用边界。
- 防止测试脚本、模拟数据、验证代码和测试说明散落到业务目录、仓库根目录或无归属路径。
- 为后续测试命名、测试程序编写和测试文档整理提供稳定位置基础。

## 自动触发信号

- 新增或修改测试目录、测试文件、测试脚本、验证程序、fixture、mock 数据、`testdata`、`__tests__`、`e2e`、`integration` 等测试资源。
- 需要决定新的测试资产应放在项目级测试根目录下的哪个任务目录。
- 发现测试资源散落在业务目录、仓库根目录、临时目录或文档目录中。
- 在编码审查通过后，准备进入测试资源管理阶段时，需要先确认测试资产归属位置。
- 一个会话中出现多次不同功能的测试需求时，每次都需要创建独立的测试目录。

## 进入后先做什么

1. 先确认当前改动是否属于测试资源，而不是生产代码归属问题。
2. 区分当前对象属于测试代码、测试数据、测试脚本、测试文档还是公共测试辅助资源。
3. 确认使用统一的 `test/` 根目录作为测试资产存放位置。
4. 为当前测试任务创建独立的子目录，命名格式为 `yyyy-MM-DD_HHmmss_测试任务中文简介`。
5. 如果同一需求需要多个测试程序或多轮独立验证，分别创建多个测试任务子目录。

## 默认执行流程

1. 默认先读 `references/test-root-and-task-folders.md`，确定测试根目录和任务目录划分规则。
2. 如果发现测试资产散落、混放到业务目录或根目录，再读 `references/forbidden-scattered-assets.md`。
3. 如果需要对照正反例或判断目录结构，再读 `references/location-examples.md`。
4. 输出推荐落点、禁止落点、是否需要迁移已有资源，以及迁移时的最小改动建议。
5. 确保每个测试任务子目录都包含 README.md，说明本次测试的目的、执行方式、依赖数据、覆盖范围和验证结论。
6. 测试资源落点未稳定前，不进入 `test-naming-rules`、`test-program-rules` 和 `test-doc-rules`。

## 权责边界与不负责事项

- 只负责测试资源位置与归属，不负责测试名称本身。
- 不负责测试程序的实现细节、脚手架结构和断言写法，那属于 `test-program-rules`。
- 不负责测试说明文档内容和章节结构，那属于 `test-doc-rules`。
- 不负责功能是否正确、覆盖是否充分，那属于 `functional-validation-rules` 和 `test-regression-rules`。
- 如果争议本质是生产代码放错目录、模块分层错误或跨层依赖问题，应转交 `code-placement-review-rules`。
- 不负责旧测试目录的归档操作，但会提醒以每周日为时间间隔进行归档。

## 需要暂停并确认的条件

- 仓库中还没有 `test/` 根目录，需要确认是否创建。
- 同一批测试资源既可能做项目级公共复用，也可能只是当前任务私有，归属不清。
- 现有历史测试资产分布混乱，需要大范围迁移才能统一落点。
- 为了整理测试目录需要顺带改动大量无关文件，已经偏离最小改动原则。

## 执行通过 / 驳回标准

- 通过：当前测试资产已经明确落入 `test/` 根目录下的独立任务子目录中，且不存在明显散落、混放或归属冲突问题；每个任务子目录都包含 README.md。
- 驳回：测试资源仍然散落在业务代码目录、仓库根目录或无归属位置，继续进入后续测试环节会放大维护和排查成本。

## 执行结果归档要求

- 将测试资源落点结论记录到当前任务测试目录下的 README.md 中。
- 归档内容至少包含测试目的、执行方式、依赖数据、覆盖范围和验证结论。
- 如果只是轻量调整，也应在任务记录、评审记录或测试说明中保留最终落点结论。
- 提醒用户旧的测试目录以每周日为时间间隔，归档到 `test/` 目录下（可创建 `archive/` 子目录存放归档内容）。

## references 读取规则

- 默认先读 `references/test-root-and-task-folders.md`。
- 只有在发现散落、混放和迁移争议时，再读 `references/forbidden-scattered-assets.md`。
- 只有在需要正反例和项目结构对照时，再读 `references/location-examples.md`。
