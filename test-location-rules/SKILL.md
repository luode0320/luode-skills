# 测试目录落点规则

只在"测试资产应该放到哪里"这个问题上使用这个 skill。
如果当前争议是测试名称怎么起，请转交 `test-naming-rules`；如果是测试程序怎么写，请转交 `test-program-rules`；如果是测试说明怎么组织，请转交 `test-doc-rules`。

## 铁律：所有测试资产必须统一在 `artifact-storage-rules` 定义的测试根目录下

**绝对禁止任何例外！** 以下所有与测试相关的内容，无论形式如何、用途如何、临时与否，**必须统一放在 `artifact-storage-rules` 定义的测试根目录下**：

- 测试脚本、测试程序、验证代码
- Mock 数据、Fixture 数据、测试输入输出样例
- 测试说明文档、验证记录、测试结论
- 临时测试文件、临时验证脚本
- 测试用例定义、测试场景描述
- 仅供测试使用的常量、辅助函数、伪造实现
- 测试截图、测试日志、测试报告
- 任何其他与测试相关的数据文件或说明文件

**任何测试相关的文件，只要发现不在中央约定的测试根目录下，立即迁移到正确位置。**

## Go 冲突决策分支：白盒同包诉求与禁放规则

- 统一规则不变：源码目录绝对不允许出现 `*_test.go`。
- 白盒单测需要同包访问未导出符号时，不允许以“同包落地”作为特例。
- 处理方式是改生产代码 seam（导出接口、可注入依赖、测试钩子等），再让测试资产继续落在中央测试根目录的 ASCII 镜像路径。
- 该分支是默认规则的一部分，不是特例通道。

## 新基线：时间戳根目录 + 中文说明目录 + 真实代码路径镜像

每次新增测试任务，统一使用下面的目录结构：

```text
<artifact-storage-rules 定义的测试根目录>/
└── <时间戳根目录>/
    ├── 测试任务中文简介/
    │   └── README.md
    └── <真实代码路径镜像，必须保持 ASCII>
```

- 中央约定的测试时间戳根目录是当前测试任务的唯一根目录，目录名只允许时间戳，不再把中文直接拼进根目录名。
- 中央约定的测试任务主说明 `README.md` 只负责中文说明；这个中文目录默认只存放这一个 `README.md`。
- 真实测试代码、测试脚本、mock、fixture、截图、日志、报告产物等，都放在同一个时间戳根目录下，按真实代码路径镜像组织，并保持编译路径为 ASCII。
- 对于会被 Go 编译链路直接扫描的目录，例如会参与 `go test ./...` 的包目录，绝对不能把中文放进编译路径；中文只能出现在说明目录中，不能出现在 Go 包目录路径中。

## Skill 作用与适用场景

- 作为测试资源管理链的第一道规则，先稳定测试资产落点。
- 统一测试资源根目录、时间戳任务根目录、中文说明目录和真实代码路径镜像目录的使用边界。
- 防止测试脚本、模拟数据、验证代码和测试说明散落到业务目录、仓库根目录或无归属路径。
- 防止中文目录名进入 Go 编译路径，破坏 `go test ./...`。
- 为后续测试命名、测试程序编写和测试文档整理提供稳定位置基础。

## 自动触发信号

- 新增或修改测试目录、测试文件、测试脚本、验证程序、fixture、mock 数据、`testdata`、`__tests__`、`e2e`、`integration` 等测试资源。
- 需要决定新的测试资产应放在项目级测试根目录下的哪个任务目录。
- 发现测试资源散落在业务目录、仓库根目录、临时目录或文档目录中。
- 在编码审查通过后，准备进入测试资源管理阶段时，需要先确认测试资产归属位置。
- 一个会话中出现多次不同功能的测试需求时，每次都需要创建独立的测试目录。
- 当前测试会参与 Go 工具链编译，需要避免中文路径进入包目录或 import path。
- 当前计划为了白盒访问未导出符号，尝试在源码目录新增同包 `*_test.go`。

## 进入后先做什么

1. 先确认当前改动是否属于测试资源，而不是生产代码归属问题。
2. 区分当前对象属于测试代码、测试数据、测试脚本、测试文档还是公共测试辅助资源。
3. 确认使用 `artifact-storage-rules` 定义的统一测试根目录作为测试资产存放位置。
4. 为当前测试任务创建独立的时间戳根目录，命名格式为 `yyyy-MM-DD_HHmmss`。
5. 在时间戳根目录下创建中文说明目录 `测试任务中文简介/`，且该目录默认只存放 `README.md`。
6. 将真实测试代码、脚本、mock、fixture、截图和报告产物，按真实代码路径镜像放在同一个时间戳根目录下的 ASCII 目录中。
7. 如果同一需求需要多个测试程序或多轮独立验证，分别创建多个时间戳根目录。
8. 对 Go 场景额外检查：源码目录是否出现 `*_test.go`；若有则判定落点不合规并改为 seam 方案。

## 默认执行流程

1. 默认先读 `../artifact-storage-rules/references/path-map.yaml`、`../artifact-storage-rules/references/naming-templates.md` 与 `../artifact-storage-rules/references/update-policy.md`，确定当前有效的测试根目录、时间戳根目录、中文说明目录规则，以及同一轮测试是复用现有根目录还是新建根目录。
2. 再读 `references/test-root-and-task-folders.md`，判断测试资源如何映射到中央约定结构中，并应用 Go 白盒冲突决策分支。
3. 如果发现测试资产散落、混放到业务目录或根目录，再读 `references/forbidden-scattered-assets.md`。
4. 如果需要对照正反例或判断目录结构，再读 `references/location-examples.md`。
5. 输出推荐落点、禁止落点、是否需要迁移已有资源，以及迁移时的最小改动建议。
6. 确保每个时间戳根目录都包含一个中文说明目录，且其中的 `README.md` 说明本次测试的目的、执行方式、依赖数据、覆盖范围和验证结论。
7. 测试资源落点未稳定前，不进入 `test-naming-rules`、`test-program-rules` 和 `test-doc-rules`。

## 权责边界与不负责事项

- 只负责测试资源位置与归属，不负责测试名称本身。
- 不负责测试程序的实现细节、脚手架结构和断言写法，那属于 `test-program-rules`。
- 不负责测试说明文档内容和章节结构，那属于 `test-doc-rules`。
- 不负责功能是否正确、覆盖是否充分，那属于 `functional-validation-rules` 和 `test-regression-rules`。
- 如果争议本质是生产代码放错目录、模块分层错误或跨层依赖问题，应转交 `code-placement-review-rules`。
- 不负责旧测试目录的归档操作，但会提醒以每周日为时间间隔进行归档。
- 不负责测试文件具体如何命名，但会强制要求 Go 可编译路径保持 ASCII。

## 需要暂停并确认的条件

- 仓库中还没有中央约定的测试根目录，需要确认是否创建。
- 同一批测试资源既可能做项目级公共复用，也可能只是当前任务私有，归属不清。
- 现有历史测试资产分布混乱，需要大范围迁移才能统一落点。
- 为了整理测试目录需要顺带改动大量无关文件，已经偏离最小改动原则。
- 测试目录准备使用中文路径承载 Go 可编译包，可能直接破坏 `go test ./...`。

## 执行通过 / 驳回标准

- 通过：当前测试资产已经明确落入中央约定的测试根目录下的独立时间戳根目录中；该根目录包含中文说明目录和 `README.md`，同时真实测试资产按真实代码路径镜像组织；会被 Go 编译的路径保持 ASCII；Go 源码目录不存在 `*_test.go`；不存在明显散落、混放或归属冲突问题。
- 驳回：测试资源仍然散落在业务代码目录、仓库根目录或无归属位置；时间戳根目录结构不完整；中文路径进入 Go 可编译目录；或 Go 源码目录出现 `*_test.go` 并试图以白盒同包作为例外，继续进入后续测试环节会直接放大维护和自动化测试风险。

## 执行结果归档要求

- 将测试资源落点结论记录到 `artifact-storage-rules` 约定的测试任务主说明 `README.md` 中。
- 归档内容至少包含测试目的、执行方式、依赖数据、覆盖范围、验证结论，以及真实测试资产所在的镜像路径。
- 测试任务主说明位置、目录命名模板和同一轮测试的复用策略统一遵循 `../artifact-storage-rules/references/path-map.yaml`、`../artifact-storage-rules/references/naming-templates.md` 与 `../artifact-storage-rules/references/update-policy.md`。
- 如果只是轻量调整，也应在上述 `README.md` 中保留最终落点结论。
- 提醒用户旧的时间戳根目录以每周日为时间间隔，归档到中央约定的测试根目录下（可创建 `archive/` 子目录存放归档内容）。

- 再读 `references/test-root-and-task-folders.md`。
