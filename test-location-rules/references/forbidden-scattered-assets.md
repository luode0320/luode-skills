# 禁止散落规则

## 用途

用于识别测试资源落点错误、散落混放和伪装成业务代码的测试资产。

## 铁律：所有测试资产必须统一在 test/ 目录下

**绝对禁止任何例外！** 以下所有与测试相关的内容，无论形式如何、用途如何、临时与否，**必须统一放在 `test/` 目录下**：

- 测试脚本、测试程序、验证代码
- Mock 数据、Fixture 数据、测试输入输出样例
- 测试说明文档、验证记录、测试结论
- 临时测试文件、临时验证脚本
- 测试用例定义、测试场景描述
- 仅供测试使用的常量、辅助函数、伪造实现
- 测试截图、测试日志、测试报告
- 任何其他与测试相关的数据文件或说明文件

## 明确禁止

- 把测试脚本直接放到仓库根目录或临时目录。
- 把 mock 数据、fixture、测试输入输出样例放进生产代码目录。
- 把一次性验证脚本长期留在业务模块根目录下。
- 把测试说明文档散落到普通业务文档目录，导致无法和对应测试资产对应。
- 把仅供测试使用的常量、辅助函数、伪造实现塞进生产 `utils`、`common`、`service` 等目录。
- 在 `src/`、`app/`、`lib/` 等真实业务代码目录创建任何临时测试程序、测试脚本或补充验证代码。
- 使用 `tests/`、`qa/`、`verify/`、`tmp-test/` 等非 `test/` 名称作为测试根目录。
- 在 `docs/`、`wiki/` 等业务文档目录存放测试说明或测试记录。
- 在 `scripts/`、`tools/` 等目录存放临时测试脚本而不放入 `test/`。
- 在项目根目录直接创建任何测试相关文件或目录。

## 常见风险

- 后续维护者无法判断文件是正式业务代码还是测试辅助代码。
- 测试资源位置分散，导致回归验证和问题复现时找不到对应资产。
- 业务目录被测试残留污染，增加误引用和误发布风险。
- 目录迁移成本被拖大，后续统一治理代价更高。
- 违反团队统一的测试目录规范，造成协作混乱。

## 处理原则

- 优先把散落资产归并到 `test/` 目录下的对应任务目录中。
- 迁移时只移动与当前任务直接相关的测试资产，避免借机大扫除。
- 如果散落问题已经涉及大量历史资产，先记录待迁移清单，不在当前任务中无限扩张。
- 如果某文件兼具生产和测试双重职责，先拆分职责，再分别归位。
- 为迁移的测试资产创建符合规范的任务目录，包含 README.md。
- **任何测试相关的文件，只要发现不在 `test/` 目录下，立即迁移到正确位置。**

## 散落资产识别清单

- 文件名包含 `test`、`spec`、`mock`、`fixture`、`demo`、`example`、`verify`、`validate` 等关键词但不在 `test/` 目录下。
- 包含测试断言、测试框架引用的文件放在业务代码目录。
- 临时验证脚本放在 `scripts/`、`tmp/`、`temp/` 等目录且长期未清理。
- 测试数据文件（JSON、CSV、XML 等）放在业务模块目录中。
- 测试说明文档、验证记录放在 `docs/` 或其他业务文档目录中。
- 任何与测试相关的文件，只要不在 `test/` 目录下，就是违规的。
