# 测试根目录与任务目录规则

## 用途

用于先确定测试资源应该挂在哪一级目录，再决定后续命名和文件组织方式。

## 根目录判定顺序

1. 先遵循项目现有测试目录约定，不新造平行目录。
2. 如果项目已有统一测试根目录，优先放入统一测试根目录。
3. 如果项目长期采用模块就近测试目录，优先放入模块内测试目录。
4. 只有在现有约定无法承载当前测试资源时，才新增任务级子目录。

## 常见目录层级

- 项目级测试根目录：适合放集成测试、端到端测试、共享 fixture、共享 mock、通用测试工具。
- 模块级测试目录：适合放与单个模块、单个组件强绑定的单元测试和局部测试数据。
- 任务级测试目录：适合放当前需求、当前 Bug、当前联调用例专属的测试程序、数据和说明。

## 放置原则

- 可跨模块复用的测试辅助资源，放到项目级公共测试目录。
- 仅服务单个模块的测试资源，放到对应模块测试目录。
- 只服务当前需求或当前 Bug 的验证资产，放到当前任务目录，避免污染公共目录。
- 测试代码、测试数据、测试说明可以同属一个任务目录，但不要与生产代码混放。

## 检查清单

- 当前项目是否已有稳定测试根目录。
- 当前资源是否属于公共复用还是任务私有。
- 当前目录是否便于后续 `test-naming-rules` 和 `test-doc-rules` 继续套用。
- 是否能让排查者快速从任务或模块定位到对应测试资产。
