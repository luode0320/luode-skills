# 测试根目录与任务目录规则

## 用途

用于先确定测试资源应该挂在哪一级目录，再决定后续命名和文件组织方式。

## 根目录约定

- **强制使用 `test/` 作为统一测试根目录**，所有测试资产必须存放在项目根目录的 `test/` 下。
- 不得在真实业务代码目录创建临时测试程序、测试脚本或补充验证代码。
- 对 Go 项目，真实业务源码目录（如 `internal/`、`common/`、`api/`、`cmd/`）不得出现 `*_test.go`。
- 不得使用 `tests/`、`qa/`、`verify/`、`tmp-test/` 等其他名称作为测试根目录。

## Go 白盒冲突统一处理

- 白盒单测需要访问未导出符号时，仍不允许在源码目录同包落地 `*_test.go`。
- 统一处理方式：先在生产代码引入可外部测试 seam（导出接口、依赖注入点、测试钩子），再把测试资产放在 `test/<时间戳>/` 的 ASCII 镜像路径下。
- 这是统一规则，不是临时特例。

## 时间戳根目录命名规范

每次新增测试任务，必须在 `test/` 下创建独立的时间戳根目录，命名格式为：

```text
yyyy-MM-DD_HHmmss
```

示例：
- `2026-03-28_143022`
- `2026-03-28_154511`
- `2026-03-28_162033`

## 中文说明目录规则

每个时间戳根目录下必须创建一个中文说明目录，目录名就是本次测试任务的中文简介，例如：

```text
test/2026-03-28_143022/用户登录功能验证/README.md
```

规则如下：
- 中文说明目录默认只存放这一个 `README.md`。
- `README.md` 用于说明本次测试的目的、执行方式、依赖数据、覆盖范围和验证结论。
- 中文目录只承载说明，不承载真实测试代码、测试脚本、mock、fixture 或任何会参与 Go 编译的资产。

## 真实代码路径镜像规则

所有真实测试资产都放在同一个时间戳根目录下，并按真实代码路径镜像组织，保持 ASCII 路径。例如：

- 被测源码是 `internal/service/history_client.go`
- 对应测试资产放在 `test/2026-03-28_143022/internal/service/` 下
- 其中测试文件可以命名为 `history_client_test.go`、`history_client_mock.json`、`history_client_cases.md`

补充要求：
- 参与 Go 编译的目录路径必须保持 ASCII，不能把中文目录放进 `go test ./...` 会扫描到的包目录中。
- 中文说明目录与真实测试资产目录并列存在，不能互相替代。
- 如果当前测试对象无法精确映射到单一源码文件，至少按最接近的真实代码模块路径镜像，仍然保持 ASCII。

## 多测试任务处理规则

- 若同一需求需要多个测试程序或多轮独立验证，应分别创建多个时间戳根目录。
- 在一个会话中一个提问可能会出现多次不同功能的测试，都需要建立不同的测试目录。
- 每个时间戳根目录仅存放对应测试任务的脚本、代码、用例、模拟数据和验证说明，不得散落到其他目录。

## 时间戳根目录内容要求

每个时间戳根目录至少应包含：
- 一个中文说明目录，且其中有 `README.md`
- 一个或多个按真实代码路径镜像组织的 ASCII 测试资产目录

其中 `README.md` 说明内容包括：
- 本次测试的目的
- 执行方式
- 依赖数据
- 覆盖范围
- 验证结论
- 真实测试资产路径

## 旧测试目录归档规则

- 旧的时间戳根目录以每周日为时间间隔，归档到 `test/` 目录下。
- 建议在 `test/` 下创建 `archive/` 子目录，按周或按月组织归档内容。
- 归档目录命名建议：`archive/2026-W13/`、`archive/2026-03/` 等。

## 常见目录结构示例

```
项目根目录/
├── src/                    # 业务代码目录（禁止存放测试资产）
├── test/                   # 统一测试根目录
│   ├── 2026-03-28_143022/
│   │   ├── 用户登录功能验证/
│   │   │   └── README.md   # 只放说明
│   │   └── internal/
│   │       └── service/
│   │           ├── history_client_test.go
│   │           ├── fixtures/
│   │           │   └── history_client_response.json
│   │           └── artifacts/
│   │               └── run.log
│   ├── 2026-03-28_154511/
│   │   ├── 订单创建接口测试/
│   │   │   └── README.md
│   │   └── api/
│   │       └── order/
│   │           └── create_order_test.go
│   └── archive/             # 旧测试归档目录
│       └── 2026-W12/
│           └── ...
└── ...
```

## 放置原则

- 所有测试资产统一存放在项目根目录 `test/` 下。
- 每次测试先创建时间戳根目录，再在其中拆分中文说明目录和真实代码路径镜像目录。
- 只服务当前需求或当前 Bug 的验证资产，放到当前时间戳根目录，避免污染公共目录。
- 测试说明只放在中文说明目录的 `README.md` 中；真实代码、脚本、数据和产物不要放进中文目录。
- 测试代码、测试数据和测试说明可以同属一个时间戳根目录，但不要与生产代码混放。
- 每个时间戳根目录保持独立，互不干扰。
- 对 Go 项目，参与编译的测试目录必须保持 ASCII，避免中文路径进入 import path。

## 检查清单

- [ ] 当前测试资产是否已放在 `test/` 根目录下。
- [ ] 是否为当前测试任务创建了独立的时间戳根目录。
- [ ] 时间戳根目录命名是否符合 `yyyy-MM-DD_HHmmss` 格式。
- [ ] 是否创建了中文说明目录，且其中只放 `README.md`。
- [ ] `README.md` 是否包含测试目的、执行方式、依赖数据、覆盖范围、验证结论和真实测试资产路径。
- [ ] 真实测试资产是否按真实代码路径镜像组织，并保持 ASCII。
- [ ] 同一需求的多个测试是否分别创建了独立的时间戳根目录。
- [ ] 是否有测试资产散落在业务代码目录或其他位置。
- [ ] Go 源码目录是否不存在 `*_test.go`；若存在，是否已改为 seam 方案并迁移到 `test/`。
