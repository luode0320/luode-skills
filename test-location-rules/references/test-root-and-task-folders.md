# 测试根目录与任务目录规则

## 用途

用于先确定测试资源应该挂在哪一级目录，再决定后续命名和文件组织方式。

## 根目录约定

- **强制使用 `test/` 作为统一测试根目录**，所有测试资产必须存放在项目根目录的 `test/` 下。
- 不得在真实业务代码目录创建临时测试程序、测试脚本或补充验证代码。
- 不得使用 `tests/`、`qa/`、`verify/`、`tmp-test/` 等其他名称作为测试根目录。

## 任务目录命名规范

每次新增测试任务，必须在 `test/` 下创建独立子目录，命名格式为：

```
yyyy-MM-DD_HHmmss_测试任务中文简介
```

示例：
- `2026-03-28_143022_用户登录功能验证`
- `2026-03-28_154511_订单创建接口测试`
- `2026-03-28_162033_数据导出功能验证`

## 多测试任务处理规则

- 若同一需求需要多个测试程序或多轮独立验证，应分别创建多个测试任务子目录。
- 在一个会话中一个提问可能会出现多次不同功能的测试，都需要建立不同的测试目录。
- 每个子目录仅存放对应测试任务的脚本、代码、用例、模拟数据和验证说明，不得散落到其他目录。

## 任务目录内容要求

每个测试任务子目录必须包含 `README.md`，说明内容包括：
- 本次测试的目的
- 执行方式
- 依赖数据
- 覆盖范围
- 验证结论

## 旧测试目录归档规则

- 旧的测试目录以每周日为时间间隔，归档到 `test/` 目录下。
- 建议在 `test/` 下创建 `archive/` 子目录，按周或按月组织归档内容。
- 归档目录命名建议：`archive/2026-W13/`、`archive/2026-03/` 等。

## 常见目录结构示例

```
项目根目录/
├── src/                    # 业务代码目录（禁止存放测试资产）
├── test/                   # 统一测试根目录
│   ├── 2026-03-28_143022_用户登录功能验证/
│   │   ├── README.md       # 必须包含
│   │   ├── test-login.js   # 测试脚本
│   │   ├── fixtures/       # 测试数据
│   │   └── cases/          # 测试用例
│   ├── 2026-03-28_154511_订单创建接口测试/
│   │   ├── README.md
│   │   └── ...
│   └── archive/             # 旧测试归档目录
│       └── 2026-W12/
│           └── ...
└── ...
```

## 放置原则

- 所有测试资产统一存放在项目根目录 `test/` 下。
- 可跨模块复用的测试辅助资源，可放到 `test/shared/` 目录（如果确实需要）。
- 只服务当前需求或当前 Bug 的验证资产，放到当前任务目录，避免污染公共目录。
- 测试代码、测试数据、测试说明可以同属一个任务目录，但不要与生产代码混放。
- 每个任务目录保持独立，互不干扰。

## 检查清单

- [ ] 当前测试资产是否已放在 `test/` 根目录下。
- [ ] 是否为当前测试任务创建了独立的子目录。
- [ ] 目录命名是否符合 `yyyy-MM-DD_HHmmss_测试任务中文简介` 格式。
- [ ] 每个任务子目录是否包含 README.md。
- [] README.md 是否包含测试目的、执行方式、依赖数据、覆盖范围和验证结论。
- [ ] 同一需求的多个测试是否分别创建了独立目录。
- [ ] 是否有测试资产散落在业务代码目录或其他位置。
