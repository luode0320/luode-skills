# Bug 根因定位规则

只在通过静态证据定位 Bug 时使用这个 skill。
如果仅靠代码、日志、trace 和调用链已经无法继续收敛，应退出并转入 `bug-runtime-debug-rules`。

## Skill 作用与适用场景

- 通过代码、日志、trace、调用链和上下游行为静态收敛根因。
- 区分实现问题、状态问题、依赖问题和设计问题。
- 给出根因结论或明确承认证据不足。
- 避免在证据不足时强行下结论。

## 自动触发信号

- Bug 入口信息已经基本清楚，准备进入代码分析。
- 已拿到日志、报错、trace、调用链或关键上下游线索。
- 需要判断异常是在哪一层、哪个分支、哪次状态转换中产生。
- 需要在不进入运行时调试的前提下先尝试静态定位。

## 进入后先做什么

1. 先锁定异常现象对应的代码入口和业务链路。
2. 再梳理日志、trace、调用链与关键状态路径。
3. 找出最可能的异常点、错误分支或错误状态转换。
4. 判断当前证据是否足以形成根因结论。

## 默认执行流程

1. 默认先读 `references/static-analysis-path.md`，按静态定位路径逐步收敛。
2. 如果需要判断根因结论是否成立，再读 `references/root-cause-evidence.md`。
3. 如果需要判断是否已经到达静态定位极限，再读 `references/when-to-stop-static-analysis.md`。
4. 输出根因结论、证据链或“证据不足需升级运行时诊断”的结论。
5. 未形成稳定证据前，不把猜测当根因。

## 权责边界与不负责事项

- 只负责静态根因定位，不代替 `bug-runtime-debug-rules` 做运行时诊断。
- 不代替 `bug-intake-rules` 补基础现象描述。
- 不代替 `bug-fix-proposal-rules` 输出修复方案。
- 不把“看到可疑代码”直接当成“已经证实的根因”。

## 需要暂停并确认的条件

- 关键日志、trace 或调用链证据缺失。
- 多个根因假设都成立，静态证据无法排除。
- 当前链路太长，静态分析已经无法继续有效收敛。
- 继续静态分析只会重复阅读代码，无法带来新的证据。

## 执行通过 / 驳回标准

- 通过：能够给出清晰根因结论和对应证据链，或明确说明静态证据不足并建议升级运行时诊断。
- 驳回：只有主观怀疑，没有证据链；或在证据不足时硬性下结论。

## 执行结果归档要求

- 将静态定位路径、关键证据、根因结论或证据不足结论记录到 `analysis/`。
- 归档内容至少包含分析入口、关键链路、证据点、当前判断和下一步建议。
- 如果已确认需要升级到运行时诊断，应明确记录为什么静态定位无法继续收敛。

## references 读取规则

- 默认先读 `references/static-analysis-path.md`。
- 只有在评估根因证据是否成立时，再读 `references/root-cause-evidence.md`。
- 只有在判断是否该停止静态分析时，再读 `references/when-to-stop-static-analysis.md`。
