# 条件字段判定规则

## 1. 本次改动点

- 触发条件：本轮存在任何文件改动（代码或文档）。
- 输出要求：列出本轮核心改动点，避免流水账。

## 2. 验证结果

- 触发条件：本轮执行了验证动作（测试命令、构建检查、人工验证步骤等）。
- 输出要求：写清“验证了什么、结果如何、是否通过、是否有残留风险”。
- 真实验证降级：仅执行 build、lint、静态搜索、代码审查或实现自审时，只能标记“仅静态验证”；不得宣称功能或接口已验证可用。

## 3. 图形化总览

- 触发条件：本轮存在多步骤处理、闸门、分支、回滚、跨角色/跨服务交互、状态迁移、任务依赖、执行阶段、文件/模块影响面或真实数字对照等适合图形表达的内容。
- 输出位置：状态摘要和一句话结论之后、`## 🛠 执行证据` 之前；适用时先图后文，不得把图形放在长段执行说明之后。
- 默认表达：流程、依赖和执行链使用 Mermaid `flowchart`；跨角色、跨服务和异步调用使用 `sequenceDiagram`；状态迁移、阻断和恢复使用 `stateDiagram-v2` 或等价状态图；量化结果只有在当前渲染端可靠支持时才使用 Mermaid 图表，否则在图形化总览中明确写 `图形化表达：N/A`、原因和证据，再用 markdown 表格承载数据，且不得把表格伪称为图形。
- 数量限制：复杂任务默认 1 张主图；只有第二张图表达不同且必要的信息时才允许增加，总数最多 2 张。
- 图形前置标注：每张 Mermaid 图前必须紧邻写 `图形目的：...` 和 `关联 ID：...`。
- 事实约束：图形只能表达本轮已确认事实；节点、参与者、状态、数字、连线和结论不得虚构，图内术语必须与正文和证据一致。
- 不适用处理：明确属于简单单点任务时可以省略本节；复杂任务判断为 Mermaid 无法准确表达、数据不足或当前渲染能力不可靠时，输出 `图形化表达：N/A`、`原因：...`、`证据：...`，不得生成无意义占位图。
- 渲染约束：Mermaid 代码块必须使用 `mermaid` 语言标记，不得嵌套进 `text`、无语言代码围栏或 HTML；表格只能作为结构化数据补充，不得把“只有表格”伪称为图形化总览。

## 4. 函数注释核对

- 本节不再生成独立的“函数注释核对”用户可见区块。
- 本轮存在代码改动时，只在「Skill 执行证据」中记录 `comment-completion-gate-rules: PASS/FAIL` 及证据定位；上层闸门不得复制函数字段、步骤编号、补丁说明或核对清单细则。
- `code-change-finalization-gate-rules` 为注释链的代码收口消费者；它只消费 PASS/FAIL，不重新定义 comment-completion 或 comment-placement 的规则。

## 5. Obsidian 知识流状态

- 轻量判断：仓库任务可以只完成 `Obsidian:<检索/沉淀/不适用/阻断>` 判断；该判断本身不等于已经执行检索或沉淀。
- 检索状态触发条件：本轮命中 `obsidian-knowledge-flow`，且真实执行过 Obsidian CLI 检索或读取笔记动作，例如 `search`、`search:context`、`read` 或等价 CLI 命令。
- 检索状态输出位置：只在「方案与根因」中追加一行简短摘要，说明检索主题、读取数量或关键笔记路径；不要输出完整笔记内容。
- 沉淀状态触发条件：本轮命中 `obsidian-knowledge-flow`，且真实执行过 Obsidian CLI 创建、追加、更新或沉淀笔记动作，例如 `create`、`append` 或等价 CLI 命令。
- 沉淀状态输出位置：只在「结果与结论」中追加一行简短摘要，说明写入/更新的笔记类型或路径；不要输出完整笔记正文。
- 阻断或跳过：CLI 不可用、vault 未注册、检索无命中、判断为 `不适用`、判断后无需写入、或本轮没有真实执行对应动作时，不输出成功态；如阻断影响本轮结论，可在验证或结论中写阻断事实。

## 6. 不适用处理

- 条件字段未触发时，允许不输出该节。
- 不允许伪造验证结果、注释闸门状态或 Obsidian 检索/沉淀结果。
- 没有真实 CLI 证据时，不得写“已检索”或“已沉淀”。

## 7. 后续内容

- 默认不输出后续内容；没有合法后续时直接结束，禁止输出“等待用户新指令”“无需继续动作”“当前无需补充动作”“无下一步”等占位句。
- 非阻断状态下，仅两类情况允许输出：
  1. 原执行计划、完成定义或验收标准内仍存在未完成必需项。
  2. 用户当前轮明确要求后续建议、优化建议或 backlog。
- 真实 `blocked/manual_handoff` 不使用本节，统一进入“任务阻断收口”；阻断恢复计划由第 8 节的共享契约渲染。
- 输出格式固定为 `## 后续内容` 加 1-3 条有序列表；每条用动作动词开头，并尽量包含动作对象、完成判据或验证入口。
- 来源只允许是原执行计划、完成定义、验收标准或用户显式要求；可选优化、额外整理、未来迭代、体验提升、文档再润色不得默认扩散。
- 排序规则：计划内未完成必需项优先，用户显式要求的 backlog 其次；计划内必需项已占满 3 条时，可以不展示 backlog。
- 示例语义：
  - 测试落点缺口应写“迁移违规测试文件 → 补外部测试 seam → 重跑测试并回写证据”，不得泛化成“完善测试”。
  - 注释闸门缺口只写“重新执行 comment-completion 并达到 PASS”，不得在本节复制注释字段或编号细则。
  - 原目标已完成且不存在用户显式建议请求时，不输出本节。

## 8. 任务阻断收口

- 触发条件：运行时已真实收口为 `blocked` 或 `manual_handoff`，且存在符合 `artifact-delivery-gate-rules/references/task-blocker-closure-contract.md` 的共享阻断事实。
- 输出 owner：只有 `reasoning-summary-structure-rules` 能输出用户可见的 `## ⛔ 任务阻断收口`；其他 skill 只能生产、更新或校验共享事实。
- 输出位置：必须是最终总结的最后一个区块；本轮改动点、验证结果和一般后续内容均不得位于其后。
- 输出字段：状态、阻断阶段、阻断依据与证据、已尝试与停止条件、影响、按优先级排序的 1-3 条解决计划、恢复验证 / 重入点。
- 解决计划的每一条都必须包含 owner、前置条件、动作、完成判据和验证入口；同一根因只展示一套去重后的计划。
- 非触发条件：`completed`、`limited`、`not_applicable`、仅静态验证、尚未真实收口的候选问题都不得输出该区块；它们按既有结果或受限交接处理。
