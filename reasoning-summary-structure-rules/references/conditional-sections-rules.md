# 条件字段判定规则

## 1. 本次改动点

- 触发条件：本轮存在任何文件改动（代码或文档）。
- 输出要求：列出本轮核心改动点，避免流水账。

## 2. 验证结果

- 触发条件：本轮执行了验证动作（测试命令、构建检查、人工验证步骤等）。
- 输出要求：写清“验证了什么、结果如何、是否通过、是否有残留风险”。

## 3. 函数注释核对

- 触发条件：本轮存在代码改动。
- 输出要求：
  - 若改动涉及函数/方法：列出已核对的函数/方法位点。
  - 若代码改动不涉及函数/方法：明确写“函数位点 0 个”。

## 4. Obsidian 知识流状态

- 检索状态触发条件：本轮命中 `obsidian-knowledge-flow`，且真实执行过 Obsidian CLI 检索或读取笔记动作，例如 `search`、`search:context`、`read` 或等价 CLI 命令。
- 检索状态输出位置：只在「方案与根因」中追加一行简短摘要，说明检索主题、读取数量或关键笔记路径；不要输出完整笔记内容。
- 沉淀状态触发条件：本轮命中 `obsidian-knowledge-flow`，且真实执行过 Obsidian CLI 创建、追加、更新或沉淀笔记动作，例如 `create`、`append` 或等价 CLI 命令。
- 沉淀状态输出位置：只在「结果与结论」中追加一行简短摘要，说明写入/更新的笔记类型或路径；不要输出完整笔记正文。
- 阻断或跳过：CLI 不可用、vault 未注册、检索无命中、判断后无需写入、或本轮没有真实执行对应动作时，不输出成功态；如阻断影响本轮结论，可在验证或结论中写阻断事实。

## 5. 不适用处理

- 条件字段未触发时，允许不输出该节。
- 不允许伪造“验证结果”或“函数注释核对”内容。
- 不允许伪造 Obsidian 检索或沉淀结果；没有真实 CLI 证据时，不得写“已检索”或“已沉淀”。

## 6. 后续内容

- 默认不输出后续内容。
- 仅三类情况允许输出：
  1. 原执行计划内仍存在未完成必需项。
  2. 当前存在阻断项，导致原计划必需项无法继续。
  3. 用户明确要求提供后续建议、优化建议或 backlog。
- 可选优化、额外整理、未来迭代、体验提升、文档再润色等内容，若不属于原计划必需项，不得作为默认后续内容输出。
