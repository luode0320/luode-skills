## 正例（需求对照清晰）

```text
Skill 命中检查：
- 命中 skill：`reasoning-summary-structure-rules`

Skill 执行证据：
- 已按模板完成结构化总结。

当前要解决的问题：
- 用户原始需求：希望总结里不仅写结论，还要能看出模型最后理解的需求。
- 模型理解的需求：在最终总结中显式写出“用户原始需求 / 模型理解的需求 / 两者是否一致”，方便对照是否理解偏了。
- 两者是否一致：不一致，模型最初只理解成“补充总结模板”，漏掉了“需求对照”这一层。

当前问题的解决方案、问题根因：
- 解决方案：在总结结构里固定增加“用户原始需求 / 模型理解的需求 / 两者是否一致”。
- 根因：没有需求对照层时，用户很难快速判断模型有没有理解偏题。

当前解决结果、结论：
- 已把需求对照纳入最终总结结构，后续可以直接比对用户提问和模型理解是否一致。

函数注释核对：
- 函数位点 0 个。

下一步状态：
1. 在真实任务里观察 2-3 次，确认需求对照是否足够清楚。
2. 如果用户还想看“模型误解了哪一步”，再补一个更细的差异说明字段。
```

```text
Skill 命中检查：
- 命中 skill：`reasoning-summary-structure-rules`

- 已完成总结。

当前要解决的问题：
- 需要把这次任务总结一下。

当前问题的解决方案、问题根因：
- 解决方案：按模板输出。
- 根因：总结要结构化。

当前解决结果、结论：
- 已完成。

下一步状态：
1. 继续补齐当前原始用户目标仍未完成的必要动作。
```

不通过原因：

- 没有写出用户原始需求。
- 没有写出模型理解的需求。
- 没有判断两者是否一致。

## 反例（缺必填项）

```text
结论：已完成。
```

不通过原因：

- 缺“Skill 命中检查”。
- 缺“Skill 执行证据”。
- 缺“当前问题 / 解决方案与根因 / 结果结论”；若原始目标未完成或用户明确要求建议时，还缺“下一步状态”。
