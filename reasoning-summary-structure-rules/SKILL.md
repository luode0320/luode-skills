# 推理总结结构闸门规则

只在“本轮准备输出最终总结”时使用这个 skill。
目标是统一最终总结结构，避免关键信息缺失或顺序混乱。

## Skill 作用与适用场景

- 在最终回复前检查总结结构字段是否齐全。
- 强制字段缺失时不允许给出“已完成”式收口结论。
- 对“有条件字段”按本轮真实上下文做显式判定，避免漏报或伪报。
- 让不同任务类型（需求、Bug、实现、评审、测试、交付）都能输出可复核总结。

## 自动触发信号

- 准备输出本轮最终回复。
- 输出中出现“总结”“结论”“收口”“最终说明”“最终回复”等结束信号。
- 任务已经进入收口阶段，需要统一总结结构。

## 进入后先做什么

1. 先收集本轮事实：命中 skill、执行动作、问题定义、处理方案、结果状态。
2. 判断条件字段是否适用：
   - 是否有代码/文档改动。
   - 是否有验证动作与验证结论。
3. 按固定顺序组织总结结构。
4. 若必填字段缺失，先补信息再输出最终总结。

## 默认执行流程

1. 读取 `references/summary-structure-template.md`，按模板组织输出框架。
2. 读取 `references/conditional-sections-rules.md`，判定条件字段是否必须出现。
3. 按模板输出最终总结。
4. 若本轮存在代码改动，必须输出“函数注释核对”条目（无函数改动也要写“函数位点 0 个”）。

## 输出要求（固定顺序）

最终总结必须按以下顺序输出：

1. Skill 命中检查（必须）
2. Skill 执行证据（必须）
3. 当前要解决的问题（必须）
4. 当前问题的解决方案、问题根因（必须）
5. 当前解决结果、结论（必须）
6. 本次改动点（有改动时必须）
7. 验证结果（有验证时必须）
8. 函数注释核对（有代码改动时必须）
9. 下一步建议（必须）

## 权责边界与不负责事项

- 只负责“最终总结结构完整性”，不替代业务执行。
- 不替代需求澄清、Bug 根因分析、编码实现、测试执行。
- 不要求虚构字段；不存在的条件字段应明确“本轮不适用”或按规则省略。

## 执行通过 / 驳回标准

- 通过：9 项结构满足“必填/条件触发”要求，顺序正确、内容可复核。
- 驳回：缺任一必填项，或条件项应出现却未出现，或结构顺序明显错乱。

## references 读取规则

- 默认先读 `references/summary-structure-template.md`。
- 只有在判断条件字段时再读 `references/conditional-sections-rules.md`。
