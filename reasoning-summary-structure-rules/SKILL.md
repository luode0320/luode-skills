# 推理总结结构闸门规则

只在“本轮准备输出最终总结”时使用这个 skill。
目标是统一最终总结结构，并在信息适合图形表达时优先用 Mermaid 呈现关键流程、依赖、状态和执行关系，避免关键信息缺失、顺序混乱或长段文字掩盖主链路。

## Skill 作用与适用场景

- 在最终回复前检查总结结构字段是否齐全。
- 强制字段缺失时不允许给出“已完成”式收口结论。
- 对“有条件字段”按本轮真实上下文做显式判定，避免漏报或伪报。
- 让不同任务类型（需求、Bug、实现、评审、测试、交付）都能输出可复核总结。
- 对流程、依赖、状态、执行链、跨角色交互和量化结果执行图形适用性判断；适用时先图后文，简单单点任务保持简洁。

## 自动触发信号

- 准备输出本轮最终回复。
- 输出中出现“总结”“结论”“收口”“最终说明”“最终回复”等结束信号。
- 任务已经进入收口阶段，需要统一总结结构。

## 进入后先做什么

1. 先收集本轮事实：命中 skill、执行动作、问题定义、处理方案、结果状态。
2. 判断条件字段是否适用：
   - 是否存在适合图形表达的流程、依赖、状态、执行链、跨角色交互或量化结果。
   - 是否有代码/文档改动。
   - 是否有验证动作与验证结论。
   - 是否已完成 Obsidian 轻量捕获 / 检索判断；只有 `obsidian-knowledge-flow` 真实执行过检索、读取、捕获或沉淀动作时，才在最终总结输出对应 Obsidian 条件行。
   - 是否存在真实状态为 `blocked` 或 `manual_handoff` 的共享阻断事实；只读 `artifact-delivery-gate-rules/references/task-blocker-closure-contract.md` 定义的字段，不重建平行字段。
3. 按固定顺序组织总结结构。
4. 若必填字段缺失，先补信息再输出最终总结。
5. 在“当前要解决的问题”里同时写清：
   - 用户原始需求
   - 模型理解的需求
   - 两者是否一致
6. 若当前运行环境存在 goal / plan / task 等显式状态收口机制，且总结结论准备写“任务已完成”，必须先确认该运行时状态已被真实收口；总结结构不能替代对应工具或协议收口动作。
7. 最终收口默认不输出后续内容；只有存在原执行计划内未完成必需项、阻断项，或用户明确要求提供后续建议时，才允许输出后续内容。若用户明确说“结束”“停止”“到此为止”“不要继续”“不要下一步建议”“不要扩散”等结束指令，或原始用户目标已完成且仅剩可选优化，本 skill 只允许组织最小收口结论；不得输出“下一步状态 / 下一步建议”区块，也不得用“等待用户新指令 / 无需继续动作”替代。

## 默认执行流程

1. 读取 `references/summary-structure-template.md`，按模板组织输出框架。
2. 读取 `references/conditional-sections-rules.md`，判定图形化总览及其他条件字段是否必须出现。
3. 先做图形适用性判断：适用时选择最贴合事实的一种 Mermaid 图形；只有两种图分别承载不同且必要的信息时才允许输出第二张图，总数最多 2 张。
4. 按模板输出最终总结：状态摘要和一句话结论之后先输出图形化总览，再输出执行证据与文字字段。
5. 按「总结视觉规范」对最终总结排版：先输出 `---` 分隔线，再用 `# 📋 本轮总结` 容器承载固定字段，与推理过程视觉分界。
6. 发送前强制自检（硬闸，不可跳过）：在最终回复真正发出前，逐条核对草稿是否满足以下四项；任意一项不满足，禁止发送，必须先改写为标准模板再输出，不允许以“先前已声明命中”为由直接放行：
   - 是否已以 `---` + `# 📋 本轮总结` 开头，与推理过程 / 中间进度视觉分界；
   - 是否已覆盖「输出要求（固定顺序）」的全部必填小节，且顺序未错乱；
   - 复杂任务是否已按事实判断图形适用性；适用时图形是否先于长段文字，不适用时是否按条件规则省略或说明 `N/A + 原因 + 证据`；
   - 每张图是否有图形目的、关联 ID、完整节点/连线，且图内术语、状态和结论与正文及真实证据一致。
   - 推理过程、中间进度与最终总结正文是否为简体中文；发现成段非中文（英文 / 日文等）自然语言必须改回简体中文再发送，代码符号、命令、路径、原始字段名、报错原文等技术片段除外。
   若本轮同时命中 `skill-execution-compliance-gate-rules` 或 `code-change-finalization-gate-rules`，本步骤与其收口检查合并执行、共用同一份证据，不重复渲染两套结论。

## 输出要求（固定顺序）

最终总结必须按以下顺序输出：

1. Skill 命中检查（必须，主标题后的状态摘要行）
2. 图形化总览（内容适用时必须，位于一句话结论之后、执行证据之前；复杂任务判断为不适用或无法可靠表达时按条件规则写 `N/A + 原因 + 证据`，明确简单单点任务可省略）
3. Skill 执行证据（必须）
4. 当前要解决的问题（必须）
5. 当前问题的解决方案、问题根因（必须）
   - 若 `obsidian-knowledge-flow` 本轮真实自动检索并读取过笔记，在本节用一行简短说明检索情况；未真实检索时不输出。
6. 验证结果（有验证时必须）
7. 当前解决结果、结论（必须）
   - 必须写清本次解决的用户问题 / 诉求、采用的方法、最终确认结果。
   - 若用户提及的问题已解决，结果区必须明确写出“已解决 / 已确认解决”；若仍未完全解决，必须说明剩余卡点。
   - 若 `obsidian-knowledge-flow` 本轮真实捕获或沉淀了知识，在本节用一行简短说明沉淀结果；未真实写入时不输出。
8. 后续内容（仅非阻断状态下存在原执行计划内未完成必需项，或用户明确要求后续建议时出现；默认不出现）
9. 本次改动点（有改动且不存在真实阻断时放在总结最后）
10. 任务阻断收口（仅真实 `blocked` 或 `manual_handoff` 时必须出现，并作为总结最后的最终状态区）

> `任务阻断收口` 只由本 skill 渲染。审查、验收、验证、失败恢复和合规闸门只能按共享契约提供或校验结构化事实，不得生成第二套用户可见的阻断模板、后续内容或解决计划。

> 不再单列“函数注释核对”节点：函数注释核对仍由 `code-change-finalization-gate-rules` 与 `comment-completion-gate-rules` 作为闸门内部执行证据完成，但不在最终总结中展示为节点。

## 图形化优先规则（强制）

### 适用性与图形选择

| 内容特征 | 默认表达 | 使用要求 |
|---|---|---|
| 多步骤处理、闸门、分支、回滚 | Mermaid `flowchart` | 突出主流程、关键分支和停止条件 |
| 跨角色、跨服务、异步调用、工具链 | Mermaid `sequenceDiagram` | 展示调用顺序、责任边界和结果回流 |
| 状态迁移、阻断、恢复、验收状态 | Mermaid `stateDiagram-v2` 或状态流转图 | 展示状态、进入条件和恢复路径 |
| 任务依赖、执行阶段、文件或模块影响面 | Mermaid `flowchart` / 依赖图 | 展示执行顺序、依赖关系和最大推进边界 |
| 有真实数字对照或趋势 | 当前渲染端可靠支持时使用 Mermaid 图表；否则在图形化总览中写 `N/A + 原因 + 证据`，并用 markdown 表格承载数据 | 不得把表格伪称为图形，也不得虚构、补齐或推算未验证数据 |
| 单文件简单修改、单命令说明、纯问答 | 不强制生成图 | 保持简洁，必要时使用表格或列表 |

- 复杂任务默认输出 1 张主图；只有第二张图表达不同且必要的信息时才允许增加，总数最多 2 张。
- 图形只表达本轮已确认事实，不得补造不存在的动作、角色、状态、结果、数字或依赖。
- 流程、时序、状态、依赖和执行关系优先使用 Mermaid；表格是结构化数据的补充，不得把“只有表格”宣称为已完成图形化总览。
- 真实图片只用于 Mermaid 无法准确表达的视觉证据或已有产物，不替代流程、时序、状态和依赖图。

### 图形标注与一致性

每个 Mermaid 图形前必须紧邻写明：

- `图形目的：...`
- `关联 ID：...`

并同时满足：

- 图形代码块使用 `mermaid` 语言标记，不得嵌套进 `text`、无语言代码围栏或 HTML。
- 图形必须非空，节点、参与者、状态和连线完整；图内术语必须与正文、证据和稳定 ID 一致。
- 图形位于状态摘要和一句话结论之后、`## 🛠 执行证据` 之前，确保用户先看到主链路再阅读细节。
- 若复杂任务已进入图形适用性判断但 Mermaid 无法准确表达、数据不足或当前渲染能力不可靠，输出 `图形化表达：N/A`、`原因：...`、`证据：...`；不得用无意义占位图绕过。
- 明确属于简单单点任务时可以省略整个图形化总览小节，不要求输出形式化 N/A。

## 总结视觉规范（强制）

最终总结必须与上方推理过程、中间进度在视觉上明确分界，并采用统一的严谨 markdown 排版，让用户一眼分辨“哪里开始是总结”。本节只约束排版与分界，不改变「输出要求（固定顺序）」的字段与顺序。

### 视觉分界

- 总结区块前必须插入一条 `---` 水平分隔线，与推理过程、工具叙述、中间进度切开。
- 分隔线下用固定一级主标题作为容器起点：`# 📋 本轮总结`（一级标题，字号最大、天然加粗；标题文案固定，便于辨认）。
- 容器外不得再散落总结性结论；中间进度（首条命中检查两行、工具调用叙述）不套用总结容器。

### 排版要求

- 紧随主标题输出一行状态摘要：`**命中检查:通过** · <闸门>:PASS ✅`（存在合规闸门时带上结论）。
- 用一个 `>` 引用块写“一句话结论”，让核心结果最突出。
- 固定字段一律用二级标题 `## ` 小节承载，字号明显大于正文且天然加粗；形成“一级主标题 > 二级小节标题 > 正文”的三级字号层级，层级之间字号必须有明显区分。
- 标题必须使用 markdown 标题语法（`#` / `##`），禁止用加粗文本冒充小标题；确保标题比正文更大且加粗。
- 小节可配区分性 emoji：📊 图形化总览 / 🛠 执行证据 / 🎯 要解决的问题 / 🔧 方案与根因 / ✅ 验证 / 📌 结果与结论 / 📦 改动点。
- 多文件改动、需求对照、状态对照优先用 markdown 表格，不堆长串冒号列表。
- 状态用徽章式标记：通过 ✅、警告 ⚠️、失败 ❌、阻断 ⛔。
- 无真实阻断时，「本次改动点」放在所有内容节点的最后；真实阻断时置于最终阻断区块之前。
- `obsidian-knowledge-flow` 的检索/沉淀状态只作为条件行嵌入「方案与根因」或「结果与结论」，不得新增独立 Obsidian 小节，也不得输出完整笔记正文。
- 不再单列“函数注释核对”节点；该核对作为闸门内部证据完成，不在总结展示。
- 统一使用 markdown，不依赖 HTML 渲染：HTML 在 Claude Desktop、纯 CLI、Codex 等多数 agent 不渲染，会退化成原文噪声并破坏机器解析；视觉靠语义层级（`#` / `##` 标题、`---` 分隔线、表格、`>` 引用块、徽章 emoji）而非绝对字号，字号由各 agent 渲染端决定，需要更大时调客户端缩放，不在输出里塞 HTML。
- 普通说明、方案、流程、总结、审查报告、线程拆分和状态回报必须使用普通 Markdown 段落、列表、表格或引用块；不得用 ` ```text `、无语言代码围栏、缩进代码块或 HTML 包裹整段自然语言输出。
- 代码围栏只用于真实代码、命令、配置片段、日志片段、JSON/YAML 等需要等宽保真的内容；若只是为了展示可读结构，应改用 Markdown 列表或表格。
- 字段顺序仍严格遵循「输出要求（固定顺序）」；图形适用时必须先于执行证据和长段文字，视觉规范不得省略必填项。
- 若存在真实阻断，`## ⛔ 任务阻断收口` 必须位于总结最后，替代“后续内容”；本次改动点（如有）必须置于其前，避免任何文本覆盖最终状态。

## 权责边界与不负责事项

- 只负责“最终总结结构完整性”，不替代业务执行。
- 不替代需求澄清、Bug 根因分析、编码实现、测试执行。
- 不要求虚构字段；不存在的条件字段应明确“本轮不适用”或按规则省略。

## 执行通过 / 驳回标准

- 通过：必填与条件触发项满足要求、顺序正确、内容可复核，且解释性自然语言保持简体中文；复杂任务已完成图形适用性判断，适用时先图后文且每张图具备图形目的、关联 ID 与图文一致性，不适用时处理符合条件规则；无真实阻断时改动点位于总结最后；标题层级与字号符合视觉规范；结果与结论明确写出本次解决的问题 / 诉求、采用的方法和确认结果；真实 `blocked/manual_handoff` 时引用共享契约完整渲染状态、阶段、依据与证据、已尝试与停止条件、影响、1-3 条解决计划和恢复验证 / 重入点，并将该区块置于最后；若不存在原执行计划内未完成必需项、真实阻断，且用户未明确要求后续建议，可直接以结果结论收口，不追加“下一步状态 / 下一步建议 / 等待用户新指令 / 无需继续动作”等任何后续占位。
- 驳回：缺任一必填项，或条件项应出现却未出现，或结构顺序明显错乱；或存在明显流程、依赖、状态、交互、执行链或量化结果却未做图形适用性判断；或图形位于长段文字之后、缺图形目的/关联 ID、图文术语漂移、节点连线不完整、虚构事实数据、使用无意义占位图，或简单任务为了形式强制造图；或无真实阻断时改动点未放在最后；或真实阻断时未将“任务阻断收口”置于最后、缺少共享契约字段、渲染了多套解决计划，或把 `limited/not_applicable` 误报为任务阻断；或标题未用 markdown 标题语法、字号层级不分；或把普通说明、方案、流程、总结、审查报告、线程拆分或状态回报包进 ` ```text ` / 无语言代码围栏 / 缩进代码块 / HTML；或仍单列“函数注释核对”节点；或结果与结论未明确回写本次解决的问题 / 诉求与确认状态；或不存在三类合法后续（原执行计划内未完成必需项 / 阻断项 / 用户显式要求的建议）却仍强行追加新的执行建议、下一步区块、等待指令文案、无需继续动作或其他扩散性后续；或当前运行环境仍有显式状态未收口却只写完成文案、没有真实状态收口动作；或已声明命中本 skill 却在发出前跳过第 5 步发送前自检，直接以未套用总结容器的口语化文本收尾；或推理过程、中间进度或最终总结出现成段非中文（英文 / 日文等）自然语言，且非代码符号、命令、路径、原始字段名、报错原文等技术片段。
