# 阶段阻断规则

## 用途

用于判断当前任务是否具备进入下一阶段的条件，防止跳过关键前置环节。

## 阶段阻断表

| 当前阶段 | 典型阻断条件 | 默认动作 |
| --- | --- | --- |
| 需求阶段 | 目标不清、关键缺口未补、边界不清、验收标准缺失、关键依赖未确认、需求拆分未稳定 | 暂停推进，留在需求域继续澄清、拆分或重算变更 |
| Bug 阶段 | 无法稳定复现、问题归属未判清、根因证据不足、运行时诊断目标不清、修复建议未经确认 | 暂停进入编码，继续补缺口、复现、定位或确认方案 |
| 编码阶段 | 仍存在阶段争议、位点不明、改动范围失控 | 暂停继续扩写代码，先回总控层或需求 / Bug 域纠偏 |
| 编码审查阶段 | 语法未过、引用未清、调试残留未处理、代码归位错误、最小改动原则不满足 | 不允许进入测试域 |
| 测试阶段 | 测试策略未定、当前需求未验证、联调未打通、回归范围未覆盖、关键场景结论缺失、Bug 修复闭环未完成 | 不允许进入交付域 |
| 交付阶段 | 提交未整理、正式评审未完成、发布检查未过、回滚条件不清、交付说明缺失 | 不允许进入正式合并或发布 |

## 附加规则

- 阻断不是终止。阻断表示必须先补齐前置条件。
- 如果前一阶段的结论被推翻，应允许流程重启到更早阶段。
- 如果当前路线已被证明显著错误，应终止当前推进并重新拆分任务。
