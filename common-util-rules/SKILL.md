# 公共工具规则

只在判断“这段代码到底该不该抽成公共工具、抽了以后该放哪里”时使用这个 skill。
如果当前代码仍然明显带有业务语义，不要因为“以后可能复用”就提前抽成公共层。

## Skill 作用与适用场景

- 判断当前逻辑是否真的属于通用工具、常量或公共组件。
- 统一公共代码的存放位置和调用方式。
- 防止以“复用”为名把业务逻辑过早抽成公共层。
- 保持公共工具层稳定、通用、低耦合。

## 自动触发信号

- 新增或修改工具类、工具函数、辅助方法、公共组件。
- 新增或修改通用常量、枚举、公共包装逻辑。
- 打算把业务代码抽成 `utils`、`common`、`shared`、`helpers` 等目录。
- 不确定一段逻辑该留在业务层还是抽到公共层。

## 进入后先做什么

1. 先判断这段逻辑是否脱离当前业务后仍然成立。
2. 再判断它是通用工具、稳定常量、通用组件，还是当前业务逻辑。
3. 明确应该落在公共层、业务层还是现有目录中。
4. 确认抽象后不会引入反向依赖、上下文泄漏或过度复用。

## 默认执行流程

1. 默认先读 `references/util-qualification.md`，判断什么才算通用工具。
2. 如果涉及落点和调用方式，再读 `references/util-placement.md`。
3. 如果需要对照典型正反例，再读 `references/util-examples.md`。
4. 输出“应抽 / 不应抽”的结论、落点建议和调用边界。
5. 如果结论是不应抽，保留在业务层，不强推公共化。

## 权责边界与不负责事项

- 只负责公共工具判定和落点，不代替 `package-structure-rules` 决定整个项目分层结构。
- 不代替具体位点 skill 设计接口、数据库、错误处理或日志实现。
- 不以“可能复用”作为唯一抽取理由。
- 不把强业务语义、强上下文依赖代码伪装成工具函数。

## 需要暂停并确认的条件

- 当前代码只在一个业务流程内成立，却计划抽成公共层。
- 抽取后需要把业务上下文大量参数化，导致调用复杂度上升。
- 抽取会让公共层开始依赖业务层或具体位点层。
- 当前公共层已经膨胀，但仍打算继续往里塞业务逻辑。

## 执行通过 / 驳回标准

- 通过：能明确说明这段代码为何通用、为何应抽、应放在哪、调用边界是什么。
- 驳回：只是因为“看起来可以复用”就提前抽象，或把业务逻辑、流程编排、领域判断塞进公共工具层。

## 执行结果归档要求

- 如果本次新增了公共工具、公共常量或公共组件，将结论记录到 `analysis/` 或 `review/`。
- 归档内容至少包含抽取理由、使用范围、落点和不应继续扩张的边界说明。
- 如果结论是不应抽，也应在必要时记录保留在业务层的原因。

## references 读取规则

- 默认先读 `references/util-qualification.md`。
- 只有在判断存放位置和调用方式时，再读 `references/util-placement.md`。
- 只有在对照正反例时，再读 `references/util-examples.md`。
