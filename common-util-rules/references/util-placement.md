# 公共工具落点

## 用途

用于决定通用工具、常量和公共组件应放在哪一层。

## 建议

- 小型无状态工具放 `utils` 或语言生态等价目录。
- `utils`、`common`、`global`、`middleware` 根目录先按职责拆子目录，再放实现文件。
- 避免在上述根目录直接放 `xx.go`，例如应使用 `utils/log/logger.go`，而不是 `utils/logger.go`。
- 稳定常量放 `constants`。
- 通用 UI 组件放明确的公共组件目录。
- 仍带业务语义的代码留在业务层，不放公共层。

## 原则

- 落点应与职责一致。
- 公共层不反向依赖业务层。
- 公共根目录承担命名空间职责，具体实现放在子目录以降低命名冲突和循环依赖风险。
- 不要为了一个函数就新建模糊公共目录。
