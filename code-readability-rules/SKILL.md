# 代码可读性规则

只在判断“当前实现是否清晰、自然、容易接手”时使用这个 skill。
如果当前争议是改动范围、项目风格、注释颗粒度或目录归位，请转交相邻 skill。

## Skill 作用与适用场景

- 保持函数结构清晰、逻辑顺序自然。
- 控制单个函数或模块的复杂度，避免隐式跳转和过深嵌套。
- 让后续维护者能快速看懂输入、处理过程和输出。
- 避免炫技式写法、过度封装、过度压缩和“为了短而短”的表达。
- 在行为不变的前提下，优先简化最近改动代码中明显绕、密、重的表达。
- 会写库、发请求、改状态、发事件、写文件或触发异步任务的函数，必须在名称、调用位置或职责层级上暴露副作用。
- 查询型函数默认不应修改状态；命令型函数默认不应伪装成纯查询，除非事务边界或项目约定明确要求合并。
- 大多数代码默认不需要封装；只有封装后主流程更清楚、重复更少、边界更稳定、调用方更明确时，才把直白代码提炼出去。
- 抽象应优先形成“深模块”：用简单接口隐藏真实复杂度；仅转发、仅改名或仅包一层的浅封装默认降低可读性。
- 线性流程代码（中间件、handler、pipeline、初始化等按时间/步骤顺序执行的逻辑）默认“单主线从上到下”表达：主线本身就是最自然的顺序，读者按时间读、代码按时间排；只有当某段逻辑真正复杂（有分支、递归、需真实复用或需独立测试）时才拆成函数，一次性、无分支、约 5 行以内的取值/赋值逻辑就地内联，不升格为带签名的独立函数。
- 在“每个函数必须带 doc 头模板（如 `[参数]`/`[返回]`/`最近修改时间`）”的项目里，函数数量本身就是注释成本：多拆一个一次性小函数 = 多背一套模板 + 多一次跳转；此时控制样板注释与阅读跳转的正确杠杆是“少拆一次性小函数”，而不是“把注释写短”（注释模板是硬规则不可删）。
- 职责清晰不等于“每个职责都必须独立函数”。极短的局部检查、判空、匹配器取用、简单 scope/flag 选择等逻辑，如果只有一个调用点、无副作用、无复杂分支、无独立测试价值，优先留在当前函数内，并用步骤注释或局部注释解释业务含义；不要为了保留一个函数名而增加跳转和注释模板成本。
- 业务方法默认只访问 DTO/对象字段，不直接在业务流程中散落 JSON 字符串 key 读取。
- 完成结构调整后，确保改动位点触发并完成 `comment-placement-granularity-rules` 与 `comment-completion-gate-rules` 注释检查与补齐。
- 当单文件体量已达 500 行及以上且仍在持续加功能时，强制触发按职责拆分评估。
- Go 场景下函数签名默认单行参数；参数过多时优先提炼参数结构体，而不是直接改成多行参数列表。
- Go 接入第三方 API 时默认必须定义响应结构体（建议落在 `internal/entity/<domain>/`）；非必要禁止 `map[string]any` + 字符串 key 的弱类型解析。
- Go 格式化日志调用优先单行；如果单行过长必须换行，优先采用“日志模板一行、参数下一行”的两段式，参数可以按语义分组，不要把同一条日志拆成每个参数一行的碎片化多行。
- 简单业务流程、单实现调用链或局部辅助逻辑，默认直接写具体实现；不要仅因为“以后也许能替换”“测试可能会 mock”就先抽一层接口。
- 接口抽象必须提升可读性或隔离真实边界。若只有一个实现、一个调用边界或只是为了“规范感”，接口 + 实现体的双跳转会降低可读性，应使用具体类型或具体函数。
- 在已有文件中新增函数时默认追加到文件末尾，不要插入到已有函数中间；类型、结构体等定义可以放在文件靠前位置。

## 自动触发信号

- 新增业务函数、服务方法、控制器逻辑、工具函数。
- 修改已有复杂逻辑。
- 修复 Bug 后需要重排逻辑顺序或拆小函数。
- 编写脚本或任务流程代码时需要控制步骤可读性。
- 同一文件超过 500 行且本轮还在新增方法、分支或流程步骤。
- Go 接入第三方 API 且文档仅有 curl 示例、缺少响应字段说明。

## 进入后先做什么

1. 先识别当前实现的主流程。
2. 确认读代码的人能否快速看出输入、关键分支和输出。
3. 找出过深嵌套、职责混杂、命名模糊、控制流跳跃、嵌套三元和过密表达的位置。
4. 检查写库、发请求、改状态、发事件、写文件或触发异步任务等副作用是否从命名和位置可见；如果查询型函数暗中修改状态，应要求拆分或改名。
5. 判断哪些地方是真正影响理解的冗余抽象或包装，尤其检查是否为了“规范感”平白引入单实现接口、接口+实现双跳转或没有真实替换需求的抽象层；不要为了“简化”拆掉有价值的结构。
6. 检查新增 helper、wrapper、factory、manager、adapter、interface 是否真的降低阅读成本；如果只是转发、改名或把局部短逻辑藏到另一处，应要求内联或取消封装。
7. 在不扩大改动范围、且功能完全不变的前提下优化结构表达。
8. 若本轮目标文件已达 500 行及以上，先按“功能域/流程阶段/上下游边界”给出最小拆分方案（拆方法、拆文件或拆子目录）。

## 默认执行流程

1. 先读取 `references/readability-general.md`，确认本次实现的可读性底线。
2. 如果涉及函数拆分、步骤组织和控制流安排，再读取 `references/function-structure-rules.md`。
3. 如果需要判断某种写法是否虽然可运行但可读性很差，再读取 `references/readability-examples.md`。
4. 先保证主流程直观，再处理局部细节；清晰优先于短小，显式优先于炫技。
5. 如果改善可读性会明显扩大改动范围，暂停并交还给 `code-minimal-change-rules` 判断。
6. 若目标文件已达 500 行及以上且仍持续膨胀，优先拆分到职责更清晰的文件；业务流程明显增多时，联动 `package-structure-rules` 评估子目录/子包拆分。
7. 结构改动落地后，联动注释双 skill：用 `comment-placement-granularity-rules` 检查注释必要性 / 位置 / 颗粒度，再用 `comment-completion-gate-rules` 检查改动位点补齐与 `1/2/3` 就近注释。

## 权责边界与不负责事项

- 只负责结构可读性，不替代 `code-minimal-change-rules` 控制改动范围。
- 不替代 `code-style-consistency-rules` 处理项目风格一致性。
- 不替代注释类 skill 决定中文注释的颗粒度和写法。
- 不替代 `naming-rules` 单独决定业务术语命名。

## 需要避免

- 在路由文件中，若接口数量较少（例如 <= 10），禁止为“复用”引入额外抽象层（如 route struct + 批量注册函数），优先显式逐条注册。
- 可读性优先于“看起来通用”的抽象；读者应一眼看到完整接口路径与处理函数映射关系。
- 函数名、注释或所在层级表现为 `get`、`list`、`query`、`find`、`calc`、`build`、`resolve` 等查询 / 计算语义时，禁止暗中写库、发请求、改状态、发事件或写文件。
- 同一函数不要同时承担 query 与 command 两种职责；确因事务一致性、幂等控制或项目约定必须合并时，名称必须体现命令语义，并把查询步骤与副作用步骤在函数内部清楚分段。
- 若当前只有一个稳定实现、调用关系也不复杂，禁止仅为了“面向接口编程”新增 `Xxx` 接口 + `XxxImpl` 实现的双层结构。
- 简单处理逻辑默认不需要抽象接口；只有在存在多个真实实现、明确替换需求、跨边界装配需求或测试替身确实能显著降低成本时，才允许引入接口。
- 抽接口前必须能写出现有或本轮即将落地的第二实现与调用边界；列不出时，接口层只会制造跳转成本。
- 大多数代码不需要额外封装；封装必须让调用点更少、主流程更清楚、边界更明确或复用更真实，否则优先直写。
- 不要把三五行局部逻辑搬到 helper / wrapper / manager 里制造跳转；读者能在原位置一眼看懂时，原位置就是更好的位置。
- 不要为了“每个步骤一个函数”把线性主流程（如中间件时间序列、请求处理管线）打散成一串一次性小函数：主线顺序本身就是最好的表达，过度拆分会稀释主线，并在带 doc 头模板的项目里把样板注释成倍放大（如 5 行取值逻辑配 4 行 doc 头）。判断标准是“这段是否真的复杂或被复用”，不是“能不能单独成函数”。
- 不要把极短、单调用点、无副作用的检查职责单独抽成函数来换取表面职责分离；这种场景应优先在调用处保留直白代码，并用清晰注释补足业务语义。只有当检查规则会复用、规则名本身是稳定业务术语、逻辑超过局部可读范围、存在副作用或需要独立测试时，才保留独立函数。
- 不要创建仅转发、仅改名、仅换参数顺序、仅包 try/catch 且不承接真实边界的浅模块；抽象必须隐藏实际复杂度、稳定协议或跨边界差异。
- 不要把“未来可能扩展”“看起来更规范”“先留好扩展点”当成简单场景抽接口的充分理由。
- Go 多参数函数若单行签名已明显影响理解，应优先提炼 `xxxParams` 结构体；不建议仅通过参数换行规避可读性问题。
- Go 第三方 API 响应解析默认禁止使用 `map[string]any` + key 硬编码读取；仅在“字段未知、临时探测响应”场景可短期使用，且必须在同任务内或最近一次迭代收敛为结构体解析。
- 已存在可复用公共工具时，禁止再写一份同语义工具函数；“重复封装”不属于可读性优化。
- 若第三方 API 响应结构缺文档，先通过测试脚本探测真实响应，再定义结构体；若仍无法确定结构，暂停并反馈用户补充响应结构。
- 不要把一条格式化日志拆成很多段交错换行；如果必须换行，先保留模板，再把参数按语义分组下移，不要一参一行地铺开。
- 禁止在业务方法中直接使用 `jsonObject.optString("xxx")`、`map["xxx"]` 这类字符串 key 取值；应在解析层集中映射为 DTO/对象后再进入业务流程。
- 业务逻辑内部函数默认不要对入参做防御性字符串清洗（如 `strings.TrimSpace`、大小写归一化），除非已有证据表明这条数据路径会产生脏数据；输入规范化应在请求解析、DTO 校验等边界层完成一次，业务函数应直接信任已进入内部的参数，不要逐层重复清洗。

## 格式化日志示例

推荐示例：

```go
applog.Errorf("上游接口请求失败，order_id=%s stage=%s endpoint=%s timeout_ms=%d latency_ms=%d err=%v",
	orderID, stage, resolved.URL,
	timeout.Milliseconds(), time.Since(startAt).Milliseconds(), err,
)
```

反例：

```go
applog.Errorf(
	"上游接口请求失败，order_id=%s stage=%s endpoint=%s timeout_ms=%d latency_ms=%d err=%v",
	orderID,
	stage,
	resolved.URL,
	timeout.Milliseconds(),
	time.Since(startAt).Milliseconds(),
	err,
)
```

## 需要暂停并确认的条件

- 为了提升可读性必须做大范围重构。
- 当前实现牵涉多个职责，拆分后会明显改变模块边界。
- 为了让代码更清晰，必须同步调整接口、数据结构或目录归属。
- 用户更关心快速止血，不接受当前阶段做结构梳理。
- 目标文件已超过 500 行，但拆分将引发跨模块接口重排或发布风险，需要先确认拆分边界。

## 执行通过 / 驳回标准

- 通过：主流程清晰、关键分支可追、函数职责相对单一、读者能快速理解执行顺序，且没有用更短但更难读的写法替代清楚表达；副作用从命名、位置或流程分段中可见，查询型与命令型职责没有隐蔽混杂；新增封装确实降低了复杂度或提升了真实复用，并形成简单接口隐藏真实复杂度的深模块；对 500 行及以上且继续扩展的文件已完成拆分或给出本轮可执行拆分方案；改动位点注释补齐已联动完成；同语义能力优先复用既有公共工具，无重复封装。
- 驳回：虽然代码能跑，但逻辑跳跃、嵌套过深、职责混杂、阅读成本过高，或为了“更优雅”而引入更难懂的抽象、嵌套三元和过密一行表达；或查询型函数暗中写库、发请求、改状态、发事件、写文件，导致副作用不可见；或在简单场景里无真实多实现、复用、边界隔离或复杂度下降证据，却强行新增接口层、接口实现对、helper、factory、manager、adapter 或仅转发/仅改名/仅包一层的浅封装；或 500 行及以上文件仍持续堆功能却未做拆分评估；或结构改动后未联动注释补齐；或已存在公共工具却重复封装同语义函数。

## 执行结果归档要求

- 如果为了控制可读性而主动放弃某些重构点，将结论记录到 `analysis/` 或 `doc/6-审查/`。
- 归档内容至少包含当前可读性问题、保留原因和后续建议。
- 如果本次只做常规可读性约束且无争议，可以不单独归档。
