# 代码可读性规则

只在判断“当前实现是否清晰、自然、容易接手”时使用这个 skill。
如果当前争议是改动范围、项目风格、注释颗粒度或目录归位，请转交相邻 skill。

## Skill 作用与适用场景

- 保持函数结构清晰、逻辑顺序自然。
- 控制单个函数或模块的复杂度，避免隐式跳转和过深嵌套。
- 让后续维护者能快速看懂输入、处理过程和输出。
- 避免炫技式写法、过度封装、过度压缩和“为了短而短”的表达。
- 在行为不变的前提下，优先简化最近改动代码中明显绕、密、重的表达。
- 完成结构调整后，确保改动位点触发并完成 `code-comment-rules` 注释补齐。

## 自动触发信号

- 新增业务函数、服务方法、控制器逻辑、工具函数。
- 修改已有复杂逻辑。
- 修复 Bug 后需要重排逻辑顺序或拆小函数。
- 编写脚本或任务流程代码时需要控制步骤可读性。

## 进入后先做什么

1. 先识别当前实现的主流程。
2. 确认读代码的人能否快速看出输入、关键分支和输出。
3. 找出过深嵌套、职责混杂、命名模糊、控制流跳跃、嵌套三元和过密表达的位置。
4. 判断哪些地方是真正影响理解的冗余抽象或包装，不要为了“简化”拆掉有价值的结构。
5. 在不扩大改动范围、且功能完全不变的前提下优化结构表达。

## 默认执行流程

1. 先读取 `references/readability-general.md`，确认本次实现的可读性底线。
2. 如果涉及函数拆分、步骤组织和控制流安排，再读取 `references/function-structure-rules.md`。
3. 如果需要判断某种写法是否虽然可运行但可读性很差，再读取 `references/readability-examples.md`。
4. 先保证主流程直观，再处理局部细节；清晰优先于短小，显式优先于炫技。
5. 如果改善可读性会明显扩大改动范围，暂停并交还给 `code-minimal-change-rules` 判断。
6. 结构改动落地后，联动 `code-comment-rules` 检查改动位点注释是否补齐（含 `1/2/3` 就近注释）。

## 权责边界与不负责事项

- 只负责结构可读性，不替代 `code-minimal-change-rules` 控制改动范围。
- 不替代 `code-style-consistency-rules` 处理项目风格一致性。
- 不替代注释类 skill 决定中文注释的颗粒度和写法。
- 不替代 `naming-rules` 单独决定业务术语命名。

## 需要避免

- 在路由文件中，若接口数量较少（例如 <= 10），禁止为“复用”引入额外抽象层（如 route struct + 批量注册函数），优先显式逐条注册。
- 可读性优先于“看起来通用”的抽象；读者应一眼看到完整接口路径与处理函数映射关系。

## 需要暂停并确认的条件

- 为了提升可读性必须做大范围重构。
- 当前实现牵涉多个职责，拆分后会明显改变模块边界。
- 为了让代码更清晰，必须同步调整接口、数据结构或目录归属。
- 用户更关心快速止血，不接受当前阶段做结构梳理。

## 执行通过 / 驳回标准

- 通过：主流程清晰、关键分支可追、函数职责相对单一、读者能快速理解执行顺序，且没有用更短但更难读的写法替代清楚表达；改动位点注释补齐已联动完成。
- 驳回：虽然代码能跑，但逻辑跳跃、嵌套过深、职责混杂、阅读成本过高，或为了“更优雅”而引入更难懂的抽象、嵌套三元和过密一行表达；或结构改动后未联动注释补齐。

## 执行结果归档要求

- 如果为了控制可读性而主动放弃某些重构点，将结论记录到 `analysis/` 或 `review/`。
- 归档内容至少包含当前可读性问题、保留原因和后续建议。
- 如果本次只做常规可读性约束且无争议，可以不单独归档。

## references 读取规则

- 默认先读 `references/readability-general.md`。
- 只有在安排函数结构和步骤顺序时，再读 `references/function-structure-rules.md`。
- 只有在判断写法优劣或做正反例对照时，再读 `references/readability-examples.md`。
