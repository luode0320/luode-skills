# 可读性正反例

## 正例

### 正例 1：先校验再主流程

- 先做参数校验和前置条件判断。
- 通过后再进入主流程。
- 主流程结束后统一返回结果。

### 正例 2：长流程拆成几个明确步骤

- 查询数据。
- 组装业务上下文。
- 执行核心逻辑。
- 输出结果。

### 正例 3：多条件分支写清楚

- 多个条件判断使用清晰的 `if/else` 或早返回。
- 不为了少几行把判断塞成嵌套三元。

### 正例 4：多参数函数先尝试单行，再按需提炼参数结构体

- 函数签名参数在可读前提下保持单行，读者可一眼看全输入。
- 当参数语义复杂度继续上升时，提炼 `xxxParams` 结构体再传入。
- 结果是签名清晰且可扩展，不靠参数换行硬撑可读性。

### 正例 5：第三方 API 响应先探测再结构体解析

- 第三方文档只有 curl 示例时，先用测试脚本调用获取真实响应样例。
- 基于样例定义 `internal/entity/<domain>/` 响应结构体，再用 `json.Unmarshal` 到结构体。
- 结果是字段语义稳定，后续维护和重构成本更低。

### 正例 6：简单逻辑保持原地直写

- 当前流程只需要补三五行局部判断或字段组装。
- 直接在主流程附近写清楚输入、判断和输出。
- 结果是读者不需要跳到 helper、manager 或 wrapper 才能理解简单逻辑。

### 正例 7：副作用从名称和位置可见

- 查询函数只负责读取和组装结果，不写库、不发事件、不修改状态。
- 写库、发请求、改状态或发事件的步骤放在命令型函数中，名称体现创建、更新、提交、同步或发送。
- 结果是读者看到调用点就能判断这里会不会改变外部世界。

### 正例 8：抽象形成深模块

- 一个对接上游的模块用简单方法暴露业务动作，内部处理鉴权、重试、字段映射、错误转换和日志。
- 调用方只需要理解稳定输入输出，不需要知道上游协议细节。
- 结果是简单接口隐藏真实复杂度，抽象降低了阅读和维护成本。

## 反例

### 反例 1：一个函数什么都做

- 既查库、又转 DTO、又发消息、又记录审计、又做格式化。
- 结果是主流程无法一眼看懂。

### 反例 2：为了“高级感”引入难懂抽象

- 明明几步直白流程即可完成，却套多层高阶封装或链式技巧。
- 结果是阅读成本高于直接写清楚的普通流程。

### 反例 3：为了“更短”压缩表达

- 多个条件写成嵌套三元。
- 多步处理写成难读的一行链式调用。
- 结果是代码更短，但理解成本更高。

### 反例 4：多参数函数直接拆成多行签名

- 参数较多时直接改成“每个参数一行”，没有先评估单行签名或参数结构体。
- 函数声明需要跨多行拼接语义，阅读入口成本上升。
- 结果是形式上“规整”，但可读性没有真正提升。

### 反例 5：第三方 API 响应用 map + key 硬编码长期解析

- 用 `map[string]any` 接收第三方响应，再通过字符串 key 层层断言读取字段。
- 没有脚本探测响应样例，也没有定义结构体模型。
- 结果是类型不稳定、异常分支隐式增多，维护时容易踩空字段。

### 反例 6：为了封装而封装

- 只有一个调用方、一个稳定实现和几行直白逻辑，却新增 `Helper`、`Factory`、`Manager`、`Wrapper` 或 `Adapter`。
- 主流程从一处可读变成多文件跳转。
- 结果是形式上更“工程化”，实际阅读成本更高。

### 反例 7：为单实现创建接口层

- 当前只有一个 `Reader` 实现，也没有插件、外部适配或真实替换需求。
- 仍新增 `Reader` 接口、`DefaultReader` 实现和构造注入，只因为“面向接口编程”或“以后可能 mock”。
- 结果是调用链多一跳，读者需要跨接口和实现体才能看清原本简单的流程。

### 反例 8：查询函数暗中产生副作用

- 函数名叫 `getUserProfile` 或 `buildOrderPreview`，内部却顺手更新数据库、发送消息或改缓存状态。
- 调用方以为只是读取或计算，实际会改变外部状态。
- 结果是副作用不可见，后续排查回归时很难定位状态从哪里被改掉。

### 反例 9：浅封装制造跳转

- 新增 `OrderService.process()` 只是调用 `orderRepo.save()`，没有校验、事务、映射、错误转换或跨边界适配。
- 新增 `formatHelper()` 只是给一行直白表达换了名字，也没有复用。
- 结果是代码看似分层，实际只是让读者多跳文件，未隐藏任何真实复杂度。
