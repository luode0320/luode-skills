# 可读性正反例

## 正例

### 正例 1：先校验再主流程

- 先做参数校验和前置条件判断。
- 通过后再进入主流程。
- 主流程结束后统一返回结果。

### 正例 2：长流程拆成几个明确步骤

- 查询数据。
- 组装业务上下文。
- 执行核心逻辑。
- 输出结果。

### 正例 3：多条件分支写清楚

- 多个条件判断使用清晰的 `if/else` 或早返回。
- 不为了少几行把判断塞成嵌套三元。

### 正例 4：多参数函数先尝试单行，再按需提炼参数结构体

- 函数签名参数在可读前提下保持单行，读者可一眼看全输入。
- 当参数语义复杂度继续上升时，提炼 `xxxParams` 结构体再传入。
- 结果是签名清晰且可扩展，不靠参数换行硬撑可读性。

### 正例 5：第三方 API 响应先探测再结构体解析

- 第三方文档只有 curl 示例时，先用测试脚本调用获取真实响应样例。
- 基于样例定义 `internal/entity/<domain>/` 响应结构体，再用 `json.Unmarshal` 到结构体。
- 结果是字段语义稳定，后续维护和重构成本更低。

## 反例

### 反例 1：一个函数什么都做

- 既查库、又转 DTO、又发消息、又记录审计、又做格式化。
- 结果是主流程无法一眼看懂。

### 反例 2：为了“高级感”引入难懂抽象

- 明明几步直白流程即可完成，却套多层高阶封装或链式技巧。
- 结果是阅读成本高于直接写清楚的普通流程。

### 反例 3：为了“更短”压缩表达

- 多个条件写成嵌套三元。
- 多步处理写成难读的一行链式调用。
- 结果是代码更短，但理解成本更高。

### 反例 4：多参数函数直接拆成多行签名

- 参数较多时直接改成“每个参数一行”，没有先评估单行签名或参数结构体。
- 函数声明需要跨多行拼接语义，阅读入口成本上升。
- 结果是形式上“规整”，但可读性没有真正提升。

### 反例 5：第三方 API 响应用 map + key 硬编码长期解析

- 用 `map[string]any` 接收第三方响应，再通过字符串 key 层层断言读取字段。
- 没有脚本探测响应样例，也没有定义结构体模型。
- 结果是类型不稳定、异常分支隐式增多，维护时容易踩空字段。
