# 函数结构规则

## 用途

用于组织函数内部步骤、控制流和拆分方式。

## 默认结构

1. 先处理前置校验。
2. 再处理主业务流程。
3. 最后处理结果返回、清理或补充动作。

## 建议

- 让同一层级的逻辑处在相近抽象层次。
- 对长流程优先按步骤拆小，而不是继续叠加分支。
- 对异常分支优先早返回或早失败，减少主流程被嵌套包裹。
- 查询步骤和副作用步骤优先分开；同一函数确实要先查再写时，应先读上下文，再集中执行写库、发请求、改状态、发事件等命令步骤。
- 会产生副作用的函数名称应体现动作语义，例如创建、更新、提交、同步、发送、持久化；不要用 `get`、`list`、`query`、`calc`、`build` 伪装副作用。
- 拆出的辅助函数名称应能直接说明作用，避免 `processData`、`handleThing` 这类模糊命名。
- 拆出的辅助函数必须隐藏真实复杂度或稳定边界；仅转发、仅改名、仅包一层的函数优先内联回主流程。
- 职责拆分要看收益，不按“每个职责一个函数”机械执行。极短的局部检查、判空、匹配器取用、scope/flag 选择等逻辑，如果只有一个调用点、无副作用、无复杂分支、无独立测试价值，优先就地内联，并用步骤注释说明业务含义。
- 当检查规则会复用、规则名本身是稳定业务术语、逻辑超过局部可读范围、存在副作用或需要独立测试时，再拆成独立函数。
- 多条件分支不要写成嵌套三元，优先用清晰的 `if/else` 或分支函数。
- Go 函数签名参数默认保持单行；若参数过多导致语义负担上升，优先提炼参数结构体而不是直接多行换参。
