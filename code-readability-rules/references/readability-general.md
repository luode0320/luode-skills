# 可读性总则

## 用途

用于定义什么样的代码结构更容易被团队持续维护，什么样的写法虽然能运行但不利于阅读。

## 基本要求

- 主流程应尽量按业务顺序展开。
- 关键输入、关键判断和关键输出应容易看见。
- 函数应尽量保持单一职责，不把初始化、校验、核心逻辑、落库、通知混成一团。
- 能直接表达的逻辑，不要用过度抽象或过度技巧包装。
- 清晰优先于短小，显式优先于过度压缩。
- 单文件达到 500 行及以上且仍持续新增功能时，默认应按职责拆分到多个文件；业务流程明显增多时应进一步拆子目录/子包。

## 常见反模式

- 过深嵌套。
- 隐式副作用过多。
- 一个函数承担多个阶段职责。
- 过度追求“短”，导致语义被压缩得难以理解。
- 多条件判断塞成嵌套三元。
- 明明几步更清楚，却强行压成一行链式表达。
- 第三方 API 响应长期使用 `map[string]any` + key 硬编码解析，导致字段语义分散、类型不稳定、维护成本上升。

## 第三方 API 响应解析补充

- Go 接入第三方 API 时默认优先结构体解析，建议把响应结构体放在 `internal/entity/<domain>/`。
- 当文档缺失响应模型时，应先用测试脚本调用并获取真实响应样例，再补结构体。
- 若无法通过文档或测试脚本确认响应结构，应暂停并反馈用户补充响应字段说明，避免盲写 map 解析。
