# 可读性总则

## 用途

用于定义什么样的代码结构更容易被团队持续维护，什么样的写法虽然能运行但不利于阅读。

## 基本要求

- 主流程应尽量按业务顺序展开。
- 关键输入、关键判断和关键输出应容易看见。
- 函数应尽量保持单一职责，不把初始化、校验、核心逻辑、落库、通知混成一团。
- 能直接表达的逻辑，不要用过度抽象或过度技巧包装。
- 清晰优先于短小，显式优先于过度压缩。
- 会写库、发请求、改状态、发事件、写文件或触发异步任务的函数，必须让副作用从名称、调用位置或职责层级上可见。
- 查询型函数默认不修改状态；命令型函数默认不伪装成纯查询。确因事务边界必须合并时，名称和流程分段都要暴露命令语义。
- 简单流程、单实现依赖或局部辅助处理，优先直接使用具体实现；不要为了“面向接口”或“先留扩展点”默认多抽一层接口。
- 抽接口前必须能说清真实第二实现、稳定调用边界或显著测试替身收益；说不清时，具体类型和具体函数更清楚。
- 大多数代码默认不需要封装。封装只有在减少真实重复、隔离稳定边界、降低主流程复杂度或承接多个真实实现时，才是可读性收益。
- 好抽象应形成深模块：简单接口背后隐藏真实复杂度、协议差异或稳定边界。
- 如果封装只是把几行直白逻辑藏到 helper、wrapper、manager、factory 或 adapter 里，让读者多跳一次文件，它降低的是可读性。
- 单文件达到 500 行及以上且仍持续新增功能时，默认应按职责拆分到多个文件；业务流程明显增多时应进一步拆子目录/子包。

## 常见反模式

- 过深嵌套。
- 隐式副作用过多。
- 函数名像查询或计算，实际却写库、发请求、改状态、发事件或写文件。
- 同一个函数同时 query 和 command，且名称、分段和调用位置都看不出副作用边界。
- 一个函数承担多个阶段职责。
- 过度追求“短”，导致语义被压缩得难以理解。
- 多条件判断塞成嵌套三元。
- 明明几步更清楚，却强行压成一行链式表达。
- 明明只有一个稳定实现，却为了“规范”引入接口、实现体、构造函数多跳转，导致读者需要跨多个文件才能看清简单逻辑。
- 明明没有第二实现，却为了“以后可能会用”或“方便 mock”创建接口层，导致调用链多一跳但事实仍只有一个实现。
- 明明局部直写更清楚，却为了“封装一下”新增无复用 helper / wrapper / manager，导致简单逻辑分散在多处。
- 明明只是转发、改名或换参数顺序，却包装成 service / helper / adapter，让读者多跳一层仍看不到新语义。
- 第三方 API 响应长期使用 `map[string]any` + key 硬编码解析，导致字段语义分散、类型不稳定、维护成本上升。

## 第三方 API 响应解析补充

- Go 接入第三方 API 时默认优先结构体解析，建议把响应结构体放在 `internal/entity/<domain>/`。
- 当文档缺失响应模型时，应先用测试脚本调用并获取真实响应样例，再补结构体。
- 若无法通过文档或测试脚本确认响应结构，应暂停并反馈用户补充响应字段说明，避免盲写 map 解析。
