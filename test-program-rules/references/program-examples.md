# 测试程序正反例

## 正例

- 在正式测试文件中只保留场景编排和断言，把复杂数据构造抽到独立 builder。
- 把共享 mock 客户端放到测试公共辅助目录，由多个接口测试复用。
- 将测试环境初始化和清理逻辑拆成独立 setup / teardown 脚本，而不是堆在主测试程序里。
- 将一次性调试脚本清理后，把有长期价值的验证逻辑沉淀为正式测试用例。

## 反例

- 一个脚本同时做环境切换、构造数据、发请求、验结果、导报告、清资源。
- 为了测试方便，把假实现和测试常量直接写进生产 `service` 或 `utils`。
- 把临时调试脚本直接提交进正式测试目录，却没有说明生命周期和清理条件。
- 明明只服务一个任务，却过度抽象成大型公共测试框架。

## 判定提示

- 如果脚本去掉注释后已经看不出主职责，通常说明拆分不够。
- 如果测试程序为了复用而引入过多间接层，导致阅读成本高于复制少量代码，通常说明抽象过度。
- 如果当前逻辑只为定位问题存在，先按调试资产处理，不要直接纳入正式测试链路。
