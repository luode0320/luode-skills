# 测试程序正反例

## 正例

- `test/2026-03-28_180000/internal/service/history_client_test.go` 只保留场景编排和断言，把复杂数据构造抽到独立文件。
- `test/2026-03-28_180000/internal/service/history_client_mock.json` 作为同一镜像路径下的 mock 数据，与主测试程序就近归档。
- `test/2026-03-28_180000/历史客户端查询验证/README.md` 只说明测试目标、运行方式和真实资产路径，不承载实际脚本。
- 将一次性调试脚本清理后，把有长期价值的验证逻辑沉淀为正式测试用例。

## 反例

- 在 `test/2026-03-28_180000/历史客户端查询验证/` 目录里直接放 `history_client_test.go`，把真实测试代码塞进中文目录。
- 一个脚本同时做环境切换、构造数据、发请求、验结果、导报告、清资源。
- 为了测试方便，把假实现和测试常量直接写进生产 `service` 或 `utils`。
- 把临时调试脚本直接提交进正式测试目录，却没有说明生命周期和清理条件。
- 把 Go 测试包目录命名成 `中文目录`，导致 `go test ./...` 失败。

## 判定提示

- 如果测试代码没有按真实代码路径镜像放置，通常说明落点先出了问题。
- 如果脚本去掉注释后已经看不出主职责，通常说明拆分不够。
- 如果当前逻辑只为定位问题存在，先按调试资产处理，不要直接纳入正式测试链路。
