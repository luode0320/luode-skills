# 测试程序类型与拆分规则

## 用途

用于区分不同测试程序的职责，并明确它们应该落在时间戳根目录中的哪个 ASCII 镜像路径下。

## 真实路径镜像基线

- 所有真实测试资产都放在 `test/yyyy-MM-DD_HHmmss/<ASCII 真实代码路径镜像>/` 下。
- 如果源文件是 `internal/service/history_client.go`，则测试资产默认镜像到 `test/yyyy-MM-DD_HHmmss/internal/service/`。
- 对应文件名建议沿用源文件语义，例如：
  - `history_client_test.go`
  - `history_client_mock.json`
  - `history_client_cases.json`
  - `history_client_run.ps1`
- 中文只允许出现在 `test/yyyy-MM-DD_HHmmss/测试任务中文简介/README.md`，不能进入 Go 可编译路径。

## 常见程序类型

- 正式测试程序：承载断言和验证流程，是长期保留的测试主体。
- 模拟程序：提供 mock、stub、fake、假服务、假依赖。
- 数据构造脚本：专门生成测试输入、测试前置数据和样例数据。
- 初始化 / 清理脚本：负责测试环境准备、资源清理、前后置动作。
- 共享测试辅助代码：沉淀通用断言、公共构造器、公共客户端、公共装配逻辑。

## Go 场景矩阵（同一规则）

- 白盒单测（需要访问未导出符号）：
  - 禁止在源码目录同包落地 `*_test.go`。
  - 必须先补可外部测试 seam（导出接口 / 依赖注入点 / 测试钩子），再把测试程序放到 `test/<时间戳>/<ASCII 镜像路径>/`。
- 黑盒单测：
  - 直接放在 `test/<时间戳>/<ASCII 镜像路径>/`，不进源码目录。
- 集成测试：
  - 直接放在 `test/<时间戳>/<ASCII 镜像路径>/`，不进源码目录。

结论：三类场景是同一落点规则，不存在白盒例外。

## 拆分原则

- 一个测试程序只承担一种主职责，避免混合过多阶段动作。
- 正式测试程序优先聚焦场景编排和断言，不承载过多数据拼装细节。
- 数据构造和环境初始化应根据复用度拆离，不强行塞进主测试脚本。
- 共享辅助代码只抽离重复且稳定的逻辑，避免把一次性逻辑过早抽公共。
- 一切真实测试程序都必须先落到 ASCII 镜像路径，再谈拆分，不允许直接丢进中文说明目录。
- Go 场景无论白盒/黑盒/集成，一律不在源码目录新增 `*_test.go`。

## 检查清单

- 当前测试程序是否能一句话说清主职责。
- 当前脚本里是否混入环境切换、调试输出、数据准备、断言、报告生成等多种职责。
- 当前辅助代码是否真的存在复用价值。
- 当前拆分是否会让后续功能验证更清晰，而不是更绕。
- 当前程序是否位于 `test/yyyy-MM-DD_HHmmss/<ASCII 真实代码路径镜像>/` 下，而没有散落到其他目录。
- Go 源码目录是否不存在 `*_test.go`；若有白盒诉求，是否已经改为 seam 方案。
