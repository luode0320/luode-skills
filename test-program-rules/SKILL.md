# 测试程序规则

只在“测试程序本身应该怎么拆、怎么组织职责”这个问题上使用这个 skill。
如果当前争议是测试目录放哪里，请转交 `test-location-rules`；如果只是测试名称不统一，请转交 `test-naming-rules`。

## Skill 作用与适用场景

- 作为测试资源管理链的第三道规则，在位置和命名稳定后约束测试程序结构。
- 统一测试程序、模拟程序、验证脚本、数据构造脚本、测试辅助代码的职责边界和拆分方式。
- 防止临时验证代码、一次性调试脚本、伪业务辅助代码混进正式测试资产或生产目录。
- 提升测试程序的可读性、可维护性和复用稳定性。

## 自动触发信号

- 新增或修改测试程序、mock server、stub、fake、数据构造脚本、测试初始化脚本、验证脚本。
- 需要决定测试逻辑应写进正式测试文件、共享测试辅助代码还是一次性调试脚本。
- 发现测试脚本同时承担数据构造、环境初始化、业务调用、断言、报告输出等多种职责。
- 发现测试辅助逻辑被塞进生产 `utils`、`common`、`service` 等目录中。
- 发现测试程序准备放到 `testing/`、根目录、业务目录或其他非 `test/` 任务目录位置时。

## 进入后先做什么

1. 先确认测试目录落点和命名已经稳定，不在当前 skill 内重新决定位置和名称。
2. 确认当前程序位于 `test/` 根目录下的当前任务目录中；若同一需求存在多个独立测试程序或多轮验证，应拆到多个任务目录。
3. 区分当前对象属于正式测试程序、模拟程序、数据构造脚本、初始化脚本、清理脚本还是共享测试辅助代码。
4. 判断当前代码是长期保留的正式测试资产，还是仅用于临时排障的诊断代码。
5. 明确每个测试程序只承担一种主职责，避免脚本持续膨胀。

## 默认执行流程

1. 默认先读 `references/program-types-and-splitting.md`，确定测试程序类型和拆分原则。
2. 如果发现测试程序与业务代码、调试代码或联调脚本混用，再读 `references/program-boundaries.md`。
3. 如果需要判断当前拆分方式是否合理，再读 `references/program-examples.md` 对照正反例。
4. 输出推荐的程序拆分方案、需要抽离的辅助代码、禁止保留的临时实现和最小改动建议。
5. 测试程序结构未稳定前，不进入 `test-doc-rules`、`functional-validation-rules` 和 `test-regression-rules`。

## 权责边界与不负责事项

- 只负责测试程序的职责划分和辅助代码边界，不负责决定测试资源目录落点，那属于 `test-location-rules`；但本 skill 必须服从其 `test/` 根目录和任务目录规则。
- 不负责测试目录、文件、脚本的命名，那属于 `test-naming-rules`。
- 不负责测试说明文档内容和结构，那属于 `test-doc-rules`。
- 不负责功能是否通过验证，也不负责回归范围判定，那属于 `functional-validation-rules` 和 `test-regression-rules`。
- 如果当前代码只是为定位问题而临时加的断点、日志、断言、一次性脚本，应优先交由 `bug-runtime-debug-rules` 或 `integration-debugging-rules` 处理。

## 需要暂停并确认的条件

- 当前测试程序既像正式测试资产，又像临时排障脚本，保留策略不清。
- 一个脚本承载多个职责，拆分后会影响大量既有引用，但影响面尚未确认。
- 测试辅助逻辑是否应沉淀为共享能力存在争议，贸然抽公共可能导致过度设计。
- 为整理测试程序结构需要顺带改动大量无关测试代码，已经偏离最小改动原则。

## 执行通过 / 驳回标准

- 通过：当前测试程序、模拟程序和辅助脚本职责清晰，正式测试资产与临时诊断代码边界明确，且没有把测试逻辑混入生产代码目录。
- 驳回：测试程序职责混杂、临时脚本长期滞留、测试辅助逻辑侵入生产代码或公共目录边界失衡，继续进入后续测试阶段会放大维护风险。

## 执行结果归档要求

- 将测试程序拆分结论统一记录到 `test/` 根目录下当前测试任务目录的 `README.md` 或同目录说明文件中，不得再写到 `testing/` 或 `analysis/`。
- 归档内容至少包含程序类型、职责划分、共享辅助代码范围、禁止保留项和待拆分项。
- 如果同一需求需要多个测试程序或多轮独立验证，应分别创建多个测试任务目录，并在各自目录内记录对应程序结论。
- 如果保留了临时诊断代码或桥接脚本，必须记录其清理条件和生命周期。

## references 读取规则

- 默认先读 `references/program-types-and-splitting.md`。
- 只有在出现业务代码混入、临时脚本滞留或职责冲突时，再读 `references/program-boundaries.md`。
- 只有在需要正反例时，再读 `references/program-examples.md`。
