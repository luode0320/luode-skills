# 自主连续执行规则

## 目标

本 Skill 是自主连续执行的唯一自动触发入口，只负责决定“能否继续、何时暂停、何时交接、何时结束”。领域实现、测试、审查、验收、安全审批和最终用户渲染仍由对应 Owner 负责。

详细契约按职责下沉：

- 执行授权与计划边界：`references/execution-authorization-and-plan-boundary.md`
- 继续、暂停与高风险边界：`references/continuation-and-pause.md`
- 阻断状态收口与事实交接：`references/blocker-handoff.md`
- 用户结束、无下一步与防循环：`references/termination-loop-prevention.md`

## 执行许可状态（新增）

只维护三态：`confirmed`、`unknown`、`revoked`。进入首次编码、上下文压缩恢复或用户改变边界时，必须按执行授权契约刷新状态；不得把聊天建议、口头方向或开工短句直接视为无边界执行许可。

完整状态转换、计划必填项和采纳建议场景见 `references/execution-authorization-and-plan-boundary.md`。

## 触发信号

- 自动触发：多步骤任务尚未闭环，且存在原执行计划内、完成原始目标所必需、当前可安全执行的下一步。
- 不触发或立即停止：用户明确结束、原始目标已完成、仅剩可选优化、执行许可不足、计划边界缺失、上下文压缩后编码许可未确认，或已进入关键/高风险节点。
- 自动触发不依赖用户再次说“继续”，也不得把“可继续优化”升级成新目标。

触发与排除样例见 `references/continuation-and-pause.md` 和 `references/termination-loop-prevention.md`。

## 默认执行策略

1. 先确认有效计划、完成定义、停止条件、最大推进边界和验证点。
2. 获得 `confirmed` 后，按当前计划连续推进，不在普通步骤重复询问；正式实施周期首次进入执行时先由 `task-plan-rehydration-rules` 持久化任务投影，再调用 `update_plan`。
3. 每个最小任务必须独立完成“实现 -> 真实测试 -> 审查 -> 验收”，不得批量实现后统一验证；每次状态迁移都先原子更新投影，再刷新悬浮任务列表。
4. 当前周期未收口不得跳到后续周期；当前周期收口且仍有计划内必需周期时自动继续。当前投影全部完成后先写为 `inactive`，后续回合不得重放。
5. 中间进度只说明已完成与正在推进的必需动作，不交还非关键执行权；`update_plan` 不可用或失败时只能说明磁盘投影已保存，不得声称 UI 已刷新。
6. 遇到关键节点、高风险、越界、冲突或授权不足时立即暂停。
7. 真实 `blocked/manual_handoff` 先收口运行时状态，再交接结构化事实。
8. 原始目标完成后只允许三类合法后续；无合法后续时直接结束。

完整执行节奏见 `references/continuation-and-pause.md`。

## 必须暂停确认的关键节点

- 首次从需求、Bug 分析或方案进入编码，且许可不是 `confirmed`。
- 开工指令缺少正式计划或停止边界；采纳建议但建议尚未转成执行计划。
- 上下文压缩后准备开始或继续编码，且编码许可未重新确认。
- 进入提交、推送或 PR 等 Git 动作，但当前轮没有明确 Git 授权。
- 方案越界、高风险或不可逆操作、成本显著上升、证据冲突无法收敛。
- 用户明确结束、无合法后续、运行时状态尚未真实收口或已满足真实阻断条件。

逐项处理见 `references/continuation-and-pause.md`、`references/blocker-handoff.md` 与 `references/termination-loop-prevention.md`。

## 关键节点提问格式（强制）

仅在确需用户决策时，简要说明当前阶段和节点类型，给出 2-3 个互斥可执行选项，推荐项置首；禁止默认开放式追问“接下来做什么”。用户已明确 Git 动作时直接让路给 `git-collaboration-rules`，不得重复询问是否提交。

## 与其他 Skill 的协作

- `git-collaboration-rules`：独占当前轮 Git 授权、提交和推送边界。
- `skill-execution-compliance-gate-rules` / `code-change-finalization-gate-rules`：独占执行合规与代码最终收口；自主执行不得跳过。
- `context-compression-rules`：提供压缩恢复事实；编码许可仍按本 Skill 的授权契约复核。
- `task-plan-rehydration-rules`：独占任务投影持久化、校验、失活和 `update_plan` payload；UI 重建不恢复本 Skill 的执行许可。
- 需求、Bug、实施、测试、审查、验收 Skill：保留领域决策和阶段门禁，本 Skill 只控制推进节奏。
- `reasoning-summary-structure-rules`：唯一负责面向用户渲染阻断与最终状态；本 Skill 只交接结构化事实。

## 禁止事项

- 不得绕过系统安全限制、权限审批、外部副作用确认或高风险防护。
- 不得以连续执行为由跳过正式计划、真实验证、审查、验收或证据。
- 不得继承上下文压缩前无法证明的编码许可。
- 不得在用户结束或无合法后续后继续工具调用、建议或占位输出。

## 通过 / 驳回标准

通过：单入口自动触发；授权状态正确；计划内必需动作连续推进；每个最小任务独立闭环；关键节点准确暂停；阻断事实完整交接；用户结束和无下一步语义生效；安全审批未被绕过。

驳回：重复询问普通步骤；无计划直接实施；压缩后直接编码；跨任务批量验证；越界或高风险静默继续；伪造运行时收口；自行渲染重复阻断；任务完成后继续扩散。

## 快速示例

- “开始执行”但没有执行计划：先形成受限计划并停在计划收口，不进入编码。
- 当前最小任务实现完成且测试、审查、验收尚未完成：继续完成该任务闭环，不询问是否继续。
- 用户说“结束”：立即停止自动继续和扩散，只做必要的真实状态收口与最小结论。
- 原始目标完成且无三类合法后续：直接结束，不输出等待、无需动作或下一步占位文案。
