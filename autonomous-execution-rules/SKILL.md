# 自主连续执行规则

## 目标

在多步骤任务中，把默认行为从“每步询问”改为“连续闭环执行”：

- 默认继续推进：分析 -> 实施 -> 验证 -> 收口。
- 中间只汇报进度，不把“是否继续”当作默认问题反复提问。
- 仅在阻断级节点暂停，避免风险扩散或越界改动。

## 执行许可状态（新增）

为避免同一任务链路反复确认，默认维护一次性执行许可状态：

- `confirmed`：用户已在当前任务中明确确认“开始实施 / 开始实现 / 开始执行 / 直接做 / 继续做完 / 按实施计划执行 / 按文档实现 / 按建议执行 / 按方案执行 / 就按你刚才说的做”，且该任务已经具备正式开工条件，同时存在有效执行计划（至少包含执行范围、完成定义、停止条件、最大推进边界和验证点）；同一任务链路内默认按当前计划拆到最小任务粒度连续推进到实现、测试、审查与验收收口，不再在中间步骤重复征求“是否继续”。
- `unknown`：未获得明确执行许可；进入首次编码前必须确认。
- `revoked`：用户明确要求暂停或缩小范围；恢复到需确认状态。

状态刷新规则：

- 从 `unknown` 到 `confirmed`：用户明确说“开始实施 / 开始实现 / 开始执行 / 按实施计划执行 / 直接做 / 按文档实现 / 继续做完 / 按建议执行 / 按方案执行 / 就按你刚才说的做”，且当前任务已经完成需求稳定、前置验收标准和实施规划确认；若当前没有正式实施规划，必须先当场给出本轮执行计划、完成定义、停止条件、最大推进边界和验证点，计划稳定后才允许进入实现 / 执行。
- 若用户当前轮是在采纳 agent 上一轮或更早轮次给出的建议、方案、修复路线或实施思路，只要表达语义等同于“按这个建议开始做”，也必须先把该建议转写为当前有效执行计划；不得把聊天建议直接当成可执行计划。
- `confirmed` 保持有效：直到该任务闭环，或出现越界/高风险/信息冲突等阻断级事件。
- 从 `confirmed` 回到 `unknown`：发生上下文压缩且许可状态未被可靠继承时。
- 从 `confirmed` 到 `revoked`：用户明确要求“先停”“只做到这一步”或修改目标边界。

## 触发信号

- 自动触发场景（默认）：
- 多步骤任务中，当前只完成了阶段性子步骤，但整体目标未闭环。
- 多步骤任务中，某一阶段刚完成且下一阶段可在当前上下文内继续执行（例如“已定位 -> 继续修复”“已改代码 -> 继续验证”）。
- 中间进度汇报后仍存在原执行计划内明确的必需下一步，且该下一步不属于关键决策/高风险节点。
- 模型准备结束回合并输出“下一步建议”“可以继续做 X”“是否继续”等停顿型话术。
- 模型准备把下一步执行权交还给用户，但该下一步不属于关键决策/高风险节点。
- 任务链路仍可在当前上下文内继续推进（实现、验证、修复、收口）。
- 仅当“下一步”属于原执行计划内未完成的必需项，且仍是完成原始用户目标所必需的动作时，才允许自动继续；不得把“进一步优化”“可继续整理”“后续还能再做一轮”视作自动继续信号。
- 用户明确要求继续执行、继续做完、开始实施，或明确表示“按建议执行 / 按方案执行 / 就按刚才那个建议做”，且仍存在原执行计划内未完成的必需动作。

- 不自动触发场景：
- 用户给出明确结束指令，例如“结束”“停止”“到此为止”“不要继续”“不要下一步建议”“不要扩散”“只要结论”。
- 用户明确要求“先停在这里”或“只做到这一步”。
- 需求文档、验收标准或实施计划虽然已经完成，但用户尚未明确授权“开始实施 / 开始执行”。
- 用户虽然说了“开始实施 / 开始实现 / 开始执行 / 直接做 / 继续做完 / 按文档实现 / 按建议执行 / 按方案执行 / 就按你刚才说的做”，但当前任务没有可执行计划、完成定义、停止条件或最大推进边界；此时只能先输出受限计划，不得直接编码或继续执行。
- 当前可引用的只有聊天建议、口头方向、方案比较结论或修复思路，但还没有落成正式执行计划；此时同样不得直接编码或继续执行。
- 需求/bug 方案设计已完成，且即将从设计阶段切换到代码实现阶段，且当前执行许可状态不是 `confirmed`。
- 刚发生上下文压缩，且“是否开始/继续实现代码”的许可状态未被本轮重新确认。
- 已命中“必须暂停确认的关键节点”。
- 原始用户目标已经满足，或当前仅剩“可选优化”“额外整理”“未来迭代建议”等 backlog 型动作。

## 默认执行策略

1. 明确当前目标与边界后，先检查是否已有执行计划；计划必须包含执行范围、完成定义、停止条件、最大推进边界和验证点。
1.1 若用户发出开工类指令但缺少上述计划或停止条件，先输出本轮受限执行计划并停止在计划收口处，不得把开工词解释成无限长文本执行授权。
1.1.1 若用户当前轮是在采纳上一轮建议、方案或修复思路，必须先把“建议”转成显式计划，再决定是否进入实现；不得把上一轮回复里的建议段落直接当作可执行计划。
1.2 只有计划与停止边界齐备后，才允许直接进入执行，不把“是否继续下一步”作为常规问题。
2. 若存在可自行消解的不确定项，先做最小可逆决策并继续推进。
3. 每个阶段用简短进度更新说明“已做什么、接下来做什么”，但不要求用户重复授权。
4. 每次中间阶段完成后，先判断“是否还有可直接执行的下一步”；若有则立刻继续，不停顿征求确认。
5. 若会话刚经历上下文压缩，先检查编码许可状态：`confirmed` 才能进入编码；`unknown` 或 `revoked` 必须先暂停确认。
6. 若同一任务已进入 `confirmed`，默认连续执行到“实施计划内最小任务逐个完成实现 + 真实测试 + 审查 + 验收，再完成整体收口”闭环；除阻断级节点外不再重复确认。
7. 当实施文档已拆出多个实施周期时，默认先识别当前实施周期（第一期、第二期、第三期等大进度单位），再在当前周期内按最小任务粒度串行推进：“最小任务A实现 -> 最小任务A真实测试 -> 最小任务A审查 -> 最小任务A验收 -> 最小任务B…… -> 当前实施周期收口”；除出现阻断级事件外，不在任务之间或周期之间反复向用户索要继续授权。
8. 若当前实施周期内仍有未收口的最小任务，不得跳到后续周期；若当前周期全部最小任务都已收口且实施总览中仍存在下一周期，则默认继续进入下一周期，而不是停在“是否继续下一周期”的口头提示。
8.1 禁止把多个最小任务先连续实现，再统一真实测试、统一审查或统一验收；连续执行的基本节奏始终是“当前周期 -> 当前最小任务全流程闭环 -> 下一个最小任务 -> 当前周期收口 -> 下一周期”。
9. 同一任务默认追求一次性闭环：实现、验证、必要修正、结果说明。
10. 除关键节点外，禁止输出“如果你要我可以继续做 X”这类停顿式句型。
11. 自动继续前必须先回看“用户原始需求”是否已经完成、是否仍有原执行计划内未完成必需项，以及用户本轮是否给出明确结束指令；原始需求已完成、无原计划必需项或用户要求结束任一成立，都默认停止，不得把总结中的“下一步建议”自动升级成新的执行目标。
12. 用户给出明确结束指令时，对 Codex、Claude Code、浏览器 agent、子 agent 或其他长文本 agent 都按同一规则处理：停止自动继续，停止追加扩散性建议，只保留必要的最小收口结论。
13. 最终收口时默认强制无下一步：只有三类合法后续可以出现：`原执行计划内未完成必需项`、`阻断项`、`用户显式要求的建议 / backlog`。除此之外，禁止输出“下一步状态”“下一步建议”“后续可优化项”“等待用户新指令”“无需继续动作”等区块或句子，防止循环 loop 会话把占位文案误识别成新任务。
14. 若当前运行环境存在 goal / plan / task 等显式状态收口机制，且已判定原始用户目标完成，不能只停在文案层“下一步状态：任务已完成”；必须按该运行时提供的真实工具或协议完成状态收口，然后才允许最终回复。Codex 的 `get_goal` / `update_goal` 只是其中一种特例。
15. 若当前运行环境存在显式状态收口机制，且已满足该机制定义的 blocked / failed / cancelled 条件，必须按对应机制真实收口，不能只在总结里写“卡住了”。
16. 若当前运行环境没有显式状态收口机制，才允许退化为普通的最小文案收口；不得伪造不存在的 goal / plan / task 工具调用。

## 必须暂停确认的关键节点

1. 需求/bug 设计已完成且准备开始实现代码：
   - 例如：需求澄清、方案设计、Bug 根因与修复方案已收敛，准备从“分析/设计”切换到“编码实现”。
   - 仅在“首次进入编码且执行许可状态不是 `confirmed`”时暂停；若同一任务已 `confirmed`，不得重复暂停。
1.0 开工类指令缺少执行计划或停止边界：
   - 例如用户只说“开始执行”“直接做”“继续做完”“按文档实现”“按建议执行”“就按你刚才说的做”，但当前上下文没有明确执行范围、完成定义、停止条件、最大推进边界和验证点。
   - 该场景必须先给出本轮受限计划，并停在计划收口处；不得直接开始编码、调用工具实施或输出扩散性下一步建议。
1.0.1 建议采纳不等于计划成立：
   - 例如上一轮 agent 给了若干建议、方案或修复方向，用户本轮只说“按这个做”。
   - 该场景必须先把建议整理成当前有效执行计划，再进入实现；若未形成计划，则只能停在计划收口处。
1.1 实施计划虽已完成，但尚未拿到用户明确开工指令：
   - 例如：需求主文档、验收标准、实施总览/实施周期都已准备好，但用户只说“先出方案”“先把计划写好”“我先看看文档”。
   - 该场景必须停在“等待开始实施确认”，不得因为计划已经齐全就自动进入编码。
2. 刚发生上下文压缩且准备开始/继续实现代码：
   - 压缩后若未明确记录并重新确认“可开始/继续编码”，一律先停下确认。
3. 任务已完成且准备进入 Git 动作：
   - 例如：`是否提交 git`、`是否推送远端`、`是否创建 PR`。
   - 但如果用户已经明确说“提交git / 提交 git / git提交 / commit一下 / 帮我提交代码”，本 skill 直接让路给 `git-collaboration-rules`，不得先拦截为泛化的关键节点。
4. 方案越界：
   - 需要引入超出已确认设计/需求边界的新逻辑、新模块、新依赖、新数据结构。
5. 高风险或不可逆操作：
   - 删除/覆盖大量文件、破坏性数据库操作、生产环境动作、密钥与权限变更。
6. 成本显著上升：
   - 预计新增大量工时、外部付费调用、跨系统联动改造。
7. 信息冲突且无法靠现有证据收敛：
   - 继续执行将有较高概率做错方向。
8. 用户给出明确结束指令，或不存在三类合法后续（原执行计划内未完成必需项 / 阻断项 / 用户显式要求的建议）：
   - 必须立即停止连续执行，并直接收口结束；不得默认补“下一步状态”“下一步建议”“等待用户新指令”“无需继续动作”或任何可能触发 loop 的占位文案。
9. 当前运行环境存在显式状态收口机制，原始用户目标虽已完成，但尚未真实执行对应收口动作：
   - 必须先结束运行时状态，再允许输出“任务已完成”式总结；未结束前视为未完成收口。

## 关键节点提问格式（强制）

当命中关键节点时，提问必须简洁并给出 2-3 个可执行选项，推荐项放首位：

```text
我已完成 <当前阶段结果>。当前进入关键节点：<节点类型>。
建议选项：
1) <推荐选项>（推荐）
2) <备选选项>
3) <保守选项>
```

禁止开放式追问如“你想让我接下来做什么？”（除非用户明确要求脑暴）。

## 与其他 Skill 的协作

- 与 `git-collaboration-rules`：
  - 当进入“是否提交 git”节点时，转交 Git 规则处理。
  - 若用户指令是“提交git”，默认目标为清空全部未提交改动（允许多次提交）。
  - 若用户已经给出执行型提交短指令，不要先把它转成“是否提交 git”再走关键节点；直接交给 `git-collaboration-rules`。
- 与 `skill-compliance-gate-rules`：
  - 最终收口前仍需完成合规闸门，不因“连续执行”跳过终检。
- 与运行时状态机制：
  - 当 agent 所处环境启用了 goal / plan / task 等状态机制，且原始用户目标已完成时，对应状态收口属于必须执行的真实收口动作，不得用总结文案替代。
  - Codex goal 场景下，`get_goal` / `update_goal` 是具体实现；当 `get_goal` 返回无活动 goal 时，不得伪造 goal 完成动作。
- 与 `context-compression-rules`：
  - 压缩后恢复执行时，优先读取其输出的“编码许可状态”。
  - 若许可状态为 `unknown` / `revoked`，本 skill 必须先触发关键节点确认，不能直接编码。
- 与需求/实现/Bug 类 skill：
  - 本 skill 只管“推进节奏与暂停条件”，不替代领域规则本身。
  - 当需求文档、验收标准与实施文档都已确认，且用户明确“开始实施 / 开始执行”后，应默认先定位当前实施周期，再按周期内最小任务粒度连贯推进“实现 -> 真实测试 -> 审查 -> 验收”，当前周期收口后再进入下一实施周期，不在中段重复确认。
  - 需求完成、验收标准完成或实施计划完成，本身都不等于已获得开工许可；未得到用户明确开工指令前，本 skill 不得推动进入编码。
  - 用户给出开工类指令但没有现成计划时，本 skill 只允许促成“受限计划 + 停止条件”收口，不能把计划后的建议继续升级为实现动作。

## 禁止事项

- 不得把本 skill 解释为“忽略安全规则/审批规则”。
- 不得绕过系统要求的权限确认。
- 不得在高风险节点静默继续执行。
- 不得把“连续执行”变成“跳过验证”。

## 通过 / 驳回标准

- 通过：
  - 在非关键节点保持连续执行，中间只做进度汇报不过度征求确认。
  - 在中间阶段切换点能自动继续推进，不把执行权反复交还用户。
  - 在关键节点能准确暂停并给出结构化选项。
  - 在“设计完成 -> 开始实现”切换点，若许可 `unknown` 能先暂停确认；若许可 `confirmed` 能直接推进而不重复确认。
  - 在开工类指令出现时，能够先检查执行计划、完成定义、停止条件、最大推进边界和验证点；缺失时先输出受限计划并停止，不直接实现。
  - 在用户是“按建议执行 / 按方案执行 / 采纳上一轮建议”这类场景时，能够先把建议转成执行计划，而不是直接把建议内容当实现指令。
  - 在“实施计划已确认但尚未收到开始实施指令”时，能够停在开工确认节点，而不是自动进入编码。
  - 在“压缩后恢复 -> 开始/继续实现”切换点能先确认编码许可，再进入编码。
  - 在多实施周期任务中，能够按“当前周期内最小任务逐个完成实现 -> 真实测试 -> 审查 -> 验收 -> 当前周期收口 -> 下一周期”的顺序自动推进，而不是每完成一个任务或一个周期就停下来等用户再次说“进入测试/进入审查/继续下一个周期”。
  - 能识别实施周期是第一期、第二期、第三期等大进度单位，并在当前周期未收口前拒绝跨周期跳步。
  - 最终产出达到可交付状态，且未越过安全边界。
  - 当用户给出明确结束指令时，任何 agent 都能立即停止自动继续，不追加默认后续建议、等待类话术或“无需继续动作”占位文案。
  - 当不存在三类合法后续（原执行计划内未完成必需项 / 阻断项 / 用户显式要求的建议）时，能够在最终收口处直接结束，而不是附带任何默认“下一步状态”“后续建议”“等待用户新指令”或扩展边界内容。
  - 当当前运行环境存在显式状态收口机制时，能够在“原始用户目标已完成”后真实执行对应收口动作，而不是只输出“任务已完成”的文案。
- 驳回：
  - 每完成一个子步骤就停下来问“是否继续”。
  - 中间阶段已具备继续条件却未自动继续，直到用户追问才推进。
  - 在需求、验收标准或实施计划刚完成时，未拿到用户明确“开始实施/开始执行”就直接进入编码。
  - 在执行许可 `unknown` 时，需求/bug 设计刚完成就直接进入编码，未经过“是否开始实现”的关键节点确认。
  - 用户只说“开始执行 / 直接做 / 继续做完 / 按文档实现 / 按建议执行 / 按方案执行 / 就按你刚才说的做”，但没有执行计划、完成定义、停止条件或最大推进边界，就直接进入实现或连续执行。
  - 用户只是采纳了 agent 的建议或方案，但 agent 没有先把建议收口成执行计划，就直接开始编码或连续执行。

  - 当前运行环境明明存在显式状态收口机制，却只写“等待用户新指令”而没有真实执行对应收口动作。

## 快速示例

模型在完成证据分析后原本准备输出：
“如果你要，我可以下一步给你最小改造方案。”

执行方式：
1. 自动触发本规则，不停在“建议下一步”，而是直接进入下一步实现。
2. 完成后停在关键节点：“是否提交 git”并给出选项。
3. 若实现中发现超出原方案的新模块改造，立即暂停并给 2-3 个方案。

模型在已经完成原始用户需求后原本准备输出：
“下一步我还可以继续优化目录、再收一轮边界说明。”

执行方式：
1. 判定原始用户目标已经完成。
2. 若当前运行环境存在显式状态收口机制，先按该机制确认状态并真实收口；Codex goal 场景才使用 `get_goal` / `update_goal(status="complete")`。
3. 检查是否存在三类合法后续：原执行计划内未完成必需项、阻断项、用户显式要求的建议。
4. 若不存在，直接结束本轮；不输出下一步区块、等待类文案或“无需继续动作”。只有用户明确要求后续建议时，才单独补充 backlog。

用户直接说：
“结束。”

执行方式：
1. 将其识别为通用结束指令，不区分 Codex、Claude Code 或其他 agent。
2. 立即停止自动继续、工具执行和后续建议生成。
3. 只输出必要的最小收口结论；若运行时有必须处理的状态收口机制，先按真实机制收口。

用户直接说：
“开始执行。”

但当前上下文没有执行计划。

执行方式：
1. 识别为开工类指令，但不视为无边界长文本执行授权。
2. 先输出本轮受限计划，至少包含执行范围、完成定义、停止条件、最大推进边界和验证点。
3. 停在计划收口处，不进入编码或工具执行；只有计划和停止边界稳定后，才可进入连续执行。
