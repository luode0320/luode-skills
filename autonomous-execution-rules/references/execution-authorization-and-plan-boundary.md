# 执行授权与计划边界

本文件承载自主执行入口的执行许可、开工计划和上下文压缩后编码许可语义；`SKILL.md` 保持唯一自动触发入口。

### 许可状态

- `confirmed`：用户在当前任务链路明确要求开始或继续实施，且已有有效执行计划。计划至少包含执行范围、完成定义、停止条件、最大推进边界和验证点。
- `unknown`：尚未获得明确执行许可，或许可不能从当前可靠上下文证明；首次编码前必须确认。
- `revoked`：用户明确暂停、结束、缩小范围或改变目标；恢复执行前必须重新确认。

### 状态转换

- `unknown -> confirmed`：用户明确说“开始实施 / 开始实现 / 开始执行 / 直接做 / 继续做完 / 按实施计划执行 / 按文档实现 / 按建议执行 / 按方案执行 / 就按你刚才说的做”，并且正式计划与停止边界齐备。
- `confirmed -> unknown`：发生上下文压缩，且许可状态未被可靠继承或本轮未重新确认是否开始/继续编码。
- `confirmed -> revoked`：用户说“先停”“只做到这一步”“结束”“不要继续”，或明确修改执行边界。
- `confirmed` 仅在同一目标、同一计划边界内持续有效；越界、高风险、信息冲突或授权范围变化会使连续执行暂停。

### 开工类指令边界

- 开工短句不是无边界长文本执行授权。缺少计划、完成定义、停止条件、最大推进边界或验证点时，只能先形成受限计划并停在计划收口处。
- 用户采纳上一轮建议、方案、修复路线或实施思路时，必须先将其转写为当前有效执行计划；聊天建议不能直接替代执行计划。
- 需求文档、验收标准或实施计划完成，本身不等于用户已经授权进入编码。
- 同一任务已 `confirmed` 后，普通阶段切换不再重复确认；只有关键、高风险或越界节点重新暂停。

### 上下文压缩后的编码许可

压缩恢复时先读取规则、当前任务状态和计划证据，再判断许可。只有本轮能够可靠确认 `confirmed` 才能开始或继续编码；`unknown` 或 `revoked` 必须暂停。不得凭“压缩前大概已经同意”推定授权。

### 开工指令示例

用户直接说“开始执行”，但当前上下文没有正式执行计划时：

1. 识别为开工类指令，但不视为无边界长文本执行授权。
2. 先形成包含执行范围、完成定义、停止条件、最大推进边界和验证点的正式计划或受限计划。
3. 计划未收敛前停在计划阶段，不进入编码、改码或其他实施动作。
