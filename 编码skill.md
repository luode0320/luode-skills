# 编码 Skill 体系整体计划（审查稿）

## 文档状态

- 状态：审查稿
- 目标：先固化整体设计，经过多轮审查后再开始初始化 skill、拆分 references 和迁移种子内容
- 当前约束：本阶段只做规划，不开始正式落地

## 一、建设目标

这套 skill 体系的唯一核心目标是：

让 Codex 在参与团队研发时，能够从需求、Bug、编码、编码审查、测试、交付的全流程中，按任务阶段和代码改动位置自动触发对应规则，不依赖人工手动拼接 skill。

希望最终达到的效果：

- 需求不清时，自动进入澄清、补全和边界确认
- Bug 处理时，自动进入问题提出、复现、静态定位、运行时诊断、根因分析、修复建议和验证链路
- 开始写代码时，自动进入可读性、命名、注释和风格一致性规则
- 改到特定代码位置时，自动命中特定 skill
- 写完代码后，自动进入编码审查，检查是否符合编码规范、注释规范、存放位置规范、最小改动原则，以及语法正确性、未使用导入与引用清理、调试残留清理和代码格式化要求
- 写完代码后，自动进入测试目录、命名、测试程序位置、测试文档说明等规范
- 进入上线或交付阶段时，自动进入发布、回滚、评审和交付说明规则

## 二、总体设计原则

### 1. 自动触发优先

所有 skill 都以“自动触发”为目标设计，不以手动显式调用为主。

### 2. 触发描述必须具体

每个 skill 的 `description` 必须写清楚触发信号，不能只写抽象用途。

正确风格示例：

- 当新增或修改 `.env`、`application.yml`、`settings.py`、`config.ts` 时触发
- 当新增或修改 migration、DDL、表字段、索引时触发
- 当新增或修改 controller、router、handler、HTTP 方法、DTO 时触发
- 当新增或修改异常类、全局异常处理、重试、降级、fallback 时触发
- 当新增或修改测试目录、测试脚本、测试 README 时触发

### 3. 以“阶段 + 问题类型 + 代码位点”三维组合设计

这套体系不只按技术分类，而是同时考虑三个维度：

- 阶段维度：当前处于需求、Bug、编码、编码审查、测试、交付哪个阶段
- 问题类型维度：当前是新需求，还是问题修复
- 代码位点维度：当前改动的是配置、数据库、API、响应、错误处理、日志、测试目录等哪一类内容

### 4. 细分，但不混乱

细分是为了方便未来持续补规则、替换规则、迁移规则。

每个 skill 只负责一种清晰职责，不做大杂烩。

### 5. 预留扩展性

未来允许继续：

- 新增 skill
- 合并 skill
- 拆分 skill
- 调整 skill 触发范围
- 把某类规则从一个 skill 迁移到另一个 skill

## 三、总体系结构

计划采用 9 个域：

| 域         | 作用                                                             |
| ---------- | ---------------------------------------------------------------- |
| 总控层     | 统一流程和 skill 分流，负责冲突裁决与流程中断管控，不承载细粒度编码规则 |
| 需求域     | 处理新功能、新页面、新接口、新模块等需求类任务                   |
| Bug 域     | 处理问题描述、复现、静态定位、运行时诊断、根因、修复建议和验证   |
| 编码基线域 | 所有编码任务都自动遵守的基础质量规则                             |
| 代码位点域 | 改到哪类代码就触发哪类 skill                                     |
| 编码审查域 | 代码写完后、测试前的实现自审，检查规范、注释、语法、引用、格式、存放位置和变更清理 |
| 测试域     | 管理测试策略、测试位置、测试命名、测试文档和联调                 |
| 交付域     | 管理评审、提交流程、交付说明、发布和回滚                         |
| 体系维护域 | 管理 skill 自身的拆分、迁移、演进规范                            |

## 四、总控层

总控层新增两项核心能力：

1. 冲突裁决能力

- 当多个 skill 同时触发、规则出现重叠或互相冲突时，由总控层统一裁决
- 裁决优先级固定为：阶段优先、位点次之、类型兜底
- 阶段优先：先保证当前阶段该做的事情不被跳过，例如需求未澄清前不能直接进入编码，编码审查未通过前不能直接进入测试
- 位点次之：在阶段正确的前提下，再按当前实际改动的代码位点选择最相关的 skill
- 类型兜底：如果阶段和位点都不能完全覆盖，再由需求类型、Bug 类型或交付类型规则兜底
- 当多个 skill 都有价值但不冲突时，总控层允许并行生效；当多个 skill 明显重复或互斥时，总控层负责去重和裁断

2. 流程中断管控

- 当需求澄清未通过、Bug 定位未完成、编码审查不达标、测试条件未满足或交付信息不完整时，总控层有权阻止流程进入下一阶段
- 总控层统一管理流程的暂停、重启、继续和终止条件，避免跳过关键环节
- “暂停”用于信息缺失、证据不足、前置步骤未完成
- “重启”用于需求变化、定位结论推翻、修复方案失效、回归验证失败
- “终止”用于发现当前路线错误、成本过高或继续推进会造成明显风险
- 各子 skill 可以提出暂停建议，但真正的阶段切换权限由总控层统一收口

总控层的定位是“流程调度与异常管控中心”，而不是另一个大杂烩规范 skill。

| Skill 名字               | 自动触发 description                                                                                         | 核心职责                                              |
| ------------------------ | ------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------- |
| `team-development-rules` | 当任务属于团队常规研发协作时自动触发，统一中文沟通、阶段判断、skill 分流、冲突裁决、流程中断管控、最小改动原则、测试收尾和交付说明。 | 作为总控入口，负责全流程调度、冲突裁决和中断管控，不承载数据库、API、错误处理等细节规则。 |

| `acceptance-criteria-rules`   | 当任务准备进入实现前确认，或交付前需要验收标准时自动触发。                         | 补齐可验证、可测试的验收标准。       |

### 需求域与 Bug 域边界判定

需求域和 Bug 域虽然都可能表现为“用户说有问题”，但分流标准必须明确：

1. 需求实现过程中发现的原有功能缺陷、历史遗留问题

- 归入 Bug 域处理
- 不纳入当前需求域范围
- 即使是在开发新需求时顺带暴露出来，只要问题本身属于旧逻辑缺陷、旧链路异常、历史行为错误，仍优先按 Bug 流程处理

2. 需求变更引发的新增问题、逻辑调整

- 归入需求域处理
- 不按 Bug 流程处理
- 这类情况本质上是“需求本身发生变化”，而不是“现有实现出现缺陷”
- 应优先进入 `requirement-change-rules`、`requirement-boundary-rules` 等需求域 skill，而不是直接进入 Bug 根因定位

3. 需求验收不通过、不符合预期

- 先判断是需求理解偏差，还是代码缺陷
- 如果是需求描述、验收标准、业务理解、边界判断出现偏差，归入需求域
- 如果是实现错误、运行错误、状态异常、历史逻辑缺陷，归入 Bug 域
- 验收不通过不能直接默认当成 Bug，也不能直接默认当成需求变更，必须先做归因判断

4. 默认判定优先级

- 先判断问题属于“原有实现缺陷”还是“当前需求变化”
- 再判断问题属于“理解偏差”还是“代码缺陷”
- 如果两者同时存在，由总控层决定是否拆成“需求调整任务 + Bug 修复任务”并行管理

## 六、Bug 域

Bug 属于需求的一个子类，但处理流程不同，因此单独建域。

Bug 域采用两条互补路径：

- 静态定位路径：通过阅读代码、追调用链、查日志、查 trace、分析模块边界来定位问题
- 运行时诊断路径：通过断点调试、打印 debug 日志、程序断言、变量快照、关键分支观测来缩小问题范围

有些 bug 只看代码无法稳定定位，必须通过运行中的状态变化、输入输出、分支命中和异常上下文才能找到真实位置，因此运行时诊断是 Bug 域的正式组成部分，不是临时技巧。

| Skill 名字                  | 自动触发 description                                                                                       | 核心职责                           |
| --------------------------- | ---------------------------------------------------------------------------------------------------------- | ---------------------------------- |
| `bug-intake-rules`          | 当用户描述报错、异常行为、结果不符、线上问题、偶发问题、接口异常、页面异常、数据错误、性能异常时自动触发。 | 把问题描述标准化。                 |
| `bug-gap-rules`             | 当 bug 描述缺少复现条件、环境信息、输入数据、报错日志、影响范围时自动触发。                                | 补齐定位所需基础信息。             |
| `bug-reproduction-rules`    | 当问题需要复现、构造步骤、确定触发条件、判断是否稳定发生时自动触发。                                       | 输出复现步骤和复现结论。           |
| `bug-scoping-rules`         | 当需要判断问题属于哪个模块、哪个业务链路、哪个服务、哪个页面、哪个接口、哪个数据流时自动触发。             | 明确问题归属和影响范围。           |
| `bug-root-cause-rules`      | 当开始分析代码、追调用链、看现有日志、查 trace、结合上下游行为定位根因时自动触发。                         | 找出根因并区分实现问题与设计问题。 |
| `bug-runtime-debug-rules`   | 当仅靠静态分析不能稳定定位 bug，且需要在运行过程中通过断点、单步执行、变量观察、调用栈观察、条件命中判断来定位时自动触发。 | 通过运行时调试缩小问题范围并定位异常位置。 |
| `bug-debug-log-rules`       | 当定位 bug 需要临时增加 debug 日志、关键变量输出、关键分支日志、上下游输入输出日志时自动触发。             | 通过可回收的临时日志补充运行期证据。 |
| `bug-assertion-diagnostic-rules` | 当怀疑状态异常、顺序错误、数据污染、不变量被破坏，且需要通过程序断言或诊断性检查快速暴露问题位置时自动触发。 | 用断言和诊断检查缩小 bug 发生区间。 |
| `bug-fix-proposal-rules`    | 当问题已定位，需要形成修改建议、风险评估、备选方案并等待用户确认时自动触发。                               | 先给修改建议，再决定是否实施。     |
| `bug-regression-risk-rules` | 当修复可能影响公共方法、共享模块、已有接口、数据库行为或兼容性时自动触发。                                 | 识别回归风险。                     |
| `bug-validation-rules`      | 当 bug 修复后需要验证是否修好、是否引入副作用、是否需要补回归测试时自动触发。                              | 负责修复后的验证闭环。             |

补充约束：

- `bug-debug-log-rules` 和 `bug-assertion-diagnostic-rules` 产出的临时诊断代码，在进入测试或交付前，必须交由 `cleanup-format-review-rules` 检查是否应删除、降级、保留或改造为正式日志/正式校验
- 断点调试属于定位手段，不是交付结果；最终交付说明中应记录定位结论，而不是保留调试动作本身

## 九、编码审查域

这部分位于“编码完成”和“开始测试”之间，负责实现后的自审，不等同于 PR 评审。

编码审查域重点覆盖：

- 是否符合编码规范、命名规范、注释规范和最小改动原则
- 是否存在语法错误、类型错误、引用缺失、构建失败风险
- 是否存在未使用导入、未使用变量、未使用方法引用、死代码、调试残留和临时代码
- 是否存在多余换行、格式不一致、代码块排版混乱等可自动整理问题
- 是否存在代码落点不合理、目录归属错误、跨层调用越界等结构问题

| Skill 名字                    | 自动触发 description                                                                                                                                             | 核心职责                                                 |
| ----------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------- |
| `implementation-review-rules` | 当功能代码已经完成、准备进入测试前验证时自动触发，检查是否符合编码规范、命名规范、注释规范、风格一致性和最小改动原则。                                           | 对刚完成的实现做一次测试前规范自审。                     |
| `syntax-check-review-rules`   | 当新增或修改代码后准备进入测试前验证，且需要确认语法、类型、依赖引用、构建基础正确性时自动触发。                                                                 | 检查语法错误、引用缺失、类型问题和明显构建失败风险。     |
| `cleanup-format-review-rules` | 当新增或修改代码后准备进入测试前验证，且存在未使用导入、未使用变量、未使用方法引用、调试残留、临时代码、多余换行或格式不一致风险时自动触发。                     | 清理代码噪音并统一基础格式。                             |
| `code-placement-review-rules` | 当新增文件、移动文件、扩展模块、跨层调用、工具类落点或目录归属可能不合理时，在进入测试前自动触发。                                                               | 检查代码存放位置、模块归属、层级边界和依赖方向是否合理。 |

### 编码审查域与测试域边界判定

编码审查域与测试域虽然都发生在“代码写完之后”，但职责必须严格分开：

1. 编码审查域

- 聚焦代码本身质量
- 负责语法、类型、格式、规范、注释、代码归属、依赖引用、未使用导入清理、调试残留清理、冗余代码清理等静态检查
- 属于开发自测环节
- 不负责功能逻辑是否真的满足业务预期

2. 测试域

- 聚焦功能正确性
- 负责测试用例、场景覆盖、联调验证、输入输出验证、功能闭环、回归验证等
- 属于功能验证环节
- 不负责代码格式、导入清理、注释风格、目录归属等静态规范问题

3. 默认分流规则

- 能通过阅读代码、静态分析、格式检查、语法检查、引用检查发现的问题，归入编码审查域
- 必须通过运行程序、执行测试、构造输入输出、联调上下游、观察实际结果才能确认的问题，归入测试域
- “代码看起来规范”不等于“功能正确”
- “功能跑通了”也不等于“代码质量合格”

4. 阶段顺序

- 默认先完成编码审查域，再进入测试域
- 编码审查未通过时，不应直接跳过进入测试
- 测试过程中如果暴露出静态质量问题，可以回流到编码审查域处理后再继续测试

## 十、测试域

测试阶段单独细分，避免测试规则继续被编码规则吞没。

| Skill 名字                    | 自动触发 description                                                             | 核心职责                     |
| ----------------------------- | -------------------------------------------------------------------------------- | ---------------------------- |
| `test-strategy-rules`         | 当功能实现和编码审查完成后，开始补测试、写验证脚本、做回归检查时自动触发。       | 决定测试层级和覆盖重点。     |
| `test-location-rules`         | 当新增测试目录、测试脚本、测试程序、模拟数据、验证代码时自动触发。               | 统一测试资源位置。           |
| `test-naming-rules`           | 当创建新的测试任务目录、测试文件、测试脚本、测试数据目录时自动触发。             | 统一测试目录和文件命名。     |
| `test-program-rules`          | 当新增测试程序、模拟程序、验证脚本、数据构造脚本时自动触发。                     | 统一测试程序和辅助脚本位置。 |
| `test-doc-rules`              | 当新增或修改测试 `README.md`、验证说明、测试报告、覆盖说明时自动触发。           | 统一测试说明文档结构。       |
| `functional-validation-rules` | 当需要验证新功能、修改后的功能、接口行为、页面交互、输入输出结果是否满足当前需求与验收标准时自动触发。 | 负责当前需求对应的功能正确性验证。 |
| `regression-validation-rules` | 当功能修改、Bug 修复、重构、依赖调整、配置调整后，需要确认旧能力未被破坏时自动触发。 | 负责回归范围识别和回归验证。 |
| `integration-debugging-rules` | 当联调、排查接口不通、字段不一致、环境差异、trace 断链、测试环境问题时自动触发。 | 统一联调和排障证据记录。     |

### 测试域内部边界判定

测试域内部继续细分为五类职责：

1. 测试策略

- 由 `test-strategy-rules` 负责
- 重点回答“测什么、测到什么程度、优先覆盖哪些场景”
- 不负责决定测试资源放在哪里，也不负责亲自承担功能验证结论

2. 测试资源管理

- 由 `test-location-rules`、`test-naming-rules`、`test-program-rules`、`test-doc-rules` 共同负责
- 重点解决测试目录、命名、测试程序落点、测试说明文档等组织问题
- 只管理测试资产如何存放、如何命名、如何描述
- 不负责判断功能是否正确

3. 功能验证

- 由 `functional-validation-rules` 负责
- 聚焦当前需求或当前修改本身是否满足业务预期、接口预期、页面交互预期和验收标准
- 只关心“这次要做的东西是否真的做对了”
- 不负责历史能力是否被破坏

4. 联调验证

- 由 `integration-debugging-rules` 负责
- 聚焦上下游系统、接口协议、环境配置、链路串联、字段一致性、trace 连通性等跨边界验证
- 适用于“单点功能看起来没问题，但整个链路跑不通”的场景
- 不替代功能验证，也不替代回归验证

5. 回归验证

- 由 `regression-validation-rules` 负责
- 聚焦当前改动是否破坏旧逻辑、共享模块、公共接口、历史页面、已有流程和兼容行为
- 重点回答“这次改动有没有带坏别的地方”
- 不负责决定测试资产存放规范

### 测试域默认分流规则

- 讨论“测什么、覆盖哪些场景、优先级怎么排”时，进入 `test-strategy-rules`
- 讨论“测试目录建在哪、怎么命名、README 怎么写”时，进入测试资源管理相关 skill
- 讨论“当前需求是否实现正确”时，进入 `functional-validation-rules`
- 讨论“上下游系统是否打通、环境是否一致、链路是否闭环”时，进入 `integration-debugging-rules`
- 讨论“旧功能有没有被这次改动带坏”时，进入 `regression-validation-rules`

### 测试域内部顺序

- 先由 `test-strategy-rules` 决定验证层级和重点
- 再由测试资源管理类 skill 规范测试资产位置、命名和文档
- 然后执行 `functional-validation-rules`
- 涉及跨系统或跨环境时，再进入 `integration-debugging-rules`
- 最后执行 `regression-validation-rules`，确认改动没有破坏原有能力

## 十八、后续审查重点

后续审查建议重点看这几件事：

1. 领域划分是否合理
2. skill 是否拆得过细或还不够细
3. 每个 skill 的自动触发 description 是否足够具体
4. 总控层的冲突裁决规则是否足够明确，是否真的能处理多 skill 同时触发
5. 总控层的流程暂停、重启、终止权限边界是否清晰
6. 有没有遗漏的重要位点类 skill
7. 哪些 skill 应该先做，哪些可以后做
8. 哪些 rules 应该进 `SKILL.md`，哪些应该进 `references/`
9. Bug 域和需求域的交接边界是否清晰，尤其是“历史遗留缺陷”“需求变更引发的问题”“验收不通过”这三类临界场景
10. Bug 域是否已经充分覆盖运行时诊断、断点调试、debug 日志和断言定位
11. 编码审查域和测试域的边界是否清晰，尤其是“静态质量自审”和“功能正确性验证”是否已经彻底分开
12. 测试域内部的“策略、资源管理、功能验证、联调验证、回归验证”边界是否已经彻底分开
13. 编码完成后的自审 skill 是否还需要继续拆细
14. PR 评审和测试前自审是否已经区分清楚
15. 语法检查、引用清理和格式化规则是否还要按语言继续拆细

## 十九、当前结论

当前整体方向已经确定为：

- 不再依赖手动拼 skill
- 采用全流程自动触发设计
- 采用“阶段 + 问题类型 + 代码位点”的混合触发模型
- 总控层负责多 skill 冲突裁决与流程中断管控
- Bug 域采用“静态定位 + 运行时诊断”的双路径设计
- 明确引入“编码完成后、测试前”的编码审查阶段
- 允许长期演进和持续拆分

本文件作为后续多轮审查的基础版本。
