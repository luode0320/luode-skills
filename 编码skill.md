# 编码 Skill 体系整体计划（审查稿）

## 文档状态

- 状态：审查稿
- 目标：先固化整体设计，经过多轮审查后再开始初始化 skill、拆分 references 和迁移种子内容
- 当前约束：本阶段只做规划，不开始正式落地

## 一、建设目标

这套 skill 体系的唯一核心目标是：

让 Codex 在参与团队研发时，能够从需求、Bug、编码、编码审查、测试、交付的全流程中，按任务阶段和代码改动位置自动触发对应规则，不依赖人工手动拼接 skill。

希望最终达到的效果：

- 需求不清时，自动进入澄清、补全和边界确认
- Bug 处理时，自动进入问题提出、复现、定位、根因分析、修复建议和验证链路
- 开始写代码时，自动进入可读性、命名、注释和风格一致性规则
- 改到特定代码位置时，自动命中特定 skill
- 写完代码后，自动进入编码审查，检查是否符合编码规范、注释规范、存放位置规范、最小改动原则，以及语法正确性、未使用导入与引用清理、调试残留清理和代码格式化要求
- 写完代码后，自动进入测试目录、命名、测试程序位置、测试文档说明等规范
- 进入上线或交付阶段时，自动进入发布、回滚、评审和交付说明规则

## 二、总体设计原则

### 1. 自动触发优先

所有 skill 都以“自动触发”为目标设计，不以手动显式调用为主。

### 2. 触发描述必须具体

每个 skill 的 `description` 必须写清楚触发信号，不能只写抽象用途。

正确风格示例：

- 当新增或修改 `.env`、`application.yml`、`settings.py`、`config.ts` 时触发
- 当新增或修改 migration、DDL、表字段、索引时触发
- 当新增或修改 controller、router、handler、HTTP 方法、DTO 时触发
- 当新增或修改异常类、全局异常处理、重试、降级、fallback 时触发
- 当新增或修改测试目录、测试脚本、测试 README 时触发

### 3. 以“阶段 + 问题类型 + 代码位点”三维组合设计

这套体系不只按技术分类，而是同时考虑三个维度：

- 阶段维度：当前处于需求、Bug、编码、编码审查、测试、交付哪个阶段
- 问题类型维度：当前是新需求，还是问题修复
- 代码位点维度：当前改动的是配置、数据库、API、响应、错误处理、日志、测试目录等哪一类内容

### 4. 细分，但不混乱

细分是为了方便未来持续补规则、替换规则、迁移规则。

每个 skill 只负责一种清晰职责，不做大杂烩。

### 5. 预留扩展性

未来允许继续：

- 新增 skill
- 合并 skill
- 拆分 skill
- 调整 skill 触发范围
- 把某类规则从一个 skill 迁移到另一个 skill

## 三、总体系结构

计划采用 9 个域：

| 域         | 作用                                                             |
| ---------- | ---------------------------------------------------------------- |
| 总控层     | 统一流程和 skill 分流，不承载细粒度编码规则                      |
| 需求域     | 处理新功能、新页面、新接口、新模块等需求类任务                   |
| Bug 域     | 处理问题描述、复现、定位、根因、修复建议和验证                   |
| 编码基线域 | 所有编码任务都自动遵守的基础质量规则                             |
| 代码位点域 | 改到哪类代码就触发哪类 skill                                     |
| 编码审查域 | 代码写完后、测试前的实现自审，检查规范、注释、语法、引用、格式、存放位置和变更清理 |
| 测试域     | 管理测试策略、测试位置、测试命名、测试文档和联调                 |
| 交付域     | 管理评审、提交流程、交付说明、发布和回滚                         |
| 体系维护域 | 管理 skill 自身的拆分、迁移、演进规范                            |

## 六、Bug 域

Bug 属于需求的一个子类，但处理流程不同，因此单独建域。

| Skill 名字                  | 自动触发 description                                                                                       | 核心职责                           |
| --------------------------- | ---------------------------------------------------------------------------------------------------------- | ---------------------------------- |
| `bug-intake-rules`          | 当用户描述报错、异常行为、结果不符、线上问题、偶发问题、接口异常、页面异常、数据错误、性能异常时自动触发。 | 把问题描述标准化。                 |
| `bug-gap-rules`             | 当 bug 描述缺少复现条件、环境信息、输入数据、报错日志、影响范围时自动触发。                                | 补齐定位所需基础信息。             |
| `bug-reproduction-rules`    | 当问题需要复现、构造步骤、确定触发条件、判断是否稳定发生时自动触发。                                       | 输出复现步骤和复现结论。           |
| `bug-scoping-rules`         | 当需要判断问题属于哪个模块、哪个业务链路、哪个服务、哪个页面、哪个接口、哪个数据流时自动触发。             | 明确问题归属和影响范围。           |
| `bug-root-cause-rules`      | 当开始分析代码、追调用链、看日志、查 trace、定位根因时自动触发。                                           | 找出根因并区分实现问题与设计问题。 |
| `bug-fix-proposal-rules`    | 当问题已定位，需要形成修改建议、风险评估、备选方案并等待用户确认时自动触发。                               | 先给修改建议，再决定是否实施。     |
| `bug-regression-risk-rules` | 当修复可能影响公共方法、共享模块、已有接口、数据库行为或兼容性时自动触发。                                 | 识别回归风险。                     |
| `bug-validation-rules`      | 当 bug 修复后需要验证是否修好、是否引入副作用、是否需要补回归测试时自动触发。                              | 负责修复后的验证闭环。             |

## 九、编码审查域

这部分位于“编码完成”和“开始测试”之间，负责实现后的自审，不等同于 PR 评审。

编码审查域重点覆盖：

- 是否符合编码规范、命名规范、注释规范和最小改动原则
- 是否存在语法错误、类型错误、引用缺失、构建失败风险
- 是否存在未使用导入、未使用变量、未使用方法引用、死代码、调试残留和临时代码
- 是否存在多余换行、格式不一致、代码块排版混乱等可自动整理问题
- 是否存在代码落点不合理、目录归属错误、跨层调用越界等结构问题

| Skill 名字                    | 自动触发 description                                                                                                                                             | 核心职责                                                 |
| ----------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------- |
| `implementation-review-rules` | 当功能代码已经完成、准备进入测试前验证时自动触发，检查是否符合编码规范、命名规范、注释规范、风格一致性和最小改动原则。                                           | 对刚完成的实现做一次测试前规范自审。                     |
| `syntax-check-review-rules`   | 当新增或修改代码后准备进入测试前验证，且需要确认语法、类型、依赖引用、构建基础正确性时自动触发。                                                                 | 检查语法错误、引用缺失、类型问题和明显构建失败风险。     |
| `cleanup-format-review-rules` | 当新增或修改代码后准备进入测试前验证，且存在未使用导入、未使用变量、未使用方法引用、调试残留、临时代码、多余换行或格式不一致风险时自动触发。                     | 清理代码噪音并统一基础格式。                             |
| `code-placement-review-rules` | 当新增文件、移动文件、扩展模块、跨层调用、工具类落点或目录归属可能不合理时，在进入测试前自动触发。                                                               | 检查代码存放位置、模块归属、层级边界和依赖方向是否合理。 |

## 十三、建议的每个 Skill 标准结构

为了保证未来继续扩展时不乱，所有 skill 都建议按相同结构编写：

1. 适用场景
2. 自动触发信号
3. 进入后必须先做什么
4. 什么时候暂停并向用户确认
5. 本 skill 产出什么结果
6. 下一步应交给哪个 skill
7. 应读取哪些 references

## 十四、推荐的 references 组织方式

- 不写正式 `references/` 细则
- 不做自动触发描述的最终定稿

## 十八、后续审查重点

后续审查建议重点看这几件事：

1. 领域划分是否合理
2. skill 是否拆得过细或还不够细
3. 每个 skill 的自动触发 description 是否足够具体
4. 有没有遗漏的重要位点类 skill
5. 哪些 skill 应该先做，哪些可以后做
6. 哪些 rules 应该进 `SKILL.md`，哪些应该进 `references/`
7. Bug 域和需求域的交接边界是否清晰
8. 编码审查域和测试域的边界是否清晰
9. 编码完成后的自审 skill 是否还需要继续拆细
10. PR 评审和测试前自审是否已经区分清楚
11. 语法检查、引用清理和格式化规则是否还要按语言继续拆细

## 十九、当前结论

当前整体方向已经确定为：

- 不再依赖手动拼 skill
- 采用全流程自动触发设计
- 采用“阶段 + 问题类型 + 代码位点”的混合触发模型
- 明确引入“编码完成后、测试前”的编码审查阶段
- 允许长期演进和持续拆分

本文件作为后续多轮审查的基础版本。
