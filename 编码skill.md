# 编码 Skill 体系整体计划（审查稿）

## 文档状态

- 状态：审查稿
- 目标：先固化整体设计，经过多轮审查后再开始初始化 skill、拆分 references 和迁移种子内容
- 当前约束：本阶段只做规划，不开始正式落地

## 一、建设目标

这套 skill 体系的唯一核心目标是：

让 Codex 在参与团队研发时，能够从需求、Bug、编码、编码审查、测试、交付的全流程中，按任务阶段和代码改动位置自动触发对应规则，不依赖人工手动拼接 skill。

希望最终达到的效果：

- 需求不清时，自动进入澄清、补全和边界确认
- 新会话刚开始且缺少上下文时，自动先压缩最近 3 天的需求、测试、Bug、文档和 Git 活动，再进入真正主域
- 需要回看历史方案、历史修复或项目历程时，自动进入历史检索和时间线分析
- 开发过程中如果发现当前已命中的 skill 不完善，自动进入 skill 演进判断，先决定补哪个 skill、怎么补、是否先暂停主流程，再在回补后继续当前任务
- 需求已稳定但实现路径仍复杂时，自动进入实施计划拆解，先把任务顺序、模块边界和验证闭环写清
- Bug 处理时，自动进入问题提出、复现、静态定位、运行时诊断、根因分析、修复建议和验证链路
- 开始写代码时，自动进入可读性、命名、注释和风格一致性规则
- 改到特定代码位置时，自动命中特定 skill
- 写完代码后，自动进入编码审查，检查是否符合编码规范、注释规范、存放位置规范、最小改动原则，以及语法正确性、未使用导入与引用清理、调试残留清理和代码格式化要求
- 写完代码后，自动进入测试目录、命名、测试程序位置、测试文档说明等规范
- 需要通过真实浏览器打开页面、点击、输入、登录、截图、提取页面数据或验证前端交互时，自动进入浏览器联动测试规则
- 进入上线或交付阶段时，自动进入提交流程、交付说明和团队发布前置收口规则

## 二、总体设计原则

### 1. 自动触发优先

所有 skill 都以“自动触发”为目标设计，不以手动显式调用为主。

### 2. 触发描述必须具体

每个 skill 的 `description` 必须写清楚触发信号，不能只写抽象用途。

正确风格示例：

- 当新增或修改 `.env`、`application.yml`、`settings.py`、`config.ts` 时触发
- 当新增或修改 migration、DDL、表字段、索引时触发
- 当新增或修改 controller、router、handler、HTTP 方法、DTO 时触发
- 当新增或修改异常类、全局异常处理、重试、降级、fallback 时触发
- 当新增或修改测试目录、测试脚本、测试 README 时触发

### 3. 以“阶段 + 问题类型 + 代码位点”三维组合设计

这套体系不只按技术分类，而是同时考虑三个维度：

- 阶段维度：当前处于需求、Bug、编码、编码审查、测试、交付哪个阶段
- 问题类型维度：当前是新需求，还是问题修复
- 代码位点维度：当前改动的是配置、数据库、API、响应、错误处理、日志、测试目录等哪一类内容

### 4. 细分，但不混乱

细分是为了方便未来持续补规则、替换规则、迁移规则。

每个 skill 只负责一种清晰职责，不做大杂烩。

### 5. 预留扩展性

未来允许继续：

- 新增 skill
- 合并 skill
- 拆分 skill
- 调整 skill 触发范围
- 把某类规则从一个 skill 迁移到另一个 skill

## 三、总体系结构

计划采用 9 个域：

| 域         | 作用                                                             |
| ---------- | ---------------------------------------------------------------- |
| 总控层     | 作为弱触发协调层与全局基础约定承载层，负责阶段分析、路由分流、冲突裁决、流程中断管控和跨域目录命名等统一约定，不承载细粒度编码规则，也不替代小 skill 执行 |
| 记忆域     | 处理新会话近期上下文预热、跨会话历史检索、历史方案回忆、项目时间线分析和长期上下文补回 |
| 需求域     | 处理新功能、新页面、新接口、新模块等需求类任务                   |
| Bug 域     | 处理问题描述、复现、静态定位、运行时诊断、根因、修复建议和验证   |
| 编码基线域 | 所有编码任务都自动遵守的基础质量规则                             |
| 代码位点域 | 改到哪类代码就触发哪类 skill                                     |
| 编码审查域 | 代码写完后、测试前的实现自审，检查规范、注释、语法、引用、格式、存放位置和变更清理 |
| 测试域     | 管理测试策略、测试位置、测试命名、测试文档、浏览器联动测试、功能验证与回归 |
| 交付域     | 管理提交流程、协作边界和交付说明                                 |

## 四、总控层

总控层不是“总规则全集”，而是“流程协调 / 阶段分析 / 路由总控”层。

它的作用是：

- 判断当前任务处于哪个阶段
- 判断当前任务更像需求、Bug、编码、测试还是交付问题
- 在多个 skill 同时触发时进行协调和裁决
- 在领域边界不清时承担备用路由功能
- 在流程应暂停、重启、继续或终止时统一收口
- 对跨域共享的目录、命名和归档基础约定提供统一入口

它不负责：

- 代替代码位点类 skill 执行具体规则
- 代替编码基线类 skill 执行具体规范
- 代替测试域、Bug 域或交付域执行细则
- 把所有团队规则重新集中写成一个大 skill

总控层新增两项核心能力：

1. 冲突裁决能力

- 当多个 skill 同时触发、规则出现重叠或互相冲突时，由总控层统一裁决
- 裁决优先级固定为：阶段优先、位点次之、类型兜底
- 阶段优先：先保证当前阶段该做的事情不被跳过，例如需求未澄清前不能直接进入编码，编码审查未通过前不能直接进入测试
- 位点次之：在阶段正确的前提下，再按当前实际改动的代码位点选择最相关的 skill
- 类型兜底：如果阶段和位点都不能完全覆盖，再由需求类型、Bug 类型或交付类型规则兜底
- 当多个 skill 都有价值但不冲突时，总控层允许并行生效；当多个 skill 明显重复或互斥时，总控层负责去重和裁断

2. 流程中断管控

- 当需求澄清未通过、Bug 定位未完成、编码审查不达标、测试条件未满足或交付信息不完整时，总控层有权阻止流程进入下一阶段
- 总控层统一管理流程的暂停、重启、继续和终止条件，避免跳过关键环节
- “暂停”用于信息缺失、证据不足、前置步骤未完成
- “重启”用于需求变化、定位结论推翻、修复方案失效、回归验证失败
- “终止”用于发现当前路线错误、成本过高或继续推进会造成明显风险
- 各子 skill 可以提出暂停建议，但真正的阶段切换权限由总控层统一收口

### 总控层触发边界

总控层必须采用“弱触发”原则，不能写成默认覆盖所有研发任务的大 skill。

应触发的场景：

- 新任务刚进入时，尚未明确属于需求域、Bug 域、编码域、测试域还是交付域
- 任务描述较宽泛，无法直接判断由哪个小 skill 接手
- 多个 skill 同时触发，且可能发生冲突、重复或阶段错位
- 需求域与 Bug 域、编码审查域与测试域等边界不清，需要先分流
- 流程需要暂停、重启、继续或终止
- 用户明确要求按团队完整研发流程处理

不应优先由总控层触发的场景：

- 单纯改 SQL、改 API、改响应、改错误处理、改日志、改包结构、改工具类
- 单纯补注释、做格式清理、做语法检查
- 单纯写测试、补回归测试、做联调验证
- 任何已经有明确小 skill 能直接接手的单域任务

默认执行原则：

- 小独立 skill 是主执行层
- 总控层是协调层和备用路由层
- 当总控层与位点类 / 基线类 / 测试类 skill 同时出现时，总控层只负责判断和分流，不替代执行 skill 的具体规则

总控层的定位是“流程调度与异常管控中心”，而不是另一个大杂烩规范 skill，也不是默认全程强触发的总入口。

| Skill 名字               | 自动触发 description                                                                                         | 核心职责                                              |
| ------------------------ | ------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------- |
| `team-development-rules` | 当任务阶段不明确、领域边界不清、多个 skill 同时触发、流程需要暂停 / 重启 / 继续 / 终止，或用户明确要求按团队完整研发流程处理时自动触发。 | 作为弱触发协调层，负责阶段分析、路由分流、冲突裁决和中断管控，不承载数据库、API、错误处理等细节规则，也不替代小 skill 执行。 |
| `artifact-storage-rules` | 当需要定义、调整或解释 `ment/`、`bug/`、`test/`、`doc/` 以及根目录 `项目设计.md` 等研发产物主入口、命名模板或同任务复用策略时自动触发。 | 作为跨域统一约定 skill，提供目录、命名和复用策略的单一真相源，供需求、Bug、测试、记忆、项目设计和交付类 skill 统一引用。 |
| `project-design-doc-rules` | 当用户要求分析整个项目、梳理架构 / 模块 / 主链路、同步根目录 `项目设计.md`，或在完成全项目分析后补建该文档时自动触发。 | 负责把根目录项目设计类文档当作弱参考源读取，按代码与当前文档优先原则判断偏移，并统一同步或补建根目录 `项目设计.md`。 |
| `skill-evolution-rules` | 当研发任务已经命中某个现有 skill，但执行中发现该 skill 的触发不准、规则缺失、边界不清、references 不足或无法覆盖当前稳定高频场景，继续推进只能依赖临时口头补充时自动触发。 | 负责判断这是业务问题还是 skill 问题，明确应补哪个现有 skill、是否需要新增相邻 skill、给出最小完善建议，并在必要时先暂停当前任务，待 skill 更新并重新加载后再继续。 |

### 总控层默认分流决策表

总控层不直接代替其他 skill 执行细则，但必须具备足够明确的分流能力，否则“弱触发协调层”会沦为空泛概念。

默认分流顺序建议固定为：

1. 先判定当前处于哪个阶段
2. 再判定这是需求类还是 Bug 类
3. 再判定是否已经进入具体代码位点
4. 再判定是否进入编码审查、测试或交付收口

可执行的默认决策如下：

| 当前信号 | 优先进入的域 / Skill | 说明 |
| -------- | -------------------- | ---- |
| 当前争议是研发文档、Bug 记录、测试任务目录、项目通用文档或根目录项目设计主文档应该放到哪里、叫什么、是否复用原记录 | 总控层 / `artifact-storage-rules` | 先确定全局目录和命名约定，再回到对应主域继续执行 |
| 当前主流程已经明确，但执行中发现已命中的某个 skill 触发不准、规则缺失、边界不清、references 不足，继续推进只能靠临时口头规则兜底 | 总控辅助 / `skill-evolution-rules` | 先判断这是不是 skill gap，再决定补旧 skill、补相邻 skill 还是新增独立 skill；阻断级 gap 应先暂停当前任务 |
| 当前会话刚开始，用户直接提当前项目里的需求、Bug、编码或测试问题，但本轮还没有前情上下文 | 记忆域 / `recent-context-bootstrap-rules` | 先压缩最近 3 天的项目活动，再进入真正主域 |
| 用户要求分析整个项目、梳理架构 / 模块 / 主链路，或同步 / 生成 / 补建根目录项目设计主文档 | 总控层 / `project-design-doc-rules` | 先把根目录项目设计类文档作为弱参考源对照真实代码和当前文档，再决定是更新还是补建 |
| 用户明确问“上次怎么做的”“之前有没有修过/讨论过/做过”，或当前任务明显依赖历史会话结论 | 记忆域 / `history-recall-rules` | 先补回历史上下文，再决定进入需求、Bug、编码还是交付 |
| 用户要求输出项目开发历程、关键决策回顾、项目时间线报告 | 记忆域 / `project-timeline-rules` | 先按项目维度组织历史，再决定是否还需要交付域摘要 |
| 用户刚提出一个新功能、新页面、新接口、新模块，但还没有开始写代码 | 需求域 / `requirement-intake-rules` | 先理解目标、背景、输入输出和上下游，不直接进入编码 |
| 用户提出的需求明显缺字段、缺流程、缺边界、缺验收标准 | 需求域 / `requirement-gap-rules`、`requirement-boundary-rules`、`acceptance-criteria-rules` | 先补信息、补边界、补验收，再决定是否进入实现 |
| 用户描述报错、异常、结果不符、线上故障、偶发问题、历史行为错误 | Bug 域 / `bug-intake-rules` | 先标准化问题，再进入复现、定位和根因分析 |
| Bug 仅靠读代码无法定位，需要运行中观察 | Bug 域 / `bug-runtime-debug-rules`、`bug-debug-log-rules`、`bug-assertion-diagnostic-rules` | 进入运行时诊断路径，不应继续停留在纯静态分析 |
| 需求或 Bug 已澄清，开始真正新增 / 修改代码 | 编码基线域 + 代码位点域 | 基线域默认并行生效，再按改动位点附加对应 skill |
| 代码已经完成，准备进入测试，但还没做静态自审 | 编码审查域 | 先做实现自审、语法检查、清理格式和代码归位 |
| 编码审查通过，准备写测试、补测试、做功能验证或联调 | 测试域 | 先定测试策略，再看资源位置、功能验证、环境差异处理和回归 |
| 测试已基本完成，开始准备提交、交付说明或进入团队发布流程 | 交付域 | 进入 Git 协作与交付说明收口流程 |
| 当前任务描述同时像需求又像 Bug，或同时命中多个阶段 | 总控层继续保留控制权，先做分流裁决 | 这属于总控层必须介入的场景，不能直接让任一小 skill 抢占 |

### 总控层的阶段阻断表

总控层除了分流，还必须控制“什么时候绝对不能进入下一阶段”。

默认阻断规则如下：

| 当前阶段 | 典型阻断条件 | 默认动作 |
| -------- | ------------ | -------- |
| 需求阶段 | 目标不清、边界不清、验收标准缺失、关键依赖未确认 | 暂停推进，留在需求域继续澄清 |
| Bug 阶段 | 无法稳定复现、问题归属未判清、根因证据不足、修复建议未经确认 | 暂停进入编码，继续复现 / 定位 / 方案确认 |
| 编码阶段 | 仍存在阶段争议、位点不明、改动范围失控 | 暂停继续扩写代码，先回总控层或需求 / Bug 域纠偏 |
| 编码审查阶段 | 语法未过、引用未清、调试残留未处理、代码归位错误、最小改动原则不满足 | 不允许进入测试域 |
| 测试阶段 | 当前需求未验证、联调未打通、回归范围未覆盖、关键场景结论缺失 | 不允许进入交付域 |
| 交付阶段 | 提交未整理、交付说明缺失、团队发布前置信息未齐 | 不允许进入正式合并 / 发布 |

补充原则：

- 阻断不是终止，阻断意味着必须先补齐前置条件
- 如果前一阶段结论被推翻，总控层有权要求流程重启到更早阶段
- 如果当前方向被证明确实错误，总控层有权终止当前路线并要求重新立项或重新拆分任务

## 五、记忆域

记忆域不是编码规范域，也不是交付收口域，而是横切辅助域。

它的作用是：

- 当用户需要回看历史方案、历史修复、历史决策和历史实现时，补回跨会话上下文
- 当新会话刚开始且缺少上下文时，先压缩 `artifact-storage-rules` 约定目录中最近 3 天的项目活动，为后续主域预热
- 当任务是整项目分析或架构梳理时，允许弱读取根目录项目设计类文档作为总览参考，但不能当成当前实现真相
- 当用户需要项目发展过程、关键决策链路、阶段演进分析时，输出时间线型报告
- 当真实持久记忆后端不可用时，降级为基于 `artifact-storage-rules` 约定目录、Git 记录和本地文档的历史回溯

它不负责：

- 代替需求域做需求补齐和边界判断
- 代替 Bug 域做复现、定位和根因分析
- 代替交付域输出当前一次交付的提交摘要
- 把“本地材料推断”伪装成“真实永久记忆检索结果”

| Skill 名字 | 自动触发 description | 核心职责 |
| ---------- | -------------------- | -------- |
| `recent-context-bootstrap-rules` | 当当前会话刚开始、缺少历史上下文、用户直接提出与当前项目相关的需求、Bug、编码、测试或交付问题时自动触发。 | 负责优先从 `artifact-storage-rules` 约定目录和最近 Git 提交中压缩前置上下文；如果当前任务涉及整项目分析，可额外弱读取根目录项目设计类文档，再把任务转交真正主域。 |
| `history-recall-rules` | 当用户明确询问“上次怎么做的”“之前有没有修过 / 做过 / 讨论过”“以前是否处理过类似需求 / 类似 Bug / 类似接口 / 类似表结构”，或当前任务明显依赖过去会话结论时自动触发。 | 负责检索跨会话历史、历史方案和历史修复记录，补回长期上下文。 |
| `project-timeline-rules` | 当用户要求生成项目开发历程、阶段总结、技术演进、关键决策回顾、项目时间线报告或完整历史分析时自动触发。 | 负责按项目维度组织长期历史、输出时间线和演进报告。 |

### 记忆域与需求 / Bug / 交付域的边界判定

1. 新会话近期预热不是历史深回忆

- 如果当前会话刚开始、用户问题与当前项目强相关、但本轮还没有前情上下文，应先进入 `recent-context-bootstrap-rules`
- `recent-context-bootstrap-rules` 只负责压缩最近 3 天的近期活动，不负责回答更早历史上“以前怎么做的”
- 只有在用户明确要查历史方案、历史修复或长期决策时，才进入 `history-recall-rules` 或 `project-timeline-rules`

2. 历史回忆不是需求澄清

- 如果当前主要问题是“需求本身不完整、边界不清、字段不清”，仍归需求域处理
- 只有在“需要先查以前是否做过、以前怎么做的”时，才先进入记忆域补上下文
- 历史回忆完成后，应回到 `requirement-intake-rules`、`requirement-gap-rules` 或 `requirement-boundary-rules` 继续推进

3. 历史回忆不是 Bug 定位

- 如果当前主要问题是“程序现在为什么错”，仍归 Bug 域处理
- 只有在“需要先确认历史上是否修过类似问题、当时怎么定位和怎么修”时，才先进入记忆域
- 记忆域只负责补历史证据，不代替当前根因定位

4. 项目时间线报告不是交付摘要

- `project-timeline-rules` 面向长期历史、项目阶段、技术演进和关键决策
- `delivery-summary-rules` 面向当前这次交付的改动、验证、风险和遗留项
- 两者都可能生成总结，但时间尺度和目标不同，不应混用

5. 默认判定优先级

- 当前会话刚开始且缺少上下文时，先进入 `recent-context-bootstrap-rules`
- 用户明确要查“以前怎么做过”时，进入 `history-recall-rules`
- 用户明确要看长期阶段演进和项目历程时，进入 `project-timeline-rules`
- 如果历史信息只是辅助当前任务，补回结论后必须返回对应主域继续处理

### 需求域与 Bug 域边界判定

需求域和 Bug 域虽然都可能表现为“用户说有问题”，但分流标准必须明确：

1. 需求实现过程中发现的原有功能缺陷、历史遗留问题

- 归入 Bug 域处理
- 不纳入当前需求域范围
- 即使是在开发新需求时顺带暴露出来，只要问题本身属于旧逻辑缺陷、旧链路异常、历史行为错误，仍优先按 Bug 流程处理

2. 需求变更引发的新增问题、逻辑调整

- 归入需求域处理
- 不按 Bug 流程处理
- 这类情况本质上是“需求本身发生变化”，而不是“现有实现出现缺陷”
- 应优先进入 `requirement-change-rules`、`requirement-boundary-rules` 等需求域 skill，而不是直接进入 Bug 根因定位

3. 需求验收不通过、不符合预期

- 先判断是需求理解偏差，还是代码缺陷
- 如果是需求描述、验收标准、业务理解、边界判断出现偏差，归入需求域
- 如果是实现错误、运行错误、状态异常、历史逻辑缺陷，归入 Bug 域
- 验收不通过不能直接默认当成 Bug，也不能直接默认当成需求变更，必须先做归因判断

4. 默认判定优先级

- 先判断问题属于“原有实现缺陷”还是“当前需求变化”
- 再判断问题属于“理解偏差”还是“代码缺陷”
- 如果两者同时存在，由总控层决定是否拆成“需求调整任务 + Bug 修复任务”并行管理

## 七、Bug 域

Bug 属于需求的一个子类，但处理流程不同，因此单独建域。

Bug 域采用两条互补路径：

- 静态定位路径：通过阅读代码、追调用链、查日志、查 trace、分析模块边界来定位问题
- 运行时诊断路径：通过断点调试、打印 debug 日志、程序断言、变量快照、关键分支观测来缩小问题范围

有些 bug 只看代码无法稳定定位，必须通过运行中的状态变化、输入输出、分支命中和异常上下文才能找到真实位置，因此运行时诊断是 Bug 域的正式组成部分，不是临时技巧。

| --------------------------- | ---------------------------------------------------------------------------------------------------------- | ---------------------------------- |
| `bug-intake-rules`          | 当用户描述报错、异常行为、结果不符、线上问题、偶发问题、接口异常、页面异常、数据错误、性能异常时自动触发。 | 把问题描述标准化。                 |
| `bug-gap-rules`             | 当 bug 描述缺少复现条件、环境信息、输入数据、报错日志、影响范围时自动触发。                                | 补齐定位所需基础信息。             |

### 代码位点域的多 Skill 并行触发规则

代码位点域不能按“一个改动只能命中一个 skill”来设计，因为真实编码任务通常会同时改到多个位点。

默认规则如下：

1. 位点类 skill 允许并行命中

- 同一次改动可以同时触发多个位点类 skill
- 例如新增一个接口时，往往会同时命中 `api-endpoint-rules`、`api-request-rules`、`api-response-rules`
- 如果该接口还涉及日志记录或异常映射，则还会继续命中 `logging-trace-rules`、`error-handling-rules`

2. 基线域默认并行生效

- 只要进入编码，`code-minimal-change-rules`、`code-readability-rules`、`code-style-consistency-rules`、`naming-rules`、注释类 skill 默认一起生效
- 位点类 skill 只负责“改到哪里就约束哪里”
- 基线域负责“无论改哪里都应保持一致的基础质量”
- 两者关系是并行叠加，不是二选一

3. 结构类位点优先于实现类位点

- 如果改动涉及目录、包名、模块归属、跨层依赖调整，应先由 `package-structure-rules` 判断代码该落在哪一层、是否允许跨层调用
- 在结构位置确定后，再继续应用数据库、接口、日志、错误处理等实现类 skill
- 这样可以避免“代码内容看起来规范，但代码落点本身错误”的问题

4. 数据结构位点优先于数据访问位点

- 如果同时修改表结构和 SQL 查询，应先由 `database-schema-rules` 约束字段、索引、DDL、迁移和回滚
- 结构稳定后，再由 `database-query-rules` 约束 CRUD、事务、锁、分页和性能
- 这样可以避免查询规则先行、但底层 schema 设计本身不合理

5. 接口入口位点与横切位点并行叠加

- `api-endpoint-rules`、`api-request-rules`、`api-response-rules` 负责接口主干
- `error-handling-rules`、`logging-trace-rules` 负责当前保留的横切约束
- 横切类 skill 不替代主干 skill，只在主干上叠加要求

6. 同文件多位点命中时，按“主位点 + 横切位点”理解

- 主位点是本次改动最核心的代码归属，例如配置、数据库、接口、前端组件、工具类
- 横切位点是伴随出现的公共要求，例如日志、错误处理、性能、安全、请求头
- 主位点决定主要参考哪一组 references
- 横切位点补充额外的限制和检查项

7. 前端冲突时 `frontend-design` 优先

- 当 `frontend-design` 与 `frontend-ui-visual-rules`、`frontend-component-rules` 同时命中时，由 `frontend-design` 先主导整体前端落地。
- `frontend-ui-visual-rules` 退为团队内部视觉约束补充，`frontend-component-rules` 退为组件工程边界补充。
- 也就是说，`frontend-design` 负责“整体成品怎么设计并落地”，内部前端 skill 负责“按团队习惯补约束”，不能反过来抢主导权。

8. 发生重复约束时由总控层裁决

- 如果两个位点类 skill 对同一问题给出重复要求，总控层负责去重
- 如果两个位点类 skill 的要求明显冲突，按“结构优先、主位点优先、横切位点补充”的原则裁决
- 无法直接裁决时，再回到“阶段优先、位点次之、类型兜底”的全局规则

  重点约束 `controller`、`service`、`repository`、`entity`、`dto`、`config`、`exception` 等分层
- `node-python-module-layout.md`
  重点约束模块拆分、工具层、配置层、接口层和测试层的目录归属

当前审查阶段先固化“需要按语言映射”的原则，不在本文件中展开到具体包级细则，避免现在就把实现细节写死。

## 十、编码审查域

这部分位于“编码完成”和“开始测试”之间，负责实现后的自审，不等同于 PR 评审。

编码审查域重点覆盖：

- 是否符合编码规范、命名规范、注释规范和最小改动原则
- 是否存在语法错误、类型错误、引用缺失、构建失败风险
- 是否存在未使用导入、未使用变量、未使用方法引用、死代码、调试残留和临时代码
- 是否存在多余换行、格式不一致、代码块排版混乱等可自动整理问题
- 是否存在代码落点不合理、目录归属错误、跨层调用越界等结构问题

| Skill 名字                    | 自动触发 description                                                                                                                                             | 核心职责                                                 |
| ----------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------- |
| `implementation-review-rules` | 当功能代码已经完成、准备进入测试前验证时自动触发，检查是否符合编码规范、命名规范、注释规范、风格一致性和最小改动原则。                                           | 对刚完成的实现做一次测试前规范自审。                     |
| `syntax-check-review-rules`   | 当新增或修改代码后准备进入测试前验证，且需要确认语法、类型、依赖引用、构建基础正确性时自动触发。                                                                 | 检查语法错误、引用缺失、类型问题和明显构建失败风险。     |
| `cleanup-format-review-rules` | 当新增或修改代码后准备进入测试前验证，且存在未使用导入、未使用变量、未使用方法引用、调试残留、临时代码、多余换行或格式不一致风险时自动触发。                     | 清理代码噪音并统一基础格式。                             |
| `code-placement-review-rules` | 当新增文件、移动文件、扩展模块、跨层调用、工具类落点或目录归属可能不合理时，在进入测试前自动触发。                                                               | 检查代码存放位置、模块归属、层级边界和依赖方向是否合理。 |

### 编码审查域与测试域边界判定

编码审查域与测试域虽然都发生在“代码写完之后”，但职责必须严格分开：

1. 编码审查域

- 聚焦代码本身质量
- 负责语法、类型、格式、规范、注释、代码归属、依赖引用、未使用导入清理、调试残留清理、冗余代码清理等静态检查
- 属于开发自测环节
- 不负责功能逻辑是否真的满足业务预期

2. 测试域

- 聚焦功能正确性
- 负责测试用例、场景覆盖、联调验证、输入输出验证、功能闭环、回归验证等
- 属于功能验证环节
- 不负责代码格式、导入清理、注释风格、目录归属等静态规范问题

3. 默认分流规则

- 能通过阅读代码、静态分析、格式检查、语法检查、引用检查发现的问题，归入编码审查域
- 必须通过运行程序、执行测试、构造输入输出、联调上下游、观察实际结果才能确认的问题，归入测试域
- “代码看起来规范”不等于“功能正确”
- “功能跑通了”也不等于“代码质量合格”

4. 阶段顺序

- 默认先完成编码审查域，再进入测试域
- 编码审查未通过时，不应直接跳过进入测试
- 测试过程中如果暴露出静态质量问题，可以回流到编码审查域处理后再继续测试

## 十一、测试域

测试阶段单独细分，避免测试规则继续被编码规则吞没。

| Skill 名字                    | 自动触发 description                                                             | 核心职责                     |
| ----------------------------- | -------------------------------------------------------------------------------- | ---------------------------- |
| `test-strategy-rules`         | 当功能实现和编码审查完成后，开始补测试、写验证脚本、做回归检查时自动触发。       | 决定测试层级和覆盖重点。     |
| `test-location-rules`         | 当新增测试目录、测试脚本、测试程序、模拟数据、验证代码时自动触发。               | 统一测试资源位置。           |
| `test-naming-rules`           | 当创建新的测试任务目录、测试文件、测试脚本、测试数据目录时自动触发。             | 统一测试目录和文件命名。     |
| `test-program-rules`          | 当新增测试程序、模拟程序、验证脚本、数据构造脚本时自动触发。                     | 统一测试程序和辅助脚本位置。 |
| `test-doc-rules`              | 当新增或修改测试 `README.md`、验证说明、测试报告、覆盖说明时自动触发。           | 统一测试说明文档结构。       |
| `agent-browser`               | 当需要打开本地或远程 Web 页面、点击按钮、填写表单、登录站点、截图、提取页面数据、查看网络请求或通过真实浏览器验证前端交互时自动触发。 | 统一浏览器自动化测试与页面交互执行。 |
| `functional-validation-rules` | 当需要验证新功能、修改后的功能、接口行为、页面交互、输入输出结果是否满足当前需求与验收标准时自动触发。 | 负责当前需求对应的功能正确性验证。 |
| `test-regression-rules`       | 当 Bug 修复、原有功能迭代、公共模块修改后，准备执行测试时自动触发。 | 明确回归测试的范围、用例选取、验证要点，针对改动点关联的功能、上下游链路做全覆盖验证，防止修复旧 Bug 引入新问题，保障功能兼容性。 |

### 测试域内部边界判定

测试域内部继续细分为六类职责：

1. 测试策略

- 由 `test-strategy-rules` 负责
- 重点回答“测什么、测到什么程度、优先覆盖哪些场景”
- 不负责决定测试资源放在哪里，也不负责亲自承担功能验证结论

2. 测试资源管理

- 由 `test-location-rules`、`test-naming-rules`、`test-program-rules`、`test-doc-rules` 共同负责
- 重点解决测试目录、命名、测试程序落点、测试说明文档等组织问题
- 只管理测试资产如何存放、如何命名、如何描述
- 不负责判断功能是否正确

3. 浏览器联动执行

- 由 `agent-browser` 负责
- 聚焦“如何通过真实浏览器执行页面打开、点击、输入、登录、截图、抓取页面证据和查看网络请求”
- 适用于本地前端页面、测试环境页面、线上页面和需要真实浏览器行为验证的 Web 链路
- 不替代功能验证、回归验证或联调归因，只负责浏览器执行手段与操作能力

4. 功能验证

- 由 `functional-validation-rules` 负责
- 聚焦当前需求或当前修改本身是否满足业务预期、接口预期、页面交互预期和验收标准
- 只关心“这次要做的东西是否真的做对了”
- 不负责历史能力是否被破坏

5. 联调与环境差异处理

- 当前不再单独保留联调专用 skill
- 遇到上下游系统、接口协议、环境配置、链路串联、字段一致性、trace 连通性等跨边界验证问题时，先回到 `test-strategy-rules` 重新分流
- 需要浏览器侧验证时进入 `agent-browser`
- 需要确认当前功能是否正确时进入 `functional-validation-rules`
- 需要确认旧能力是否被带坏时进入 `test-regression-rules`

6. 回归测试与回归验证

- 由 `test-regression-rules` 负责
- 聚焦当前改动是否破坏旧逻辑、共享模块、公共接口、历史页面、已有流程和兼容行为
- 重点回答“这次改动有没有带坏别的地方”
- 明确回归范围、用例选取和验证要点
- 不负责决定测试资产存放规范

### 测试域默认分流规则

- 讨论“测什么、覆盖哪些场景、优先级怎么排”时，进入 `test-strategy-rules`
- 讨论“测试目录建在哪、怎么命名、README 怎么写”时，进入测试资源管理相关 skill
- 讨论“需要用真实浏览器打开页面、点击、输入、登录、截图、抓取页面数据或查看页面网络请求”时，进入 `agent-browser`
- 讨论“当前需求是否实现正确”时，进入 `functional-validation-rules`
- 讨论“上下游系统是否打通、环境是否一致、链路是否闭环”时，先进入 `test-strategy-rules` 重新拆分验证路径，必要时升级到总控层协调
- 讨论“旧功能有没有被这次改动带坏、回归范围怎么定、回归用例怎么选”时，进入 `test-regression-rules`

### 测试域内部顺序

- 先由 `test-strategy-rules` 决定验证层级和重点
- 再由测试资源管理类 skill 规范测试资产位置、命名和文档
- 涉及页面交互、前端链路或真实浏览器操作时，进入 `agent-browser`
- 然后执行 `functional-validation-rules`
- 涉及跨系统或跨环境时，回到 `test-strategy-rules` 重新确认验证路径与证据收集方式
- 最后执行 `test-regression-rules`，确认改动没有破坏原有能力，并完成回归范围验证

## 十二、交付域

| Skill 名字                | 自动触发 description                                                                      | 核心职责                                         |
| ------------------------- | ----------------------------------------------------------------------------------------- | ------------------------------------------------ |
| `git-collaboration-rules` | 当创建提交、整理提交说明、准备 PR、合并前检查时自动触发。                                 | 统一 Git 协作规则。                              |
| `delivery-summary-rules`  | 当准备交付说明、变更说明、影响范围说明、风险说明时自动触发。                              | 统一交付文档结构。                               |

### 交付域与测试域边界判定

测试域结束后，不代表任务天然具备交付条件，交付域负责的是“可协作、可说明、可追溯”的收口动作。

1. 测试域

- 负责证明功能是否正确、链路是否打通、回归是否充分
- 输出的是测试结论、覆盖情况、联调结论和回归结果
- 不负责 Git 提交质量、PR 评审流程、发布检查表、上线窗口、回滚预案和交付说明组织

2. 交付域

- 负责把已经完成实现、编码审查和测试验证的结果，整理成可协作、可说明、可追溯的交付物
- 关注点是协作流程、交付收口、风险说明和交付记录
- 不替代测试域做功能正确性判断

3. 默认分流规则

- 讨论“功能是否正确、是否覆盖到位、上下游是否打通”时，归入测试域
- 讨论“提交怎么拆、PR 怎么写、要给什么交付说明、进入团队发布流程前还缺什么信息”时，归入交付域
- “测试通过”只能说明功能层面具备继续推进条件，不等于交付环节已经完成

### 交付域内部边界判定

交付域当前只保留两类职责，避免把协作和说明混成一个 skill，同时不为低频环节额外拆 skill。

1. Git 协作

- 由 `git-collaboration-rules` 负责
- 聚焦提交粒度、提交信息、分支协作、PR 准备、合并前基础检查
- 重点解决“怎么把改动以可协作的形式提交出去”
- 不负责对代码内容做深入质量判断

2. 交付说明

- 由 `delivery-summary-rules` 负责
- 聚焦变更摘要、影响范围、风险提示、使用说明、上线说明、回滚说明和后续观察项
- 重点解决“要把什么信息交给评审者、测试者、运维者、业务方或下一个接手人”
- 不负责代替团队发布流程做上线决策

补充说明：

- 正式代码评审仍然存在，但目前不单独拆成 skill；默认由编码审查域先完成作者自审，再结合团队现有评审流程处理
- 发布、灰度、回滚也仍然存在，但当前不再单独建 skill；如需上线动作，统一通过团队发布流程执行，并由 `delivery-summary-rules` 补充说明

### 交付域默认顺序

- 先由 `git-collaboration-rules` 整理提交粒度和提交说明
- 最后由 `delivery-summary-rules` 固化交付记录、变更说明和观察事项

如果项目流程要求先写交付说明再整理提交，可允许 `delivery-summary-rules` 前置，但不能跳过编码审查域和测试域的既有职责。

## 十四、建议的每个 Skill 标准结构

为了保证未来继续扩展时不乱，所有 skill 都建议按相同结构编写：

1. 适用场景
2. 自动触发信号
3. 进入后必须先做什么
4. 什么时候暂停并向用户确认
5. 本 skill 产出什么结果
6. 下一步应交给哪个 skill
7. 应读取哪些 references
8. 本 skill 执行通过 / 驳回的判定标准
9. 执行结果归档要求

补充说明：

- “本 skill 执行通过 / 驳回的判定标准”必须明确合格与不合格的边界，避免只写原则、不写判定条件
- 判定标准应尽量可检查、可复核，避免出现“基本合理”“大致完成”这类模糊措辞
- “执行结果归档要求”必须明确产出内容保存到哪里、采用什么记录格式、是否需要附带说明或结论
- 归档要求的目标是保证后续复盘、回看、审计和追溯时，能找到本次执行留下的证据和结论

### 统一产物目录约定（已落地）

为了避免每个 skill 各自定义一套目录、命名和复用策略，现已将这部分规则独立抽出为 `artifact-storage-rules`，作为跨域单一真相源。

当前统一原则如下：

- 需求域相关文档统一遵循 `artifact-storage-rules` 定义的需求文档根目录、命名模板和同文档更新策略
- Bug 域相关记录统一遵循 `artifact-storage-rules` 定义的 Bug 根目录、主入口 `README.md` 和同 Bug 复用策略
- 测试域相关资产统一遵循 `artifact-storage-rules` 定义的测试根目录、时间戳根目录和主说明入口
- 项目级总体设计、架构总览和模块大纲统一遵循 `artifact-storage-rules` 定义的根目录 `项目设计.md` 主入口
- 项目通用说明和交付总结统一遵循 `artifact-storage-rules` 定义的项目文档根目录

默认记录格式建议如下：

- 过程说明、结论摘要、判定结果优先使用 `Markdown`，文件后缀为 `.md`
- 结构化配置、检查结果、清单类数据优先使用 `.json`、`.yaml` 或 `.yml`
- 原始日志、命令输出、运行快照优先使用 `.log` 或 `.txt`
- 截图、录屏、调用链导出、trace 附件统一放入对应任务目录下的 `attachments/` 子目录

默认归档命名建议如下：

- 任务级目录优先使用 `yyyy-MM-dd-HHmmss-中文任务简述`
- 文件名应体现“阶段 + 主题 + 结果”，避免使用 `temp.md`、`result.txt` 这类无意义命名
- 同一任务下的归档内容应能从目录名和文件名直接看出来源与用途

默认归档最小要求如下：

- 只要一个 skill 产生了可复用结论、明确判定、关键证据或后续要继续接力的信息，就必须归档
- 归档内容至少应包含：任务背景、执行结论、通过 / 驳回判定、关键证据位置、下一步动作
- 纯临时调试动作本身不要求长期保留，但由此得出的定位结论、修复依据和关键证据应保留

特殊说明：

- 其他 skill 如需知道具体目录名、文件名模板或扫描范围，应优先读取 `artifact-storage-rules/references/path-map.yaml`
- 其他 skill 如需知道“更新原记录还是新建新记录”，应优先读取 `artifact-storage-rules/references/update-policy.md`
- 总控层只负责把这类争议分流到 `artifact-storage-rules`，不再自己承载目录模板细节

## 十五、推荐的 references 组织方式

建议每个 skill 都采用“精简 `SKILL.md` + 结构化 `references/`”的方式组织，不建议把全部规则堆在 `SKILL.md` 里。

### 1. `SKILL.md` 保留内容

`SKILL.md` 只保留最核心的、必须第一时间被加载的内容，原则上只写：

- 核心触发逻辑
- 执行流程
- 权责边界
- 暂停 / 确认条件
- 通过 / 驳回判定标准
- references 读取入口与选择规则

约束要求：

- `SKILL.md` 必须保持精简，严禁堆砌大量细则
- 不把语言细节、框架细节、场景枚举、长篇示例直接塞进 `SKILL.md`
- 一切可拆分、可复用、可按需加载的细节，都应进入 `references/`

### 2. `references/*.md` 拆分规则

`references/` 用于承载细则，必须按可维护、可复用、可独立更新的原则拆分。

拆分维度优先级建议如下：

1. 按语言拆分

- 例如：`java-*.md`、`python-*.md`、`typescript-*.md`

2. 按框架或技术栈拆分

- 例如：`mybatis-*.md`、`jpa-*.md`、`spring-*.md`
- 例如：`vue-*.md`、`react-*.md`

3. 按场景拆分

- 例如：`bug-runtime-debug.md`
- 例如：`api-response-envelope.md`
- 例如：`test-regression-scope.md`

4. 按代码位点拆分

- 例如：`database-sql-rules.md`
- 例如：`request-header-trace.md`
- 例如：`common-util-placement.md`

典型示例：

- 数据库规则可拆成：
  - `sql-rules.md`
  - `mybatis-rules.md`
  - `jpa-rules.md`
- 前端规则可拆成：
  - `vue-rules.md`
  - `react-rules.md`
- 包结构规则可拆成：
  - `go-package-layout-rules.md`
  - `go-dependency-boundary-rules.md`
  - `java-package-layout-rules.md`
  - `python-module-layout-rules.md`
- 安全规则可拆成：
  - `fastapi-security.md`
  - `django-security.md`
  - `react-security.md`

拆分原则：

- 一份 reference 只负责一类相对稳定的主题
- 同一份 reference 不要同时混合语言、框架、位点、流程四种维度
- 只要某一部分会独立演进、独立增补、独立复用，就应该拆出去

### 3. references 命名规范

全体系统一使用小写字母 + 连字符命名，不使用空格，不使用中文文件名。

推荐命名格式：

- `主题-规则.md`
- `语言-主题-规则.md`
- `框架-主题-规则.md`
- `位点-主题-规则.md`

命名示例：

- `sql-rules.md`
- `mybatis-rules.md`
- `jpa-rules.md`
- `react-rules.md`
- `vue-rules.md`
- `error-response-rules.md`
- `trace-header-rules.md`
- `runtime-debug-rules.md`

禁止情况：

- `规则1.md`
- `临时.md`
- `最终版.md`
- `各种规范汇总.md`
- 含义不清、不能从文件名判断用途的命名

### 4. references 目录层级规范

默认建议每个 skill 只保留一层 `references/`，不要做过深目录嵌套。

推荐结构：

```text
skill-name/
├── SKILL.md
├── agents/
│   └── openai.yaml
└── references/
    ├── general-rules.md
    ├── java-xxx-rules.md
    ├── python-xxx-rules.md
    ├── react-xxx-rules.md
    └── ...
```

目录层级控制原则：

- 优先平铺，不优先深层级
- 如果 references 文件数量较少，全部放在 `references/` 根下
- 只有当某个 skill 的 references 数量明显变多且维度非常稳定时，才考虑再分一级目录
- 即使分目录，也只建议一层，例如：
  - `references/languages/`
  - `references/frameworks/`
  - `references/scenarios/`

### 5. 全体系统一性要求

所有 skill 的 references 组织方式要尽量统一，避免不同 skill 各写一套风格。

统一要求包括：

- 命名风格统一
- 目录层级统一
- 拆分维度统一
- 文件职责边界统一
- 不在一个 skill 平铺、另一个 skill 深层嵌套、第三个 skill 全堆进 `SKILL.md`

目标是：

- 后续任何人补规则时，都知道该把内容放在哪
- 后续任何人审查 skill 时，都能快速找到对应 reference
- 后续迁移、合并、拆分 skill 时，references 结构可以直接复用

- 如果未来确实需要恢复，按“重新立项、重新命名、重新审查边界”的方式处理，而不是直接沿用旧规划

## 十八、后续维护原则

当前维护重点从“继续扩张 skill 数量”，调整为“把现有 skill 用稳、用准、用清楚”。

后续只按下面原则推进：

1. 优先维护已落地 skill

- 优先修正触发 description 是否够具体
- 优先修正与相邻 skill 的边界是否清晰
- 优先补充必要的 `references/`，而不是继续新建 skill

2. 对空档场景先做吸收，不急着拆新 skill

- 如果发现某个空档场景只是现有 skill 还没写细，应优先补到相邻现有 skill
- 只有在高频、边界稳定、现有 skill 明显无法承接时，才重新讨论新增 skill

3. 不再保留“为了完整而完整”的规划项

- 低频、边界不稳、容易与现有 skill 重叠的规则，不再提前挂在主规划里占位
- 没有进入真实高频使用前，不再预留独立 skill 名称

4. 仍然保持串行、保守的维护节奏

- 每次只处理一个明确问题
- 一个 skill 完整审查通过后，再进入下一个
- 不因为发现可扩展点，就顺手恢复已经放弃的规划 skill

5. 如果未来再次新增 skill，仍然遵守最小标准

- 先明确目标、边界和触发 description
- 再确定是否真的需要独立 skill
- 最后才创建目录、补 `SKILL.md` 和 `references/`

19. 串行实施计划是否合理，是否需要调整各波次顺序

## 二十一、当前结论

当前整体方向已经确定为：

- 不再依赖手动拼 skill
- 采用全流程自动触发设计
- 采用“阶段 + 问题类型 + 代码位点”的混合触发模型
- 小独立 skill 是主执行层
- 总控层是弱触发的流程协调、阶段分析与备用路由层
- 总控层负责多 skill 冲突裁决与流程中断管控，但不替代小 skill 执行
- Bug 域采用“静态定位 + 运行时诊断”的双路径设计
- 明确引入“编码完成后、测试前”的编码审查阶段
- 正式落地采用“串行、安全、单 skill 完整验收后再进入下一个”的实施策略
- 允许长期演进和持续拆分

本文件作为后续多轮审查的基础版本。
