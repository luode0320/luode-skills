# 编码 Skill 体系整体计划（审查稿）

## 文档状态

- 状态：审查稿
- 目标：先固化整体设计，经过多轮审查后再开始初始化 skill、拆分 references 和迁移种子内容
- 当前约束：本阶段只做规划，不开始正式落地

## 一、建设目标

这套 skill 体系的唯一核心目标是：

让 Codex 在参与团队研发时，能够从需求、Bug、编码、编码审查、测试、交付的全流程中，按任务阶段和代码改动位置自动触发对应规则，不依赖人工手动拼接 skill。

希望最终达到的效果：

- 需求不清时，自动进入澄清、补全和边界确认
- Bug 处理时，自动进入问题提出、复现、静态定位、运行时诊断、根因分析、修复建议和验证链路
- 开始写代码时，自动进入可读性、命名、注释和风格一致性规则
- 改到特定代码位置时，自动命中特定 skill
- 写完代码后，自动进入编码审查，检查是否符合编码规范、注释规范、存放位置规范、最小改动原则，以及语法正确性、未使用导入与引用清理、调试残留清理和代码格式化要求
- 写完代码后，自动进入测试目录、命名、测试程序位置、测试文档说明等规范
- 进入上线或交付阶段时，自动进入发布、回滚、评审和交付说明规则

## 二、总体设计原则

### 1. 自动触发优先

所有 skill 都以“自动触发”为目标设计，不以手动显式调用为主。

### 2. 触发描述必须具体

每个 skill 的 `description` 必须写清楚触发信号，不能只写抽象用途。

正确风格示例：

- 当新增或修改 `.env`、`application.yml`、`settings.py`、`config.ts` 时触发
- 当新增或修改 migration、DDL、表字段、索引时触发
- 当新增或修改 controller、router、handler、HTTP 方法、DTO 时触发
- 当新增或修改异常类、全局异常处理、重试、降级、fallback 时触发
- 当新增或修改测试目录、测试脚本、测试 README 时触发

### 3. 以“阶段 + 问题类型 + 代码位点”三维组合设计

这套体系不只按技术分类，而是同时考虑三个维度：

- 阶段维度：当前处于需求、Bug、编码、编码审查、测试、交付哪个阶段
- 问题类型维度：当前是新需求，还是问题修复
- 代码位点维度：当前改动的是配置、数据库、API、响应、错误处理、日志、测试目录等哪一类内容

### 4. 细分，但不混乱

细分是为了方便未来持续补规则、替换规则、迁移规则。

每个 skill 只负责一种清晰职责，不做大杂烩。

### 5. 预留扩展性

未来允许继续：

- 新增 skill
- 合并 skill
- 拆分 skill
- 调整 skill 触发范围
- 把某类规则从一个 skill 迁移到另一个 skill

## 三、总体系结构

计划采用 9 个域：

| 域         | 作用                                                             |
| ---------- | ---------------------------------------------------------------- |
| 总控层     | 作为弱触发协调层，负责阶段分析、路由分流、冲突裁决与流程中断管控，不承载细粒度编码规则，也不替代小 skill 执行 |
| 需求域     | 处理新功能、新页面、新接口、新模块等需求类任务                   |
| Bug 域     | 处理问题描述、复现、静态定位、运行时诊断、根因、修复建议和验证   |
| 编码基线域 | 所有编码任务都自动遵守的基础质量规则                             |
| 代码位点域 | 改到哪类代码就触发哪类 skill                                     |
| 编码审查域 | 代码写完后、测试前的实现自审，检查规范、注释、语法、引用、格式、存放位置和变更清理 |
| 测试域     | 管理测试策略、测试位置、测试命名、测试文档和联调                 |
| 交付域     | 管理评审、提交流程、交付说明、发布和回滚                         |
| 体系维护域 | 管理 skill 自身的拆分、迁移、演进规范                            |

## 四、总控层

总控层不是“总规则全集”，而是“流程协调 / 阶段分析 / 路由总控”层。

它的作用是：

- 判断当前任务处于哪个阶段
- 判断当前任务更像需求、Bug、编码、测试还是交付问题
- 在多个 skill 同时触发时进行协调和裁决
- 在领域边界不清时承担备用路由功能
- 在流程应暂停、重启、继续或终止时统一收口

它不负责：

- 代替代码位点类 skill 执行具体规则
- 代替编码基线类 skill 执行具体规范
- 代替测试域、Bug 域或交付域执行细则
- 把所有团队规则重新集中写成一个大 skill

总控层新增两项核心能力：

1. 冲突裁决能力

- 当多个 skill 同时触发、规则出现重叠或互相冲突时，由总控层统一裁决
- 裁决优先级固定为：阶段优先、位点次之、类型兜底
- 阶段优先：先保证当前阶段该做的事情不被跳过，例如需求未澄清前不能直接进入编码，编码审查未通过前不能直接进入测试
- 位点次之：在阶段正确的前提下，再按当前实际改动的代码位点选择最相关的 skill
- 类型兜底：如果阶段和位点都不能完全覆盖，再由需求类型、Bug 类型或交付类型规则兜底
- 当多个 skill 都有价值但不冲突时，总控层允许并行生效；当多个 skill 明显重复或互斥时，总控层负责去重和裁断

2. 流程中断管控

- 当需求澄清未通过、Bug 定位未完成、编码审查不达标、测试条件未满足或交付信息不完整时，总控层有权阻止流程进入下一阶段
- 总控层统一管理流程的暂停、重启、继续和终止条件，避免跳过关键环节
- “暂停”用于信息缺失、证据不足、前置步骤未完成
- “重启”用于需求变化、定位结论推翻、修复方案失效、回归验证失败
- “终止”用于发现当前路线错误、成本过高或继续推进会造成明显风险
- 各子 skill 可以提出暂停建议，但真正的阶段切换权限由总控层统一收口

### 总控层触发边界

总控层必须采用“弱触发”原则，不能写成默认覆盖所有研发任务的大 skill。

应触发的场景：

- 新任务刚进入时，尚未明确属于需求域、Bug 域、编码域、测试域还是交付域
- 任务描述较宽泛，无法直接判断由哪个小 skill 接手
- 多个 skill 同时触发，且可能发生冲突、重复或阶段错位
- 需求域与 Bug 域、编码审查域与测试域等边界不清，需要先分流
- 流程需要暂停、重启、继续或终止
- 用户明确要求按团队完整研发流程处理

不应优先由总控层触发的场景：

- 单纯改 SQL、改 API、改响应、改错误处理、改日志、改包结构、改工具类
- 单纯补注释、做格式清理、做语法检查
- 单纯写测试、补回归测试、做联调验证
- 任何已经有明确小 skill 能直接接手的单域任务

默认执行原则：

- 小独立 skill 是主执行层
- 总控层是协调层和备用路由层
- 当总控层与位点类 / 基线类 / 测试类 skill 同时出现时，总控层只负责判断和分流，不替代执行 skill 的具体规则

总控层的定位是“流程调度与异常管控中心”，而不是另一个大杂烩规范 skill，也不是默认全程强触发的总入口。

| Skill 名字               | 自动触发 description                                                                                         | 核心职责                                              |
| ------------------------ | ------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------- |
| `team-development-rules` | 当任务阶段不明确、领域边界不清、多个 skill 同时触发、流程需要暂停 / 重启 / 继续 / 终止，或用户明确要求按团队完整研发流程处理时自动触发。 | 作为弱触发协调层，负责阶段分析、路由分流、冲突裁决和中断管控，不承载数据库、API、错误处理等细节规则，也不替代小 skill 执行。 |

## 五、需求域

| Skill 名字                    | 自动触发 description                                                               | 核心职责                             |
| ----------------------------- | ---------------------------------------------------------------------------------- | ------------------------------------ |
| `requirement-intake-rules`    | 当用户提出新需求、新功能、新页面、新接口、新模块，且任务刚进入研发阶段时自动触发。 | 先理解目标、背景、上下游和输入输出。 |
| `requirement-gap-rules`       | 当需求描述不完整、缺少前提、缺少数据结构、缺少交互细节、缺少业务规则时自动触发。   | 识别缺失信息并暂停盲目实现。         |
| `requirement-boundary-rules`  | 当需求边界不清、影响范围不明、兼容性不明确、是否允许改旧逻辑不清楚时自动触发。     | 明确改动边界和影响面。               |
| `requirement-splitting-rules` | 当需求较大、涉及多个模块、多个接口、多个页面、多个步骤时自动触发。                 | 负责任务拆分、模块拆分和实施顺序。   |
| `requirement-change-rules`    | 当编码过程中需求被补充、修正、插入新条件、改变优先级时自动触发。                   | 重新确认变更范围和影响。             |
| `acceptance-criteria-rules`   | 当任务准备进入实现前确认，或交付前需要验收标准时自动触发。                         | 补齐可验证、可测试的验收标准。       |

### 需求域与 Bug 域边界判定

需求域和 Bug 域虽然都可能表现为“用户说有问题”，但分流标准必须明确：

1. 需求实现过程中发现的原有功能缺陷、历史遗留问题

- 归入 Bug 域处理
- 不纳入当前需求域范围
- 即使是在开发新需求时顺带暴露出来，只要问题本身属于旧逻辑缺陷、旧链路异常、历史行为错误，仍优先按 Bug 流程处理

2. 需求变更引发的新增问题、逻辑调整

- 归入需求域处理
- 不按 Bug 流程处理
- 这类情况本质上是“需求本身发生变化”，而不是“现有实现出现缺陷”
- 应优先进入 `requirement-change-rules`、`requirement-boundary-rules` 等需求域 skill，而不是直接进入 Bug 根因定位

3. 需求验收不通过、不符合预期

- 先判断是需求理解偏差，还是代码缺陷
- 如果是需求描述、验收标准、业务理解、边界判断出现偏差，归入需求域
- 如果是实现错误、运行错误、状态异常、历史逻辑缺陷，归入 Bug 域
- 验收不通过不能直接默认当成 Bug，也不能直接默认当成需求变更，必须先做归因判断

4. 默认判定优先级

- 先判断问题属于“原有实现缺陷”还是“当前需求变化”
- 再判断问题属于“理解偏差”还是“代码缺陷”
- 如果两者同时存在，由总控层决定是否拆成“需求调整任务 + Bug 修复任务”并行管理

## 六、Bug 域

Bug 属于需求的一个子类，但处理流程不同，因此单独建域。

Bug 域采用两条互补路径：

- 静态定位路径：通过阅读代码、追调用链、查日志、查 trace、分析模块边界来定位问题
- 运行时诊断路径：通过断点调试、打印 debug 日志、程序断言、变量快照、关键分支观测来缩小问题范围

有些 bug 只看代码无法稳定定位，必须通过运行中的状态变化、输入输出、分支命中和异常上下文才能找到真实位置，因此运行时诊断是 Bug 域的正式组成部分，不是临时技巧。

| Skill 名字                  | 自动触发 description                                                                                       | 核心职责                           |
| --------------------------- | ---------------------------------------------------------------------------------------------------------- | ---------------------------------- |
| `bug-intake-rules`          | 当用户描述报错、异常行为、结果不符、线上问题、偶发问题、接口异常、页面异常、数据错误、性能异常时自动触发。 | 把问题描述标准化。                 |
| `bug-gap-rules`             | 当 bug 描述缺少复现条件、环境信息、输入数据、报错日志、影响范围时自动触发。                                | 补齐定位所需基础信息。             |
| `bug-reproduction-rules`    | 当问题需要复现、构造步骤、确定触发条件、判断是否稳定发生时自动触发。                                       | 输出复现步骤和复现结论。           |
| `bug-scoping-rules`         | 当需要判断问题属于哪个模块、哪个业务链路、哪个服务、哪个页面、哪个接口、哪个数据流时自动触发。             | 明确问题归属和影响范围。           |
| `bug-root-cause-rules`      | 当开始分析代码、追调用链、看现有日志、查 trace、结合上下游行为定位根因时自动触发。                         | 找出根因并区分实现问题与设计问题。 |
| `bug-runtime-debug-rules`   | 当仅靠静态分析不能稳定定位 bug，且需要在运行过程中通过断点、单步执行、变量观察、调用栈观察、条件命中判断来定位时自动触发。 | 通过运行时调试缩小问题范围并定位异常位置。 |
| `bug-debug-log-rules`       | 当定位 bug 需要临时增加 debug 日志、关键变量输出、关键分支日志、上下游输入输出日志时自动触发。             | 通过可回收的临时日志补充运行期证据。 |
| `bug-assertion-diagnostic-rules` | 当怀疑状态异常、顺序错误、数据污染、不变量被破坏，且需要通过程序断言或诊断性检查快速暴露问题位置时自动触发。 | 用断言和诊断检查缩小 bug 发生区间。 |
| `bug-fix-proposal-rules`    | 当问题已定位，需要形成修改建议、风险评估、备选方案并等待用户确认时自动触发。                               | 先给修改建议，再决定是否实施。     |
| `bug-regression-risk-rules` | 当修复可能影响公共方法、共享模块、已有接口、数据库行为或兼容性时自动触发。                                 | 识别回归风险。                     |
| `bug-validation-rules`      | 当 bug 修复后需要验证是否修好、是否引入副作用、是否需要补回归测试时自动触发。                              | 负责修复后的验证闭环。             |

补充约束：

- `bug-debug-log-rules` 和 `bug-assertion-diagnostic-rules` 产出的临时诊断代码，在进入测试或交付前，必须交由 `cleanup-format-review-rules` 检查是否应删除、降级、保留或改造为正式日志/正式校验
- 断点调试属于定位手段，不是交付结果；最终交付说明中应记录定位结论，而不是保留调试动作本身

## 九、编码审查域

这部分位于“编码完成”和“开始测试”之间，负责实现后的自审，不等同于 PR 评审。

编码审查域重点覆盖：

- 是否符合编码规范、命名规范、注释规范和最小改动原则
- 是否存在语法错误、类型错误、引用缺失、构建失败风险
- 是否存在未使用导入、未使用变量、未使用方法引用、死代码、调试残留和临时代码
- 是否存在多余换行、格式不一致、代码块排版混乱等可自动整理问题
- 是否存在代码落点不合理、目录归属错误、跨层调用越界等结构问题

| Skill 名字                    | 自动触发 description                                                                                                                                             | 核心职责                                                 |
| ----------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------- |
| `implementation-review-rules` | 当功能代码已经完成、准备进入测试前验证时自动触发，检查是否符合编码规范、命名规范、注释规范、风格一致性和最小改动原则。                                           | 对刚完成的实现做一次测试前规范自审。                     |
| `syntax-check-review-rules`   | 当新增或修改代码后准备进入测试前验证，且需要确认语法、类型、依赖引用、构建基础正确性时自动触发。                                                                 | 检查语法错误、引用缺失、类型问题和明显构建失败风险。     |
| `cleanup-format-review-rules` | 当新增或修改代码后准备进入测试前验证，且存在未使用导入、未使用变量、未使用方法引用、调试残留、临时代码、多余换行或格式不一致风险时自动触发。                     | 清理代码噪音并统一基础格式。                             |
| `code-placement-review-rules` | 当新增文件、移动文件、扩展模块、跨层调用、工具类落点或目录归属可能不合理时，在进入测试前自动触发。                                                               | 检查代码存放位置、模块归属、层级边界和依赖方向是否合理。 |

### 编码审查域与测试域边界判定

编码审查域与测试域虽然都发生在“代码写完之后”，但职责必须严格分开：

1. 编码审查域

- 聚焦代码本身质量
- 负责语法、类型、格式、规范、注释、代码归属、依赖引用、未使用导入清理、调试残留清理、冗余代码清理等静态检查
- 属于开发自测环节
- 不负责功能逻辑是否真的满足业务预期

2. 测试域

- 聚焦功能正确性
- 负责测试用例、场景覆盖、联调验证、输入输出验证、功能闭环、回归验证等
- 属于功能验证环节
- 不负责代码格式、导入清理、注释风格、目录归属等静态规范问题

3. 默认分流规则

- 能通过阅读代码、静态分析、格式检查、语法检查、引用检查发现的问题，归入编码审查域
- 必须通过运行程序、执行测试、构造输入输出、联调上下游、观察实际结果才能确认的问题，归入测试域
- “代码看起来规范”不等于“功能正确”
- “功能跑通了”也不等于“代码质量合格”

4. 阶段顺序

- 默认先完成编码审查域，再进入测试域
- 编码审查未通过时，不应直接跳过进入测试
- 测试过程中如果暴露出静态质量问题，可以回流到编码审查域处理后再继续测试

## 十、测试域

测试阶段单独细分，避免测试规则继续被编码规则吞没。

| Skill 名字                    | 自动触发 description                                                             | 核心职责                     |
| ----------------------------- | -------------------------------------------------------------------------------- | ---------------------------- |
| `test-strategy-rules`         | 当功能实现和编码审查完成后，开始补测试、写验证脚本、做回归检查时自动触发。       | 决定测试层级和覆盖重点。     |
| `test-location-rules`         | 当新增测试目录、测试脚本、测试程序、模拟数据、验证代码时自动触发。               | 统一测试资源位置。           |
| `test-naming-rules`           | 当创建新的测试任务目录、测试文件、测试脚本、测试数据目录时自动触发。             | 统一测试目录和文件命名。     |
| `test-program-rules`          | 当新增测试程序、模拟程序、验证脚本、数据构造脚本时自动触发。                     | 统一测试程序和辅助脚本位置。 |
| `test-doc-rules`              | 当新增或修改测试 `README.md`、验证说明、测试报告、覆盖说明时自动触发。           | 统一测试说明文档结构。       |
| `functional-validation-rules` | 当需要验证新功能、修改后的功能、接口行为、页面交互、输入输出结果是否满足当前需求与验收标准时自动触发。 | 负责当前需求对应的功能正确性验证。 |
| `test-regression-rules`       | 当 Bug 修复、原有功能迭代、公共模块修改后，准备执行测试时自动触发。 | 明确回归测试的范围、用例选取、验证要点，针对改动点关联的功能、上下游链路做全覆盖验证，防止修复旧 Bug 引入新问题，保障功能兼容性。 |
| `integration-debugging-rules` | 当联调、排查接口不通、字段不一致、环境差异、trace 断链、测试环境问题时自动触发。 | 统一联调和排障证据记录。     |

### 测试域内部边界判定

测试域内部继续细分为五类职责：

1. 测试策略

- 由 `test-strategy-rules` 负责
- 重点回答“测什么、测到什么程度、优先覆盖哪些场景”
- 不负责决定测试资源放在哪里，也不负责亲自承担功能验证结论

2. 测试资源管理

- 由 `test-location-rules`、`test-naming-rules`、`test-program-rules`、`test-doc-rules` 共同负责
- 重点解决测试目录、命名、测试程序落点、测试说明文档等组织问题
- 只管理测试资产如何存放、如何命名、如何描述
- 不负责判断功能是否正确

3. 功能验证

- 由 `functional-validation-rules` 负责
- 聚焦当前需求或当前修改本身是否满足业务预期、接口预期、页面交互预期和验收标准
- 只关心“这次要做的东西是否真的做对了”
- 不负责历史能力是否被破坏

4. 联调验证

- 由 `integration-debugging-rules` 负责
- 聚焦上下游系统、接口协议、环境配置、链路串联、字段一致性、trace 连通性等跨边界验证
- 适用于“单点功能看起来没问题，但整个链路跑不通”的场景
- 不替代功能验证，也不替代回归验证

5. 回归测试与回归验证

- 由 `test-regression-rules` 负责
- 聚焦当前改动是否破坏旧逻辑、共享模块、公共接口、历史页面、已有流程和兼容行为
- 重点回答“这次改动有没有带坏别的地方”
- 明确回归范围、用例选取和验证要点
- 不负责决定测试资产存放规范

### 测试域默认分流规则

- 讨论“测什么、覆盖哪些场景、优先级怎么排”时，进入 `test-strategy-rules`
- 讨论“测试目录建在哪、怎么命名、README 怎么写”时，进入测试资源管理相关 skill
- 讨论“当前需求是否实现正确”时，进入 `functional-validation-rules`
- 讨论“上下游系统是否打通、环境是否一致、链路是否闭环”时，进入 `integration-debugging-rules`
- 讨论“旧功能有没有被这次改动带坏、回归范围怎么定、回归用例怎么选”时，进入 `test-regression-rules`

### 测试域内部顺序

- 先由 `test-strategy-rules` 决定验证层级和重点
- 再由测试资源管理类 skill 规范测试资产位置、命名和文档
- 然后执行 `functional-validation-rules`
- 涉及跨系统或跨环境时，再进入 `integration-debugging-rules`
- 最后执行 `test-regression-rules`，确认改动没有破坏原有能力，并完成回归范围验证

## 十三、建议的每个 Skill 标准结构

为了保证未来继续扩展时不乱，所有 skill 都建议按相同结构编写：

1. 适用场景
2. 自动触发信号
3. 进入后必须先做什么
4. 什么时候暂停并向用户确认
5. 本 skill 产出什么结果
6. 下一步应交给哪个 skill
7. 应读取哪些 references
8. 本 skill 执行通过 / 驳回的判定标准
9. 执行结果归档要求

补充说明：

- “本 skill 执行通过 / 驳回的判定标准”必须明确合格与不合格的边界，避免只写原则、不写判定条件
- 判定标准应尽量可检查、可复核，避免出现“基本合理”“大致完成”这类模糊措辞
- “执行结果归档要求”必须明确产出内容保存到哪里、采用什么记录格式、是否需要附带说明或结论
- 归档要求的目标是保证后续复盘、回看、审计和追溯时，能找到本次执行留下的证据和结论

### 统一归档目录约定（审查建议）

为了避免每个 skill 各自定义一套归档位置，默认建议采用统一归档根目录。若项目本身已经存在明确目录约定，则优先遵循项目约定；以下规则作为团队 skill 的默认方案。

默认归档根目录建议如下：

- `analysis/`
  用于需求澄清记录、需求边界判定、Bug 定位记录、根因分析、修复建议、运行时诊断结论
- `review/`
  用于编码审查结果、代码评审结论、发布前检查记录、风险复核记录
- `test/`
  用于测试资产、测试任务目录、测试脚本、模拟数据、测试 README、测试结论
- `reports/`
  用于交付说明、安全报告、上线说明、回滚记录、复盘文档、专项分析报告

默认记录格式建议如下：

- 过程说明、结论摘要、判定结果优先使用 `Markdown`，文件后缀为 `.md`
- 结构化配置、检查结果、清单类数据优先使用 `.json`、`.yaml` 或 `.yml`
- 原始日志、命令输出、运行快照优先使用 `.log` 或 `.txt`
- 截图、录屏、调用链导出、trace 附件统一放入对应任务目录下的 `attachments/` 子目录

默认归档命名建议如下：

- 任务级目录优先使用 `yyyy-MM-dd-HHmmss-中文任务简述`
- 文件名应体现“阶段 + 主题 + 结果”，避免使用 `temp.md`、`result.txt` 这类无意义命名
- 同一任务下的归档内容应能从目录名和文件名直接看出来源与用途

默认归档最小要求如下：

- 只要一个 skill 产生了可复用结论、明确判定、关键证据或后续要继续接力的信息，就必须归档
- 归档内容至少应包含：任务背景、执行结论、通过 / 驳回判定、关键证据位置、下一步动作
- 纯临时调试动作本身不要求长期保留，但由此得出的定位结论、修复依据和关键证据应保留

特殊说明：

- 测试域继续遵循 `test/` 目录作为测试资源根目录的原则
- Bug 域的运行时诊断产物如果最终保留，应优先归档到 `analysis/` 下对应任务目录，而不是散落在代码目录中
- 交付域产生的正式说明、检查记录和报告，应优先归档到 `reports/` 或 `review/`，避免混入测试目录

## 十四、推荐的 references 组织方式

建议每个 skill 都采用“精简 `SKILL.md` + 结构化 `references/`”的方式组织，不建议把全部规则堆在 `SKILL.md` 里。

### 1. `SKILL.md` 保留内容

`SKILL.md` 只保留最核心的、必须第一时间被加载的内容，原则上只写：

- 核心触发逻辑
- 执行流程
- 权责边界
- 暂停 / 确认条件
- 通过 / 驳回判定标准
- references 读取入口与选择规则

约束要求：

- `SKILL.md` 必须保持精简，严禁堆砌大量细则
- 不把语言细节、框架细节、场景枚举、长篇示例直接塞进 `SKILL.md`
- 一切可拆分、可复用、可按需加载的细节，都应进入 `references/`

### 2. `references/*.md` 拆分规则

`references/` 用于承载细则，必须按可维护、可复用、可独立更新的原则拆分。

拆分维度优先级建议如下：

1. 按语言拆分

- 例如：`java-*.md`、`python-*.md`、`typescript-*.md`

2. 按框架或技术栈拆分

- 例如：`mybatis-*.md`、`jpa-*.md`、`spring-*.md`
- 例如：`vue-*.md`、`react-*.md`

3. 按场景拆分

- 例如：`bug-runtime-debug.md`
- 例如：`api-response-envelope.md`
- 例如：`test-regression-scope.md`

4. 按代码位点拆分

- 例如：`database-sql-rules.md`
- 例如：`request-header-trace.md`
- 例如：`common-util-placement.md`

典型示例：

- 数据库规则可拆成：
  - `sql-rules.md`
  - `mybatis-rules.md`
  - `jpa-rules.md`
- 前端规则可拆成：
  - `vue-rules.md`
  - `react-rules.md`
- 包结构规则可拆成：
  - `go-package-layout-rules.md`
  - `go-dependency-boundary-rules.md`
  - `java-package-layout-rules.md`
  - `python-module-layout-rules.md`
- 安全规则可拆成：
  - `fastapi-security.md`
  - `django-security.md`
  - `react-security.md`

拆分原则：

- 一份 reference 只负责一类相对稳定的主题
- 同一份 reference 不要同时混合语言、框架、位点、流程四种维度
- 只要某一部分会独立演进、独立增补、独立复用，就应该拆出去

### 3. references 命名规范

全体系统一使用小写字母 + 连字符命名，不使用空格，不使用中文文件名。

推荐命名格式：

- `主题-规则.md`
- `语言-主题-规则.md`
- `框架-主题-规则.md`
- `位点-主题-规则.md`

命名示例：

- `sql-rules.md`
- `mybatis-rules.md`
- `jpa-rules.md`
- `react-rules.md`
- `vue-rules.md`
- `error-response-rules.md`
- `trace-header-rules.md`
- `runtime-debug-rules.md`

禁止情况：

- `规则1.md`
- `临时.md`
- `最终版.md`
- `各种规范汇总.md`
- 含义不清、不能从文件名判断用途的命名

### 4. references 目录层级规范

默认建议每个 skill 只保留一层 `references/`，不要做过深目录嵌套。

推荐结构：

```text
skill-name/
├── SKILL.md
├── agents/
│   └── openai.yaml
└── references/
    ├── general-rules.md
    ├── java-xxx-rules.md
    ├── python-xxx-rules.md
    ├── react-xxx-rules.md
    └── ...
```

目录层级控制原则：

- 优先平铺，不优先深层级
- 如果 references 文件数量较少，全部放在 `references/` 根下
- 只有当某个 skill 的 references 数量明显变多且维度非常稳定时，才考虑再分一级目录
- 即使分目录，也只建议一层，例如：
  - `references/languages/`
  - `references/frameworks/`
  - `references/scenarios/`

### 5. 全体系统一性要求

所有 skill 的 references 组织方式要尽量统一，避免不同 skill 各写一套风格。

统一要求包括：

- 命名风格统一
- 目录层级统一
- 拆分维度统一
- 文件职责边界统一
- 不在一个 skill 平铺、另一个 skill 深层嵌套、第三个 skill 全堆进 `SKILL.md`

目标是：

- 后续任何人补规则时，都知道该把内容放在哪
- 后续任何人审查 skill 时，都能快速找到对应 reference
- 后续迁移、合并、拆分 skill 时，references 结构可以直接复用

## 十七、实施落地计划（串行安全版）

整体落地原则不是“同时铺很多个 skill”，而是“一个一个做，做完一个、审完一个、验证一个，再开始下一个”。

### 1. 实施总原则

- 严格串行，不并行落地多个 skill
- 每个 skill 必须完成定义、结构、样例、校验、审查、回归后，才能进入下一个
- 先做模板和样板，再批量扩展
- 先做高频、高价值、边界清晰的 skill，再做复杂或依赖更多上下文的 skill
- 每次落地只解决一个明确目标，不在同一轮里顺手做额外扩展

### 2. 单个 Skill 的标准落地步骤

每个 skill 一律按以下顺序落地：

1. 固化目标

- 明确这个 skill 解决什么问题
- 明确它和相邻 skill 的边界
- 明确自动触发 description

2. 固化结构

- 创建 skill 目录骨架
- 写精简版 `SKILL.md`
- 规划 `references/` 文件清单
- 明确通过 / 驳回标准
- 明确归档要求

3. 写第一版内容

- 只写当前 skill 自己必须有的规则
- 不提前把未来规则塞进来
- 不跨 skill 抢职责

4. 做本地静态校验

- 检查 `SKILL.md` frontmatter
- 检查命名、目录、references 结构
- 检查 description 是否过宽或过窄

5. 做场景验证

- 至少准备 3 类真实场景：
  - 应该触发的场景
  - 不该触发的场景
  - 容易混淆边界的场景
- 验证 skill 是否真的符合预期边界

6. 做审查修订

- 记录本轮发现的问题
- 只修这个 skill 自身的问题
- 不连带开启下一个 skill

7. 冻结当前版本

- 在文档中记录当前 skill 已审查通过
- 记录遗留问题和后续优化项

8. 再进入下一个 skill

### 3. 单个 Skill 的完成判定

一个 skill 只有同时满足下面条件，才算“完成”，否则不能进入下一个：

- 目标和边界明确
- 自动触发 description 定稿
- `SKILL.md` 已完成精简化
- `references/` 结构已完成第一版
- 通过 / 驳回标准已写清
- 归档要求已写清
- 至少经过一轮场景验证
- 至少经过一轮人工审查修订

### 4. 推荐实施波次

为了降低风险，建议按“先模板、再样板、再扩展”的顺序落地。

#### 第一波：基础模板与协调层

目标：先把整套体系的模板、约定和总控层收稳。

- `team-development-rules`
- `skill-evolution-rules`

完成标准：

- 总控层定位稳定为弱触发协调层
- 标准结构、归档规范、references 规范全部定稿
- 后续 skill 可以直接复用模板

第一波详细实施步骤：

1. 固化总控层最终定义

- 只处理 `team-development-rules`
- 不同时处理其他业务 skill
- 最终确认总控层的定位、触发边界、冲突裁决规则、流程中断规则
- 输出物：
  - 总控层最终定义审查稿
  - 总控层触发 / 不触发场景清单

2. 固化公共模板规范

- 只处理所有 skill 的公共模板，不开始写具体业务细则
- 最终确认：
  - skill 标准结构
  - references 拆分规则
  - references 命名与层级规范
  - 归档目录约定
  - 通过 / 驳回标准写法
- 输出物：
  - 公共模板规范稿
  - 公共 references 结构规范稿

3. 落地 `skill-evolution-rules` 规划

- 先不急着扩其他 skill
- 优先把“后续 skill 怎么拆、怎么改、怎么迁移”的元规则稳定下来
- 输出物：
  - `skill-evolution-rules` 的目标、边界、触发定义草案

4. 做第一波场景验证

- 至少验证 3 类场景：
  - 只有小 skill 应该触发，总控层不应抢占
  - 阶段不明确，总控层应介入分流
  - 多个 skill 同时命中，总控层应负责裁决
- 输出物：
  - 第一波验证样例列表
  - 每个样例的预期路由结果

5. 做第一波复盘

- 审核第一波里是否仍存在：
  - 总控层过宽
  - 模板过重
  - references 结构不统一
  - 归档要求不可执行
- 输出物：
  - 第一波复盘记录
  - 第一波遗留问题清单

第一波退出条件：

- `team-development-rules` 已被确认是弱触发协调层，而不是总规则 skill
- `skill-evolution-rules` 的规划已成型
- 所有 skill 都有统一标准结构模板
- `references/` 组织规范已稳定
- 归档规范已稳定
- 总控层与小 skill 的主次关系已经在样例验证中跑通

只有第一波退出条件全部满足，才能进入第二波。

#### 第二波：编码主干样板

目标：选最核心、最常用、边界最清晰的编码 skill 做样板。

- `code-minimal-change-rules`
- `code-readability-rules`
- `code-style-consistency-rules`
- `package-structure-rules`

完成标准：

- 编码基线类 skill 的写法稳定
- 代码位点类 skill 的触发描述写法稳定
- “基线域”和“位点域”的边界基本跑通

第二波固定顺序：

1. `code-minimal-change-rules`
2. `code-readability-rules`
3. `code-style-consistency-rules`
4. `package-structure-rules`

第二波详细实施步骤：

1. 先落地 `code-minimal-change-rules`

- 把它作为第一个编码样板，不并行处理其他编码 skill
- 先明确“最小改动”的边界：
  - 什么属于合理改动
  - 什么属于无关修改
  - 什么属于冗余改动
  - 什么属于过度优化
- 先形成第一版通过 / 驳回判定标准
- 输出物：
  - `code-minimal-change-rules` 边界说明稿
  - 最小改动正反例清单

2. 再落地 `code-readability-rules`

- 必须以前一个 skill 稳定为前提
- 重点明确可读性规则与最小改动规则之间的边界
- 避免把“为了更好看而顺手大改”误包装成可读性优化
- 输出物：
  - `code-readability-rules` 第一版结构稿
  - 可读性与最小改动冲突场景清单

3. 再落地 `code-style-consistency-rules`

- 在前两个 skill 稳定后再进入
- 重点确认风格一致性是否服从最小改动原则
- 只允许“跟随现有项目风格”的规则，不允许顺手改造成另一套风格
- 输出物：
  - 风格一致性判定稿
  - 项目既有风格跟随清单模板

4. 最后落地 `package-structure-rules`

- 作为第二波中唯一一个更偏代码位点的样板
- 重点验证包结构 / 目录结构 / 模块分层类 skill 的写法是否能独立成立
- 先从 Go 语言包定义场景做样板，再预留其他语言扩展位
- 输出物：
  - Go 包结构样板清单
  - 包职责与依赖方向样板表

5. 做第二波联合验证

- 至少验证 4 类场景：
  - 小范围 Bug 修复，只应命中最小改动和可读性，不应误触发包结构调整
  - 新增少量业务逻辑，应命中可读性和风格一致性
  - 新增公共目录或模块，应命中包结构规则
  - 仅代码格式和组织问题，不应被误判为功能位点规则
- 输出物：
  - 第二波验证样例列表
  - 每个样例的预期命中 skill 清单

6. 做第二波复盘

- 审核第二波里是否仍存在：
  - 编码基线域之间边界重叠
  - 最小改动原则被其他规则稀释
  - 包结构规则写得过宽，压制普通编码 skill
  - 代码位点类 description 不够具体
- 输出物：
  - 第二波复盘记录
  - 第二波遗留问题清单

第二波退出条件：

- `code-minimal-change-rules` 已稳定，并能作为后续编码 skill 的基础约束
- `code-readability-rules` 与 `code-minimal-change-rules` 的边界已稳定
- `code-style-consistency-rules` 已明确服从项目现有风格，不会反客为主
- `package-structure-rules` 已能独立表达包结构 / 分层 / 依赖方向规则
- 编码基线域与代码位点域的边界已通过样例验证跑通
- 第二波样板已经能作为后续编码类 skill 的模板

只有第二波退出条件全部满足，才能进入第三波。

#### 第三波：需求与 Bug 样板

目标：把需求流和 Bug 流跑通，验证阶段分流是否成立。

- `requirement-gap-rules`
- `requirement-boundary-rules`
- `bug-intake-rules`
- `bug-root-cause-rules`
- `bug-runtime-debug-rules`
- `bug-fix-proposal-rules`

完成标准：

- 需求域与 Bug 域边界稳定
- Bug 的静态定位与运行时诊断双路径可执行
- 总控层的分流价值被验证

第三波固定顺序：

1. `requirement-gap-rules`
2. `requirement-boundary-rules`
3. `bug-intake-rules`
4. `bug-root-cause-rules`
5. `bug-runtime-debug-rules`
6. `bug-fix-proposal-rules`

第三波详细实施步骤：

1. 先落地 `requirement-gap-rules`

- 先把“需求信息不足时必须暂停”的边界写稳
- 重点明确哪些信息缺失会阻止编码继续推进
- 避免把普通不确定项写得过重，导致所有需求都被过度拦截
- 输出物：
  - 需求缺失信息判定清单
  - 需求暂停触发条件清单

2. 再落地 `requirement-boundary-rules`

- 在上一项稳定后，再处理需求边界
- 重点明确：
  - 什么属于当前需求范围
  - 什么属于历史问题
  - 什么属于需求变更
  - 什么属于兼容性影响面
- 输出物：
  - 需求边界判定清单
  - 需求变更与历史问题区分样例

3. 再落地 `bug-intake-rules`

- 把 Bug 的描述标准化作为 Bug 域入口
- 重点明确异常现象、影响范围、环境条件、期望结果与实际结果的记录方式
- 避免一开始就进入代码分析，先把问题说清楚
- 输出物：
  - Bug 描述模板
  - Bug 基础信息必填项清单

4. 再落地 `bug-root-cause-rules`

- 先以静态定位为主样板
- 重点明确：
  - 什么时候优先看代码、日志、trace、调用链
  - 什么时候可以形成根因结论
  - 什么时候必须承认“静态定位证据不足”
- 输出物：
  - 静态定位路径清单
  - 根因结论形成条件清单

5. 再落地 `bug-runtime-debug-rules`

- 只有在静态定位不能稳定收敛时再进入
- 重点明确：
  - 什么时候允许进入断点 / debug 日志 / 运行时观察
  - 什么时候运行时诊断结束
  - 运行时诊断产物如何回流到 Bug 结论
- 避免把运行时调试写成默认第一选择
- 输出物：
  - 运行时诊断进入条件清单
  - 运行时诊断结束条件清单
  - 静态定位与运行时诊断切换样例

6. 最后落地 `bug-fix-proposal-rules`

- 作为需求域 / Bug 域通往编码域之前的最后一个样板
- 重点明确：
  - 什么情况下必须先给修改建议
  - 修改建议里必须包含哪些内容
  - 什么情况下允许直接进入编码，什么情况下必须等确认
- 输出物：
  - 修复建议模板
  - 修复建议通过 / 驳回标准

7. 做第三波联合验证

- 至少验证 5 类场景：
  - 需求描述不完整，应命中需求域，不应误入 Bug 域
  - 需求实现中发现历史遗留缺陷，应转入 Bug 域
  - 需求变更引发逻辑调整，应继续留在需求域
  - 仅靠静态定位可以收敛的 Bug，不应过早进入运行时调试
  - 静态证据不足的 Bug，应进入运行时诊断，再回流到修复建议
- 输出物：
  - 第三波验证样例列表
  - 每个样例的预期分流路径

8. 做第三波复盘

- 审核第三波里是否仍存在：
  - 需求域与 Bug 域边界不清
  - Bug 域入口信息要求过重或过轻
  - 静态定位与运行时诊断的切换条件不明确
  - 修复建议模板不足以支撑后续编码决策
- 输出物：
  - 第三波复盘记录
  - 第三波遗留问题清单

第三波退出条件：

- `requirement-gap-rules` 已能稳定拦截信息不足场景
- `requirement-boundary-rules` 已能稳定区分当前需求、需求变更与历史问题
- `bug-intake-rules` 已形成统一问题描述模板
- `bug-root-cause-rules` 已能独立表达静态定位路径
- `bug-runtime-debug-rules` 已明确只在静态定位不足时介入
- `bug-fix-proposal-rules` 已能把 Bug 域稳定交接到编码域
- 需求域与 Bug 域边界已通过样例验证跑通
- 总控层在第三波中的分流价值已被验证，而不是替代执行

只有第三波退出条件全部满足，才能进入第四波。

#### 第四波：接口与公共逻辑样板

目标：覆盖最常见业务开发位点。

- `api-endpoint-rules`
- `api-request-rules`
- `api-response-rules`

完成标准：

- 公共逻辑、错误处理、日志链路开始形成统一规范

第四波固定顺序：

1. `api-endpoint-rules`
2. `api-request-rules`
3. `api-response-rules`
4. `error-handling-rules`
5. `logging-trace-rules`
6. `common-util-rules`

第四波详细实施步骤：

1. 先落地 `api-endpoint-rules`

- 先把接口入口定义清楚
- 重点明确：
  - controller / router / handler 的职责边界
  - 路径命名与 HTTP 方法语义
  - CRUD 接口的基本设计边界
  - 哪些属于接口入口问题，哪些不属于
- 输出物：
  - 接口入口设计规则稿
  - 路径命名与方法语义样例清单

2. 再落地 `api-request-rules`

- 必须在接口入口样板稳定后再进入
- 重点明确：
  - path / query / body / DTO 的边界
  - 参数校验、必填字段、可选字段的表达方式
  - 请求结构规则与业务校验规则的边界
- 输出物：
  - 请求模型与参数校验规则稿
  - 请求参数正反例清单

3. 再落地 `api-response-rules`

- 在请求规则稳定后再处理响应
- 重点明确：
  - 成功响应结构
  - 错误响应结构
  - 分页结构
  - 兼容字段与版本字段的放置规则
- 避免把错误处理逻辑整体写进响应结构规则里
- 输出物：
  - 响应结构模板
  - 成功 / 错误 / 分页响应样例

4. 再落地 `error-handling-rules`

- 作为接口类 skill 之后的横切规则
- 重点明确：
  - 异常分类
  - 全局异常处理与局部处理的边界
  - 重试、超时、降级、fallback 的适用条件
  - 错误处理与错误响应的分工
- 输出物：
  - 错误分类清单
  - 错误处理路径样例
  - 错误处理与错误响应边界说明

5. 再落地 `logging-trace-rules`

- 在错误处理之后进入
- 重点明确：
  - 日志字段
  - trace / span 透传规则
  - 关键链路记录点
  - 日志与调试日志、审计日志的边界
- 避免把业务日志、调试日志、审计日志混成一套
- 输出物：
  - 日志字段清单
  - trace 透传样例
  - 日志类型边界说明

6. 最后落地 `common-util-rules`

- 放在第四波最后处理
- 重点明确：
  - 什么才算通用工具
  - 什么只是当前业务逻辑，不应抽成公共代码
  - 公共方法、工具类、通用组件、常量的存放位置与调用方式
  - 公共工具与包结构规则、位点规则之间的边界
- 避免以“复用”为名把业务逻辑过早抽成公共工具
- 输出物：
  - 公共工具判定清单
  - 工具类 / 常量 / 通用组件落点样例
  - 公共工具与业务逻辑边界说明

7. 做第四波联合验证

- 至少验证 6 类场景：
  - 新增接口路由，应优先命中接口入口规则
  - 修改请求参数结构，应命中请求规则，不应误触发响应规则
  - 修改返回包裹结构，应命中响应规则，不应把错误处理整体拉进来
  - 修改异常中间件或重试逻辑，应命中错误处理规则
  - 修改日志字段或 trace 透传，应命中日志链路规则
  - 抽取公共方法或常量时，应验证是否真的应该进入公共工具层
- 输出物：
  - 第四波验证样例列表
  - 每个样例的预期命中 skill 清单

8. 做第四波复盘

- 审核第四波里是否仍存在：
  - 接口入口、请求、响应三者边界交叉
  - 错误处理和错误响应职责混淆
  - 日志规则覆盖过宽或过窄
  - 公共工具规则过度鼓励抽象，导致公共层膨胀
- 输出物：
  - 第四波复盘记录
  - 第四波遗留问题清单

第四波退出条件：

- `api-endpoint-rules` 已能独立表达接口入口设计边界
- `api-request-rules` 与 `api-response-rules` 已清楚分离
- `error-handling-rules` 与 `api-response-rules` 的边界已稳定
- `logging-trace-rules` 已能独立表达日志与 trace 规则，而不依附于错误处理
- `common-util-rules` 已能稳定区分“通用工具”与“业务逻辑”
- 后端高频位点之间的边界已通过样例验证跑通
- 第四波样板已能作为后续后端位点类 skill 的模板

只有第四波退出条件全部满足，才能进入第五波。

#### 第五波：测试样板

目标：把“写完代码 -> 编码审查 -> 测试 -> 回归”链路打通。

- `implementation-review-rules`
- `syntax-check-review-rules`
- `cleanup-format-review-rules`
- `code-placement-review-rules`
- `test-location-rules`
- `test-naming-rules`
- `test-program-rules`
- `test-doc-rules`
- `functional-validation-rules`
- `test-regression-rules`

完成标准：

- 编码审查域与测试域边界稳定
- 测试资源管理和功能验证、回归验证分离清楚

第五波固定顺序：

1. `implementation-review-rules`
2. `syntax-check-review-rules`
3. `cleanup-format-review-rules`
4. `code-placement-review-rules`
5. `test-location-rules`
6. `test-naming-rules`
7. `test-program-rules`
8. `test-doc-rules`
9. `functional-validation-rules`
10. `test-regression-rules`

第五波详细实施步骤：

1. 先落地 `implementation-review-rules`

- 把它作为“编码完成后的第一道自审口”
- 重点明确：
  - 什么属于编码规范问题
  - 什么属于注释与风格问题
  - 什么属于最小改动原则违背

- 输出物：
  - 实现自审检查清单

2. 再落地 `syntax-check-review-rules`

- 作为编码审查域中的第二道检查
- 重点明确：
  - 语法错误、类型错误、引用缺失、明显构建失败风险的判定标准
  - 语法检查和格式清理的边界
- 输出物：
  - 语法 / 类型 / 引用检查清单
  - 语法类问题样例

3. 再落地 `cleanup-format-review-rules`

- 在语法检查之后进入
- 重点明确：
  - 未使用导入、未使用变量、未使用方法引用
  - 调试残留、临时代码、死代码
  - 多余换行、格式不一致、代码块排版问题
- 避免把功能性修改包装成“格式清理”
- 输出物：
  - 清理与格式化检查清单
  - 可自动整理问题样例

4. 再落地 `code-placement-review-rules`

- 作为编码审查域的最后一道结构检查
- 重点明确：
  - 文件归属、模块归属、目录归属
  - 跨层调用、依赖方向
  - 是否出现位置合理但职责不合理的伪通过情况
- 输出物：
  - 代码归属检查清单
  - 目录 / 分层正反例样例

5. 再落地 `test-location-rules`

- 从这里开始进入测试域中的资源管理部分
- 重点明确测试资源根目录、任务目录和禁止散落规则
- 输出物：
  - 测试目录落点规则稿
  - 测试资源放置样例

6. 再落地 `test-naming-rules`

- 在测试位置稳定后再进入
- 重点明确测试任务目录、测试文件、测试数据目录命名
- 输出物：
  - 测试命名规范清单
  - 命名正反例样例

7. 再落地 `test-program-rules`

- 重点明确测试程序、模拟程序、验证脚本、数据构造脚本的边界和落点

- 输出物：
  - 测试程序落点规则稿
  - 测试程序与业务代码边界样例

8. 再落地 `test-doc-rules`

- 重点明确测试 README、验证说明、测试报告、覆盖说明的最小要求
- 输出物：
  - 测试文档模板
  - 测试文档通过 / 驳回标准

9. 再落地 `functional-validation-rules`

- 作为测试域中真正的功能验证样板
- 重点明确：
  - 当前需求的验证范围
  - 功能正确性的判定方式
  - 功能验证与联调验证、回归验证的边界
- 输出物：
  - 功能验证样例集
  - 功能验证结论模板

10. 最后落地 `test-regression-rules`

- 作为第五波收尾 skill
- 重点明确：
  - 回归范围怎么定
  - 回归用例怎么选
  - 公共模块修改、Bug 修复、功能迭代后的回归链路如何覆盖
- 输出物：
  - 回归范围判定清单
  - 回归用例选取模板
  - 回归验证结论模板

11. 做第五波联合验证

- 至少验证 6 类场景：
  - 编码规范不通过，应被拦在编码审查域，不能直接进测试域
  - 语法和格式问题应由编码审查域收口，不应进入功能验证
  - 测试资源位置和命名问题应由测试资源管理类 skill 接管
  - 功能正确性问题应命中 `functional-validation-rules`
  - Bug 修复后的兼容性问题应命中 `test-regression-rules`
  - 编码审查通过但功能验证失败，应回流编码域或 Bug 域，而不是停留在测试资源管理层
- 输出物：
  - 第五波验证样例列表
  - 每个样例的预期流转路径

12. 做第五波复盘

- 审核第五波里是否仍存在：
  - 编码审查域内部职责交叉
  - 测试资源管理与功能验证职责混淆
  - 功能验证与回归验证边界不清
  - 编码审查不通过却仍能进入测试的漏洞
- 输出物：
  - 第五波复盘记录
  - 第五波遗留问题清单

第五波退出条件：

- `implementation-review-rules` 已能稳定作为测试前第一道自审关口
- `syntax-check-review-rules` 与 `cleanup-format-review-rules` 边界已稳定
- `code-placement-review-rules` 已能独立表达结构与归属检查
- `test-location-rules`、`test-naming-rules`、`test-program-rules`、`test-doc-rules` 已形成稳定测试资源管理链
- `functional-validation-rules` 与 `test-regression-rules` 已清楚分离
- 编码审查域与测试域的边界已通过样例验证跑通
- “编码完成 -> 编码审查 -> 测试资源管理 -> 功能验证 -> 回归验证”的完整链路已可执行

只有第五波退出条件全部满足，才能进入第六波。

#### 第六波：剩余位点与增强项

目标：在前五波稳定后，再补剩余位点和增强能力。

- `config-change-rules`
- `database-schema-rules`
- `database-query-rules`
- `request-header-rules`
- `auth-security-rules`
- `performance-caching-rules`
- `integration-debugging-rules`
- 其他后续新增 skill

第六波固定顺序：

1. `config-change-rules`
2. `database-schema-rules`
3. `database-query-rules`
4. `request-header-rules`
5. `auth-security-rules`
6. `performance-caching-rules`
7. `integration-debugging-rules`

第六波详细实施步骤：

1. 先落地 `config-change-rules`

- 作为第六波入口，优先处理配置与环境差异问题
- 重点明确：
  - 环境变量与配置文件边界
  - 配置分层与默认值规则
  - 密钥占位与敏感配置边界
  - 配置修改与业务逻辑修改的边界
- 输出物：
  - 配置分层规则稿
  - 默认值与密钥处理清单

2. 再落地 `database-schema-rules`

- 在配置规则稳定后进入
- 重点明确：
  - 表、字段、索引、约束、迁移脚本的边界
  - 结构变更与数据变更的边界
  - 迁移安全、回滚和兼容性要求
- 输出物：
  - 数据库结构变更规则稿
  - 迁移与回滚样例清单

3. 再落地 `database-query-rules`

- 必须在 schema 规则之后进入
- 重点明确：
  - SQL、Repository、DAO、Mapper、QueryBuilder 的边界
  - 查询性能、事务、锁、批量操作的适用条件
  - 数据访问规则与业务逻辑规则的分工
- 输出物：
  - 数据访问规则稿
  - 查询 / 事务 / 锁样例清单

4. 再落地 `request-header-rules`

- 作为接口增强位点进入
- 重点明确：
  - 认证头、trace 头、span 头、租户头、幂等键、客户端上下文头的职责
  - Header 规则与请求参数规则的边界
  - Header 透传与日志 / trace 规则的边界
- 输出物：
  - 请求头分类清单
  - Header 使用与透传样例

5. 再落地 `auth-security-rules`

- 在接口、配置、数据库、请求头样板稳定后再进入
- 重点明确：
  - 认证、鉴权、输入校验、敏感信息处理
  - 上传下载安全与外部请求安全
  - 安全规则与普通接口 / 错误处理 / Header 规则的边界
- 输出物：
  - 安全能力边界清单
  - 安全场景正反例样例

6. 再落地 `performance-caching-rules`

- 放在安全规则之后
- 重点明确：
  - SQL 性能、接口耗时、缓存读取写入、缓存 key、缓存失效、前端渲染热点的边界
  - 性能优化与过度优化的边界
  - 缓存策略与数据库 / 接口规则之间的分工
- 输出物：
  - 性能与缓存规则稿
  - 性能优化与过度优化区分样例

7. 最后落地 `integration-debugging-rules`

- 作为第六波收尾 skill
- 重点明确：
  - 上下游系统联调
  - 接口字段不一致
  - 环境差异、链路不通、trace 断链
  - 联调问题与功能验证 / Bug 运行时诊断的边界
- 输出物：
  - 联调验证规则稿
  - 环境差异与链路问题样例

8. 做第六波联合验证

- 至少验证 7 类场景：
  - 修改环境变量与默认值，应命中配置规则
  - 修改表结构与迁移脚本，应命中 schema 规则
  - 修改查询逻辑与事务控制，应命中 query 规则
  - 新增 trace 头或幂等键，应命中请求头规则
  - 增加认证 / 鉴权 / 敏感信息处理，应命中安全规则
  - 做缓存或性能优化时，应验证没有被误包装成业务逻辑修改
  - 联调时出现环境差异、链路不通，应命中联调增强规则
- 输出物：
  - 第六波验证样例列表
  - 每个样例的预期命中 skill 清单

9. 做第六波复盘

- 审核第六波里是否仍存在：
  - 配置规则与业务逻辑边界不清
  - schema 与 query 规则交叉
  - Header、安全、日志 trace 三者边界混淆
  - 性能缓存规则过宽，鼓励过度优化
  - 联调规则和 Bug 运行时诊断规则重叠
- 输出物：
  - 第六波复盘记录
  - 第六波遗留问题清单

第六波退出条件：

- `config-change-rules` 已能独立表达配置与默认值边界
- `database-schema-rules` 与 `database-query-rules` 已清楚分离
- `request-header-rules` 已能稳定表达 Header 规则，不与请求参数规则混淆
- `auth-security-rules` 已能独立覆盖安全增强场景
- `performance-caching-rules` 已能稳定区分性能优化与过度优化
- `integration-debugging-rules` 已能独立表达联调增强场景
- 第六波样板已能作为剩余增强类 skill 的模板

当第六波退出条件满足后，整套首版实施计划视为闭环完成，后续新增 skill 按 `skill-evolution-rules` 继续滚动演进。

### 5. 落地节奏建议

建议采用保守节奏：

- 每次只处理 1 个 skill
- 一个 skill 至少经过“初稿 -> 审查 -> 修订 -> 再审”两轮
- 每一波完成后，不立刻进入下一波，先复盘共性问题
- 如果某一波中发现模板本身有问题，先回修模板，再继续后续 skill
- 如果当前 skill 的边界仍不稳定，不允许以“先做下一个再回头修”的方式推进
- 如果当前波次的退出条件未满足，不允许提前进入下一波

### 6. 当前建议的第一步

如果正式开始落地，不建议一上来就建很多目录。

最稳的开始方式是：

1. 先把 `team-development-rules` 改造成正式的弱触发协调层
2. 同时把 `skill-evolution-rules` 做出来，作为后续体系维护规则
3. 然后只落地一个编码样板 skill：
   `code-minimal-change-rules`
4. 等这一套模板验证稳定后，再进入第二个 skill

补充说明：

- 这里的“做出来”优先指完成规划、结构、场景验证和审查，不默认等于立刻批量建目录
- 如果 `team-development-rules` 和 `skill-evolution-rules` 仍有争议，不能提前进入 `code-minimal-change-rules`
- `code-minimal-change-rules` 之所以作为第一个编码样板，是因为它对后续所有 skill 都有约束意义，适合作为编码主干模板的第一块

### 7. 当前实施态度

这套计划默认采用“慢工出细活”的策略：

- 不追求短时间铺满全部 skill
- 追求每个 skill 都能独立成立、独立维护、独立复用
- 追求每完成一个，就真的能成为后续 skill 的模板和参照物

## 十八、当前不执行的事项

在审查通过前，暂不执行以下动作：

- 不初始化新 skill 目录
- 不拆分现有 `team-development-rules`
- 不把 seeds 正式并入新 skill
- 不写正式 `references/` 细则
- 不做自动触发描述的最终定稿

## 十九、后续审查重点

后续审查建议重点看这几件事：

1. 领域划分是否合理
2. skill 是否拆得过细或还不够细
3. 每个 skill 的自动触发 description 是否足够具体
4. 总控层的冲突裁决规则是否足够明确，是否真的能处理多 skill 同时触发
5. 总控层的流程暂停、重启、终止权限边界是否清晰
6. 总控层是否已经足够“弱触发”，不会压制小独立 skill 的命中
7. 有没有遗漏的重要位点类 skill
8. 哪些 skill 应该先做，哪些可以后做
9. 哪些 rules 应该进 `SKILL.md`，哪些应该进 `references/`
10. Bug 域和需求域的交接边界是否清晰，尤其是“历史遗留缺陷”“需求变更引发的问题”“验收不通过”这三类临界场景
11. Bug 域是否已经充分覆盖运行时诊断、断点调试、debug 日志和断言定位
12. 编码审查域和测试域的边界是否清晰，尤其是“静态质量自审”和“功能正确性验证”是否已经彻底分开
13. 测试域内部的“策略、资源管理、功能验证、联调验证、回归验证”边界是否已经彻底分开
14. 编码完成后的自审 skill 是否还需要继续拆细
15. PR 评审和测试前自审是否已经区分清楚
16. 语法检查、引用清理和格式化规则是否还要按语言继续拆细
17. 统一归档目录约定是否合理，是否需要按领域再细化子目录
18. references 的拆分维度、命名规范和目录层级是否足够统一，是否便于长期维护
19. 串行实施计划是否合理，是否需要调整各波次顺序
20. 每个 skill 的完成判定是否足够严格，能否防止半成品进入下一阶段

## 二十、当前结论

当前整体方向已经确定为：

- 不再依赖手动拼 skill
- 采用全流程自动触发设计
- 采用“阶段 + 问题类型 + 代码位点”的混合触发模型
- 小独立 skill 是主执行层
- 总控层是弱触发的流程协调、阶段分析与备用路由层
- 总控层负责多 skill 冲突裁决与流程中断管控，但不替代小 skill 执行
- Bug 域采用“静态定位 + 运行时诊断”的双路径设计
- 明确引入“编码完成后、测试前”的编码审查阶段
- 正式落地采用“串行、安全、单 skill 完整验收后再进入下一个”的实施策略
- 允许长期演进和持续拆分
