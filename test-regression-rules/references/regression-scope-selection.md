# 回归范围判定与用例选取

## 用途

用于回答“这次改动之后，哪些旧能力必须回归验证”。

## 常见回归触发类型

- Bug 修复：重点验证原问题路径、相邻路径、同类边界条件。
- 原有功能迭代：重点验证旧入口、旧交互、旧返回和兼容行为。
- 公共模块或共享逻辑修改：重点验证所有主要调用方和关键上下游链路。
- 接口兼容性变化：重点验证老参数、老字段、老客户端或旧流程是否仍可用。

## 选取原则

- 先覆盖与改动点直接相连的旧功能。
- 再覆盖最容易被共享逻辑波及的关键链路。
- 对公共能力修改，优先选取代表性调用方，而不是盲目全量展开。
- 对未覆盖项必须说明原因，不用“默认没问题”代替回归结论。

## 检查清单

- 是否已识别直接影响点和共享依赖
- 是否覆盖了最关键的旧路径和兼容场景
- 是否说明了未覆盖项和剩余风险
- 是否能回答“有没有带坏别的地方”
