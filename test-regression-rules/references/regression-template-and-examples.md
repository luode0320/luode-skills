# 回归结论模板与样例

结论：本模板用于说明改动后是否仍保持既有关键行为；影响：相关人员可判断回归风险和是否需要继续处理；范围：覆盖改动类型、用例选择、结果、风险和证据；非范围：不把环境异常或未执行范围误判为产品回归；变化：先说明业务结论，再把运行和追踪细节归入附录；完成标准：已覆盖范围、未覆盖风险和结论依据均可复核；术语说明：无技术术语需要解释；验证状态：回归执行后以 local 环境的真实结果确认。

## 结论模板

主结论写入 `doc/5-tests/yyyy-MM-DD_HHmmss/<测试任务中文主题>/README.md`，至少包含：

- 改动类型：
- 关联需求 / Bug：
- 回归范围：
- 选取用例：
- 执行环境：
- 实际结果：
- 未覆盖项 / 风险项：
- 详细证据路径：
- 结论：通过 / 驳回 / 待补测
- 必要流转动作（仅驳回、待补测、存在阻断风险或用户明确要求建议时填写；回归闭环且无必需动作时不写）：

## 正例

- 修复订单超时 Bug 后，除了验证修复路径，还回归同类重试链路和主要下游通知流程，并在 README 中写明详细案例位于 `doc/5-tests/2026-03-28_180000/internal/service/order_timeout_regression.md`。
- 修改公共鉴权中间件后，选取主要调用方、不同权限角色和关键异常路径做兼容验证，执行日志放到对应的 ASCII 镜像路径目录中。
- 调整页面公共组件后，回归主要使用页面和关键交互，不只验证当前页面本身。

## 反例

- 只验证当前新功能通过，就直接宣称没有回归风险。
- 公共模块改动后只测一个调用方，却不给未覆盖范围任何说明。
- 联调环境异常导致链路失败，却直接判定为回归失败。
- 用“时间不够”跳过关键旧路径，但没有记录待补测项和风险。
- 把回归结论单独扔到 `testing/`、`analysis/`、仓库根目录或中文说明目录之外的错误位置，导致和当前时间戳任务目录脱离。

## 执行附录

- local 环境、选取用例、精确命令、样本、运行日志、失败项、清理方式和人工复核步骤。
- 环境异常与实现回归的区分依据，避免用错误归因替代真实验证。

## 追踪附录

- 来源对象、测试 ID、详细证据路径、风险项、未覆盖项和机器可读定位。
- 附录维护规则：执行附录与追踪附录连续置于文档末尾，后续正文不得插入两者之间或其后。
