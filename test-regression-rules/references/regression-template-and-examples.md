# 回归结论模板与样例

## 结论模板

主结论写入 `test/yyyy-MM-DD_HHmmss/测试任务中文简介/README.md`，至少包含：

- 改动类型：
- 关联需求 / Bug：
- 回归范围：
- 选取用例：
- 执行环境：
- 实际结果：
- 未覆盖项 / 风险项：
- 详细证据路径：
- 结论：通过 / 驳回 / 待补测
- 下一步流转建议：

## 正例

- 修复订单超时 Bug 后，除了验证修复路径，还回归同类重试链路和主要下游通知流程，并在 README 中写明详细案例位于 `test/2026-03-28_180000/internal/service/order_timeout_regression.md`。
- 修改公共鉴权中间件后，选取主要调用方、不同权限角色和关键异常路径做兼容验证，执行日志放到对应的 ASCII 镜像路径目录中。
- 调整页面公共组件后，回归主要使用页面和关键交互，不只验证当前页面本身。

## 反例

- 只验证当前新功能通过，就直接宣称没有回归风险。
- 公共模块改动后只测一个调用方，却不给未覆盖范围任何说明。
- 联调环境异常导致链路失败，却直接判定为回归失败。
- 用“时间不够”跳过关键旧路径，但没有记录待补测项和风险。
- 把回归结论单独扔到 `testing/`、`analysis/`、仓库根目录或中文说明目录之外的错误位置，导致和当前时间戳任务目录脱离。
