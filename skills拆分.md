# Skill 拆分 Skill 设计（功能不删减版）

更新时间：2026-04-02 22:13:57

## 1. 目标

定义一个独立 skill（建议名：`skill-split-preserve-rules`）：

- 当任意已有 skill 被更新、优化、补充时自动触发。
- 在“原有功能不删减”的前提下，把目标 skill 拆分为多个独立 skill。
- 让拆分结果可验证、可回滚、可追溯，避免“拆完更轻但能力丢失”。

## 2. 触发条件

满足任一条件即触发本 skill：

- 对某个现有 skill 做“新增规则/补充规则/优化规则”。
- 发现某个 skill 的职责混合、触发歧义、正文过长，准备拆分。
- 需要把一个 skill 的多类职责拆成多个平级独立 skill。

不触发场景：

- 仅修正错别字、排版、链接等非功能编辑。
- 仅调整语句表达但不改变触发、流程、边界、验收标准。

### 2.1 触发门槛（建议写死）

为了避免“所有有效更新都命中拆分 skill”，建议在自动触发前再增加一道进入门槛：

- 只有当目标 skill 同时满足“本轮发生功能性变化”与“原边界已难以自然承接新变化”时，才进入拆分。
- 至少命中以下任一结构信号，才建议正式启动拆分：
  1. 同一 skill 已出现两个以上可独立命中的并列职责组。
  2. `description` 已同时覆盖多个不同触发对象，继续补规则会放大触发歧义。
  3. `SKILL.md` 与 `references/` 已明显膨胀，新增规则难以继续在原结构中清晰归位。
  4. 本轮新增规则若继续留在原 skill，会让“不要用它代替...”边界声明失效或明显变弱。

### 2.2 需要暂停而不是立刻拆分的场景

- 当前只是感到文档“有点长”，但还无法指出独立职责组。
- 当前 skill 仍主要围绕同一触发对象，只是例子变多、细则变多。
- 本轮只是信息补充，还没有形成新的独立类别。
- 当前团队尚未确认旧 skill 是否允许在拆分后直接删除，直接拆分会造成删除策略不一致。

## 3. 核心约束（必须）

本 skill 的第一原则：**功能不删减**。
第二原则：**原规则零丢失**。
拆分策略第一优先级：**按分类二分拆分（每次拆一半类别）**。

拆分过程中必须满足：

- 原 skill 的每一条可执行规则，都必须在新 skill 集合中找到落点。
- 原 skill 的每一个“必须/默认/先做/阻断/通过标准”语句，不得丢失。
- 原 skill 中任何看似奇怪、低频、历史兼容、特殊路径规则，默认都按“特例规则”处理并强制保留，绝对不能丢弃。
- 原 skill 的 references/scripts/assets 中有用内容，不得静默丢弃。
- 原 skill 的边界声明（不要用它代替...）必须被继承或显式重写。
- 不允许用“由经验补齐”替代显式规则迁移。
- 新 skill 可以补充和增强规则，但不得删除、弱化或覆盖原规则约束强度。
- 只要当前规则仍可按“分类集合”拆成两组，就必须优先按分类二分，不得直接跳到其它维度拆分。

## 3.1 分类二分拆分策略（强优先级）

- 拆分时优先识别“并列类别集合”（如目录/文件/脚本/验证程序/fixture/mock/文档落点）。
- 每次拆分只做一次二分：把类别集合拆成 A/B 两组，尽量接近一半。
- 若某一组后续仍过长或职责仍混合，继续对该组递归执行“分类二分”。
- 只有在“单类触发已无法再按分类二分”时，才允许降级到更细维度（如新增 vs 修改、前置条件、路径层级）。
- 禁止在仍可分类二分时直接做过细拆分，避免一次性碎片化。

### 3.1.1 分类识别清单（建议写死）

只有当候选类别同时满足以下大部分条件时，才视为可用于二分的“并列类别”：

- 能自然并列罗列，而不是必须依赖顺序理解。
- 每个类别都能独立改写成“当...时触发”的触发描述。
- 各类别共享同一上位目标，但处理对象不同。
- 拆开后仍能分别执行、分别验证，而不是拆成互相等待的伪子任务。
- 若删除其中一类，其他类别仍成立，不会破坏整体语义闭环。

若某组规则只是流程步骤先后不同、但处理对象没有变化，默认不视为并列类别，优先保持在同一 skill 内。

### 3.1.2 停止继续拆分条件（建议写死）

出现以下任一情况时，应停止继续递归拆分：

- 当前候选组已无法再识别出新的并列类别集合。
- 再拆只会形成语义差异极小的新 skill，例如只换了近义词或局部例子。
- 拆分后新 skill 必须依赖“父 skill 上下文”才能理解职责。
- 拆分后同类任务平均命中 skill 数预计会稳定超过 3。
- 新 skill 无法形成独立的通过标准、阻断条件或边界声明。

### 3.1.3 分类二分示例（test-location 场景）

原触发：
- 当新增或修改测试目录、测试文件、测试脚本、验证程序、fixture、mock 数据、测试说明文档落点时触发。

第一次拆分（按分类二分）：
1. 当新增或修改测试目录、测试文件、测试脚本时触发。
2. 当新增或修改验证程序、fixture、mock 数据、测试说明文档落点时触发。

第二次拆分（仍按分类优先；若分类不可再拆，再降级维度）：
- 对“当新增或修改测试目录、测试文件、测试脚本时触发”继续拆分，可得到：
1. 当新增测试目录、测试文件、测试脚本时触发。
2. 当修改测试目录、测试文件、测试脚本时触发。

## 3.2 特例规则保护（强制）

- 任何“不符合直觉”的原规则，在未获得明确废弃指令前，一律视为有效特例。
- 特例规则必须在映射表标记 `特例=是`，并给出落点。
- 若确需移除某条原规则，必须有明确的“废弃决议来源”；无决议时禁止删除。

## 4. 输入与输出

输入：

- 被拆分 skill 目录（例如 `xxx-rules/`）。
- 本次更新/优化/补充需求说明。

输出：

1. 拆分后的独立 skill 列表（平级命名，不走父子链，名称根据新 skill 的 `description` 命名）。
2. 每个新 skill 的职责边界与触发描述。
3. 功能覆盖映射表（原规则 -> 新 skill 落点）。
4. 特例规则保护说明（包含所有“看似奇怪规则”的迁移去向）。
5. 迁移结果与删除策略（旧 skill 何时删除、删除前如何校验、删除后由谁承接）。

## 5. 标准流程

1. 基线采集
- 读取目标 skill 的 `SKILL.md`、`description`、`references/`、`scripts/`、`assets/`、`agents/openai.yaml`。
- 抽取“完整规则清单”和“资源清单”，包含必须、建议、示例、注释性约束与历史兼容说明。
- 为每条原规则分配唯一规则 ID，避免迁移时漏项。

2. 规则原子化
- 把原规则按触发对象拆成最小职责单元。
- 标记每条规则的优先级（必须/建议/说明）。
- 建议补充原子化口径：
  - 每个包含“必须/默认/先做/阻断/通过标准/不得”的句子，默认拆成单独规则条目。
  - 一条语句如果同时包含“主规则 + 例外 + 历史兼容说明”，应拆成至少两条：主规则条目、例外或兼容条目。
  - 普通示例默认不计入主规则总数；若示例本身承载兼容语义或特例语义，则必须单独入表。
  - 规则原子化后应能独立回答三个问题：何时触发、要做什么、是否有例外。

3. 拆分设计
- 按“单一触发对象”把原子规则分组。
- 优先按“分类集合二分”生成两组候选，单次只拆一半类别。
- 对仍过长的候选组递归执行分类二分；无法再二分时再切换更细维度。
- 形成多个独立 skill 候选，保证职责不重叠、边界可判定。

4. 独立化落地
- 为每个新 skill 生成完整结构：`SKILL.md` + 按需 `references/` + 按需 `scripts/`/`assets/`。
- description 统一写法：`当...时触发` + `负责...` + `不要用它代替...`。

5. 覆盖校验
- 生成“功能覆盖映射表”，逐条核对原规则是否 100% 覆盖。
- 任一原规则无落点，则判定拆分失败并阻断提交。
- 若某条规则被改写，必须给出“等价或增强证明”；不能证明则回退为原规则表达。
- 覆盖映射完成后仍不算拆分完成，必须继续进入详细、多轮、不同模式的测试验证。
- 测试至少覆盖两类核心问题：
  1. 新 skill 集合对旧 skill 的任何规则都没有丢失。
  2. 新 skill 集合仍能自动被触发，而不是写出来后无法自动命中。
- 建议补充覆盖口径：
  - 原规则 ID 建议按章节顺序生成，如 `R-3.1-001`。
  - 一条原规则允许映射到多个新 skill，但必须指定一个“主落点”。
  - 若新 skill 通过多条规则共同覆盖一条原规则，必须写明“组合覆盖说明”。
  - 只要触发条件、约束强度或例外范围发生变化，就必须补“等价性说明”，不能只写“已覆盖”。

6. 删除收口
- 旧 skill 不进入长期兼容期；当新 skill 集合完成覆盖校验后，旧 skill 直接删除。
- 删除前必须写清：原规则已由哪些新 skill 承接、删除后不再依赖旧 skill 命中。
- 删除动作必须晚于覆盖率 100% 校验，禁止“先删再补”。
- 删除动作必须晚于多轮多模式测试通过，禁止“先删再测”。

### 5.1 旧 skill 删除状态机（建议写死）

建议把旧 skill 的生命周期收口为固定三态：

- `active`：原 skill 正常使用，尚未开始拆分。
- `splitting`：开始拆分，允许补拆分材料与迁移说明，禁止继续混入无关新规则。
- `deleted`：覆盖校验通过后删除旧 skill，由新 skill 集合完全承接。

状态切换建议：

- `active -> splitting`：已确认存在独立类别集合，且本轮要正式启动拆分。
- `splitting -> deleted`：覆盖映射表 100% 完成、多轮多模式测试通过，删除前检查通过，旧 skill 不再承担任何独有规则价值。

### 5.2 删除前承接检查模板（建议写死）

由于旧 skill 会被直接删除，删除前必须显式写清“删掉以后由谁承接”。建议至少固定以下模板字段：

| 字段 | 说明 |
|---|---|
| 旧 skill 名称 | 当前准备删除的 skill 名称 |
| 当前状态 | 固定写 `splitting` 或 `deleted` 前检查 |
| 主承接新 skill | 当前主要承接原命中的新 skill |
| 次承接新 skill | 仅在原规则被拆到多个 skill 时填写 |
| 删除前保留内容 | 删除前临时保留的迁移说明或校验材料 |
| 删除后承接方式 | 删除旧 skill 后如何由新 skill 集合承接 |
| 删除动作 | 何时删除目录、何时删除引用 |
| 删除前检查 | 删除前必须满足的条件 |

建议正文模板：

```text
删除前承接检查：
- 旧 skill：`<old-skill>`
- 当前状态：`splitting`
- 主承接新 skill：`<new-skill-a>`
- 次承接新 skill：`<new-skill-b>`（如无可省略）
- 删除前临时保留：迁移说明、覆盖校验材料
- 删除前测试材料：静态覆盖对照、自动触发验证、真实场景演练
- 删除后承接方式：
  1. 原旧 skill 覆盖的每条规则都必须能在映射表中定位到新 skill。
  2. 若存在多个可能落点，则按映射表中的主落点承接。
  3. 若仍无法判定唯一承接关系，则阻断删除并回到拆分设计阶段补齐映射。
- 删除动作：
  1. 覆盖校验通过后删除旧 skill 目录。
  2. 删除旧 skill 在索引、引用或说明中的主入口位置。
- 删除前检查：
  1. 覆盖映射表 100% 完成。
  2. 多轮多模式测试已完成，且未发现旧规则丢失、弱化或承接空洞。
  3. 新 skill 自动触发验证通过，边界已稳定。
  4. 删除后不再依赖旧 skill 作为命中入口。
  5. 原特例规则均已在新 skill 中找到落点。
```

若删除前无法写清“删除后由谁承接”，则不允许删除旧 skill。

## 6. 功能覆盖映射表（强制）

每次拆分必须产出映射表，建议格式：

| 原 skill 规则ID | 原规则摘要 | 特例标记 | 新 skill 名称 | 新规则落点（文件/章节） | 迁移动作 | 状态 |
|---|---|---|---|---|---|---|
| R-001 | ... | 否/是 | ... | ... | 原样迁移/等价改写/增强补充 | 已覆盖 |

强制要求：

- 覆盖率必须为 100%。
- 状态不得出现“待补充”进入提交阶段。
- 若规则被重写，必须给出“等价性说明”。
- 映射统计口径以“原规则条目总数”为准，不允许合并条目来虚增覆盖率。
- 每条标记为特例的规则都必须有落点，不允许“按常规合并省略”。
- 若一个原规则映射到多个新 skill，必须额外标记“主落点”。
- 若原规则被多条新规则组合覆盖，必须给出组合覆盖说明，不允许只写“已覆盖”。
- 映射表完成后仍不算拆分完成，必须继续进入多轮多模式测试。

### 6.2 多轮多模式测试（强制）

拆分后的新 skill 集合必须经过详细测试，至少包含：

1. 静态覆盖对照
  - 用映射表逐条确认旧规则没有丢失。
2. 边界与自动触发验证
  - 用真实或拟真提问验证新 skill 能自动命中。
3. 真实场景演练
  - 用真实 skill 样本、真实仓库或高保真场景做端到端验证。

测试至少要回答四个问题：

- 是否存在旧规则丢失
- 是否存在约束弱化
- 是否存在承接空洞或歧义
- 是否存在自动触发失效

只要任一问题回答为“是”，就不得删除旧 skill。

### 6.1 映射表填写模板（建议写死）

建议把映射表字段解释一并写死，减少执行时的主观理解：

| 字段 | 填写要求 |
|---|---|
| 原 skill 规则ID | 必填，按原文档章节顺序唯一编号，例如 `R-3.1-001` |
| 原规则摘要 | 必填，保留原意，不得在摘要阶段偷改语义 |
| 特例标记 | 必填，只能写 `是` / `否` |
| 新 skill 名称 | 必填，可写一个或多个；多个时需在备注中指出主落点 |
| 新规则落点（文件/章节） | 必填，写到具体文件和章节，不接受“见新 skill”这类笼统描述 |
| 迁移动作 | 必填，只能写 `原样迁移` / `等价改写` / `增强补充` |
| 状态 | 必填，只能写 `已覆盖` / `阻断`，禁止写 `待补充` 进入提交阶段 |
| 主落点 | 多 skill 映射时必填，单 skill 映射时建议也写明 |
| 等价性说明 | 只要不是“原样迁移”就必须填写 |
| 组合覆盖说明 | 只有一条原规则被多条新规则共同覆盖时填写 |

建议最小示例：

| 原 skill 规则ID | 原规则摘要 | 特例标记 | 新 skill 名称 | 新规则落点（文件/章节） | 迁移动作 | 状态 | 主落点 | 等价性说明 | 组合覆盖说明 |
|---|---|---|---|---|---|---|---|---|---|
| R-3.1-001 | 当新增或修改测试目录、测试文件、测试脚本时触发 | 否 | `test-location-core-rules` | `SKILL.md / 触发条件` | 原样迁移 | 已覆盖 | `test-location-core-rules` | 无 | 无 |
| R-3.2-004 | 历史兼容目录规则必须保留 | 是 | `test-location-compat-rules`、`test-location-core-rules` | `references/compat.md / 历史兼容规则` | 等价改写 | 已覆盖 | `test-location-compat-rules` | 约束强度不变，仅将兼容入口与主规则分离 | 主规则由 core 承接，兼容入口由 compat 承接 |

填写约束：

- `原样迁移`：语义、触发、约束强度均不变。
- `等价改写`：表达位置或组织方式变化，但语义和约束强度不变。
- `增强补充`：在保留原规则的前提下新增更细口径，不能弱化原约束。
- 若无法判断属于哪一类，默认按“阻断”处理，而不是先写“已覆盖”。

## 7. 命名规则（独立 skill）

- 禁止父子序号命名（例如 `xxx-rules-10-yyy`）。
- 新拆分 skill 的名称必须根据该新 skill 自己的 `description` 命名，而不是沿用旧 skill 名字做机械裁剪。
- 名称应可直接体现职责，无需依赖“父 skill 上下文”。

### 7.1 命名生成原则（建议写死）

新 skill 名称应从其 `description` 中抽取两个核心信息：

1. 新 skill 主要在“当什么对象 / 什么位点 / 什么场景发生变化时触发”
2. 新 skill “负责什么核心职责”

推荐做法：

- 先写清新 skill 的 `description`，再反推名称。
- 名称优先取 `description` 中最稳定的“触发对象 + 职责焦点”。
- 触发对象应优先使用最具体、最可区分的名词，而不是泛词。
- 职责焦点应优先使用能区分相邻 skill 的词，而不是过宽词。
- 最终名称仍保持 kebab-case，并以 `-rules` 结尾。

### 7.2 命名步骤（建议写死）

1. 先产出新 skill 的完整 `description`。
2. 从 `description` 中提取“主触发对象”。
3. 再提取“主职责焦点”。
4. 以“主触发对象 + 主职责焦点 + rules”生成候选名称。
5. 若出现多个候选名称，优先选择与 `description` 语义最贴近、且最不依赖旧 skill 上下文的名称。

推荐格式：

- `<trigger-object>-<focus>-rules`

这里的 `<trigger-object>` 与 `<focus>` 不是预设字段，而是从新 skill 的 `description` 中反推出的稳定语义。

### 7.3 命名约束（建议写死）

- 禁止仅因为原 skill 名称好记，就直接沿用原 skill 前缀再硬拼一个后缀。
- 禁止名称与新 skill 的 `description` 脱节，导致“看名字像 A，description 实际是 B”。
- 禁止为了追求统一而使用过宽名称，例如 `common-rules`、`misc-rules`、`other-rules`。
- 若 `description` 变化到已经改变主触发对象或主职责焦点，应重新评估 skill 名称，而不是默认沿用旧名。
- 若多个新 skill 的 `description` 高度相似到无法稳定命名，优先回退检查拆分是否过细，而不是强行造多个近义名称。

### 7.4 命名示例（建议写死）

示例一：

- 新 skill description：当新增或修改测试目录、测试文件、测试脚本落点时触发。负责统一测试基础落点规则；不要用它代替验证程序、fixture、mock 或测试说明文档落点规则。
- 推荐名称：`test-location-core-rules`

示例二：

- 新 skill description：当新增或修改验证程序、fixture、mock 数据落点时触发。负责统一测试支撑资源落点规则；不要用它代替测试目录、测试文件或测试脚本落点规则。
- 推荐名称：`test-support-location-rules`

示例三：

- 新 skill description：当旧 skill 拆分完成、准备删除旧 skill 前触发。负责核对删除前承接关系与删除条件；不要用它代替覆盖映射校验本身。
- 推荐名称：`skill-retirement-check-rules`

## 8. 验收标准

拆分完成判定必须同时满足：

- 新增 skill 全部可独立触发、独立执行。
- 原 skill 可执行规则覆盖率 100%。
- 原 skill 完整规则（含特例）覆盖率 100%。
- 拆分路径符合“分类二分优先”，且每轮拆分都有分组依据记录。
- 原资源迁移完成，无悬空引用。
- 新 skill 边界清晰，互不替代。
- 同类任务平均命中 skill 数稳定在 2~3（防止过度拆分）。

## 9. 阻断条件

出现以下任一情况，禁止收口：

- 发现原规则无新落点（覆盖率 < 100%）。
- 发现任何特例规则缺失、被弱化或被“常规化吞并”。
- 在仍可按分类二分时，直接跳过分类拆分进入过细维度拆分。
- 发现关键 references/scripts 被遗漏。
- 新 skill 之间职责冲突，无法判定触发边界。
- 拆分完成后旧 skill 仍未删除，且没有明确阻断原因。
- 缺少类别识别依据，无法说明为什么这几组算“并列类别”。
- 缺少停止递归条件，导致拆分可能无限细化或明显过拆。
- 映射表未标明主落点、等价性说明或组合覆盖说明。
- 删除前检查未完成，却先删除旧 skill。
- 多轮多模式测试未完成，却先宣布拆分完成或删除旧 skill。
- 测试发现旧规则丢失、弱化或自动触发失败，却未回滚修正。

## 10. 回滚策略

- 若覆盖校验不通过，回滚到“拆分前基线版本”。
- 先补齐映射缺口，再重新执行拆分流程。
- 回滚后保留问题清单，避免重复遗漏。

## 11. 推荐的 Skill 描述草案（可直接用于新 skill）

```text
当新增、优化或补充任意现有 skill 内容，且该 skill 存在多职责混合或触发歧义、需要拆分为多个独立 skill 时触发。负责在原有功能不删减前提下优先执行“分类二分拆分（每次拆一半类别）”、递归拆分、资源迁移与覆盖校验，并输出原规则到新 skill 的 100% 映射关系；任何看似奇怪的原规则默认按特例保留且必须有落点，不得删除；不要用它代替业务需求分析、Bug 定位或具体功能实现。
```
