# 最小改动总则

## 用途

用于定义哪些改动属于本次任务的最小必要范围，哪些改动属于应当被排除的无关修改。

## 允许的改动

- 直接完成当前需求所需的最小实现。
- 直接修复当前 Bug 所需的最小修正。
- 为了让本次改动可运行、可编译、可验证而必须补的最小支撑代码。
- 与当前改动直接耦合、否则会立刻出错的最小连带调整。

## 应排除的改动

- 与当前目标无关的历史问题修复。
- 与当前目标无关的大范围重构。
- 与当前目标无关的统一命名、统一格式、统一风格清理。
- 与当前目标无关的性能优化、缓存优化、结构优化。
- 与当前目标无关，或不能证明真实收益的封装、抽象、helper 拆分、wrapper 包装、factory / manager / adapter 层。
- 不能列出现有或本轮即将落地第二实现、稳定跨边界契约或真实替换需求的接口抽象。
- 对“最近修改超过7天且被多处复用”的通用代码直接改旧行为（应优先新增兼容能力）。

## 默认原则

1. 能不改的旧代码，不改。
2. 能局部修正的，不扩大到整层或整模块。
3. 能新增小范围代码解决的，不先改大面积旧逻辑。
4. 能直写清楚的代码，不默认封装；封装必须先证明收益。
5. 能用具体类型或具体函数表达的，不先抽接口；接口必须先证明真实第二实现或稳定边界。
6. 本次不处理的问题，记录并留给后续独立任务。

## 编码前先思考（并入 Karpathy 准则）

- 开始实现前先写明关键假设，避免带着“想当然”直接落代码。
- 如果需求存在多种合理解释，先收敛理解再编码，不默选其一。
- 若发现更简单方案，应优先采用更简单方案并说明取舍。
- 若有关键不确定项，先暂停澄清，不得用临时猜测硬推进。

## 简单优先（并入 Karpathy 准则）

- 用能解决问题的最少代码，不做猜测性扩展。
- 不为一次性改动提前抽象“未来能力”。
- 对简单处理不要顺手引入接口层、实现体拆分或无明确收益的依赖注入包装。
- 大多数代码默认不需要封装。新增封装前先回答：是否已有多个真实调用方、是否隔离了真实边界、是否让主流程更短更清楚、是否降低了测试或维护成本；任一答案都说不清时，保持直写。
- 大多数代码默认不需要抽象接口。抽接口前先回答：是否已有真实第二实现、是否存在插件 / 外部适配边界、是否有稳定跨模块契约、是否能显著降低测试替身成本；答不上来时，使用具体实现。
- 不把“看起来更工程化”“以后可能复用”“先留扩展点”当成封装理由。
- 不把“面向接口编程”“以后可能会替换”“方便 mock”当成单实现接口的充分理由。
- 不引入未被需求要求的可配置性或分支复杂度。
- 若发现实现明显过度，应先收缩再提交。

## 目标驱动验证（并入 Karpathy 准则）

- 每次改动都必须绑定一个可验证目标，而不是“看起来差不多”。
- 推荐按“步骤 -> 检查项”执行最小闭环，避免完成定义模糊。
- 结束前复核：每一行主要改动都能追溯到当前目标。
