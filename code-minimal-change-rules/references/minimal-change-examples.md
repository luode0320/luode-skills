# 最小改动正反例

## 正例

### 正例 1：修复接口参数校验

- 当前目标：修复某接口缺少必填参数校验。
- 合理改动：补参数校验、补必要错误返回、补对应最小测试。
- 不需要顺手做：重构整个 controller、统一所有 DTO 命名。

### 正例 2：修复查询 Bug

- 当前目标：修复一条 SQL 条件写错的问题。
- 合理改动：修正 SQL、补充最小验证。
- 不需要顺手做：把整个 Repository 全部改写成另一种风格。

### 正例 3：直写简单流程

- 当前目标：新增一个只有当前入口使用的三步数据转换。
- 合理改动：在当前方法或同文件局部函数中直接写清转换步骤，并补最小验证。
- 不需要顺手做：提前抽 `TransformerFactory`、`ConvertManager`、`IConverter` 或单独 wrapper 文件。

## 反例

### 反例 1：借修 Bug 顺手重构模块

- 当前目标：修复一个空指针异常。
- 越界行为：顺手把整段服务逻辑重构成另一套分层。
- 结论：驳回，另立重构任务。

### 反例 2：借新增功能顺手统一风格

- 当前目标：新增一个接口。
- 越界行为：顺手把同目录几十个文件全部重新格式化、改命名、改注释风格。
- 结论：驳回，除非这些改动直接阻碍当前功能落地。

### 反例 3：把简单代码包装成“看起来规范”的封装

- 当前目标：在一个现有流程中补一个简单判断和一次字段组装。
- 越界行为：新增 `Helper`、`Manager`、`Factory`、`Adapter`、接口和实现体，只为承接这几行逻辑。
- 结论：驳回。没有多个真实调用方、边界隔离或复杂度下降证据时，保持直写。

### 反例 4：为单实现提前抽象接口

- 当前目标：接入一个当前只有一种实现的数据读取流程。
- 越界行为：新增 `Reader` 接口、`DefaultReader` 实现体和装配层，只因为“以后可能替换”或“方便 mock”。
- 结论：驳回。列不出现有或本轮即将落地的真实第二实现时，优先使用具体类型或具体函数。
