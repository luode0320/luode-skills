# 最小改动边界

## 用途

用于区分最小改动原则与相邻编码 skill 的职责边界。

## 与可读性规则的边界

- 当前 skill 只判断“该不该改这么多”。
- 可读性规则才判断“写法是否清晰、结构是否自然”。
- 如果只是最近改动代码里少量表达层冗余，可在当前范围内顺手收口；如果已经扩散到历史代码大面积“优化表达”，仍然属于越界。
- 如果为了“可读性”新增封装，必须先证明它减少了真实复杂度；只是把三五行直白逻辑搬到 helper / wrapper / manager 里，通常不是可读性优化，而是跳转成本。

## 与封装抽象的边界

- 默认立场是“不封装，除非有证据”。封装证据包括多个真实调用方、明确跨边界隔离、复杂流程被拆成稳定阶段、或存在多个真实实现。
- 单实现、单调用、局部短流程、一次性适配、临时整理，不足以触发新增接口、factory、manager、adapter 或 wrapper。
- 抽接口前必须列出现有或本轮即将落地的多个实现、调用边界和替换场景；列不出时使用具体类型、具体函数或局部实现。
- 封装后如果只是多了命名、多了文件、多了跳转，但主流程没有变清楚，应判定为过度封装并收回。

## 与风格一致性规则的边界

- 当前 skill 不允许借机做大面积风格统一。
- 风格一致性规则只在当前必要改动范围内约束写法，不扩展范围。

## 与代码归位规则的边界

- 如果代码位置明显错误，允许做最小必要归位。
- 如果只是想顺手整理整个目录结构，不属于当前 skill 的允许范围。

## 与测试支撑的边界

- 为了验证本次改动而补的最小测试支撑代码可以接受。
- 借机重写整套测试目录、测试框架或历史测试资产，不属于最小改动。
- 不要为了“方便 mock”默认抽接口；只有真实测试替身能显著降低验证成本，且不引入额外结构复杂度时，才允许作为封装理由。
