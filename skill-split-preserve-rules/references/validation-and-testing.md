# 多轮多模式测试验证

## 1. 为什么测试是拆分完成的强制条件

拆分 skill 不是“新目录写完就算完成”。
只有当新 skill 集合经过详细、多轮、不同模式的测试后，才能证明拆分真的成立。

测试的核心目标只有两个，且都属于阻断级：

1. 新 skill 集合对旧 skill 的任何规则都没有丢失。
2. 新 skill 集合仍能在真实提问中被自动触发，而不是只存在于目录中。

只要任一目标未通过，就不得宣告拆分完成，更不得删除旧 skill。

## 2. 测试总原则

- 测试轮次必须不少于 2 轮，推荐至少 3 轮。
- 测试模式必须不同，禁止只换几个文案做同一类测试。
- 每轮测试都要写清：测试目的、输入、期望命中、实际结论、是否通过。
- 若当前是首个实战样本，默认允许保留旧 skill 作为冻结对照基线，直到测试完成。
- 测试结论必须能直接回答：
  - 是否存在旧规则丢失
  - 是否存在承接弱化
  - 是否存在自动触发失效
  - 是否存在触发边界漂移

## 3. 必测主题

### 3.1 旧规则零丢失验证

至少覆盖：

- 原规则逐条映射后，是否都能在新 skill 集合中找到落点。
- 原 skill 的低频特例、兼容规则、看起来奇怪的规则是否仍能找到落点。
- 若一条原规则需要多个新 skill 组合承接，组合后是否仍与旧约束强度等价。
- 是否存在“旧 skill 能回答，新 skill 集合反而回答不了”的空洞。

判定口径：

- 发现任一原规则无落点，立即判定失败。
- 发现原规则被弱化、放宽、改写成更模糊表述，也视为失败。

### 3.2 自动触发验证

至少覆盖：

- 新 skill 的 `description` 是否能支撑自动命中。
- 同一类真实提问下，新 skill 是否能被正确命中。
- 拆分后是否出现“应该命中的新 skill 没命中”。
- 是否出现“命中了错误 skill，而正确 skill 没命中”的边界漂移。

判定口径：

- 新 skill 需要在真实或拟真提问中稳定自动命中。
- 若必须依赖额外口头补充才能命中，视为触发设计未完成。

## 4. 推荐测试轮次

### 第 1 轮：静态覆盖对照

目的：

- 用映射表和原规则清单验证零丢失。

最少要做：

- 逐条检查原规则 ID 是否都有承接落点。
- 检查特例规则和 resources 是否都已迁移。
- 标出所有“多新 skill 组合承接”的规则，确认主落点和组合覆盖说明完整。

### 第 2 轮：边界与路由验证

目的：

- 验证新 skill 是否真的能按 `description` 自动命中。

最少要做：

- 为每个新 skill 准备若干条“应命中”提问。
- 为相邻 skill 准备若干条“不得误命中”提问。
- 验证旧 skill 拆分后是否出现边界重叠、空洞或抢职责。

### 第 3 轮：真实场景演练

目的：

- 在真实任务或拟真高保真场景中验证拆分后的整体可用性。

最少要做：

- 用一个真实现有 skill 作为样本做端到端演练。
- 或使用一个真实仓库 / 真实场景 / 历史案例做完整试跑。
- 记录旧 skill 与新 skill 集合的结论差异，确认无丢失、无触发失效。

## 5. 测试文档最小字段

每一轮测试至少记录：

- 测试轮次
- 测试模式
- 测试目标
- 输入样本
- 旧 skill 预期结论
- 新 skill 集合预期结论
- 实际结论
- 是否通过
- 发现的问题
- 回到哪一步修正

## 6. 删除前测试闸门

旧 skill 进入 `deleted` 前，测试必须至少证明：

1. 原规则 100% 可追溯到新 skill 集合。
2. 不存在旧规则丢失、弱化或承接空洞。
3. 新 skill 在真实提问中可以自动触发。
4. 新 skill 的命中边界没有明显漂移到相邻 skill。

若上述任一项无法证明，则：

- 旧 skill 不得删除。
- 当前拆分状态最多停留在 `splitting` 或 `comparing`。
- 必须回到拆分设计阶段补齐后重新测试。

## 7. 失败后的回流要求

- 若静态覆盖失败，回到规则原子化与映射阶段。
- 若自动触发失败，回到 `description`、边界声明和相邻 skill 路由阶段。
- 若真实场景演练失败，回到分类拆分设计阶段，重新评估是否拆得过细或职责分组错误。
- 不允许跳过失败轮次，直接用后续轮次覆盖前面的失败结论。
