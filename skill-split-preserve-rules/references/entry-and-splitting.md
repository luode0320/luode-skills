# 进入与拆分判定

## 1. 进入前先检查

只有当目标 skill 同时满足下面两类条件时，才进入拆分：

- 本轮发生的是功能性变化，而不是错别字、排版、链接或纯表述润色。
- 原边界已经难以自然承接新增规则，继续往原 skill 填内容只会放大触发歧义或职责混合。

至少再命中以下一个结构信号，才正式启动拆分：

- 同一 skill 已出现两个以上可独立命中的并列职责组。
- `description` 已明显同时覆盖多个触发对象。
- `SKILL.md` 与 `references/` 已膨胀到新增规则难以清晰归位。
- 若继续留在原 skill，会削弱“不要用它代替...”的边界声明。

## 2. 需要暂停而不是立刻拆分的场景

出现以下情况时，先暂停，不要强拆：

- 只是感觉文档有点长，但说不清独立职责组在哪里。
- 当前仍围绕同一触发对象，只是例子或细则变多。
- 本轮只是补信息，还没有形成新的独立类别。
- 当前任务并不允许拆完后删除旧 skill。

## 3. 分类识别清单

只有当候选类别同时满足以下大部分条件时，才视为可用于二分的“并列类别”：

- 能自然并列罗列，而不是必须按顺序理解。
- 每个类别都能独立改写成“当...时触发”。
- 各类别共享同一上位目标，但处理对象不同。
- 拆开后仍能分别执行、分别验证。
- 删除其中一类，其他类别仍成立，不会破坏整体闭环。

反向提醒：

- 只是流程步骤先后不同，但处理对象没有变化，默认不算并列类别。
- 只是同义改写、局部例子或轻微语气差异，默认不算独立类别。

## 4. 分类二分流程

1. 先列出所有候选类别，并给出每类的触发对象与职责摘要。
2. 只做一次二分，把类别集合拆成 A/B 两组，尽量接近一半。
3. 检查 A/B 两组是否都能形成独立触发描述、边界声明和验证闭环。
4. 只有某一组仍明显过载时，才继续对那一组递归执行“分类二分”。
5. 只有当前组已无法再按类别拆分时，才允许降级到“新增 / 修改”“前置 / 执行 / 收口”等更细维度。

## 5. 停止继续拆分条件

出现以下任一情况时，应停止继续递归拆分：

- 当前候选组已无法再识别新的并列类别集合。
- 再拆只会形成语义差异极小的新 skill。
- 新 skill 必须依赖“父 skill 上下文”才能理解职责。
- 拆分后同类任务平均命中 skill 数预计会稳定超过 3。
- 新 skill 无法形成独立的通过标准、阻断条件或边界声明。

## 6. 进入结果结论格式

进入结论至少要明确三件事：

- 这次是“进入拆分 / 暂停拆分 / 不触发拆分”哪一种结果。
- 触发拆分的结构信号是什么。
- 如果不进入拆分，当前规则为什么仍能留在原 skill 内自然承接。
