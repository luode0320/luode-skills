# 映射、承接与删除

## 1. 规则原子化口径

基线采集后，先把旧 skill 拆成最小可迁移规则条目：

- 每个包含“必须 / 默认 / 先做 / 阻断 / 通过标准 / 不得”的句子，默认单独成条。
- 一条语句如果同时包含“主规则 + 例外 + 历史兼容说明”，至少拆成两条。
- 普通示例默认不计入主规则总数；若示例承载兼容语义或特例语义，必须单独入表。
- 原子化后的每条规则都应能单独回答三个问题：何时触发、要做什么、是否有例外。

规则 ID 建议按章节顺序生成，例如 `R-3.1-001`。

## 2. 映射表字段口径

| 字段 | 填写要求 |
|---|---|
| 原 skill 规则ID | 必填，按原文档章节顺序唯一编号 |
| 原规则摘要 | 必填，保留原意，不得在摘要阶段偷改语义 |
| 特例标记 | 必填，只能写 `是` / `否` |
| 新 skill 名称 | 必填，可写一个或多个；多个时需指出主落点 |
| 新规则落点 | 必填，写到具体文件和章节 |
| 迁移动作 | 必填，只能写 `原样迁移` / `等价改写` / `增强补充` |
| 状态 | 必填，只能写 `已覆盖` / `阻断` |
| 主落点 | 多 skill 映射时必填 |
| 等价性说明 | 只要不是“原样迁移”就必须填写 |
| 组合覆盖说明 | 一条原规则被多条新规则共同覆盖时填写 |

最小模板：

| 原 skill 规则ID | 原规则摘要 | 特例标记 | 新 skill 名称 | 新规则落点 | 迁移动作 | 状态 | 主落点 | 等价性说明 | 组合覆盖说明 |
|---|---|---|---|---|---|---|---|---|---|
| R-3.1-001 | ... | 否 | ... | ... | 原样迁移 | 已覆盖 | ... | 无 | 无 |

## 3. 迁移动作判定口径

- `原样迁移`：语义、触发、约束强度均不变。
- `等价改写`：表达位置或组织方式变化，但语义与约束强度不变。
- `增强补充`：在保留原规则的前提下新增更细口径，不能弱化原约束。
- 若无法判断属于哪一类，默认按 `阻断` 处理，而不是先写“已覆盖”。

## 4. 特例与资源承接

- 任何看起来奇怪、低频、历史兼容的规则，默认标记为特例并强制保留。
- `references/`、`scripts/`、`assets/`、`agents/openai.yaml` 都属于迁移对象，不得只迁 `SKILL.md`。
- 若某项资源不再需要，必须给出明确废弃依据；没有依据时默认保留并迁移。

## 5. 旧 skill 删除状态机

- `active`：原 skill 正常使用，尚未开始拆分。
- `splitting`：已经开始拆分，允许补迁移材料，禁止继续混入无关新规则。
- `comparing`：拆分结果已生成，但旧 skill 仅作为冻结对照基线暂时保留，用于核对是否有遗漏或弱化。
- `deleted`：覆盖校验通过后删除旧 skill，由新 skill 集合完全承接。

状态切换：

- `active -> splitting`：已确认存在独立类别集合，且本轮正式启动拆分。
- `splitting -> comparing`：用户明确要求先保留旧 skill 做首轮对照验证，且新 skill 集合已具备基础承接能力。
- `splitting -> deleted`：无需对照保留，且映射表 100% 完成、多轮多模式测试通过、删除前承接检查通过、旧 skill 不再承担任何独有价值。
- `comparing -> deleted`：对照完成，未发现遗漏、弱化或自动触发失效，删除前检查通过。

`comparing` 不是长期兼容期，只允许：

- 保留旧 skill 目录作为冻结基线，不再新增规则。
- 产出旧 skill 与新 skill 集合的覆盖对照结论。
- 产出多轮多模式测试结论，至少覆盖“旧规则零丢失”与“新 skill 自动触发”两类验证。
- 在确认无遗漏后继续进入删除动作。

## 6. 删除前承接检查

删除旧 skill 前，至少写清以下字段：

- 旧 skill 名称
- 当前状态
- 主承接新 skill
- 次承接新 skill
- 删除前临时保留内容
- 删除前测试材料
- 删除后承接方式
- 删除动作
- 删除前检查

建议正文模板：

```text
删除前承接检查：
- 旧 skill：`<old-skill>`
- 当前状态：`splitting`
- 主承接新 skill：`<new-skill-a>`
- 次承接新 skill：`<new-skill-b>`（如无可省略）
- 删除前临时保留：迁移说明、覆盖校验材料
- 删除前测试材料：静态覆盖对照、自动触发验证、真实场景演练
- 删除后承接方式：
  1. 原旧 skill 的每条规则都必须能在映射表中定位到新 skill。
  2. 若存在多个可能落点，则按映射表中的主落点承接。
  3. 若仍无法判定唯一承接关系，则阻断删除并回到拆分设计阶段。
- 删除动作：
  1. 覆盖校验通过后删除旧 skill 目录。
  2. 删除旧 skill 在索引、引用或说明中的主入口位置。
- 删除前检查：
  1. 覆盖映射表 100% 完成。
  2. 多轮多模式测试已完成，且未发现旧规则丢失、弱化或承接空洞。
  3. 新 skill 自动触发验证通过，边界已稳定。
  4. 删除后不再依赖旧 skill 作为命中入口。
  5. 原特例规则均已在新 skill 中找到落点。
```

若删除前无法写清“删除后由谁承接”，则不允许删除旧 skill。

## 7. 测试与验证闸门

- 拆分完成后必须执行多轮多模式测试，不能只做一轮静态映射检查。
- 测试至少覆盖两类核心目标：
  - 旧 skill 的任何规则都没有丢失。
  - 新 skill 仍能自动被触发。
- 若测试发现旧规则丢失、约束弱化、承接歧义或自动触发失效，当前拆分直接判定未完成。
- 若当前是首个实战样本，推荐至少完成：
  - 静态覆盖对照
  - 边界与路由验证
  - 真实场景演练

## 8. 失败与回滚

- 任一原规则无落点时，判定拆分失败并阻断删除。
- 若处于 `comparing` 状态且发现新 skill 集合存在遗漏、弱化、承接歧义或自动触发失效，保持旧 skill 不删并回到拆分设计阶段修正。
- 删除前检查未完成时，不得先删旧 skill 再补材料。
- 覆盖校验不通过时，回滚到拆分前基线版本，补齐缺口后再重新执行拆分流程。
