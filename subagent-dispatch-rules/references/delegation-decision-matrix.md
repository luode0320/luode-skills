# 委派判定矩阵

## 判定顺序

1. 是否存在“明确禁止委派”信号。
2. 是否可切分为独立子任务。
3. 子任务是否阻塞主路径。
4. 是否存在写集冲突或高风险操作。
5. 满足条件后再委派，否则本地执行。

## 默认策略

- 默认允许并推荐优先使用 subagent。
- 只有命中“明确禁止委派”或其他阻断条件时，才回退主 agent 本地执行。

## 可委派（满足全部）

- 未出现用户明确禁止委派信号。
- 子任务输入、输出和验收标准可写清。
- 子任务不阻塞主 agent 的下一步关键动作。
- 子任务写集与主路径写集不冲突，或天然只读。

## 不可委派（命中任一）

- 用户明确禁止 subagent / delegation / 并行代理。
- 任务边界无法定义，或依赖实时交互决策。
- 下一步就是主路径关键阻塞步骤。
- 涉及高风险不可逆操作且必须主 agent 统一控制。

## 主 agent 可见性要求

- 只要启动 subagent，主 agent 必须立即输出“已启动 + 执行 skill + 任务摘要”。
- subagent 完成后，主 agent 必须输出“已完成 + 执行 skill + 结果摘要”。

## 建议优先委派类型

- 注释补齐、注释位点检查、注释一致性对照。
- 静态审查类检查（格式、语法、规则核对）。
- 可并行的文档同步与清单更新。

## 建议本地执行类型

- 关键实现路径上的实时编码决策。
- 风险高、强耦合、难以拆分的重构。
- 需要连续上下文推演的主链路分析。
