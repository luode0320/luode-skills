# 子代理分发规则

任一 skill 命中并进入执行前，必须先进行 subagent 委派判定。
默认自动分析、默认允许并优先尝试委派；不需要用户明确要求才启动。
只有明确阻断条件命中时，才回退本地执行。

## Skill 作用与适用场景

- 为已命中的 skill 增加统一的“是否委派 subagent”前置判定。
- 明确 subagent 委派是自动判定机制，不以用户显式要求为前置条件。
- 在不阻塞主路径的前提下，优先把可并行、边界清晰的子任务交给 subagent。
- 降低主 agent 串行处理多个独立子任务导致的等待时间。
- 对代码注释、代码审查、规则核对等可切分任务优先尝试委派。
- 统一回收 subagent 结果并并入主流程，避免并行结果丢失或重复执行。
- 强制主 agent 对外可见地汇报 subagent 的启动、执行中与完成状态，确保过程可追踪。

## 自动触发信号

- 强制前置触发：任一 skill 已命中，且即将进入实质执行阶段。
- 实际优先委派条件：当前任务存在可拆分、可并行的独立子任务。
- 实际优先委派条件：当前任务出现“可本地做但会阻塞主路径”的旁路工作（如审查、注释补齐、规则对照）。
- 用户没有明确要求 subagent 不构成回退理由；只有明确禁止、不可切分、风险不可控、写集冲突或环境不支持时才回退本地执行。

## 进入后先做什么

1. 先自动执行委派判定，不等待用户要求“请用 subagent / 并行代理”。
2. 确认当前轮是否存在“明确禁止 subagent”的信号；若无，默认允许并优先委派。
3. 先识别主路径阻塞步骤，禁止把关键阻塞步骤直接外包导致主流程停摆。
4. 再识别可并行的 sidecar 子任务，并明确每个子任务的输入、输出和责任边界。
5. 若子任务涉及改代码，必须先定义写集，确保不同 subagent 写集不重叠。
6. 判断是否触发阻断条件；若阻断则回退主 agent 本地执行，且回退原因不得写成“用户没有明确要求委派”。
7. 一旦启动 subagent，主 agent 立即输出可见进度消息，说明“已启动谁、执行哪个 skill、在做什么”。

## 默认执行流程

1. 先读 `references/delegation-decision-matrix.md`，判断“可委派 / 不可委派 / 需本地优先”。
2. 可委派时，再读 `references/subagent-task-templates.md`，按模板组装委派任务。
3. 若涉及高风险或边界不清，读 `references/blockers-and-fallbacks.md`，决定回退策略。
4. 先委派 sidecar 子任务，并立即在主 agent 输出“启动公告”进度消息。
5. 委派后继续主 agent 的非重叠工作，不空等。
6. 回收 subagent 结果时，主 agent 输出“完成公告”进度消息，再做最终整合、冲突处理和收口结论。

## 主 agent 可见公告（强制）

- 启动 subagent 后，主 agent 必须先输出一条可见进度：

```text
Subagent 状态：
- 已启动：<subagent-id 或昵称>
- 执行 skill：<skill-name>
- 任务：<一句话任务摘要>
```

- subagent 返回后，主 agent 必须再输出一条可见进度：

```text
Subagent 状态：
- 已完成：<subagent-id 或昵称>
- 执行 skill：<skill-name>
- 结果：<一句话结果摘要>
```

## 子任务优先委派清单

- 代码注释补齐与注释位点核查（如 `comment-placement-granularity-rules`、`comment-completion-gate-rules`）。
- 编码审查类检查（如 `implementation-review-rules`、`cleanup-format-review-rules`、`syntax-check-review-rules`）。
- 规则核对与文档同步类任务（不阻塞主路径时）。
- 多模块独立改造且写集可天然分离的任务。

## 必须主 agent 本地执行的场景

- 下一步是当前主路径的关键阻塞步骤。
- 子任务边界不清、上下游强耦合、无法给出明确输入输出。
- 需要统一决策的高风险操作（删除/迁移/发布关键路径）。
- 用户明确禁止 subagent，或当前环境不支持 subagent 能力。
- 不允许把“用户没有明确要求委派”作为本地执行理由。

## 权责边界与不负责事项

- 只负责“是否委派 + 如何切分 + 如何回收”，不代替具体业务 skill 的执行细则。
- 不因为可委派就把所有工作都外包；主路径关键步骤仍由主 agent 主导。
- 不允许在写集冲突未消解时并行委派多个改代码子任务。
- 不允许绕过系统安全、权限审批或高风险操作保护。
- 不允许只在内部委派而不对外输出主 agent 可见状态。
- 不允许输出“本轮命中 subagent-dispatch-rules，但用户没有明确要求委派/并行代理”作为不委派理由。

## 需要暂停并确认的条件

- 用户明确禁止委派，但任务计划依赖 subagent 才能推进。
- 子任务写集重叠，且无法快速拆分出互斥边界。
- 子任务结果会直接影响关键业务决策，且无可验证回收标准。
- subagent 返回结果与主路径证据冲突，无法自动裁决。

## 执行通过 / 驳回标准

- 通过：委派判定有据可依、子任务边界清晰、写集不冲突、主路径持续推进；且主 agent 已输出 subagent 启动/完成可见状态并最终收口。
- 驳回：在明确禁止或高风险场景强行委派、把关键阻塞步骤外包后空等、并行任务互相覆盖、未输出主 agent 可见状态导致过程不可追踪，或仅因用户未明确要求 subagent 就跳过可委派任务。

## references 读取规则

- 只有在需要组织委派消息时，再读 `references/subagent-task-templates.md`。
- 只有在命中阻断条件或需要回退时，再读 `references/blockers-and-fallbacks.md`。
