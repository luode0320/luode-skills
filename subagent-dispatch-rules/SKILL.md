# 子代理分发规则

任一 skill 命中并进入执行前，必须先进行 subagent 委派判定。
默认自动分析是否值得委派；真实启动前必须先确认当前工具授权策略允许启动。
本仓库已声明 subagent 完全授权模式，视为用户对自动 subagent / delegation / parallel agent work 的项目级显式授权；只有当前项目没有完全授权声明且当前轮也没有授权时，才回退本地执行。
委派判定不是终点；满足任务条件且工具授权允许时必须发起真实子线程，而不是只给一份“建议并行”的文字结果。

## Skill 作用与适用场景

- 为已命中的 skill 增加统一的“是否委派 subagent”前置判定。
- 明确 subagent 委派判定是自动机制；本仓库完全授权声明满足工具显式授权前置条件，真实启动仍需满足任务可切分、写集不冲突和风险可控。
- 在不阻塞主路径的前提下，优先把可并行、边界清晰的子任务交给 subagent。
- 支持主 agent 基于任务结构自主识别可委派子任务；不要求子任务已经落在固定 skill 映射或固定并行清单中。
- 降低主 agent 串行处理多个独立子任务导致的等待时间。
- 对代码注释、代码审查、规则核对等可切分任务优先尝试委派。
- 对项目分析、需求完善侦察、Bug 分诊、资料/日志/调用链证据收集等可拆分任务，默认先尝试形成 sidecar 子 agent 计划；在完全授权模式下允许真实启动，否则按授权状态回退本地执行。
- 统一回收 subagent 结果并并入主流程，避免并行结果丢失或重复执行。
- 强制主 agent 对外可见地汇报 subagent 的启动、执行中与完成状态，确保过程可追踪。

## 自动触发信号

- 强制前置触发：任一 skill 已命中，且即将进入实质执行阶段。
- 实际优先委派条件：当前任务存在可拆分、可并行且能在窄上下文内完成的独立子任务。
- 实际优先委派条件：当前任务出现“可本地做但会阻塞主路径”的旁路工作（如审查、注释补齐、规则对照）。
- 实际优先委派条件：当前任务存在多个独立问题、独立证据来源、独立资料源、独立目录或独立模块可同时侦察。
- 实际优先委派条件：项目分析、找 Bug、需求完善、测试策略、影响面盘点等任务可以拆出只读 sidecar 子任务。
- 上游 `parallel-task-dispatch-rules` 已给出“可并行”或“条件并行”判定，并完成线程拆分。
- 单纯“用户没有主动提到 subagent”不构成回退理由；本仓库完全授权声明已经覆盖该授权。只有当前项目没有完全授权声明且当前轮也没有授权时，才属于“工具授权不支持自动启动”的环境阻断。

## 进入后先做什么

1. 先自动执行委派判定，不等待用户要求“请用 subagent / 并行代理”。
2. 读取或确认当前环境真实可用的 subagent / multi-agent / thread 工具元数据，并确认当前项目是否声明 subagent 完全授权模式；完全授权模式视为满足显式授权要求。
3. 确认当前轮是否存在“明确禁止 subagent”的信号；若无且当前轮授权或项目级完全授权成立，默认允许并优先委派，但前提是通过上下文重复读取与启动成本门槛。
4. 先识别主路径阻塞步骤，禁止把关键阻塞步骤直接外包导致主流程停摆。
5. 若上游已有并行线程方案，优先复用其线程边界、文件归属和收口负责人；不要重新发明一套冲突的线程拆分。
6. 再识别可并行的 sidecar 子任务，并明确每个子任务的输入、输出和责任边界。
6.1 先估计上下文重复读取成本：若 subagent 需要为了入门重复阅读主 agent 已掌握的大段共享上下文、重复扫描同一批核心文件，或任务只覆盖单文件 / 少量文件且主 agent 一次聚焦读取即可完成，直接回退主 agent 本地执行。
7. 若任务没有固定 skill 映射，主 agent 仍必须按“独立问题 / 独立证据来源 / 独立目录 / 独立模块 / 独立文件集”自行拆分候选子任务。
8. 若子任务涉及改代码，必须先定义写集，确保不同 subagent 写集不重叠。
9. 判断是否触发阻断条件；若阻断则回退主 agent 本地执行。回退原因不得写成泛化的“用户没有明确要求委派”；在本仓库完全授权模式下，也不得把“缺少逐次授权”写成阻断。
10. 启动前必须核对当前活跃 subagent 数（已启动且未回收）是否已达到「并发上限与空闲回收」章节规定的上限 5；已达上限时新任务必须先排队，等待现有 subagent 回收释放槽位后才能启动，不得强行超发。
11. 一旦启动 subagent，主 agent 立即输出可见进度消息，说明“已启动谁、执行哪个 skill、在做什么”。
12. 启动完成后，必须核对“计划线程数”和“实际启动线程数”；若少于计划值，必须立即补齐回退说明。
13. subagent 完成、失败或被放弃后，主 agent 必须尽快回收结果并调用真实的 `close_agent` / 等价关闭工具，避免已结束子 agent 长时间占用并发槽位。
14. 若启动工具返回平台昵称，必须记录“逻辑中文名 -> 平台昵称 -> agent_id”映射；禁止把平台英文昵称误报成“中文命名失败”。

## 默认执行流程

1. 先读 `references/delegation-decision-matrix.md`，判断“可委派 / 不可委派 / 需本地优先”。
2. 可委派时，再读 `references/subagent-task-templates.md`，按模板组装委派任务。
3. 需要批量启动时，再读 `references/launch-plan-schema.md`，并优先运行 `scripts/generate_subagent_plan.py` 生成结构化启动计划。
4. 若涉及高风险或边界不清，读 `references/blockers-and-fallbacks.md`，决定回退策略。
5. 使用当前环境真实可用且当前轮授权或项目级完全授权允许的 subagent / multi-agent / thread 工具发起委派；禁止只在正文里宣称“已并行”却没有真实工具调用。
6. 先委派 sidecar 子任务，并立即在主 agent 输出“启动公告”进度消息。
7. 委派后继续主 agent 的非重叠工作，不空等。
8. 回收 subagent 结果时，主 agent 输出“完成公告”进度消息，再做最终整合、冲突处理和收口结论。
9. 结果已收回且不再需要该子 agent 后，立即调用 `close_agent` / 等价关闭工具，并核对“已关闭线程数”与“已完成线程数”。

## 主 agent 可见公告（强制）

- 启动 subagent 后，主 agent 必须先输出一条可见进度：

```text
Subagent 状态：
- 已启动：<subagent-id 或昵称>
- 逻辑名：<任务简要中文-线程标识>
- 平台昵称：<spawn 返回昵称；没有则写未提供>
- 线程：<A / B / C ...>
- 执行 skill：<skill-name>
- 任务：<一句话任务摘要>
- 并发槽位：<当前活跃数>/5
```

- subagent 返回后，主 agent 必须再输出一条可见进度：

```text
Subagent 状态：
- 已完成：<subagent-id 或昵称>
- 逻辑名：<任务简要中文-线程标识>
- 平台昵称：<spawn 返回昵称；没有则写未提供>
- 线程：<A / B / C ...>
- 执行 skill：<skill-name>
- 结果：<一句话结果摘要>
- 回收：<已调用 close_agent / 回收失败原因>
- 并发槽位：<回收后当前活跃数>/5
```

## 真实启动证据（强制）

- 只要对外宣称“已启动 subagent / 已并行执行”，就必须存在真实的 subagent / multi-agent / thread 工具调用结果。
- 若上游 `parallel-task-dispatch-rules` 已给出线程计划，必须逐一核对每个计划线程是否都有对应的实际启动记录。
- 若使用 `scripts/generate_subagent_plan.py`，必须核对脚本输出中的 `planned_thread_count`、`threads[*].agent_name` / `threads[*].logical_agent_name` 与真实启动结果一致。
- `threads[*].agent_name` / `threads[*].logical_agent_name` 表示主 agent 计划中的中文逻辑名；真实 UI 中显示的昵称以启动工具返回值为准，当前若工具未提供命名参数，不得声称可以强制改成中文。
- 若只输出了线程分配、并行技能列表、启动公告文案，但没有真实工具调用结果，视为“未启动”。
- 若环境不支持真实 subagent / multi-agent / thread 能力，必须显式输出“环境不支持真实并行执行”，并回退串行或本地执行；不得把这类场景写成“已并行”。
- 若工具元数据明确要求用户显式授权 subagent / delegation / parallel agent work，当前项目没有完全授权声明且当前轮也没有授权，必须显式输出“工具授权不支持自动启动”，计划线程数可大于 0，但实际启动数必须为 0；不得尝试调用启动工具。
- subagent 已完成不等于已回收；只要结果已经收回且不再需要，主 agent 仍必须继续调用 `close_agent` / 等价关闭工具。
- 最终至少要能回答三件事：
  - 计划启动几个线程
  - 实际启动几个线程
  - 实际关闭几个线程
  - 哪些线程未启动或未关闭以及为什么
- 若使用 `scripts/generate_subagent_plan.py` 的批次拆分输出，必须核对 `max_concurrent`、`batch_count`、`threads[*].batch_index` 与真实启动顺序一致：同一批次内线程数不得超过 `max_concurrent`，且必须等上一批全部回收关闭后才能启动下一批。

## 平台能力矩阵（新增）

不同平台对"子代理并发管理"的实现机制不同，回收/关闭动作必须按当前平台实际可用工具执行，不得对不存在的工具名生搬硬套：

| 平台 | 并发槽位模型 | `close_agent` 等价物 | 执行动作 |
| --- | --- | --- | --- |
| Codex | 存在多代理/线程平台机制，需显式关闭回收 | 平台提供的真实 `close_agent` 或等价工具 | 结果收回后立即调用，维护活跃数计数 |
| Claude Code | 前台子代理调用（未使用 `run_in_background`）阻塞至返回，天然只有一个子代理在执行；后台任务（`run_in_background`）可用 `TaskStop`/`TaskGet`/`TaskOutput` 等延迟工具查询/终止 | 当前无独立 `close_agent` 工具；"回收"等价于前台调用自然返回，或对后台任务调用 `TaskStop`（若已提供） | 前台调用完成即视为已回收，无需额外关闭动作；后台任务完成或不再需要时调用 `TaskStop`；不得假装调用不存在的 `close_agent` |
| 其他 agent | 视具体工具元数据而定 | 仅在真实存在时使用 | 无对应工具时显式跳过并记录 |

## 并发上限与空闲回收（强制）

- 同一会话内，“已启动且未回收（尚未调用 `close_agent` / 等价关闭工具）”的 subagent 总数（活跃数）任意时刻不得超过 5。
- 活跃数由主 agent 自行维护的运行时清单（逻辑名 -> 平台昵称 -> agent_id -> 状态）计数；当前平台工具集不提供“查询活跃 agent 数”的接口，因此本上限是主 agent 自律约束，不是工具强制拦截，违反本条视为流程违规。
- 已达上限 5 时，新的委派请求必须先排队；只有等现有 subagent 被回收、活跃数降到 5 以下后，才能启动排队中的下一个。
- 批量启动时优先用 `scripts/generate_subagent_plan.py` 的批次拆分能力：脚本按 `max_concurrent`（固定为 5）把线程分成若干批，每批 `threads[*].batch_index` 相同；主 agent 必须按批次顺序启动，同一批次内的线程数不得超过 5，且不得在上一批次全部关闭前提前启动下一批次。
- 以下任一信号命中即视为该 subagent “闲置 / 不再使用”，主 agent 必须立即调用 `close_agent` / 等价关闭工具释放并发槽位，不得等到全部任务完成后再统一关闭：
  - 结果已收回且不再需要该 subagent 继续工作。
  - 已判定该 subagent 放弃或长时间无进展（例如反复失败、明显跑偏且无法自我纠正）。
  - 用户明确要求取消该子任务。
- 不设固定超时分钟数作为空闲判定条件；判定依据是上述三类状态信号，而不是计时器。
- 不允许以“稍后可能还要用到”为由，在已命中任一空闲信号后继续占用槽位。
- Claude Code 平台说明（新增）：若当前会话为前台阻塞式子代理调用（无 `run_in_background`），一次只能有一个子代理在执行，"上限 5"与显式 `close_agent` 回收在此形态下不构成实际约束，可视为天然满足；若使用 `run_in_background` 并行多个后台任务，应改为对照当前会话实际暴露的任务管理工具（如 `TaskList`/`TaskGet`/`TaskStop`）维护活跃数，逻辑上仍遵循"同时活跃不超过 5、完成或放弃后及时标记回收"的精神，但不强行调用名为 `close_agent` 的工具。

## 子任务优先委派清单

- 代码注释补齐与注释位点核查（如 `comment-placement-granularity-rules`、`comment-completion-gate-rules`）。
- 编码审查类检查（如 `implementation-review-rules`、`project-change-review-rules`）。
- 规则核对与文档同步类任务（不阻塞主路径时）。
- 多模块独立改造且写集可天然分离的任务。
- 项目分析中的目录结构、入口链路、配置/部署面、模块职责等只读侦察。
- 需求完善中的代码现状、既有文档、外部官方资料、竞品/API 资料等独立证据收集。
- Bug 分诊中的复现路径、日志/报错、近期改动、调用链和影响面并行收集。
- 测试策略、回归范围、风险清单等可按模块或场景拆分的只读分析。

## 必须主 agent 本地执行的场景

- 下一步是当前主路径的关键阻塞步骤。
- 子任务边界不清、上下游强耦合、无法给出明确输入输出。
- 需要统一决策的高风险操作（删除/迁移/发布关键路径）。
- 单一根因、需求边界、接口契约、schema 或架构方向需要主 agent 统一裁决。
- 用户明确禁止 subagent，或当前环境不支持 subagent 能力，或当前工具授权策略要求用户显式授权且当前轮/项目级完全授权均不存在。
- 不允许把“用户没有主动提到委派”作为泛化本地执行理由；本仓库完全授权模式下，逐次未提 subagent 不是环境阻断。

## 权责边界与不负责事项

- 只负责“是否委派 + 如何切分 + 如何回收”，不代替具体业务 skill 的执行细则。
- 不因为可委派就把所有工作都外包；主路径关键步骤仍由主 agent 主导。
- 不允许在写集冲突未消解时并行委派多个改代码子任务。
- 不允许绕过系统安全、权限审批或高风险操作保护。
- 不允许只在内部委派而不对外输出主 agent 可见状态。
- 不允许输出“本轮命中 subagent-dispatch-rules，但用户没有明确要求委派/并行代理”作为泛化不委派理由；若真实工具元数据要求显式授权，必须先检查当前项目是否已有完全授权声明。
- 不允许把“已经完成委派判定”当作“已经真实启动子线程”。
- 不允许把平台返回的英文昵称误写成“脚本没有生成中文任务名”；两者不是同一个概念。

## 需要暂停并确认的条件

- 用户明确禁止委派，但任务计划依赖 subagent 才能推进。
- 子任务写集重叠，且无法快速拆分出互斥边界。
- 子任务结果会直接影响关键业务决策，且无可验证回收标准。
- subagent 返回结果与主路径证据冲突，无法自动裁决。

## 执行通过 / 驳回标准

- 通过：委派判定有据可依、子任务边界清晰、写集不冲突、主路径持续推进；若当前轮授权或项目级完全授权允许启动，主 agent 已真实调用可用的 subagent / multi-agent / thread 工具，已输出 subagent 启动/完成可见状态，已记录逻辑中文名与平台昵称映射，并能对齐“计划线程数 = 实际启动线程数 = 已回收关闭线程数”；若当前轮授权与项目级完全授权均不存在，已明确记录“工具授权不支持自动启动”，实际启动数为 0，并回退本地执行；任意时刻活跃 subagent 数不超过 5，且命中空闲信号的 subagent 已被及时 `close_agent` 回收。
- 驳回：在明确禁止或高风险场景强行委派、把关键阻塞步骤外包后空等、并行任务互相覆盖、未输出主 agent 可见状态导致过程不可追踪、只做委派判定未真实启动、结果收回后未关闭子 agent，或仅因用户未明确要求 subagent 就跳过可委派任务；活跃数超过 5 仍强行启动新 subagent；或 subagent 已命中空闲信号（结果已收回 / 判定放弃 / 用户要求取消）却未及时关闭、长期占用并发槽位。
