# Bug 侦察输出模板

## 推荐结构

```markdown
## 现象复述

- 用户原始描述：
- 截图 / 报错关键信息：
- agent 当前理解：

## 已侦察来源

| 来源 | 类型 | 结论 | 证据等级 |
| ---- | ---- | ---- | -------- |

## 代码侦察

- 相关模块 / 文件：
- 可疑逻辑 / 分支 / 状态转换：
- 调用链（上游 -> 下游）：

## 数据侦察（本地库，只读）

- 连接的本地库：
- 执行的只读 SQL（脱敏）：
- 期望数据 vs 实际数据对比：
- 数据结论：数据本身错 / 写入逻辑错 / 读取逻辑错 / 数据正常

## 根因候选

| 候选 | 依据 | 证据等级 | 排除方式 |
| ---- | ---- | -------- | -------- |

## 风险与边界

- 影响范围：
- 是否涉及公共方法 / 共享模块 / 兼容性：
- 数据风险：

## 仍需用户确认

1. ...

## 需要手动执行的 SQL（如有）

- 目的：
- 脚本：
- 影响行 / 回滚方式：

## 记忆回写建议

- 需要写入 `PROJECT_MEMORY.md` 的线索：
- 不写入原因：

## 下一步

- 转交 skill：
- 进入条件：
```

## 输出要求

- 待确认问题不超过 3 个，除非现象本身包含多个互不相关的问题。
- 每个根因候选必须至少有一个证据来源，并标注证据等级。
- 如果没查到有效证据，要明确写出查过哪里、为什么没有结论。
- 连库部分必须体现“只读、仅本地”；任何增删改只能以 SQL 脚本形式交用户手动执行。
- 若已能复现或定位，下一步转 `bug-reproduction-rules` 或 `bug-root-cause-rules`；怀疑运行期状态转 `bug-runtime-debug-rules`；仍缺关键信息转 `bug-gap-rules`。
