# 接口基线对账规则

本文件定义扫描结果与现有接口基线之间的对账方式，避免基线长期漂移。

## 差异分类

### 新增
- 代码或文档中出现，但基线中不存在的接口。
- 必须补入基线，并写明发现来源、发现证据、完整度和待确认项。

### 删除
- 基线中存在，但当前代码、文档和测试资产都已无法找到的接口。
- 必须标记为废弃或删除状态，不得继续纳入门禁。

### 变更
- 同一接口的路径、HTTP 方法、鉴权方式、请求参数 schema、响应结构摘要、成功/失败判定、依赖数据、风险等级发生变化。
- 必须先更新基线，再决定是否沿用历史测试结果。
- 请求参数 schema、响应结构、鉴权方式、参数来源、可提供字段或业务成功判定变化时，必须同步影响 `reusable-params.yaml`：相关参数先标记为 `stale`，本轮复用前必须复验。

### 待确认
- 扫描发现冲突，但当前无法确定哪份信息是正确的。
- 必须写明待确认原因和待确认字段，阻断该接口的自动放行结论。

## 对账输出要求
- 对账结果至少包含：新增数、删除数、变更数、待确认数。
- 每条差异都要给出接口标识、差异类型、来源证据、处理动作。
- 对账完成后，必须先更新基线文件，再生成测试计划。
- 对账结果必须包含 `affected_reusable_params`，列出因本轮漂移需要标记 `stale` 的参数 key、影响接口、影响字段和原因。
- 若发现新的 provider / consumer 关系或参数来源，必须同步更新 `dependency-graph.yaml` 与 `parameter-sources.yaml`。

## 阻断条件
- 存在待确认差异且影响接口是否必测或影响成功判定。
- 存在关键接口（P0/P1）的路径、方法、鉴权或请求响应结构漂移，但未回写基线。
- 对账结果未落盘，只在口头说明里提到差异。
- 可复用参数受漂移影响但未标记 `stale`，仍被直接用于 P0 / P1 或写接口测试。
