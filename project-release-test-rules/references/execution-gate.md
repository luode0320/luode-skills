# 上线接口测试门禁判定规则

本文件定义上线测试的最终结论判定规则，结论直接作为是否允许上线的依据，分为PASS、FAIL、PARTIAL三个等级，必须严格按照规则判定，不得随意调整等级。

## 基础规则
- 所有判定必须基于实际执行的接口测试结果，没有执行测试不能给出任何结论。
- 没有完整的接口明细报告，不能给出PASS结论。
- 所有P0级接口必须有测试结果，不得遗漏。
- 待确认状态的接口必须有明确的待确认理由，且不能默认按通过处理。

---

## 结论等级判定规则
### 1. PASS（允许上线）
同时满足以下所有条件：
1. 所有必测P0级接口全部通过，无遗漏、无不通过、无待确认、无跳过。
2. 必测P1级接口通过率100%，或者不通过的P1接口经过评估不影响本次上线功能，且有明确的修复计划和上线后补测安排。
3. 无阻断级的接口问题（如核心服务不可用、数据错乱、鉴权失效等）。
4. 所有接口测试结果完整，判定理由充分。
5. 风险项都经过评估，且影响可控。

### 2. FAIL（阻断上线）
满足以下任一条件即为FAIL：
1. 存在任一P0级接口不通过。
2. 存在任一P0级接口待确认，且无法在上线前明确结论。
3. 存在任一P0级接口被跳过，无合理理由。
4. 超过2个及以上P1级接口不通过，且影响核心业务。
5. 存在阻断级的全局问题（如服务全不可用、所有接口鉴权失败、数据库报错等）。
6. 测试范围覆盖不全，大量必测接口未执行。

### 3. PARTIAL（风险上线/需要人工确认）
同时满足以下所有条件：
1. 没有P0级接口不通过、待确认、跳过的情况。
2. 存在少量P1级接口不通过，但不影响本次上线的核心功能，或者是低影响的已知问题，有明确的修复计划。
3. 存在部分P2级接口不通过，但不影响整体业务。
4. 存在待确认的接口，但不涉及核心功能，风险可控。
5. 所有问题都有明确的说明、影响范围评估和后续修复安排。

---

## 阻断规则
出现以下任一情况时，必须给出FAIL结论，阻断上线：
1. 支付、交易、资金类接口不通过，可能导致资金损失。
2. 鉴权、权限类接口不通过，可能导致权限泄露、越权访问。
3. 核心订单、数据写入接口不通过，可能导致业务不可用、数据错乱。
4. 接口存在敏感信息泄露（明文密码、密钥、用户隐私）。
5. 大量核心接口返回5xx错误，服务不可用。
6. 接口执行后导致数据污染、脏数据无法清理，影响线上业务。

---

## 风险处理规则
对于PARTIAL结论，必须明确列出以下内容：
1. 每个风险接口的风险等级、影响范围、严重程度。
2. 风险是否在可接受范围内的评估依据。
3. 上线后的修复计划、补测计划。
4. 回滚方案（如果风险发生如何处理）。
5. 需要业务方/技术负责人确认的点。

---

## 结论使用规则
1. PASS结论可以直接作为最终验收的输入，允许上线。
2. FAIL结论必须修复问题后重新测试，重新生成报告，直到得到PASS或经过特批的PARTIAL。
3. PARTIAL结论必须经过相关负责人确认风险可接受后，才能作为允许上线的依据，不能直接自动放行。
4. 最终结论必须写入测试报告主README.md，作为上线审批的附件留存。
