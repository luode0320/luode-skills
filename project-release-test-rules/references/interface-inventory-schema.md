# 项目接口基线字段结构

本文件定义项目级接口测试基线的标准化字段，用于长期维护所有接口的定义、判定规则和测试记录，支持跨上线迭代复用。

## 基线核心字段（必填）

| 字段名 | 说明 | 示例 |
| --- | --- | --- |
| 接口标识 | 全局唯一的接口 ID，建议使用模块名_接口名_方法 | order_create_POST |
| 接口名称 | 接口的中文业务名称 | 创建订单接口 |
| HTTP 方法 | 标准 HTTP 方法 | POST / GET / PUT / DELETE / PATCH |
| 接口路径 | 接口的全路径 | /api/order/v2/create |
| 所属模块 | 接口所属的业务模块 | 订单模块 / 用户模块 / 支付模块 |
| 鉴权要求 | 接口是否需要鉴权，以及鉴权类型 | 需要 Bearer Token / 不需要鉴权 / 需要签名 |
| 请求参数 schema | 核心请求参数的 JSON 结构定义，必填字段、类型、说明 | `{"userId":"string(必填，用户ID)","amount":"number(必填，订单金额)"}` |
| 响应结构摘要 | 核心响应字段的 JSON 结构定义，必填字段、类型、说明 | `{"httpStatus":"number","code":"number","message":"string","data":{"orderId":"string"}}` |
| 业务成功判定 | agent 判断接口业务成功的规则 | code == 0 && data.orderId != "" |
| 业务失败判定 | agent 判断接口业务失败的规则 | code != 0 || data.orderId == "" |
| 依赖数据 | 测试需要的依赖数据说明 | 需要已认证用户 token、有效币种配置 |
| 数据副作用 | 接口执行后对业务数据的影响 | 创建新订单记录、扣减用户可用额度 |
| 清理方式 | 测试后清理脏数据的方法 | 删除测试订单、回滚额度扣减 |
| 风险等级 | 接口的风险级别 | P0 / P1 / P2 |
| 是否上线必测 | 是否每次上线都必须测试该接口 | 是 / 否 |
| 最近扫描时间 | 上次扫描该接口事实的时间 | 2026-06-30 20:35:00 |
| 最近测试时间 | 上次测试该接口的时间 | 2026-06-30 21:10:00 |
| 最近测试结论 | 上次测试的结论 | 通过 / 不通过 / 待确认 |

## 扫描与对账字段（必填）

以下字段用于支撑冷启动、每次执行前的增量扫描和基线对账：

| 字段名 | 说明 | 示例 |
| --- | --- | --- |
| 发现来源 | 当前接口事实来自哪类扫描来源 | route / controller / swagger / test / doc |
| 发现证据 | 扫描命中的文件路径、符号名或文档片段 | backend/router/order.go:42 |
| 完整度 | 当前基线信息完整程度 | 完整 / 部分 / 待确认 |
| 待确认项 | 当前仍缺失或冲突的关键字段列表 | ["鉴权要求","业务成功判定"] |
| 最近扫描提交 | 本次扫描对应的提交或工作区标识 | 86e52d2 |

## 推荐扩展字段

以下字段不是强制必填，但强烈建议在项目成熟后补齐：

- 接口负责人：接口的业务或技术负责人
- 上线时间：接口首次上线时间
- 废弃标记：是否已废弃，废弃时间和替代接口
- 限流规则：接口的限流策略
- 幂等规则：接口是否幂等，幂等键说明
- 第三方依赖：接口依赖的外部服务名称
- 鉴权获取方式：测试时如何拿到 token、cookie 或签名
- 调用顺序依赖：是否依赖前置接口先调用
- 成功后关键副作用：成功后必须额外核验的写入、副作用或状态流转
- 失败后预期语义：失败响应在业务上应如何解释
- 回滚/清理步骤：测试结束后如何清理脏数据、恢复环境

## 基线存储规则

- 每个项目统一存放在 `doc/5-tests/基线/interface-inventory.yaml`，YAML 格式，可被脚本直接读取。
- 新增、修改、删除接口必须更新基线。
- 每次上线测试前必须先扫描当前接口事实，并与基线对账。
- 每次上线测试后必须更新对应接口的最近扫描时间、最近测试时间和结论。
- 基线禁止随意删除；废弃接口应标记废弃状态，而不是静默移除。
- 冷启动生成的初版基线允许包含 `待确认项`，但不得缺少 `发现来源`、`发现证据` 和 `完整度`。
