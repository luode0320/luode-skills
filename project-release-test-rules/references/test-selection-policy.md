# 上线前接口测试范围选择策略

本文件定义上线前接口测试的范围选择规则，agent必须严格按照该策略筛选本次必测接口，不得随意跳过核心接口，也不能无限制扩大测试范围导致效率低下。

## 核心原则
- 风险导向：风险越高的接口测试优先级越高，必须覆盖。
- 改动导向：与本次上线改动直接相关、间接影响的接口必须覆盖。
- 核心导向：系统核心链路接口无论是否改动，每次上线必须覆盖。
- 透明原则：每个被选的接口必须说明选择理由，每个跳过的接口必须说明跳过理由。

## 风险等级划分规则
首先将所有接口按风险等级划分为P0、P1、P2三级：

### P0级接口（最高风险，每次上线必测，无例外）
满足以下任一条件即为P0：
- 涉及资金、交易、支付、结算、扣款、退款相关的接口
- 涉及用户身份、鉴权、权限、登录、授权相关的接口
- 涉及订单、合同、单据、核心数据写入的接口
- 涉及核心链路状态变更、流转的接口
- 涉及用户敏感信息（身份证、手机号、银行卡、密码）的接口
- 影响全系统可用性的基础服务接口（如网关、认证中心、配置中心）
- 线上出过严重故障的接口
- 业务方明确要求每次上线必测的接口

### P1级接口（高风险，按改动影响面选择必测）
满足以下任一条件即为P1：
- 核心业务链路的查询、导出、统计类接口
- 非核心的写入类接口，只影响局部业务
- 业务高频使用的查询接口
- 依赖外部服务的接口
- 本次上线改动模块内的所有接口

### P2级接口（低风险，可选测或周期性全量覆盖）
满足以下任一条件即为P2：
- 后台管理的低频操作接口
- 业务边缘功能的接口
- 只读、静态数据查询接口
- 内部运维、监控类接口
- 已废弃即将下线的接口

## 本次上线必测接口选择规则
符合以下任一条件的接口必须纳入本次测试范围：
1. 所有P0级接口，无论是否改动，必测，无例外。
2. 本次上线改动代码直接对应的接口，必测。
3. 本次上线改动模块内的所有P1级接口，必测。
4. 调用了本次改动代码的上游接口，必测。
5. 被本次改动代码调用的下游接口，必测。
6. 和本次改动接口共享数据、共享逻辑的关联接口，必测。
7. 业务方明确要求本次必须测试的接口，必测。
8. 近3个月出过线上故障的接口，必测。
9. 上次上线测试不通过、待确认的接口，必测。

## 可选测接口选择规则
符合以下条件的接口可以纳入可选测试范围，根据测试时间和资源决定：
1. 本次改动模块周边的P2级接口
2. 其他非关联模块的P1级接口
3. 近6个月未测试过的P2级接口
4. 用户主动要求补充测试的接口

## 可跳过接口规则
只有满足以下所有条件的接口可以跳过本次测试，且必须在报告中说明跳过理由：
1. 确认和本次上线改动无任何关联
2. 近1个月内已经测试过且结论为通过
3. 风险等级为P2级
4. 近3个月未出过线上故障
5. 不属于本次上线需要覆盖的业务范围
- **禁止将任何P0、P1级接口无理由跳过**

## 选择输出要求
最终选择结果必须输出以下内容：
1. P0级接口总数、必测P0接口列表、每个接口选择理由
2. P1级接口总数、必测P1接口列表、每个接口选择理由
3. P2级接口总数、可选测P2接口列表、每个接口选择理由
4. 跳过接口列表、每个接口跳过理由
5. 本次测试范围总览：必测接口数、选测接口数、跳过接口数、风险评估
