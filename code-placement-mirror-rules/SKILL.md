# 代码归位镜像规则

## 目标

只检查本轮代码是否满足 `code-placement-review-rules` 的归位标准，不执行修改。

## 源规则

以下判定严格沿用源 skill：

- 文件归属、模块归属和目录落点是否合理
- 跨层调用、依赖方向和职责归位是否正确
- `utils` / `common` / `global` / `middleware` 根目录不得直接堆实现文件
- Go `internal/service` 根目录不得直接堆业务实现文件
- Go `internal/service` 中请求/响应/第三方结果结构体应归位到 `internal/entity/<domain>/`
- `test/` 之外不得出现 `*_test.go`
- 500 行及以上且持续膨胀的业务文件必须拆分评估

## 自动触发信号

- 新增文件、移动文件或扩展模块。
- 出现跨层调用、工具类落点争议或目录归属争议。
- 编码完成后需要确认实现是不是放对了层、对了目录。
- Go 代码改动涉及测试文件、服务实现或服务实现中的结构体定义。
- 单文件超过 500 行且本轮仍继续新增方法或流程分支。

## 进入后先做什么

1. 先识别当前文件属于入口层、业务层、数据层、公共层还是测试层。
2. 再检查它与相邻目录和模块的职责是否一致。
3. 确认依赖方向是否越层或反向。
4. 检查 `utils` / `common` / `global` / `middleware` 根目录是否直接出现实现文件。
5. 检查 Go `internal/service` 根目录是否直接出现业务实现文件。
6. 对 Go 项目执行禁放扫描和结构体定义扫描。
7. 检查本轮目标文件是否已达 500 行及以上且仍持续堆功能。
8. 判断是否存在“文件位置正确但逻辑职责错误”的情况。

## 默认执行流程

1. 先读取 `code-placement-review-rules`。
2. 如果涉及依赖方向和跨层调用，再读源 skill 的对应 references。
3. 输出归位结论、越层问题和需回改的文件。
4. 归位未通过时，不进入测试域。

## 需要暂停并确认的条件

- 当前代码跨越多个层级，无法单点纠偏。
- 纠正归位会显著扩大本次改动范围。
- 模块职责本身在项目里就有冲突，无法直接判定应该跟随哪一侧。
- 当前问题已不是“归位复查”，而是更上层的架构调整。
- Go 场景扫描到 `test/` 外存在 `*_test.go`，但是否改 seam 方案尚未达成一致。
- Go 服务层结构体迁移涉及接口联动改造，但迁移范围和兼容边界尚未达成一致。
- 500+ 行文件拆分会引发跨模块调用改造，且当前迭代窗口无法承接完整拆分。

## 执行通过 / 驳回标准

- 通过：文件、模块、目录和依赖方向都与职责一致，没有明显越层或错位；Go 场景下 `internal/service` 未散落请求/响应/第三方结果结构体，相关结构体已归位到 `internal/entity`；对 500 行及以上且持续膨胀的文件已完成拆分或给出本轮可执行拆分方案。
- 驳回：位置、职责、依赖方向存在明显错位，`utils` / `common` / `global` / `middleware` 根目录直接放实现文件，Go `internal/service` 根目录直接堆业务实现文件，或 Go 项目中 `test/` 之外存在 `*_test.go`；或 Go 请求/响应/第三方结果结构体散落在 `internal/service` 实现文件；或 500 行及以上文件继续堆功能却没有拆分动作/方案。

## 执行结果归档要求

- 将归位结论、越层问题和回改建议记录到 `review/`。
- 归档内容至少包含检查对象、归属结论、依赖方向和下一步动作。
- 如果本次归位检查无争议，可以简化记录，但应保留结论。

## 输出格式

- 写出归位结论、越层问题和需回改的文件。

## 读取规则

- 默认先读 `code-placement-review-rules`。
- 只有在判断依赖方向时，再读源 skill 的对应 references。
