# 错误处理规则

只在判断“异常应该怎么分类、在哪一层处理、何时重试、何时降级或 fallback”时使用这个 skill。
如果当前问题是错误响应结构本身，请转交 `api-response-rules`；如果当前问题是日志和 trace 记录点，请转交 `logging-trace-rules`。

## Skill 作用与适用场景

- 统一异常分类和错误处理路径。
- 区分全局异常处理与局部处理的职责边界。
- 约束重试、超时、降级和 fallback 的适用条件。
- 防止把错误处理逻辑和错误响应结构混为一谈。

## 自动触发信号

- 新增或修改异常类、全局异常处理器、错误中间件。
- 新增或修改 try/catch、错误映射、超时、重试、降级或 fallback 逻辑。
- 不确定某个异常应该在哪里处理。
- 不确定当前错误是应直接失败、重试还是降级。

## 进入后先做什么

1. 先识别当前异常属于输入错误、业务错误、依赖错误还是系统错误。
2. 再判断应在局部处理还是交给全局处理。
3. 明确当前是否适合重试、超时、降级或 fallback。
4. 确认错误处理和错误响应是两层不同职责。

## 默认执行流程

1. 默认先读 `references/error-classification.md`，确定异常分类。
2. 如果涉及处理层次和路径，再读 `references/handling-paths.md`。
3. 如果涉及重试、超时、降级或 fallback，再读 `references/resilience-boundaries.md`。
4. 输出异常分类、处理层次、处理路径和不适用的机制。
5. 如果当前只是响应格式问题，停止停留在错误处理层并转给 `api-response-rules`。

## 权责边界与不负责事项

- 只负责错误处理机制，不代替 `api-response-rules` 设计错误响应结构。
- 不代替 `logging-trace-rules` 设计日志字段、trace 字段和排障记录点。
- 不代替业务层决定业务规则本身，只决定异常发生后怎么处理。
- 不默认所有错误都适合重试、降级或 fallback。

## 需要暂停并确认的条件

- 当前异常分类不清，无法判断处理路径。
- 重试或降级会改变业务语义、幂等性或一致性。
- fallback 可能掩盖严重错误，导致问题长期隐身。
- 处理策略调整会影响旧链路稳定性或兼容行为。

## 执行通过 / 驳回标准

- 通过：能明确异常分类、处理层次、处理路径，以及重试 / 超时 / 降级 / fallback 的适用边界。
- 驳回：把不同类型错误混成一个处理器，或对所有异常一律重试、一律吞掉、一律降级。

## 执行结果归档要求

- 如果本次定义了新的错误分类、处理路径或弹性策略，将结论记录到 `analysis/` 或 `review/`。
- 归档内容至少包含异常分类、处理结论、风险点和兼容影响。
- 如果只是沿用既有清晰机制且无争议，可以不单独归档。

## references 读取规则

- 默认先读 `references/error-classification.md`。
- 只有在判断局部 / 全局处理边界时，再读 `references/handling-paths.md`。
- 只有在判断重试、超时、降级和 fallback 时，再读 `references/resilience-boundaries.md`。
