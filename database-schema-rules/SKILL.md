# 数据库结构规则

只在“表结构怎么变、迁移怎么写、回滚和兼容性怎么保证”这个问题上使用这个 skill。
如果当前问题是 SQL 查询、事务、锁、分页或批量 CRUD，请转交 `database-query-rules`。

## Skill 作用与适用场景

- 统一表、字段、索引、唯一约束、外键、迁移脚本和 DDL 的结构变更规则。
- 区分结构变更、数据修复、查询实现和业务逻辑的职责边界。
- 约束迁移安全、回滚策略、兼容性处理和上线风险说明。
- 防止把危险结构变更、不可回滚迁移和破坏兼容的 DDL 直接推进到后续环节。

## 自动触发信号

- 新增或修改表、字段、索引、唯一约束、外键、迁移脚本、DDL、实体结构、schema 定义。
- 需要判断字段类型、空值策略、默认值、约束、索引方案是否合理。
- 需要拆分“结构变更”和“数据迁移 / 数据修复”时。
- 需要确认迁移是否安全、是否可回滚、是否会影响旧版本兼容时。

## 进入后先做什么

1. 先识别当前变更是结构定义、迁移脚本、数据修复脚本还是查询实现。
2. 明确本轮只处理 schema 和迁移边界，不提前扩展到查询逻辑。
3. 判断当前变更会不会影响现有数据、现有接口契约或旧版本兼容。
4. 确认当前配置和环境差异规则已经稳定，避免把环境问题混进 schema 变更。

## 默认执行流程

1. 默认先读 `references/schema-boundaries.md`，判断表、字段、索引、约束和数据变更的边界。
2. 如果涉及迁移执行、安全上线和回滚，再读 `references/migration-safety-and-rollback.md`。
3. 如果需要对照正反例或判断兼容性风险，再读 `references/schema-examples.md`。
4. 输出结构变更结论、迁移顺序、兼容风险、回滚策略和不应一起混入的改动。
5. schema 方案未稳定前，不继续进入 `database-query-rules`。

## 权责边界与不负责事项

- 只负责数据库结构、迁移和回滚边界，不负责 SQL 查询写法、事务、锁和分页，那属于 `database-query-rules`。
- 不负责环境变量、连接串、数据库开关和配置来源，那属于 `config-change-rules`。
- 不负责发布窗口、灰度、线上发布回滚执行，那属于交付域或发布类 skill。
- 不把业务逻辑调整伪装成字段含义变化或默认值变化；这种情况应回流需求域或对应实现位点。
- 不默认所有结构变更都能在一次迁移里安全完成，必要时应拆分多步迁移。

## 需要暂停并确认的条件

- 当前 DDL 会删除字段、缩窄类型、修改主键或直接重写大表结构，兼容风险高。
- 当前迁移同时承担结构变更、数据修复和业务开关切换，多重职责混杂。
- 回滚方案不清，或回滚后会造成数据不可逆丢失。
- 结构变更会影响旧版本代码、历史接口或上下游系统，但兼容窗口尚未确认。

## 执行通过 / 驳回标准

- 通过：结构变更边界清楚，迁移顺序合理，兼容与回滚策略可解释，且没有把查询实现或业务逻辑混入 schema 层。
- 驳回：危险 DDL 无兼容保护、回滚不可解释、结构变更与数据逻辑混杂，或直接引入高风险不可逆变更。

## 执行结果归档要求

- 将 schema 变更结论记录到迁移说明、任务记录、评审记录或部署说明中。
- 归档内容至少包含变更对象、迁移顺序、兼容策略、回滚方式、影响范围和风险说明。
- 如果涉及不可逆或高风险迁移，必须明确标记执行前提、观察点和人工确认条件。

## references 读取规则

- 默认先读 `references/schema-boundaries.md`。
- 只有在涉及迁移执行、上线安全和回滚时，再读 `references/migration-safety-and-rollback.md`。
- 只有在需要对照正反例时，再读 `references/schema-examples.md`。
