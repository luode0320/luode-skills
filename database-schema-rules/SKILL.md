# 数据库结构规则

只在"表结构怎么变、迁移怎么写、回滚和兼容性怎么保证"这个问题上使用这个 skill。
如果当前问题是 SQL 查询、事务、锁、分页或批量 CRUD，请转交 `database-query-rules`。

## Skill 作用与适用场景

- 统一表、字段、索引、唯一约束、外键、迁移脚本和 DDL 的结构变更规则。
- **铁律 1：数据库的字段必须要定义数据类型、默认值、是否需要索引、字段 CHARSET=utf8mb4、ENGINE=InnoDB、注释说明等写清楚，不要遗漏，否则会导致自动创建表出现不可控的因素。**
- **铁律 1.1：SQL 标识符（表名、字段名、索引名）统一使用反引号包裹，养成固定习惯；例如 enabled 必须写成 `enabled`。**
- **铁律 2：所有金额相关的要强制使用字符串，避免任何出现精度问题的情况。**
- **铁律 3：所有表必须包含 created_at 和 updated_at 字段，由数据库自动管理。**
- **铁律 4：必须冗余一个毫秒级时间戳的创建时间，避免数据库的时区问题影响不同的时间格式。**
- **铁律 5：所有表必须包含逻辑删除字段，1 的状态标识删除，不是 1 代表正常非删除状态，默认 0=非删除。**
- 区分结构变更、数据修复、查询实现和业务逻辑的职责边界。
- 约束迁移安全、回滚策略、兼容性处理和上线风险说明。
- 防止把危险结构变更、不可回滚迁移和破坏兼容的 DDL 直接推进到后续环节。

## 自动触发信号

- 新增或修改表、字段、索引、唯一约束、外键、迁移脚本、DDL、实体结构、schema 定义。
- 需要判断字段类型、空值策略、默认值、约束、索引方案是否合理。
- 需要拆分"结构变更"和"数据迁移 / 数据修复"时。
- 需要确认迁移是否安全、是否可回滚、是否会影响旧版本兼容时。
- DDL 定义中缺少数据类型、默认值、索引、CHARSET=utf8mb4、ENGINE=InnoDB、注释说明等。
- 金额字段使用 DECIMAL、DOUBLE、FLOAT、INT 等数值类型。
- 缺少 created_at 和 updated_at 字段，或定义不正确。
- 缺少冗余毫秒级时间戳的创建时间字段。
- 缺少逻辑删除字段，或定义不正确。

## 进入后先做什么

1. 先识别当前变更是结构定义、迁移脚本、数据修复脚本还是查询实现。
2. **确认 DDL 定义中包含数据类型、默认值、是否需要索引、CHARSET=utf8mb4、ENGINE=InnoDB、注释说明等，不要遗漏**。
3. **确认 SQL 标识符（表名、字段名、索引名）统一使用反引号包裹；例如 `enabled`。**
4. **确认所有金额字段使用 VARCHAR 或 CHAR 字符串类型**。
5. **确认所有表包含 created_at 和 updated_at 字段，定义正确**。
6. **确认包含冗余毫秒级时间戳的创建时间字段**。
7. **确认包含逻辑删除字段，定义正确**。
8. 明确本轮只处理 schema 和迁移边界，不提前扩展到查询逻辑。
9. 判断当前变更会不会影响现有数据、现有接口契约或旧版本兼容。
10. 确认当前配置和环境差异规则已经稳定，避免把环境问题混进 schema 变更。

## 默认执行流程

1. 默认先读 `references/schema-boundaries.md`，判断表、字段、索引、约束和数据变更的边界。
2. 如果涉及迁移执行、安全上线和回滚，再读 `references/migration-safety-and-rollback.md`。
3. 如果需要对照正反例或判断兼容性风险，再读 `references/schema-examples.md`。
4. **特别强调：数据库的字段必须要定义数据类型、默认值、是否需要索引、字段 CHARSET=utf8mb4、ENGINE=InnoDB、注释说明等写清楚，不要遗漏**。
5. **特别强调：SQL 标识符（表名、字段名、索引名）统一使用反引号包裹；例如 `enabled`。**
6. **特别强调：所有金额相关的要强制使用字符串，避免任何出现精度问题的情况**。
7. **特别强调：所有表必须包含 created_at 和 updated_at 字段，由数据库自动管理**。
8. **特别强调：必须冗余一个毫秒级时间戳的创建时间，避免数据库的时区问题影响不同的时间格式**。
9. **特别强调：所有表必须包含逻辑删除字段，1 的状态标识删除，不是 1 代表正常非删除状态，默认 0=非删除**。
10. 输出结构变更结论、迁移顺序、兼容风险、回滚策略和不应一起混入的改动。
11. schema 方案未稳定前，不继续进入 `database-query-rules`。

## 权责边界与不负责事项

- 只负责数据库结构、迁移和回滚边界，不负责 SQL 查询写法、事务、锁和分页，那属于 `database-query-rules`。
- 不负责环境变量、连接串、数据库开关和配置来源，那属于项目配置约束。
- 不负责发布窗口、灰度、线上发布回滚执行，那属于交付域或发布类 skill。
- 不把业务逻辑调整伪装成字段含义变化或默认值变化；这种情况应回流需求域或对应实现位点。
- 不默认所有结构变更都能在一次迁移里安全完成，必要时应拆分多步迁移。
- **必须确保 DDL 定义中包含数据类型、默认值、是否需要索引、CHARSET=utf8mb4、ENGINE=InnoDB、注释说明等，不要遗漏**。
- **必须确保 SQL 标识符（表名、字段名、索引名）统一使用反引号包裹，避免裸写；例如 `enabled`。**
- **必须确保所有金额字段使用 VARCHAR 或 CHAR 字符串类型**。
- **必须确保所有表包含 created_at 和 updated_at 字段**。
- **必须确保包含冗余毫秒级时间戳的创建时间字段**。
- **必须确保包含逻辑删除字段**。

## 需要暂停并确认的条件

- 当前 DDL 会删除字段、缩窄类型、修改主键或直接重写大表结构，兼容风险高。
- 当前迁移同时承担结构变更、数据修复和业务开关切换，多重职责混杂。
- 回滚方案不清，或回滚后会造成数据不可逆丢失。
- 结构变更会影响旧版本代码、历史接口或上下游系统，但兼容窗口尚未确认。
- DDL 定义中缺少数据类型、默认值、索引、CHARSET=utf8mb4、ENGINE=InnoDB、注释说明等。
- SQL 标识符存在裸写（如 enabled 未写成 `enabled`）。
- 金额字段使用 DECIMAL、DOUBLE、FLOAT、INT 等数值类型。
- 缺少 created_at 和 updated_at 字段，或定义不正确。
- 缺少冗余毫秒级时间戳的创建时间字段。
- 缺少逻辑删除字段，或定义不正确。

## 执行通过 / 驳回标准

- 通过：结构变更边界清楚，迁移顺序合理，兼容与回滚策略可解释，且没有把查询实现或业务逻辑混入 schema 层；DDL 定义中包含数据类型、默认值、是否需要索引、CHARSET=utf8mb4、ENGINE=InnoDB、注释说明等，没有遗漏；SQL 标识符统一使用反引号包裹（如 `enabled`）；所有金额字段使用 VARCHAR 或 CHAR 字符串类型；所有表包含 created_at 和 updated_at 字段；包含冗余毫秒级时间戳的创建时间字段；包含逻辑删除字段。
- 驳回：危险 DDL 无兼容保护、回滚不可解释、结构变更与数据逻辑混杂，或直接引入高风险不可逆变更；DDL 定义中缺少数据类型、默认值、索引、CHARSET=utf8mb4、ENGINE=InnoDB、注释说明等，会导致自动创建表出现不可控的因素；SQL 标识符裸写（例如 `enabled` 未加反引号）；金额字段使用数值类型；缺少 created_at、updated_at、冗余毫秒级时间戳或逻辑删除字段。

## 执行结果归档要求

- 将 schema 变更结论记录到迁移说明、任务记录、评审记录或部署说明中。
- 归档内容至少包含变更对象、迁移顺序、兼容策略、回滚方式、影响范围和风险说明。
- 如果涉及不可逆或高风险迁移，必须明确标记执行前提、观察点和人工确认条件。
- 确保 DDL 定义的完整说明已记录，包括数据类型、默认值、索引、CHARSET=utf8mb4、ENGINE=InnoDB、注释说明等。
- 确保 SQL 标识符反引号规范已记录并执行（表名、字段名、索引名统一反引号包裹）。
- 确保金额字段、时间字段、逻辑删除字段的使用规范已记录。
